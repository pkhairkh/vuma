# VUMA Gold Standard Test Suite ŌĆö Summary

> Generated by Task 12-d as the final report of the 48-wave gold-standard
> testing effort. All numbers below are sourced from the in-tree result

## Overview

- **Total test programs:** 704 in the gold-standard suite (+ 47 original
  `examples/*.vuma` = **751 `.vuma` programs** project-wide).
- **Categories:** 16
- **Backends tested:** 8 ŌĆö `x86_64`, `AArch64`, `RISC-V 64`, `ARM32`,
  `MIPS64LE`, `PPC64`, `LoongArch64`, `WASM32`
  (7 of the 8 are exercised under QEMU user-mode in CI; `WASM32` is
  covered by the Rust integration test `vuma/src/tests/src/wasm_validation.rs`
  because it has no native execution path).
- **Testing infrastructure:** 9 binaries / tools
  - `compile_dump` ŌĆö compile & run a single `.vuma` program (with optional
    `diag` mode that emits a per-program pass/fail verdict).
  - `differential_test` ŌĆö compile every program against 7 backends and
    flag exit-code / stdout disagreements.
  - `opt_level_test` ŌĆö compile each program at `O0` and `O3` and diff the
    behaviour (exit code + stdout).
  - `test_harness` ŌĆö master orchestrator that wires the above together.
  - `vuma_fuzzer` ŌĆö Csmith-equivalent typed expression generator.
  - `fuzzer_stmt_gen` ŌĆö statement-level generator that emits valid
    straight-line `.vuma` bodies.
  - `fuzzer_mem_gen` ŌĆö memory-operation + function generator
    (alloc/store/load/free patterns).
  - `fuzz_driver` ŌĆö integrated driver that seeds the three generators and
    runs N random programs through `compile_dump`.
  - `property_tests` ŌĆö IVE safety property tests (double-free, leak,
    use-after-free, buffer overflow, etc.).
- **Source layout:** `tests/gold_standard/<category>/*.vuma` plus
  per-category `README.md`, top-level `manifest.json`, `README.md`,
  `RUN_TESTS.md`, and the baseline result snapshots
  (`results_baseline.txt`, `differential_results.txt`,
  `differential_raw.tsv`, `o0_o3_results.txt`).
- **CI entry points:** [`scripts/ci_run_tests.sh`](../../scripts/ci_run_tests.sh)
  invoked by [`.github/workflows/vuma-tests.yml`](../../.github/workflows/vuma-tests.yml).

## O0 vs O3 Comparison

Source: Task 11-d, `o0_o3_results.txt`. 97 programs (47 original examples +
50 stratified gold-standard sample), x86_64 native, 3 s per-execution
timeout, comparing exit code + stdout.

| Outcome                | Original 47 | GS sample 50 | Combined 97 |
|------------------------|------------:|-------------:|------------:|
| Pass (O0 == O3)        | 12 (25.5%)  |  7 (14.0%)   | 19 (19.6%)  |
| Miscompilation         | 19 (40.4%)  | 35 (70.0%)   | 54 (55.7%)  |
| Crash asymmetry        |  9 (19.1%)  |  4 (8.0%)    | 13 (13.4%)  |
| Timeout asymmetry      |  7 (14.9%)  |  4 (8.0%)    | 11 (11.3%)  |
| Compile failures       |  0          |  0           |  0          |

- **O3 miscompilation rate:** 55.7 % (54/97) when both levels run; if you
  also count crash/timeout asymmetries the overall **any-divergence rate
  is 80.4 %** (78/97).
- The 36/54 miscompilations where O3 returns the wrong exit code while
  O0 returns the right one are dominated by the return-value
  propagation bug at O3 (MOV-to-RAX dropped/clobbered). 17/54 produce
  suspicious multiples-of-16 exit codes (16, 32, 48, 64, 80, 96, 112,
  128, ŌĆ”) suggesting a stack-slot or register-allocation bug on
  struct/pointer store patterns.
- **Recommendation:** **O3 should be disabled for release builds** until
  the four identified optimizer bugs are fixed. The O0 path is the only
  reliable compilation path today.

## Differential Testing

Source: Task 9-a/9-b, `differential_results.txt`. 100 programs ├Ś 7
backends (x86_64 native; arm32, mips64, aarch64, riscv64, ppc64,
loongarch64 under QEMU) = 700 differential runs, plus 100 ├Ś 2 O0/O3
runs = 900 total runs.

| Metric                                  | Value |
|-----------------------------------------|------:|
| Total programs swept                    | 100   |
| All 7 backends agree (exit code)        | 58 / 100 = **58.0 %** |
| Differential failure (exit/stdout differ) |  7 / 100 = 7.0 % |
| Crash on at least 1 backend             | 13 / 100 = 13.0 % |
| Timeout on at least 1 backend           | 22 / 100 = 22.0 % |
| O0-vs-O3 agreement (x86_64)             | 11 / 100 = 11.0 % |

- **Cross-backend agreement rate: 58 %.** The bulk of disagreements come
  from the universal codegen gaps (control-flow body assignment drop,
  atomic bridge, store-loaded-variable) which produce different
  wrong-but-stable exit codes per backend rather than crashes.
- **Most common disagreement:** exit code 0 on most backends vs. exit
  218 / 37 / 255 from one or two backends hitting a guest fault
  captured by QEMU. Examples: `arith2_abundant_number`,
  `arith2_catalan`, `arith2_chinese_remainder`, `cf2_if_assign_var`,
  `cf2_if_else_assign`.
- The 11 % O0-vs-O3 agreement rate (lower than the cross-backend
  agreement rate) confirms that the **O3 optimizer is the single
  largest source of incorrect program behaviour today** ŌĆö bigger than
  any backend-specific gap.

## Bugs Found

The gold-standard suite + differential tester + O0/O3 comparator + IVE
property tests + fuzzer together surfaced the following bugs (each is
cross-referenced to the worklog Task ID that documented it):

1. **If-body assignment propagation** ŌĆö assignments inside an `if`/`else`
   body do not always propagate to the outer scope (Task 6-e).
2. **For/while loop body assignment propagation** ŌĆö same class for loop
   bodies; affects ~32 control_flow tests + every nested_loops test
   (Task 6-e).
3. **Immediate-RHS bitwise op drop** ŌĆö `x & 0xFF` etc. silently drop
   the immediate operand; workaround is to load the immediate into a
   variable first (Task 7-d).
4. **Store-loaded-variable drop** ŌĆö `*ptr = loaded_var` silently drops
   the store; `*ptr = loaded_var + 0` works (Task 7-d).
5. **Hex-literal store bug** ŌĆö `*ptr = 0xNN` followed by a load
   sometimes returns 0 (Task 7-d).
6. **First-allocation overwrite bug** ŌĆö the first `allocate` in a
   program can be silently overwritten by subsequent allocations
   (Task 7-d).
7. **SCG-to-codegen atomic bridge gap** ŌĆö `atomic_store` /
   `atomic_load` / `atomic_cas` are silently dropped during SCGŌåÆIR
   lowering (Task 6-d / 7-d).
8. **Dynamic-size `allocate()` lowering** ŌĆö `allocate(<runtime expr>)`
   lowers to `IRInstr::Alloc{size:0}` instead of
   `Call("__vuma_alloc", expr)` (Task 6-d / 11-c).
9. **Struct-literal store lowering** ŌĆö `*ptr = StructLiteral{ŌĆ”}`
   lowers to a single-byte store of `Immediate(0)` instead of
   per-field stores (Task 7-d / 11-c).
10. **O3 optimizer miscompilations** ŌĆö 80 %+ of programs miscompile
    at O3 (Task 11-d). Dominant sub-bugs: return-value MOV-to-RAX
    dropped/clobbered, syscall argument registers clobbered,
    struct/pointer field-offset computation wrong (multiples-of-16
    exit codes), calling-convention save/restore broken for
    value-returning functions.
11. **Atomic bridge gap (same as #7)** ŌĆö listed separately in the
    task brief; unified here.
12. **IVE false positive on valid programs** ŌĆö the SCG builder does
    not always populate `Deallocation.allocation_node`, so the
    `LivenessVerifier` reports a spurious leak on valid programs
    that call `free` (Task 10-a).
13. **ARM64 STR/STRB/STRH/LDRSW decode bit-pattern bug** ŌĆö decode
    patterns matched the wrong opc field, causing round-trip
    failures (Task 1-c, fixed).
14. **x86_64 Select isel test assertion** ŌĆö assertion was too strict
    on the REX prefix encoding for high-register-file operands
    (Task 1-c, fixed).
15. **scg_to_ir `resolve_expr` silent-zero soundness hole** ŌĆö
    `resolve_expr` returned `Ok(Immediate(0))` for unknown variables
    instead of `Err(UnknownVariable)` (Task 1-b, fixed).
16. **`resolve_phis` removing phi nodes** ŌĆö phi nodes were stripped
    after SSA destruction, breaking `control_flow.rs` loop trip-count
    analysis and the `test_*_phi_*` tests (Task 1-b, fixed).
17. **LoongArch64 broken call/return path** ŌĆö 21/47 crashes and 21/47
    timeouts on the original examples; function-call path is broken
    on this backend specifically (Task 8-a, open).
18. **PPC64 Address-typed-return calling-convention bug** ŌĆö 7/47
    timeouts; `*_func_load`, `struct2_func_load`, `*_address_return`
    tests exit 0 because the Address return is dropped (Task 9-b,
    open).
19. **MIPS64 big-endian ELF vs little-endian QEMU** ŌĆö the mips64
    backend emits BE MIPS64 ELF but only `qemu-mips64el` (LE) is
    available; QEMU rejects the binaries (Task 9-b, open).
20. **`compile_dump` exec-bit inconsistency** ŌĆö direct path wrote
    binaries 0o644 (no exec bit) while diag path wrote 0o755;
    QEMU user-mode behaves differently depending on the exec bit,
    producing apparent non-determinism (Task 11-c, open).

## IVE Property Tests

Source: `tests/property_tests.rs` and Task 10-a worklog. The IVE
property-test suite exercises the Invariant Verification Engine with
programs that are deliberately buggy and checks whether IVE flags them.

| Property                  | IVE status                                |
|---------------------------|-------------------------------------------|
| Double-free               | Ō£ō **Detected** (Cleanup invariant)        |
| Memory leak (no `free`)   | Ō£ō **Detected** (Liveness invariant)       |
| Use-after-free            | Ō£Ś Not detected (known gap)                |
| Buffer overflow (read)    | Ō£Ś Not detected (known gap)                |
| Buffer overflow (write)   | Ō£Ś Not detected (known gap)                |
| Null pointer deref        | Ō£Ś Not detected (known gap)                |
| Uninitialized memory      | Ō£Ś Not detected (known gap)                |
| Valid program w/ `free`   | Ō£Ś **False positive** (spurious "leak")    |

The false-positive on valid programs is caused by the SCG builder not
always populating `Deallocation.allocation_node`, so `LivenessVerifier`
cannot see the matching free. The Cleanup invariant does see the free,
so a fix likely involves propagating the allocation-node link from the
Cleanup pass back into the Liveness pass.

## How to Run

```bash
# Build all tools (one-time per checkout)
cd /tmp/my-project
cargo build --release

# Run all tests (CI happy path ŌĆö ~2 minutes on x86_64)
./scripts/ci_run_tests.sh

# Run a single category on the reference backend
for f in tests/gold_standard/bitwise/*.vuma; do
    ./target/release/compile_dump "$f" /tmp/t.bin x86_64 2>/dev/null
    chmod +x /tmp/t.bin
    timeout 3 /tmp/t.bin
    echo "$(basename $f): exit=$?"
done

# Run on a non-x86 backend (QEMU user-mode required)
./target/release/compile_dump diag aarch64 \
    tests/gold_standard/bitwise /tmp/qemu_bins/qemu-aarch64

# Run differential testing across all 7 backends
./target/release/differential_test

# Run O0-vs-O3 comparison on a directory
./target/release/opt_level_test tests/gold_standard/arithmetic 3

# Run the fuzzer
./target/release/fuzz_driver --count 100 --seed 42
```

## Pass / Fail Criteria (Reminder)

- **Pass (any)** ŌĆö binary ran to completion without crash or timeout
  (exit code not in {124, 134, 136, 139}). This is `compile_dump diag`'s
  own definition.
- **Pass (strict)** ŌĆö exit code matches the file's
  `Expected exit code: N` header comment (or the file has no such
  header and ran without crash/timeout). This is what the per-category
  README baseline tables report.
- **Crash** ŌĆö process killed by a signal (`SIGSEGV=139`, `SIGABRT=134`,
  `SIGFPE=136`).
- **Timeout** ŌĆö `timeout 3` killed the process (exit 124).
- **CompileFail** ŌĆö `compile_dump` failed to produce a binary at any
  pipeline stage (parse / SCG / IR / regalloc / encode).
- **WrongExit** ŌĆö clean run whose exit code does not match the
  documented expected value.

## Next Actions

1. **Disable O3 for release builds.** Add a build-profile guard that
   forces `OptLevel::O0` until the four identified optimizer bugs are
   fixed (Task 11-d recommendation).
2. **Fix the universal codegen gaps** (Tasks 6-d, 6-e, 7-d): for/while
   body assignment propagation, atomic bridge, store-loaded-variable,
   hex-literal store, first-allocation overwrite, dynamic-size
   `allocate()`, struct-literal store. Fixing these will lift the
   x86_64 strict-pass rate from 76.5 % to >95 % and resolve most
   cross-backend differential failures in one stroke.
3. **Fix the IVE `Deallocation.allocation_node` population** so the
   Liveness invariant sees the matching free (eliminates the false
   positive on valid programs).
4. **Close the IVE detection gaps** for use-after-free, buffer overflow,
   null-deref, and uninitialized-memory detection ŌĆö these are the
   properties the IVE was designed to enforce but currently does not.
5. **Fix the LoongArch64 call/return path** ŌĆö currently the worst
   backend by a wide margin (5/47 strict pass on the original
   examples).
6. **Fix the PPC64 Address-typed-return calling convention** ŌĆö 7/47
   timeouts concentrated on programs that return `Address` from a
   function.
7. **Provide a `qemu-mips64` (big-endian) binary or switch the mips64
   backend to emit little-endian MIPS64 ELF** so the MIPS numbers in
   the differential tester reflect real execution.
8. **Re-run the full 704-program baseline** after the universal
   codegen fixes land and update `results_baseline.txt` and the
   pass-rate tables in `README.md` and this `SUMMARY.md`.
9. **Re-run the O0/O3 comparison** after the optimizer fixes land and
   confirm the miscompilation rate drops below 5 %.
10. **Promote the 54 O3-miscompiling programs + 13 crash-asymmetry +
    11 timeout-asymmetry programs to regression tests** against the O3
    optimizer; they should all PASS once O3 is fixed.
