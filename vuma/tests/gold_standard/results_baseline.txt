HOW TO READ
============================================================
- "Pass (any)"        = binary ran to completion without crash or timeout
                       (exit code need not match the documented "Expected exit code")
- "Pass (strict)"     = exit code matches the file's "Expected exit code: N" header,
                       OR the file has no such header and ran without crash/timeout
- "Crash"             = process killed by signal (SIGSEGV=139, SIGABRT=134, SIGFPE=136, etc.)
- "Timeout"           = `timeout 3` killed the process (exit 124)
- "CompileFail"       = compile_dump could not produce a binary (parse / SCG / IR / regalloc / encode)
- "WrongExit"         = binary ran cleanly but the exit code does NOT match the documented expected value

Native QEMU backends:  x86_64 (native), aarch64, riscv64, arm32, mips64 (qemu-mips64el),
                      ppc64, loongarch64.  Wasm32 skipped (no native execution).

Notable findings (x86_64 reference backend):
  * 100% pass (any) on 9/16 categories (bitwise, complex_stores, control_flow,
    crypto_patterns, functions, linked_structures, memory, multi_function,
    nested_loops, u32_arith).
  * 100% pass (strict) on 7/16 categories (bitwise, complex_stores, functions,
    linked_structures, multi_function, nested_loops, u32_arith).
  * atomics: 4/32 strict pass — all 27 wrong-exit cases return exit 0 instead of
    the expected nonzero value. This is the documented SCG-to-codegen bridge gap
    for atomic_store / atomic_load / atomic_cas (worklog Task 6-d / 7-d): atomic
    stores are silently dropped, so the post-atomic read returns 0.
  * control_flow: 15/47 strict pass — all 32 wrong-exit cases return 0 (or some
    other small value) instead of the documented sum/count. This is the
    documented "for/while body assignment propagation" bug (worklog Task 6-e):
    variables assigned inside loop bodies do not propagate back to the outer
    scope.
  * memory: 49/70 strict pass — the 21 wrong-exit cases all return 0 instead of
    the documented post-store value. This is the documented "store-loaded-
    variable" bug (worklog Task 7-d): `*ptr = loaded_var` silently drops the
    store, so subsequent loads return the cell's previous (zero) value.
  * pointers: 23/33 strict pass — same store-loaded-variable bug plus two
    pointer-deref crashes (ptr_multi_level, ptr_pointer_to_pointer).
  * structs: 25/32 strict pass — same store-loaded-variable bug surfaces in
    struct field-update patterns (struct_swap_fields returns 2 instead of 1
    because the swap is silently dropped — worklog Task 7-d confirmed this).
  * atomics_demo (atomics) and self_exec, thread_pool (concurrency) and
    float_math (edge_cases) and struct_demo (structs) and arith_lcm (arithmetic)
    all CRASH on x86_64 (sig 11). float_math is a known FP intrinsic gap
    (worklog Task 1). self_exec / thread_pool fork/spawn and are expected to
    interact poorly with the minimal VUMA runtime. atomics_demo and struct_demo
    crashes need investigation.

KEY FINDINGS (read alongside worklog Tasks 6-d, 6-e, 7-d, 7-c)
============================================================
1. Universal codegen gaps (hit by all 7 backends uniformly):
   a. for/while body assignment propagation: variables assigned inside a loop
      body do not propagate to the outer scope. Affects ~32 control_flow tests
      on x86_64 and 3/3 sampled control_flow tests on every backend.
   b. SCG-to-codegen atomic bridge: atomic_store / atomic_load / atomic_cas are
      silently dropped. Affects ~27 atomics tests on x86_64 and 3/3 sampled
      atomics tests on every backend.
   c. Store-loaded-variable: `*ptr = loaded_var` silently drops the store;
      `*ptr = loaded_var + 0` works. Affects ~21 memory tests and ~8 pointer
      tests on x86_64, plus shared structs/struct_swap_fields.

2. Backend-specific gaps:
   a. aarch64: 14/47 examples crash (sig 11), 8/47 timeout. The crashes are
      NON-DETERMINISTIC: re-running the same binary gives different exit codes
      on different runs (exit=1 vs exit=139 for arena_allocator). Indicates
      uninitialized-memory or ASLR-sensitive codegen.
   b. loongarch64: 21/47 examples crash (mix of sig 11 and sig 5 = trap),
      21/47 timeout. Function call / return path appears broken: 0/3 sampled
      functions tests pass, 0/3 sampled multi_function tests pass.
   c. ppc64: 7/47 timeouts (atomics_demo, ffi_demo, float_math,
      lock_free_queue, test_print, test_print2, thread_pool). No crashes.
   d. riscv64: 1/47 timeout (lock_free_queue only). Otherwise solid.
   e. x86_64, arm32, mips64: 47/47 examples pass. The three most stable
      backends. On the 50-test gold_standard sample, x86_64/riscv64/mips64/ppc64
      all hit 37/50 strict pass; arm32 hit 35/50.

3. Differential agreement:
   - 0% on the original 47 examples (every example diverges across backends on
     exit code or stdout). Most of the divergence is `exit=0` on some backends
     vs. `exit=255` (qemu's "target did something bad") on others — i.e., a
     mix of "ran cleanly returning 0" and "QEMU captured a guest fault and
     returned 255".
   - 52% on the 50-test gold-standard sample (which contains many more
     straight-line tests that avoid the universal codegen gaps).

4. Documentation/manifest drift: the gold_standard/README.md says 47 programs
   across 16 categories; the manifest.json says 77. Actual at run time: 527
   programs across 16 categories. The README/manifest should be regenerated.

5. Test-file growth during the run: 467 -> 497 -> 512 -> 527 files observed
   over ~20 minutes as concurrent test-generation waves (Tasks 7-c through 7-e
   and beyond) added new files in arithmetic/, bitwise/, control_flow/, memory/,
   and pointers/.
