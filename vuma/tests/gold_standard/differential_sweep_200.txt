=== Extended Analysis ===

Agreement rates (various groupings):
- All 7 backends agree:           140 / 208 = 67.3%
- 6 backends agree (excl Loong):  187 / 208 = 89.9%
- 5 backends agree (excl ARM32 + Loong): 193 / 208 = 92.8%
- 5 mature backends (x86_64, mips64, aarch64, riscv64, ppc64) agree: 193 / 208 = 92.8%

Disagreement root-cause categories:

1. LoongArch64 broken call/return path (47 cases): Every program that
   makes a function call (and many that don't) either times out (124)
   or segfaults (139) on LoongArch64. This is the dominant failure
   pattern — 47 of 68 disagreements (69%).
   - 28 timeouts (exit 124) — program entered an infinite loop or
     stalled in the broken call/return path.
   - 13 segfaults (exit 139) — broken stack frame / return address.
   - 5 nested-loop programs (nl_diagonal=3, nl_floyd=1, nl_identity=4,
     nl_pascals=2, nl_outer_product=4) returned DIFFERENT non-zero
     values than the other backends' 0. These tests document the
     "for-body assignment drop" bug; LoongArch64 seems to execute the
     for body partially, producing small values that differ from x86_64's
     (also buggy) 0.
   - 1 case (crypto2_bit_reverse) returned 192 vs others' 240.

2. ARM32 division/modulo bug (8 cases): ARM32 returns 218 for any
   u32/u8/u64 division or modulo. Confirmed by direct minimal tests:
     - `4/4` (u32) → x86_64 returns 1, ARM32 returns 218
     - `22%15` (u32) → x86_64 returns 7, ARM32 returns 218
     - `4/4` (u8) → ARM32 returns 218
     - `4/4` (u64) → ARM32 returns 218
   The 218 result is consistent across types and operands — it appears
   to be a fixed value emitted by the ARM32 backend for div/mod ops
   (possibly an unimplemented-instruction trap return value or a
   compiler-inserted placeholder). 7 of the 8 affected tests are
   arith2_* programs that use div/mod for primality, modular-inverse,
   CRT, etc. arith2_happy_number returns 94 instead of 1 — likely
   the same div/mod bug surfacing as an intermediate computation
   producing a wrong sum.

3. cf2_if_*_assign tests (3 cases: cf2_if_assign_var, cf2_if_else_assign,
   cf2_if_nested_assign): Only x86_64 and ppc64 return the "expected"
   non-zero value. ARM32, mips64, aarch64, riscv64, loongarch64 all
   return 0. This is the inverse of pattern #1 — here x86_64+ppc64
   are in the minority. The 5 returning-0 backends all suffer from
   the same if-body assignment-drop bug; x86_64 and ppc64 happen to
   propagate the assignment. This is a documented VUMA bug; the
   "correct" answer per the test headers is the non-zero value.

4. AArch64 + RISC-V matrix/nested-matrix bugs (2 cases each):
   - matrix: x86_64=mips64=arm32=ppc64=0, but aarch64=144, riscv64=176
   - nl_matrix_mul: same pattern — aarch64=32, riscv64=64
   The aarch64 and riscv64 backends produce non-zero (and distinct)
   values for 2D-matrix nested loops where x86_64/mips64/ppc64 produce
   0. This suggests both backends partially execute the inner loop
   body (whereas the others' inner-body assignments get dropped
   silently). aarch64 and riscv64 give DIFFERENT non-zero values,
   so they disagree with each other too — neither matches x86_64.

5. ARM32 bitwise bug (1 case): bit2_priority_encode returns 0 on ARM32
   and mips64, but 7 on x86_64/aarch64/riscv64/ppc64. Isolated
   bit-manipulation discrepancy.

6. PPC64 quirks (5 cases):
   - fibonacci: ppc64 returns 1 instead of 0 (off-by-one — likely
     the same loop-body-drop bug but with different propagation).
   - doubly_linked_list + arena_allocator: ppc64 times out (124)
     instead of crashing (139) like the others. Both are documented
