VUMA O0 vs O3 Comparison Results
Date: Sat Jun 20 12:27:39 UTC 2026
Tool: ./target/release/opt_level_test (Task 9-b artifact)
Backend: x86_64, native execution under `timeout 3`
Pipeline: parser -> SCG -> SCG transforms (OptLevel) -> IR -> regalloc -> encode -> run

Methodology
-----------
For each .vuma program, compile twice with the same pipeline differing only in
OptLevel (O0 vs O3), execute both ELFs natively, and compare exit code + stdout.
A program PASSES iff O0 and O3 produce identical exit code AND identical stdout.
"Miscompilation" = both levels ran to completion but exit code or stdout differ.
"Crash asymmetry"  = exactly one level crashed (segfault / abort / killed by signal).
"Timeout asymmetry" = exactly one level hit the 3s wall-clock limit (timeout returns 124).

Two independent runs were performed:
  Run A: 47 original example programs from /tmp/my-project/examples/
  Run B: 50 programs sampled stratified from /tmp/my-project/tests/gold_standard/
         (3 from each of the 16 categories = 48, plus 2 extra from `arithmetic`
         to reach 50; selected via alphabetical sort within each category)
Total: 97 programs compared.

=== Section 5: Pattern analysis ===

1. O3 frequently loses the program's intended return value.
   Of the 54 miscompilations, 36 (~67%) show O3 exit=0 where O0 produced a
   non-zero return value (e.g. test_exit: O0=42 -> O3=0; debug_info: O0=35
   -> O3=0). This is consistent with the O3 register allocator or
   instruction selector either dropping the final MOV of the return-value
   register, or clobbering %rax after main() sets it.

2. O3 frequently emits "suspicious" exit codes that are multiples of 16.
   17 of the 54 miscompilations have O3 exit codes in {16, 32, 48, 64, 80,
   96, 112, 128, 144, 160, 192, 208, 240}. These look like raw byte values
   extracted from a mis-aligned stack slot or a clobbered register -- not
   like program-logic return values. Examples:
     hello_memory: O0=42, O3=144   (gold)   / O3=48    (examples)
     struct2_func_load: O0=42, O3=96
     struct2_func_store: O0=42, O3=112
     cs2_after_alloc:   O0=42, O3=160
     cs2_before_free:   O0=42, O3=208
     ls_copy:           O0=42, O3=192

3. O3 frequently breaks stdout emission for print syscalls.
   test_print and test_print2 are the only STDOUT-DIFFERS cases (vs the
   EXIT-CODE-DIFFERS pattern in the others): O0 prints the expected bytes
   ("42\n0\n12345\n"), O3 emits nothing. This indicates O3 is dropping the
   syscall instruction or clobbering its argument registers.

4. Asymmetries skew overwhelmingly toward O0 being broken, not O3.
   Of the 13 crash asymmetries, 12 are O0-crash-only and 1 is O3-crash-only.
   Of the 11 timeout asymmetries, 11 are O0-timeout-only and 0 are
   O3-timeout-only.
   -> O3 actually FIXES more crash/timeout cases than it introduces.
   -> BUT, O3 introduces dramatic semantic regressions on the 54 programs
      that DO run to completion at both levels (wrong exit codes, lost
      stdout, suspicious "multiple-of-16" exits).

5. Categories that consistently pass:
   - simple constant-return programs (minimal, edge2_empty_func)
   - plain counted loops over local vars (cf2_for_count, cf2_for_sum,
     nl_2x2, nl_3x3, fibonacci)
   - programs whose output is purely an exit code in the small-constant
     range that happens to round-trip through both opt levels.
   Categories that consistently fail:
   - anything with a function call that returns a value (mf_*, fn2_*)
   - struct field stores (struct_*, cs2_*)
   - atomics (atom_*)
   - pointer arithmetic / store+load round trips (ptr2_*, test_endian,
     hello_memory)
   - syscalls (test_print, test_print2, test_hex)

diverse / extreme language features).


=== Recommendation ===
The O3 optimizer has a 80.4% miscompilation rate on the combined 97-program
suite (78/97 programs produce different observable behaviour at O3 than at
O0, including miscompilations, crash asymmetries, and timeout asymmetries).
On the strict "both levels ran, but output differs" definition, the rate
is 55.7% (54/97).

The O3 path is NOT production-safe. It either must be:
  (a) disabled entirely (force OptLevel::O0 for all release builds until
      fixed), OR
  (b) rewritten with a proper correctness proof / differential test gate
      before being re-enabled.

The O0 path is the only reliable compilation path. While O0 itself still
has 12 crash-only and 11 timeout-only failures (out of 97 programs), all
of those failures are programs that exercise bleeding-edge features
(FFI, atomics syscalls, linked-structure traversals) where O0 produces
an ELF that segfaults natively -- the O3 optimizer happens to elide the
crashing instruction sequence but at the cost of producing wrong output.

Specific bugs to investigate first (high signal-to-noise):
  1. Return-value propagation through register allocator at O3: 36+
     programs show O3 exit=0 where O0 returned a meaningful value.
     Likely a missing MOV-to-RAX at function exit, or RAX clobbered by
     a dead-store elimination pass.
  2. Syscall argument registers clobbered at O3: test_print/test_print2
     emit no stdout at O3 but correct stdout at O0.
  3. Struct field offset computation at O3: the suspicious
     multiples-of-16 exit codes (16/32/48/64/80/96/112/128/144/160/192/
     208/240) consistently appear on struct/pointer-store programs at
     O3, suggesting the field offset is being read from the wrong
     stack slot or computed in bytes-vs-dwords incorrectly.
  4. Calling-convention save/restore at O3: fn2_*, mf_*, struct2_func_*
     and similar function-call-returning-value programs fail at O3
     but pass at O0, indicating a callee-saved register or stack-cleanup
     bug in the O3 prologue/epilogue.
