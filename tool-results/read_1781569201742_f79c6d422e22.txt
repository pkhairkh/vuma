    32в†’## 1. Quick Start
    33в†’
    34в†’A VUMA program consists of one or more function definitions. The entry point is always `fn main()`. The compiler produces a native binary (or Wasm module) that can be executed directly or via QEMU for cross-architecture testing.
    35в†’
    36в†’### Minimal Program
    37в†’
    38в†’```vuma
    39в†’fn main() -> i32 {
    40в†’    return 0;
    41в†’}
    42в†’```
    43в†’
    44в†’This program exits with code 0.
    45в†’
    46в†’### Compile and Run
    47в†’
    48в†’VUMA compiles `.vuma` source files to native binaries. The build system uses Cargo under the hood. To build and test:
    49в†’
    50в†’```bash
    51в†’# Build the VUMA compiler
    52в†’cargo build --workspace
    53в†’
    54в†’# Run all tests
    55в†’cargo test --workspace
    56в†’
    57в†’# Cross-architecture testing uses QEMU
    58в†’# x86_64:
    59в†’qemu-x86_64 ./output_binary
    60в†’# AArch64:
    61в†’qemu-aarch64 ./output_binary
    62в†’# RISC-V 64:
    63в†’qemu-riscv64-static ./output_binary
    64в†’# ARM32:
    65в†’qemu-arm ./output_binary
    66в†’# MIPS64:
    67в†’qemu-mips64-static ./output_binary
    68в†’# PPC64:
    69в†’qemu-ppc64 ./output_binary
    70в†’# LoongArch64:
    71в†’qemu-loongarch64 ./output_binary
    72в†’```
    73в†’
    74в†’### Program with Memory
    75в†’
    76в†’```vuma
    77в†’fn main() -> i32 {
    78в†’    msg = allocate(3);
    79в†’    *(msg + 0) = 97;   // 'a'
    80в†’    *(msg + 1) = 98;   // 'b'
    81в†’    *(msg + 2) = 99;   // 'c'
    82в†’    free(msg);
    83в†’    return 0;
    84в†’}
    85в†’```
    86в†’
    87в†’**Key facts**:
    88в†’- `main()` returns `i32` (the process exit code) or `u64`.
    89в†’- Every `allocate()` must be paired with a `free()`.
    90в†’- Pointer dereference uses `*(ptr + offset)` syntax.
    91в†’- Bytes are stored as integer values (e.g., `97` for ASCII `'a'`).
    92в†’

    95в†’## 2. Types
    96в†’
    97в†’VUMA supports the following primitive types:
    98в†’
    99в†’| Type | Size (bytes) | Range | Signed |
   100в†’|------|-------------|-------|--------|
   101в†’| `i8`  | 1 | -128 to 127 | Yes |
   102в†’| `i16` | 2 | -32768 to 32767 | Yes |
   103в†’| `i32` | 4 | -2147483648 to 2147483647 | Yes |
   104в†’| `i64` | 8 | -9223372036854775808 to 9223372036854775807 | Yes |
   105в†’| `u8`  | 1 | 0 to 255 | No |
   106в†’| `u16` | 2 | 0 to 65535 | No |
   107в†’| `u32` | 4 | 0 to 4294967295 | No |
   108в†’| `u64` | 8 | 0 to 18446744073709551615 | No |
   109в†’| `Address` | 8 (4 on 32-bit targets) | N/A (opaque pointer) | No |
   110в†’| `f32` | 4 | IEEE 754 single-precision | N/A |
   111в†’| `f64` | 8 | IEEE 754 double-precision | N/A |
   112в†’| `bool` | 1 | `true` or `false` | N/A |
   113в†’| `void` | 0 | N/A | N/A |
   114в†’
   115в†’### Type Annotation Syntax
   116в†’
   117в†’```vuma
   118в†’x: i32 = 42;
   119в†’y: u32 = 0xFFFFFFFF;
   120в†’ptr: Address = allocate(64);
   121в†’big: u64 = 18446744073709551615;
   122в†’flag: bool = true;
   123в†’pi: f64 = 3.141592653589793;
   124в†’angle: f32 = 1.5707964;
   125в†’```
   126в†’
   127в†’### Critical Constraint: u32 on 64-bit Registers
   128в†’
   129в†’**This is the single most important thing to know about VUMA types.** All 64-bit target platforms (x86_64, AArch64, RISC-V 64, MIPS64, PPC64, LoongArch64) use 64-bit general-purpose registers. A `u32` value stored in a 64-bit register can have garbage bits in the upper 32 bits after arithmetic operations. You **must** manually mask u32 results with `& 4294967295` (which is `0xFFFFFFFF` in decimal) after any operation that could set bits above bit 31:
   130в†’
   131в†’```vuma
   132в†’// WRONG: bits above bit 31 may be set
   133в†’sum: u32 = a + b;
   134в†’
   135в†’// CORRECT: mask to 32 bits after addition
   136в†’sum: u32 = (a + b) & 4294967295;
   137в†’```
   138в†’
   139в†’See [Section 14: Pitfalls](#14-pitfalls) for the complete list of operations that require masking.
   140в†’
   141в†’### Address Type
   142в†’
   143в†’`Address` is the type for pointers and memory regions. It is 8 bytes on 64-bit targets and 4 bytes on 32-bit targets (ARM32, Wasm32). The `allocate()` built-in returns an `Address`.
   144в†’
   145в†’```vuma
   146в†’buf: Address = allocate(256);
   147в†’```
   148в†’

   151в†’## 3. Functions
   152в†’
   153в†’### Function Declaration
   154в†’
   155в†’```vuma
   156в†’fn function_name(param1: type1, param2: type2) -> return_type {
   157в†’    // body
   158в†’    return value;
   159в†’}
   160в†’```
   161в†’
   162в†’- The `fn` keyword introduces a function.
   163в†’- Parameters have mandatory type annotations when declared in the signature.
   164в†’- The return type follows `->`. If omitted, the function returns `void`.
   165в†’- The body is a block enclosed in `{ }`.
   166в†’- Use `return expr;` to return a value. The last expression in a block without `return` is also the return value in some contexts, but `return` is preferred for clarity.
   167в†’
   168в†’### Examples
   169в†’
   170в†’```vuma
   171в†’// Function returning a value
   172в†’fn add(a: i32, b: i32) -> i32 {
   173в†’    return a + b;
   174в†’}
   175в†’
   176в†’// Function with no return value (void)
   177в†’fn store_byte(buf: Address, offset: u64, val: u32) {
   178в†’    *(buf + offset) = val & 255;
   179в†’}
   180в†’
   181в†’// Function with Address parameter
   182в†’fn read_u32_be(buf: Address, offset: u64) -> u32 {
   183в†’    b0: u32 = *(buf + offset);
   184в†’    b1: u32 = *(buf + offset + 1);
   185в†’    b2: u32 = *(buf + offset + 2);
   186в†’    b3: u32 = *(buf + offset + 3);
   187в†’    return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
   188в†’}
   189в†’
   190в†’// Main entry point вЂ” must return i32 or u64
   191в†’fn main() -> i32 {
   192в†’    return 0;
   193в†’}
   194в†’```
   195в†’
   196в†’### Function Call
   197в†’
   198в†’```vuma
   199в†’result: i32 = add(10, 20);
   200в†’store_byte(buffer, 0, 65);
   201в†’```
   202в†’
   203в†’Functions are called by name with parenthesized arguments. Function calls can appear as expressions (when the function returns a value) or as statements (when the function returns void or the return value is discarded).
   204в†’

   207в†’## 4. Variables
   208в†’
   209в†’### Declaration with Type Annotation
   210в†’
   211в†’```vuma
   212в†’name: type = expression;
   213в†’```
   214в†’
   215в†’Variables are declared with a name, a type annotation, and an initial value. The type annotation is required in most cases.
   216в†’
   217в†’```vuma
   218в†’x: i32 = 42;
   219в†’y: u32 = 0xFFFFFFFF;
   220в†’ptr: Address = allocate(64);
   221в†’sum: u32 = (a + b) & 4294967295;
   222в†’```
   223в†’
   224в†’### Declaration Without Type Annotation (Inferred)
   225в†’
   226в†’VUMA can infer the type from the right-hand side in some cases, but explicit annotation is strongly recommended to avoid ambiguity:
   227в†’
   228в†’```vuma
   229в†’result = add1(41);       // type inferred from function return type
   230в†’msg = allocate(3);        // type inferred as Address
   231в†’```
   232в†’
   233в†’**When writing VUMA code as an LLM, always use explicit type annotations** except for variables assigned from `allocate()` (which always returns `Address`).
   234в†’
   235в†’### Assignment
   236в†’
   237в†’Once declared, variables can be reassigned without repeating the type:
   238в†’
   239в†’```vuma
   240в†’x: u32 = 10;
   241в†’x = 20;              // reassignment
   242в†’x = (x + 1) & 4294967295;  // reassignment with expression
   243в†’```
   244в†’
   245в†’### Compound Assignment
   246в†’
   247в†’VUMA supports compound assignment operators:
   248в†’
   249в†’```vuma
   250в†’x += 1;     // x = x + 1
   251в†’x -= 1;     // x = x - 1
   252в†’x *= 2;     // x = x * 2
   253в†’x &= 255;   // x = x & 255
   254в†’x |= mask;  // x = x | mask
   255в†’x ^= key;   // x = x ^ key
   256в†’x <<= 4;    // x = x << 4
   257в†’x >>= 4;    // x = x >> 4
   258в†’```
   259в†’
   260в†’**Note**: Compound assignments on `u32` values still need the `& 4294967295` mask. The compound form does **not** automatically mask. Prefer explicit assignment with masking:
   261в†’
   262в†’```vuma
   263в†’// WRONG for u32 on 64-bit registers:
   264в†’x += 1;
   265в†’
   266в†’// CORRECT:
   267в†’x = (x + 1) & 4294967295;
   268в†’```

   272в†’## 5. Control Flow
   273в†’
   274в†’### For Loop (Range)
   275в†’
   276в†’The `for` loop iterates over an exclusive range `start..end`. The loop variable takes each integer value from `start` (inclusive) to `end` (exclusive).
   277в†’
   278в†’```vuma
   279в†’for i in 0..10 {
   280в†’    // i = 0, 1, 2, ..., 9
   281в†’}
   282в†’
   283в†’for i in 0..64 {
   284в†’    // i = 0, 1, 2, ..., 63
   285в†’}
   286в†’
   287в†’for i in 16..64 {
   288в†’    // i = 16, 17, ..., 63
   289в†’}
   290в†’```
   291в†’
   292в†’**The range is always exclusive on the upper bound**: `0..10` means `0 <= i < 10`.
   293в†’
   294в†’### For Loop with Memory Access
   295в†’
   296в†’```vuma
   297в†’// Copy 32 bytes from src to dst
   298в†’fn copy32(dst: Address, src: Address) {
   299в†’    i: u64 = 0;
   300в†’    for i in 0..32 {
   301в†’        *(dst + i) = *(src + i);
   302в†’    }
   303в†’}
   304в†’```
   305в†’
   306в†’### While Loop
   307в†’
   308в†’```vuma
   309в†’current = (*list).next;
   310в†’while current != list {
   311в†’    process(current);
   312в†’    current = (*current).next;
   313в†’}
   314в†’```
   315в†’
   316в†’### If / Else
   317в†’
   318в†’```vuma
   319в†’if x > 100 {
   320в†’    y = 2;
   321в†’} else {
   322в†’    y = 1;
   323в†’}
   324в†’```
   325в†’
   326в†’```vuma
   327в†’if value & 1 == 1 {
   328в†’    // odd
   329в†’}
   330в†’```
   331в†’
   332в†’Nested conditionals:
   333в†’
   334в†’```vuma
   335в†’if x > 0 {
   336в†’    if x > 100 {
   337в†’        result = 2;
   338в†’    } else {
   339в†’        result = 1;
   340в†’    }
   341в†’} else {
   342в†’    result = 0;
   343в†’}
   344в†’```
   345в†’
   346в†’### Infinite Loop
   347в†’
   348в†’```vuma
   349в†’loop {
   350в†’    // runs forever (useful in embedded/bare-metal contexts)
   351в†’}
   352в†’```
   353в†’
   354в†’### Break and Continue
   355в†’
   356в†’```vuma
   357в†’for i in 0..100 {
   358в†’    if i == 50 {
   359в†’        break;
   360в†’    }
   361в†’    if i % 2 == 0 {
   362в†’        continue;
   363в†’    }
   364в†’    process(i);
   365в†’}
   366в†’```

   370в†’## 6. Memory
   371в†’
   372в†’VUMA provides two first-class memory operations: `allocate` and `free`. There is no garbage collector; memory management is manual.
   373в†’
   374в†’### allocate(n)
   375в†’
   376в†’Reserves `n` bytes of memory and returns an `Address` (pointer to the start of the block). The allocated memory is uninitialized.
   377в†’
   378в†’```vuma
   379в†’buf = allocate(64);     // 64-byte buffer
   380в†’state = allocate(32);   // 32-byte state buffer
   381в†’k_table = allocate(256); // 256-byte constant table
   382в†’```
   383в†’
   384в†’**The returned `Address` does not need a type annotation** вЂ” it is always inferred as `Address`:
   385в†’
   386в†’```vuma
   387в†’state = allocate(32);   // state has type Address
   388в†’```
   389в†’
   390в†’### free(ptr)
   391в†’
   392в†’Releases the memory block pointed to by `ptr`. After `free(ptr)`, the pointer is invalid and must not be used.
   393в†’
   394в†’```vuma
   395в†’free(state);
   396в†’free(k_table);
   397в†’free(buf);
   398в†’```
   399в†’
   400в†’**Every `allocate` must be matched with a `free`**. Failing to free memory is a resource leak. The IVE (Invariant Verification Engine) checks this at compile time.
   401в†’
   402в†’### Memory Layout Example
   403в†’
   404в†’```vuma
   405в†’fn sha256d(msg: Address, msg_len: u64, out: Address) {
   406в†’    // Allocate working memory
   407в†’    state = allocate(32);
   408в†’    k = allocate(256);
   409в†’    w = allocate(256);
   410в†’    block = allocate(64);
   411в†’    inner = allocate(32);
   412в†’
   413в†’    // ... use the memory ...
   414в†’
   415в†’    // Clean up all allocations
   416в†’    free(state);
   417в†’    free(k);
   418в†’    free(w);
   419в†’    free(block);
   420в†’    free(inner);
   421в†’}
   422в†’```
   423в†’
   424в†’### Region Declarations
   425в†’
   426в†’Named regions can be declared with the `region` keyword:
   427в†’
   428в†’```vuma
   429в†’region pool = allocate(4096);
   430в†’```

   434в†’## 7. Pointer Operations
   435в†’
   436в†’### Dereference: Load
   437в†’
   438в†’Read a byte from memory at `ptr + offset`:
   439в†’
   440в†’```vuma
   441в†’b0: u32 = *(buf + offset);
   442в†’```
   443в†’
   444в†’The result is a single byte zero-extended to the type of the variable (typically `u32`).
   445в†’
   446в†’### Dereference: Store
   447в†’
   448в†’Write a byte to memory at `ptr + offset`:
   449в†’
   450в†’```vuma
   451в†’*(buf + offset) = value & 255;
   452в†’```
   453в†’
   454в†’Only the low 8 bits of `value` are stored.
   455в†’
   456в†’### Pointer Arithmetic
   457в†’
   458в†’Pointer + integer produces a new `Address` offset by that many bytes:
   459в†’
   460в†’```vuma
   461в†’ptr = base + 128;      // 128 bytes forward
   462в†’byte3 = *(buf + 3);    // read byte at offset 3
   463в†’*(buf + 4) = 99;       // write byte at offset 4
   464в†’```
   465в†’
   466в†’The offset can be an expression:
   467в†’
   468в†’```vuma
   469в†’*(buf + i * 4) = val;  // write at byte offset i*4
   470в†’byte_val = *(buf + i); // read at byte offset i
   471в†’```
   472в†’
   473в†’### Read/Write Multi-byte Values (u32)
   474в†’
   475в†’VUMA does not have native multi-byte load/store. You must compose byte-level accesses. For big-endian u32:
   476в†’
   477в†’```vuma
   478в†’fn write_u32_be(buf: Address, offset: u64, val: u32) {
   479в†’    *(buf + offset) = (val >> 24) & 255;
   480в†’    *(buf + offset + 1) = (val >> 16) & 255;
   481в†’    *(buf + offset + 2) = (val >> 8) & 255;
   482в†’    *(buf + offset + 3) = val & 255;
   483в†’}
   484в†’
   485в†’fn read_u32_be(buf: Address, offset: u64) -> u32 {
   486в†’    b0: u32 = *(buf + offset);
   487в†’    b1: u32 = *(buf + offset + 1);
   488в†’    b2: u32 = *(buf + offset + 2);
   489в†’    b3: u32 = *(buf + offset + 3);
   490в†’    return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
   491в†’}
   492в†’```
   493в†’
   494в†’**Why the `& 4294967295` mask?** After left-shifting `b0` by 24 bits in a 64-bit register, bits above bit 31 may be set. The mask clears them. This is the most common source of bugs in VUMA code. See [Section 14: Pitfalls](#14-pitfalls).
   495в†’
   496в†’### Address-Of Operator
   497в†’
   498в†’The `@` operator takes the address of a variable:
   499в†’
   500в†’```vuma
   501в†’let x: i32 = 42;
   502в†’ptr = @x;       // ptr has type Address, points to x
   503в†’*ptr = 100;     // modifies x through the pointer
   504в†’```
   505в†’
   506в†’### Derive
   507в†’
   508в†’The `derive(ptr, region)` expression explicitly creates a derived pointer within a specific region. This is the formal mechanism for sub-allocation:
   509в†’
   510в†’```vuma
   511в†’inner = derive(ptr, arena_region);
   512в†’```
   513в†’
   514в†’In practice, pointer arithmetic (`base + offset`) implicitly derives a new pointer.
   515в†’

   518в†’## 8. Bitwise Operations
   519в†’
   520в†’VUMA supports the standard bitwise operations:
   521в†’
   522в†’| Operator | Name | Example |
   523в†’|----------|------|---------|
   524в†’| `&` | Bitwise AND | `a & 0xFF` |
   525в†’| `\|` | Bitwise OR | `a \| b` |
   526в†’| `^` | Bitwise XOR | `a ^ 0xFFFFFFFF` |
   527в†’| `<<` | Left shift | `a << 8` |
   528в†’| `>>` | Right shift | `a >> 4` |
   529в†’
   530в†’### No Built-in NOT Operator
   531в†’
   532в†’**VUMA does not have a reliable bitwise NOT (`~`) operator for u32 values.** The `~x` operator inverts all 64 bits of the register, which corrupts the upper 32 bits. Instead, use XOR with `0xFFFFFFFF`:
   533в†’
   534в†’```vuma
   535в†’// WRONG: ~x inverts bits 32-63 which corrupts u32 values
   536в†’not_x = ~x;
   537в†’
   538в†’// CORRECT: XOR with 0xFFFFFFFF inverts only the lower 32 bits
   539в†’not_x: u32 = x ^ 4294967295;
   540в†’```
   541в†’
   542в†’**Always use `x ^ 4294967295` instead of `~x` for u32 bitwise NOT.**
   543в†’
   544в†’### Bitwise AND for Masking
   545в†’
   546в†’```vuma
   547в†’low_byte: u32 = value & 255;          // mask to 8 bits
   548в†’low_16: u32 = value & 65535;          // mask to 16 bits
   549в†’low_32: u32 = value & 4294967295;     // mask to 32 bits
   550в†’```
   551в†’
   552в†’### Bitwise OR for Combining
   553в†’
   554в†’```vuma
   555в†’combined: u32 = (high << 8) | low;
   556в†’```
   557в†’
   558в†’### Left Shift
   559в†’
   560в†’```vuma
   561в†’shifted: u32 = (value << 24) & 4294967295;  // MUST mask after left shift
   562в†’```
   563в†’
   564в†’**Left shift on u32 values always requires masking** because the result can have bits set above bit 31.
   565в†’
   566в†’### Right Shift
   567в†’
   568в†’```vuma
   569в†’upper: u32 = value >> 24;   // no mask needed for right shift
   570в†’```
   571в†’
   572в†’Right shift does not require masking because it only clears bits (never sets new bits above the existing ones).
   573в†’
   574в†’### Rotate Right (Compose from Shifts)
   575в†’
   576в†’VUMA does not have a built-in rotate operator. Compose it from shifts:
   577в†’
   578в†’```vuma
   579в†’fn rotr32(x: u32, n: u32) -> u32 {
   580в†’    return ((x >> n) | (x << (32 - n))) & 4294967295;
   581в†’}
   582в†’```
   583в†’
   584в†’The mask is required because `x << (32 - n)` sets bits above bit 31.
   585в†’
   586в†’---
   587в†’
   588в†’## 9. Arithmetic
   589в†’
   590в†’### Basic Operators
   591в†’
   592в†’| Operator | Name | Example |
   593в†’|----------|------|---------|
   594в†’| `+` | Addition | `a + b` |
   595в†’| `-` | Subtraction | `a - b` |
   596в†’| `*` | Multiplication | `a * b` |
   597в†’| `/` | Division | `a / b` |
   598в†’| `%` | Modulo | `a % b` |
   599в†’
   600в†’### u32 Masking Rule
   601в†’
   602в†’**All u32 arithmetic results must be masked with `& 4294967295`.** VUMA uses 64-bit registers on 64-bit targets. Arithmetic on u32 values can produce carries into the upper 32 bits:
   603в†’
   604в†’```vuma
   605в†’// WRONG: overflow bits pollute upper 32 bits
   606в†’sum: u32 = a + b;
   607в†’
   608в†’// CORRECT: mask to 32 bits
   609в†’sum: u32 = (a + b) & 4294967295;
   610в†’```
   611в†’
   612в†’This applies to **all** arithmetic operations on u32 values:
   613в†’
   614в†’```vuma
   615в†’sum: u32 = (a + b) & 4294967295;
   616в†’diff: u32 = (a - b) & 4294967295;
   617в†’prod: u32 = (a * b) & 4294967295;
   618в†’quot: u32 = (a / b) & 4294967295;
   619в†’rem: u32 = (a % b) & 4294967295;
   620в†’```
   621в†’
   622в†’Division and modulo technically cannot overflow for u32, but masking is still recommended for consistency and safety.
   623в†’
   624в†’### u64 Arithmetic
   625в†’
   626в†’u64 arithmetic does **not** require masking:
   627в†’
   628в†’```vuma
   629в†’big_sum: u64 = a + b;    // OK, no mask needed
   630в†’```
   631в†’
   632в†’### i32 Arithmetic
   633в†’
   634в†’Signed 32-bit arithmetic should also be masked for correctness:
   635в†’
   636в†’```vuma
   637в†’signed_result: i32 = (a + b) & 4294967295;
   638в†’```
   639в†’
   640в†’### Chained Arithmetic
   641в†’
   642в†’When multiple u32 operations are chained, mask the **final** result:
   643в†’
   644в†’```vuma
   645в†’// SHA-256 T1 computation
   646в†’t1: u32 = (h + big_sigma1(e) + ch(e, f, g) + ki + wi) & 4294967295;
   647в†’```
   648в†’
   649в†’Intermediate results in sub-expressions (like `big_sigma1(e)`) should also be masked in their function definitions. Every function that returns u32 must mask its return value.
   650в†’

   756в†’## 12. Calling Convention
   757в†’
   758в†’### Argument Passing
   759в†’
   760в†’On 64-bit targets, the first 8 integer/pointer arguments are passed in registers (X0-X7 on AArch64, RDI/RSI/RDX/RCX/R8/R9 on x86_64, a0-a7 on RISC-V). Additional arguments are passed on the stack.
   761в†’
   762в†’On 32-bit targets (ARM32, Wasm32), the first 4 arguments use registers (r0-r3 on ARM32, the value stack on Wasm32).
   763в†’
   764в†’### Return Values
   765в†’
   766в†’The return value is passed in the first return register (X0 on AArch64, RAX on x86_64, a0 on RISC-V). For void functions, no return register is used.
   767в†’
   768в†’### Address Parameters
   769в†’
   770в†’`Address` parameters are passed like integer values (pointer-sized). On 64-bit targets, they occupy 8 bytes; on 32-bit targets, 4 bytes.
   771в†’
   772в†’### Function Call Example
   773в†’
   774в†’```vuma
   775в†’fn add1(x: i32) -> i32 {
   776в†’    return x + 1;
   777в†’}
   778в†’
   779в†’fn main() -> i32 {
   780в†’    return add1(41);  // returns 42
   781в†’}
   782в†’```
   783в†’
   784в†’### Cross-Function u32 Values
   785в†’
   786в†’When a function returns `u32`, the caller receives the value in a 64-bit register. The upper 32 bits may contain garbage. The **returning function** must mask the value before returning, and the **calling function** should not assume the upper bits are clear unless the callee guarantees it.
   787в†’
   788в†’```vuma
   789в†’fn safe_add(a: u32, b: u32) -> u32 {
   790в†’    return (a + b) & 4294967295;  // callee masks the result
   791в†’}
   792в†’
   793в†’fn main() -> i32 {
   794в†’    result: u32 = safe_add(0xFFFFFFFF, 1);  // result = 0
   795в†’    return 0;
   796в†’}
   797в†’```
   798в†’
   799в†’### Stack Layout (AArch64)
   800в†’
   801в†’```
   802в†’Higher addresses
   803в†’  +----------------------+
   804в†’  | Incoming stack args   |  FP+16, FP+24, ...
   805в†’  +----------------------+
   806в†’  | Saved FP (X29)       |  FP+0
   807в†’  | Saved LR (X30)       |  FP+8
   808в†’  +----------------------+
   809в†’  | Callee-saved regs    |  FP-8, FP-16, ...
   810в†’  | (X19..X28)           |
   811в†’  +----------------------+
   812в†’  | Local variables      |
   813в†’  | (aligned)            |
   814в†’  +----------------------+
   815в†’  | Outgoing stack args  |
   816в†’  | (for nested calls)   |
   817в†’  +----------------------+  SP
   818в†’Lower addresses
   819в†’```

   826в†’
   827в†’### 13.1 The `fmt` Module (String Formatting)
   828в†’
   829в†’The `fmt` module provides VUMA-verified string formatting primitives вЂ” the equivalent of C's `printf`/`sprintf` family. These functions convert numeric values to text representations in various bases, format floating-point numbers, pad strings, and write formatted output into byte buffers.
   830в†’
   831в†’All `fmt` functions are pure (CapD: { Read, Compare }) except for buffer-writing functions which also declare { Write }.
   832в†’
   833в†’#### Integer Formatting
   834в†’
   835в†’```vuma
   836в†’// Format a signed integer in the given base (2вЂ“36) with minimum width
   837в†’s = fmt::format_int(value: i64, base: u32, width: u32);
   838в†’
   839в†’// Format an unsigned integer in the given base with minimum width
   840в†’s = fmt::format_uint(value: u64, base: u32, width: u32);
   841в†’
   842в†’// Convenience wrappers for common bases:
   843в†’s = fmt::format_hex(value: u64, width: u32);     // base 16, e.g. "0x000000ff"
   844в†’s = fmt::format_binary(value: u64, width: u32);  // base 2, e.g. "11111111"
   845в†’s = fmt::format_octal(value: u64, width: u32);   // base 8, e.g. "377"
   846в†’
   847в†’// Format a pointer address as "0x" + 16-char hex
   848в†’s = fmt::format_pointer(addr: u64);
   849в†’```
   850в†’
   851в†’**Key behaviors:**
   852в†’- `base` must be 2вЂ“36; values outside that range are treated as base 10.
   853в†’- `width` specifies the minimum number of digits; the output is left-padded with `'0'` to reach this width.
   854в†’- For `format_int`, a negative value is prefixed with `'-'`; the sign occupies one character but is not counted as a digit for width purposes.
   855в†’- `format_hex` uses lowercase hex digits (`a`вЂ“`f`).
   856в†’
   857в†’**Examples:**
   858в†’
   859в†’```vuma
   860в†’// Decimal formatting
   861в†’s = fmt::format_int(42, 10, 0);          // "42"
   862в†’s = fmt::format_int(-7, 10, 4);          // "-0007"
   863в†’s = fmt::format_uint(255, 10, 0);        // "255"
   864в†’
   865в†’// Hex formatting
   866в†’s = fmt::format_hex(255, 4);             // "00ff"
   867в†’s = fmt::format_hex(0xDEADBEEF, 8);      // "deadbeef"
   868в†’
   869в†’// Binary formatting
   870в†’s = fmt::format_binary(5, 8);            // "00000101"
   871в†’
   872в†’// Pointer formatting
   873в†’s = fmt::format_pointer(0x7FFE4321);     // "0x000000007ffe4321"
   874в†’```
   875в†’
   876в†’#### Floating-Point Formatting
   877в†’
   878в†’```vuma
   879в†’// Format a float with exactly `precision` digits after the decimal point
   880в†’s = fmt::format_float(value: f64, precision: u32);
   881в†’```
   882в†’
   883в†’**Special values:**
   884в†’- NaN в†’ `"nan"`
   885в†’- Positive infinity в†’ `"inf"`
   886в†’- Negative infinity в†’ `"-inf"`
   887в†’- Zero with precision 0 в†’ `"0"`, with precision > 0 в†’ `"0.00..."`
   888в†’
   889в†’**Examples:**
   890в†’
   891в†’```vuma
   892в†’s = fmt::format_float(3.14159, 2);       // "3.14"
   893в†’s = fmt::format_float(3.14159, 6);       // "3.141590"
   894в†’s = fmt::format_float(0.0, 0);           // "0"
   895в†’s = fmt::format_float(0.0/0.0, 2);       // "nan"
   896в†’```
   897в†’
   898в†’#### String Padding and Joining
   899в†’
   900в†’```vuma
   901в†’// Left-pad a string to `width` characters with `fill`
   902в†’s = fmt::pad_left(s: &str, width: u32, fill: char);
   903в†’
   904в†’// Right-pad a string to `width` characters with `fill`
   905в†’s = fmt::pad_right(s: &str, width: u32, fill: char);
   906в†’
   907в†’// Join string slices with a separator
   908в†’s = fmt::join(parts: &[&str], separator: &str);
   909в†’```
   910в†’
   911в†’**Examples:**
   912в†’
   913в†’```vuma
   914в†’s = fmt::pad_left("42", 6, '0');         // "000042"
   915в†’s = fmt::pad_left("hello", 3, ' ');      // "hello" (already wider)
   916в†’s = fmt::pad_right("abc", 6, '-');        // "abc---"
   917в†’s = fmt::join(["a", "b", "c"], ", ");     // "a, b, c"
   918в†’```

  1035в†’### 13.3 The `crypto` Module (Cryptographic Primitives)
  1036в†’
  1037в†’The `crypto` module provides SHA-256 constants, logical functions, byte access helpers, and constant-time operations:
  1038в†’
  1039в†’```vuma
  1040в†’// SHA-256 logical functions
  1041в†’result: u32 = crypto::sha256_ch(x: u32, y: u32, z: u32);
  1042в†’result: u32 = crypto::sha256_maj(a: u32, b: u32, c: u32);
  1043в†’result: u32 = crypto::sha256_big_sigma0(x: u32);
  1044в†’result: u32 = crypto::sha256_big_sigma1(x: u32);
  1045в†’result: u32 = crypto::sha256_small_sigma0(x: u32);
  1046в†’result: u32 = crypto::sha256_small_sigma1(x: u32);
  1047в†’
  1048в†’// Big-endian byte access
  1049в†’val: u32 = crypto::sha256_read_u32_be(buf: Address, offset: u64);
  1050в†’crypto::sha256_write_u32_be(buf: Address, offset: u64, val: u32);
  1051в†’
  1052в†’// Constant-time operations (no data-dependent branches вЂ” safe against timing attacks)
  1053в†’equal: u32 = crypto::ct_eq_u32(a: u32, b: u32);     // 1 if equal, 0 otherwise
  1054в†’neq: u32 = crypto::ct_ne_u32(a: u32, b: u32);        // 1 if not equal
  1055в†’selected: u32 = crypto::ct_select_u32(flag: u32, a: u32, b: u32);  // a if flag=1, b if flag=0
  1056в†’lt: u32 = crypto::ct_lt_u32(a: u32, b: u32);         // 1 if a < b
  1057в†’gte: u32 = crypto::ct_gte_u32(a: u32, b: u32);       // 1 if a >= b
  1058в†’
  1059в†’// SHA-256 constants
  1060в†’// crypto::SHA256_K вЂ” 64 u32 round constants
  1061в†’// crypto::SHA256_H вЂ” 8 u32 initial hash values
  1062в†’```
  1063в†’
  1064в†’**Critical:** Constant-time functions execute in the same number of cycles regardless of input values. Use them whenever comparing secret data to prevent timing side-channels. All 8 backends implement these with branchless codegen.
  1065в†’
  1066в†’### 13.4 The `string` Module (String and Memory Operations)
  1067в†’
  1068в†’```vuma
  1069в†’// String operations
  1070в†’len: u64 = string::strlen(ptr: Address);
  1071в†’cmp: i32 = string::strcmp(a: Address, b: Address);
  1072в†’
  1073в†’// Memory operations
  1074в†’string::memcpy(dst: Address, src: Address, len: u64);
  1075в†’string::memset(ptr: Address, value: u32, len: u64);
  1076в†’```
  1077в†’

  1080в†’VUMA also provides these modules (available via `import`):
  1081в†’
  1082в†’- **`io`**: I/O traits (`VumaReader`, `VumaWriter`), buffered I/O, low-level syscall wrappers (`read_bytes`, `write_bytes`), little-endian byte access (`read_u32_le`, `write_u32_le`)
  1083в†’- **`sync`**: `Mutex`, `RwLock`, `Channel`, `Barrier` вЂ” all BD-annotated
  1084в†’- **`collections`**: `Vec`, `HashMap`, `RingBuffer`, `DoublyLinkedList` вЂ” VUMA-VERIFIED
  1085в†’- **`alloc`**: `GlobalAllocator`, `ArenaAllocator`, `BumpAllocator`, `PoolAllocator`, `FreeListAllocator`
  1086в†’- **`fs`**: Filesystem operations with capability-based access control
  1087в†’- **`net`**: TCP/UDP socket I/O

  1089в†’- **`path`**: Path manipulation and normalization
  1090в†’- **`process`**: Process management
  1091в†’- **`thread`**: Thread creation and management

  1096в†’## 14. Foreign Function Interface (FFI)
  1097в†’
  1098в†’VUMA supports calling external C functions through the `extern "C"` block syntax. This allows VUMA programs to interface with the system's C library, Linux syscalls, and other native code.
  1099в†’
  1100в†’### 14.1 The `extern "C"` Block
  1101в†’
  1102в†’Use an `extern "C" { ... }` block at the top level to declare external functions:
  1103в†’
  1104в†’```vuma
  1105в†’extern "C" {
  1106в†’    fn write(fd: i64, buf: Address, count: i64) -> i64;
  1107в†’    fn read(fd: i64, buf: Address, count: i64) -> i64;
  1108в†’    fn exit(code: i64);
  1109в†’}
  1110в†’```
  1111в†’
  1112в†’**Syntax rules:**
  1113в†’- The block must appear at the top level (not inside a function).
  1114в†’- The convention string is `"C"` (the only currently supported convention; `"system"` is reserved).
  1115в†’- Each function signature ends with `;` (no body вЂ” the implementation is external).
  1116в†’- Parameters use the same type syntax as regular VUMA functions.
  1117в†’- Functions without a return type are `void` (like `exit` above).
  1118в†’
  1119в†’### 14.2 Calling External Functions
  1120в†’
  1121в†’Once declared, extern functions are called like any other VUMA function:
  1122в†’
  1123в†’```vuma
  1124в†’fn main() -> i64 {
  1125в†’    let msg_addr: Address = 0x400000;
  1126в†’    let msg_len: i64 = 21;
  1127в†’    let result: i64 = write(1, msg_addr, msg_len);
  1128в†’    exit(0);
  1129в†’    return result;   // unreachable вЂ” exit() does not return
  1130в†’}
  1131в†’```
  1132в†’
  1133в†’### 14.3 How It Works (Codegen Details)
  1134в†’
  1135в†’When the compiler encounters a call to an `extern` function:
  1136в†’
  1137в†’1. **No local `BL` instruction** вЂ” Instead of emitting a direct branch, the codegen produces a **relocation** entry in the `.rela.text` section of the ELF object file.
  1138в†’2. **Symbol table** вЂ” The extern function is recorded as an `SHN_UNDEF` symbol in the ELF symbol table, indicating it must be resolved by the linker.
  1139в†’3. **Linking** вЂ” The system linker (`ld`) resolves the relocations against the C library or other object files.
  1140в†’
  1141в†’**Compilation pipeline for FFI programs:**
  1142в†’
  1143в†’```bash
  1144в†’# Compile VUMA source to a relocatable object (.o)
  1145в†’vuma compile --format obj --target aarch64 ffi_demo.vuma -o ffi_demo.o
  1146в†’
  1147в†’# Link with the C library
  1148в†’ld -o ffi_demo ffi_demo.o -lc
  1149в†’```
  1150в†’
  1151в†’### 14.4 Built-in Syscall Bindings
  1152в†’
  1153в†’The VUMA compiler includes built-in bindings for common Linux syscalls and C library functions. These are available without explicit `extern` declarations when compiling for bare-metal:
  1154в†’
  1155в†’**Linux syscalls:**
  1156в†’- `write(fd, buf, count)` вЂ” write bytes to a file descriptor
  1157в†’- `read(fd, buf, count)` вЂ” read bytes from a file descriptor
  1158в†’- `exit(code)` вЂ” terminate the process
  1159в†’- `mmap(addr, len, prot, flags, fd, offset)` вЂ” map memory
  1160в†’- `munmap(addr, len)` вЂ” unmap memory
  1161в†’- `brk(addr)` вЂ” change data segment size
  1162в†’
  1163в†’**C library functions:**
  1164в†’- `memcpy(dst, src, n)` вЂ” copy memory
  1165в†’- `memset(dst, c, n)` вЂ” fill memory
  1166в†’- `malloc(size)` вЂ” allocate memory
  1167в†’- `free(ptr)` вЂ” free memory
  1168в†’
  1169в†’### 14.5 Pitfalls for FFI
  1170в†’
  1171в†’- **No type checking across the boundary**: The compiler cannot verify the types of external C functions. If you declare `fn write(fd: i32, ...)` but the C function expects `i64`, the behavior is undefined.
  1172в†’- **Calling convention must match**: The `"C"` convention uses the platform's standard C ABI (System V AMD64 on Linux/x86_64, AAPCS64 on AArch64, etc.).
  1173в†’- **Linking is required**: FFI programs cannot run as standalone ET_REL objects. They must be linked with `ld` or `gcc`.
  1174в†’- **E037 relocation errors**: If a relocation cannot be resolved (e.g., symbol not found, overflow), the linker emits error E037.
  1175в†’

  1178в†’## 15. Atomic Operations
  1179в†’
  1180в†’VUMA provides three atomic memory operations for concurrent programming. These compile to target-specific atomic instructions.
  1181в†’
  1182в†’### 15.1 AtomicLoad
  1183в†’
  1184в†’Atomically loads a value from memory with acquire semantics.
  1185в†’
  1186в†’```vuma
  1187в†’value = AtomicLoad(addr);
  1188в†’```
  1189в†’
  1190в†’**Target-specific lowering:**
  1191в†’- AArch64: `LDAXR`
  1192в†’- x86_64: `LOCK` prefix or plain `MOV` (x86 is already atomic for aligned loads)
  1193в†’- RISC-V: `LR.D`
  1194в†’
  1195в†’### 15.2 AtomicStore
  1196в†’
  1197в†’Atomically stores a value to memory with release semantics.
  1198в†’
  1199в†’```vuma
  1200в†’AtomicStore(value, addr);
  1201в†’```
  1202в†’
  1203в†’**Target-specific lowering:**
  1204в†’- AArch64: `STLXR`
  1205в†’- x86_64: `LOCK` prefix or plain `MOV`
  1206в†’- RISC-V: `SC.D`
  1207в†’
  1208в†’### 15.3 AtomicCas (Compare-and-Swap)
  1209в†’
  1210в†’Atomically compares the value at `addr` with `expected`. If equal, writes `desired` to `addr`. Returns the old value at `addr`.
  1211в†’
  1212в†’```vuma
  1213в†’old_value = AtomicCas(addr, expected, desired);
  1214в†’```
  1215в†’
  1216в†’**Target-specific lowering:**
  1217в†’- AArch64: `LDAXR` / `CMP` / `B.NE` / `STLXR` loop
  1218в†’- x86_64: `LOCK CMPXCHG`
  1219в†’- RISC-V: `LR.D` / `BNE` / `SC.D` loop
  1220в†’
  1221в†’### 15.4 Example: Atomic Counter
  1222в†’
  1223в†’```vuma
  1224в†’fn atomic_increment(counter: Address) -> u64 {
  1225в†’    old: u64 = AtomicLoad(counter);
  1226в†’    loop {
  1227в†’        new_val: u64 = old + 1;
  1228в†’        prev: u64 = AtomicCas(counter, old, new_val);
  1229в†’        if prev == old {
  1230в†’            return new_val;
  1231в†’        }
  1232в†’        old = prev;
  1233в†’    }
  1234в†’}
  1235в†’```
  1236в†’
  1237в†’**Note:** Atomic operations are only available on 64-bit backends (x86_64, AArch64, RISC-V 64). The ARM32 and Wasm32 backends have limited atomic support.

  1300в†’## 17. Common Patterns
  1301в†’
  1302в†’### Pattern 1: SHA256d (Double SHA-256)
  1303в†’
  1304в†’This is the canonical VUMA program. It exercises all major language features: functions, memory, pointer arithmetic, bitwise operations, u32 masking, and loops.
  1305в†’
  1306в†’```vuma
  1307в†’fn rotr32(x: u32, n: u32) -> u32 {
  1308в†’    return ((x >> n) | (x << (32 - n))) & 4294967295;
  1309в†’}
  1310в†’
  1311в†’fn ch(x: u32, y: u32, z: u32) -> u32 {
  1312в†’    return (x & y) ^ ((x ^ 4294967295) & z);
  1313в†’}
  1314в†’
  1315в†’fn maj(a: u32, b: u32, c: u32) -> u32 {
  1316в†’    return (a & b) ^ (a & c) ^ (b & c);
  1317в†’}
  1318в†’
  1319в†’fn big_sigma0(x: u32) -> u32 {
  1320в†’    return rotr32(x, 2) ^ rotr32(x, 13) ^ rotr32(x, 22);
  1321в†’}
  1322в†’
  1323в†’fn big_sigma1(x: u32) -> u32 {
  1324в†’    return rotr32(x, 6) ^ rotr32(x, 11) ^ rotr32(x, 25);
  1325в†’}
  1326в†’
  1327в†’fn small_sigma0(x: u32) -> u32 {
  1328в†’    return (rotr32(x, 7) ^ rotr32(x, 18) ^ (x >> 3)) & 4294967295;
  1329в†’}
  1330в†’
  1331в†’fn small_sigma1(x: u32) -> u32 {
  1332в†’    return (rotr32(x, 17) ^ rotr32(x, 19) ^ (x >> 10)) & 4294967295;
  1333в†’}
  1334в†’
  1335в†’fn write_u32_be(buf: Address, offset: u64, val: u32) {
  1336в†’    *(buf + offset) = (val >> 24) & 255;
  1337в†’    *(buf + offset + 1) = (val >> 16) & 255;
  1338в†’    *(buf + offset + 2) = (val >> 8) & 255;
  1339в†’    *(buf + offset + 3) = val & 255;
  1340в†’}
  1341в†’
  1342в†’fn read_u32_be(buf: Address, offset: u64) -> u32 {
  1343в†’    b0: u32 = *(buf + offset);
  1344в†’    b1: u32 = *(buf + offset + 1);
  1345в†’    b2: u32 = *(buf + offset + 2);
  1346в†’    b3: u32 = *(buf + offset + 3);
  1347в†’    return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
  1348в†’}
  1349в†’
  1350в†’fn main() -> i32 {
  1351в†’    msg = allocate(3);
  1352в†’    *(msg + 0) = 97;
  1353в†’    *(msg + 1) = 98;
  1354в†’    *(msg + 2) = 99;
  1355в†’    // ... SHA-256 compression ...
  1356в†’    free(msg);
  1357в†’    return 0;
  1358в†’}
  1359в†’```
  1360в†’
  1361в†’### Pattern 2: Memory Buffer (Read/Write Bytes)
  1362в†’
  1363в†’```vuma
  1364в†’fn main() -> i32 {
  1365в†’    buf = allocate(16);
  1366в†’

  1368в†’    *(buf + 0) = 0x4f;
  1369в†’    *(buf + 1) = 0x8b;
  1370в†’
  1371в†’    // Read bytes back
  1372в†’    b0: u32 = *(buf + 0);
  1373в†’    b1: u32 = *(buf + 1);
  1374в†’
  1375в†’    free(buf);
  1376в†’    return 0;

  1379в†’
  1380в†’### Pattern 3: Byte Manipulation (Extract/Insert)
  1381в†’
  1382в†’```vuma
  1383в†’// Extract byte n from a u32 (big-endian)
  1384в†’fn get_byte(val: u32, n: u32) -> u32 {
  1385в†’    return (val >> (n * 8)) & 255;
  1386в†’}
  1387в†’
  1388в†’// Set byte n in a u32 (big-endian)
  1389в†’fn set_byte(val: u32, n: u32, byte: u32) -> u32 {
  1390в†’    shift: u32 = n * 8;
  1391в†’    mask: u32 = 255 << shift;
  1392в†’    return (val & (mask ^ 4294967295)) | ((byte & 255) << shift) & 4294967295;
  1393в†’}
  1394в†’```
  1395в†’
  1396в†’### Pattern 4: Memory Copy
  1397в†’
  1398в†’```vuma
  1399в†’fn copy_bytes(dst: Address, src: Address, len: u64) {
  1400в†’    i: u64 = 0;

  1402в†’        *(dst + i) = *(src + i);

  1407в†’### Pattern 5: Memory Set (Fill with Value)
  1408в†’
  1409в†’```vuma
  1410в†’fn mem_set(buf: Address, len: u64, val: u32) {
  1411в†’    i: u64 = 0;
  1412в†’    for i in 0..len {
  1413в†’        *(buf + i) = val & 255;
  1414в†’    }
  1415в†’}
  1416в†’```
  1417в†’
  1418в†’### Pattern 6: Array-style Access via Stride
  1419в†’
  1420в†’When storing an array of u32 values in a byte buffer, each element occupies 4 bytes (stride = 4):
  1421в†’
  1422в†’```vuma
  1423в†’fn w_store(w_base: Address, idx: u64, val: u32) {
  1424в†’    write_u32_be(w_base, idx * 4, val);
  1425в†’}
  1426в†’
  1427в†’fn w_load(w_base: Address, idx: u64) -> u32 {
  1428в†’    return read_u32_be(w_base, idx * 4);
  1429в†’}
  1430в†’```
  1431в†’
  1432в†’### Pattern 7: SHA-256 Compression Round
  1433в†’
  1434в†’```vuma
  1435в†’for i in 0..64 {
  1436в†’    ki: u32 = read_u32_be(k, i * 4);
  1437в†’    wi: u32 = w_load(w, i);
  1438в†’    t1: u32 = (h + big_sigma1(e) + ch(e, f, g) + ki + wi) & 4294967295;
  1439в†’    t2: u32 = (big_sigma0(a) + maj(a, b, c)) & 4294967295;
  1440в†’    h = g;
  1441в†’    g = f;
  1442в†’    f = e;
  1443в†’    e = (d + t1) & 4294967295;
  1444в†’    d = c;
  1445в†’    c = b;
  1446в†’    b = a;
  1447в†’    a = (t1 + t2) & 4294967295;
  1448в†’}
  1449в†’```

  1453в†’## 18. Pitfalls
  1454в†’
  1455в†’This section documents the most common mistakes when writing VUMA code. **Read this section carefully.**
  1456в†’
  1457в†’### Pitfall 1: u32 Arithmetic Without Masking
  1458в†’
  1459в†’**Problem**: VUMA uses 64-bit registers. When two u32 values are added, the 64-bit result can have bits set above bit 31 (the carry). Subsequent operations on this value (especially right shifts) will produce wrong results.
  1460в†’
  1461в†’```vuma
  1462в†’// WRONG
  1463в†’sum: u32 = a + b;        // upper 32 bits may have carry
  1464в†’rotated: u32 = sum >> 4;  // wrong! shift includes garbage bits
  1465в†’
  1466в†’// CORRECT
  1467в†’sum: u32 = (a + b) & 4294967295;  // clear upper 32 bits
  1468в†’rotated: u32 = sum >> 4;          // now correct
  1469в†’```
  1470в†’
  1471в†’**Rule**: After every arithmetic operation (`+`, `-`, `*`) on u32 values, apply `& 4294967295`.
  1472в†’
  1473в†’### Pitfall 2: Using `~` for Bitwise NOT
  1474в†’
  1475в†’**Problem**: The `~` operator inverts all 64 bits. For a u32 value in a 64-bit register, this means bits 32-63 are also inverted, producing `0xFFFFFFFF????????` instead of `0x00000000????????`.
  1476в†’
  1477в†’```vuma
  1478в†’// WRONG
  1479в†’not_x = ~x;  // produces 0xFFFFFFFF???????? for u32 x
  1480в†’
  1481в†’// CORRECT
  1482в†’not_x: u32 = x ^ 4294967295;  // flips only lower 32 bits
  1483в†’```
  1484в†’
  1485в†’**Rule**: Never use `~` for u32 bitwise NOT. Always use `x ^ 4294967295`.
  1486в†’
  1487в†’### Pitfall 3: Left Shift Without Masking
  1488в†’
  1489в†’**Problem**: Left-shifting a u32 value by N bits in a 64-bit register moves bits into the upper 32 bits.
  1490в†’
  1491в†’```vuma
  1492в†’// WRONG
  1493в†’shifted: u32 = value << 24;  // bits above bit 31 are set
  1494в†’
  1495в†’// CORRECT
  1496в†’shifted: u32 = (value << 24) & 4294967295;  // clear upper 32 bits
  1497в†’```
  1498в†’
  1499в†’**Rule**: After any left shift of a u32 value, apply `& 4294967295`.
  1500в†’
  1501в†’### Pitfall 4: Rotate Without Masking
  1502в†’
  1503в†’**Problem**: A rotate-right composed from shifts produces bits above bit 31 from the left-shift component.
  1504в†’
  1505в†’```vuma
  1506в†’// WRONG
  1507в†’rotated: u32 = (x >> n) | (x << (32 - n));  // upper bits polluted
  1508в†’
  1509в†’// CORRECT
  1510в†’rotated: u32 = ((x >> n) | (x << (32 - n))) & 4294967295;
  1511в†’```
  1512в†’
  1513в†’**Rule**: Always mask the result of a rotate operation.
  1514в†’
  1515в†’### Pitfall 5: Forgetting to Free Memory
  1516в†’
  1517в†’**Problem**: Every `allocate()` must be matched with a `free()`. The IVE checks this, but it is easy to forget in complex control flow.
  1518в†’
  1519в†’```vuma
  1520в†’// WRONG вЂ” memory leak
  1521в†’fn bad() -> i32 {
  1522в†’    buf = allocate(64);
  1523в†’    return 0;  // buf is never freed!
  1524в†’}
  1525в†’
  1526в†’// CORRECT
  1527в†’fn good() -> i32 {
  1528в†’    buf = allocate(64);
  1529в†’    // ... use buf ...
  1530в†’    free(buf);
  1531в†’    return 0;
  1532в†’}
  1533в†’```
  1534в†’
  1535в†’**Rule**: Always pair `allocate()` with `free()`. Free in reverse order of allocation to match the typical stack discipline.
  1536в†’
  1537в†’### Pitfall 6: Using `!x` for Bitwise NOT
  1538в†’
  1539в†’**Problem**: The `!` operator is logical NOT (converts non-zero to 0 and 0 to 1), not bitwise NOT.
  1540в†’
  1541в†’```vuma
  1542в†’// WRONG вЂ” logical NOT, not bitwise NOT
  1543в†’not_x = !x;  // produces 0 or 1, not bitwise complement
  1544в†’
  1545в†’// CORRECT вЂ” bitwise NOT via XOR
  1546в†’not_x: u32 = x ^ 4294967295;
  1547в†’```
  1548в†’
  1549в†’**Rule**: Use `x ^ 4294967295` for bitwise NOT, never `!x` or `~x`.
  1550в†’
  1551в†’### Pitfall 7: Hex Literals as Byte Values in Stores
  1552в†’
  1553в†’**Problem**: When storing a value to memory via `*(ptr + offset) = value`, only the low 8 bits are stored. Make sure you are not accidentally storing a truncated value.
  1554в†’
  1555в†’```vuma
  1556в†’// This stores only the low 8 bits (0xC2)
  1557в†’*(buf + 0) = 0x4FC2;  // stores 0xC2, NOT 0x4FC2
  1558в†’
  1559в†’// To store a specific byte, mask explicitly:
  1560в†’*(buf + 0) = (0x4FC2 >> 8) & 255;  // stores 0x4F
  1561в†’*(buf + 1) = 0x4FC2 & 255;        // stores 0xC2
  1562в†’```
  1563в†’
  1564в†’### Pitfall 8: Exit Code Truncation
  1565в†’
  1566в†’**Problem**: The process exit code is typically 8 bits (0-255) on Linux. When `main()` returns a u32 or i32, only the low byte is used as the exit code.
  1567в†’
  1568в†’```vuma
  1569в†’fn main() -> i32 {
  1570в†’    // To exit with code 79 (0x4F):
  1571в†’    return 79;  // exit code = 79
  1572в†’}
  1573в†’```
  1574в†’
  1575в†’If you need to verify a specific byte from a u32 computation, mask it and return:
  1576в†’
  1577в†’```vuma
  1578в†’fn main() -> i32 {
  1579в†’    result: u32 = compute_something();
  1580в†’    return result & 255;  // return low byte as exit code
  1581в†’}
  1582в†’```
  1583в†’
  1584в†’### Pitfall 9: Reading Bytes as u32 Without Masking
  1585в†’
  1586в†’When you read a byte from memory into a `u32` variable, the value is zero-extended by the load. However, if you then shift and combine bytes, the result must be masked:
  1587в†’
  1588в†’```vuma
  1589в†’// Each byte read is zero-extended to u32, but after shifting
  1590в†’// and combining, upper bits may be set
  1591в†’b0: u32 = *(buf + 0);  // e.g., 0x61
  1592в†’combined: u32 = (b0 << 24);  // 0x61000000 in 64-bit register, but
  1593в†’                               // might have upper bits
  1594в†’combined: u32 = (b0 << 24) & 4294967295;  // CORRECT
  1595в†’```
  1596в†’
  1597в†’### Pitfall 10: Range Bounds in For Loops
  1598в†’
  1599в†’The range `a..b` is exclusive on the upper bound. `0..64` means 0 through 63, not 0 through 64:
  1600в†’
  1601в†’```vuma
  1602в†’for i in 0..64 {
  1603в†’    // i goes from 0 to 63 inclusive (64 iterations)
  1604в†’}
  1605в†’
  1606в†’for i in 16..64 {
  1607в†’    // i goes from 16 to 63 inclusive (48 iterations)
  1608в†’}
  1609в†’```
  1610в†’
  1611в†’### Pitfall 11: FFI Type Mismatches
  1612в†’
  1613в†’**Problem**: The compiler cannot verify that the parameter types declared in an `extern "C"` block match the actual C function signature. A mismatch causes undefined behavior at runtime.
  1614в†’
  1615в†’```vuma
  1616в†’// WRONG: C's write() expects size_t (u64) for count, not i32
  1617в†’extern "C" {
  1618в†’    fn write(fd: i64, buf: Address, count: i32) -> i64;  // BUG: count should be i64
  1619в†’}
  1620в†’
  1621в†’// CORRECT: match the C ABI exactly
  1622в†’extern "C" {
  1623в†’    fn write(fd: i64, buf: Address, count: i64) -> i64;
  1624в†’}
  1625в†’```
  1626в†’
  1627в†’**Rule**: Always consult the C header documentation when declaring `extern "C"` functions. Use `i64`/`u64` for pointer-sized C types (`size_t`, `ssize_t`, `intptr_t`).
  1628в†’
  1629в†’### Pitfall 12: AtomicCas Retry Loop
  1630в†’
  1631в†’**Problem**: `AtomicCas` can fail spuriously (the value at `addr` was changed by another thread between the load and the CAS). You must loop until the CAS succeeds:
  1632в†’
  1633в†’```vuma
  1634в†’// WRONG: single CAS attempt may fail
  1635в†’fn bad_increment(counter: Address) -> u64 {
  1636в†’    old: u64 = AtomicLoad(counter);
  1637в†’    new_val: u64 = old + 1;
  1638в†’    prev: u64 = AtomicCas(counter, old, new_val);
  1639в†’    return new_val;  // may return wrong value if CAS failed!
  1640в†’}
  1641в†’
  1642в†’// CORRECT: retry loop
  1643в†’fn atomic_increment(counter: Address) -> u64 {
  1644в†’    old: u64 = AtomicLoad(counter);
  1645в†’    loop {
  1646в†’        new_val: u64 = old + 1;
  1647в†’        prev: u64 = AtomicCas(counter, old, new_val);
  1648в†’        if prev == old {
  1649в†’            return new_val;
  1650в†’        }
  1651в†’        old = prev;
  1652в†’    }
  1653в†’}
  1654в†’```
  1655в†’
  1656в†’**Rule**: Always wrap `AtomicCas` in a retry loop. Check that the returned old value matches the expected value.

  1744в†’## 20. Target Platforms
  1745в†’
  1746в†’VUMA compiles to native machine code for 8 architectures plus Wasm. Each target has its own backend that translates the IR to machine code.
  1747в†’
  1748в†’### Supported Targets
  1749в†’
  1750в†’| Target | Status | Pointer Size | Register Width | Notes |
  1751в†’|--------|--------|-------------|----------------|-------|
  1752в†’| x86_64 | Stable | 8 bytes | 64-bit | System V AMD64 ABI, DWARF debug info, FFI, atomics |
  1753в†’| AArch64 | Stable | 8 bytes | 64-bit | AAPCS64 calling convention, DWARF debug info, FFI, atomics |
  1754в†’| RISC-V 64 | Stable | 8 bytes | 64-bit | RV64GC, LP64 ABI, DWARF debug info, FFI, atomics |
  1755в†’| ARM32 | Stable | 4 bytes | 32-bit | AAPCS, no u32 masking needed*, DWARF debug info, FFI |
  1756в†’| MIPS64 | Stable | 8 bytes | 64-bit | N64 ABI, big-endian, DWARF debug info, FFI |
  1757в†’| PPC64 | Stable | 8 bytes | 64-bit | ELFv2 ABI, big-endian, DWARF debug info, FFI |
  1758в†’| LoongArch64 | Stable | 8 bytes | 64-bit | LP64 ABI, DWARF debug info, FFI, QEMU slow |
  1759в†’| Wasm32 | Experimental | 4 bytes | 32-bit | WebAssembly MVP, limited atomics |
  1760в†’
  1761в†’*ARM32 uses 32-bit registers, so u32 masking is not needed on this target. However, **write code that works on all targets** by always masking u32 results.
  1762в†’

  1789в†’
  1790в†’### Target-Specific Considerations
  1791в†’
  1792в†’**64-bit targets (x86_64, AArch64, RISC-V 64, MIPS64, PPC64, LoongArch64)**:
  1793в†’- All use 64-bit general-purpose registers
  1794в†’- u32 values stored in 64-bit registers **must** be masked with `& 4294967295` after arithmetic and left shifts
  1795в†’- `Address` is 8 bytes
  1796в†’- Pointer arithmetic uses 64-bit addition
  1797в†’
  1798в†’**32-bit targets (ARM32, Wasm32)**:
  1799в†’- Use 32-bit registers
  1800в†’- u32 masking is technically unnecessary but recommended for portability
  1801в†’- `Address` is 4 bytes
  1802в†’- Pointer arithmetic uses 32-bit addition
  1803в†’
  1804в†’**Wasm32 specifics**:
  1805в†’- No native ROR/ROL instructions; rotates are synthesized from shift+or sequences
  1806в†’- Type inference can be tricky; explicit type annotations are important
  1807в†’- Stack machine architecture (no registers); the backend manages a virtual value stack
  1808в†’
  1809в†’### Portability Rules
  1810в†’
  1811в†’When writing VUMA code, follow these rules to ensure it works on all 8 targets:
  1812в†’
  1813в†’1. **Always mask u32 results** with `& 4294967295` after addition, subtraction, multiplication, and left shift.
  1814в†’2. **Use `x ^ 4294967295`** for bitwise NOT, never `~x`.
  1815в†’3. **Compose rotates from shifts** with a final mask: `((x >> n) | (x << (32 - n))) & 4294967295`.
  1816в†’4. **Use explicit type annotations** for all variable declarations.
  1817в†’5. **Use `u64` for offsets and lengths**, not `u32`, to avoid truncation on large buffers.
  1818в†’6. **Always pair `allocate()` with `free()`**.
