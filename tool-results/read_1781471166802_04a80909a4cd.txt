     1вЖТ# VUMA Project Glossary
     2вЖТ
     3вЖТThis document defines every term used across the VUMA project. Each entry
     4вЖТincludes the term, its pronunciation when non-obvious, a detailed definition
     5вЖТof at least 50 words, and cross-references to related terms. This glossary is
     6вЖТthe single source of truth for terminology; all other documents should use
     7вЖТterms as defined here.
     8вЖТ
     9вЖТ---
    10вЖТ
    11вЖТ## Project Core Terms
    12вЖТ
    13вЖТ### SCG
    14вЖТ
    15вЖТ**Pronunciation:** /…Ыs-siЋР-d ТiЋР/ (spell out: S-C-G)
    16вЖТ
    17вЖТ**Semantic Computation Graph** вАФ The primary representation of a program in the
    18вЖТVUMA framework. The SCG is a directed, acyclic graph where nodes represent
    19вЖТcomputational operations (function application, type construction, effect
    20вЖТexecution, resource allocation, memory access, deallocation, control flow, type
    21вЖТcasts, and phantom markers for analysis), edges represent data flow, control
    22вЖТflow, derivation, and annotation relationships between operations, annotations
    23вЖТcarry type information, constraints, invariants, and metadata attached to nodes
    24вЖТand edges, and regions delineate scopes, phases, security boundaries, and
    25вЖТdeployment targets within the graph. The SCG is not derived from source code вАФ
    26вЖТit *is* the program. Any textual or visual representation is a projection of the
    27вЖТSCG. Two programs with the same semantics have the same SCG, regardless of how
    28вЖТthey were constructed (unique canonical form). The SCG is compositional
    29вЖТ(subgraphs combine through formally defined composition operators),
    30вЖТtransformable (semantics-preserving graph transformations are provably correct
    31вЖТby construction), and queryable (the IVE can ask arbitrary questions about
    32вЖТprogram properties through the SCG query engine). The SCG is implemented in the
    33вЖТ`vuma-scg` crate and defines node types (`NodeType` enum with variants
    34вЖТComputation, Allocation, Deallocation, Access, Cast, Effect, Control, Phantom),
    35вЖТedge types (`EdgeKind` with DataFlow, ControlFlow, Derivation, Annotation),
    36вЖТand region types (`SCGRegion` with `DeploymentTarget`).
    37вЖТ
    38вЖТ**See also:** IVE, Projection, BD, COR, MSG
    39вЖТ
    40вЖТ---
    41вЖТ
    42вЖТ### IVE
    43вЖТ
    44вЖТ**Pronunciation:** /a…™v/ (rhymes with "hive")
    45вЖТ
    46вЖТ**Inference and Verification Engine** вАФ The unified reasoning system that
    47вЖТreplaces the traditional compiler's type checker, borrow checker, and static
    48вЖТanalyzer. The IVE operates on the SCG and performs four core functions: (1) type
    49вЖТinference, deriving all types from SCG structure without requiring human
    50вЖТannotation by propagating Behavioral Descriptors through the graph; (2)
    51вЖТconstraint inference, deriving temporal constraints, resource flow constraints,
    52вЖТsecurity boundaries, and complexity bounds from program structure; (3)
    53вЖТverification, constructing formal proofs or counterexamples for properties that
    54вЖТcannot be inferred, checking the five VUMA global invariants (liveness,
    55вЖТexclusivity, interpretation, origin, cleanup) against the Memory State Graph;
    56вЖТand (4) gradual verification, maintaining a "verification debt" of properties
    57вЖТbelieved true but not yet proven, and continuously working to reduce this debt.
    58вЖТThe IVE is the reasoning heart of the VUMA framework вАФ it is what makes
    59вЖТverified-unsafe memory access possible, because it proves safety through global
    60вЖТreasoning rather than local restriction. The IVE is implemented in the
    61вЖТ`vuma-ive` crate with per-invariant modules (`liveness.rs`, `exclusivity.rs`,
    62вЖТ`interpretation.rs`, `origin.rs`, `cleanup.rs`), an inference engine
    63вЖТ(`inference.rs`), a constraint system (`constraint.rs`), debt tracking
    64вЖТ(`debt.rs`), and an invariant aggregator that runs all passes and produces
    65вЖТunified results (`invariant_aggregator.rs`).
    66вЖТ
    67вЖТ**See also:** SCG, VUMA, MSG, Verification Debt, Verification Confidence
    68вЖТ
    69вЖТ---
    70вЖТ
    71вЖТ### COR
    72вЖТ
    73вЖТ**Pronunciation:** /k…ФЋРr/ (rhymes with "more")
    74вЖТ
    75вЖТ**Continuous Optimization Runtime** вАФ The runtime system that replaces the
    76вЖТtraditional compile-link-run pipeline. The COR maintains the SCG in an
    77вЖТalways-compiled state: edits trigger incremental recompilation of affected
    78вЖТsubgraphs, eliminating the concept of "build time." The COR performs
    79вЖТprofile-guided optimization by collecting runtime profile data (including
    80вЖТhardware PMU counters for cycle count, instruction count, cache misses, and
    81вЖТbranch misses) and feeding it back to the IVE for optimization decisions. It
    82вЖТsupports speculative optimization (pre-optimizing likely execution paths with
    83вЖТtransparent fallback via a snapshot-and-rollback mechanism) and adaptive
    84вЖТdeployment (moving computation between nodes in a distributed system based on
    85вЖТlatency, cost, and availability). The COR includes a deployment manager with
    86вЖТhot-swap support (6-phase state machine: Idle, PreparingShadow,
    87вЖТAwaitingSafePoint, Swapping, Completed, Failed), delta deployment (block-level
    88вЖТbinary diffing for minimal transfer), and version logging with rollback
    89вЖТcapability. The COR treats the SCG's region annotations as guidance for
    90вЖТdeployment topology decisions and is implemented in the `vuma-cor` crate.
    91вЖТ
    92вЖТ**See also:** SCG, IVE, Projection
    93вЖТ

    95вЖТ
    96вЖТ### BD
    97вЖТ
    98вЖТ**Pronunciation:** /biЋР-diЋР/ (spell out: B-D)
    99вЖТ
   100вЖТ**Behavioral Descriptor** вАФ A triple `(RepD, CapD, RelD)` that describes data
   101вЖТby what it *does* rather than what it *is called*. The BD replaces traditional
   102вЖТnominal types in the VUMA framework. Where a traditional type bundles memory
   103вЖТlayout, valid operations, and relationships into a single nominal category, the
   104вЖТBD decomposes these into three orthogonal dimensions that can each vary
   105вЖТindependently. The IVE infers and verifies all three components of a BD from
   106вЖТprogram structure, function signatures, and execution context. The programmer
   107вЖТcan add explicit descriptors as optional refinements, but they are never
   108вЖТmandatory annotations. Two values with the same BD are interchangeable
   109вЖТregardless of their nominal "type name," and two values with different BDs are
   110вЖТdistinct even if they share a name. The BD is implemented as a struct in the
   111вЖТ`vuma-bd` crate (`descriptor::BD`) with fields for each component, plus methods
   112вЖТfor compatibility checking (`compatible()`), refinement ordering (`refines()`),
   113вЖТand composition. The crate also provides a context solver for BD inference and
   114вЖТa unification engine for BD compatibility.
   115вЖТ
   116вЖТ**See also:** RepD, CapD, RelD, IVE
   117вЖТ
   118вЖТ---
   119вЖТ
   120вЖТ### RepD
   121вЖТ
   122вЖТ**Pronunciation:** /r…Ыp-diЋР/ (rhymes with "step-dee")
   123вЖТ
   124вЖТ**Representation Descriptor** вАФ The component of a Behavioral Descriptor that
   125вЖТspecifies the physical layout of data in memory: size, alignment, field offsets,
   126вЖТand bit-level structure. A RepD is not a type; it is a *memory map*. Multiple
   127вЖТRepDs can describe the same memory at different granularities simultaneously вАФ
   128вЖТa 128-byte region can be described as `bytes[128]`, as `float32[32]`, or as
   129вЖТ`struct { header: uint32; payload: bytes[124] }`. The RepD does not choose
   130вЖТamong these; it enumerates all valid interpretations. This multi-view capability
   131вЖТis what enables zero-copy interoperation and eliminates type-conversion bugs,
   132вЖТbecause there is no conversion вАФ only a shift in perspective verified by the
   133вЖТIVE. The RepD is implemented as an enum in `vuma-bd::repd::RepD` with variants
   134вЖТfor byte-level, pointer, struct, array, and primitive representations, along
   135вЖТwith compatibility and refinement queries in `repd_compat.rs`. In the VUMA
   136вЖТstandard library, RepD is used to annotate the memory layout of primitive types
   137вЖТlike `Ptr<T>`, `RegionPtr<T>`, `Slice<T>`, `VumaResult<T, E>`, `VumaOption<T>`,
   138вЖТand `Range`.
   139вЖТ
   140вЖТ**See also:** BD, CapD, RelD, VUMA, Interpretation
   141вЖТ
   142вЖТ---
   143вЖТ
   144вЖТ### CapD
   145вЖТ
   146вЖТ**Pronunciation:** /k√¶p-diЋР/ (rhymes with "map-dee")
   147вЖТ
   148вЖТ**Capability Descriptor** вАФ The component of a Behavioral Descriptor that
   149вЖТspecifies what operations are valid on data in a given context. A CapD enumerates
   150вЖТpermissions: read, write, iterate, send over network, persist to disk, execute
   151вЖТas code, derive pointer from, compare for equality, hash, serialize. A CapD is
   152вЖТa set of *permissions*, not a type class. The same data can have different
   153вЖТcapabilities in different contexts вАФ a buffer is readable and writable during
   154вЖТprocessing but only readable during transmission. CapD captures this
   155вЖТcontext-dependence natively, enabling polymorphism through capability matching:
   156вЖТany value with the required capabilities satisfies the constraint, regardless
   157вЖТof its nominal type. The CapD forms a lattice (implemented in
   158вЖТ`vuma-bd::capd_lattice`) where the join is union and the meet is intersection,
   159вЖТwith a sub-capability ordering. The CapD is implemented as a struct in
   160вЖТ`vuma-bd::capd::CapD` with a bitflag-based `Capability` enum and methods for
   161вЖТstrengthening, weakening, union, intersection, and sub-capability checking.
   162вЖТ
   163вЖТ**See also:** BD, RepD, RelD, Capability Calculi
   164вЖТ
   165вЖТ---
   166вЖТ
   167вЖТ### RelD
   168вЖТ
   169вЖТ**Pronunciation:** /r…Ыl-diЋР/ (rhymes with "tell-dee")
   170вЖТ
   171вЖТ**Relational Descriptor** вАФ The component of a Behavioral Descriptor that
   172вЖТspecifies relationships between data values: temporal co-occurrence, structural
   173вЖТcontainment, dependency ordering, semantic equivalence, and security-level flow.
   174вЖТA RelD captures the web of relationships that nominal types express through
   175вЖТinheritance hierarchies and trait implementations, but with far greater
   176вЖТexpressiveness. For example, a RelD can express: "this value is semantically
   177вЖТequivalent to that value but represented differently" (a database row and a
   178вЖТprotobuf message describing the same entity), or "this value must not outlive
   179вЖТthat value" (a slice and its backing buffer), or "this value's security level
   180вЖТis derived from the maximum security level of its sources" (a computed result
   181вЖТcombining public and secret inputs). The RelD is implemented as a struct in
   182вЖТ`vuma-bd::reld::RelD` with a `Relation` enum for relationship kinds
   183вЖТ(Containment, Liveness, Aliasing, DataFlow, RegionBound, Ownership) and methods
   184вЖТfor composition, refinement, and intersection, with a separate refinement
   185вЖТengine in `reld_refine.rs`.
   186вЖТ
   187вЖТ**See also:** BD, RepD, CapD
   188вЖТ
   189вЖТ---
   190вЖТ
   191вЖТ### VUMA
   192вЖТ
   193вЖТ**Pronunciation:** /vuЋРm…СЋР/ (rhymes with "llama" with a v)
   194вЖТ
   195вЖТ**Verified-Unsafe Memory Access** вАФ The memory model at the core of this
   196вЖТproject, which permits unrestricted raw memory access (pointers, addresses,
   197вЖТmanual allocation, arbitrary casts) and establishes safety through global
   198вЖТverification rather than local restriction. In the VUMA model, all data access
   199вЖТis pointer-based. There are no "safe references" and "unsafe pointers" вАФ there
   200вЖТare only addresses, and the IVE verifies that every access through every address
   201вЖТis valid at the point of access. The memory model has three primitives: Address
   202вЖТ(a numeric value identifying a location, implemented as `vuma_core::address::Address`),
   203вЖТRegion (a contiguous range of addresses with allocation status, ownership
   204вЖТcontext, and access history, implemented as `vuma_core::region::Region`), and
   205вЖТAccess (a read or write operation targeting an address, verified by the IVE
   206вЖТbefore execution, implemented as `vuma_core::access::Access`). VUMA inverts the
   207вЖТsafety model: instead of "restrict by default, permit with explicit unsafe
   208вЖТblocks," it proposes "permit by default, verify by global reasoning, flag only
   209вЖТwhat cannot be proven safe." The VUMA core is implemented in the `vuma` crate
   210вЖТwith modules for regions, addresses, accesses, derivations, synchronization,
   211вЖТthe Memory State Graph, and SCG-to-MSG conversion.
   212вЖТ
   213вЖТ**See also:** IVE, MSG, Liveness, Exclusivity, Interpretation, Origin, Cleanup
   214вЖТ
   215вЖТ---
   216вЖТ
   217вЖТ### MSG
   218вЖТ
   219вЖТ**Pronunciation:** /…Ыm-…Ыs-d ТiЋР/ (spell out: M-S-G)
   220вЖТ
   221вЖТ**Memory State Graph** вАФ The IVE's formal model of the program's entire memory
   222вЖТbehavior. The MSG captures every allocation point and the region it creates,
   223вЖТevery pointer derivation and the path from allocation to dereference, every
   224вЖТdeallocation point and the region it destroys, every concurrent access and the
   225вЖТsynchronization that orders it, and every cast or reinterpretation and the
   226вЖТrepresentation descriptors involved. The IVE proves global memory invariants
   227вЖТ(liveness, exclusivity, interpretation, origin, cleanup) against the MSG. The
   228вЖТMSG is the data structure that makes VUMA possible вАФ it is the comprehensive
   229вЖТmodel against which all memory safety proofs are constructed. The MSG is
   230вЖТimplemented as `vuma_core::msg::MSG` with an incremental update mechanism
   231вЖТ(`vuma_core::msg_incremental`) that supports delta computation and application
   232вЖТfor efficient incremental verification when the SCG changes. The MSG is
   233вЖТconstructed from the SCG via `vuma_core::scg_to_msg::scg_to_msg()`, which
   234вЖТperforms a topological walk of SCG nodes and creates the corresponding regions,
   235вЖТderivations, accesses, and synchronization edges.
   236вЖТ
   237вЖТ**See also:** VUMA, IVE, Liveness, Exclusivity, Interpretation, Origin, Cleanup
   238вЖТ
   239вЖТ---
   240вЖТ
   241вЖТ### Projection
   242вЖТ
   243вЖТ**Pronunciation:** /pr…ЩЋИd Т…Ыk Г…Щn/
   244вЖТ
   245вЖТA view of the SCG rendered for human consumption. Projections come in four
   246вЖТforms: (1) Textual projections вАФ traditional code-like views customized to the
   247вЖТviewer's role (systems programmer, domain expert, security auditor); (2) Visual
   248вЖТprojections вАФ dataflow diagrams, call graphs, state machines, memory layout
   249вЖТviews that are live and interactive; (3) Conversational projections вАФ natural
   250вЖТlanguage dialogue where the human describes intent and the system translates it
   251вЖТinto SCG modifications; and (4) Diff projections вАФ change descriptions in human
   252вЖТterms ("the authentication flow now requires 2FA for admin accounts") rather
   253вЖТthan line-level diffs. Projections are bidirectional: changes to the projection
   254вЖТare propagated back to the SCG and validated by the IVE before being applied.
   255вЖТThe projection system is implemented in the `vuma-projection` crate with
   256вЖТmodules for textual, visual, conversational, and diff projections, plus a
   257вЖТbidirectional synchronization engine.
   258вЖТ
   259вЖТ**See also:** SCG, IVE
   260вЖТ
   261вЖТ---
   262вЖТ
   263вЖТ### Outcome Space
   264вЖТ
   265вЖТ**Pronunciation:** /ЋИa Кtk Мm spe…™s/
   266вЖТ
   267вЖТThe complete set of possible outcomes of a computation, including all failure
   268вЖТmodes. In the VUMA framework, the type of a function includes its complete
   269вЖТoutcome space вАФ not just `Result<T, E>` with a generic error type, but a
   270вЖТstructured enumeration of every possible outcome with its conditions and
   271вЖТhandlers. The IVE verifies that every outcome is handled, either explicitly by
   272вЖТthe programmer or implicitly by a verified safe default. As the program executes
   273вЖТand invariants are established, the outcome space shrinks вАФ for example, after
   274вЖТauthentication succeeds, the "unauthorized" outcome is removed from the
   275вЖТpossibility space for subsequent operations. The VUMA standard library provides
   276вЖТ`VumaResult<T, E>` and `VumaOption<T>` types with embedded Behavioral
   277вЖТDescriptors that track the outcome space, enabling the IVE to reason about
   278вЖТwhich outcomes are possible at each program point and verify that all are
   279вЖТhandled appropriately.
   280вЖТ
   281вЖТ**See also:** IVE, BD, CapD
   282вЖТ
   283вЖТ---
   284вЖТ
   285вЖТ### Verification Debt
   286вЖТ
   287вЖТ**Pronunciation:** /ЋМv…Ыr…™f…™ЋИke…™ Г…Щn d…Ыt/
   288вЖТ
   289вЖТProperties believed true but not yet formally proven by the IVE. The IVE
   290вЖТmaintains verification debt as a prioritized work list: it continuously works
   291вЖТto reduce debt, prioritizing properties that affect correctness and security.
   292вЖТThe concept mirrors technical debt in software engineering, but applies to
   293вЖТformal guarantees rather than code quality. Verification debt is acceptable in
   294вЖТpractice because not all properties need to be proven at all times вАФ but the
   295вЖТsystem tracks and communicates the current debt level so that stakeholders can
   296вЖТmake informed decisions about risk. In the codebase, verification debt is
   297вЖТtracked by the `vuma_ive::debt::VerificationDebt` struct, which maintains a
   298вЖТlist of `DebtItem` entries with priorities, statuses, and descriptions. Code
   299вЖТthat has not yet been verified by the IVE is marked with `// IVE-TODO`
   300вЖТannotations, which are the code-level representation of verification debt.
   301вЖТ
   302вЖТ**See also:** IVE, Verification Confidence, IVE-TODO
   303вЖТ
   304вЖТ---
   305вЖТ
   306вЖТ### Verification Confidence
   307вЖТ
   308вЖТ**Pronunciation:** /ЋМv…Ыr…™f…™ЋИke…™ Г…Щn ЋИk…Тnf…™d…Щns/
   309вЖТ
   310вЖТA tiered assessment of proof strength assigned to each verified property by the
   311вЖТIVE: (1) **Proven safe** вАФ the IVE has constructed a formal proof, represented
   312вЖТby `VerificationStatus::ProvenSafe` with the proof steps and evidence; (2)
   313вЖТ**Probably safe given stated assumptions** вАФ the IVE has proven safety
   314вЖТconditional on assumptions that it cannot verify but are documented and tracked,
   315вЖТrepresented by `VerificationStatus::ProbablySafe` with the list of assumptions;
   316вЖТ(3) **Unverified** вАФ the IVE has not yet been able to establish safety,
   317вЖТrepresented by `VerificationStatus::Unverified` with the reason why verification
   318вЖТcould not be completed. Deployment policies can require minimum confidence
   319вЖТlevels for different environments, allowing the system to gracefully degrade
   320вЖТwhen full verification is not achievable. The verification confidence levels are
   321вЖТdefined in `vuma_ive::result::VerificationStatus` and are used throughout the
   322вЖТverification pipeline to communicate proof strength to the programmer and the
   323вЖТruntime.
   324вЖТ
   325вЖТ**See also:** IVE, Verification Debt, VUMA
   326вЖТ

   328вЖТ
   329вЖТ## Verification Invariant Terms
   330вЖТ
   331вЖТ### Liveness
   332вЖТ
   333вЖТ**Pronunciation:** /ЋИla…™vn…Щs/
   334вЖТ
   335вЖТThe VUMA invariant requiring that every memory access targets a region that is
   336вЖТallocated at that program point. The IVE proves liveness by tracking allocation
   337вЖТand deallocation events through the MSG and verifying that no execution path
   338вЖТleads to a dereference of a freed or unallocated region. If the IVE cannot
   339вЖТprove liveness for a specific access, it flags that access and provides the
   340вЖТexecution path that leads to the potential use-after-free. Liveness is the first
   341вЖТof the five VUMA global invariants and corresponds to the elimination of
   342вЖТuse-after-free bugs. The liveness verification pass is implemented in
   343вЖТ`vuma_ive::liveness::LivenessVerifier`, which performs a forward dataflow
   344вЖТanalysis over the MSG tracking resource events (allocate, access, free) and
   345вЖТchecking that every access is preceded by an allocation and not followed by a
   346вЖТdeallocation at the same program point. The pass produces a
   347вЖТ`LivenessVerificationResult` with either proof obligations satisfied or
   348вЖТspecific `LivenessViolation` entries containing the offending address, program
   349вЖТpoint, and execution path.
   350вЖТ
   351вЖТ**See also:** VUMA, MSG, Exclusivity, Origin, Cleanup
   352вЖТ
   353вЖТ---
   354вЖТ
   355вЖТ### Exclusivity
   356вЖТ
   357вЖТ**Pronunciation:** /ЋМ…ЫkskluЋРЋИs…™v…™ti/
   358вЖТ
   359вЖТThe VUMA invariant requiring that every write access does not overlap with a
   360вЖТsimultaneous read or write access through a different address. The IVE proves
   361вЖТexclusivity by analyzing concurrent access patterns through the MSG, verifying
   362вЖТthat either accesses are properly ordered by synchronization (happens-before
   363вЖТedges, atomic operations, mutex acquisitions) or that they target non-overlapping
   364вЖТregions. If the IVE cannot prove exclusivity, it flags the potential data race
   365вЖТand provides the concurrent execution paths. Exclusivity is the second VUMA
   366вЖТinvariant and corresponds to the elimination of data races. The exclusivity
   367вЖТverification pass is implemented in `vuma_ive::exclusivity::ExclusivityVerifier`,
   368вЖТwhich constructs an interference graph of concurrent accesses and checks for
   369вЖТconflicts. The pass considers synchronization edges (represented as
   370вЖТ`vuma_core::sync::SyncEdge` with orderings like HappensBefore, AcquireRelease,
   371вЖТSequentiallyConsistent) that establish ordering between accesses. ARM64 barrier
   372вЖТinstructions (DMB, DSB, ISB) and atomic instructions (LDXR/STXR, CAS) are
   373вЖТmodeled as synchronization edges by the IVE.
   374вЖТ
   375вЖТ**See also:** VUMA, MSG, Liveness, Interpretation, DMB, DSB, LDXR, STXR
   376вЖТ
   377вЖТ---
   378вЖТ
   379вЖТ### Interpretation
   380вЖТ
   381вЖТ**Pronunciation:** /…™nЋМt…ЬЋРpr…™ЋИte…™ Г…Щn/
   382вЖТ
   383вЖТThe VUMA invariant requiring that every memory access interprets the target
   384вЖТbytes according to a valid representation descriptor (RepD). If the IVE cannot
   385вЖТprove the interpretation is valid (for example, reading uninitialized memory as
   386вЖТa pointer, or interpreting a floating-point bit pattern as an integer when the
   387вЖТsubsequent operations require an integer), it flags the access. Interpretation
   388вЖТis the third VUMA invariant and ensures that the bytes at any accessed address
   389вЖТare meaningful under the operation being performed, eliminating type confusion
   390вЖТand uninitialized-memory bugs. The interpretation verification pass is
   391вЖТimplemented in `vuma_ive::interpretation::InterpretationVerifier`, which traces
   392вЖТwrite-read pairs across the MSG and verifies that the RepD at the write point
   393вЖТis compatible with the RepD expected at the read point. The pass uses the
   394вЖТ`repd_compat` module from `vuma-bd` to check RepD compatibility, and produces
   395вЖТ`InterpretationViolation` entries with the write-read pair and the incompatible
   396вЖТRepDs when violations are found.
   397вЖТ
   398вЖТ**See also:** VUMA, MSG, RepD, Liveness
   399вЖТ
   400вЖТ---
   401вЖТ
   402вЖТ### Origin
   403вЖТ
   404вЖТ**Pronunciation:** /ЋИ…Тr…™d Т…™n/
   405вЖТ
   406вЖТThe VUMA invariant requiring that every address can be traced back to a valid
   407вЖТallocation point through a well-formed derivation chain. If an address is
   408вЖТcomputed through arithmetic that the IVE cannot trace to an allocation (for
   409вЖТexample, a hardcoded constant like `0xDEADBEEF`, or a value read from an
   410вЖТuntrusted source without validation), the IVE flags the computation. Origin is
   411вЖТthe fourth VUMA invariant and prevents the creation of "phantom pointers" that
   412вЖТpoint to memory the program never allocated, which could corrupt arbitrary
   413вЖТprocess memory. The origin verification pass uses taint analysis: each source
   414вЖТof pointer values is assigned a trust level, and the IVE verifies that every
   415вЖТpointer used in an access operation has a trust level that meets the required
   416вЖТthreshold. The origin verification is implemented in
   417вЖТ`vuma_ive::origin` and supported by the proof infrastructure in
   418вЖТ`vuma_proof::origin_proofs`, which provides `OriginProof`, `DerivationChainProof`,
   419вЖТtaint labels, and proof tactics for establishing origin.
   420вЖТ
   421вЖТ**See also:** VUMA, MSG, Liveness, Cleanup, Derivation Chain
   422вЖТ
   423вЖТ---
   424вЖТ
   425вЖТ### Cleanup
   426вЖТ
   427вЖТ**Pronunciation:** /ЋИkliЋРn Мp/
   428вЖТ
   429вЖТThe VUMA invariant requiring that every allocated region is eventually freed,
   430вЖТor explicitly marked as intentionally leaked (for example, a long-lived arena
   431вЖТor a global static). If the IVE detects a potential leak, it flags the
   432вЖТallocation. Cleanup is the fifth VUMA invariant and corresponds to the
   433вЖТelimination of memory leaks. The "explicitly marked as intentionally leaked"
   434вЖТexception accommodates patterns where memory is intentionally never freed
   435вЖТ(program-lifetime arenas, static globals), distinguishing them from accidental
   436вЖТleaks. The cleanup verification pass is implemented in
   437вЖТ`vuma_ive::cleanup::CleanupVerifier`, which constructs a cleanup graph tracking
   438вЖТallocation-deallocation pairs and verifies that every allocation has a matching
   439вЖТdeallocation on all execution paths. The pass produces `CleanupViolation`
   440вЖТentries with the specific `ViolationKind` (Leaked, PartialLeak, DoubleFree) and
   441вЖТthe offending allocation's program point. The proof infrastructure in
   442вЖТ`vuma_proof::cleanup_proofs` provides formal proof objects for cleanup
   443вЖТverification results.
   444вЖТ
   445вЖТ**See also:** VUMA, MSG, Liveness, Origin
   446вЖТ
   447вЖТ---
   448вЖТ
   449вЖТ## ARM64 / AArch64 Terms
   450вЖТ
   451вЖТ### AAPCS64
   452вЖТ
   453вЖТ**Pronunciation:** /√¶pks-s…™kstiЋР-f…ФЋРr/ (spell out acronym)
   454вЖТ
   455вЖТ**ARM Architecture Procedure Call Standard for 64-bit** вАФ The ABI (Application
   456вЖТBinary Interface) that governs how functions are called on ARM64 processors.
   457вЖТAAPCS64 defines which registers are used for arguments (x0вАУx7 for integer/pointer
   458вЖТarguments, v0вАУv7 for floating-point/SIMD arguments), which registers are
   459вЖТcaller-saved (x0вАУx18, excluding SP and XZR) vs. callee-saved (x19вАУx28), stack
   460вЖТframe layout with 16-byte stack alignment, and return value conventions (x0 for
   461вЖТscalar returns, x8 for indirect result locations). VUMA codegen must comply with
   462вЖТAAPCS64 to interoperate with C libraries and the operating system. The
   463вЖТregister classification is implemented in `vuma_codegen::arm64::Register` with
   464вЖТmethods `is_callee_saved()`, `is_caller_saved()`, `arg_register(index)`, and
   465вЖТ`arg_index()`. The codegen's register allocator (`vuma_codegen::regalloc`) must
   466вЖТrespect AAPCS64 calling conventions when allocating registers for function calls.
   467вЖТ
   468вЖТ**See also:** Cortex-A76, Register
   469вЖТ
   470вЖТ---
   471вЖТ
   472вЖТ### DMB
   473вЖТ
   474вЖТ**Pronunciation:** /diЋР-…Ыm-biЋР/ (spell out)
   475вЖТ
   476вЖТ**Data Memory Barrier** вАФ An ARM64 instruction that ensures that all explicit
   477вЖТmemory accesses before the DMB complete before any explicit memory accesses
   478вЖТafter the DMB are observed. DMB is used to enforce ordering between memory
   479вЖТoperations in multi-processor or DMA scenarios. DMB accepts a barrier option
   480вЖТthat specifies the shareability domain: `SY` (full system), `ISH` (inner
   481вЖТshareable), `OSH` (outer shareable), plus load/store variants (`ISHLD`,
   482вЖТ`ISHST`, `OSHLD`, `OSHST`, `LD`, `ST`). In VUMA, the IVE must understand DMB
   483вЖТsemantics to correctly verify the exclusivity invariant across concurrent
   484вЖТaccess patterns. DMB does not cause a pipeline flush; it only orders memory
   485вЖТoperations as observed by other observers. The DMB instruction is represented
   486вЖТas `Instruction::DMB { option: BarrierOption }` in the codegen, with encoding
   487вЖТimplemented in the `Instruction::encode()` method.
   488вЖТ
   489вЖТ**See also:** DSB, ISB, Exclusivity, BarrierOption
   490вЖТ
   491вЖТ---
   492вЖТ
   493вЖТ### DSB
   494вЖТ
   495вЖТ**Pronunciation:** /diЋР-…Ыs-biЋР/ (spell out)
   496вЖТ
   497вЖТ**Data Synchronization Barrier** вАФ An ARM64 instruction stronger than DMB: it
   498вЖТensures that all explicit memory accesses before the DSB complete before any
   499вЖТinstruction after the DSB is executed. DSB is required when the CPU must wait
   500вЖТfor memory operations to be truly complete (for example, after writing to a
   501вЖТdevice register before reading a status register). In VUMA, DSB is used in
   502вЖТdevice-driver code and in memory-mapped I/O scenarios where
   503вЖТthe IVE must model device-visible memory ordering. The DSB instruction is
   504вЖТrepresented as `Instruction::DSB { option: BarrierOption }` in the codegen and
   505вЖТaccepts the same barrier options as DMB (`SY`, `ISH`, `OSH`, and their load/store
   506вЖТvariants). The DSB is modeled as a full memory fence in the MSG's synchronization
   507вЖТmodel, creating happens-before edges between all preceding and subsequent memory
   508вЖТoperations.
   509вЖТ
   510вЖТ**See also:** DMB, ISB
   511вЖТ
   512вЖТ---
   513вЖТ
   514вЖТ### ISB
   515вЖТ
   516вЖТ**Pronunciation:** /a…™-…Ыs-biЋР/ (spell out)
   517вЖТ
   518вЖТ**Instruction Synchronization Barrier** вАФ An ARM64 instruction that flushes the
   519вЖТprocessor pipeline so that all instructions after the ISB are fetched from the
   520вЖТinstruction cache or memory after the ISB completes. ISB is required after
   521вЖТwriting to system registers (such as the SCTLR, TLB entries, or branch predictor
   522вЖТsettings) to ensure the change takes effect before subsequent instructions
   523вЖТexecute. In VUMA, ISB is relevant for context-switching code and low-level
   524вЖТsystem programming. The ISB instruction is represented as
   525вЖТ`Instruction::ISB` (no operands) in the codegen. The IVE models ISB as a
   526вЖТfull execution barrier that also implies DSB semantics, ensuring all prior
   527вЖТmemory and system register writes are visible before subsequent instructions
   528вЖТexecute.
   529вЖТ
   530вЖТ**See also:** DMB, DSB
   531вЖТ
   532вЖТ---
   533вЖТ
   534вЖТ### LDXR
   535вЖТ
   536вЖТ**Pronunciation:** /…Ыl-diЋР-…Ыks-…СЋРr/ (spell out)
   537вЖТ
   538вЖТ**Load Exclusive Register** вАФ An ARM64 instruction that loads a value from
   539вЖТmemory and tags the memory address for exclusive access monitoring. LDXR is one
   540вЖТhalf of the ARM64 exclusive access pair (with STXR) that implements lock-free
   541вЖТatomic operations. The hardware tracks whether the tagged address has been
   542вЖТwritten to by another agent between the LDXR and the subsequent STXR. In VUMA,
   543вЖТLDXR/STXR pairs are the codegen target for atomic compare-and-swap operations
   544вЖТ(the fallback path when ARMv8.1-LSE CAS is not available), and the IVE must
   545вЖТunderstand their semantics to verify the exclusivity invariant for lock-free
   546вЖТdata structures. The LDXR instruction is represented as `Instruction::LDXR {
   547вЖТrt, rn }` in the codegen, encoding as a 32-bit word with the base register (Rn)
   548вЖТand destination register (Rt). The exclusive monitor is modeled in the MSG as
   549вЖТa special synchronization edge that provides conditional ordering.
   550вЖТ
   551вЖТ**See also:** STXR, CAS, Exclusivity
   552вЖТ
   553вЖТ---
   554вЖТ
   555вЖТ### STXR
   556вЖТ
   557вЖТ**Pronunciation:** /…Ыs-tiЋР-…Ыks-…СЋРr/ (spell out)
   558вЖТ
   559вЖТ**Store Exclusive Register** вАФ An ARM64 instruction that attempts to store a
   560вЖТvalue to memory, succeeding only if the target address has not been written to
   561вЖТby another agent since the corresponding LDXR. STXR returns a status value in
   562вЖТa register: 0 indicates success, 1 indicates failure (the exclusive monitor was
   563вЖТlost). In VUMA, STXR is the codegen target for the store side of atomic
   564вЖТcompare-and-swap. Failed STXR operations typically cause a retry loop, and the
   565вЖТIVE must verify that such loops eventually succeed (a liveness property) and
   566вЖТthat the retry logic correctly handles the failure case. The STXR instruction
   567вЖТis represented as `Instruction::STXR { rs, rt, rn }` in the codegen, with the
   568вЖТstatus register (Rs), value register (Rt), and base address register (Rn).
   569вЖТ
   570вЖТ**See also:** LDXR, CAS, Liveness, Exclusivity
   571вЖТ
   572вЖТ---
   573вЖТ
   574вЖТ### CAS
   575вЖТ
   576вЖТ**Pronunciation:** /siЋР-e…™-…Ыs/ (spell out)
   577вЖТ
   578вЖТ**Compare and Swap** вАФ An ARM64 atomic instruction (ARMv8.1-LSE) that
   579вЖТatomically compares the value at a memory address with an expected value and,
   580вЖТif they match, replaces it with a new value. CAS is the single-instruction
   581вЖТequivalent of the LDXR/STXR loop pair, providing better performance on
   582вЖТARMv8.1-capable processors. CAS variants
   583вЖТinclude acquire (`CASA`), release (`CASL`), and acquire-release (`CASAL`)
   584вЖТordering semantics. In VUMA, the codegen prefers CAS over LDXR/STXR when the
   585вЖТtarget supports ARMv8.1-LSE, falling back to LDXR/STXR otherwise. The CAS
   586вЖТinstruction is represented as `Instruction::CAS { rs, rt, rn }` in the codegen.
   587вЖТThe IVE models CAS as an atomic operation that provides both exclusivity and
   588вЖТhappens-before ordering, making it the strongest synchronization primitive
   589вЖТavailable in the ARM64 instruction set.
   590вЖТ
   591вЖТ**See also:** LDXR, STXR, Exclusivity
   592вЖТ
   593вЖТ---
   594вЖТ
   595вЖТ### Cortex-A76
   596вЖТ
   597вЖТ**Pronunciation:** /ЋИk…ФЋРt…Ыks e…™ s…Ыv…Щnti-s…™ks/
   598вЖТ
   599вЖТAn ARM-compatible 64-bit out-of-order, superscalar CPU core implementing the
   600вЖТARMv8.2-A architecture. It features a 4-wide decode pipeline, branch prediction,
   601вЖТand a deep reorder buffer that allows significant instruction-level parallelism.
   602вЖТFor VUMA, the Cortex-A76's memory model (weakly ordered with multi-copy atomicity)
   603вЖТis the target for the IVE's exclusivity verification, and its instruction set
   604вЖТ(including ARMv8.1-LSE atomic instructions like CAS, LDADD, and STLXR) is the
   605вЖТtarget for the ARM64 codegen backend. The Cortex-A76 supports the AAPCS64
   606вЖТcalling convention and has 31 general-purpose registers (X0вАУX30), the stack
   607вЖТpointer (SP), and the zero register (XZR).
   608вЖТ
   609вЖТ**See also:** BCM2712, AAPCS64

   661вЖТ## Type Theory Terms
   662вЖТ
   663вЖТ### Nominal Types
   664вЖТ
   665вЖТ**Pronunciation:** /ЋИn…Тm…™n…Щl ta…™ps/
   666вЖТ
   667вЖТTypes defined by their *name* and explicit declaration. Two nominal types are
   668вЖТconsidered distinct even if they have identical structure вАФ for example,
   669вЖТ`struct UserId(u64)` and `struct OrderId(u64)` are different types in Rust
   670вЖТdespite having the same memory layout, because they have different names.
   671вЖТNominal typing is the dominant paradigm in mainstream languages (C, Java, Rust,
   672вЖТSwift) because it aligns with how humans categorize the world. In VUMA, nominal
   673вЖТtypes are superseded by Behavioral Descriptors, which define data by what it
   674вЖТ*does* (representation, capabilities, relationships) rather than by what it is
   675вЖТ*called*. This means that two values with different names but the same BD are
   676вЖТinterchangeable, and two values with the same name but different BDs are
   677вЖТdistinct вАФ a more flexible and precise system than nominal typing alone.
   678вЖТ
   679вЖТ**See also:** Structural Types, Behavioral Types, BD
   680вЖТ
   681вЖТ---
   682вЖТ
   683вЖТ### Structural Types
   684вЖТ
   685вЖТ**Pronunciation:** /ЋИstr Мkt Г…Щr…Щl ta…™ps/
   686вЖТ
   687вЖТTypes defined by their *structure* rather than their name. Two structural types
   688вЖТare considered equivalent if they have the same shape, regardless of what they
   689вЖТare called. TypeScript interfaces, Go structural typing for interfaces, and ML
   690вЖТrecord types are examples. Structural typing is more flexible than nominal
   691вЖТtyping for interoperation (any value with the right shape satisfies the
   692вЖТinterface), but it can accidentally equate types that happen to share structure
   693вЖТbut have different semantic intent. VUMA's Behavioral Descriptors generalize
   694вЖТstructural types: instead of comparing shape, they compare capabilities and
   695вЖТrelationships, capturing the useful properties of structural typing while
   696вЖТavoiding accidental equivalence through the RelD component that captures
   697вЖТsemantic relationships.
   698вЖТ
   699вЖТ**See also:** Nominal Types, Behavioral Types, CapD, RelD
   700вЖТ
   701вЖТ---
   702вЖТ
   703вЖТ### Behavioral Types
   704вЖТ
   705вЖТ**Pronunciation:** /b…™ЋИhe…™vj…Щr…Щl ta…™ps/
   706вЖТ
   707вЖТA type formalism that describes the *behavior* of a component вАФ the sequence
   708вЖТand conditions of interactions it may engage in вАФ rather than just its data
   709вЖТlayout. Behavioral types originate in concurrency theory (session types,
   710вЖТbehavioral contracts) and specify communication protocols, ordering constraints,
   711вЖТand resource usage patterns. VUMA's Behavioral Descriptors are a form of
   712вЖТbehavioral type that extends the concept from communication protocols to all
   713вЖТproperties of data: what operations are valid (CapD), what memory layout is used
   714вЖТ(RepD), and what relationships exist with other data (RelD). This makes VUMA's
   715вЖТtype system a full behavioral type system, not just a data classification system,
   716вЖТenabling the IVE to reason about behavioral properties like protocol compliance
   717вЖТand resource usage patterns in addition to memory safety.
   718вЖТ
   719вЖТ**See also:** Nominal Types, Structural Types, BD, CapD
   720вЖТ
   721вЖТ---
   722вЖТ
   723вЖТ### Capability Calculi
   724вЖТ
   725вЖТ**Pronunciation:** /ЋМke…™p…ЩЋИb…™l…™ti ЋИk√¶lkj Кla…™/
   726вЖТ
   727вЖТA family of formal calculi that model capabilities as first-class, unforgeable
   728вЖТtokens that grant specific access rights. Capability calculi (such as the
   729вЖТobject-capability model, capability-based security, and the calculus of
   730вЖТcapabilities) provide formal foundations for reasoning about which components
   731вЖТcan access which resources. VUMA's CapD system draws on capability calculi
   732вЖТtheory: a CapD is essentially a capability set, and the IVE verifies that no
   733вЖТoperation exceeds the capabilities granted to its inputs. The key difference
   734вЖТfrom traditional capability systems is that VUMA capabilities are *inferred* by
   735вЖТthe IVE rather than explicitly managed by the programmer, reducing the
   736вЖТcognitive burden while maintaining the formal guarantees. The CapD lattice
   737вЖТstructure (join = union, meet = intersection, sub-capability ordering) is
   738вЖТimplemented in `vuma-bd::capd_lattice` and provides the algebraic foundation
   739вЖТfor capability reasoning.
   740вЖТ
   741вЖТ**See also:** CapD, BD, IVE
   742вЖТ

   745вЖТ## Additional Project Terms
   746вЖТ
   747вЖТ### Derivation Chain
   748вЖТ
   749вЖТ**Pronunciation:** /d…Ыr…™ЋИve…™ Г…Щn t Гe…™n/
   750вЖТ
   751вЖТThe sequence of operations by which a pointer is computed from an original
   752вЖТallocation. For example, if `base = allocate(bytes[1024])`, `offset = base +
   753вЖТ64`, and `field_ptr = offset as *Header`, then the derivation chain for
   754вЖТ`field_ptr` is `[base, offset, field_ptr]`. The IVE tracks derivation chains
   755вЖТthrough the MSG to verify the origin invariant (every address traces back to an
   756вЖТallocation) and to verify that derived pointers remain within the bounds of
   757вЖТtheir originating region. Derivation chains are modeled in the MSG as
   758вЖТ`vuma_core::derivation::Derivation` entries with kinds (Direct, Offset, Cast,
   759вЖТIndex) and provenance ranges (base address + size from the original allocation).
   760вЖТThe SCG query engine (`vuma_scg::query::find_derivation_chains`) can compute
   761вЖТderivation chains from any node to its ancestors, and the proof infrastructure
   762вЖТ(`vuma_proof::origin_proofs::DerivationChainProof`) provides formal proof
   763вЖТobjects for derivation chain well-formedness.
   764вЖТ
   765вЖТ**See also:** MSG, Origin, VUMA, Derivation
   766вЖТ
   767вЖТ---
   768вЖТ
   769вЖТ### Region (SCG)
   770вЖТ
   771вЖТ**Pronunciation:** /ЋИriЋРd Т…Щn/
   772вЖТ
   773вЖТA subgraph of the SCG that delineates a scope, phase, security boundary, or
   774вЖТdeployment target. SCG regions are the unit of composition, optimization, and
   775вЖТdeployment. The COR uses region annotations to guide adaptive deployment
   776вЖТdecisions (moving computation between nodes based on latency and cost
   777вЖТconstraints). In the VUMA memory model, a Region is also a contiguous range of
   778вЖТaddresses with associated metadata (allocation status, ownership context, access
   779вЖТhistory). The dual usage is intentional: SCG regions correspond to ownership
   780вЖТcontexts, and memory regions are the physical manifestation of those contexts.
   781вЖТSCG regions are implemented as `vuma_scg::region::SCGRegion` with a
   782вЖТ`DeploymentTarget` enum (Heap, Stack, Global, Mmap, Gpu, Device) and a set of
   783вЖТnode IDs. Memory regions are implemented as `vuma_core::region::Region` with
   784вЖТ`RegionId`, base address, size, `RegionStatus` (Allocated, Freed, Stack, Device,
   785вЖТStatic), and program points for allocation and deallocation.
   786вЖТ
   787вЖТ**See also:** SCG, COR, MSG
   788вЖТ

   791вЖТ### VUMA-VERIFIED
   792вЖТ
   793вЖТ**Pronunciation:** /ЋИvuЋРm…СЋР ЋИv…Ыr…™fa…™d/
   794вЖТ
   795вЖТA code annotation comment (`// VUMA-VERIFIED`) used in the VUMA Rust codebase
   796вЖТto mark methods and functions whose safety has been formally verified by the
   797вЖТIVE. This comment replaces the `unsafe` keyword in semantic intent: where Rust
   798вЖТwould require an `unsafe` block to acknowledge that the programmer takes
   799вЖТresponsibility for safety, `// VUMA-VERIFIED` indicates that the IVE has taken
   800вЖТthat responsibility and proven safety. No VUMA stdlib code should contain bare
   801вЖТ`unsafe` blocks вАФ all raw memory access should be either `// VUMA-VERIFIED` or
   802вЖТ`// IVE-TODO`. The annotation must include a brief description of what was
   803вЖТverified: `// VUMA-VERIFIED: IVE proves this dereference targets a live,
   804вЖТaligned region`. The CI pipeline scans for `unsafe` blocks without the required
   805вЖТannotations and rejects any PR that introduces an un-annotated `unsafe` block.
   806вЖТ
   807вЖТ**See also:** IVE-TODO, VUMA, Verification Confidence
   808вЖТ
   809вЖТ---
   810вЖТ
   811вЖТ### IVE-TODO
   812вЖТ
   813вЖТ**Pronunciation:** /a…™v tuЋРЋИduЋР/
   814вЖТ
   815вЖТA code annotation comment (`// IVE-TODO`) used in the VUMA Rust codebase to
   816вЖТmark methods and functions where IVE verification has not yet been implemented.
   817вЖТThis is the equivalent of Rust's `unsafe` block in terms of acknowledging that
   818вЖТsafety has not been formally proven, but it carries a different connotation: it
   819вЖТis a *temporary* state, not an accepted risk. The project tracks IVE-TODO items
   820вЖТas verification debt that must be resolved before stable releases. The annotation
   821вЖТmust include a brief description of what needs to be verified:
   822вЖТ`// IVE-TODO: verify origin and liveness for this pointer arithmetic`. The
   823вЖТCI pipeline scans for bare `unsafe` blocks and rejects any that lack either
   824вЖТ`// VUMA-VERIFIED` or `// IVE-TODO` annotations. The verification debt tracker
   825вЖТ(`vuma_ive::debt::VerificationDebt`) maintains a list of all IVE-TODO items
   826вЖТwith their priorities and statuses, enabling the project to track progress
   827вЖТtoward full verification coverage.
   828вЖТ
   829вЖТ**See also:** VUMA-VERIFIED, Verification Debt, IVE
   830вЖТ

   832вЖТ
   833вЖТ### Invariant Aggregator
   834вЖТ
   835вЖТ**Pronunciation:** /…™nЋИv…Ы…Щri…Щnt ЋИ√¶…°r…™…°e…™t…Щr/
   836вЖТ
   837вЖТThe IVE component that runs all verification passes and produces a unified
   838вЖТverification summary. The aggregator executes each invariant check (liveness,
   839вЖТexclusivity, interpretation, origin, cleanup) against the MSG and combines the
   840вЖТresults into an `AggregatedResult` with an `OverallVerdict` (AllPassed,
   841вЖТPartialPass, Failed), per-invariant deltas (`InvariantDelta` showing what
   842вЖТchanged since the last verification run), and a `DiagnosticsReport` with
   843вЖТactionable information. The aggregator is implemented in
   844вЖТ`vuma_ive::invariant_aggregator::InvariantAggregator` and is the primary entry
   845вЖТpoint for running full verification. It also supports incremental verification
   846вЖТby comparing the current verification state against a previous snapshot and
   847вЖТonly re-running the invariant checks that are affected by the changes.
   848вЖТ
   849вЖТ**See also:** IVE, MSG, Liveness, Exclusivity, Interpretation, Origin, Cleanup
   850вЖТ
   851вЖТ---
   852вЖТ
   853вЖТ### Proof Object
   854вЖТ
   855вЖТ**Pronunciation:** /pruЋРf ЋИ…Тbd Т…Ыkt/
   856вЖТ
   857вЖТA structured representation of a formal proof constructed by the IVE during
   858вЖТverification. A proof object contains a goal (the property being proven), a
   859вЖТsequence of proof steps (each applying an inference rule to derive a new fact
   860вЖТfrom previous facts), and a conclusion (the final derived fact that establishes
   861вЖТthe goal). Proof objects are checked by the `vuma_proof::checker::ProofChecker`,
   862вЖТwhich verifies that each step follows from previous steps using the stated
   863вЖТinference rules and detects circular reasoning. When a proof cannot be
   864вЖТconstructed, the IVE generates a `CounterExample` from `vuma_proof::counterexample`
   865вЖТthat describes a minimal execution trace demonstrating the violation, helping
   866вЖТthe programmer understand and fix the issue.
   867вЖТ
   868вЖТ**See also:** IVE, Verification Confidence, CounterExample
   869вЖТ
