     3‚Üí## Task 2-f: Fix ARM32 calling convention for functions with >4 arguments
     4‚Üí
     5‚Üí**Date**: 2026-03-05
     6‚Üí**File**: `src/codegen/src/arm32/mod.rs`
     7‚Üí
     8‚Üí### Bug Description
     9‚Üí
    10‚ÜíThe ARM32 backend silently dropped function arguments beyond the first 4. Under AAPCS, the first 4 arguments are passed in registers R0‚ÄìR3, while arguments 5+ must be passed on the stack. The code at line 3594 had a `TODO: handle > 4 args via stack` comment, and arguments at index ‚â•4 were simply ignored in both the caller (call emission) and callee (prologue/argument setup).
    11‚Üí
    12‚Üí### Root Cause
    13‚Üí
    14‚ÜíThree related gaps in the ARM32 codegen:
    15‚Üí
    16‚Üí1. **Call emission** (`IRInstr::Call`): Only loaded args 0‚Äì3 into R0‚ÄìR3; args ‚â•4 were skipped entirely.
    17‚Üí2. **Function prologue**: Only stored R0‚ÄìR3 params to their vreg stack slots; stack-passed params (index ‚â•4) were never loaded from the stack into their vreg slots.
    18‚Üí3. **Missing helper**: No `emit_add_sp` counterpart to `emit_sub_sp`, needed for caller-side stack cleanup after calls with stack-passed args.
    19‚Üí
    20‚Üí### AAPCS Stack Layout
    21‚Üí
    22‚ÜíAfter the prologue (SUB SP, ADD R11=SP+frame_size, save LR/R11):
    23‚Üí```
    24‚ÜíHigher addresses
    25‚Üí  ‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê
    26‚Üí  ‚îÇ arg N-1      ‚îÇ  ‚Üê [R11 + 8 + (N-5)*4]
    27‚Üí  ‚îÇ ...          ‚îÇ
    28‚Üí  ‚îÇ arg 5        ‚îÇ  ‚Üê [R11 + 12]
    29‚Üí  ‚îÇ arg 4        ‚îÇ  ‚Üê [R11 + 8]
    30‚Üí  ‚îÇ saved LR     ‚îÇ  ‚Üê [R11 + 4]
    31‚Üí  ‚îÇ saved R11    ‚îÇ  ‚Üê [R11 + 0]
    32‚Üí  ‚îú‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚î§  ‚Üê R11 (frame pointer)
    33‚Üí  ‚îÇ vreg slots   ‚îÇ  ‚Üê [R11 - offset]
    34‚Üí  ‚îÇ ...          ‚îÇ
    35‚Üí  ‚îú‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚î§  ‚Üê SP
    36‚ÜíLower addresses
    37‚Üí```
    38‚Üí
    39‚ÜíStack-passed arg at 0-based index `i` (i ‚â• 4) is at `[R11 + 8 + (i - 4) * 4]`.
    40‚Üí
    41‚Üí### Fixes Applied
    42‚Üí
    43‚Üí#### Fix 1: Added `ss_load_from_r11_plus` helper (after line 3300)
    44‚Üí
    45‚ÜíNew closure that loads a 32-bit word from `[R11 + positive_offset]` into a destination register. Handles both the common case (offset ‚â§ 4095, single LDR instruction) and the rare large-offset case (compute address into R12 via ADD, then LDR indirect). This is the positive-offset counterpart to `ss_load_from_slot` which uses negative offsets.
    46‚Üí
    47‚Üí#### Fix 2: Added `emit_add_sp` helper (after line 3165)
    48‚Üí
    49‚ÜíNew closure that emits `ADD SP, SP, #imm` with the same large-value handling as the existing `emit_sub_sp` ‚Äî tries ARM rotated-immediate encoding first, falls back to loading the value into R12 and using `ADD SP, SP, R12`.
    50‚Üí
    51‚Üí#### Fix 3: Updated function prologue ‚Äî stack-passed args (lines 3444‚Äì3481)
    52‚Üí
    53‚ÜíExtended the parameter-storing loop to handle index ‚â•4. For each stack-passed param:
    54‚Üí1. Load the value from `[R11 + 8 + (i-4)*4]` into R0 using `ss_load_from_r11_plus`
    55‚Üí2. Store R0 to the param's vreg stack slot using `ss_store_to_slot`
    56‚Üí
    57‚ÜíUses R0 (not R12) as the scratch register because `ss_store_to_slot` documents that `src_reg` must NOT be R12 for large offsets (it uses R12 internally). R0 is safe because it was already saved for param 0 earlier in the loop.
    58‚Üí
    59‚Üí#### Fix 4: Updated call emission ‚Äî stack-passed args (lines 3862‚Äì3941)
    60‚Üí
    61‚ÜíCompletely rewrote the `IRInstr::Call` handler to support >4 arguments:
    62‚Üí
    63‚Üí1. **Allocate stack space**: `SUB SP, SP, #(num_stack_args * 4)` before setting up any args
    64‚Üí2. **Store args 5+ to stack**: For each arg at index ‚â•4:
    65‚Üí   - Normal case (offset ‚â§ 4095): Load arg into R12, `STR R12, [SP, #offset]`
    66‚Üí   - Large-offset case: Compute `SP + offset` into R12, load arg into R0, `STR R0, [R12, #0]`
    67‚Üí3. **Load args 0‚Äì3 into R0‚ÄìR3** (existing behavior, unchanged)
    68‚Üí4. **BL** (existing, unchanged)
    69‚Üí5. **Caller cleanup**: `ADD SP, SP, #(num_stack_args * 4)` after the call returns
    70‚Üí6. **Store return value** (existing, unchanged)
    71‚Üí
    72‚ÜíThe order (stack args first, then register args) ensures R12 is free for scratch use when storing stack args, and R0‚ÄìR3 are loaded last just before the BL.
    73‚Üí
    74‚Üí### Argument Access in Function Body
    75‚Üí
    76‚ÜíNo changes needed. The function body accesses all parameters (including stack-passed ones) through `vreg_stack_slots` using `ss_load_value`. The prologue is the bridge that copies stack-passed args from their AAPCS locations `[R11 + 8 + ...]` into the same vreg stack slot mechanism used by all other variables. After the prologue runs, the function body treats all params identically regardless of how they were passed.
    77‚Üí
    78‚Üí### Backward Compatibility
    79‚Üí
    80‚ÜíFunctions with ‚â§4 arguments are completely unaffected ‚Äî `num_stack_args` evaluates to 0, so no SUB SP, stack stores, or ADD SP instructions are emitted. The prologue loop only enters the `else` branch for params at index ‚â•4.

    82‚Üí## Task 2-c: Fix RISC-V 64 atomic CAS loop ‚Äî branch labels never inserted into label_offsets
    83‚Üí
    84‚Üí**Date**: 2026-03-05
    85‚Üí**File**: `src/codegen/src/riscv64.rs`
    86‚Üí
    87‚Üí### Bug Description
    88‚Üí
    89‚ÜíThe RISC-V 64 backend's atomic operations (AtomicStore and AtomicCas) had broken branch label handling. The code generated labels like `__atomic_store_retry_{offset}` and branch fixups referencing those labels, but never inserted the labels into the `label_offsets` HashMap. When Phase 4 (branch fixup) ran, it looked up each branch's `target_label` in `label_offsets`, got `None`, and silently skipped the fixup ‚Äî leaving all branch offsets at 0. This meant:
    90‚Üí
    91‚Üí- **AtomicStore**: SC.D failure was never retried; BNE with offset 0 branched to the next instruction (a no-op fall-through).
    92‚Üí- **AtomicCas**: No branch fixups were created at all (only a `TODO` comment). Both BNE instructions (not-equal‚Üídone and SC-failed‚Üíretry) kept offset 0, so the CAS loop never looped back on SC.D failure and never skipped to done on mismatch.
    93‚Üí
    94‚Üí### Additional Bug in AtomicStore
    95‚Üí
    96‚ÜíThe original AtomicStore code emitted `SC.D` without a preceding `LR.D`. Per the RISC-V A extension, SC.D requires a valid reservation established by a prior LR.D; without it, SC.D always reports failure (rd=1), making the retry loop infinite even if branching worked.
    97‚Üí
    98‚Üí### Root Cause Analysis
    99‚Üí
   100‚ÜíThe `label_offsets` HashMap is populated in two places:
   101‚Üí1. **Block labels** (line 4164): `label_offsets.insert(block.label.clone(), current_byte_offset)` ‚Äî works correctly for IR block labels.
   102‚Üí2. **Atomic labels**: Never inserted ‚Äî the code created label name strings and `BranchFixup` entries referencing them, but never called `label_offsets.insert(...)`.
   103‚Üí
   104‚Üí### Fixes Applied
   105‚Üí
   106‚Üí#### Fix 1: AtomicStore (lines 4659‚Äì4691)
   107‚Üí
   108‚Üí- **Added LR.D before SC.D** in the retry loop to properly establish a memory reservation. The loop is now: `LR.D T2, (T0)` ‚Üí `SC.D T2, T1, (T0)` ‚Üí `BNE T2, zero, retry`.
   109‚Üí- **Inserted the retry label into `label_offsets`** at the absolute byte offset of the LR.D instruction: `label_offsets.insert(retry_label.clone(), retry_abs_offset)`.
   110‚Üí- **Removed stale TODO comments** about "proper label support for atomic loops" since this IS the proper fix.
   111‚Üí- Cleaned up variable naming: `retry_offset` ‚Üí `retry_abs_offset` and `retry_label` for clarity.
   112‚Üí
   113‚Üí#### Fix 2: AtomicCas (lines 4693‚Äì4752)
   114‚Üí
   115‚Üí- **Inserted the retry label** (`__atomic_cas_retry_{offset}`) into `label_offsets` at the absolute byte offset of the LR.D instruction.
   116‚Üí- **Inserted the done label** (`__atomic_cas_done_{offset}`) into `label_offsets` at the absolute byte offset after the second BNE (where the dst store begins).
   117‚Üí- **Added branch fixup for BNE1** (current != expected ‚Üí done): Creates a `BranchFixup` targeting the `done_label` with `bne_rs1=T2, bne_rs2=T1`.
   118‚Üí- **Added branch fixup for BNE2** (SC.D failed ‚Üí retry): Creates a `BranchFixup` targeting the `retry_label` with `bne_rs1=T4, bne_rs2=Zero`.
   119‚Üí- **Removed stale TODO comment** about "proper label fixups for the retry/done branches".
   120‚Üí
   121‚Üí### How the Fix Works
   122‚Üí
   123‚ÜíBoth fixes follow the same pattern used by IR-level branches (e.g., `CondBranch`):
   124‚Üí
   125‚Üí1. Compute the absolute byte offset of the label position using `current_byte_offset + code.len() as u64`.
   126‚Üí2. Create a unique label name (e.g., `__atomic_cas_retry_128`) incorporating the absolute offset for uniqueness.
   127‚Üí3. Insert the label into `label_offsets` immediately ‚Äî no need to wait, since `label_offsets` is just a HashMap consulted later in Phase 4.
   128‚Üí4. Create `BranchFixup` entries referencing the label name.
   129‚Üí5. Phase 4's existing fixup loop resolves all labels correctly, computing `rel_offset = target_offset - branch_abs_offset` and patching the encoded BNE/JAL instructions.
   130‚Üí
   131‚Üí### Instruction Sequences After Fix
   132‚Üí
   133‚Üí**AtomicStore**:
   134‚Üí```
   135‚Üíretry:  LR.D   T2, (T0)       ; establish reservation
   136‚Üí        SC.D   T2, T1, (T0)   ; attempt store; T2=0 success, T2=1 failure
   137‚Üí        BNE    T2, zero, retry ; if failed, branch back to LR.D
   138‚Üí```
   139‚Üí
   140‚Üí**AtomicCas**:
   141‚Üí```
   142‚Üíretry:  LR.D   T2, (T0)       ; load current value & establish reservation
   143‚Üí        BNE    T2, T1, done   ; if current != expected, skip to done
   144‚Üí        SC.D   T4, T3, (T0)   ; attempt to store desired; T4=0 success, T4=1 failure
   145‚Üí        BNE    T4, zero, retry; if SC failed, branch back to LR.D
   146‚Üídone:   <store T2 to dst>     ; store the old (current) value to destination
   147‚Üí```

   149‚Üí## Task 2-d: Implement proper Compare-and-Swap (CAS) for the Wasm32 backend
   150‚Üí
   151‚Üí**Date**: 2026-03-05
   152‚Üí**File**: `src/codegen/src/wasm32/mod.rs`
   153‚Üí
   154‚Üí### Bug Description
   155‚Üí
   156‚ÜíThe Wasm32 backend's `AtomicCas` lowering was a placeholder that emitted a simple `Load` instruction, completely ignoring the `expected` and `desired` operands. Similarly, `AtomicLoad` and `AtomicStore` were lowered to their non-atomic equivalents (plain `i32.load`/`i32.store`), which do not provide any atomicity guarantees. The code had a comment stating "Atomic operations ‚Äî not natively supported in Wasm MVP", but the Wasm Threads proposal (now standardized) defines proper atomic instructions with the `0xFE` prefix byte.
   157‚Üí
   158‚Üí### Root Cause
   159‚Üí
   160‚ÜíThe original implementation predated the Wasm Threads proposal's standardization. It assumed Wasm had no native atomic support and fell back to single-threaded semantics. This is incorrect for any multi-threaded Wasm execution environment.
   161‚Üí
   162‚Üí### Changes Made
   163‚Üí
   164‚Üí#### 1. Added `shared` field to `WasmLimits` (line ~973)
   165‚Üí
   166‚ÜíWasm atomic instructions require **shared linear memory**. Added a `shared: bool` field to `WasmLimits` and updated the binary encoding to use the correct flag bits:
   167‚Üí- `0x00`: min only, not shared
   168‚Üí- `0x01`: min + max, not shared
   169‚Üí- `0x02`: min only, shared
   170‚Üí- `0x03`: min + max, shared
   171‚Üí
   172‚ÜíThe flag is computed as: `flag = (has_max as u8) | ((shared as u8) << 1)`.
   173‚Üí
   174‚Üí#### 2. Added atomic `WasmInstr` variants (lines 364‚Äì418)
   175‚Üí
   176‚ÜíAdded instruction variants covering the full set of Wasm atomic memory operations:
   177‚Üí
   178‚Üí**Atomic loads** (0xFE prefix):
   179‚Üí- `I32AtomicLoad` (0x10), `I64AtomicLoad` (0x11)
   180‚Üí- `I32AtomicLoad8U` (0x12), `I32AtomicLoad16U` (0x13)
   181‚Üí- `I64AtomicLoad8U` (0x14), `I64AtomicLoad16U` (0x15), `I64AtomicLoad32U` (0x16)
   182‚Üí
   183‚Üí**Atomic stores** (0xFE prefix):
   184‚Üí- `I32AtomicStore` (0x17), `I64AtomicStore` (0x18)

   188‚Üí**Fence**:
   189‚Üí- `MemoryAtomicFence` (0x1E, with reserved byte 0x00)
   190‚Üí
   191‚Üí**Compare-and-swap** (0xFE prefix):

   225‚Üí
   226‚Üí- `test_i32_atomic_load_encoding` ‚Äî verifies 0xFE 0x10 + alignment + offset

   228‚Üí- `test_i32_atomic_store_encoding` ‚Äî verifies 0xFE 0x17
   229‚Üí- `test_i64_atomic_store_encoding` ‚Äî verifies 0xFE 0x18

   231‚Üí- `test_i64_atomic_rmw_cmpxchg_encoding` ‚Äî verifies 0xFE 0x49
   232‚Üí- `test_subword_atomic_cmpxchg_encoding` ‚Äî verifies 0xFE 0x4A (8-bit) and 0xFE 0x4B (16-bit)
   233‚Üí- `test_memory_atomic_fence_encoding` ‚Äî verifies 0xFE 0x1E + reserved 0x00
   234‚Üí- `test_shared_memory_limits_encoding` ‚Äî verifies flag byte encoding for shared/non-shared memories
   235‚Üí
   236‚Üí### Key Design Decisions
   237‚Üí
   238‚Üí1. **No LL/SC loop needed**: Unlike AArch64 and RISC-V which require explicit load-linked/store-conditional loops, Wasm's `cmpxchg` instruction handles the retry loop internally. This is a major simplification.
   239‚Üí
   240‚Üí2. **Natural alignment enforced**: All atomic instructions use naturally aligned addresses (align = log2(access_width)). Misaligned atomic accesses trap in Wasm.
   241‚Üí
   242‚Üí3. **Sub-word support**: 8-bit and 16-bit atomic CAS is supported via the `rmw8.cmpxchg_u` and `rmw16.cmpxchg_u` instructions, which zero-extend the result to the Wasm stack type (i32 or i64).
   243‚Üí
   244‚Üí4. **Shared memory required**: The Wasm spec mandates that atomic instructions can only operate on shared memory. The production module now emits `shared: true` in its memory definition.

   246‚Üí## Task 2-h: Fix MIPS64 Ror/Rol incomplete implementation + LoongArch64 unimplemented terminators
   247‚Üí
   248‚Üí**Date**: 2026-03-05
   249‚Üí**Files**: `src/codegen/src/mips64/mod.rs`, `src/codegen/src/loongarch64/mod.rs`, `src/codegen/src/loongarch64/stack_slot_isel.rs`, `src/codegen/src/loongarch64/reg_alloc_isel.rs`
   250‚Üí
   251‚Üí### Bug 1: MIPS64 Ror/Rol Incomplete Implementation
   252‚Üí
   253‚Üí#### Bug Description
   254‚Üí
   255‚ÜíThe MIPS64 backend had two separate ROR/ROL bugs:
   256‚Üí
   257‚Üí1. **`lower_binop` function (register-allocated path, dead code)**: ROR only emitted `DSRLV` and ROL only emitted `DSLLV` ‚Äî each was missing the complementary shift + OR to complete the rotation. For example, ROR should produce `(n >> r) | (n << (64-r))` but only produced `(n >> r)`.
   258‚Üí
   259‚Üí2. **Stack-slot ISel path (active code generation)**: ROR and ROL were hardcoded to 32-bit rotation (using `imm: 32` in ADDIU), ignoring the `is_32bit` flag. 64-bit types (I64/U64) would produce incorrect results because they need a 64-bit rotation width, and they don't need the zero-extend/mask-to-32-bits steps.
   260‚Üí
   261‚Üí#### Root Cause
   262‚Üí
   263‚ÜíThe `lower_binop` function had comments acknowledging the issue: "Simplified: just emit dsrlv as placeholder (lower_ir_instr is dead code)". The stack-slot ISel path was implemented only for the I32/U32 case and never updated for 64-bit types.
   264‚Üí
   265‚Üí#### Fixes Applied
   266‚Üí
   267‚Üí##### Fix 1: `lower_binop` ‚Äî Complete ROR/ROL sequences (lines ~1831‚Äì1922)
   268‚Üí
   269‚ÜíReplaced the single-instruction placeholder with proper multi-instruction sequences, following the same pattern as `SLe` (push to `result`, return early):
   270‚Üí
   271‚Üí**ROR (64-bit)**: `(n >> r) | (n << (64-r))`
   272‚Üí```
   273‚ÜíDSRLV T2, lhs, rhs       // T2 = lhs >> rhs
   274‚ÜíDADDIU T3, $zero, 64     // T3 = 64
   275‚ÜíDSUBU  T3, T3, rhs       // T3 = 64 - rhs
   276‚ÜíDSLLV  T3, lhs, T3       // T3 = lhs << (64-rhs)
   277‚ÜíOR     dst, T2, T3       // dst = (lhs >> rhs) | (lhs << (64-rhs))
   278‚Üí```
   279‚Üí
   280‚Üí**ROL (64-bit)**: `(n << r) | (n >> (64-r))`
   281‚Üí```
   282‚ÜíDSLLV  T2, lhs, rhs      // T2 = lhs << rhs
   283‚ÜíDADDIU T3, $zero, 64     // T3 = 64
   284‚ÜíDSUBU  T3, T3, rhs       // T3 = 64 - rhs
   285‚ÜíDSRLV  T3, lhs, T3       // T3 = lhs >> (64-rhs)
   286‚ÜíOR     dst, T2, T3       // dst = (lhs << rhs) | (lhs >> (64-rhs))
   287‚Üí```
   288‚Üí

   301‚Üí
   302‚Üí#### Bug Description
   303‚Üí
   304‚ÜíThe LoongArch64 backend's Switch, Invoke, TailCall, and Resume terminators all emitted a `BREAK` instruction (intentional crash), making any IR using these terminators immediately trap at runtime. Both `stack_slot_isel.rs` and `reg_alloc_isel.rs` had this issue.

   311‚Üí
   312‚Üí#### Fixes Applied
   313‚Üí
   314‚Üí##### Fix 0: Vreg collection (both stack_slot_isel.rs and reg_alloc_isel.rs)
   315‚Üí
   316‚ÜíExtended the terminator vreg tracking `match` to include:
   317‚Üí- `Switch { discr, .. }` ‚Üí track `discr` register
   318‚Üí- `Invoke { args, .. }` ‚Üí track all `args` registers
   319‚Üí- `TailCall { args, .. }` ‚Üí track all `args` registers
   320‚Üí- `Resume { value }` ‚Üí track `value` register
   321‚Üí
   322‚Üí##### Fix 1: Switch ‚Äî Cascade of BEQ comparisons (both isel files)
   323‚Üí
   324‚Üí```loongarch
   325‚Üí# Load discriminator into S0/T0
   326‚Üíld.d   S0, fp, discr_offset      # (or read_vreg for reg_alloc_isel)
   327‚Üí# For each (value, target):
   328‚Üílu12i.w S1, value_hi              # load comparison value
   329‚Üíori    S1, S1, value_lo
   330‚Üíbeq    S0, S1, target_label       # branch if match (patched later)
   331‚Üí# Default:
   332‚Üíb      default_label               # unconditional branch to default
   333‚Üí```
   334‚Üí
   335‚ÜíNo jump table is used (PCADDU12I + LD.W + JR would require emitting data in the code section, which the framework doesn't support). The cascade-of-comparisons approach is correct and handles all switch sizes, though it's O(n) in the number of targets.
   336‚Üí
   337‚Üí##### Fix 2: Invoke ‚Äî Call + branch to normal (both isel files)
   338‚Üí
   339‚ÜíInvoke is implemented identically to Call for the invocation part, with an additional unconditional branch to the `normal` continuation block after the call returns:
   340‚Üí
   341‚Üí1. Flush caller-saved registers (reg_alloc_isel) / set up args from stack slots (stack_slot_isel)
   342‚Üí2. Load arguments into $a0‚Äì$a7 (+ stack-passed args if >8)
   343‚Üí3. BL to function (with R_LARCH_B26 relocation)
   344‚Üí4. Deallocate stack arg space (if any)
   345‚Üí5. Store return value to dst's stack slot / assign to $a0 in reg cache
   346‚Üí6. B normal (branch to normal continuation, patched later)
   347‚Üí
   348‚ÜíThe `unwind` block is tracked for control flow correctness (it's a successor of the Invoke block) but doesn't get explicit landing-pad setup ‚Äî that requires DWARF exception info which the framework doesn't emit. The unwind handler would be reached by the unwinder at runtime.
   349‚Üí
   350‚Üí##### Fix 3: TailCall ‚Äî Epilogue + B to target (both isel files)
   351‚Üí
   352‚ÜíTail call reuses the caller's stack frame. The implementation:
   353‚Üí
   354‚Üí1. Load register args into $a0‚Äì$a7 (while frame is still intact)
   355‚Üí2. For stack-passed args (>8): place them at old_sp offsets (above current frame, in the caller's arg area which is no longer needed)
   356‚Üí3. Epilogue: restore $ra, $fp, deallocate frame (same as Return, minus JIRL)
   357‚Üí4. B to target (with R_LARCH_B26 relocation, no return address save)
   358‚Üí
   359‚ÜíSince `B` doesn't modify `$ra`, the callee returns directly to the tail-caller's caller ‚Äî the key semantic of a tail call.
   360‚Üí
   361‚ÜíIn `stack_slot_isel.rs`, the stack args are placed by computing `old_sp = $sp + frame_size` and storing at `old_sp + (i-8)*8`. In `reg_alloc_isel.rs`, we flush all registers and load args from the reg cache before the epilogue.
   362‚Üí
   363‚Üí##### Fix 4: Resume ‚Äî Call __Unwind_Resume (both isel files)
   364‚Üí
   365‚ÜíResume transfers control to the unwinder, which never returns:
   366‚Üí
   367‚Üí1. Load exception value into $a0
   368‚Üí2. BL __Unwind_Resume (with R_LARCH_B26 relocation)
   369‚Üí3. BREAK (safety trap ‚Äî __Unwind_Resume should never return)
   370‚Üí
   371‚ÜíThis is a minimal but correct implementation. In a full EH system, the personality function and unwinder would use DWARF info to find the correct landing pad.
   372‚Üí
   373‚Üí### Bonus Fix: LoongArch64 mod.rs ROL

   421‚Üí
   422‚ÜíThe CAS loop uses a fixed-size instruction sequence with precomputed branch offsets:
   423‚Üí
   424‚Üí```
   425‚Üí; Register allocation: R3=addr, R1=expected, R2=desired, R0=old/result, R12=status
   426‚Üí  DMB SY                    ; +0  ‚Äî acquire barrier
   427‚Üíretry:
   428‚Üí  LDREX{,B,H} R0, [R3]     ; +4  ‚Äî load exclusive
   429‚Üí  CMP R0, R1                ; +8  ‚Äî compare old with expected
   430‚Üí  BNE done                  ; +12 ‚Äî offset_words = +2
   431‚Üí  STREX{,B,H} R12, R2, [R3]; +16 ‚Äî try store; R12 = status
   432‚Üí  CMP R12, #0               ; +20 ‚Äî check status
   433‚Üí  BNE retry                 ; +24 ‚Äî offset_words = -7
   434‚Üídone:
   435‚Üí  DMB SY                    ; +28 ‚Äî release barrier
   436‚Üí```
   437‚Üí
   438‚ÜíBranch offset derivation (ARM formula: `offset_words = (target - (branch + 8)) / 4`):
   439‚Üí- BNE done: `(28 - (12 + 8)) / 4 = +2`
   440‚Üí- BNE retry: `(4 - (24 + 8)) / 4 = -7`
   441‚Üí
   442‚ÜíType-specific variants:
   443‚Üí- I8/U8 ‚Üí LDREXB + STREXB
   444‚Üí- I16/U16 ‚Üí LDREXH + STREXH
   445‚Üí- I32/U32/I64/U64 ‚Üí LDREX + STREX
   446‚Üí
   447‚ÜíR0 holds the old value throughout the loop because STREX writes its status to R12 (not R0), so R0 is preserved for the result store.
   448‚Üí
   449‚Üí#### MIPS64: LLD/SCD-based CAS loop
   450‚Üí
   451‚ÜíAdded 5 new instruction variants to the `Instruction` enum, 4 new opcode constants, and 1 new function code:
   452‚Üí
   453‚Üí| Constant | Value | Description |
   454‚Üí|----------|-------|-------------|
   455‚Üí| `OPC_LL` | 0x30 | Load Linked opcode |
   456‚Üí| `OPC_LLD` | 0x34 | Load Linked Doubleword opcode |
   457‚Üí| `OPC_SC` | 0x38 | Store Conditional opcode |
   458‚Üí| `OPC_SCD` | 0x3C | Store Conditional Doubleword opcode |
   459‚Üí| `FN_SYNC` | 0x0F | SYNC function code |
   460‚Üí
   461‚ÜíNew instruction variants:
   462‚Üí- `Ll { rt, base, offset }` ‚Äî Load Linked (32-bit)
   463‚Üí- `Lld { rt, base, offset }` ‚Äî Load Linked Doubleword (64-bit)
   464‚Üí- `Sc { rt, base, offset }` ‚Äî Store Conditional (32-bit)
   465‚Üí- `Scd { rt, base, offset }` ‚Äî Store Conditional Doubleword (64-bit)
   466‚Üí- `Sync { stype }` ‚Äî Memory synchronization barrier
   467‚Üí
   468‚ÜíAlso added `encode()`, `mnemonic()`, and `Display` support for all new instructions.
   469‚Üí
   470‚ÜíThe CAS loop follows the MIPS64 LL/SC pattern with delay slot NOPs:
   471‚Üí
   472‚Üí```
   473‚Üí; Register allocation: T0=addr, T1=expected, T2=desired, V0=old/result, T3=temp/status
   474‚Üí  SYNC                      ; +0  ‚Äî acquire barrier
   475‚Üíretry:
   476‚Üí  LLD/LL V0, 0(T0)         ; +4  ‚Äî load linked
   477‚Üí  NOP                       ; +8  ‚Äî load-use hazard
   478‚Üí  BNE V0, T1, done          ; +12 ‚Äî if old != expected, skip
   479‚Üí  NOP                       ; +16 ‚Äî delay slot
   480‚Üí  DADDU T3, $zero, T2       ; +20 ‚Äî copy desired to temp (SCD overwrites rt)
   481‚Üí  SCD/SC T3, 0(T0)          ; +24 ‚Äî store conditional; T3 = 1 success, 0 failure
   482‚Üí  NOP                       ; +28 ‚Äî hazard
   483‚Üí  BEQ T3, $zero, retry      ; +32 ‚Äî if failed, retry
   484‚Üí  NOP                       ; +36 ‚Äî delay slot
   485‚Üídone:
   486‚Üí  SYNC                      ; +40 ‚Äî release barrier
   487‚Üí```
   488‚Üí
   489‚ÜíBranch offset derivation (MIPS formula: `offset_bytes = target - (branch + 4)`):
   490‚Üí- BNE done: `40 - (12 + 4) = +24` bytes ‚Üí 6 words
   491‚Üí- BEQ retry: `4 - (32 + 4) = -32` bytes ‚Üí -8 words
   492‚Üí
   493‚ÜíType-specific variants:
   494‚Üí- I32/U32 ‚Üí LL + SC
   495‚Üí- I64/U64/Ptr ‚Üí LLD + SCD
   496‚Üí
   497‚ÜíThe `DADDU T3, $zero, T2` instruction copies the desired value to T3 before SCD because SCD overwrites its `rt` register with the status flag (1=success, 0=failure). Without this copy, the desired value would be lost on retry. V0 holds the old value from LLD and is never overwritten by SCD (which writes to T3), so V0 is available for the result store after the loop.
   498‚Üí
   499‚Üí### Key Design Decisions
   500‚Üí
   501‚Üí1. **Precomputed branch offsets**: Both backends use fixed-size CAS loop sequences, allowing branch offsets to be computed at compile time rather than requiring label tracking and fixup passes. This simplifies the implementation and avoids interaction with the existing branch fixup mechanisms.
   502‚Üí
   503‚Üí2. **Memory barriers**: Both implementations use full memory barriers (DMB SY on ARM32, SYNC on MIPS64) before and after the CAS loop, providing sequential consistency semantics. The acquire barrier before the loop prevents reordering of the load with prior memory operations, and the release barrier after prevents reordering of the store with subsequent operations.
   504‚Üí
   505‚Üí3. **Sub-word support (ARM32 only)**: ARM32 has dedicated LDREXB/STREXB and LDREXH/STREXH instructions for byte and halfword CAS. MIPS64 does not have sub-word LL/SC variants; the implementation uses LL/SC for 32-bit types and LLD/SCD for 64-bit types only.
   506‚Üí
   507‚Üí4. **Conservative NOPs (MIPS64)**: The MIPS64 CAS loop includes NOPs after LLD/LL (load-use hazard) and SCD/SC (conservative hazard avoidance), matching the existing code generation pattern in the backend.

   593‚Üí### Register Conflict Handling (Register-Allocated Path)
   594‚Üí
   595‚ÜíThe CAS loop writes to the destination register `d` via `ldarx`, which can clobber `expected` or `desired` if they share the same physical register. The implementation handles this by:
   596‚Üí
   597‚Üí1. Using R0 and R11 as scratch registers for conflict resolution
   598‚Üí2. Copying conflicting expected/desired values to scratch registers before the CAS loop
   599‚Üí3. When both expected and desired conflict with `d`, moving expected from R0 to R11 and desired to R0
   600‚Üí
   601‚Üí### Branch Offset Computation
   602‚Üí
   603‚ÜíThe CAS loop uses fixed-size instruction sequences, allowing precomputed branch offsets:
   604‚Üí- `bc 12, 2, +3` at index 3: target = 3 + 3 = 6 (final `sync`)
   605‚Üí- `bc 12, 2, -4` at index 5: target = 5 + (-4) = 1 (retry at `ldarx`)

   622‚Üí## Task 2-a: Implement LoongArch64 Atomic Operations
   623‚Üí
   624‚Üí**Date**: 2026-03-05
   625‚Üí**Files**: `src/codegen/src/loongarch64/mod.rs`, `src/codegen/src/loongarch64/stack_slot_isel.rs`
   626‚Üí
   627‚Üí### Bug Description
   628‚Üí
   629‚ÜíThe LoongArch64 backend silently dropped atomic operations. `AtomicLoad`, `AtomicStore`, and `AtomicCas` in the stack-slot instruction selection path all returned `Vec::new()`, meaning any atomic operation in a LoongArch64-compiled program was a no-op ‚Äî producing no machine code at all.
   630‚Üí
   631‚Üí### Root Cause
   632‚Üí
   633‚ÜíThe stack-slot ISel at `stack_slot_isel.rs:1099-1100` had a single match arm for all three atomic IR instructions that returned an empty vector:
   634‚Üí
   635‚Üí```rust
   636‚ÜíIRInstr::AtomicLoad { .. } | IRInstr::AtomicStore { .. } | IRInstr::AtomicCas { .. } => {
   637‚Üí    Vec::new() // TODO: implement LoongArch64 atomic lowering
   638‚Üí}
   639‚Üí```
   640‚Üí
   641‚ÜíAdditionally, the `Instruction` enum in `mod.rs` lacked the DBAR (data barrier) and AMSWAP (atomic memory swap) instructions needed for proper atomic lowering.
   642‚Üí
   643‚Üí### LoongArch64 Atomic Instructions
   644‚Üí
   645‚ÜíReferencing the LoongArch ISA manual:
   646‚Üí
   647‚Üí- **LL.D Rd, Rj, Si14** ‚Äî Load-Linked Doubleword (64-bit, address in Rj + Si14)
   648‚Üí- **SC.D Rd, Rj, Si14** ‚Äî Store-Conditional Doubleword (stores Rd to [Rj+Si14]; Rd ‚Üê 0 on success, 1 on failure)
   649‚Üí- **LL.W / SC.W** ‚Äî 32-bit variants
   650‚Üí- **AMSWAP.D Rd, Rj, Rk** ‚Äî Atomic Memory Swap Doubleword (Rd = old [Rj], [Rj] = Rk)
   651‚Üí- **AMSWAP.W** ‚Äî 32-bit variant
   652‚Üí- **DBAR hint** ‚Äî Data Barrier (hint=0 is full barrier)
   653‚Üí
   654‚Üí### New Instructions Added
   655‚Üí
   656‚ÜíAdded to `Instruction` enum in `mod.rs`:
   657‚Üí
   658‚Üí| Instruction | Format | Opcode | Encoding |
   659‚Üí|------------|--------|--------|----------|
   660‚Üí| `AmswapW { rd, rj, rk }` | 3R | 0x00C0 (bits[31:15]) | `encode_3r(0x00C0, rk, rj, rd)` |
   661‚Üí| `AmswapD { rd, rj, rk }` | 3R | 0x00C2 (bits[31:15]) | `encode_3r(0x00C2, rk, rj, rd)` |
   662‚Üí| `Dbar { hint }` | 2RI12 | 0x0E7 (bits[31:22]) | `encode_2ri12(0x0E7, hint, 0, 0)` |
   663‚Üí
   664‚Üí### Atomic Lowering Implementation
   665‚Üí
   666‚Üí#### AtomicLoad (dst, addr, ty)
   667‚Üí
   668‚Üí```
   669‚Üídbar 0                    ; acquire fence
   670‚Üíll.{d,w} S0, S0, 0       ; load-linked from [S0+0]
   671‚Üídbar 0                    ; fence after load
   672‚Üí; store S0 to dst vreg slot
   673‚Üí```
   674‚Üí
   675‚ÜíUses LL.D for 64-bit types (I64/U64/Ptr/Func) and LL.W for 32-bit types (I8/I16/I32/U8/U16/U32).
   676‚Üí
   677‚Üí#### AtomicStore (value, addr, ty)
   678‚Üí
   679‚Üí```
   680‚Üídbar 0                         ; release fence
   681‚Üíamswap.{d,w} $r0, S1, S0      ; atomically store S0 to [S1], discard old value in $r0
   682‚Üídbar 0                         ; fence after store
   683‚Üí```
   684‚Üí
   685‚ÜíUses AMSWAP.D for 64-bit and AMSWAP.W for 32-bit types. The `$r0` destination discards the old value.
   686‚Üí
   687‚Üí#### AtomicCas (dst, addr, expected, desired, ty)
   688‚Üí
   689‚Üí```
   690‚Üí; S0=addr, S1=expected, S2=desired
   691‚Üídbar 0                         ; offset 0:  full fence
   692‚Üíll.{d,w} S3, S0, 0             ; offset 4:  S3 = current value at [S0]
   693‚Üíbne S3, S1, +5                 ; offset 8:  if current != expected, skip to dbar
   694‚Üíor S3, S2, $r0                  ; offset 12: S3 = desired (reload before each SC)
   695‚Üísc.{d,w} S3, S0, 0             ; offset 16: store desired; S3 = 0 (ok) / 1 (fail)
   696‚Üíbnez S3, -4                    ; offset 20: retry if SC failed
   697‚Üíor S3, S1, $r0                  ; offset 24: SC succeeded: old value = expected = S1
   698‚Üídbar 0                         ; offset 28: fence after CAS
   699‚Üí; store S3 (old value) to dst vreg slot
   700‚Üí```
   701‚Üí
   702‚ÜíKey design decisions:
   703‚Üí- The `or S3, S2, $r0` before SC.D reloads the desired value (from S2, which is never overwritten) because SC.D overwrites its rd register with the success flag.
   704‚Üí- When BNE fires (current ‚âÝ expected), S3 still holds the old value from LL.D, and we jump directly to dbar at offset 28.
   705‚Üí- When SC succeeds, S3 = 0, and the old value = expected = S1. The `or S3, S1, $r0` at offset 24 moves the expected value into S3 for a unified store path.
   706‚Üí- Branch offsets are precomputed since the instruction sequence has fixed size (8 instructions = 32 bytes).
   707‚Üí
   708‚Üí### Disassembler Updates
