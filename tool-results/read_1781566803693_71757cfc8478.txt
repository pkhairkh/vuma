    27вҶ’    27вҶ’## 1. Quick Start
    28вҶ’    28вҶ’
    29вҶ’    29вҶ’A VUMA program consists of one or more function definitions. The entry point is always `fn main()`. The compiler produces a native binary (or Wasm module) that can be executed directly or via QEMU for cross-architecture testing.
    30вҶ’    30вҶ’
    31вҶ’    31вҶ’### Minimal Program
    32вҶ’    32вҶ’
    33вҶ’    33вҶ’```vuma
    34вҶ’    34вҶ’fn main() -> i32 {
    35вҶ’    35вҶ’    return 0;
    36вҶ’    36вҶ’}
    37вҶ’    37вҶ’```
    38вҶ’    38вҶ’
    39вҶ’    39вҶ’This program exits with code 0.
    40вҶ’    40вҶ’
    41вҶ’    41вҶ’### Compile and Run
    42вҶ’    42вҶ’
    43вҶ’    43вҶ’VUMA compiles `.vuma` source files to native binaries. The build system uses Cargo under the hood. To build and test:
    44вҶ’    44вҶ’
    45вҶ’    45вҶ’```bash
    46вҶ’    46вҶ’# Build the VUMA compiler
    47вҶ’    47вҶ’cargo build --workspace
    48вҶ’    48вҶ’
    49вҶ’    49вҶ’# Run all tests
    50вҶ’    50вҶ’cargo test --workspace
    51вҶ’    51вҶ’
    52вҶ’    52вҶ’# Cross-architecture testing uses QEMU
    53вҶ’    53вҶ’# x86_64:
    54вҶ’    54вҶ’qemu-x86_64 ./output_binary
    55вҶ’    55вҶ’# AArch64:
    56вҶ’    56вҶ’qemu-aarch64 ./output_binary
    57вҶ’    57вҶ’# RISC-V 64:
    58вҶ’    58вҶ’qemu-riscv64-static ./output_binary
    59вҶ’    59вҶ’# ARM32:
    60вҶ’    60вҶ’qemu-arm ./output_binary
    61вҶ’    61вҶ’# MIPS64:
    62вҶ’    62вҶ’qemu-mips64-static ./output_binary
    63вҶ’    63вҶ’# PPC64:
    64вҶ’    64вҶ’qemu-ppc64 ./output_binary
    65вҶ’    65вҶ’# LoongArch64:
    66вҶ’    66вҶ’qemu-loongarch64 ./output_binary
    67вҶ’    67вҶ’```
    68вҶ’    68вҶ’
    69вҶ’    69вҶ’### Program with Memory
    70вҶ’    70вҶ’
    71вҶ’    71вҶ’```vuma
    72вҶ’    72вҶ’fn main() -> i32 {
    73вҶ’    73вҶ’    msg = allocate(3);
    74вҶ’    74вҶ’    *(msg + 0) = 97;   // 'a'
    75вҶ’    75вҶ’    *(msg + 1) = 98;   // 'b'
    76вҶ’    76вҶ’    *(msg + 2) = 99;   // 'c'
    77вҶ’    77вҶ’    free(msg);
    78вҶ’    78вҶ’    return 0;
    79вҶ’    79вҶ’}
    80вҶ’    80вҶ’```
    81вҶ’    81вҶ’
    82вҶ’    82вҶ’**Key facts**:
    83вҶ’    83вҶ’- `main()` returns `i32` (the process exit code) or `u64`.
    84вҶ’    84вҶ’- Every `allocate()` must be paired with a `free()`.
    85вҶ’    85вҶ’- Pointer dereference uses `*(ptr + offset)` syntax.
    86вҶ’    86вҶ’- Bytes are stored as integer values (e.g., `97` for ASCII `'a'`).

    90вҶ’    90вҶ’## 2. Types
    91вҶ’    91вҶ’
    92вҶ’    92вҶ’VUMA supports the following primitive types:
    93вҶ’    93вҶ’
    94вҶ’    94вҶ’| Type | Size (bytes) | Range | Signed |
    95вҶ’    95вҶ’|------|-------------|-------|--------|
    96вҶ’    96вҶ’| `i8`  | 1 | -128 to 127 | Yes |
    97вҶ’    97вҶ’| `i16` | 2 | -32768 to 32767 | Yes |
    98вҶ’    98вҶ’| `i32` | 4 | -2147483648 to 2147483647 | Yes |
    99вҶ’    99вҶ’| `i64` | 8 | -9223372036854775808 to 9223372036854775807 | Yes |
   100вҶ’   100вҶ’| `u8`  | 1 | 0 to 255 | No |
   101вҶ’   101вҶ’| `u16` | 2 | 0 to 65535 | No |
   102вҶ’   102вҶ’| `u32` | 4 | 0 to 4294967295 | No |
   103вҶ’   103вҶ’| `u64` | 8 | 0 to 18446744073709551615 | No |
   104вҶ’   104вҶ’| `Address` | 8 (4 on 32-bit targets) | N/A (opaque pointer) | No |
   105вҶ’   105вҶ’| `bool` | 1 | `true` or `false` | N/A |
   106вҶ’   106вҶ’| `void` | 0 | N/A | N/A |
   107вҶ’   107вҶ’
   108вҶ’   108вҶ’### Type Annotation Syntax
   109вҶ’   109вҶ’
   110вҶ’   110вҶ’```vuma
   111вҶ’   111вҶ’x: i32 = 42;
   112вҶ’   112вҶ’y: u32 = 0xFFFFFFFF;
   113вҶ’   113вҶ’ptr: Address = allocate(64);
   114вҶ’   114вҶ’big: u64 = 18446744073709551615;
   115вҶ’   115вҶ’flag: bool = true;
   116вҶ’   116вҶ’```
   117вҶ’   117вҶ’
   118вҶ’   118вҶ’### Critical Constraint: u32 on 64-bit Registers
   119вҶ’   119вҶ’
   120вҶ’   120вҶ’**This is the single most important thing to know about VUMA types.** All 64-bit target platforms (x86_64, AArch64, RISC-V 64, MIPS64, PPC64, LoongArch64) use 64-bit general-purpose registers. A `u32` value stored in a 64-bit register can have garbage bits in the upper 32 bits after arithmetic operations. You **must** manually mask u32 results with `& 4294967295` (which is `0xFFFFFFFF` in decimal) after any operation that could set bits above bit 31:
   121вҶ’   121вҶ’
   122вҶ’   122вҶ’```vuma
   123вҶ’   123вҶ’// WRONG: bits above bit 31 may be set
   124вҶ’   124вҶ’sum: u32 = a + b;
   125вҶ’   125вҶ’
   126вҶ’   126вҶ’// CORRECT: mask to 32 bits after addition
   127вҶ’   127вҶ’sum: u32 = (a + b) & 4294967295;
   128вҶ’   128вҶ’```
   129вҶ’   129вҶ’
   130вҶ’   130вҶ’See [Section 14: Pitfalls](#14-pitfalls) for the complete list of operations that require masking.
   131вҶ’   131вҶ’
   132вҶ’   132вҶ’### Address Type
   133вҶ’   133вҶ’
   134вҶ’   134вҶ’`Address` is the type for pointers and memory regions. It is 8 bytes on 64-bit targets and 4 bytes on 32-bit targets (ARM32, Wasm32). The `allocate()` built-in returns an `Address`.
   135вҶ’   135вҶ’
   136вҶ’   136вҶ’```vuma
   137вҶ’   137вҶ’buf: Address = allocate(256);
   138вҶ’   138вҶ’```

   142вҶ’   142вҶ’## 3. Functions
   143вҶ’   143вҶ’
   144вҶ’   144вҶ’### Function Declaration
   145вҶ’   145вҶ’
   146вҶ’   146вҶ’```vuma
   147вҶ’   147вҶ’fn function_name(param1: type1, param2: type2) -> return_type {
   148вҶ’   148вҶ’    // body
   149вҶ’   149вҶ’    return value;
   150вҶ’   150вҶ’}
   151вҶ’   151вҶ’```
   152вҶ’   152вҶ’
   153вҶ’   153вҶ’- The `fn` keyword introduces a function.
   154вҶ’   154вҶ’- Parameters have mandatory type annotations when declared in the signature.
   155вҶ’   155вҶ’- The return type follows `->`. If omitted, the function returns `void`.
   156вҶ’   156вҶ’- The body is a block enclosed in `{ }`.
   157вҶ’   157вҶ’- Use `return expr;` to return a value. The last expression in a block without `return` is also the return value in some contexts, but `return` is preferred for clarity.
   158вҶ’   158вҶ’
   159вҶ’   159вҶ’### Examples
   160вҶ’   160вҶ’
   161вҶ’   161вҶ’```vuma
   162вҶ’   162вҶ’// Function returning a value
   163вҶ’   163вҶ’fn add(a: i32, b: i32) -> i32 {
   164вҶ’   164вҶ’    return a + b;
   165вҶ’   165вҶ’}
   166вҶ’   166вҶ’
   167вҶ’   167вҶ’// Function with no return value (void)
   168вҶ’   168вҶ’fn store_byte(buf: Address, offset: u64, val: u32) {
   169вҶ’   169вҶ’    *(buf + offset) = val & 255;
   170вҶ’   170вҶ’}
   171вҶ’   171вҶ’
   172вҶ’   172вҶ’// Function with Address parameter
   173вҶ’   173вҶ’fn read_u32_be(buf: Address, offset: u64) -> u32 {
   174вҶ’   174вҶ’    b0: u32 = *(buf + offset);
   175вҶ’   175вҶ’    b1: u32 = *(buf + offset + 1);
   176вҶ’   176вҶ’    b2: u32 = *(buf + offset + 2);
   177вҶ’   177вҶ’    b3: u32 = *(buf + offset + 3);
   178вҶ’   178вҶ’    return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
   179вҶ’   179вҶ’}
   180вҶ’   180вҶ’
   181вҶ’   181вҶ’// Main entry point вҖ” must return i32 or u64
   182вҶ’   182вҶ’fn main() -> i32 {
   183вҶ’   183вҶ’    return 0;
   184вҶ’   184вҶ’}
   185вҶ’   185вҶ’```
   186вҶ’   186вҶ’
   187вҶ’   187вҶ’### Function Call
   188вҶ’   188вҶ’
   189вҶ’   189вҶ’```vuma
   190вҶ’   190вҶ’result: i32 = add(10, 20);
   191вҶ’   191вҶ’store_byte(buffer, 0, 65);
   192вҶ’   192вҶ’```
   193вҶ’   193вҶ’
   194вҶ’   194вҶ’Functions are called by name with parenthesized arguments. Function calls can appear as expressions (when the function returns a value) or as statements (when the function returns void or the return value is discarded).

   198вҶ’   198вҶ’## 4. Variables
   199вҶ’   199вҶ’
   200вҶ’   200вҶ’### Declaration with Type Annotation
   201вҶ’   201вҶ’
   202вҶ’   202вҶ’```vuma
   203вҶ’   203вҶ’name: type = expression;
   204вҶ’   204вҶ’```
   205вҶ’   205вҶ’
   206вҶ’   206вҶ’Variables are declared with a name, a type annotation, and an initial value. The type annotation is required in most cases.
   207вҶ’   207вҶ’
   208вҶ’   208вҶ’```vuma
   209вҶ’   209вҶ’x: i32 = 42;
   210вҶ’   210вҶ’y: u32 = 0xFFFFFFFF;
   211вҶ’   211вҶ’ptr: Address = allocate(64);
   212вҶ’   212вҶ’sum: u32 = (a + b) & 4294967295;
   213вҶ’   213вҶ’```
   214вҶ’   214вҶ’
   215вҶ’   215вҶ’### Declaration Without Type Annotation (Inferred)
   216вҶ’   216вҶ’
   217вҶ’   217вҶ’VUMA can infer the type from the right-hand side in some cases, but explicit annotation is strongly recommended to avoid ambiguity:
   218вҶ’   218вҶ’
   219вҶ’   219вҶ’```vuma
   220вҶ’   220вҶ’result = add1(41);       // type inferred from function return type
   221вҶ’   221вҶ’msg = allocate(3);        // type inferred as Address
   222вҶ’   222вҶ’```
   223вҶ’   223вҶ’
   224вҶ’   224вҶ’**When writing VUMA code as an LLM, always use explicit type annotations** except for variables assigned from `allocate()` (which always returns `Address`).
   225вҶ’   225вҶ’
   226вҶ’   226вҶ’### Assignment
   227вҶ’   227вҶ’
   228вҶ’   228вҶ’Once declared, variables can be reassigned without repeating the type:
   229вҶ’   229вҶ’
   230вҶ’   230вҶ’```vuma
   231вҶ’   231вҶ’x: u32 = 10;
   232вҶ’   232вҶ’x = 20;              // reassignment
   233вҶ’   233вҶ’x = (x + 1) & 4294967295;  // reassignment with expression
   234вҶ’   234вҶ’```
   235вҶ’   235вҶ’
   236вҶ’   236вҶ’### Compound Assignment
   237вҶ’   237вҶ’
   238вҶ’   238вҶ’VUMA supports compound assignment operators:
   239вҶ’   239вҶ’
   240вҶ’   240вҶ’```vuma
   241вҶ’   241вҶ’x += 1;     // x = x + 1
   242вҶ’   242вҶ’x -= 1;     // x = x - 1
   243вҶ’   243вҶ’x *= 2;     // x = x * 2
   244вҶ’   244вҶ’x &= 255;   // x = x & 255
   245вҶ’   245вҶ’x |= mask;  // x = x | mask
   246вҶ’   246вҶ’x ^= key;   // x = x ^ key
   247вҶ’   247вҶ’x <<= 4;    // x = x << 4
   248вҶ’   248вҶ’x >>= 4;    // x = x >> 4
   249вҶ’   249вҶ’```
   250вҶ’   250вҶ’
   251вҶ’   251вҶ’**Note**: Compound assignments on `u32` values still need the `& 4294967295` mask. The compound form does **not** automatically mask. Prefer explicit assignment with masking:
   252вҶ’   252вҶ’
   253вҶ’   253вҶ’```vuma
   254вҶ’   254вҶ’// WRONG for u32 on 64-bit registers:
   255вҶ’   255вҶ’x += 1;
   256вҶ’   256вҶ’
   257вҶ’   257вҶ’// CORRECT:
   258вҶ’   258вҶ’x = (x + 1) & 4294967295;
   259вҶ’   259вҶ’```

   263вҶ’   263вҶ’## 5. Control Flow
   264вҶ’   264вҶ’
   265вҶ’   265вҶ’### For Loop (Range)
   266вҶ’   266вҶ’
   267вҶ’   267вҶ’The `for` loop iterates over an exclusive range `start..end`. The loop variable takes each integer value from `start` (inclusive) to `end` (exclusive).
   268вҶ’   268вҶ’
   269вҶ’   269вҶ’```vuma
   270вҶ’   270вҶ’for i in 0..10 {
   271вҶ’   271вҶ’    // i = 0, 1, 2, ..., 9
   272вҶ’   272вҶ’}
   273вҶ’   273вҶ’
   274вҶ’   274вҶ’for i in 0..64 {
   275вҶ’   275вҶ’    // i = 0, 1, 2, ..., 63
   276вҶ’   276вҶ’}
   277вҶ’   277вҶ’
   278вҶ’   278вҶ’for i in 16..64 {
   279вҶ’   279вҶ’    // i = 16, 17, ..., 63
   280вҶ’   280вҶ’}
   281вҶ’   281вҶ’```
   282вҶ’   282вҶ’
   283вҶ’   283вҶ’**The range is always exclusive on the upper bound**: `0..10` means `0 <= i < 10`.
   284вҶ’   284вҶ’
   285вҶ’   285вҶ’### For Loop with Memory Access
   286вҶ’   286вҶ’
   287вҶ’   287вҶ’```vuma
   288вҶ’   288вҶ’// Copy 32 bytes from src to dst
   289вҶ’   289вҶ’fn copy32(dst: Address, src: Address) {
   290вҶ’   290вҶ’    i: u64 = 0;
   291вҶ’   291вҶ’    for i in 0..32 {
   292вҶ’   292вҶ’        *(dst + i) = *(src + i);
   293вҶ’   293вҶ’    }
   294вҶ’   294вҶ’}
   295вҶ’   295вҶ’```
   296вҶ’   296вҶ’
   297вҶ’   297вҶ’### While Loop
   298вҶ’   298вҶ’
   299вҶ’   299вҶ’```vuma
   300вҶ’   300вҶ’current = (*list).next;
   301вҶ’   301вҶ’while current != list {
   302вҶ’   302вҶ’    process(current);
   303вҶ’   303вҶ’    current = (*current).next;
   304вҶ’   304вҶ’}
   305вҶ’   305вҶ’```
   306вҶ’   306вҶ’
   307вҶ’   307вҶ’### If / Else
   308вҶ’   308вҶ’
   309вҶ’   309вҶ’```vuma
   310вҶ’   310вҶ’if x > 100 {
   311вҶ’   311вҶ’    y = 2;
   312вҶ’   312вҶ’} else {
   313вҶ’   313вҶ’    y = 1;
   314вҶ’   314вҶ’}
   315вҶ’   315вҶ’```
   316вҶ’   316вҶ’
   317вҶ’   317вҶ’```vuma
   318вҶ’   318вҶ’if value & 1 == 1 {
   319вҶ’   319вҶ’    // odd
   320вҶ’   320вҶ’}
   321вҶ’   321вҶ’```
   322вҶ’   322вҶ’
   323вҶ’   323вҶ’Nested conditionals:
   324вҶ’   324вҶ’
   325вҶ’   325вҶ’```vuma
   326вҶ’   326вҶ’if x > 0 {
   327вҶ’   327вҶ’    if x > 100 {
   328вҶ’   328вҶ’        result = 2;
   329вҶ’   329вҶ’    } else {
   330вҶ’   330вҶ’        result = 1;
   331вҶ’   331вҶ’    }
   332вҶ’   332вҶ’} else {
   333вҶ’   333вҶ’    result = 0;
   334вҶ’   334вҶ’}
   335вҶ’   335вҶ’```
   336вҶ’   336вҶ’
   337вҶ’   337вҶ’### Infinite Loop
   338вҶ’   338вҶ’
   339вҶ’   339вҶ’```vuma
   340вҶ’   340вҶ’loop {
   341вҶ’   341вҶ’    // runs forever (useful in embedded/bare-metal contexts)
   342вҶ’   342вҶ’}
   343вҶ’   343вҶ’```
   344вҶ’   344вҶ’
   345вҶ’   345вҶ’### Break and Continue
   346вҶ’   346вҶ’

   348вҶ’   348вҶ’for i in 0..100 {

   361вҶ’   361вҶ’## 6. Memory
   362вҶ’   362вҶ’
   363вҶ’   363вҶ’VUMA provides two first-class memory operations: `allocate` and `free`. There is no garbage collector; memory management is manual.
   364вҶ’   364вҶ’
   365вҶ’   365вҶ’### allocate(n)
   366вҶ’   366вҶ’
   367вҶ’   367вҶ’Reserves `n` bytes of memory and returns an `Address` (pointer to the start of the block). The allocated memory is uninitialized.
   368вҶ’   368вҶ’
   369вҶ’   369вҶ’```vuma
   370вҶ’   370вҶ’buf = allocate(64);     // 64-byte buffer
   371вҶ’   371вҶ’state = allocate(32);   // 32-byte state buffer
   372вҶ’   372вҶ’k_table = allocate(256); // 256-byte constant table
   373вҶ’   373вҶ’```
   374вҶ’   374вҶ’
   375вҶ’   375вҶ’**The returned `Address` does not need a type annotation** вҖ” it is always inferred as `Address`:
   376вҶ’   376вҶ’
   377вҶ’   377вҶ’```vuma
   378вҶ’   378вҶ’state = allocate(32);   // state has type Address
   379вҶ’   379вҶ’```
   380вҶ’   380вҶ’
   381вҶ’   381вҶ’### free(ptr)
   382вҶ’   382вҶ’
   383вҶ’   383вҶ’Releases the memory block pointed to by `ptr`. After `free(ptr)`, the pointer is invalid and must not be used.
   384вҶ’   384вҶ’
   385вҶ’   385вҶ’```vuma
   386вҶ’   386вҶ’free(state);
   387вҶ’   387вҶ’free(k_table);
   388вҶ’   388вҶ’free(buf);
   389вҶ’   389вҶ’```
   390вҶ’   390вҶ’
   391вҶ’   391вҶ’**Every `allocate` must be matched with a `free`**. Failing to free memory is a resource leak. The IVE (Invariant Verification Engine) checks this at compile time.
   392вҶ’   392вҶ’
   393вҶ’   393вҶ’### Memory Layout Example
   394вҶ’   394вҶ’
   395вҶ’   395вҶ’```vuma
   396вҶ’   396вҶ’fn sha256d(msg: Address, msg_len: u64, out: Address) {
   397вҶ’   397вҶ’    // Allocate working memory
   398вҶ’   398вҶ’    state = allocate(32);
   399вҶ’   399вҶ’    k = allocate(256);
   400вҶ’   400вҶ’    w = allocate(256);
   401вҶ’   401вҶ’    block = allocate(64);
   402вҶ’   402вҶ’    inner = allocate(32);
   403вҶ’   403вҶ’
   404вҶ’   404вҶ’    // ... use the memory ...
   405вҶ’   405вҶ’
   406вҶ’   406вҶ’    // Clean up all allocations
   407вҶ’   407вҶ’    free(state);
   408вҶ’   408вҶ’    free(k);
   409вҶ’   409вҶ’    free(w);
   410вҶ’   410вҶ’    free(block);
   411вҶ’   411вҶ’    free(inner);
   412вҶ’   412вҶ’}
   413вҶ’   413вҶ’```
   414вҶ’   414вҶ’
   415вҶ’   415вҶ’### Region Declarations
   416вҶ’   416вҶ’
   417вҶ’   417вҶ’Named regions can be declared with the `region` keyword:
   418вҶ’   418вҶ’
   419вҶ’   419вҶ’```vuma
   420вҶ’   420вҶ’region pool = allocate(4096);
   421вҶ’   421вҶ’```
   422вҶ’   422вҶ’
   423вҶ’   423вҶ’---
   424вҶ’   424вҶ’
   425вҶ’   425вҶ’## 7. Pointer Operations
   426вҶ’   426вҶ’
   427вҶ’   427вҶ’### Dereference: Load
   428вҶ’   428вҶ’
   429вҶ’   429вҶ’Read a byte from memory at `ptr + offset`:
   430вҶ’   430вҶ’
   431вҶ’   431вҶ’```vuma
   432вҶ’   432вҶ’b0: u32 = *(buf + offset);
   433вҶ’   433вҶ’```
   434вҶ’   434вҶ’
   435вҶ’   435вҶ’The result is a single byte zero-extended to the type of the variable (typically `u32`).
   436вҶ’   436вҶ’
   437вҶ’   437вҶ’### Dereference: Store
   438вҶ’   438вҶ’
   439вҶ’   439вҶ’Write a byte to memory at `ptr + offset`:
   440вҶ’   440вҶ’
   441вҶ’   441вҶ’```vuma
   442вҶ’   442вҶ’*(buf + offset) = value & 255;
   443вҶ’   443вҶ’```
   444вҶ’   444вҶ’
   445вҶ’   445вҶ’Only the low 8 bits of `value` are stored.
   446вҶ’   446вҶ’
   447вҶ’   447вҶ’### Pointer Arithmetic
   448вҶ’   448вҶ’
   449вҶ’   449вҶ’Pointer + integer produces a new `Address` offset by that many bytes:
   450вҶ’   450вҶ’
   451вҶ’   451вҶ’```vuma
   452вҶ’   452вҶ’ptr = base + 128;      // 128 bytes forward
   453вҶ’   453вҶ’byte3 = *(buf + 3);    // read byte at offset 3
   454вҶ’   454вҶ’*(buf + 4) = 99;       // write byte at offset 4
   455вҶ’   455вҶ’```
   456вҶ’   456вҶ’
   457вҶ’   457вҶ’The offset can be an expression:
   458вҶ’   458вҶ’
   459вҶ’   459вҶ’```vuma
   460вҶ’   460вҶ’*(buf + i * 4) = val;  // write at byte offset i*4
   461вҶ’   461вҶ’byte_val = *(buf + i); // read at byte offset i
   462вҶ’   462вҶ’```
   463вҶ’   463вҶ’
   464вҶ’   464вҶ’### Read/Write Multi-byte Values (u32)
   465вҶ’   465вҶ’
   466вҶ’   466вҶ’VUMA does not have native multi-byte load/store. You must compose byte-level accesses. For big-endian u32:
   467вҶ’   467вҶ’
   468вҶ’   468вҶ’```vuma
   469вҶ’   469вҶ’fn write_u32_be(buf: Address, offset: u64, val: u32) {
   470вҶ’   470вҶ’    *(buf + offset) = (val >> 24) & 255;
   471вҶ’   471вҶ’    *(buf + offset + 1) = (val >> 16) & 255;
   472вҶ’   472вҶ’    *(buf + offset + 2) = (val >> 8) & 255;
   473вҶ’   473вҶ’    *(buf + offset + 3) = val & 255;
   474вҶ’   474вҶ’}
   475вҶ’   475вҶ’
   476вҶ’   476вҶ’fn read_u32_be(buf: Address, offset: u64) -> u32 {
   477вҶ’   477вҶ’    b0: u32 = *(buf + offset);
   478вҶ’   478вҶ’    b1: u32 = *(buf + offset + 1);
   479вҶ’   479вҶ’    b2: u32 = *(buf + offset + 2);
   480вҶ’   480вҶ’    b3: u32 = *(buf + offset + 3);
   481вҶ’   481вҶ’    return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
   482вҶ’   482вҶ’}
   483вҶ’   483вҶ’```
   484вҶ’   484вҶ’
   485вҶ’   485вҶ’**Why the `& 4294967295` mask?** After left-shifting `b0` by 24 bits in a 64-bit register, bits above bit 31 may be set. The mask clears them. This is the most common source of bugs in VUMA code. See [Section 14: Pitfalls](#14-pitfalls).
   486вҶ’   486вҶ’
   487вҶ’   487вҶ’### Address-Of Operator
   488вҶ’   488вҶ’
   489вҶ’   489вҶ’The `@` operator takes the address of a variable:
   490вҶ’   490вҶ’
   491вҶ’   491вҶ’```vuma
   492вҶ’   492вҶ’let x: i32 = 42;
   493вҶ’   493вҶ’ptr = @x;       // ptr has type Address, points to x
   494вҶ’   494вҶ’*ptr = 100;     // modifies x through the pointer
   495вҶ’   495вҶ’```
   496вҶ’   496вҶ’
   497вҶ’   497вҶ’### Derive
   498вҶ’   498вҶ’
   499вҶ’   499вҶ’The `derive(ptr, region)` expression explicitly creates a derived pointer within a specific region. This is the formal mechanism for sub-allocation:
   500вҶ’   500вҶ’
   501вҶ’   501вҶ’```vuma
   502вҶ’   502вҶ’inner = derive(ptr, arena_region);
   503вҶ’   503вҶ’```
   504вҶ’   504вҶ’
   505вҶ’   505вҶ’In practice, pointer arithmetic (`base + offset`) implicitly derives a new pointer.
   506вҶ’   506вҶ’
   507вҶ’   507вҶ’---
   508вҶ’   508вҶ’
   509вҶ’   509вҶ’## 8. Bitwise Operations
   510вҶ’   510вҶ’
   511вҶ’   511вҶ’VUMA supports the standard bitwise operations:
   512вҶ’   512вҶ’
   513вҶ’   513вҶ’| Operator | Name | Example |
   514вҶ’   514вҶ’|----------|------|---------|
   515вҶ’   515вҶ’| `&` | Bitwise AND | `a & 0xFF` |
   516вҶ’   516вҶ’| `\|` | Bitwise OR | `a \| b` |
   517вҶ’   517вҶ’| `^` | Bitwise XOR | `a ^ 0xFFFFFFFF` |
   518вҶ’   518вҶ’| `<<` | Left shift | `a << 8` |
   519вҶ’   519вҶ’| `>>` | Right shift | `a >> 4` |
   520вҶ’   520вҶ’
   521вҶ’   521вҶ’### No Built-in NOT Operator
   522вҶ’   522вҶ’
   523вҶ’   523вҶ’**VUMA does not have a reliable bitwise NOT (`~`) operator for u32 values.** The `~x` operator inverts all 64 bits of the register, which corrupts the upper 32 bits. Instead, use XOR with `0xFFFFFFFF`:
   524вҶ’   524вҶ’
   525вҶ’   525вҶ’```vuma
   526вҶ’   526вҶ’// WRONG: ~x inverts bits 32-63 which corrupts u32 values
   527вҶ’   527вҶ’not_x = ~x;
   528вҶ’   528вҶ’
   529вҶ’   529вҶ’// CORRECT: XOR with 0xFFFFFFFF inverts only the lower 32 bits
   530вҶ’   530вҶ’not_x: u32 = x ^ 4294967295;
   531вҶ’   531вҶ’```
   532вҶ’   532вҶ’
   533вҶ’   533вҶ’**Always use `x ^ 4294967295` instead of `~x` for u32 bitwise NOT.**
   534вҶ’   534вҶ’
   535вҶ’   535вҶ’### Bitwise AND for Masking
   536вҶ’   536вҶ’
   537вҶ’   537вҶ’```vuma
   538вҶ’   538вҶ’low_byte: u32 = value & 255;          // mask to 8 bits
   539вҶ’   539вҶ’low_16: u32 = value & 65535;          // mask to 16 bits
   540вҶ’   540вҶ’low_32: u32 = value & 4294967295;     // mask to 32 bits
   541вҶ’   541вҶ’```
   542вҶ’   542вҶ’
   543вҶ’   543вҶ’### Bitwise OR for Combining
   544вҶ’   544вҶ’
   545вҶ’   545вҶ’```vuma
   546вҶ’   546вҶ’combined: u32 = (high << 8) | low;
   547вҶ’   547вҶ’```
   548вҶ’   548вҶ’
   549вҶ’   549вҶ’### Left Shift
   550вҶ’   550вҶ’
   551вҶ’   551вҶ’```vuma
   552вҶ’   552вҶ’shifted: u32 = (value << 24) & 4294967295;  // MUST mask after left shift
   553вҶ’   553вҶ’```
   554вҶ’   554вҶ’
   555вҶ’   555вҶ’**Left shift on u32 values always requires masking** because the result can have bits set above bit 31.
   556вҶ’   556вҶ’
   557вҶ’   557вҶ’### Right Shift
   558вҶ’   558вҶ’
   559вҶ’   559вҶ’```vuma
   560вҶ’   560вҶ’upper: u32 = value >> 24;   // no mask needed for right shift
   561вҶ’   561вҶ’```
   562вҶ’   562вҶ’
   563вҶ’   563вҶ’Right shift does not require masking because it only clears bits (never sets new bits above the existing ones).
   564вҶ’   564вҶ’
   565вҶ’   565вҶ’### Rotate Right (Compose from Shifts)
   566вҶ’   566вҶ’
   567вҶ’   567вҶ’VUMA does not have a built-in rotate operator. Compose it from shifts:
   568вҶ’   568вҶ’
   569вҶ’   569вҶ’```vuma
   570вҶ’   570вҶ’fn rotr32(x: u32, n: u32) -> u32 {
   571вҶ’   571вҶ’    return ((x >> n) | (x << (32 - n))) & 4294967295;
   572вҶ’   572вҶ’}
   573вҶ’   573вҶ’```
   574вҶ’   574вҶ’
   575вҶ’   575вҶ’The mask is required because `x << (32 - n)` sets bits above bit 31.
   576вҶ’   576вҶ’
   577вҶ’   577вҶ’---
   578вҶ’   578вҶ’
   579вҶ’   579вҶ’## 9. Arithmetic
   580вҶ’   580вҶ’
   581вҶ’   581вҶ’### Basic Operators
   582вҶ’   582вҶ’
   583вҶ’   583вҶ’| Operator | Name | Example |
   584вҶ’   584вҶ’|----------|------|---------|
   585вҶ’   585вҶ’| `+` | Addition | `a + b` |
   586вҶ’   586вҶ’| `-` | Subtraction | `a - b` |
   587вҶ’   587вҶ’| `*` | Multiplication | `a * b` |
   588вҶ’   588вҶ’| `/` | Division | `a / b` |
   589вҶ’   589вҶ’| `%` | Modulo | `a % b` |
   590вҶ’   590вҶ’
   591вҶ’   591вҶ’### u32 Masking Rule
   592вҶ’   592вҶ’
   593вҶ’   593вҶ’**All u32 arithmetic results must be masked with `& 4294967295`.** VUMA uses 64-bit registers on 64-bit targets. Arithmetic on u32 values can produce carries into the upper 32 bits:
   594вҶ’   594вҶ’
   595вҶ’   595вҶ’```vuma
   596вҶ’   596вҶ’// WRONG: overflow bits pollute upper 32 bits
   597вҶ’   597вҶ’sum: u32 = a + b;
   598вҶ’   598вҶ’
   599вҶ’   599вҶ’// CORRECT: mask to 32 bits
   600вҶ’   600вҶ’sum: u32 = (a + b) & 4294967295;
   601вҶ’   601вҶ’```
   602вҶ’   602вҶ’
   603вҶ’   603вҶ’This applies to **all** arithmetic operations on u32 values:
   604вҶ’   604вҶ’
   605вҶ’   605вҶ’```vuma
   606вҶ’   606вҶ’sum: u32 = (a + b) & 4294967295;
   607вҶ’   607вҶ’diff: u32 = (a - b) & 4294967295;
   608вҶ’   608вҶ’prod: u32 = (a * b) & 4294967295;
   609вҶ’   609вҶ’quot: u32 = (a / b) & 4294967295;
   610вҶ’   610вҶ’rem: u32 = (a % b) & 4294967295;
   611вҶ’   611вҶ’```
   612вҶ’   612вҶ’
   613вҶ’   613вҶ’Division and modulo technically cannot overflow for u32, but masking is still recommended for consistency and safety.
   614вҶ’   614вҶ’
   615вҶ’   615вҶ’### u64 Arithmetic
   616вҶ’   616вҶ’
   617вҶ’   617вҶ’u64 arithmetic does **not** require masking:
   618вҶ’   618вҶ’
   619вҶ’   619вҶ’```vuma
   620вҶ’   620вҶ’big_sum: u64 = a + b;    // OK, no mask needed
   621вҶ’   621вҶ’```
   622вҶ’   622вҶ’
   623вҶ’   623вҶ’### i32 Arithmetic
   624вҶ’   624вҶ’
   625вҶ’   625вҶ’Signed 32-bit arithmetic should also be masked for correctness:
   626вҶ’   626вҶ’
   627вҶ’   627вҶ’```vuma
   628вҶ’   628вҶ’signed_result: i32 = (a + b) & 4294967295;
   629вҶ’   629вҶ’```
   630вҶ’   630вҶ’
   631вҶ’   631вҶ’### Chained Arithmetic
   632вҶ’   632вҶ’
   633вҶ’   633вҶ’When multiple u32 operations are chained, mask the **final** result:
   634вҶ’   634вҶ’
   635вҶ’   635вҶ’```vuma
   636вҶ’   636вҶ’// SHA-256 T1 computation
   637вҶ’   637вҶ’t1: u32 = (h + big_sigma1(e) + ch(e, f, g) + ki + wi) & 4294967295;
   638вҶ’   638вҶ’```
   639вҶ’   639вҶ’
   640вҶ’   640вҶ’Intermediate results in sub-expressions (like `big_sigma1(e)`) should also be masked in their function definitions. Every function that returns u32 must mask its return value.
   641вҶ’   641вҶ’

   725вҶ’   725вҶ’### Important: Decimal vs Hex for Masking
   726вҶ’   726вҶ’
   727вҶ’   727вҶ’The u32 mask value `4294967295` is `0xFFFFFFFF`. Both notations are equivalent:
   728вҶ’   728вҶ’
   729вҶ’   729вҶ’```vuma
   730вҶ’   730вҶ’x = value & 4294967295;   // decimal
   731вҶ’   731вҶ’x = value & 0xFFFFFFFF;   // hex (equivalent)
   732вҶ’   732вҶ’```
   733вҶ’   733вҶ’
   734вҶ’   734вҶ’**In existing VUMA code, the decimal form `4294967295` is conventional for u32 masking.** When generating VUMA code, prefer the decimal form for consistency with the existing codebase.

   747вҶ’   747вҶ’## 12. Calling Convention
   748вҶ’   748вҶ’
   749вҶ’   749вҶ’### Argument Passing
   750вҶ’   750вҶ’
   751вҶ’   751вҶ’On 64-bit targets, the first 8 integer/pointer arguments are passed in registers (X0-X7 on AArch64, RDI/RSI/RDX/RCX/R8/R9 on x86_64, a0-a7 on RISC-V). Additional arguments are passed on the stack.
   752вҶ’   752вҶ’
   753вҶ’   753вҶ’On 32-bit targets (ARM32, Wasm32), the first 4 arguments use registers (r0-r3 on ARM32, the value stack on Wasm32).
   754вҶ’   754вҶ’
   755вҶ’   755вҶ’### Return Values
   756вҶ’   756вҶ’
   757вҶ’   757вҶ’The return value is passed in the first return register (X0 on AArch64, RAX on x86_64, a0 on RISC-V). For void functions, no return register is used.
   758вҶ’   758вҶ’
   759вҶ’   759вҶ’### Address Parameters
   760вҶ’   760вҶ’
   761вҶ’   761вҶ’`Address` parameters are passed like integer values (pointer-sized). On 64-bit targets, they occupy 8 bytes; on 32-bit targets, 4 bytes.
   762вҶ’   762вҶ’
   763вҶ’   763вҶ’### Function Call Example
   764вҶ’   764вҶ’
   765вҶ’   765вҶ’```vuma
   766вҶ’   766вҶ’fn add1(x: i32) -> i32 {
   767вҶ’   767вҶ’    return x + 1;
   768вҶ’   768вҶ’}
   769вҶ’   769вҶ’
   770вҶ’   770вҶ’fn main() -> i32 {
   771вҶ’   771вҶ’    return add1(41);  // returns 42
   772вҶ’   772вҶ’}
   773вҶ’   773вҶ’```
   774вҶ’   774вҶ’
   775вҶ’   775вҶ’### Cross-Function u32 Values
   776вҶ’   776вҶ’
   777вҶ’   777вҶ’When a function returns `u32`, the caller receives the value in a 64-bit register. The upper 32 bits may contain garbage. The **returning function** must mask the value before returning, and the **calling function** should not assume the upper bits are clear unless the callee guarantees it.
   778вҶ’   778вҶ’
   779вҶ’   779вҶ’```vuma
   780вҶ’   780вҶ’fn safe_add(a: u32, b: u32) -> u32 {
   781вҶ’   781вҶ’    return (a + b) & 4294967295;  // callee masks the result
   782вҶ’   782вҶ’}
   783вҶ’   783вҶ’
   784вҶ’   784вҶ’fn main() -> i32 {
   785вҶ’   785вҶ’    result: u32 = safe_add(0xFFFFFFFF, 1);  // result = 0
   786вҶ’   786вҶ’    return 0;
   787вҶ’   787вҶ’}
   788вҶ’   788вҶ’```
   789вҶ’   789вҶ’
   790вҶ’   790вҶ’### Stack Layout (AArch64)

   814вҶ’   814вҶ’## 13. Common Patterns
   815вҶ’   815вҶ’
   816вҶ’   816вҶ’### Pattern 1: SHA256d (Double SHA-256)
   817вҶ’   817вҶ’
   818вҶ’   818вҶ’This is the canonical VUMA program. It exercises all major language features: functions, memory, pointer arithmetic, bitwise operations, u32 masking, and loops.
   819вҶ’   819вҶ’
   820вҶ’   820вҶ’```vuma
   821вҶ’   821вҶ’fn rotr32(x: u32, n: u32) -> u32 {
   822вҶ’   822вҶ’    return ((x >> n) | (x << (32 - n))) & 4294967295;
   823вҶ’   823вҶ’}
   824вҶ’   824вҶ’
   825вҶ’   825вҶ’fn ch(x: u32, y: u32, z: u32) -> u32 {
   826вҶ’   826вҶ’    return (x & y) ^ ((x ^ 4294967295) & z);
   827вҶ’   827вҶ’}
   828вҶ’   828вҶ’
   829вҶ’   829вҶ’fn maj(a: u32, b: u32, c: u32) -> u32 {
   830вҶ’   830вҶ’    return (a & b) ^ (a & c) ^ (b & c);
   831вҶ’   831вҶ’}
   832вҶ’   832вҶ’
   833вҶ’   833вҶ’fn big_sigma0(x: u32) -> u32 {
   834вҶ’   834вҶ’    return rotr32(x, 2) ^ rotr32(x, 13) ^ rotr32(x, 22);
   835вҶ’   835вҶ’}
   836вҶ’   836вҶ’
   837вҶ’   837вҶ’fn big_sigma1(x: u32) -> u32 {
   838вҶ’   838вҶ’    return rotr32(x, 6) ^ rotr32(x, 11) ^ rotr32(x, 25);
   839вҶ’   839вҶ’}
   840вҶ’   840вҶ’
   841вҶ’   841вҶ’fn small_sigma0(x: u32) -> u32 {
   842вҶ’   842вҶ’    return (rotr32(x, 7) ^ rotr32(x, 18) ^ (x >> 3)) & 4294967295;
   843вҶ’   843вҶ’}
   844вҶ’   844вҶ’
   845вҶ’   845вҶ’fn small_sigma1(x: u32) -> u32 {
   846вҶ’   846вҶ’    return (rotr32(x, 17) ^ rotr32(x, 19) ^ (x >> 10)) & 4294967295;
   847вҶ’   847вҶ’}
   848вҶ’   848вҶ’
   849вҶ’   849вҶ’fn write_u32_be(buf: Address, offset: u64, val: u32) {
   850вҶ’   850вҶ’    *(buf + offset) = (val >> 24) & 255;
   851вҶ’   851вҶ’    *(buf + offset + 1) = (val >> 16) & 255;
   852вҶ’   852вҶ’    *(buf + offset + 2) = (val >> 8) & 255;
   853вҶ’   853вҶ’    *(buf + offset + 3) = val & 255;
   854вҶ’   854вҶ’}
   855вҶ’   855вҶ’
   856вҶ’   856вҶ’fn read_u32_be(buf: Address, offset: u64) -> u32 {
   857вҶ’   857вҶ’    b0: u32 = *(buf + offset);
   858вҶ’   858вҶ’    b1: u32 = *(buf + offset + 1);
   859вҶ’   859вҶ’    b2: u32 = *(buf + offset + 2);
   860вҶ’   860вҶ’    b3: u32 = *(buf + offset + 3);
   861вҶ’   861вҶ’    return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
   862вҶ’   862вҶ’}
   863вҶ’   863вҶ’
   864вҶ’   864вҶ’fn main() -> i32 {
   865вҶ’   865вҶ’    msg = allocate(3);
   866вҶ’   866вҶ’    *(msg + 0) = 97;
   867вҶ’   867вҶ’    *(msg + 1) = 98;
   868вҶ’   868вҶ’    *(msg + 2) = 99;
   869вҶ’   869вҶ’    // ... SHA-256 compression ...
   870вҶ’   870вҶ’    free(msg);
   871вҶ’   871вҶ’    return 0;
   872вҶ’   872вҶ’}
   873вҶ’   873вҶ’```
   874вҶ’   874вҶ’
   875вҶ’   875вҶ’### Pattern 2: Memory Buffer (Read/Write Bytes)
   876вҶ’   876вҶ’
   877вҶ’   877вҶ’```vuma
   878вҶ’   878вҶ’fn main() -> i32 {
   879вҶ’   879вҶ’    buf = allocate(16);
   880вҶ’   880вҶ’

   882вҶ’   882вҶ’    *(buf + 0) = 0x4f;
   883вҶ’   883вҶ’    *(buf + 1) = 0x8b;
   884вҶ’   884вҶ’
   885вҶ’   885вҶ’    // Read bytes back
   886вҶ’   886вҶ’    b0: u32 = *(buf + 0);
   887вҶ’   887вҶ’    b1: u32 = *(buf + 1);
   888вҶ’   888вҶ’

   893вҶ’   893вҶ’
   894вҶ’   894вҶ’### Pattern 3: Byte Manipulation (Extract/Insert)
   895вҶ’   895вҶ’
   896вҶ’   896вҶ’```vuma
   897вҶ’   897вҶ’// Extract byte n from a u32 (big-endian)
   898вҶ’   898вҶ’fn get_byte(val: u32, n: u32) -> u32 {
   899вҶ’   899вҶ’    return (val >> (n * 8)) & 255;
   900вҶ’   900вҶ’}
   901вҶ’   901вҶ’
   902вҶ’   902вҶ’// Set byte n in a u32 (big-endian)
   903вҶ’   903вҶ’fn set_byte(val: u32, n: u32, byte: u32) -> u32 {
   904вҶ’   904вҶ’    shift: u32 = n * 8;
   905вҶ’   905вҶ’    mask: u32 = 255 << shift;
   906вҶ’   906вҶ’    return (val & (mask ^ 4294967295)) | ((byte & 255) << shift) & 4294967295;
   907вҶ’   907вҶ’}
   908вҶ’   908вҶ’```
   909вҶ’   909вҶ’
   910вҶ’   910вҶ’### Pattern 4: Memory Copy
   911вҶ’   911вҶ’
   912вҶ’   912вҶ’```vuma
   913вҶ’   913вҶ’fn copy_bytes(dst: Address, src: Address, len: u64) {
   914вҶ’   914вҶ’    i: u64 = 0;
   915вҶ’   915вҶ’    for i in 0..len {
   916вҶ’   916вҶ’        *(dst + i) = *(src + i);

   920вҶ’   920вҶ’
   921вҶ’   921вҶ’### Pattern 5: Memory Set (Fill with Value)
   922вҶ’   922вҶ’
   923вҶ’   923вҶ’```vuma

   925вҶ’   925вҶ’    i: u64 = 0;
   926вҶ’   926вҶ’    for i in 0..len {
   927вҶ’   927вҶ’        *(buf + i) = val & 255;

   929вҶ’   929вҶ’}
   930вҶ’   930вҶ’```
   931вҶ’   931вҶ’
   932вҶ’   932вҶ’### Pattern 6: Array-style Access via Stride
   933вҶ’   933вҶ’
   934вҶ’   934вҶ’When storing an array of u32 values in a byte buffer, each element occupies 4 bytes (stride = 4):
   935вҶ’   935вҶ’
   936вҶ’   936вҶ’```vuma
   937вҶ’   937вҶ’fn w_store(w_base: Address, idx: u64, val: u32) {
   938вҶ’   938вҶ’    write_u32_be(w_base, idx * 4, val);
   939вҶ’   939вҶ’}
   940вҶ’   940вҶ’
   941вҶ’   941вҶ’fn w_load(w_base: Address, idx: u64) -> u32 {
   942вҶ’   942вҶ’    return read_u32_be(w_base, idx * 4);
   943вҶ’   943вҶ’}
   944вҶ’   944вҶ’```
   945вҶ’   945вҶ’
   946вҶ’   946вҶ’### Pattern 7: SHA-256 Compression Round
   947вҶ’   947вҶ’
   948вҶ’   948вҶ’```vuma
   949вҶ’   949вҶ’for i in 0..64 {
   950вҶ’   950вҶ’    ki: u32 = read_u32_be(k, i * 4);
   951вҶ’   951вҶ’    wi: u32 = w_load(w, i);
   952вҶ’   952вҶ’    t1: u32 = (h + big_sigma1(e) + ch(e, f, g) + ki + wi) & 4294967295;
   953вҶ’   953вҶ’    t2: u32 = (big_sigma0(a) + maj(a, b, c)) & 4294967295;
   954вҶ’   954вҶ’    h = g;
   955вҶ’   955вҶ’    g = f;
   956вҶ’   956вҶ’    f = e;
   957вҶ’   957вҶ’    e = (d + t1) & 4294967295;
   958вҶ’   958вҶ’    d = c;
   959вҶ’   959вҶ’    c = b;
   960вҶ’   960вҶ’    b = a;
   961вҶ’   961вҶ’    a = (t1 + t2) & 4294967295;
   962вҶ’   962вҶ’}
   963вҶ’   963вҶ’```

   967вҶ’   967вҶ’## 14. Pitfalls
   968вҶ’   968вҶ’
   969вҶ’   969вҶ’This section documents the most common mistakes when writing VUMA code. **Read this section carefully.**
   970вҶ’   970вҶ’
   971вҶ’   971вҶ’### Pitfall 1: u32 Arithmetic Without Masking
   972вҶ’   972вҶ’
   973вҶ’   973вҶ’**Problem**: VUMA uses 64-bit registers. When two u32 values are added, the 64-bit result can have bits set above bit 31 (the carry). Subsequent operations on this value (especially right shifts) will produce wrong results.
   974вҶ’   974вҶ’
   975вҶ’   975вҶ’```vuma
   976вҶ’   976вҶ’// WRONG
   977вҶ’   977вҶ’sum: u32 = a + b;        // upper 32 bits may have carry
   978вҶ’   978вҶ’rotated: u32 = sum >> 4;  // wrong! shift includes garbage bits
   979вҶ’   979вҶ’
   980вҶ’   980вҶ’// CORRECT
   981вҶ’   981вҶ’sum: u32 = (a + b) & 4294967295;  // clear upper 32 bits
   982вҶ’   982вҶ’rotated: u32 = sum >> 4;          // now correct
   983вҶ’   983вҶ’```
   984вҶ’   984вҶ’
   985вҶ’   985вҶ’**Rule**: After every arithmetic operation (`+`, `-`, `*`) on u32 values, apply `& 4294967295`.
   986вҶ’   986вҶ’
   987вҶ’   987вҶ’### Pitfall 2: Using `~` for Bitwise NOT
   988вҶ’   988вҶ’
   989вҶ’   989вҶ’**Problem**: The `~` operator inverts all 64 bits. For a u32 value in a 64-bit register, this means bits 32-63 are also inverted, producing `0xFFFFFFFF????????` instead of `0x00000000????????`.
   990вҶ’   990вҶ’
   991вҶ’   991вҶ’```vuma
   992вҶ’   992вҶ’// WRONG
   993вҶ’   993вҶ’not_x = ~x;  // produces 0xFFFFFFFF???????? for u32 x
   994вҶ’   994вҶ’
   995вҶ’   995вҶ’// CORRECT
   996вҶ’   996вҶ’not_x: u32 = x ^ 4294967295;  // flips only lower 32 bits
   997вҶ’   997вҶ’```
   998вҶ’   998вҶ’
   999вҶ’   999вҶ’**Rule**: Never use `~` for u32 bitwise NOT. Always use `x ^ 4294967295`.
  1000вҶ’  1000вҶ’
  1001вҶ’  1001вҶ’### Pitfall 3: Left Shift Without Masking
  1002вҶ’  1002вҶ’
  1003вҶ’  1003вҶ’**Problem**: Left-shifting a u32 value by N bits in a 64-bit register moves bits into the upper 32 bits.
  1004вҶ’  1004вҶ’
  1005вҶ’  1005вҶ’```vuma
  1006вҶ’  1006вҶ’// WRONG
  1007вҶ’  1007вҶ’shifted: u32 = value << 24;  // bits above bit 31 are set
  1008вҶ’  1008вҶ’
  1009вҶ’  1009вҶ’// CORRECT
  1010вҶ’  1010вҶ’shifted: u32 = (value << 24) & 4294967295;  // clear upper 32 bits
  1011вҶ’  1011вҶ’```
  1012вҶ’  1012вҶ’
  1013вҶ’  1013вҶ’**Rule**: After any left shift of a u32 value, apply `& 4294967295`.
  1014вҶ’  1014вҶ’
  1015вҶ’  1015вҶ’### Pitfall 4: Rotate Without Masking
  1016вҶ’  1016вҶ’
  1017вҶ’  1017вҶ’**Problem**: A rotate-right composed from shifts produces bits above bit 31 from the left-shift component.
  1018вҶ’  1018вҶ’
  1019вҶ’  1019вҶ’```vuma
  1020вҶ’  1020вҶ’// WRONG
  1021вҶ’  1021вҶ’rotated: u32 = (x >> n) | (x << (32 - n));  // upper bits polluted
  1022вҶ’  1022вҶ’
  1023вҶ’  1023вҶ’// CORRECT
  1024вҶ’  1024вҶ’rotated: u32 = ((x >> n) | (x << (32 - n))) & 4294967295;
  1025вҶ’  1025вҶ’```
  1026вҶ’  1026вҶ’
  1027вҶ’  1027вҶ’**Rule**: Always mask the result of a rotate operation.
  1028вҶ’  1028вҶ’
  1029вҶ’  1029вҶ’### Pitfall 5: Forgetting to Free Memory
  1030вҶ’  1030вҶ’
  1031вҶ’  1031вҶ’**Problem**: Every `allocate()` must be matched with a `free()`. The IVE checks this, but it is easy to forget in complex control flow.
  1032вҶ’  1032вҶ’
  1033вҶ’  1033вҶ’```vuma
  1034вҶ’  1034вҶ’// WRONG вҖ” memory leak
  1035вҶ’  1035вҶ’fn bad() -> i32 {
  1036вҶ’  1036вҶ’    buf = allocate(64);
  1037вҶ’  1037вҶ’    return 0;  // buf is never freed!
  1038вҶ’  1038вҶ’}
  1039вҶ’  1039вҶ’
  1040вҶ’  1040вҶ’// CORRECT
  1041вҶ’  1041вҶ’fn good() -> i32 {
  1042вҶ’  1042вҶ’    buf = allocate(64);
  1043вҶ’  1043вҶ’    // ... use buf ...
  1044вҶ’  1044вҶ’    free(buf);
  1045вҶ’  1045вҶ’    return 0;
  1046вҶ’  1046вҶ’}
  1047вҶ’  1047вҶ’```
  1048вҶ’  1048вҶ’
  1049вҶ’  1049вҶ’**Rule**: Always pair `allocate()` with `free()`. Free in reverse order of allocation to match the typical stack discipline.
  1050вҶ’  1050вҶ’
  1051вҶ’  1051вҶ’### Pitfall 6: Using `!x` for Bitwise NOT
  1052вҶ’  1052вҶ’
  1053вҶ’  1053вҶ’**Problem**: The `!` operator is logical NOT (converts non-zero to 0 and 0 to 1), not bitwise NOT.
  1054вҶ’  1054вҶ’
  1055вҶ’  1055вҶ’```vuma
  1056вҶ’  1056вҶ’// WRONG вҖ” logical NOT, not bitwise NOT
  1057вҶ’  1057вҶ’not_x = !x;  // produces 0 or 1, not bitwise complement
  1058вҶ’  1058вҶ’
  1059вҶ’  1059вҶ’// CORRECT вҖ” bitwise NOT via XOR
  1060вҶ’  1060вҶ’not_x: u32 = x ^ 4294967295;
  1061вҶ’  1061вҶ’```
  1062вҶ’  1062вҶ’
  1063вҶ’  1063вҶ’**Rule**: Use `x ^ 4294967295` for bitwise NOT, never `!x` or `~x`.
  1064вҶ’  1064вҶ’
  1065вҶ’  1065вҶ’### Pitfall 7: Hex Literals as Byte Values in Stores
  1066вҶ’  1066вҶ’
  1067вҶ’  1067вҶ’**Problem**: When storing a value to memory via `*(ptr + offset) = value`, only the low 8 bits are stored. Make sure you are not accidentally storing a truncated value.
  1068вҶ’  1068вҶ’
  1069вҶ’  1069вҶ’```vuma
  1070вҶ’  1070вҶ’// This stores only the low 8 bits (0xC2)
  1071вҶ’  1071вҶ’*(buf + 0) = 0x4FC2;  // stores 0xC2, NOT 0x4FC2
  1072вҶ’  1072вҶ’
  1073вҶ’  1073вҶ’// To store a specific byte, mask explicitly:
  1074вҶ’  1074вҶ’*(buf + 0) = (0x4FC2 >> 8) & 255;  // stores 0x4F
  1075вҶ’  1075вҶ’*(buf + 1) = 0x4FC2 & 255;        // stores 0xC2
  1076вҶ’  1076вҶ’```
  1077вҶ’  1077вҶ’
  1078вҶ’  1078вҶ’### Pitfall 8: Exit Code Truncation
  1079вҶ’  1079вҶ’
  1080вҶ’  1080вҶ’**Problem**: The process exit code is typically 8 bits (0-255) on Linux. When `main()` returns a u32 or i32, only the low byte is used as the exit code.
  1081вҶ’  1081вҶ’
  1082вҶ’  1082вҶ’```vuma
  1083вҶ’  1083вҶ’fn main() -> i32 {
  1084вҶ’  1084вҶ’    // To exit with code 79 (0x4F):
  1085вҶ’  1085вҶ’    return 79;  // exit code = 79
  1086вҶ’  1086вҶ’}
  1087вҶ’  1087вҶ’```
  1088вҶ’  1088вҶ’
  1089вҶ’  1089вҶ’If you need to verify a specific byte from a u32 computation, mask it and return:
  1090вҶ’  1090вҶ’
  1091вҶ’  1091вҶ’```vuma
  1092вҶ’  1092вҶ’fn main() -> i32 {
  1093вҶ’  1093вҶ’    result: u32 = compute_something();
  1094вҶ’  1094вҶ’    return result & 255;  // return low byte as exit code
  1095вҶ’  1095вҶ’}
  1096вҶ’  1096вҶ’```
  1097вҶ’  1097вҶ’
  1098вҶ’  1098вҶ’### Pitfall 9: Reading Bytes as u32 Without Masking
  1099вҶ’  1099вҶ’
  1100вҶ’  1100вҶ’When you read a byte from memory into a `u32` variable, the value is zero-extended by the load. However, if you then shift and combine bytes, the result must be masked:
  1101вҶ’  1101вҶ’
  1102вҶ’  1102вҶ’```vuma
  1103вҶ’  1103вҶ’// Each byte read is zero-extended to u32, but after shifting
  1104вҶ’  1104вҶ’// and combining, upper bits may be set
  1105вҶ’  1105вҶ’b0: u32 = *(buf + 0);  // e.g., 0x61
  1106вҶ’  1106вҶ’combined: u32 = (b0 << 24);  // 0x61000000 in 64-bit register, but
  1107вҶ’  1107вҶ’                               // might have upper bits
  1108вҶ’  1108вҶ’combined: u32 = (b0 << 24) & 4294967295;  // CORRECT
  1109вҶ’  1109вҶ’```
  1110вҶ’  1110вҶ’
  1111вҶ’  1111вҶ’### Pitfall 10: Range Bounds in For Loops
  1112вҶ’  1112вҶ’
  1113вҶ’  1113вҶ’The range `a..b` is exclusive on the upper bound. `0..64` means 0 through 63, not 0 through 64:
  1114вҶ’  1114вҶ’
  1115вҶ’  1115вҶ’```vuma
  1116вҶ’  1116вҶ’for i in 0..64 {
  1117вҶ’  1117вҶ’    // i goes from 0 to 63 inclusive (64 iterations)
  1118вҶ’  1118вҶ’}

  1173вҶ’  1173вҶ’### Target-Specific Considerations
  1174вҶ’  1174вҶ’
  1175вҶ’  1175вҶ’**64-bit targets (x86_64, AArch64, RISC-V 64, MIPS64, PPC64, LoongArch64)**:
  1176вҶ’  1176вҶ’- All use 64-bit general-purpose registers
  1177вҶ’  1177вҶ’- u32 values stored in 64-bit registers **must** be masked with `& 4294967295` after arithmetic and left shifts
  1178вҶ’  1178вҶ’- `Address` is 8 bytes
  1179вҶ’  1179вҶ’- Pointer arithmetic uses 64-bit addition
  1180вҶ’  1180вҶ’
  1181вҶ’  1181вҶ’**32-bit targets (ARM32, Wasm32)**:
  1182вҶ’  1182вҶ’- Use 32-bit registers
  1183вҶ’  1183вҶ’- u32 masking is technically unnecessary but recommended for portability
  1184вҶ’  1184вҶ’- `Address` is 4 bytes
  1185вҶ’  1185вҶ’- Pointer arithmetic uses 32-bit addition
  1186вҶ’  1186вҶ’
  1187вҶ’  1187вҶ’**Wasm32 specifics**:
  1188вҶ’  1188вҶ’- No native ROR/ROL instructions; rotates are synthesized from shift+or sequences
  1189вҶ’  1189вҶ’- Type inference can be tricky; explicit type annotations are important
  1190вҶ’  1190вҶ’- Stack machine architecture (no registers); the backend manages a virtual value stack
  1191вҶ’  1191вҶ’
  1192вҶ’  1192вҶ’### Portability Rules
  1193вҶ’  1193вҶ’
  1194вҶ’  1194вҶ’When writing VUMA code, follow these rules to ensure it works on all 8 targets:
  1195вҶ’  1195вҶ’
  1196вҶ’  1196вҶ’1. **Always mask u32 results** with `& 4294967295` after addition, subtraction, multiplication, and left shift.
  1197вҶ’  1197вҶ’2. **Use `x ^ 4294967295`** for bitwise NOT, never `~x`.
  1198вҶ’  1198вҶ’3. **Compose rotates from shifts** with a final mask: `((x >> n) | (x << (32 - n))) & 4294967295`.
  1199вҶ’  1199вҶ’4. **Use explicit type annotations** for all variable declarations.
  1200вҶ’  1200вҶ’5. **Use `u64` for offsets and lengths**, not `u32`, to avoid truncation on large buffers.
  1201вҶ’  1201вҶ’6. **Always pair `allocate()` with `free()`**.
