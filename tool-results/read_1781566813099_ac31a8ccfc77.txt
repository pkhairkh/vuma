    27‚Üí## 1. Quick Start
    28‚Üí
    29‚ÜíA VUMA program consists of one or more function definitions. The entry point is always `fn main()`. The compiler produces a native binary (or Wasm module) that can be executed directly or via QEMU for cross-architecture testing.
    30‚Üí
    31‚Üí### Minimal Program
    32‚Üí
    33‚Üí```vuma
    34‚Üífn main() -> i32 {
    35‚Üí    return 0;
    36‚Üí}
    37‚Üí```
    38‚Üí
    39‚ÜíThis program exits with code 0.
    40‚Üí
    41‚Üí### Compile and Run
    42‚Üí
    43‚ÜíVUMA compiles `.vuma` source files to native binaries. The build system uses Cargo under the hood. To build and test:
    44‚Üí
    45‚Üí```bash
    46‚Üí# Build the VUMA compiler
    47‚Üícargo build --workspace
    48‚Üí
    49‚Üí# Run all tests
    50‚Üícargo test --workspace
    51‚Üí
    52‚Üí# Cross-architecture testing uses QEMU
    53‚Üí# x86_64:
    54‚Üíqemu-x86_64 ./output_binary
    55‚Üí# AArch64:
    56‚Üíqemu-aarch64 ./output_binary
    57‚Üí# RISC-V 64:
    58‚Üíqemu-riscv64-static ./output_binary
    59‚Üí# ARM32:
    60‚Üíqemu-arm ./output_binary
    61‚Üí# MIPS64:
    62‚Üíqemu-mips64-static ./output_binary
    63‚Üí# PPC64:
    64‚Üíqemu-ppc64 ./output_binary
    65‚Üí# LoongArch64:
    66‚Üíqemu-loongarch64 ./output_binary
    67‚Üí```
    68‚Üí
    69‚Üí### Program with Memory
    70‚Üí
    71‚Üí```vuma
    72‚Üífn main() -> i32 {
    73‚Üí    msg = allocate(3);
    74‚Üí    *(msg + 0) = 97;   // 'a'
    75‚Üí    *(msg + 1) = 98;   // 'b'
    76‚Üí    *(msg + 2) = 99;   // 'c'
    77‚Üí    free(msg);
    78‚Üí    return 0;
    79‚Üí}
    80‚Üí```
    81‚Üí
    82‚Üí**Key facts**:
    83‚Üí- `main()` returns `i32` (the process exit code) or `u64`.
    84‚Üí- Every `allocate()` must be paired with a `free()`.
    85‚Üí- Pointer dereference uses `*(ptr + offset)` syntax.
    86‚Üí- Bytes are stored as integer values (e.g., `97` for ASCII `'a'`).
    87‚Üí

    89‚Üí
    90‚Üí## 2. Types
    91‚Üí
    92‚ÜíVUMA supports the following primitive types:
    93‚Üí
    94‚Üí| Type | Size (bytes) | Range | Signed |
    95‚Üí|------|-------------|-------|--------|
    96‚Üí| `i8`  | 1 | -128 to 127 | Yes |
    97‚Üí| `i16` | 2 | -32768 to 32767 | Yes |
    98‚Üí| `i32` | 4 | -2147483648 to 2147483647 | Yes |
    99‚Üí| `i64` | 8 | -9223372036854775808 to 9223372036854775807 | Yes |
   100‚Üí| `u8`  | 1 | 0 to 255 | No |
   101‚Üí| `u16` | 2 | 0 to 65535 | No |
   102‚Üí| `u32` | 4 | 0 to 4294967295 | No |
   103‚Üí| `u64` | 8 | 0 to 18446744073709551615 | No |
   104‚Üí| `Address` | 8 (4 on 32-bit targets) | N/A (opaque pointer) | No |
   105‚Üí| `bool` | 1 | `true` or `false` | N/A |
   106‚Üí| `void` | 0 | N/A | N/A |
   107‚Üí
   108‚Üí### Type Annotation Syntax
   109‚Üí
   110‚Üí```vuma
   111‚Üíx: i32 = 42;
   112‚Üíy: u32 = 0xFFFFFFFF;
   113‚Üíptr: Address = allocate(64);
   114‚Üíbig: u64 = 18446744073709551615;
   115‚Üíflag: bool = true;
   116‚Üí```
   117‚Üí
   118‚Üí### Critical Constraint: u32 on 64-bit Registers
   119‚Üí
   120‚Üí**This is the single most important thing to know about VUMA types.** All 64-bit target platforms (x86_64, AArch64, RISC-V 64, MIPS64, PPC64, LoongArch64) use 64-bit general-purpose registers. A `u32` value stored in a 64-bit register can have garbage bits in the upper 32 bits after arithmetic operations. You **must** manually mask u32 results with `& 4294967295` (which is `0xFFFFFFFF` in decimal) after any operation that could set bits above bit 31:
   121‚Üí
   122‚Üí```vuma
   123‚Üí// WRONG: bits above bit 31 may be set
   124‚Üísum: u32 = a + b;
   125‚Üí
   126‚Üí// CORRECT: mask to 32 bits after addition
   127‚Üísum: u32 = (a + b) & 4294967295;
   128‚Üí```
   129‚Üí
   130‚ÜíSee [Section 14: Pitfalls](#14-pitfalls) for the complete list of operations that require masking.
   131‚Üí
   132‚Üí### Address Type
   133‚Üí
   134‚Üí`Address` is the type for pointers and memory regions. It is 8 bytes on 64-bit targets and 4 bytes on 32-bit targets (ARM32, Wasm32). The `allocate()` built-in returns an `Address`.
   135‚Üí
   136‚Üí```vuma
   137‚Üíbuf: Address = allocate(256);
   138‚Üí```
   139‚Üí

   142‚Üí## 3. Functions
   143‚Üí
   144‚Üí### Function Declaration
   145‚Üí
   146‚Üí```vuma
   147‚Üífn function_name(param1: type1, param2: type2) -> return_type {
   148‚Üí    // body
   149‚Üí    return value;
   150‚Üí}
   151‚Üí```
   152‚Üí
   153‚Üí- The `fn` keyword introduces a function.
   154‚Üí- Parameters have mandatory type annotations when declared in the signature.
   155‚Üí- The return type follows `->`. If omitted, the function returns `void`.
   156‚Üí- The body is a block enclosed in `{ }`.
   157‚Üí- Use `return expr;` to return a value. The last expression in a block without `return` is also the return value in some contexts, but `return` is preferred for clarity.
   158‚Üí
   159‚Üí### Examples
   160‚Üí
   161‚Üí```vuma
   162‚Üí// Function returning a value
   163‚Üífn add(a: i32, b: i32) -> i32 {
   164‚Üí    return a + b;
   165‚Üí}
   166‚Üí
   167‚Üí// Function with no return value (void)
   168‚Üífn store_byte(buf: Address, offset: u64, val: u32) {
   169‚Üí    *(buf + offset) = val & 255;
   170‚Üí}
   171‚Üí
   172‚Üí// Function with Address parameter
   173‚Üífn read_u32_be(buf: Address, offset: u64) -> u32 {
   174‚Üí    b0: u32 = *(buf + offset);
   175‚Üí    b1: u32 = *(buf + offset + 1);
   176‚Üí    b2: u32 = *(buf + offset + 2);
   177‚Üí    b3: u32 = *(buf + offset + 3);
   178‚Üí    return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
   179‚Üí}
   180‚Üí
   181‚Üí// Main entry point ‚Äî must return i32 or u64
   182‚Üífn main() -> i32 {
   183‚Üí    return 0;
   184‚Üí}
   185‚Üí```
   186‚Üí
   187‚Üí### Function Call
   188‚Üí
   189‚Üí```vuma
   190‚Üíresult: i32 = add(10, 20);
   191‚Üístore_byte(buffer, 0, 65);
   192‚Üí```
   193‚Üí
   194‚ÜíFunctions are called by name with parenthesized arguments. Function calls can appear as expressions (when the function returns a value) or as statements (when the function returns void or the return value is discarded).
   195‚Üí
   196‚Üí---
   197‚Üí
   198‚Üí## 4. Variables
   199‚Üí
   200‚Üí### Declaration with Type Annotation
   201‚Üí
   202‚Üí```vuma
   203‚Üíname: type = expression;
   204‚Üí```
   205‚Üí
   206‚ÜíVariables are declared with a name, a type annotation, and an initial value. The type annotation is required in most cases.
   207‚Üí
   208‚Üí```vuma
   209‚Üíx: i32 = 42;
   210‚Üíy: u32 = 0xFFFFFFFF;
   211‚Üíptr: Address = allocate(64);
   212‚Üísum: u32 = (a + b) & 4294967295;
   213‚Üí```
   214‚Üí
   215‚Üí### Declaration Without Type Annotation (Inferred)
   216‚Üí
   217‚ÜíVUMA can infer the type from the right-hand side in some cases, but explicit annotation is strongly recommended to avoid ambiguity:
   218‚Üí
   219‚Üí```vuma
   220‚Üíresult = add1(41);       // type inferred from function return type
   221‚Üímsg = allocate(3);        // type inferred as Address
   222‚Üí```
   223‚Üí
   224‚Üí**When writing VUMA code as an LLM, always use explicit type annotations** except for variables assigned from `allocate()` (which always returns `Address`).
   225‚Üí
   226‚Üí### Assignment
   227‚Üí
   228‚ÜíOnce declared, variables can be reassigned without repeating the type:
   229‚Üí
   230‚Üí```vuma
   231‚Üíx: u32 = 10;
   232‚Üíx = 20;              // reassignment
   233‚Üíx = (x + 1) & 4294967295;  // reassignment with expression
   234‚Üí```
   235‚Üí
   236‚Üí### Compound Assignment
   237‚Üí
   238‚ÜíVUMA supports compound assignment operators:
   239‚Üí
   240‚Üí```vuma
   241‚Üíx += 1;     // x = x + 1
   242‚Üíx -= 1;     // x = x - 1
   243‚Üíx *= 2;     // x = x * 2
   244‚Üíx &= 255;   // x = x & 255
   245‚Üíx |= mask;  // x = x | mask
   246‚Üíx ^= key;   // x = x ^ key
   247‚Üíx <<= 4;    // x = x << 4
   248‚Üíx >>= 4;    // x = x >> 4
   249‚Üí```
   250‚Üí
   251‚Üí**Note**: Compound assignments on `u32` values still need the `& 4294967295` mask. The compound form does **not** automatically mask. Prefer explicit assignment with masking:
   252‚Üí
   253‚Üí```vuma
   254‚Üí// WRONG for u32 on 64-bit registers:
   255‚Üíx += 1;
   256‚Üí
   257‚Üí// CORRECT:
   258‚Üíx = (x + 1) & 4294967295;
   259‚Üí```
   260‚Üí

   263‚Üí## 5. Control Flow
   264‚Üí
   265‚Üí### For Loop (Range)
   266‚Üí
   267‚ÜíThe `for` loop iterates over an exclusive range `start..end`. The loop variable takes each integer value from `start` (inclusive) to `end` (exclusive).
   268‚Üí
   269‚Üí```vuma
   270‚Üífor i in 0..10 {
   271‚Üí    // i = 0, 1, 2, ..., 9
   272‚Üí}
   273‚Üí
   274‚Üífor i in 0..64 {
   275‚Üí    // i = 0, 1, 2, ..., 63
   276‚Üí}
   277‚Üí
   278‚Üífor i in 16..64 {
   279‚Üí    // i = 16, 17, ..., 63
   280‚Üí}
   281‚Üí```
   282‚Üí
   283‚Üí**The range is always exclusive on the upper bound**: `0..10` means `0 <= i < 10`.
   284‚Üí
   285‚Üí### For Loop with Memory Access
   286‚Üí
   287‚Üí```vuma
   288‚Üí// Copy 32 bytes from src to dst
   289‚Üífn copy32(dst: Address, src: Address) {
   290‚Üí    i: u64 = 0;
   291‚Üí    for i in 0..32 {
   292‚Üí        *(dst + i) = *(src + i);
   293‚Üí    }
   294‚Üí}
   295‚Üí```
   296‚Üí
   297‚Üí### While Loop
   298‚Üí
   299‚Üí```vuma
   300‚Üícurrent = (*list).next;
   301‚Üíwhile current != list {
   302‚Üí    process(current);
   303‚Üí    current = (*current).next;
   304‚Üí}
   305‚Üí```
   306‚Üí
   307‚Üí### If / Else
   308‚Üí
   309‚Üí```vuma
   310‚Üíif x > 100 {
   311‚Üí    y = 2;
   312‚Üí} else {
   313‚Üí    y = 1;
   314‚Üí}
   315‚Üí```
   316‚Üí
   317‚Üí```vuma
   318‚Üíif value & 1 == 1 {
   319‚Üí    // odd
   320‚Üí}
   321‚Üí```
   322‚Üí
   323‚ÜíNested conditionals:
   324‚Üí
   325‚Üí```vuma
   326‚Üíif x > 0 {
   327‚Üí    if x > 100 {
   328‚Üí        result = 2;
   329‚Üí    } else {
   330‚Üí        result = 1;
   331‚Üí    }
   332‚Üí} else {
   333‚Üí    result = 0;
   334‚Üí}
   335‚Üí```
   336‚Üí
   337‚Üí### Infinite Loop
   338‚Üí
   339‚Üí```vuma
   340‚Üíloop {
   341‚Üí    // runs forever (useful in embedded/bare-metal contexts)
   342‚Üí}
   343‚Üí```
   344‚Üí
   345‚Üí### Break and Continue
   346‚Üí
   347‚Üí```vuma
   348‚Üífor i in 0..100 {
   349‚Üí    if i == 50 {
   350‚Üí        break;
   351‚Üí    }
   352‚Üí    if i % 2 == 0 {
   353‚Üí        continue;
   354‚Üí    }
   355‚Üí    process(i);
   356‚Üí}
   357‚Üí```

   361‚Üí## 6. Memory
   362‚Üí
   363‚ÜíVUMA provides two first-class memory operations: `allocate` and `free`. There is no garbage collector; memory management is manual.
   364‚Üí
   365‚Üí### allocate(n)
   366‚Üí
   367‚ÜíReserves `n` bytes of memory and returns an `Address` (pointer to the start of the block). The allocated memory is uninitialized.
   368‚Üí
   369‚Üí```vuma
   370‚Üíbuf = allocate(64);     // 64-byte buffer
   371‚Üístate = allocate(32);   // 32-byte state buffer
   372‚Üík_table = allocate(256); // 256-byte constant table
   373‚Üí```
   374‚Üí
   375‚Üí**The returned `Address` does not need a type annotation** ‚Äî it is always inferred as `Address`:
   376‚Üí
   377‚Üí```vuma
   378‚Üístate = allocate(32);   // state has type Address
   379‚Üí```
   380‚Üí
   381‚Üí### free(ptr)
   382‚Üí
   383‚ÜíReleases the memory block pointed to by `ptr`. After `free(ptr)`, the pointer is invalid and must not be used.
   384‚Üí
   385‚Üí```vuma
   386‚Üífree(state);
   387‚Üífree(k_table);
   388‚Üífree(buf);
   389‚Üí```
   390‚Üí
   391‚Üí**Every `allocate` must be matched with a `free`**. Failing to free memory is a resource leak. The IVE (Invariant Verification Engine) checks this at compile time.
   392‚Üí
   393‚Üí### Memory Layout Example
   394‚Üí
   395‚Üí```vuma
   396‚Üífn sha256d(msg: Address, msg_len: u64, out: Address) {
   397‚Üí    // Allocate working memory
   398‚Üí    state = allocate(32);
   399‚Üí    k = allocate(256);
   400‚Üí    w = allocate(256);
   401‚Üí    block = allocate(64);
   402‚Üí    inner = allocate(32);
   403‚Üí
   404‚Üí    // ... use the memory ...
   405‚Üí
   406‚Üí    // Clean up all allocations
   407‚Üí    free(state);
   408‚Üí    free(k);
   409‚Üí    free(w);
   410‚Üí    free(block);
   411‚Üí    free(inner);
   412‚Üí}
   413‚Üí```
   414‚Üí
   415‚Üí### Region Declarations
   416‚Üí
   417‚ÜíNamed regions can be declared with the `region` keyword:
   418‚Üí
   419‚Üí```vuma
   420‚Üíregion pool = allocate(4096);
   421‚Üí```

   424‚Üí
   425‚Üí## 7. Pointer Operations
   426‚Üí
   427‚Üí### Dereference: Load
   428‚Üí
   429‚ÜíRead a byte from memory at `ptr + offset`:
   430‚Üí
   431‚Üí```vuma
   432‚Üíb0: u32 = *(buf + offset);
   433‚Üí```
   434‚Üí
   435‚ÜíThe result is a single byte zero-extended to the type of the variable (typically `u32`).
   436‚Üí
   437‚Üí### Dereference: Store
   438‚Üí
   439‚ÜíWrite a byte to memory at `ptr + offset`:
   440‚Üí
   441‚Üí```vuma
   442‚Üí*(buf + offset) = value & 255;
   443‚Üí```
   444‚Üí
   445‚ÜíOnly the low 8 bits of `value` are stored.
   446‚Üí
   447‚Üí### Pointer Arithmetic
   448‚Üí
   449‚ÜíPointer + integer produces a new `Address` offset by that many bytes:
   450‚Üí
   451‚Üí```vuma
   452‚Üíptr = base + 128;      // 128 bytes forward
   453‚Üíbyte3 = *(buf + 3);    // read byte at offset 3
   454‚Üí*(buf + 4) = 99;       // write byte at offset 4
   455‚Üí```
   456‚Üí
   457‚ÜíThe offset can be an expression:
   458‚Üí
   459‚Üí```vuma
   460‚Üí*(buf + i * 4) = val;  // write at byte offset i*4
   461‚Üíbyte_val = *(buf + i); // read at byte offset i
   462‚Üí```
   463‚Üí
   464‚Üí### Read/Write Multi-byte Values (u32)
   465‚Üí
   466‚ÜíVUMA does not have native multi-byte load/store. You must compose byte-level accesses. For big-endian u32:
   467‚Üí
   468‚Üí```vuma
   469‚Üífn write_u32_be(buf: Address, offset: u64, val: u32) {
   470‚Üí    *(buf + offset) = (val >> 24) & 255;
   471‚Üí    *(buf + offset + 1) = (val >> 16) & 255;
   472‚Üí    *(buf + offset + 2) = (val >> 8) & 255;
   473‚Üí    *(buf + offset + 3) = val & 255;
   474‚Üí}
   475‚Üí
   476‚Üífn read_u32_be(buf: Address, offset: u64) -> u32 {
   477‚Üí    b0: u32 = *(buf + offset);
   478‚Üí    b1: u32 = *(buf + offset + 1);
   479‚Üí    b2: u32 = *(buf + offset + 2);
   480‚Üí    b3: u32 = *(buf + offset + 3);
   481‚Üí    return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
   482‚Üí}
   483‚Üí```
   484‚Üí
   485‚Üí**Why the `& 4294967295` mask?** After left-shifting `b0` by 24 bits in a 64-bit register, bits above bit 31 may be set. The mask clears them. This is the most common source of bugs in VUMA code. See [Section 14: Pitfalls](#14-pitfalls).
   486‚Üí
   487‚Üí### Address-Of Operator
   488‚Üí
   489‚ÜíThe `@` operator takes the address of a variable:
   490‚Üí
   491‚Üí```vuma
   492‚Üílet x: i32 = 42;
   493‚Üíptr = @x;       // ptr has type Address, points to x
   494‚Üí*ptr = 100;     // modifies x through the pointer
   495‚Üí```
   496‚Üí
   497‚Üí### Derive
   498‚Üí
   499‚ÜíThe `derive(ptr, region)` expression explicitly creates a derived pointer within a specific region. This is the formal mechanism for sub-allocation:
   500‚Üí
   501‚Üí```vuma
   502‚Üíinner = derive(ptr, arena_region);
   503‚Üí```
   504‚Üí
   505‚ÜíIn practice, pointer arithmetic (`base + offset`) implicitly derives a new pointer.
   506‚Üí
   507‚Üí---
   508‚Üí
   509‚Üí## 8. Bitwise Operations
   510‚Üí
   511‚ÜíVUMA supports the standard bitwise operations:
   512‚Üí
   513‚Üí| Operator | Name | Example |
   514‚Üí|----------|------|---------|
   515‚Üí| `&` | Bitwise AND | `a & 0xFF` |
   516‚Üí| `\|` | Bitwise OR | `a \| b` |
   517‚Üí| `^` | Bitwise XOR | `a ^ 0xFFFFFFFF` |
   518‚Üí| `<<` | Left shift | `a << 8` |
   519‚Üí| `>>` | Right shift | `a >> 4` |
   520‚Üí
   521‚Üí### No Built-in NOT Operator
   522‚Üí
   523‚Üí**VUMA does not have a reliable bitwise NOT (`~`) operator for u32 values.** The `~x` operator inverts all 64 bits of the register, which corrupts the upper 32 bits. Instead, use XOR with `0xFFFFFFFF`:
   524‚Üí
   525‚Üí```vuma
   526‚Üí// WRONG: ~x inverts bits 32-63 which corrupts u32 values
   527‚Üínot_x = ~x;
   528‚Üí
   529‚Üí// CORRECT: XOR with 0xFFFFFFFF inverts only the lower 32 bits
   530‚Üínot_x: u32 = x ^ 4294967295;
   531‚Üí```
   532‚Üí
   533‚Üí**Always use `x ^ 4294967295` instead of `~x` for u32 bitwise NOT.**
   534‚Üí
   535‚Üí### Bitwise AND for Masking
   536‚Üí
   537‚Üí```vuma
   538‚Üílow_byte: u32 = value & 255;          // mask to 8 bits
   539‚Üílow_16: u32 = value & 65535;          // mask to 16 bits
   540‚Üílow_32: u32 = value & 4294967295;     // mask to 32 bits
   541‚Üí```
   542‚Üí
   543‚Üí### Bitwise OR for Combining
   544‚Üí
   545‚Üí```vuma
   546‚Üícombined: u32 = (high << 8) | low;
   547‚Üí```
   548‚Üí
   549‚Üí### Left Shift
   550‚Üí
   551‚Üí```vuma
   552‚Üíshifted: u32 = (value << 24) & 4294967295;  // MUST mask after left shift
   553‚Üí```
   554‚Üí
   555‚Üí**Left shift on u32 values always requires masking** because the result can have bits set above bit 31.
   556‚Üí
   557‚Üí### Right Shift
   558‚Üí
   559‚Üí```vuma
   560‚Üíupper: u32 = value >> 24;   // no mask needed for right shift
   561‚Üí```
   562‚Üí
   563‚ÜíRight shift does not require masking because it only clears bits (never sets new bits above the existing ones).
   564‚Üí
   565‚Üí### Rotate Right (Compose from Shifts)
   566‚Üí
   567‚ÜíVUMA does not have a built-in rotate operator. Compose it from shifts:
   568‚Üí
   569‚Üí```vuma
   570‚Üífn rotr32(x: u32, n: u32) -> u32 {
   571‚Üí    return ((x >> n) | (x << (32 - n))) & 4294967295;
   572‚Üí}
   573‚Üí```
   574‚Üí
   575‚ÜíThe mask is required because `x << (32 - n)` sets bits above bit 31.
   576‚Üí
   577‚Üí---
   578‚Üí
   579‚Üí## 9. Arithmetic
   580‚Üí
   581‚Üí### Basic Operators
   582‚Üí
   583‚Üí| Operator | Name | Example |
   584‚Üí|----------|------|---------|
   585‚Üí| `+` | Addition | `a + b` |
   586‚Üí| `-` | Subtraction | `a - b` |
   587‚Üí| `*` | Multiplication | `a * b` |
   588‚Üí| `/` | Division | `a / b` |
   589‚Üí| `%` | Modulo | `a % b` |
   590‚Üí
   591‚Üí### u32 Masking Rule
   592‚Üí
   593‚Üí**All u32 arithmetic results must be masked with `& 4294967295`.** VUMA uses 64-bit registers on 64-bit targets. Arithmetic on u32 values can produce carries into the upper 32 bits:
   594‚Üí
   595‚Üí```vuma
   596‚Üí// WRONG: overflow bits pollute upper 32 bits
   597‚Üísum: u32 = a + b;
   598‚Üí
   599‚Üí// CORRECT: mask to 32 bits
   600‚Üísum: u32 = (a + b) & 4294967295;
   601‚Üí```
   602‚Üí
   603‚ÜíThis applies to **all** arithmetic operations on u32 values:
   604‚Üí
   605‚Üí```vuma
   606‚Üísum: u32 = (a + b) & 4294967295;
   607‚Üídiff: u32 = (a - b) & 4294967295;
   608‚Üíprod: u32 = (a * b) & 4294967295;
   609‚Üíquot: u32 = (a / b) & 4294967295;
   610‚Üírem: u32 = (a % b) & 4294967295;
   611‚Üí```
   612‚Üí
   613‚ÜíDivision and modulo technically cannot overflow for u32, but masking is still recommended for consistency and safety.
   614‚Üí
   615‚Üí### u64 Arithmetic
   616‚Üí
   617‚Üíu64 arithmetic does **not** require masking:
   618‚Üí
   619‚Üí```vuma
   620‚Üíbig_sum: u64 = a + b;    // OK, no mask needed
   621‚Üí```
   622‚Üí
   623‚Üí### i32 Arithmetic
   624‚Üí
   625‚ÜíSigned 32-bit arithmetic should also be masked for correctness:
   626‚Üí
   627‚Üí```vuma
   628‚Üísigned_result: i32 = (a + b) & 4294967295;
   629‚Üí```
   630‚Üí
   631‚Üí### Chained Arithmetic
   632‚Üí
   633‚ÜíWhen multiple u32 operations are chained, mask the **final** result:
   634‚Üí
   635‚Üí```vuma
   636‚Üí// SHA-256 T1 computation
   637‚Üít1: u32 = (h + big_sigma1(e) + ch(e, f, g) + ki + wi) & 4294967295;
   638‚Üí```
   639‚Üí
   640‚ÜíIntermediate results in sub-expressions (like `big_sigma1(e)`) should also be masked in their function definitions. Every function that returns u32 must mask its return value.
   641‚Üí

   645‚Üí
   646‚Üí### Comparison Operators
   647‚Üí
   648‚Üí| Operator | Name | Example |
   649‚Üí|----------|------|---------|
   650‚Üí| `==` | Equal | `a == b` |
   651‚Üí| `!=` | Not equal | `a != b` |
   652‚Üí| `<` | Less than | `a < b` |
   653‚Üí| `<=` | Less than or equal | `a <= b` |
   654‚Üí| `>` | Greater than | `a > b` |
   655‚Üí| `>=` | Greater than or equal | `a >= b` |
   656‚Üí
   657‚Üí### Usage
   658‚Üí
   659‚Üí```vuma
   660‚Üíif x == 0 {
   661‚Üí    // handle zero case
   662‚Üí}
   663‚Üí
   664‚Üíif offset < 256 {
   665‚Üí    // within bounds
   666‚Üí}
   667‚Üí

   674‚Üí

   676‚Üífor i in 0..64 {
   677‚Üí    if i < 16 {
   678‚Üí        // first 16 iterations
   679‚Üí    } else {
   680‚Üí        // remaining iterations
   681‚Üí    }
   682‚Üí}
   683‚Üí```
   684‚Üí
   685‚Üí### Boolean Results
   686‚Üí
   687‚ÜíComparison operators produce boolean values. However, in VUMA's compiled form, booleans are represented as integers (1 for true, 0 for false). This means they can be used in arithmetic:
   688‚Üí
   689‚Üí```vuma
   690‚Üíis_zero: u32 = (x == 0);  // 1 if x is 0, 0 otherwise
   691‚Üí```

   703‚Üí```
   704‚Üí
   705‚Üí### Hexadecimal Literals
   706‚Üí
   707‚ÜíUse the `0x` prefix:
   708‚Üí
   709‚Üí```vuma
   710‚Üímask: u32 = 0xFF;           // 255
   711‚Üífull: u32 = 0xFFFFFFFF;     // 4294967295

   717‚Üí
   718‚ÜíLiterals support underscore separators for readability:
   719‚Üí
   720‚Üí```vuma

   722‚Üíhex: u32 = 0xFF00_0000;

   726‚Üí
   727‚ÜíThe u32 mask value `4294967295` is `0xFFFFFFFF`. Both notations are equivalent:
   728‚Üí
   729‚Üí```vuma
   730‚Üíx = value & 4294967295;   // decimal
   731‚Üíx = value & 0xFFFFFFFF;   // hex (equivalent)
   732‚Üí```
   733‚Üí
   734‚Üí**In existing VUMA code, the decimal form `4294967295` is conventional for u32 masking.** When generating VUMA code, prefer the decimal form for consistency with the existing codebase.

   747‚Üí## 12. Calling Convention
   748‚Üí
   749‚Üí### Argument Passing
   750‚Üí
   751‚ÜíOn 64-bit targets, the first 8 integer/pointer arguments are passed in registers (X0-X7 on AArch64, RDI/RSI/RDX/RCX/R8/R9 on x86_64, a0-a7 on RISC-V). Additional arguments are passed on the stack.
   752‚Üí
   753‚ÜíOn 32-bit targets (ARM32, Wasm32), the first 4 arguments use registers (r0-r3 on ARM32, the value stack on Wasm32).
   754‚Üí
   755‚Üí### Return Values
   756‚Üí
   757‚ÜíThe return value is passed in the first return register (X0 on AArch64, RAX on x86_64, a0 on RISC-V). For void functions, no return register is used.
   758‚Üí
   759‚Üí### Address Parameters
   760‚Üí
   761‚Üí`Address` parameters are passed like integer values (pointer-sized). On 64-bit targets, they occupy 8 bytes; on 32-bit targets, 4 bytes.
   762‚Üí
   763‚Üí### Function Call Example
   764‚Üí
   765‚Üí```vuma
   766‚Üífn add1(x: i32) -> i32 {
   767‚Üí    return x + 1;
   768‚Üí}
   769‚Üí
   770‚Üífn main() -> i32 {
   771‚Üí    return add1(41);  // returns 42
   772‚Üí}
   773‚Üí```
   774‚Üí
   775‚Üí### Cross-Function u32 Values
   776‚Üí
   777‚ÜíWhen a function returns `u32`, the caller receives the value in a 64-bit register. The upper 32 bits may contain garbage. The **returning function** must mask the value before returning, and the **calling function** should not assume the upper bits are clear unless the callee guarantees it.
   778‚Üí
   779‚Üí```vuma
   780‚Üífn safe_add(a: u32, b: u32) -> u32 {
   781‚Üí    return (a + b) & 4294967295;  // callee masks the result
   782‚Üí}
   783‚Üí
   784‚Üífn main() -> i32 {
   785‚Üí    result: u32 = safe_add(0xFFFFFFFF, 1);  // result = 0
   786‚Üí    return 0;
   787‚Üí}
   788‚Üí```
   789‚Üí
   790‚Üí### Stack Layout (AArch64)
   791‚Üí
   792‚Üí```
   793‚ÜíHigher addresses
   794‚Üí  +----------------------+
   795‚Üí  | Incoming stack args   |  FP+16, FP+24, ...
   796‚Üí  +----------------------+
   797‚Üí  | Saved FP (X29)       |  FP+0
   798‚Üí  | Saved LR (X30)       |  FP+8
   799‚Üí  +----------------------+
   800‚Üí  | Callee-saved regs    |  FP-8, FP-16, ...
   801‚Üí  | (X19..X28)           |
   802‚Üí  +----------------------+
   803‚Üí  | Local variables      |
   804‚Üí  | (aligned)            |
   805‚Üí  +----------------------+
   806‚Üí  | Outgoing stack args  |
   807‚Üí  | (for nested calls)   |
   808‚Üí  +----------------------+  SP
   809‚ÜíLower addresses
   810‚Üí```

   814‚Üí## 13. Common Patterns
   815‚Üí
   816‚Üí### Pattern 1: SHA256d (Double SHA-256)
   817‚Üí
   818‚ÜíThis is the canonical VUMA program. It exercises all major language features: functions, memory, pointer arithmetic, bitwise operations, u32 masking, and loops.
   819‚Üí
   820‚Üí```vuma
   821‚Üífn rotr32(x: u32, n: u32) -> u32 {
   822‚Üí    return ((x >> n) | (x << (32 - n))) & 4294967295;
   823‚Üí}
   824‚Üí
   825‚Üífn ch(x: u32, y: u32, z: u32) -> u32 {
   826‚Üí    return (x & y) ^ ((x ^ 4294967295) & z);
   827‚Üí}
   828‚Üí
   829‚Üífn maj(a: u32, b: u32, c: u32) -> u32 {
   830‚Üí    return (a & b) ^ (a & c) ^ (b & c);
   831‚Üí}
   832‚Üí
   833‚Üífn big_sigma0(x: u32) -> u32 {
   834‚Üí    return rotr32(x, 2) ^ rotr32(x, 13) ^ rotr32(x, 22);
   835‚Üí}
   836‚Üí
   837‚Üífn big_sigma1(x: u32) -> u32 {
   838‚Üí    return rotr32(x, 6) ^ rotr32(x, 11) ^ rotr32(x, 25);
   839‚Üí}
   840‚Üí
   841‚Üífn small_sigma0(x: u32) -> u32 {
   842‚Üí    return (rotr32(x, 7) ^ rotr32(x, 18) ^ (x >> 3)) & 4294967295;
   843‚Üí}
   844‚Üí
   845‚Üífn small_sigma1(x: u32) -> u32 {
   846‚Üí    return (rotr32(x, 17) ^ rotr32(x, 19) ^ (x >> 10)) & 4294967295;
   847‚Üí}
   848‚Üí
   849‚Üífn write_u32_be(buf: Address, offset: u64, val: u32) {
   850‚Üí    *(buf + offset) = (val >> 24) & 255;
   851‚Üí    *(buf + offset + 1) = (val >> 16) & 255;
   852‚Üí    *(buf + offset + 2) = (val >> 8) & 255;
   853‚Üí    *(buf + offset + 3) = val & 255;
   854‚Üí}
   855‚Üí
   856‚Üífn read_u32_be(buf: Address, offset: u64) -> u32 {
   857‚Üí    b0: u32 = *(buf + offset);
   858‚Üí    b1: u32 = *(buf + offset + 1);
   859‚Üí    b2: u32 = *(buf + offset + 2);
   860‚Üí    b3: u32 = *(buf + offset + 3);
   861‚Üí    return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
   862‚Üí}
   863‚Üí
   864‚Üífn main() -> i32 {
   865‚Üí    msg = allocate(3);
   866‚Üí    *(msg + 0) = 97;
   867‚Üí    *(msg + 1) = 98;
   868‚Üí    *(msg + 2) = 99;
   869‚Üí    // ... SHA-256 compression ...
   870‚Üí    free(msg);
   871‚Üí    return 0;
   872‚Üí}
   873‚Üí```
   874‚Üí
   875‚Üí### Pattern 2: Memory Buffer (Read/Write Bytes)
   876‚Üí
   877‚Üí```vuma
   878‚Üífn main() -> i32 {
   879‚Üí    buf = allocate(16);
   880‚Üí
   881‚Üí    // Write bytes
   882‚Üí    *(buf + 0) = 0x4f;
   883‚Üí    *(buf + 1) = 0x8b;
   884‚Üí
   885‚Üí    // Read bytes back
   886‚Üí    b0: u32 = *(buf + 0);
   887‚Üí    b1: u32 = *(buf + 1);
   888‚Üí
   889‚Üí    free(buf);
   890‚Üí    return 0;
   891‚Üí}
   892‚Üí```
   893‚Üí
   894‚Üí### Pattern 3: Byte Manipulation (Extract/Insert)
   895‚Üí
   896‚Üí```vuma
   897‚Üí// Extract byte n from a u32 (big-endian)
   898‚Üífn get_byte(val: u32, n: u32) -> u32 {
   899‚Üí    return (val >> (n * 8)) & 255;
   900‚Üí}
   901‚Üí
   902‚Üí// Set byte n in a u32 (big-endian)
   903‚Üífn set_byte(val: u32, n: u32, byte: u32) -> u32 {
   904‚Üí    shift: u32 = n * 8;
   905‚Üí    mask: u32 = 255 << shift;
   906‚Üí    return (val & (mask ^ 4294967295)) | ((byte & 255) << shift) & 4294967295;
   907‚Üí}
   908‚Üí```
   909‚Üí
   910‚Üí### Pattern 4: Memory Copy
   911‚Üí
   912‚Üí```vuma
   913‚Üífn copy_bytes(dst: Address, src: Address, len: u64) {
   914‚Üí    i: u64 = 0;
   915‚Üí    for i in 0..len {
   916‚Üí        *(dst + i) = *(src + i);
   917‚Üí    }
   918‚Üí}
   919‚Üí```
   920‚Üí
   921‚Üí### Pattern 5: Memory Set (Fill with Value)
   922‚Üí
   923‚Üí```vuma
   924‚Üífn mem_set(buf: Address, len: u64, val: u32) {
   925‚Üí    i: u64 = 0;
   926‚Üí    for i in 0..len {
   927‚Üí        *(buf + i) = val & 255;
   928‚Üí    }
   929‚Üí}
   930‚Üí```
   931‚Üí
   932‚Üí### Pattern 6: Array-style Access via Stride
   933‚Üí
   934‚ÜíWhen storing an array of u32 values in a byte buffer, each element occupies 4 bytes (stride = 4):
   935‚Üí
   936‚Üí```vuma
   937‚Üífn w_store(w_base: Address, idx: u64, val: u32) {
   938‚Üí    write_u32_be(w_base, idx * 4, val);
   939‚Üí}
   940‚Üí
   941‚Üífn w_load(w_base: Address, idx: u64) -> u32 {
   942‚Üí    return read_u32_be(w_base, idx * 4);
   943‚Üí}
   944‚Üí```
   945‚Üí
   946‚Üí### Pattern 7: SHA-256 Compression Round
   947‚Üí
   948‚Üí```vuma
   949‚Üífor i in 0..64 {
   950‚Üí    ki: u32 = read_u32_be(k, i * 4);
   951‚Üí    wi: u32 = w_load(w, i);
   952‚Üí    t1: u32 = (h + big_sigma1(e) + ch(e, f, g) + ki + wi) & 4294967295;
   953‚Üí    t2: u32 = (big_sigma0(a) + maj(a, b, c)) & 4294967295;
   954‚Üí    h = g;
   955‚Üí    g = f;
   956‚Üí    f = e;
   957‚Üí    e = (d + t1) & 4294967295;
   958‚Üí    d = c;
   959‚Üí    c = b;
   960‚Üí    b = a;
   961‚Üí    a = (t1 + t2) & 4294967295;
   962‚Üí}
   963‚Üí```

   966‚Üí
   967‚Üí## 14. Pitfalls
   968‚Üí
   969‚ÜíThis section documents the most common mistakes when writing VUMA code. **Read this section carefully.**
   970‚Üí
   971‚Üí### Pitfall 1: u32 Arithmetic Without Masking
   972‚Üí
   973‚Üí**Problem**: VUMA uses 64-bit registers. When two u32 values are added, the 64-bit result can have bits set above bit 31 (the carry). Subsequent operations on this value (especially right shifts) will produce wrong results.
   974‚Üí
   975‚Üí```vuma
   976‚Üí// WRONG
   977‚Üísum: u32 = a + b;        // upper 32 bits may have carry
   978‚Üírotated: u32 = sum >> 4;  // wrong! shift includes garbage bits
   979‚Üí
   980‚Üí// CORRECT
   981‚Üísum: u32 = (a + b) & 4294967295;  // clear upper 32 bits
   982‚Üírotated: u32 = sum >> 4;          // now correct
   983‚Üí```
   984‚Üí
   985‚Üí**Rule**: After every arithmetic operation (`+`, `-`, `*`) on u32 values, apply `& 4294967295`.
   986‚Üí
   987‚Üí### Pitfall 2: Using `~` for Bitwise NOT
   988‚Üí
   989‚Üí**Problem**: The `~` operator inverts all 64 bits. For a u32 value in a 64-bit register, this means bits 32-63 are also inverted, producing `0xFFFFFFFF????????` instead of `0x00000000????????`.
   990‚Üí
   991‚Üí```vuma
   992‚Üí// WRONG
   993‚Üínot_x = ~x;  // produces 0xFFFFFFFF???????? for u32 x
   994‚Üí
   995‚Üí// CORRECT
   996‚Üínot_x: u32 = x ^ 4294967295;  // flips only lower 32 bits
   997‚Üí```
   998‚Üí
   999‚Üí**Rule**: Never use `~` for u32 bitwise NOT. Always use `x ^ 4294967295`.
  1000‚Üí
  1001‚Üí### Pitfall 3: Left Shift Without Masking
  1002‚Üí
  1003‚Üí**Problem**: Left-shifting a u32 value by N bits in a 64-bit register moves bits into the upper 32 bits.
  1004‚Üí
  1005‚Üí```vuma
  1006‚Üí// WRONG
  1007‚Üíshifted: u32 = value << 24;  // bits above bit 31 are set
  1008‚Üí
  1009‚Üí// CORRECT
  1010‚Üíshifted: u32 = (value << 24) & 4294967295;  // clear upper 32 bits
  1011‚Üí```
  1012‚Üí
  1013‚Üí**Rule**: After any left shift of a u32 value, apply `& 4294967295`.
  1014‚Üí
  1015‚Üí### Pitfall 4: Rotate Without Masking
  1016‚Üí
  1017‚Üí**Problem**: A rotate-right composed from shifts produces bits above bit 31 from the left-shift component.
  1018‚Üí
  1019‚Üí```vuma
  1020‚Üí// WRONG
  1021‚Üírotated: u32 = (x >> n) | (x << (32 - n));  // upper bits polluted
  1022‚Üí
  1023‚Üí// CORRECT
  1024‚Üírotated: u32 = ((x >> n) | (x << (32 - n))) & 4294967295;
  1025‚Üí```
  1026‚Üí
  1027‚Üí**Rule**: Always mask the result of a rotate operation.
  1028‚Üí
  1029‚Üí### Pitfall 5: Forgetting to Free Memory
  1030‚Üí
  1031‚Üí**Problem**: Every `allocate()` must be matched with a `free()`. The IVE checks this, but it is easy to forget in complex control flow.
  1032‚Üí
  1033‚Üí```vuma
  1034‚Üí// WRONG ‚Äî memory leak
  1035‚Üífn bad() -> i32 {
  1036‚Üí    buf = allocate(64);
  1037‚Üí    return 0;  // buf is never freed!
  1038‚Üí}
  1039‚Üí
  1040‚Üí// CORRECT
  1041‚Üífn good() -> i32 {
  1042‚Üí    buf = allocate(64);
  1043‚Üí    // ... use buf ...
  1044‚Üí    free(buf);
  1045‚Üí    return 0;
  1046‚Üí}
  1047‚Üí```
  1048‚Üí
  1049‚Üí**Rule**: Always pair `allocate()` with `free()`. Free in reverse order of allocation to match the typical stack discipline.
  1050‚Üí
  1051‚Üí### Pitfall 6: Using `!x` for Bitwise NOT
  1052‚Üí
  1053‚Üí**Problem**: The `!` operator is logical NOT (converts non-zero to 0 and 0 to 1), not bitwise NOT.
  1054‚Üí
  1055‚Üí```vuma
  1056‚Üí// WRONG ‚Äî logical NOT, not bitwise NOT
  1057‚Üínot_x = !x;  // produces 0 or 1, not bitwise complement
  1058‚Üí
  1059‚Üí// CORRECT ‚Äî bitwise NOT via XOR
  1060‚Üínot_x: u32 = x ^ 4294967295;
  1061‚Üí```
  1062‚Üí
  1063‚Üí**Rule**: Use `x ^ 4294967295` for bitwise NOT, never `!x` or `~x`.
  1064‚Üí
  1065‚Üí### Pitfall 7: Hex Literals as Byte Values in Stores
  1066‚Üí
  1067‚Üí**Problem**: When storing a value to memory via `*(ptr + offset) = value`, only the low 8 bits are stored. Make sure you are not accidentally storing a truncated value.
  1068‚Üí
  1069‚Üí```vuma
  1070‚Üí// This stores only the low 8 bits (0xC2)
  1071‚Üí*(buf + 0) = 0x4FC2;  // stores 0xC2, NOT 0x4FC2
  1072‚Üí
  1073‚Üí// To store a specific byte, mask explicitly:
  1074‚Üí*(buf + 0) = (0x4FC2 >> 8) & 255;  // stores 0x4F
  1075‚Üí*(buf + 1) = 0x4FC2 & 255;        // stores 0xC2
  1076‚Üí```
  1077‚Üí
  1078‚Üí### Pitfall 8: Exit Code Truncation
  1079‚Üí
  1080‚Üí**Problem**: The process exit code is typically 8 bits (0-255) on Linux. When `main()` returns a u32 or i32, only the low byte is used as the exit code.
  1081‚Üí
  1082‚Üí```vuma
  1083‚Üífn main() -> i32 {
  1084‚Üí    // To exit with code 79 (0x4F):
  1085‚Üí    return 79;  // exit code = 79
  1086‚Üí}
  1087‚Üí```
  1088‚Üí
  1089‚ÜíIf you need to verify a specific byte from a u32 computation, mask it and return:
  1090‚Üí
  1091‚Üí```vuma
  1092‚Üífn main() -> i32 {
  1093‚Üí    result: u32 = compute_something();
  1094‚Üí    return result & 255;  // return low byte as exit code
  1095‚Üí}
  1096‚Üí```
  1097‚Üí
  1098‚Üí### Pitfall 9: Reading Bytes as u32 Without Masking
  1099‚Üí
  1100‚ÜíWhen you read a byte from memory into a `u32` variable, the value is zero-extended by the load. However, if you then shift and combine bytes, the result must be masked:
  1101‚Üí
  1102‚Üí```vuma
  1103‚Üí// Each byte read is zero-extended to u32, but after shifting
  1104‚Üí// and combining, upper bits may be set
  1105‚Üíb0: u32 = *(buf + 0);  // e.g., 0x61
  1106‚Üícombined: u32 = (b0 << 24);  // 0x61000000 in 64-bit register, but
  1107‚Üí                               // might have upper bits
  1108‚Üícombined: u32 = (b0 << 24) & 4294967295;  // CORRECT
  1109‚Üí```
  1110‚Üí
  1111‚Üí### Pitfall 10: Range Bounds in For Loops
  1112‚Üí
  1113‚ÜíThe range `a..b` is exclusive on the upper bound. `0..64` means 0 through 63, not 0 through 64:
  1114‚Üí
  1115‚Üí```vuma
  1116‚Üífor i in 0..64 {
  1117‚Üí    // i goes from 0 to 63 inclusive (64 iterations)
  1118‚Üí}
  1119‚Üí
  1120‚Üífor i in 16..64 {
  1121‚Üí    // i goes from 16 to 63 inclusive (48 iterations)
  1122‚Üí}
  1123‚Üí```

  1127‚Üí## 15. Target Platforms
  1128‚Üí
  1129‚ÜíVUMA compiles to native machine code for 8 architectures plus Wasm. Each target has its own backend that translates the IR to machine code.
  1130‚Üí
  1131‚Üí### Supported Targets
  1132‚Üí
  1133‚Üí| Target | Status | Pointer Size | Register Width | Notes |
  1134‚Üí|--------|--------|-------------|----------------|-------|
  1135‚Üí| x86_64 | Stable | 8 bytes | 64-bit | System V AMD64 ABI |
  1136‚Üí| AArch64 | Stable | 8 bytes | 64-bit | AAPCS64 calling convention |
  1137‚Üí| RISC-V 64 | Stable | 8 bytes | 64-bit | RV64GC, LP64 ABI |
  1138‚Üí| ARM32 | Stable | 4 bytes | 32-bit | AAPCS, no u32 masking needed* |
  1139‚Üí| MIPS64 | Stable | 8 bytes | 64-bit | N64 ABI |
  1140‚Üí| PPC64 | Stable | 8 bytes | 64-bit | ELFv2 ABI |
  1141‚Üí| LoongArch64 | Experimental | 8 bytes | 64-bit | LP64 ABI, QEMU slow |
  1142‚Üí| Wasm32 | Experimental | 4 bytes | 32-bit | WebAssembly MVP |
  1143‚Üí
  1144‚Üí*ARM32 uses 32-bit registers, so u32 masking is not needed on this target. However, **write code that works on all targets** by always masking u32 results.
  1145‚Üí
  1146‚Üí### Cross-Compilation and Testing

  1163‚Üí# Test on MIPS64

  1173‚Üí### Target-Specific Considerations
  1174‚Üí
  1175‚Üí**64-bit targets (x86_64, AArch64, RISC-V 64, MIPS64, PPC64, LoongArch64)**:
  1176‚Üí- All use 64-bit general-purpose registers
  1177‚Üí- u32 values stored in 64-bit registers **must** be masked with `& 4294967295` after arithmetic and left shifts
  1178‚Üí- `Address` is 8 bytes
  1179‚Üí- Pointer arithmetic uses 64-bit addition
  1180‚Üí
  1181‚Üí**32-bit targets (ARM32, Wasm32)**:
  1182‚Üí- Use 32-bit registers
  1183‚Üí- u32 masking is technically unnecessary but recommended for portability
  1184‚Üí- `Address` is 4 bytes
  1185‚Üí- Pointer arithmetic uses 32-bit addition
  1186‚Üí
  1187‚Üí**Wasm32 specifics**:
  1188‚Üí- No native ROR/ROL instructions; rotates are synthesized from shift+or sequences
  1189‚Üí- Type inference can be tricky; explicit type annotations are important
  1190‚Üí- Stack machine architecture (no registers); the backend manages a virtual value stack
  1191‚Üí
  1192‚Üí### Portability Rules
  1193‚Üí
  1194‚ÜíWhen writing VUMA code, follow these rules to ensure it works on all 8 targets:
  1195‚Üí
  1196‚Üí1. **Always mask u32 results** with `& 4294967295` after addition, subtraction, multiplication, and left shift.
  1197‚Üí2. **Use `x ^ 4294967295`** for bitwise NOT, never `~x`.
  1198‚Üí3. **Compose rotates from shifts** with a final mask: `((x >> n) | (x << (32 - n))) & 4294967295`.
  1199‚Üí4. **Use explicit type annotations** for all variable declarations.
  1200‚Üí5. **Use `u64` for offsets and lengths**, not `u32`, to avoid truncation on large buffers.
  1201‚Üí6. **Always pair `allocate()` with `free()`**.
