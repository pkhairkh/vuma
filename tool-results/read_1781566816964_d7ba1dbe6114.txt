    25→## 1. Introduction
    26→
    27→This document provides the formal specification for the five global invariants of the Verified-Unsafe Memory Access (VUMA) model, as introduced in Layer 6 of the AI-Native Language Design Framework (Proposal Section 3.6).
    28→
    29→The VUMA model inverts the traditional safety paradigm: instead of restricting access to prevent misuse, it permits unrestricted access and verifies safety through global reasoning. The Inference and Verification Engine (IVE) constructs a **Memory State Graph (MSG)** and proves that the program's memory behavior satisfies all five invariants simultaneously. If the IVE cannot prove an invariant for a specific access, it flags that access with the execution path leading to the potential violation.
    30→
    31→The five invariants are:
    32→
    33→| # | Invariant | Concern |
    34→|---|-----------|---------|
    35→| 1 | **Liveness** | Every access targets allocated memory |
    36→| 2 | **Exclusivity** | No conflicting concurrent accesses |
    37→| 3 | **Interpretation** | Access respects the Representation Descriptor |
    38→| 4 | **Origin** | Every address traces to a valid allocation |
    39→| 5 | **Cleanup** | Every allocation is eventually freed or explicitly leaked |
    40→
    41→These invariants are **not independent** — Liveness depends on Origin, and Cleanup refines Liveness temporally. The dependency structure is made explicit in Section 8.
    42→

    67→## 2. Memory State Graph (MSG) Formal Definition
    68→
    69→The MSG is the central data structure that the IVE constructs and reasons over. It is a typed, attributed, directed hypergraph that captures the complete memory behavior of a program.
    70→
    71→### 2.1 Definition
    72→
    73→$$
    74→\text{MSG} = (\mathcal{R},\ \mathcal{D},\ \mathcal{A},\ \mathcal{S})
    75→$$
    76→
    77→where:
    78→
    79→| Component | Type | Description |
    80→|-----------|------|-------------|
    81→| $\mathcal{R}$ | Set of **Region** | All memory regions in the program |
    82→| $\mathcal{D}$ | Set of **Derivation** | All pointer derivations |
    83→| $\mathcal{A}$ | Set of **Access** | All memory accesses |
    84→| $\mathcal{S}$ | Set of **SyncEdge** | All synchronization edges |
    85→
    86→### 2.2 Region
    87→
    88→A Region represents a contiguous range of addresses with associated metadata:
    89→
    90→$$
    91→\text{Region} = \{\\
    92→\quad \text{id} : \text{RegionId},\\
    93→\quad \text{base\_addr} : \text{Addr},\\
    94→\quad \text{size} : \mathbb{N},\\
    95→\quad \text{status} : \text{Allocated} \mid \text{Freed} \mid \text{Stack} \mid \text{Mapped},\\
    96→\quad \text{alloc\_point} : \text{PP},\\
    97→\quad \text{free\_point} : \text{PP}\ ?\\
    98→\}
    99→$$
   100→
   101→**Auxiliary definitions:**
   102→
   103→- **Address range**: $\text{range}(r) = [r.\text{base\_addr},\ r.\text{base\_addr} + r.\text{size})$
   104→- **Byte range overlap**: $\text{overlap}(r_1, r_2) \iff \text{range}(r_1) \cap \text{range}(r_2) \neq \emptyset$
   105→- **Contains address**: $\text{contains}(r, a) \iff a \in \text{range}(r)$
   106→- **Status at program point**: Defined by the allocation/deallocation ordering (see Invariant 1).
   107→
   108→### 2.3 Derivation
   109→
   110→A Derivation represents the computation of an address from another address or allocation:
   111→
   112→$$
   113→\text{Derivation} = \{\\
   114→\quad \text{id} : \text{DerivationId},\\
   115→\quad \text{source} : \text{Region} \mid \text{Derivation},\\
   116→\quad \text{offset} : \text{Offset},\\
   117→\quad \text{cast} : \text{RepD}\ ?\\
   118→\}
   119→$$
   120→
   121→**Kinds of derivation:**
   122→
   123→1. **Base derivation**: $\text{source} \in \mathcal{R}$, $\text{offset} = 0$, $\text{cast} = \text{null}$. This is the address of the first byte of a freshly allocated region.
   124→2. **Offset derivation**: $\text{source} \in \mathcal{R} \cup \mathcal{D}$, $\text{offset} \neq 0$, $\text{cast} = \text{null}$. This is pointer arithmetic: adding a byte offset to a source address.
   125→3. **Cast derivation**: $\text{source} \in \mathcal{D}$, $\text{offset} = 0$, $\text{cast} \neq \text{null}$. This is a RepD reinterpretation of the same address.
   126→
   127→A derivation chain forms a directed tree rooted at a Region:
   128→
   129→$$
   130→\text{Derivations form a forest where every tree root is a Region.}
   131→$$
   132→
   133→### 2.4 Access
   134→
   135→An Access represents a single read or write operation targeting a derived address:
   136→
   137→$$
   138→\text{Access} = \{\\
   139→\quad \text{id} : \text{AccessId},\\
   140→\quad \text{target} : \text{Derivation},\\
   141→\quad \text{kind} : \text{Read} \mid \text{Write},\\
   142→\quad \text{size} : \mathbb{N},\\
   143→\quad \text{program\_point} : \text{PP}\\
   144→\}
   145→$$
   146→
   147→**Auxiliary definitions:**
   148→
   149→- **Accessed byte range**: $\text{bytes}(a) = [\text{addr}(a.\text{target}),\ \text{addr}(a.\text{target}) + a.\text{size})$
   150→- **Access kind is write**: $\text{is\_write}(a) \iff a.\text{kind} = \text{Write}$
   151→
   152→### 2.5 SyncEdge
   153→
   154→A SyncEdge represents a synchronization relationship between two accesses, establishing an ordering:
   155→
   156→$$
   157→\text{SyncEdge} = \{\\
   158→\quad \text{access}_1 : \text{Access},\\
   159→\quad \text{access}_2 : \text{Access},\\
   160→\quad \text{ordering} : \text{HappensBefore} \mid \text{Atomic} \mid \text{Locked}\\
   161→\}
   162→$$
   163→
   164→**Ordering semantics:**
   165→
   166→| Ordering | Meaning |
   167→|----------|---------|
   168→| **HappensBefore** | $a_1$ completes before $a_2$ begins (sequential consistency, fork-join, message passing) |
   169→| **Atomic** | $a_1$ and $a_2$ access the same atomic variable; memory ordering guarantees visibility |
   170→| **Locked** | $a_1$ and $a_2$ are guarded by the same lock; mutual exclusion is guaranteed |
   171→
   172→**Ordered relation (transitive closure):**
   173→
   174→$$
   175→\text{ordered}(a_1, a_2) \iff \exists\ n \geq 1,\ \exists\ e_1, \ldots, e_n \in \mathcal{S} :\\
   176→\quad e_1.\text{access}_1 = a_1 \land e_n.\text{access}_2 = a_2 \land\\
   177→\quad \forall\ i \in [1, n):\ e_i.\text{access}_2 = e_{i+1}.\text{access}_1
   178→$$
   179→
   180→That is, `ordered` is the reflexive-transitive closure of the SyncEdge relation, establishing a partial order over accesses.
   181→
   182→### 2.6 Derived Functions
   183→
   184→**Address resolution** — compute the concrete address a derivation refers to:
   185→
   186→$$
   187→\text{addr}(d) = \begin{cases}
   188→r.\text{base\_addr} + d.\text{offset} & \text{if } d.\text{source} \in \mathcal{R} \text{ (a Region)} \\
   189→\text{addr}(d.\text{source}) + d.\text{offset} & \text{if } d.\text{source} \in \mathcal{D} \text{ (a Derivation)}
   190→\end{cases}
   191→$$
   192→
   193→**Region of a derivation** — find the root Region of the derivation tree:
   194→
   195→$$
   196→\text{region\_of}(d) = \begin{cases}
   197→d.\text{source} & \text{if } d.\text{source} \in \mathcal{R} \\
   198→\text{region\_of}(d.\text{source}) & \text{if } d.\text{source} \in \mathcal{D}
   199→\end{cases}
   200→$$
   201→
   202→**Well-formedness**: $\text{region\_of}$ is well-defined because derivations form a forest rooted at Regions (Section 2.3), ensuring termination.
   203→
   204→**Effective RepD of a derivation** — the most recent cast in the derivation chain:
   205→
   206→$$
   207→\text{repd\_of}(d) = \begin{cases}
   208→d.\text{cast} & \text{if } d.\text{cast} \neq \text{null} \\
   209→\text{repd\_of}(d.\text{source}) & \text{if } d.\text{source} \in \mathcal{D} \land d.\text{cast} = \text{null} \\
   210→\text{default\_repd}(\text{region\_of}(d)) & \text{if } d.\text{source} \in \mathcal{R} \land d.\text{cast} = \text{null}
   211→\end{cases}
   212→$$
   213→
   214→where $\text{default\_repd}(r)$ is the RepD specified at the allocation point of $r$.
   215→

   218→## 3. Invariant 1: Liveness
   219→
   220→> **Every access targets allocated memory.**
   221→
   222→### 3.1 Formal Statement
   223→
   224→$$
   225→\boxed{\forall\ a \in \mathcal{A} :\ \text{is\_allocated}(\text{region\_of}(a.\text{target}),\ a.\text{program\_point})}
   226→$$
   227→
   228→**Definition of `is_allocated`:**
   229→
   230→$$
   231→\text{is\_allocated}(r, pp) \iff \begin{cases}
   232→\text{true} & \text{if } r.\text{status} = \text{Stack} \land pp \in \text{lifetime}(r) \\
   233→\text{true} & \text{if } r.\text{status} = \text{Allocated} \land r.\text{alloc\_point} \leq_{pp} pp \land (r.\text{free\_point} = \text{null} \lor pp <_{pp} r.\text{free\_point}) \\
   234→\text{true} & \text{if } r.\text{status} = \text{Mapped} \land \text{mapping\_valid\_at}(r, pp) \\
   235→\text{false} & \text{otherwise}
   236→\end{cases}
   237→$$
   238→
   239→where $\leq_{pp}$ is the program-point ordering (the partial order defined by the SCG's control flow edges), and $\text{lifetime}(r)$ for Stack regions is the set of program points between the frame entry and frame exit.
   240→
   241→**Additional requirement — bounds check:**
   242→
   243→$$
   244→\forall\ a \in \mathcal{A} :\ \text{bytes}(a) \subseteq \text{range}(\text{region\_of}(a.\text{target}))
   245→$$
   246→
   247→That is, the accessed byte range must be fully contained within the source region.
   248→
   249→### 3.2 Proof Strategy Sketch
   250→
   251→1. **Region state analysis**: For each Region $r$, compute the set of program points where $r.\text{status} \in \{\text{Allocated}, \text{Stack}, \text{Mapped}\}$ by analyzing the allocation and deallocation points. This yields a temporal interval $[r.\text{alloc\_point},\ r.\text{free\_point})$ during which the region is live.
   252→
   253→2. **Derivation resolution**: For each Access $a$, resolve $\text{region\_of}(a.\text{target})$ by traversing the derivation chain to its root Region. This is guaranteed to terminate by the forest structure of derivations.
   254→
   255→3. **Containment check**: Verify that $\text{bytes}(a) \subseteq \text{range}(\text{region\_of}(a.\text{target}))$ by comparing the resolved address and access size against the region bounds. For offset derivations, verify that $\text{offset} + a.\text{size} \leq \text{region\_of}(a.\text{target}).\text{size}$.
   256→
   257→4. **Liveness check**: Verify that $a.\text{program\_point}$ falls within the live interval of $\text{region\_of}(a.\text{target})$. This requires reasoning about the control flow graph to determine reachability.
   258→
   259→5. **Path sensitivity**: For programs with conditional deallocation, the IVE must enumerate feasible execution paths. If liveness holds on all feasible paths, the invariant is satisfied. If any feasible path violates liveness, the IVE produces that path as a counterexample.
   260→
   261→### 3.3 Example: Satisfying Program
   262→
   263→```
   264→// Simple allocation, use, and free
   265→r = allocate(bytes[64]);     // Region r: base=0x1000, size=64, alloc_point=PP1
   266→d = derive(r, offset=32);    // Derivation d: source=r, offset=32
   267→a = read(d, size=4);         // Access a: target=d, kind=Read, size=4, pp=PP2
   268→free(r);                     // r.free_point = PP3
   269→
   270→// Verification:
   271→//   region_of(d) = r
   272→//   bytes(a) = [0x1020, 0x1024) ⊆ [0x1000, 0x1040) = range(r) ✓
   273→//   is_allocated(r, PP2): PP1 ≤ PP2 < PP3 ✓
   274→// Liveness invariant: SATISFIED
   275→```
   276→
   277→### 3.4 Example: Violating Program
   278→
   279→```
   280→// Use-after-free
   281→r = allocate(bytes[64]);     // Region r: base=0x1000, size=64, alloc_point=PP1
   282→free(r);                     // r.free_point = PP2
   283→d = derive(r, offset=0);     // Derivation d: source=r, offset=0
   284→a = read(d, size=4);         // Access a: target=d, kind=Read, size=4, pp=PP3
   285→
   286→// Verification:
   287→//   region_of(d) = r
   288→//   is_allocated(r, PP3): PP2 ≤ PP3, but r.free_point = PP2
   289→//   PP3 ≥ PP2 ⇒ NOT allocated at PP3
   290→// Liveness invariant: VIOLATED
   291→// IVE report: "Access a at PP3 targets freed region r (freed at PP2)"
   292→```

   296→## 4. Invariant 2: Exclusivity
   297→
   298→> **No conflicting concurrent accesses exist without synchronization.**
   299→
   300→### 4.1 Formal Statement
   301→
   302→$$
   303→\boxed{\forall\ a_1, a_2 \in \mathcal{A} :\ \text{conflicts}(a_1, a_2) \Rightarrow \text{ordered}(a_1, a_2)}
   304→$$
   305→
   306→**Definition of `conflicts`:**
   307→
   308→$$
   309→\text{conflicts}(a_1, a_2) \iff \text{is\_write}(a_1) \lor \text{is\_write}(a_2)\\
   310→\quad \land \text{region\_of}(a_1.\text{target}) = \text{region\_of}(a_2.\text{target})\\
   311→\quad \land \text{bytes}(a_1) \⌣\ \text{bytes}(a_2)\\
   312→\quad \land a_1 \neq a_2
   313→$$
   314→
   315→where $⌣$ denotes byte-range overlap:
   316→
   317→$$
   318→[b_1, e_1) \⌣\ [b_2, e_2) \iff b_1 < e_2 \land b_2 < e_1
   319→$$
   320→
   321→**Key observations:**
   322→
   323→- Two Read accesses never conflict (reads are always safe to overlap).
   324→- A Read and a Write conflict if they touch overlapping bytes.
   325→- Two Write accesses conflict if they touch overlapping bytes.
   326→- The `ordered` relation (transitive closure of SyncEdges, defined in Section 2.5) establishes that one access completes before the other begins, preventing simultaneous execution.
   327→
   328→### 4.2 Proof Strategy Sketch
   329→
   330→1. **Conflict pair enumeration**: For each pair of accesses $(a_1, a_2)$ where at least one is a Write, check:
   331→   - (a) They target the same Region: $\text{region\_of}(a_1.\text{target}) = \text{region\_of}(a_2.\text{target})$
   332→   - (b) Their byte ranges overlap: $\text{bytes}(a_1) \⌣\ \text{bytes}(a_2)$
   333→
   334→2. **Concurrent execution analysis**: For each conflict pair, determine whether they can execute concurrently. Two accesses are concurrent iff neither $\text{ordered}(a_1, a_2)$ nor $\text{ordered}(a_2, a_1)$ holds.
   335→
   336→3. **Synchronization graph construction**: Build the synchronization graph $G_{\text{sync}}$ from $\mathcal{S}$. Compute the transitive closure to establish the `ordered` relation.
   337→
   338→4. **Reachability check**: For each conflict pair $(a_1, a_2)$, verify that $a_2$ is reachable from $a_1$ (or vice versa) in $G_{\text{sync}}$. If neither direction is reachable, the accesses are concurrent and conflicting — the invariant is violated.
   339→
   340→5. **Lock-set analysis (optional strengthening)**: For Locked SyncEdges, verify that the accesses are guarded by the same lock instance. Different lock instances do not provide mutual exclusion.
   341→
   342→6. **Atomic access classification**: For Atomic SyncEdges, verify that the overlapping accesses are to the same atomic variable with compatible memory ordering (acquire/release, sequentially consistent).
   343→
   344→### 4.3 Example: Satisfying Program
   345→
   346→```
   347→// Mutex-protected write with concurrent read
   348→r = allocate(bytes[64]);        // Region r: base=0x1000, size=64
   349→d1 = derive(r, offset=0);       // Derivation d1 → r[0..4)
   350→d2 = derive(r, offset=0);       // Derivation d2 → r[0..4)
   351→
   352→// Thread 1
   353→lock(mutex);
   354→a1 = write(d1, size=4);         // Access a1: pp=PP1, kind=Write
   355→unlock(mutex);
   356→
   357→// Thread 2
   358→lock(mutex);
   359→a2 = read(d2, size=4);          // Access a2: pp=PP2, kind=Read
   360→unlock(mutex);
   361→
   362→// SyncEdges:
   363→//   S1 = { access1=a1, access2=a2, ordering=Locked }
   364→
   365→// Verification:
   366→//   conflicts(a1, a2): is_write(a1)=true, same region, bytes overlap ✓
   367→//   ordered(a1, a2): S1 establishes a1→a2 via Locked ordering ✓
   368→// Exclusivity invariant: SATISFIED
   369→```
   370→
   371→### 4.4 Example: Violating Program
   372→
   373→```
   374→// Data race: unsynchronized write and read
   375→r = allocate(bytes[64]);        // Region r: base=0x1000, size=64
   376→d1 = derive(r, offset=0);
   377→d2 = derive(r, offset=0);
   378→
   379→// Thread 1 (no synchronization)
   380→a1 = write(d1, size=4);         // Access a1: pp=PP1, kind=Write
   381→
   382→// Thread 2 (no synchronization)
   383→a2 = read(d2, size=4);          // Access a2: pp=PP2, kind=Read
   384→
   385→// No SyncEdge between a1 and a2
   386→
   387→// Verification:
   388→//   conflicts(a1, a2): is_write(a1)=true, same region, bytes overlap ✓
   389→//   ordered(a1, a2): no path in sync graph → NOT ordered ✗
   390→// Exclusivity invariant: VIOLATED
   391→// IVE report: "Write a1 at PP1 and Read a2 at PP2 conflict on region r[0..4) without synchronization"
   392→```

   396→## 5. Invariant 3: Interpretation
   397→
   398→> **Every access respects the Representation Descriptor (RepD) of its target.**
   399→
   400→### 5.1 Formal Statement
   401→
   402→$$
   403→\boxed{\forall\ a \in \mathcal{A} :\ \text{compatible}(\text{repd\_of}(a.\text{target}),\ \text{expected\_repd}(a))}
   404→$$
   405→
   406→where $\text{expected\_repd}(a)$ is the RepD required by the operation performed at access $a$ (e.g., a pointer dereference expects a RepD with pointer layout; an integer add expects an integer RepD).
   407→
   408→**Definition of `compatible`:**
   409→
   410→$$
   411→\text{compatible}(r_1, r_2) \iff r_1.\text{size} = r_2.\text{size} \land r_1.\text{alignment} \mid \text{addr}(a.\text{target}) \land \text{valid\_reinterpretation}(r_1, r_2)
   412→$$
   413→
   414→where $\text{valid\_reinterpretation}$ is defined as:
   415→
   416→$$
   417→\text{valid\_reinterpretation}(r_1, r_2) \iff \begin{cases}
   418→\text{true} & \text{if } r_1 = r_2 \text{ (same RepD)} \\
   419→\text{true} & \text{if } r_1 \sqsubseteq r_2 \text{ (r1 is a sub-Repd of r2, e.g., bytes ⊑ any)} \\
   420→\text{true} & \text{if cast is explicit and } r_2 \text{ is a valid union member} \\
   421→\text{false} & \text{if } r_1 \in \text{PointerRepD} \land r_2 \notin \text{PointerRepD} \cup \text{BytesRepD} \\
   422→\text{false} & \text{otherwise (needs IVE case analysis)}
   423→\end{cases}
   424→$$
   425→
   426→**Uninitialized memory restriction:**
   427→
   428→$$
   429→\forall\ a \in \mathcal{A} :\ \text{expected\_repd}(a) \in \text{PointerRepD} \Rightarrow \text{is\_initialized}(\text{region\_of}(a.\text{target}),\ \text{bytes}(a))
   430→$$
   431→
   432→Reading uninitialized memory as a pointer type is always forbidden, because an arbitrary bit pattern interpreted as a pointer may point to an arbitrary address, violating Origin (Invariant 4).
   433→
   434→### 5.2 Proof Strategy Sketch
   435→
   436→1. **RepD propagation**: For each Derivation, compute $\text{repd\_of}(d)$ by walking the derivation chain. The most recent cast in the chain determines the effective RepD.
   437→
   438→2. **Cast validation**: For each cast derivation $d$ (where $d.\text{cast} \neq \text{null}$), verify that the cast is a valid reinterpretation. This requires:
   439→   - Size compatibility: the target RepD's size must not exceed the remaining bytes in the source region from the offset.
   440→   - Alignment: the resolved address must satisfy the target RepD's alignment requirement.
   441→   - Semantic compatibility: the IVE verifies that the bit pattern, when interpreted according to the target RepD, satisfies the invariants required by subsequent operations.
   442→
   443→3. **Uninitialized memory tracking**: The IVE maintains an initialization map $\text{init\_map} : \text{Region} \times \text{ByteOffset} \to \{\text{Init}, \text{Uninit}\}$. After a Write access, the written bytes are marked Init. The IVE verifies that no Read access with a Pointer RepD targets Uninit bytes.
   444→
   445→4. **Operation-RepD matching**: For each Access $a$, verify that the operation being performed is valid given $\text{repd\_of}(a.\text{target})$. For example:
   446→   - Pointer dereference requires $\text{repd\_of}(a.\text{target}) \in \text{PointerRepD}$.
   447→   - Floating-point operation requires a float RepD.
   448→   - Byte copy accepts any RepD (bytes is the universal supertype).
   449→
   450→5. **Transitivity of reinterpretation**: If a value is cast from RepD A to RepD B and then from B to C, the IVE must verify that A → B → C is a valid reinterpretation chain, not just that each individual step is valid. This prevents chains like `pointer → int → float` where the final interpretation is semantically unsound.
   451→
   452→### 5.3 Example: Satisfying Program
   453→
   454→```
   455→// Valid cast: interpret bytes as header struct
   456→r = allocate(bytes[1024]);           // Region r, default RepD: bytes[1024]
   457→d1 = derive(r, offset=0);            // Derivation d1, RepD: bytes[1024]
   458→d2 = derive(d1, offset=0,            // Cast derivation d2
   459→            cast=Header);             //   RepD: Header { magic: uint32, size: uint32 }
   460→a = read(d2, size=8);                // Access a: Read, expected_repd=Header
   461→
   462→// Verification:
   463→//   repd_of(d2) = Header (from cast)
   464→//   compatible(Header, Header): same RepD ✓
   465→//   addr(d2) = r.base_addr, alignment(Header)=4, r.base_addr % 4 = 0 ✓
   466→//   sizeof(Header) = 8 ≤ 1024 ✓
   467→// Interpretation invariant: SATISFIED
   468→```
   469→
   470→### 5.4 Example: Violating Program
   471→
   472→```
   473→// Reading uninitialized memory as a pointer
   474→r = allocate(bytes[64]);             // Region r, all bytes Uninit
   475→d1 = derive(r, offset=0);
   476→d2 = derive(d1, offset=0,            // Cast derivation
   477→            cast=ptr<u8>);            //   RepD: ptr<u8> (pointer type)
   478→a = read(d2, size=8);                // Access a: Read, expected_repd=ptr<u8>
   479→
   480→// Verification:
   481→//   repd_of(d2) = ptr<u8>
   482→//   expected_repd(a) = ptr<u8>
   483→//   compatible(ptr<u8>, ptr<u8>): same RepD ✓
   484→//   BUT: is_initialized(r, [0, 8)) = false (no prior Write to r)
   485→//   Reading Uninit bytes as pointer: FORBIDDEN
   486→// Interpretation invariant: VIOLATED
   487→// IVE report: "Access a reads uninitialized bytes r[0..8) as pointer type ptr<u8>"
   488→```

   492→## 6. Invariant 4: Origin
   493→
   494→> **Every address traces to a valid allocation; arithmetic derivations stay within bounds.**
   495→
   496→### 6.1 Formal Statement
   497→
   498→**Part A — Trace terminates at allocation:**
   499→
   500→$$
   501→\boxed{\forall\ d \in \mathcal{D} :\ \text{trace}(d) \neq \text{diverges}}
   502→$$
   503→
   504→$$
   505→\text{trace}(d) = \begin{cases}
   506→[r] & \text{if } d.\text{source} = r \in \mathcal{R} \\
   507→\text{trace}(d.\text{source}) \mathbin{+\!\!+} [d] & \text{if } d.\text{source} \in \mathcal{D}
   508→\end{cases}
   509→$$
   510→
   511→The trace always terminates because derivations form a forest (no cycles), and every tree root is a Region. Formally, the derivation graph is a DAG where all sinks are Regions.
   512→
   513→**Part B — Arithmetic derivations stay in bounds:**
   514→
   515→$$
   516→\boxed{\forall\ d \in \mathcal{D} :\ d.\text{offset} \neq 0 \Rightarrow 0 \leq d.\text{offset} \land d.\text{offset} \leq \text{region\_of}(d).\text{size}}
   517→$$
   518→
   519→More precisely, for any derivation chain $[r, d_1, d_2, \ldots, d_n]$ where $d_n$ is the final derivation:
   520→
   521→$$
   522→\text{addr}(d_n) + \text{max\_access\_size}(d_n) \leq \text{region\_of}(d_n).\text{base\_addr} + \text{region\_of}(d_n).\text{size}
   523→$$
   524→
   525→where $\text{max\_access\_size}(d_n)$ is the maximum size of any access targeting $d_n$.
   526→
   527→**Part C — No fabrication:**
   528→
   529→$$
   530→\forall\ a \in \mathcal{A} :\ \nexists\ \text{fabrication in trace}(a.\text{target})
   531→$$
   532→
   533→An address is "fabricated" if it is introduced by a computation that the IVE cannot trace to a valid allocation. Examples include: integer literals interpreted as addresses (`0xDEADBEEF`), addresses returned by FFI functions without derivation tracking, and addresses computed from uninitialized memory.
   534→
   535→### 6.2 Proof Strategy Sketch
   536→
   537→1. **DAG verification**: Verify that the derivation graph is acyclic. This is a structural property of the MSG — if the IVE constructs it correctly from the SCG, it is a DAG by construction. Detecting cycles is a simple graph traversal.
   538→
   539→2. **Trace computation**: For each Derivation $d$, compute $\text{trace}(d)$ by walking the `source` chain. Termination is guaranteed by the DAG property. Each trace element records the offset and cast applied at that step.
   540→
   541→3. **Bounds checking**: For each offset derivation $d$ with $d.\text{offset} \neq 0$, verify:
   542→   - $d.\text{offset} \geq 0$ (no negative offsets without explicit signed arithmetic — if signed, verify it doesn't go below the region base).
   543→   - The cumulative offset from the root Region's base address does not exceed the Region's size.
   544→
   545→4. **Fabrication detection**: For each derivation, verify that its source is either a Region or another Derivation — never a raw integer constant or an untracked external value. If the program contains FFI calls that return addresses, the IVE must either:
   546→   - Treat the returned address as a new Region (with the FFI call as the allocation point), or
   547→   - Mark the address as unproven and flag it for review.
   548→
   549→5. **Alias analysis**: If two different derivation chains produce the same address, verify that they trace to the same Region. Different Regions occupying overlapping addresses (in non-overlapping lifetimes) are acceptable; different Regions occupying overlapping addresses in overlapping lifetimes may indicate an Origin violation.
   550→
   551→### 6.3 Example: Satisfying Program
   552→
   553→```
   554→// Valid pointer arithmetic within bounds
   555→r = allocate(bytes[1024]);         // Region r: base=0x1000, size=1024
   556→d1 = derive(r, offset=64);         // addr(d1) = 0x1040
   557→d2 = derive(d1, offset=128);       // addr(d2) = 0x10C0
   558→d3 = derive(d2, offset=0,          // Cast at d2's address
   559→            cast=Header);           // RepD: Header (size=8)
   560→a = read(d3, size=8);              // Access bytes [0x10C0, 0x10C8)
   561→
   562→// Verification:
   563→//   trace(d3) = [r, d1, d2, d3]
   564→//   All sources are Region or Derivation: no fabrication ✓
   565→//   Cumulative offset: 64 + 128 + 0 = 192
   566→//   addr(d3) + 8 = 0x10C8 ≤ 0x1000 + 1024 = 0x1400 ✓
   567→// Origin invariant: SATISFIED
   568→```
   569→
   570→### 6.4 Example: Violating Program
   571→
   572→```
   573→// Fabricated address from integer literal
   574→r = allocate(bytes[1024]);         // Region r: base=0x1000, size=1024
   575→d = derive_from_integer(0xDEADBEEF); // Fabrication: source is not a Region or Derivation
   576→a = read(d, size=4);               // Access at 0xDEADBEEF
   577→
   578→// Verification:
   579→//   trace(d) does not terminate at a Region
   580→//   d.source is an integer literal, not a valid derivation source
   581→// Origin invariant: VIOLATED
   582→// IVE report: "Address 0xDEADBEEF has no traceable origin to a valid allocation"
   583→```

   587→## 7. Invariant 5: Cleanup
   588→
   589→> **Every allocation is eventually freed or explicitly leaked; no region is freed twice.**
   590→
   591→### 7.1 Formal Statement
   592→
   593→**Part A — Every region is freed or explicitly leaked:**
   594→
   595→$$
   596→\boxed{\forall\ r \in \mathcal{R} :\ r.\text{free\_point} \neq \text{null} \lor r.\text{status} = \text{Leaked}}
   597→$$
   598→
   599→where $\text{Leaked}$ is an explicit annotation applied by the programmer or inferred by the IVE to indicate intentional non-deallocation (e.g., long-lived arenas, global state, process-lifetime mappings).
   600→
   601→**Part B — No double-free:**
   602→
   603→$$
   604→\boxed{\forall\ r \in \mathcal{R} :\ \text{count\_free}(r) \leq 1}
   605→$$
   606→
   607→where $\text{count\_free}(r) = |\{pp \in \text{PP} : \text{free\_op}(pp, r)\}|$, and $\text{free\_op}(pp, r)$ means a free operation targeting $r$ occurs at program point $pp$.
   608→
   609→**Part C — Freed regions are not accessed (temporal safety, overlaps Liveness):**
   610→
   611→$$
   612→\forall\ r \in \mathcal{R},\ \forall\ a \in \mathcal{A} :\ \text{region\_of}(a.\text{target}) = r \land r.\text{free\_point} = pp_f \Rightarrow a.\text{program\_point} <_{pp} pp_f
   613→$$
   614→
   615→This is a restatement of the Liveness invariant (Invariant 1) but from the perspective of the region's lifecycle rather than the access's perspective. It is included here for completeness.
   616→
   617→### 7.2 Proof Strategy Sketch
   618→
   619→1. **Leak detection**: For each Region $r$ where $r.\text{free\_point} = \text{null}$:
   620→   - If $r.\text{status} = \text{Stack}$, the region is implicitly freed when the stack frame is deallocated. The IVE verifies that the stack frame deallocation point is reachable.
   621→   - If $r.\text{status} = \text{Allocated}$, the IVE checks whether $r$ is explicitly marked $\text{Leaked}$. If not, it searches for a feasible execution path where the program terminates or the region goes out of scope without deallocation.
   622→   - If $r.\text{status} = \text{Mapped}$, the IVE verifies that the mapping is revoked before program termination or that it is explicitly marked $\text{Leaked}$.
   623→
   624→2. **Double-free detection**: For each Region $r$, verify that at most one program point executes a free operation on $r$. This requires:
   625→   - Control flow analysis: verify that no execution path reaches two different free operations on the same region.
   626→   - Alias analysis: verify that two free operations on different derivation chains do not target the same region (i.e., `region_of(d1) = region_of(d2)` where both are freed).
   627→
   628→3. **Ownership tracking (optional strengthening)**: The IVE may track an "ownership set" for each Region — the set of derivation chains that have the authority to free the region. Only derivations in the ownership set may initiate a free. This prevents accidental double-free through aliasing.
   629→
   630→4. **Path-sensitive analysis**: For programs with conditional deallocation (e.g., `if (condition) free(r);`), the IVE must verify that:
   631→   - On all paths where the condition is true, the free occurs exactly once.
   632→   - On all paths where the condition is false, either the region is Leaked or it is freed on another path.
   633→
   634→5. **Leaked annotation verification**: If a region is marked $\text{Leaked}$, the IVE verifies that the annotation is justified:
   635→   - The region is reachable from a global root (arena, static variable).
   636→   - The region's lifetime is intended to span the entire process.
   637→   - No dangling references to the region exist after the process terminates (trivially true at process exit).
   638→
   639→### 7.3 Example: Satisfying Program
   640→
   641→```
   642→// Arena allocation with explicit leak annotation
   643→arena = allocate(bytes[65536]);     // Region arena, mark as Leaked (global arena)
   644→r1 = allocate_from(arena, 64);      // Sub-region r1, within arena
   645→r2 = allocate_from(arena, 128);     // Sub-region r2, within arena
   646→free(r1);                           // r1.free_point = PP1
   647→free(r2);                           // r2.free_point = PP2
   648→// arena: free_point = null, status = Leaked ✓
   649→
   650→// Verification:
   651→//   r1.free_point = PP1 ≠ null ✓
   652→//   r2.free_point = PP2 ≠ null ✓
   653→//   arena: Leaked annotation, justified as global arena ✓
   654→//   count_free(r1) = 1 ✓
   655→//   count_free(r2) = 1 ✓
   656→// Cleanup invariant: SATISFIED
   657→```
   658→
   659→### 7.4 Example: Violating Program
   660→
   661→```
   662→// Memory leak and double-free
   663→r = allocate(bytes[256]);          // Region r: alloc_point=PP1
   664→// ... no free(r) on any execution path ...
   665→
   666→// OR:
   667→
   668→r = allocate(bytes[256]);          // Region r: alloc_point=PP1
   669→free(r);                           // r.free_point = PP2
   670→free(r);                           // Second free at PP3
   671→
   672→// Verification (leak case):
   673→//   r.free_point = null, r.status ≠ Leaked
   674→//   No reachable free operation on any execution path
   675→// Cleanup invariant: VIOLATED (leak)
   676→// IVE report: "Region r allocated at PP1 is never freed and not marked Leaked"
   677→
   678→// Verification (double-free case):
   679→//   count_free(r) = 2 (at PP2 and PP3)
   680→// Cleanup invariant: VIOLATED (double-free)
   681→// IVE report: "Region r freed at PP2 and again at PP3"
   682→```

   686→## 8. Invariant Dependency Graph
   687→
   688→The five invariants are not independent. The following diagram shows their logical dependencies:
   689→
   690→```
   691→    ┌──────────┐
   692→    │  Origin  │  (Invariant 4)
   693→    │   (4)    │
   694→    └────┬─────┘
   695→         │ region_of() is well-defined
   696→         ▼
   697→    ┌──────────┐
   698→    │ Liveness │  (Invariant 1)
   699→    │   (1)    │
   700→    └────┬─────┘
   701→         │ region is allocated at access point
   702→         ▼
   703→    ┌──────────────┐     ┌────────────────┐
   704→    │ Exclusivity  │────▷│ Interpretation │
   705→    │     (2)      │     │      (3)       │
   706→    └──────────────┘     └────────────────┘
   707→         │                      │
   708→         │ write doesn't        │ RepD is valid
   709→         │ conflict             │
   710→         ▼                      ▼
   711→    ┌──────────┐
   712→    │  Cleanup │  (Invariant 5)
   713→    │   (5)    │
   714→    └──────────┘
   715→```
   716→
   717→**Dependency explanation:**
   718→
   719→| Dependency | Reason |
   720→|-----------|--------|
   721→| Liveness → Origin | `region_of()` (used by Liveness) is well-defined only if every derivation traces to a Region (Origin). |
   722→| Exclusivity → Liveness | Exclusivity checks overlap on allocated regions; checking exclusivity on freed memory is meaningless. |
   723→| Interpretation → Liveness | Reading a RepD from unallocated memory is a Liveness violation before it can be an Interpretation violation. |
   724→| Interpretation → Exclusivity | A Write that sets up a valid RepD must be exclusive with Reads that interpret it; otherwise the RepD may be observed in an inconsistent state. |
   725→| Cleanup → Liveness, Exclusivity | Cleanup is a temporal property: it ensures that the "allocated" status in Liveness eventually transitions to "freed." Cleanup depends on Liveness to define when a region is live. |
   726→
   727→**Proof order**: The IVE should verify invariants in topological order: Origin → Liveness → (Exclusivity, Interpretation) → Cleanup.
