   133‚Üí## 2025-03-04 ‚Äî Task 4-h: Wire up C FFI system end-to-end
   134‚Üí
   135‚Üí### Summary
   136‚Üí
   137‚ÜíWired the C FFI system end-to-end so that VUMA programs can declare and call
   138‚Üíexternal C functions via `extern "C" { ... }` blocks, and the resulting ELF
   139‚Üíobject files contain proper undefined symbol entries and relocations for the
   140‚Üísystem linker to resolve.
   141‚Üí
   142‚Üí### Changes Made
   143‚Üí
   144‚Üí#### 1. Parser/SCG side (`src/parser/src/to_scg.rs`)
   145‚Üí- No changes needed ‚Äî the parser already correctly parses `extern "C"` blocks
   146‚Üí  into `ExternBlockDef` with `ExternFnDecl`s, and `emit_call_nodes` creates
   147‚Üí  `FunctionEntry`/`FunctionReturn` nodes with `call_<name>` labels.
   148‚Üí
   149‚Üí#### 2. Pipeline side (`src/pipeline.rs`)
   150‚Üí- **`extract_extern_functions(ast)`**: New function that extracts the set of
   151‚Üí  extern function names from the AST by iterating over `Item::ExternBlock`
   152‚Üí  entries.
   153‚Üí- **`bridge_scg_to_codegen_with_externs()`**: New function that accepts an
   154‚Üí  `extern_functions: &HashSet<String>` parameter. The original
   155‚Üí  `bridge_scg_to_codegen()` now delegates to it with an empty set for backward
   156‚Üí  compatibility.
   157‚Üí- **Phase 0 ‚Äî Call-site detection**: The bridge now identifies call-site
   158‚Üí  `FunctionEntry` nodes (labels prefixed with `call_`) and excludes them from
   159‚Üí  Phase 1 function boundary detection. Previously, these were incorrectly
   160‚Üí  treated as function definitions, creating spurious empty functions.
   161‚Üí- **`walk_control_flow_with_externs()`**: New function that propagates the
   162‚Üí  extern function set through the control-flow walk. When encountering a
   163‚Üí  call-site `FunctionEntry` node (label `call_<name>`), it generates a
   164‚Üí  `CallNode { is_extern: true/false }` statement based on whether the target
   165‚Üí  function is in the extern set.
   166‚Üí- **`convert_node_to_statement_with_externs()`**: New function that checks
   167‚Üí  extern function membership for `Effect` nodes (which can represent function
   168‚Üí  calls) and sets `is_extern` accordingly.
   169‚Üí- All three compilation entry points (`compile_with_path`,
   170‚Üí  `compile_with_recovery`, `compile_to_wasm`) now extract extern functions
   171‚Üí  from the AST and pass them to the bridge.
   172‚Üí
   173‚Üí#### 3. IR side (`src/codegen/src/ir.rs`)
   174‚Üí- No changes needed ‚Äî `IRInstr::Call` already has an `is_extern: bool` field
   175‚Üí  that was correctly propagated from `CallNode.is_extern` through the
   176‚Üí  SCG‚ÜíIR conversion in `scg_to_ir.rs`.
   177‚Üí
   178‚Üí#### 4. Codegen side (`src/codegen/src/emit.rs`)
   179‚Üí- **`is_extern` flag**: The ARM64 emitter now captures the `is_extern` flag
   180‚Üí  from `IRInstr::Call` (previously ignored with `is_extern: _`). Extern
   181‚Üí  calls use the same `R_AARCH64_CALL26` relocation type ‚Äî the distinction
   182‚Üí  is in the ELF symbol table (undefined vs. defined).
   183‚Üí- **Symbol table**: External symbols are now marked as `STT_FUNC` instead
   184‚Üí  of `STT_NOTYPE`, allowing the linker to properly set up PLT entries and
   185‚Üí  enforce calling-convention compatibility.
   186‚Üí- **`resolve_call_relocs()`**: Updated to use `log::debug!` instead of
   187‚Üí  `log::warn!` for unresolved symbols, since extern functions are expected
   188‚Üí  to be undefined in the current compilation unit and will be resolved by
   189‚Üí  the system linker.
   190‚Üí
   191‚Üí#### 5. FFI module (`src/ffi.rs`)
   192‚Üí- Added `HashSet` import.
   193‚Üí- Added `ExternRegistry::function_name_set()` method returning a `HashSet<String>`.
   194‚Üí- Added `ExternBlock::to_registry()` conversion method.
   195‚Üí- Added `ExternBlock::function_names_set()` method for direct name extraction.
   196‚Üí
   197‚Üí#### 6. API fix (`src/api.rs`)
   198‚Üí- Fixed pre-existing compile error in `with_related()` call (was passing
   199‚Üí  `Vec<RelatedInfo>` instead of `RelatedInfo`).
   200‚Üí
   201‚Üí#### 7. Example (`examples/ffi_demo.vuma`)
   202‚Üí- Added FFI demonstration example showing `extern "C"` block syntax with
   203‚Üí  `write`, `read`, and `exit` declarations.
   204‚Üí
   205‚Üí### Architecture
   206‚Üí
   207‚ÜíThe end-to-end FFI flow is now:
   208‚Üí
   209‚Üí```
   210‚ÜíVUMA Source
   211‚Üí  ‚îÇ
   212‚Üí  ‚ñº
   213‚ÜíParser ‚Üí AST (ExternBlockDef with ExternFnDecl)
   214‚Üí  ‚îÇ
   215‚Üí  ‚îú‚îÄ‚Üí to_scg: creates Phantom node for extern block
   216‚Üí  ‚îÇ              creates FunctionEntry("call_write") for calls
   217‚Üí  ‚îÇ
   218‚Üí  ‚ñº
   219‚ÜíPipeline:
   220‚Üí  ‚îÇ  extract_extern_functions(ast) ‚Üí {"write", "read", "exit"}
   221‚Üí  ‚îÇ  bridge_scg_to_codegen_with_externs(scg, extern_fns)
   222‚Üí  ‚îÇ    ‚îú‚îÄ Phase 0: identify call-site FunctionEntry nodes
   223‚Üí  ‚îÇ    ‚îú‚îÄ Phase 1: skip call-site entries in function boundary detection
   224‚Üí  ‚îÇ    ‚îî‚îÄ walk_control_flow: convert call-site entries ‚Üí CallNode(is_extern=true)
   225‚Üí  ‚îÇ
   226‚Üí  ‚ñº
   227‚ÜíIR: IRInstr::Call { func: "write", is_extern: true }
   228‚Üí  ‚îÇ
   229‚Üí  ‚ñº
   230‚ÜíARM64 Emitter: BL with R_AARCH64_CALL26 relocation
   231‚Üí  ‚îÇ
   232‚Üí  ‚ñº
   233‚ÜíELF (ET_REL):
   234‚Üí  ‚îú‚îÄ .text: BL instruction (offset 0, to be resolved)
   235‚Üí  ‚îú‚îÄ .rela.text: R_AARCH64_CALL26 entry pointing to symbol
   236‚Üí  ‚îú‚îÄ .symtab: STB_GLOBAL | STT_FUNC, SHN_UNDEF for "write"
   237‚Üí  ‚îî‚îÄ System linker resolves: vuma.o ‚Üí ld -o out vuma.o -lc
   238‚Üí```
   239‚Üí

   248‚Üí## 2025-03-05 ‚Äî Task 4-g: Add DWARF debug info generation infrastructure
   249‚Üí
   250‚Üí### Summary
   251‚Üí
   252‚ÜíAdded comprehensive DWARF v4 debug info generation to the VUMA compiler,
   253‚Üíenabling GDB/LLDB to debug VUMA-compiled binaries. The four most important
   254‚ÜíDWARF sections are now generated: `.debug_abbrev`, `.debug_info`, `.debug_line`,
   255‚Üíand `.debug_frame`. The existing `--debug` CLI flag now properly triggers
   256‚ÜíDWARF emission through the full pipeline.
   257‚Üí
   258‚Üí### Changes Made
   259‚Üí
   260‚Üí#### 1. DWARF module (`src/codegen/src/dwarf.rs`)
   261‚Üí
   262‚Üí**Version change**: Upgraded from DWARF v5 to DWARF v4 for wider tool support
   263‚Üí(GDB, LLDB, readelf on older systems). Updated `.debug_info` and `.debug_line`
   264‚Üíheaders to match DWARF v4 format.
   265‚Üí
   266‚Üí**New data structures**:
   267‚Üí- `SavedRegister` ‚Äî register saved in the prologue (DWARF reg number + CFA offset)
   268‚Üí- `FrameDescriptorEntry` (FDE) ‚Äî per-function call frame info
   269‚Üí- `CommonInformationEntry` (CIE) ‚Äî shared prologue pattern for all FDEs
   270‚Üí
   271‚Üí**New CIE/FDE methods**:
   272‚Üí- `set_cie_aarch64()` ‚Äî CIE for ARM64 (SP=x31, LR=x30, FP=x29)
   273‚Üí- `set_cie_x86_64()` ‚Äî CIE for x86_64 (RSP=r7, RBP=r6)
   274‚Üí- `set_cie_riscv64()` ‚Äî CIE for RISC-V 64 (SP=x2, RA=x1)
   275‚Üí- `set_cie_arm32()` ‚Äî CIE for ARM32 (SP=r13, LR=r14)
   276‚Üí- `set_cie_mips64()` ‚Äî CIE for MIPS64 (SP=$sp, RA=$ra)
   277‚Üí- `set_cie_ppc64()` ‚Äî CIE for PPC64 (SP=R1, LR=65)
   278‚Üí- `set_cie_loongarch64()` ‚Äî CIE for LoongArch64 (SP=r3, RA=r1)
   279‚Üí- `set_cie_for_backend(BackendKind)` ‚Äî auto-selects CIE based on target
   280‚Üí- `set_cie(CIE)` ‚Äî custom CIE configuration
   281‚Üí- `add_fde(name, initial_location, address_range)` ‚Äî records an FDE
   282‚Üí
   283‚Üí**New `.debug_frame` section** (`emit_debug_frame()`):
   284‚Üí- Generates a CIE with `DW_CFA_def_cfa` and `DW_CFA_offset` instructions
   285‚Üí- Generates one FDE per function with initial_location and address_range
   286‚Üí- Supports both 32-bit and 64-bit address sizes
   287‚Üí- If no CIE is configured, `.debug_frame` is empty (omitted from ELF)
   288‚Üí
   289‚Üí**Updated `DebugSections`**: Now includes `debug_frame: Vec<u8>` field
   290‚Üí
   291‚Üí**Updated `append_debug_sections_to_elf()`**: Now handles 4 debug sections
   292‚Üí(3 + `.debug_frame`), conditionally omitting `.debug_frame` if empty
   293‚Üí
   294‚Üí**DWARF v4 `.debug_info` header format**:
   295‚Üí```
   296‚Üíunit_length (4) + version (2) + debug_abbrev_offset (4) + address_size (1)
   297‚Üí```
   298‚Üí(Previously DWARF v5 had: unit_length + version + unit_type + address_size + debug_abbrev_offset)
   299‚Üí
   300‚Üí**DWARF v4 `.debug_line` header format**:
   301‚Üí- Removed `address_size` and `segment_selector_size` fields (v5 only)
   302‚Üí- Removed `max_ops_per_instruction` field (v5 only)
   303‚Üí- Switched from DWARF v5 directory/file format (entry_format_count + format +
   304‚Üí  count) to DWARF v4 null-terminated list format
   305‚Üí- File entries now use ULEB128 for dir_index, time, size (v4 format)
   306‚Üí
   307‚Üí**New tests** (4 added, 24 total):
   308‚Üí- Test 21: `.debug_frame` section with CIE and FDE

   660‚Üí### Test Design Principles
   661‚Üí
   662‚Üí- **Minimal IR construction**: Each test constructs the smallest possible
   663‚Üí  `IRFunction` that would trigger the original bug ‚Äî typically a single
   664‚Üí  instruction or terminator in a one-block function.
   665‚Üí- **Multi-backend coverage**: Bug 11 (FP conversion) tests all 8 backends
   666‚Üí  in a single test function, accepting `UnsupportedFeature` errors for
   667‚Üí  backends that haven't implemented FP yet (as long as they error
   668‚Üí  explicitly rather than silently producing wrong code).
   669‚Üí- **Opcode-level verification**: Most tests extract the `opcode` strings
   670‚Üí  from `AllocatedInstruction` records and check for the presence of
   671‚Üí  expected instruction names (e.g., "dbar", "ldrex", "cmpxchg", "beq",
   672‚Üí  "dsrlv", "scvtf") and the absence of buggy ones (e.g., "BREAK").
   673‚Üí- **Encoded-byte verification**: For bugs where the problem was zero-length
   674‚Üí  output (e.g., NOP for CtSelect/CtEq), tests verify that the total
   675‚Üí  encoded byte count exceeds a reasonable minimum.
   676‚Üí- **ELF-level verification**: Bug 13 verifies the ELF binary contains the
   677‚Üí  external symbol name string, ensuring the linker can resolve it.
