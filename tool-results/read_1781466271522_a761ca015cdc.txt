    20→## 1. Rust Coding Style (Beyond rustfmt)
    21→
    22→### 1.1 Base Standard
    23→
    24→All VUMA Rust code follows `rustfmt` with project-specific settings defined in
    25→`rustfmt.toml`. Run `cargo fmt` before every commit. The CI pipeline will reject
    26→any PR that does not pass `cargo fmt --check`.
    27→
    28→Project-specific `rustfmt.toml` settings:
    29→
    30→```toml
    31→max_width = 100        # Wider than default 100 — but doc comments are 80
    32→tab_spaces = 4
    33→edition = "2021"
    34→```
    35→
    36→Beyond `rustfmt` defaults, we enforce these additional rules:
    37→
    38→- **Maximum line length**: 100 characters for code, **80 characters for doc
    39→  comments**. Doc comments that exceed 80 characters must be wrapped. This
    40→  ensures readability in `cargo doc` output and terminal windows.
    41→- **Imports**: Group imports in the following order, separated by blank lines:
    42→  1. Standard library (`use std::...`)
    43→  2. External crates (`use serde::...`, `use petgraph::...`)
    44→  3. Internal crates (`use vuma_scg::...`, `use vuma_ive::...`)
    45→  4. Current crate (`use crate::...`)
    46→- **Trailing commas**: Always use trailing commas in multi-line constructs
    47→  (function args, struct fields, enum variants, match arms).
    48→- **Match arms**: Always use `{}` blocks for match arms, even single-expression
    49→  arms. This makes it easier to add debugging or logging later:
    50→
    51→  ```rust
    52→  // PREFERRED
    53→  match node_type {
    54→      NodeType::Computation => {
    55→          process_computation(node)
    56→      }
    57→      NodeType::Allocation => {
    58→          process_allocation(node)
    59→      }
    60→  }
    61→
    62→  // AVOID
    63→  match node_type {
    64→      NodeType::Computation => process_computation(node),
    65→      NodeType::Allocation => process_allocation(node),
    66→  }
    67→  ```
    68→
    69→- **Implicit returns**: Prefer implicit returns (omitting the semicolon on the
    70→  final expression) for short functions. Use explicit `return` only for early
    71→  returns.
    72→- **Struct literal formatting**: Use block formatting for struct literals when
    73→  any field has a long value or when there are more than two fields.
    74→- **Method chaining**: Break long method chains so that each method call is on
    75→  its own line, aligned under the dot:
    76→
    77→  ```rust
    78→  let result = scg
    79→      .nodes()
    80→      .filter(|n| n.node_type == NodeType::Access)
    81→      .map(|n| verify_access(n, msg))
    82→      .collect::<Vec<_>>();
    83→  ```
    84→
    85→### 1.2 Clippy
    86→
    87→All code must pass `cargo clippy` with no warnings. The CI pipeline runs clippy
    88→with `-D warnings`, so any clippy warning is treated as an error. The project
    89→uses `clippy.toml` with `cognitive-complexity-threshold = 50` — any function
    90→exceeding this threshold should be refactored into smaller functions.
    91→
    92→If a clippy lint is genuinely inappropriate for a specific case, add a targeted
    93→`#[allow(clippy::lint_name)]` with a comment explaining why:
    94→
    95→```rust
    96→// This match is exhaustive by construction; the wildcard is a safety net.
    97→#[allow(clippy::match_wildcard_for_single_variants)]
    98→match result {
    99→    VerificationStatus::ProvenSafe => { /* ... */ }
   100→    _ => { /* handle all other cases uniformly */ }
   101→}
   102→```
   103→
   104→### 1.3 Cognitive Complexity
   105→
   106→The project enforces a cognitive complexity threshold of 50 (configured in
   107→`clippy.toml`). Functions that exceed this threshold must be refactored. Common
   108→refactoring strategies:
   109→
   110→- Extract helper functions for nested conditionals
   111→- Use early returns to reduce nesting
   112→- Replace complex match arms with helper methods
   113→- Use the builder pattern for complex construction
   114→
   115→### 1.4 Type Safety and Newtypes
   116→
   117→Use newtype wrappers for domain-specific identifiers to prevent accidental
   118→confusion between different kinds of IDs. The project follows this pattern
   119→extensively:
   120→
   121→```rust
   122→/// Unique identifier for a node within the SCG.
   123→#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord, Hash, Serialize, Deserialize)]
   124→pub struct NodeId(pub u64);
   125→
   126→/// Unique identifier for an edge within the SCG.
   127→#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord, Hash, Serialize, Deserialize)]
   128→pub struct EdgeId(pub u64);
   129→
   130→/// Unique identifier for a region within the SCG.
   131→#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord, Hash, Serialize, Deserialize)]
   132→pub struct RegionId(pub u64);
   133→```
   134→
   135→This prevents bugs like accidentally passing an `EdgeId` where a `NodeId` is
   136→expected, which would compile fine with raw `u64` values.
   137→
   138→### 1.5 Derive Macros
   139→
   140→All data-carrying types should derive at minimum: `Debug`, `Clone`. For types
   141→that are used as map keys or in equality comparisons, also derive `PartialEq`,
   142→`Eq`, `Hash`. For types that need serialization, derive `Serialize`,
   143→`Deserialize` from serde.

   147→## 2. Error Handling Patterns
   148→
   149→### 2.1 Library Crates — Use `thiserror`
   150→
   151→All error types in library crates (`vuma-scg`, `vuma-ive`, `vuma-bd`,
   152→`vuma-codegen`, `vuma-vuma`, `vuma-proof`, `vuma-cor`, `vuma-parser`,
   153→`vuma-projection`, `vuma-pi5`, `vuma-std`) must use **`thiserror`** for error
   154→definitions. This ensures that error types are concrete, implement
   155→`std::error::Error`, and have useful `Display` implementations.
   156→
   157→```rust
   158→use thiserror::Error;
   159→
   160→#[derive(Error, Debug)]
   161→pub enum IveError {
   162→    #[error("liveness verification failed for access at {address:#x}")]
   163→    LivenessViolation {
   164→        address: u64,
   165→        path: Vec<String>,
   166→    },
   167→
   168→    #[error("exclusivity violation: concurrent write detected at {address:#x}")]
   169→    ExclusivityViolation {
   170→        address: u64,
   171→        thread_a: String,
   172→        thread_b: String,
   173→    },
   174→
   175→    #[error("verification debt limit exceeded: {pending} unverified properties")]
   176→    VerificationDebtExceeded { pending: usize },
   177→
   178→    #[error("encoding error: {0}")]
   179→    Encoding(#[from] crate::codegen::CodegenError),
   180→}
   181→```
   182→
   183→**Rules for error types:**
   184→
   185→- Use `#[error("...")]` with a human-readable message that includes all relevant
   186→  context (addresses, IDs, paths)
   187→- Use `#[from]` for automatic `From` impls when wrapping errors from other crates
   188→- Group related error variants together with a comment separator
   189→- Never use `Box<dyn std::error::Error>` in library code — always use a concrete
   190→  error enum
   191→- Never use `String` as an error type — always create a named variant with
   192→  structured fields
   193→
   194→### 2.2 Application Crates — Use `anyhow`
   195→
   196→Use **`anyhow`** for application-level error handling in CLI tools,
   197→integration tests, and example programs. This allows flexible error chaining
   198→and context attachment without defining exhaustive error enums.
   199→
   200→```rust
   201→use anyhow::{Context, Result};
   202→
   203→fn run_verification(scg: &SemanticComputationGraph) -> Result<()> {
   204→    let msg = scg
   205→        .build_memory_state_graph()
   206→        .context("failed to build Memory State Graph")?;
   207→    ive::verify_liveness(&msg)
   208→        .context("liveness verification failed")?;
   209→    Ok(())
   210→}
   211→```
   212→
   213→**Rules for `anyhow` usage:**
   214→
   215→- Always use `.context("...")` or `.with_context(|| ...)` to attach a
   216→  human-readable description of what was being attempted when the error occurred
   217→- Chain context for a causal trace: the outermost context should describe the
   218→  high-level operation, and inner contexts should describe the specific step
   219→- Use `anyhow::Result<T>` as the return type, not `std::result::Result<T, anyhow::Error>`
   220→
   221→### 2.3 No Panics in Library Code
   222→
   223→Library crates must **never panic** in production code. Use `Result` for all
   224→fallible operations. The only acceptable uses of `panic!`, `unwrap()`, or
   225→`expect()` are:
   226→
   227→1. **In test code** (`#[cfg(test)]` modules) — `unwrap()` and `expect()` are
   228→   encouraged in tests for clarity
   229→2. **For true programming errors** that indicate a bug in the caller — in this
   230→   case, use `assert_eq!`, `assert!`, or `panic!` with a message explaining
   231→   the invariant that was violated
   232→3. **In `unreachable!()` branches** after exhaustive matches where the compiler
   233→   cannot prove exhaustiveness
   234→4. **In `todo!()` markers** for unimplemented functionality (these are acceptable
   235→   during development but must not appear in released code)
   236→
   237→**Anti-patterns to avoid:**
   238→
   239→```rust
   240→// BAD: Silent unwrap that hides the failure mode
   241→let region = msg.region_of(address).unwrap();
   242→
   243→// BAD: Indexing that can panic
   244→let first = nodes[0];
   245→
   246→// GOOD: Explicit error handling
   247→let region = msg
   248→    .region_of(address)
   249→    .ok_or(IveError::RegionNotFound { address })?;
   250→
   251→// GOOD: Bounds-checked access
   252→let first = nodes
   253→    .first()
   254→    .ok_or(IveError::EmptyNodeList)?;
   255→```
   256→
   257→### 2.4 Error Propagation Across Crates
   258→
   259→When errors propagate across crate boundaries, wrap them using `#[from]` or
   260→explicit conversion:
   261→
   262→```rust
   263→// In vuma-ive (which depends on vuma-scg and vuma-codegen)
   264→#[derive(Error, Debug)]
   265→pub enum IveError {
   266→    #[error("SCG error: {0}")]
   267→    Scg(#[from] vuma_scg::ScgError),
   268→
   269→    #[error("codegen error: {0}")]
   270→    Codegen(#[from] vuma_codegen::CodegenError),
   271→}
   272→```
   273→
   274→Never leak internal error types through public APIs. Each crate should define
   275→its own error enum that wraps dependencies.

   279→## 3. Testing Conventions
   280→
   281→### 3.1 Test Categories
   282→
   283→VUMA distinguishes three categories of tests, each serving a different purpose:
   284→
   285→| Category | Location | Scope | Speed |
   286→|----------|----------|-------|-------|
   287→| **Unit** | `#[cfg(test)] mod tests` in each file | Single function or type | Fast (<1ms each) |
   288→| **Integration** | `src/tests/` workspace crate | Cross-crate workflows | Medium (<100ms each) |
   289→| **Verification** | IVE-specific test functions | Full verification pipeline | Slow (seconds each) |
   290→
   291→### 3.2 Unit Tests
   292→
   293→Unit tests live in the same file as the code they test, inside a
   294→`#[cfg(test)] mod tests` block at the bottom of the file. Unit tests should be
   295→focused, fast, and deterministic. Each test function name should describe the
   296→specific scenario being tested, following the pattern
   297→`{method}_{scenario}_{expected_outcome}`:
   298→
   299→```rust
   300→#[cfg(test)]
   301→mod tests {
   302→    use super::*;
   303→
   304→    #[test]
   305→    fn verify_liveness_catches_use_after_free() {
   306→        // Arrange: build a minimal MSG where an access targets a freed region
   307→        // Act: run liveness verification
   308→        // Assert: verification returns a LivenessViolation
   309→    }
   310→
   311→    #[test]
   312→    fn verify_liveness_accepts_valid_access() {
   313→        // Arrange: build a minimal MSG where all accesses target live regions
   314→        // Act: run liveness verification
   315→        // Assert: verification returns ProvenSafe
   316→    }
   317→
   318→    #[test]
   319→    fn encode_ldr_unsigned_offset_produces_correct_machine_code() {
   320→        let inst = Instruction::LDR {
   321→            rt: Register::X0,
   322→            rn: Register::X1,
   323→            offset: 16,
   324→        };
   325→        let encoded = inst.encode().unwrap();
   326→        assert_eq!(encoded, 0xF94080A0);
   327→    }
   328→}
   329→```
   330→
   331→**Unit test rules:**
   332→
   333→- Use the Arrange-Act-Assert pattern (or Given-When-Then)
   334→- Do not depend on external state, network, or filesystem
   335→- Do not depend on execution order — each test must be independently runnable
   336→- Avoid test interdependence; use fresh data structures for each test
   337→- Prefer specific assertions over vague ones:
   338→  - `assert_eq!(result.status, VerificationStatus::ProvenSafe)` ✓
   339→  - `assert!(result.is_ok())` — too vague, use only when the specific value
   340→    is not important
   341→
   342→### 3.3 Integration Tests
   343→
   344→Integration tests live in the `src/tests/` workspace crate (`vuma-tests`) for
   345→cross-crate integration tests. Individual crates may also have a `tests/`
   346→directory for crate-level integration tests.
   347→
   348→Integration tests exercise complete workflows across crate boundaries:
   349→
   350→```rust
   351→// src/tests/src/dlist.rs — Integration test for doubly-linked list verification
   352→#[test]
   353→fn dlist_full_pipeline_verification_passes() {
   354→    // 1. Parse the example program
   355→    let scg = build_example_scg("doubly_linked_list");
   356→
   357→    // 2. Convert SCG → MSG
   358→    let msg = vuma_core::scg_to_msg::scg_to_msg(&scg).unwrap();
   359→
   360→    // 3. Run all IVE verification passes
   361→    let result = vuma_ive::InvariantAggregator::new().verify_all(&msg);
   362→
   363→    // 4. Assert all invariants pass
   364→    assert!(result.all_passed(), "Verification failures: {:?}", result.failures());
   365→}
   366→```
   367→
   368→**Integration test rules:**
   369→
   370→- Test complete workflows, not individual functions
   371→- Cover the SCG → MSG → IVE → Codegen pipeline end-to-end
   372→- Include both positive tests (correct programs pass verification) and negative
   373→  tests (incorrect programs fail verification with expected diagnostics)
   374→- Run under `--test-threads=1` when tests share global state
   375→
   376→### 3.4 Verification Tests
   377→
   378→Verification tests are IVE-specific tests that exercise the full verification
   379→pipeline. They are slower than unit tests and may involve constructing large
   380→MSG instances. These tests are tagged with `verification` for selective running:
   381→
   382→```bash
   383→# Run only verification tests
   384→cargo test -p vuma-ive -- --test-threads=1 -- verification
   385→
   386→# Run with verbose output
   387→cargo test -p vuma-ive -- --nocapture -- verification
   388→```
   389→
   390→**Verification test rules:**
   391→
   392→- Always run with `--test-threads=1` to avoid race conditions in shared state
   393→- Test both the positive case (verification passes for correct programs) and
   394→  the negative case (verification fails with actionable counterexamples for
   395→  incorrect programs)
   396→- Test each of the five invariants independently
   397→- Test interactions between invariants (e.g., a program that passes liveness
   398→  but fails exclusivity)
   399→
   400→### 3.5 Property-Based Testing
   401→
   402→For verification-related code, prefer property-based tests using the `proptest`
   403→crate. This is especially important for the IVE, where edge cases in pointer
   404→arithmetic and graph traversal are common:
   405→
   406→```rust
   407→use proptest::prelude::*;
   408→
   409→proptest! {
   410→    #[test]
   411→    fn address_arithmetic_stays_in_region(
   412→        base in 0x1000u64..0x10000u64,
   413→        offset in 0u64..0x1000u64,
   414→    ) {
   415→        let region = Region::new(RegionId::new(1), Address::from(base), 0x1000);
   416→        let addr = Address::from(base) + offset;
   417→        prop_assert!(region.contains(&addr));
   418→    }
   419→}
   420→```
   421→
   422→### 3.6 Test Naming Convention
   423→
   424→Test functions use `snake_case` and follow the pattern:
   425→`{unit}_{scenario}_{expected_outcome}`
   426→
   427→Examples:
   428→
   429→- `verify_liveness_valid_access_returns_ok`
   430→- `verify_liveness_freed_region_returns_violation`
   431→- `codegen_ldxr_stxr_produces_valid_atomic`
   432→- `encode_add_shifted_register_matches_arm_spec`
   433→- `scg_to_msg_allocation_creates_region_with_monotonic_address`
   434→- `bd_compatible_same_type_returns_true`
   435→- `proof_checker_detects_circular_reasoning`

   439→## 4. Naming Conventions for VUMA Types
   440→
   441→### 4.1 General Rust Naming
   442→
   443→| Item | Convention | Example |
   444→|------|-----------|---------|
   445→| Crates | `snake_case` | `vuma_scg`, `vuma_ive`, `vuma_codegen` |
   446→| Modules | `snake_case` | `memory_model`, `graph_transform` |
   447→| Types (struct, enum, trait) | `CamelCase` | `SemanticComputationGraph`, `BehavioralDescriptor` |
   448→| Enum variants | `CamelCase` | `LivenessViolation`, `ExclusivityViolation` |
   449→| Functions / methods | `snake_case` | `verify_liveness`, `infer_capabilities` |
   450→| Constants | `SCREAMING_SNAKE_CASE` | `MAX_VERIFICATION_DEPTH`, `DEFAULT_REGION_SIZE` |
   451→| Static variables | `SCREAMING_SNAKE_CASE` | `GLOBAL_IVE_CONFIG` |
   452→| Type parameters | Short `CamelCase` | `T`, `N` for common; `NodeKind`, `RepDescriptor` for domain |
   453→| Lifetime parameters | Short lowercase | `'a`, `'ctx`, `'graph` |
   454→| Macros | `snake_case!` | `scg_node!`, `define_bd!` |
   455→
   456→### 4.2 VUMA-Specific Naming Conventions
   457→
   458→These conventions apply to types and functions specific to the VUMA domain:
   459→
   460→**SCG Node Types**: Named as `{Kind}Node` — the `Node` suffix is mandatory for
   461→all node payload types in the SCG:
   462→
   463→- `AllocationNode` — memory allocation operation
   464→- `DeallocationNode` — memory deallocation operation
   465→- `AccessNode` — memory read/write operation
   466→- `ComputationNode` — pure computation
   467→- `CastNode` — type conversion or coercion
   468→- `EffectNode` — side-effecting operation
   469→- `ControlNode` — control flow point
   470→- `PhantomNode` — structural/analysis placeholder
   471→
   472→**BD Components**: Named as `{DescriptorKind}Descriptor` in public APIs. The
   473→abbreviations `RepD`, `CapD`, `RelD` are acceptable in doc comments, local
   474→variables, and internal discussion, but **never** in public API names:
   475→
   476→```rust
   477→// Public API: use full names
   478→pub struct RepresentationDescriptor { /* ... */ }
   479→pub struct CapabilityDescriptor { /* ... */ }
   480→pub struct RelationalDescriptor { /* ... */ }
   481→
   482→// Doc comments and local variables: abbreviations OK
   483→/// Computes the RepD (Representation Descriptor) for a pointer type.
   484→fn compute_ptr_repd(pointee: &RepresentationDescriptor) -> RepresentationDescriptor {
   485→    // ...
   486→}
   487→```
   488→
   489→**Verification Result Types**: Named as `{Property}Result` or `{Property}Report`:
   490→
   491→- `LivenessResult` / `LivenessReport`
   492→- `ExclusivityResult` / `ExclusivityReport`
   493→- `InterpretationResult` / `InterpretationReport`
   494→- `VerificationSummary` — aggregated result from all passes
   495→- `CounterExample` — structured failure description
   496→
   497→**Verification Violation Types**: Named as `{Property}Violation`:
   498→
   499→- `LivenessViolation`
   500→- `ExclusivityViolation`
   501→- `InterpretationViolation`
   502→- `OriginViolation`
   503→- `CleanupViolation`
   504→
   505→**Error Types**: Named as `{Domain}Error` — the `Error` suffix is mandatory:
   506→
   507→- `ScgError` — errors in SCG construction or validation
   508→- `IveError` — errors in IVE verification or inference
   509→- `CodegenError` — errors in ARM64 code generation
   510→- `ConversionError` — errors in SCG → MSG conversion
   511→- `ProofError` — errors in proof construction or checking
   512→
   513→**Verifier Types**: Named as `{Property}Verifier`:
   514→
   515→- `LivenessVerifier`
   516→- `ExclusivityVerifier`
   517→- `InterpretationVerifier`
   518→- `CleanupVerifier`
   519→- `AlignmentVerifier`
   520→
   521→**ARM64 Instruction Types**: Named exactly as the ARM64 mnemonic in `CamelCase`:
   522→
   523→- `Ldxr` — Load Exclusive Register
   524→- `Stxr` — Store Exclusive Register
   525→- `Dmb` — Data Memory Barrier
   526→- `Dsb` — Data Synchronization Barrier
   527→- `Isb` — Instruction Synchronization Barrier
   528→- `Cas` — Compare and Swap
   529→
   530→**Identifier Newtypes**: Named as `{Entity}Id` — the `Id` suffix is mandatory:
   531→
   532→- `NodeId` — unique node identifier in the SCG
   533→- `EdgeId` — unique edge identifier in the SCG
   534→- `RegionId` — unique region identifier
   535→- `AccessId` — unique access identifier
   536→- `DerivationId` — unique derivation identifier
   537→- `SyncEdgeId` — unique synchronization edge identifier
   538→
   539→### 4.3 Function Naming Patterns
   540→
   541→| Pattern | Meaning | Example |
   542→|---------|---------|---------|
   543→| `verify_{property}` | Run a verification pass | `verify_liveness`, `verify_exclusivity` |
   544→| `infer_{property}` | Infer a property from structure | `infer_capabilities`, `infer_representation` |
   545→| `build_{entity}` | Construct a complex object | `build_memory_state_graph`, `build_derivation_chain` |
   546→| `compute_{value}` | Derive a computed value | `compute_dominators`, `compute_hot_paths` |
   547→| `find_{entities}` | Search and return results | `find_use_after_free`, `find_derivation_chains` |
   548→| `emit_{output}` | Generate output (codegen) | `emit_atomic_swap`, `emit_barrier` |
   549→| `lower_{source}` | Lower to a lower-level representation | `lower_to_arm64` |
   550→| `encode` | Binary encode an instruction | `instruction.encode()` |
   551→| `as_{type}` | Cheap reference conversion | `as_reg()`, `as_u64()` |
   552→| `to_{type}` | Expensive owned conversion | `to_string()`, `to_scg()` |
   553→| `is_{predicate}` | Boolean predicate | `is_callee_saved()`, `is_null()`, `is_valid()` |
   554→| `has_{property}` | Boolean property check | `has_capability()`, `has_snapshot()` |
   555→

   558→## 5. Documentation Conventions
   559→
   560→### 5.1 Doc Comments Are Mandatory
   561→
   562→**All public items** (functions, types, traits, modules, constants) must have
   563→doc comments (`///` or `//!`). Doc comments must include:
   564→
   565→1. **A summary line** (imperative mood, no trailing period):
   566→   `/// Verify liveness of all memory accesses`
   567→2. **A detailed description** (one or more paragraphs explaining the what,
   568→   why, and how)
   569→3. **At least one example** for functions and methods showing typical usage
   570→4. **Panics** section if the function can panic (explain under what conditions)
   571→5. **Errors** section if the function returns `Result` (describe what errors
   572→   can be returned)
   573→6. **Safety** section for any `// VUMA-VERIFIED` code (explain what the IVE
   574→   verified)
   575→
   576→Example:
   577→
   578→```rust
   579→/// Verify liveness of all memory accesses in the Memory State Graph
   580→///
   581→/// Traverses every access node in the MSG and checks that the target region
   582→/// is allocated at the program point where the access occurs. This is the
   583→/// first of the five VUMA global invariants (see GLOSSARY.md for details).
   584→///
   585→/// The verification algorithm performs a forward dataflow analysis over the
   586→/// MSG, tracking the allocation status of each region at every program point.
   587→/// Time complexity is O(N + E) where N is the number of nodes and E is the
   588→/// number of edges in the MSG.
   589→///
   590→/// # Examples
   591→///
   592→/// ```
   593→/// use vuma_ive::verify_liveness;
   594→/// use vuma_core::msg::MSG;
   595→///
   596→/// let msg = MSG::from_scg(&scg);
   597→/// let result = verify_liveness(&msg);
   598→/// assert!(result.is_ok(), "all accesses target live memory");
   599→/// ```
   600→///
   601→/// # Errors
   602→///
   603→/// Returns `IveError::LivenessViolation` if any access targets a freed
   604→/// or unallocated region, with the offending address and execution path.
   605→pub fn verify_liveness(msg: &MSG) -> Result<LivenessReport, IveError> {
   606→    // ...
   607→}
   608→```
   609→
   610→### 5.2 Module-Level Documentation
   611→
   612→Every crate must have a `//!` module-level doc comment in its `lib.rs` (or
   613→`main.rs`) that explains:
   614→
   615→1. What the crate does
   616→2. How it fits into the VUMA architecture
   617→3. Key types and their relationships
   618→4. Links to related crates
   619→5. A quick-start code example
   620→
   621→Example (from `src/scg/src/lib.rs`):
   622→
   623→```rust
   624→//! # VUMA SCG — Semantic Computation Graph
   625→//!
   626→//! This crate provides the core data structures and algorithms for the
   627→//! **Semantic Computation Graph (SCG)**, a central component of the VUMA
   628→//! framework for verified-unsafe memory access.
   629→//!
   630→//! ## Overview
   631→//!
   632→//! The SCG models program semantics as a directed graph where:
   633→//! - **Nodes** represent operations (computation, allocation, …)
   634→//! - **Edges** represent relationships between operations
   635→//! - **Regions** group nodes into memory scopes
   636→//!
   637→//! ## Quick Start
   638→//!
   639→//! ```
   640→//! use vuma_scg::SCG;
   641→//! let mut scg = SCG::new();
   642→//! ```
   643→```
   644→
   645→### 5.3 Internal Documentation
   646→
   647→Private functions and types should also have doc comments when their purpose
   648→is not immediately obvious from the name. A good rule of thumb: if you would
   649→need to read the function body to understand what it does, it needs a doc
   650→comment.
   651→
   652→### 5.4 Doc Comment Line Length
   653→
   654→Doc comments must not exceed **80 characters** per line. This ensures
   655→readability in `cargo doc` output, terminal windows, and code review tools.
   656→Break lines at natural boundaries (between sentences, after commas):
   657→
   658→```rust
   659→/// Verify that every memory access targets a region that is allocated at
   660→/// that program point, eliminating use-after-free bugs by proving that no
   661→/// execution path leads to a dereference of a freed or unallocated region.
   662→```
   663→
   664→### 5.5 Code Examples in Doc Comments
   665→
   666→All code examples in doc comments must be valid Rust code that compiles and
   667→runs. Use ````rust` fences for runnable examples and ````text` or ````asm`
   668→for non-Rust output (assembly, hex dumps, etc.).
   669→
   670→Prefer showing the common case over the edge case. Include assertions that
   671→verify the expected behavior:
   672→
   673→```rust
   674→/// # Examples
   675→///
   676→/// ```
   677→/// use vuma_bd::descriptor::BD;
   678→/// use vuma_bd::repd::RepD;
   679→/// use vuma_bd::capd::CapD;
   680→///
   681→/// let bd = BD::new(RepD::u64(), CapD::read_write(), RelD::empty());
   682→/// assert!(bd.capd().can_read());
   683→/// assert!(bd.capd().can_write());
   684→/// ```
   685→```

   689→## 6. Git Commit Message Format
   690→
   691→### 6.1 Conventional Commits
   692→
   693→All commit messages must follow the [Conventional Commits](https://www.conventionalcommits.org/)
   694→specification:
   695→
   696→```
   697→<type>(<scope>): <description>
   698→
   699→[optional body]
   700→
   701→[optional footer]
   702→```
   703→
   704→**Format rules:**
   705→
   706→- The **type** and **scope** are lowercase
   707→- The **description** is in imperative mood, lowercase, no trailing period
   708→  ("add liveness pass" not "Added liveness pass")
   709→- The **body** (if present) is separated from the description by a blank line
   710→  and explains the *why* of the change, not the *what*
   711→- The **footer** (if present) contains breaking change notices and issue
   712→  references: `Closes #42`, `BREAKING CHANGE: ...`
   713→
   714→### 6.2 Types
   715→
   716→| Type | Meaning | Example |
   717→|------|---------|---------|
   718→| `feat` | New feature | `feat(ive): add liveness verification pass` |
   719→| `fix` | Bug fix | `fix(codegen): correct LDXR/STXR register allocation` |
   720→| `docs` | Documentation only | `docs(glossary): add ARM64 instruction terms` |
   721→| `test` | Adding or updating tests | `test(ive): add property tests for exclusivity` |
   722→| `refactor` | Code restructuring (no behavior change) | `refactor(scg): extract node visitor trait` |
   723→| `perf` | Performance improvement | `perf(msg): use arena allocation for derivation chains` |
   724→| `chore` | Maintenance, CI, tooling | `chore(ci): update clippy lint configuration` |
   725→| `style` | Formatting only (no logic change) | `style: apply rustfmt` |
   726→| `build` | Build system changes | `build(pi5): add bare-metal linker script` |
   727→| `ci` | CI/CD configuration | `ci: add QEMU aarch64 test job` |
   728→
   729→### 6.3 Scopes
   730→
   731→Use the crate name as the scope. The valid scopes are:
   732→
   733→| Scope | Crate |
   734→|-------|-------|
   735→| `scg` | `vuma-scg` |
   736→| `ive` | `vuma-ive` |
   737→| `vuma` | `vuma` (core) |
   738→| `bd` | `vuma-bd` |
   739→| `cor` | `vuma-cor` |
   740→| `projection` | `vuma-projection` |
   741→| `parser` | `vuma-parser` |
   742→| `codegen` | `vuma-codegen` |
   743→| `pi5` | `vuma-pi5` |
   744→| `std` | `vuma-std` |
   745→| `proof` | `vuma-proof` |
   746→| `tests` | `vuma-tests` |
   747→| `docs` | Documentation (cross-cutting) |
   748→| `ci` | CI/CD configuration |
   749→| *(none)* | Changes affecting the entire workspace |
   750→
   751→### 6.4 Examples
   752→
   753→```
   754→feat(ive): add alignment verification pass
   755→
   756→Implement the alignment invariant verifier that checks every memory
   757→access targets an address satisfying the alignment requirement of the
   758→access's RepD. The pass integrates into the InvariantAggregator as the
   759→sixth verification pass.
   760→
   761→Closes #127
   762→```
   763→
   764→```
   765→fix(codegen): correct STXR encoding status register position
   766→
   767→The STXR instruction encoding was placing the status register (Rs) in
   768→the wrong bit position (bits 20:16 instead of bits 15:10). This caused
   769→atomic compare-and-swap loops to never observe success.
   770→
   771→Fixes #98
   772→```
   773→
   774→```
   775→refactor(scg): extract SCGPass trait from transform module
   776→
   777→Move the SCGPass trait definition into its own file and add a
   778→PassManager that runs passes in dependency order. No behavior change.
   779→```
   780→
   781→### 6.5 Branch Naming
   782→
   783→- Feature branches: `feat/<description>` — e.g., `feat/ive-liveness-pass`
   784→- Fix branches: `fix/<description>` — e.g., `fix/codegen-ldxr-register`
   785→- All branches are created from `main` and merged back via PR
   786→
   787→### 6.6 PR Titles
   788→
   789→PR titles follow the same conventional commit format as commit messages.
   790→Squash-merged PRs produce a single commit whose message is the PR title.
   791→

   793→
   794→*These conventions are enforced by CI and code review. When in doubt, follow
   795→the principle of making the code as easy as possible for the IVE to reason
   796→about — explicit, well-structured, and thoroughly documented.*
