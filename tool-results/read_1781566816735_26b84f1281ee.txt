    30→## Overview
    31→
    32→VUMA is a programming language framework built on a radical premise: **unsafe memory operations should not be forbidden — they should be made verifiable.** Instead of relying on a borrow checker to reject programs that cannot be statically proven safe, VUMA constructs a formal model of every memory operation and verifies global invariants against that model. Programs that pass verification run without runtime overhead; programs that fail receive precise counterexamples showing the exact execution path to the violation.
    33→
    34→### Why VUMA?
    35→
    36→Traditional languages face a fundamental trade-off: either restrict what programmers can express (Rust's borrow checker, Java's GC) or accept memory-unsafe programs (C, C++). VUMA breaks this trade-off through three innovations:
    37→
    38→1. **Semantic Computation Graphs (SCGs)** replace source code as the primary program representation. The SCG is a directed, acyclic, attributed multigraph where nodes are operations, edges are relationships, and regions delineate scopes and deployment targets.
    39→
    40→2. **Behavioral Descriptors (BDs)** replace nominal types. A BD is the triple (RepD, CapD, RelD) — representation, capabilities, and relationships — inferred from program structure rather than declared by the programmer.
    41→
    42→3. **Verification over restriction** — instead of rejecting programs that might be unsafe, VUMA verifies that they are safe and provides precise diagnostics when they are not.
    43→
    44→### The Five VUMA Invariants
    45→
    46→Every VUMA program is verified against five global memory-safety invariants:
    47→
    48→| Invariant       | Ensures                                            |
    49→|-----------------|----------------------------------------------------|
    50→| **Liveness**    | Every access targets allocated memory              |
    51→| **Exclusivity** | No conflicting concurrent accesses                 |
    52→| **Interpretation** | Every access uses a valid representation         |
    53→| **Origin**      | Every address traces to a valid allocation         |
    54→| **Cleanup**     | Every region is eventually freed or explicitly leaked |

    58→## Features
    59→
    60→### Multi-Architecture Codegen (8 Backends)
    61→
    62→VUMA compiles to **8 CPU/platform targets** with a unified backend trait architecture:
    63→
    64→| Backend | Architecture | Status | ABI |
    65→|---------|-------------|--------|-----|
    66→| x86_64 | x86-64 (SysV) | ✅ Stable | System V AMD64 |
    67→| AArch64 | ARM64 (AAPCS64) | ✅ Stable | AAPCS64 |
    68→| RISC-V 64 | RV64G | ✅ Stable | LP64D |
    69→| ARM32 | ARMv7-A (AAPCS) | ✅ Stable | AAPCS |
    70→| MIPS64 | MIPS III (N64) | ✅ Stable | N64 |
    71→| PPC64 | PowerPC v2 (ELFv2) | ✅ Stable | ELFv2 |
    72→| LoongArch64 | LA64 | 🔄 Experimental | LP64 |
    73→| Wasm32 | WebAssembly | 🔄 Experimental | Stack machine |
    74→
    75→All 6 native backends pass the full SHA256d execution test. Wasm32 generates valid modules. LoongArch64 passes individual operation tests.
    76→
    77→### AI-Native Design
    78→
    79→VUMA is designed from the ground up for programmatic consumption by AI agents:
    80→
    81→- **VumaForLLM API** — stateless `compile()`, `check()`, `analyze()`, `to_wasm()`, `explain_error()`, `suggest_fixes()`
    82→- **LSP Server** — full protocol: diagnostics, hover, go-to-definition, completion, semantic tokens
    83→- **Enhanced REPL** — `:wasm`, `:backends`, `:check`, `:diagnostics`, `:exports`, tab completion, color output
    84→- **Wasm32 Sandbox** — LLM agents compile to safe, sandboxed WebAssembly modules
    85→- **65 Diagnostic Codes** — E001-E050, W001-W010, I001-I005 with error chaining and structured suggestions
    86→
    87→### Memory Safety Verification
    88→
    89→- **10 violation types** detected at compile time: UseAfterFree, DoubleFree, MemoryLeak, BoundsCheckFailure, NullDereference, DanglingPointer, UninitializedRead, BufferOverflow, UseAfterScope, InvalidFree
    90→- **Runtime bounds checking** behind `--safe` flag
    91→- **Constant-time crypto**: Branchless `ct_select`, `ct_eq`, `ct_ne`, `ct_lt`, `ct_gte` across all 8 backends
    92→
    93→### Module System & Package Manager
    94→
    95→- **Multi-file compilation**: `import "crypto.vuma"::{sha256, sha256d};` with circular import detection
    96→- **Package manager**: `vuma pkg init`, `vuma pkg build`, `vuma pkg add` with dependency resolution
    97→
    98→### FFI & System Calls
    99→
   100→- **19 Linux syscalls** across all 8 architectures (read, write, open, close, exit, mmap, munmap, brk, ioctl, fcntl, getpid, clone, futex, etc.)

   102→- **`extern "C" { fn ...; }`** FFI blocks with is_extern flag propagation

   106→## Architecture
   107→
   108→VUMA implements a six-layer architecture where data flows from human intent through graph construction, inference, verification, code generation, and into execution on bare metal. Feedback flows upward at every stage.
   109→
   110→```
   111→┌─────────────────────────────────────────────────────────────────────┐
   112→│                    LLM Integration Layer                             │
   113→│    VumaForLLM API · LSP Server · REPL · Structured Diagnostics     │
   114→├─────────────────────────────────────────────────────────────────────┤
   115→│                    Layer 3 — Projection System                       │
   116→│    Textual · Visual · Conversational · Bidirectional · Diff         │
   117→├─────────────────────────────────────────────────────────────────────┤
   118→│                    Parser / Frontend (auxiliary)                     │
   119→│    Lexer → Parser → AST → AST-to-SCG Lowering · Module Resolution  │
   120→├─────────────────────────────────────────────────────────────────────┤
   121→│             Layers 2, 5, 6 — Reasoning Core                         │
   122→│    IVE (Inference + Verification) · BD (Descriptors) · MSG (Memory) │
   123→│    Invariants: Liveness · Exclusivity · Interpretation ·             │
   124→│                Origin · Cleanup                                      │
   125→├─────────────────────────────────────────────────────────────────────┤
   126→│                    Layer 1 — SCG (Core Representation)               │
   127→│    Nodes (ops, allocs, effects) · Edges (data flow, deps) · Regions │
   128→├─────────────────────────────────────────────────────────────────────┤
   129→│                    Layer 4 — Execution                               │
   130→│    COR Runtime (always-compiled, PGO, JIT) · Multi-ISA Codegen      │
   131→│    x86_64 · AArch64 · RISC-V 64 · ARM32 · MIPS64 · PPC64           │
   132→│    LoongArch64 · Wasm32                                             │
   133→└─────────────────────────────────────────────────────────────────────┘
   134→```
   135→
   136→### Pipeline
   137→
   138→```
   139→Source Text → Lexer → Parser → AST → SCG Lowering → Raw SCG
   140→    → Module Resolution (imports) → Merged SCG
   141→    → BD Inference (RepD + CapD + RelD fixpoint) → Annotated SCG
   142→    → MSG Builder → VUMA Verification (5 invariants) → Verified SCG
   143→    → Multi-Arch Codegen (IR → regalloc → emit) → Machine Code / Wasm
   144→    → COR Runtime → Execution → Profile Feedback → re-optimize
   145→```

   177→## VUMA for LLMs
   178→
   179→VUMA is designed as an **AI-native** language framework. Every interface is designed for programmatic consumption by AI agents. This LLM-first design allows AI coding agents to use VUMA as a verified compilation sandbox: parse source, analyze structure, run verification, compile to any of 8 backends, and inspect results — all through clean API boundaries.
   180→
   181→### LLM-Facing Interfaces
   182→
   183→| Interface | Description | Use Case |
   184→|-----------|-------------|----------|
   185→| `VumaForLLM` API | `compile()`, `check()`, `analyze()`, `to_wasm()`, `explain_error()`, `suggest_fixes()` | LLM compiles and verifies code in a sandbox |
   186→| `VumaCompiler` API | `compile()`, `parse()`, `analyze()`, `validate()`, `verify()` | Full pipeline with verification reports |
   187→| LSP Server | Full LSP protocol: diagnostics, hover, go-to-definition, completion | IDE integration and LLM agent interaction |
   188→| REPL `:wasm` | Compile to Wasm and show binary size | Quick Wasm compilation check |
   189→| REPL `:backends` | List 8 backends with status | Discover compilation capabilities |
   190→| REPL `:check` | Run IVE verification | Instant verification feedback |
   191→| REPL `:diagnostics` | Show all diagnostics as JSON | Structured error analysis |
   192→| REPL `:exports` | List all functions and signatures | Program structure inspection |
   193→
   194→### Wasm32 Sandbox for LLM Agents
   195→
   196→The Wasm32 backend enables LLM agents to compile VUMA programs into safe, sandboxed WebAssembly modules. This is the recommended execution path for LLM-generated code: the Wasm module runs in a sandboxed environment with no access to host memory or peripherals, ensuring that LLM-generated code cannot cause harm.
   197→
   198→```rust
   199→// LLM agent compiles VUMA to Wasm
   200→let result = VumaForLLM::compile(source);
   201→if let Some(wasm) = &result.wasm_binary {
   202→    println!("Wasm module: {} bytes", wasm.len());
   203→}
   204→```
   205→
   206→### Parser Hardening for LLM-Generated Code
   207→
   208→The VUMA parser is specifically hardened for code generated by LLMs:
   209→
   210→- **LLM type aliases**: `int` → `i32`, `float` → `f32`, `double` → `f64`, `String` → `string`
   211→- **C/Rust construct detection**: `println!`, `vec!`, `format!`, `panic!` produce helpful errors
   212→- **C-style for loop detection**: `for (i=0; i<n; i++)` → specific error with VUMA suggestion
   213→- **Reference type conversion**: `&T` / `&mut T` auto-convert to `*T` (pointer type)

   242→### First Program
   243→
   244→Create `hello.vuma`:
   245→
   246→```vuma
   247→region main_region {
   248→    let x: ptr<u8> = allocate(1) in main_region;
   249→    write x, 42;
   250→    let val: u8 = read x;
   251→    deallocate x from main_region;
   252→}
   253→```
   254→
   255→Compile and verify:
   256→
   257→```bash
   258→# Verify against all 5 VUMA invariants
   259→cargo run -- verify hello.vuma
   260→

   262→cargo run -- build --target x86_64 hello.vuma
   263→cargo run -- build --target aarch64 hello.vuma
   264→cargo run -- build --target wasm32 hello.vuma
   265→
   266→# Quick syntax and semantic check
   267→cargo run -- check hello.vuma
   268→
   269→# Run with memory safety analysis
   270→cargo run -- run --safe hello.vuma
   271→
   272→# Compile with debug info
   273→cargo run -- build --debug hello.vuma
   274→```

   276→The verifier will check all five invariants against the SCG derived from your program. If verification passes, you get zero-overhead machine code for any of the 8 backends. If it fails, you get a precise counterexample showing the exact path to the violation.

   280→## 8 Backend Architectures
   281→
   282→VUMA supports multi-ISA code generation across 8 backend architectures:
   283→
   284→| Backend | ELF Class | Endianness | Pointer Width | Section Alignment | Calling Convention |
   285→|---------|-----------|------------|---------------|-------------------|-------------------|
   286→| x86_64 | ELF64 | Little | 64-bit | 16 | System V AMD64 (RDI, RSI, RDX, RCX, R8, R9) |
   287→| AArch64 | ELF64 | Little | 64-bit | 16 | AAPCS64 (X0-X7, V0-V7) |
   288→| RISC-V 64 | ELF64 | Little | 64-bit | 4 | RV64G LP64D (A0-A7) |
   289→| ARM32 | ELF32 | Little | 32-bit | 4 | AAPCS (R0-R3) |
   290→| MIPS64 | ELF64 | Big | 64-bit | 8 | N64 ($a0-$a7) |
   291→| PPC64 | ELF64 | Big | 64-bit | 16 | ELFv2 (R3-R10, TOC) |
   292→| LoongArch64 | ELF64 | Little | 64-bit | 8 | LP64 ($a0-$a7) |
   293→| Wasm32 | Wasm | Little | 32-bit | — | Stack machine |
   294→
   295→All backends share a unified `Backend` trait and produce either ELF executables (7 native targets) or Wasm modules (Wasm32). The codegen pipeline is:
   296→
   297→```
   298→SCG → IR (target-independent) → Register Allocation → Instruction Selection → Binary Emission

   446→## Key Concepts
   447→
   448→### SCG — Semantic Computation Graph
   449→
   450→The SCG is the single source of truth in VUMA. There is no "source code" that the compiler translates — the SCG *is* the program, and every other representation (textual, visual, conversational, machine code) is a **projection** of the SCG. Nodes represent computational operations (allocations, accesses, computations, casts, effects, control flow), edges represent relationships (data flow, control flow, derivation, annotation), and regions delineate scopes, phases, and deployment targets.
   451→
   452→### BD — Behavioral Descriptor
   453→
   454→A Behavioral Descriptor replaces traditional nominal types with the triple (RepD, CapD, RelD):
   455→
   456→- **RepD** (Representation Descriptor): memory layout — size, alignment, field offsets, multiple simultaneous interpretations
   457→- **CapD** (Capability Descriptor): permitted operations — read, write, execute, serialize, send, persist, derive-pointer (context-dependent)
   458→- **RelD** (Relational Descriptor): relationships — temporal co-occurrence, structural containment, dependency ordering, semantic equivalence, security-level flow
   459→
   460→BDs are **inferred**, not declared. The IVE derives them from SCG structure through iterative fixpoint computation.
   461→
   462→### MSG — Memory State Graph
   463→
   464→The Memory State Graph captures every allocation point, every pointer derivation, every deallocation point, every concurrent access, and every reinterpretation. It is constructed from the annotated SCG and serves as the formal model against which the five invariants are verified.
   465→
   466→### IVE — Inference & Verification Engine
   467→
   468→The IVE is the reasoning core. It reads the SCG, infers Behavioral Descriptors, constructs the MSG, and verifies the five global invariants. It operates through iterative fixpoint computation, resolving interdependencies between RepD, CapD, and RelD inference. It supports interprocedural analysis, escape analysis, verification caching, and incremental re-verification.
   469→
   470→### COR — Continuous Optimization Runtime
   471→
   472→The COR maintains an always-compiled invariant: every reachable SCG region is kept in compiled machine code at all times. It performs incremental compilation, profile-guided optimization (using PMU counters), speculative optimization with transparent deoptimization, and adaptive deployment across heterogeneous targets.

   496→
   497→```rust
   498→use vuma::VumaForLLM;
   499→
   500→// Compile VUMA source code
   501→let result = VumaForLLM::compile(source);

   503→    println!("Compilation succeeded!");

   506→        println!("SCG: {}", scg);

   512→
   513→// Quick syntax/semantic check
   514→let diagnostics = VumaForLLM::check(source);
   515→for diag in &diagnostics {
   516→    println!("[{}] {}", diag.severity, diag.message);
   517→}
   518→
   519→// Analyze SCG structure
   520→let analysis = VumaForLLM::analyze(source)?;
   521→
   522→// Compile to Wasm sandbox
   523→let wasm = VumaForLLM::to_wasm(source)?;
   524→
   525→// Explain an error in natural language
   526→let explanation = VumaForLLM::explain_error(&diagnostic);
   527→
   528→// Get fix suggestions
   529→let fixes = VumaForLLM::suggest_fixes(&diagnostic);
   530→
   531→// List available targets
   532→let targets = VumaForLLM::targets();
   533→for t in &targets {
   534→    println!("{} ({}, {}-bit, {})", t.name, t.triple, t.pointer_width, t.endianness);
   535→}
   536→```

   538→### LSP Server
   539→
   540→```bash
   541→# Start the LSP server (for IDE integration)
   542→vuma lsp
   543→```
   544→
   545→The LSP server provides:
   546→- **Diagnostics**: real-time verification feedback with counterexamples
   547→- **Hover**: BD details, verification status, type information
   548→- **Go-to-definition**: navigate to function/variable definitions
   549→- **Completion**: VUMA keywords, functions, and types
   550→- **Document symbols**: function list with signatures
   551→- **Semantic tokens**: syntax highlighting information
