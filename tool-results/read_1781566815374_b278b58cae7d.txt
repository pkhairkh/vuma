    11→## 0. Preliminary Definitions
    12→
    13→Before specifying the construction rules, we define the core domains and structures that compose the Memory State Graph.
    14→
    15→### 0.1 Core Domains
    16→
    17→```
    18→NodeId      ∈ ℕ                          — SCG node identifiers
    19→RegionId    ∈ ℕ                          — MSG region identifiers
    20→DerivId     ∈ ℕ                          — Derivation chain identifiers
    21→AccessId    ∈ ℕ                          — Access entry identifiers
    22→Addr        ∈ ℤ                          — Abstract address values
    23→Offset      ∈ ℤ                          — Byte offsets within regions
    24→Size        ∈ ℕ⁺                         — Sizes in bytes
    25→RepD        = Layout × Alignment × Fields — Representation Descriptors
    26→CapD        = 𝒫(Operation)              — Capability Descriptors (sets of ops)
    27→RelD        = 𝒫(Relation)               — Relational Descriptors
    28→```
    29→
    30→### 0.2 MSG Structure
    31→
    32→A **Memory State Graph** is a tuple:
    33→
    34→```
    35→MSG = (R, D, A, φ)
    36→```
    37→
    38→where:
    39→- **R ⊆ Region** is the set of memory regions. Each `r ∈ R` is a record:
    40→  ```
    41→  r = (rid : RegionId, base : Addr, size : Size,
    42→       status : {Allocated, Freed, Stack, Mapped, Device},
    43→       owner : NodeId, repd : RepD, capd : CapD,
    44→       birth : ProgramPoint, death : ProgramPoint ∪ {⊥})
    45→  ```
    46→- **D ⊆ Derivation** is the set of pointer derivations. Each `d ∈ D` is a record:
    47→  ```
    48→  d = (did : DerivId, source : DerivId ∪ RegionId,
    49→       kind : {Offset, Cast, Assign, FieldAccess},
    50→       offset : Offset, targetRepD : RepD,
    51→       programPoint : ProgramPoint)
    52→  ```
    53→- **A ⊆ Access** is the set of memory accesses. Each `a ∈ A` is a record:
    54→  ```
    55→  a = (aid : AccessId, deriv : DerivId, region : RegionId,
    56→       mode : {Read, Write, Execute}, size : Size,
    57→       programPoint : ProgramPoint, path : Path)
    58→  ```
    59→- **φ : Access → {✓, ✗, ?}** is the verification status function, mapping each access to proven-safe (✓), proven-unsafe (✗), or unverified (?).
    60→
    61→### 0.3 SCG Node Types
    62→
    63→The Semantic Computation Graph defines the following node types relevant to the MSG:
    64→
    65→```
    66→N_Alloc     — AllocationNode: creates a new region
    67→N_Dealloc   — DeallocationNode: frees a region
    68→N_Access    — AccessNode: reads from or writes to memory
    69→N_Cast      — CastNode: reinterprets a pointer under a different RepD
    70→N_Arith     — ArithmeticNode: computes an offset from a base pointer
    71→N_Call      — FunctionCall: invokes a procedure
    72→N_Control   — ControlFlow: branches or merges execution paths
    73→```
    74→
    75→---
    76→
    77→## 1. MSG Construction Rules
    78→
    79→The MSG is constructed by a forward traversal of the SCG. Each SCG node type triggers a specific transformation on the MSG. We define each transformation as a formal rule relating the MSG before processing the node (MSG_pre) to the MSG after processing (MSG_post).
    80→
    81→### 1.1 AllocationNode → Creates New Region
    82→
    83→When the SCG traversal encounters an `N_Alloc` node, a new region is created in the MSG and added to the region set R.
    84→
    85→**Rule ALLOC:**
    86→
    87→```
    88→  n ∈ N_Alloc    n.allocSize = s    n.allocRepD = ρ    n.allocCapD = κ
    89→  n.programPoint = pp    rid = freshRegionId()    a = freshAddr()
    90→  ────────────────────────────────────────────────────────────────────────
    91→  MSG_post = (R ∪ {r}, D, A, φ)
    92→
    93→  where r = (rid, a, s, Allocated, n.id, ρ, κ, pp, ⊥)
    94→```
    95→
    96→**Explanation:** The allocation node `n` specifies a size `s`, a representation descriptor `ρ`, and a capability descriptor `κ`. The rule creates a fresh region `r` with a unique identifier `rid`, a fresh base address `a`, and status `Allocated`. The region's `birth` program point is set to `pp` (the node's position in the SCG), and `death` is set to `⊥` (undefined — the region is not yet freed). The owner is set to the node's identifier, linking the region back to its allocation point in the SCG. This satisfies the **origin invariant** (§3.6.2, Invariant 4): every region can be traced back to the allocation node that created it. The fresh address `a` is chosen from a monotonic counter to ensure uniqueness; in the abstract model, we require only that `a` does not overlap with any existing region in R, formally: `∀r' ∈ R_pre : [a, a+s) ∩ [r'.base, r'.base + r'.size) = ∅`.
    97→
    98→### 1.2 DeallocationNode → Sets Region Status to Freed
    99→
   100→When the SCG traversal encounters an `N_Dealloc` node, the target region's status transitions from `Allocated` to `Freed`.
   101→
   102→**Rule DEALLOC:**
   103→
   104→```
   105→  n ∈ N_Dealloc    n.targetDeriv = did    resolveRegion(did, D, R) = r
   106→  r.status = Allocated    n.programPoint = pp
   107→  ────────────────────────────────────────────────────────────────────────
   108→  MSG_post = ((R \ {r}) ∪ {r'}, D, A, φ')
   109→
   110→  where r' = r[status ↦ Freed, death ↦ pp]
   111→        φ'(a) = if a.region = r.rid ∧ a.programPoint > pp then ✗ else φ(a)
   112→```
   113→
   114→**Explanation:** The deallocation node `n` references a derivation `did` that resolves to region `r`. The rule transitions `r.status` from `Allocated` to `Freed` and records the deallocation point as `r.death = pp`. Critically, the verification function `φ` is updated: any access `a ∈ A` whose target region is `r` and whose program point is *after* `pp` is marked as `✗` (proven-unsafe) — this is a use-after-free violation. This directly enforces the **liveness invariant** (§3.6.2, Invariant 1). If the region is already `Freed`, this constitutes a double-free, and the rule signals an error. The `resolveRegion` function follows the derivation chain from `did` back to its source region: `resolveRegion(did, D, R) = r` where `r.rid = rootRegion(did, D)`.
   115→
   116→### 1.3 AccessNode → Creates Access Entry Linked to Derivation
   117→
   118→When the SCG traversal encounters an `N_Access` node, a new access entry is created in the MSG, linked to the derivation that produced the pointer being dereferenced.
   119→
   120→**Rule ACCESS:**
   121→
   122→```
   123→  n ∈ N_Access    n.derivId = did    n.mode = m ∈ {Read, Write, Execute}
   124→  n.accessSize = s    n.programPoint = pp    n.path = π
   125→  resolveRegion(did, D, R) = r    offset(did, D) = o
   126→  ────────────────────────────────────────────────────────────────────────
   127→  MSG_post = (R, D, A ∪ {a}, φ ∪ {(aid, v)})
   128→
   129→  where a = (aid, did, r.rid, m, s, pp, π)
   130→        v = verify(a, R, D, A) ∈ {✓, ✗, ?}
   131→```
   132→
   133→**Explanation:** The access node `n` specifies which derivation (`did`) produces the pointer being dereferenced, the access mode `m`, the size `s` of the access, and the program point `pp`. The rule creates an access entry `a` linking the derivation to the target region. The verification function `verify(a, R, D, A)` checks four conditions corresponding to the VUMA invariants:
   134→
   135→1. **Liveness** (Invariant 1): `r.status = Allocated ∧ pp ∈ [r.birth, r.death)` — the region is alive at the access point.
   136→2. **Bounds** (subset of Liveness): `0 ≤ o ∧ o + s ≤ r.size` — the accessed range falls within the region.
   137→3. **Exclusivity** (Invariant 2): For `m = Write`, `∄a' ∈ A : a'.region = r.rid ∧ a'.mode = Write ∧ a'.path ∥ a.path ∧ rangesOverlap(a, a')` — no conflicting concurrent write access exists.
   138→4. **Interpretation** (Invariant 3): The bytes `[o, o+s)` within region `r` are interpretable according to the derivation's `targetRepD` — the representation descriptor at the access offset is compatible with `n`'s expected interpretation.
   139→
   140→If all four checks pass, `v = ✓`. If any check definitively fails, `v = ✗`. If a check cannot be resolved (e.g., due to path sensitivity or abstraction), `v = ?`. The access is recorded regardless of verification status — the IVE may refine `v` later as more information becomes available.
   141→
   142→### 1.4 CastNode → Creates New Derivation with Cast RepD
   143→
   144→When the SCG traversal encounters an `N_Cast` node, a new derivation is created that records the reinterpretation of a pointer under a different representation descriptor.
   145→
   146→**Rule CAST:**
   147→
   148→```
   149→  n ∈ N_Cast    n.sourceDeriv = did_s    n.targetRepD = ρ_t
   150→  n.programPoint = pp    d_new_id = freshDerivId()
   151→  d_s ∈ D    d_s.did = did_s    resolveRegion(did_s, D, R) = r
   152→  offset(did_s, D) = o    ρ_t.alignment = al    o mod al = 0
   153→  ────────────────────────────────────────────────────────────────────────
   154→  MSG_post = (R, D ∪ {d_new}, A, φ)
   155→
   156→  where d_new = (d_new_id, did_s, Cast, o, ρ_t, pp)
   157→```
   158→
   159→**Explanation:** The cast node `n` takes a source derivation `did_s` and produces a new derivation `d_new` with kind `Cast`. The offset from the source derivation is preserved (casts do not change the address, only the interpretation). The `targetRepD` of the new derivation is `ρ_t`, specified by the cast node. The side condition `o mod al = 0` ensures alignment: the target representation descriptor requires alignment `al`, and the offset `o` must satisfy this alignment constraint. If the alignment check fails, the cast creates an **invalid derivation** — any subsequent access through `d_new` will fail the interpretation invariant check. This rule is critical for the **interpretation invariant** (§3.6.2, Invariant 3): the IVE must verify that the bytes at offset `o` within region `r` are valid under `ρ_t`. The derivation chain is extended: `d_new.source = did_s`, maintaining the full provenance from allocation through all intermediate derivations to the final access point.
   160→
   161→### 1.5 ArithmeticNode → Creates Offset Derivation from Source
   162→
   163→When the SCG traversal encounters an `N_Arith` node, a new derivation is created that records the computation of an offset from a base pointer.
   164→
   165→**Rule ARITH:**
   166→
   167→```
   168→  n ∈ N_Arith    n.sourceDeriv = did_s    n.computedOffset = Δ
   169→  n.programPoint = pp    d_new_id = freshDerivId()
   170→  d_s ∈ D    d_s.did = did_s    resolveRegion(did_s, D, R) = r
   171→  offset(did_s, D) = o_s    o_new = o_s + Δ
   172→  0 ≤ o_new    o_new ≤ r.size
   173→  ────────────────────────────────────────────────────────────────────────
   174→  MSG_post = (R, D ∪ {d_new}, A, φ)
   175→
   176→  where d_new = (d_new_id, did_s, Offset, o_new, d_s.targetRepD, pp)
   177→```
   178→
   179→**Explanation:** The arithmetic node `n` computes an offset `Δ` from the source derivation's current position. The new derivation `d_new` has kind `Offset` and records the absolute offset `o_new = o_s + Δ` within the source region `r`. The side conditions `0 ≤ o_new` and `o_new ≤ r.size` enforce the **bounds invariant**: the derived pointer must remain within the allocated region. If these conditions cannot be statically verified (e.g., `Δ` depends on runtime input), the derivation is marked with a bounds verification debt, and any subsequent access through this derivation will have `v = ?` until the bounds can be proven. The `targetRepD` of the new derivation defaults to the source derivation's RepD — the arithmetic operation changes the address but not the interpretation. A subsequent `CastNode` can change the RepD if needed. This two-step decomposition (offset then cast) mirrors the physical reality of pointer arithmetic in C-like systems, where computing an offset and reinterpreting the result are separate operations. The derivation chain preserves the full computation: from `r.base + o_s` to `r.base + o_new`, each step recorded with its program point, enabling the IVE to trace any pointer back to its allocation.
   180→
   181→### 1.6 FunctionCall → Inlines Callee's MSG or Creates Boundary
   182→
   183→When the SCG traversal encounters an `N_Call` node, the callee's MSG is either inlined into the caller's MSG (if the callee's MSG is available and small enough) or a function boundary is created (if the callee is external, too large, or not yet analyzed).
   184→
   185→**Rule CALL-INLINE:**
   186→
   187→```
   188→  n ∈ N_Call    n.callee = f    MSG_f = (R_f, D_f, A_f, φ_f) available
   189→  n.arguments = [(did_1, ρ_1), ..., (did_k, ρ_k)]
   190→  n.programPoint = pp    R_f' = rename(R_f, pp)
   191→  D_f' = rename(D_f, pp, did_1↦f_param_1, ..., did_k↦f_param_k)
   192→  A_f' = rename(A_f, pp, path_prefix = n.callPath)
   193→  ────────────────────────────────────────────────────────────────────────
   194→  MSG_post = (R ∪ R_f', D ∪ D_f', A ∪ A_f', φ ∪ φ_f')
   195→```
   196→
   197→**Rule CALL-BOUNDARY:**
   198→
   199→```
   200→  n ∈ N_Call    n.callee = f    MSG_f not available or too large
   201→  n.arguments = [(did_1, ρ_1), ..., (did_k, ρ_k)]
   202→  n.programPoint = pp
   203→  ────────────────────────────────────────────────────────────────────────
   204→  MSG_post = (R, D ∪ {d_boundary}, A, φ ∪ {(aid_boundary, ?)})
   205→
   206→  where d_boundary = (freshDerivId(), ⊥, Boundary, 0, ρ_unknown, pp)
   207→        ∀did_i : createBoundaryAccess(did_i, pp, n.callPath)
   208→```
   209→
   210→**Explanation:** The call-inline rule composes the caller's MSG with the callee's MSG. The `rename` function applies a namespace prefix (based on the call site's program point `pp`) to all identifiers in the callee's MSG, preventing collisions. Argument derivations from the caller (`did_1, ..., did_k`) are mapped to the callee's formal parameter derivations (`f_param_1, ..., f_param_k`), establishing the derivation chain across the call boundary. Access paths within the callee are prefixed with the call path, enabling path-sensitive analysis. The call-boundary rule handles cases where the callee's MSG cannot be inlined: an opaque derivation `d_boundary` is created with kind `Boundary` and unknown RepD. All pointer arguments are recorded as boundary accesses with verification status `?` (unverified). This conservative treatment ensures soundness: the IVE will flag any access through a boundary derivation as unverified, prompting either inline analysis of the callee or manual verification. The boundary mechanism is the MSG's interface to compositional reasoning: by summarizing a function's memory effects as a contract (precondition/postcondition on regions and derivations), the IVE can verify callers without inlining, at the cost of some precision.
   211→
   212→### 1.7 ControlFlow → Splits MSG into Paths, Merges with Join Rules
   213→
   214→When the SCG traversal encounters an `N_Control` node, the MSG is split into separate execution paths (at branch points) and merged (at join points).
   215→
   216→**Rule BRANCH:**
   217→
   218→```
   219→  n ∈ N_Control    n.kind = Branch    n.condition = c
   220→  n.trueSuccessor = n_t    n.falseSuccessor = n_f
   221→  MSG_pre = (R, D, A, φ)    π = currentPath
   222→  ────────────────────────────────────────────────────────────────────────
   223→  MSG_post = { (π·⊤, MSG_t), (π·⊥, MSG_f) }
   224→
   225→  where MSG_t = (R, D, A, φ)  — to be extended by n_t's successors
   226→        MSG_f = (R, D, A, φ)  — to be extended by n_f's successors
   227→```
   228→
   229→**Rule JOIN:**
   230→
   231→```
   232→  n ∈ N_Control    n.kind = Join
   233→  n.predecessors = {n_1, ..., n_k}
   234→  MSG_i = (R_i, D_i, A_i, φ_i) for each i ∈ {1, ..., k}
   235→  ────────────────────────────────────────────────────────────────────────
   236→  MSG_post = (⋂_i R_i^live, ⋃_i D_i, ⋃_i A_i, ⋃_i φ_i)
   237→
   238→  where R_i^live = { r ∈ R_i | r.status = Allocated }
   239→```
   240→
   241→**Explanation:** The branch rule creates two copies of the current MSG, one for each successor path. The path `π` is extended with `⊤` (true branch) or `⊥` (false branch), creating distinct path identifiers for the two executions. Each copy is then independently extended by the successors. The join rule merges the MSGs from all incoming paths. The region set is intersected over *live* regions: only regions that are allocated on *all* incoming paths survive the join. A region that is freed on one path but live on another results in a region with status that depends on the path — this is handled by the path-sensitive MSG (Section 3). The derivation and access sets are unioned, as they are path-indexed and do not conflict. The verification function is also unioned; if an access is proven safe on one path but unverified on another, the merged status is the weaker of the two (ordering: `✓ > ? > ✗`). This ensures that the join rule is sound: a merged MSG never proves more than what all individual path MSGs can prove. The join rule is the key mechanism for handling loops (where the join point is the loop header) and conditionals (where the join point is the post-dominator).

   245→## 2. Derivation Chain Construction
   246→
   247→A derivation chain records the complete provenance of a pointer from its allocation point to every point of use. This section formalizes the construction of derivation chains as inference rules.
   248→
   249→### 2.1 Definition: Derivation Chain
   250→
   251→A **derivation chain** for derivation `d_k` is a sequence of derivations:
   252→
   253→```
   254→chain(d_k) = [d_0, d_1, ..., d_k]
   255→```
   256→
   257→where `d_0` is the root derivation (whose source is a region), and for each `i > 0`, `d_i.source = d_{i-1}.did`. The chain captures every transformation applied to the pointer: offsets, casts, assignments, and field accesses.
   258→
   259→The **offset function** computes the cumulative offset from the region base:
   260→
   261→```
   262→offset(d_0, D) = 0
   263→offset(d_i, D) = offset(d_{i-1}, D) + d_i.offset   (if d_i.kind = Offset)
   264→offset(d_i, D) = offset(d_{i-1}, D)                  (otherwise)
   265→```
   266→
   267→### 2.2 Inference Rules for Derivation Chain Construction
   268→
   269→**Rule CHAIN-ROOT** — Starting from an AllocationNode, the root derivation is created:
   270→
   271→```
   272→  n ∈ N_Alloc    r = new region from Rule ALLOC
   273→  ────────────────────────────────────────
   274→  chain(d_root) = [d_root]
   275→
   276→  where d_root = (freshDerivId(), r.rid, Assign, 0, r.repd, n.programPoint)
   277→```
   278→
   279→This rule states that every allocation produces a root derivation `d_root` whose source is the region itself (not another derivation), with offset 0 and RepD equal to the region's allocation RepD.
   280→
   281→**Rule CHAIN-ASSIGN** — Each pointer assignment creates a Derivation:
   282→
   283→```
   284→  chain(d_s) = [d_0, ..., d_s]    n assigns d_s to new variable
   285→  d_new = (freshDerivId(), d_s.did, Assign, offset(d_s, D), d_s.targetRepD, pp)
   286→  ────────────────────────────────────────
   287→  chain(d_new) = [d_0, ..., d_s, d_new]
   288→```
   289→
   290→This rule extends the derivation chain with a new `Assign` derivation. An assignment does not change the offset or RepD — it creates an alias. Both `d_s` and `d_new` refer to the same memory location, but through different derivation chains. This is crucial for alias analysis: the IVE can determine that two derivations are aliases by checking whether they resolve to the same (region, offset) pair.
   291→
   292→**Rule CHAIN-OFFSET** — Each arithmetic operation extends the chain:
   293→
   294→```
   295→  chain(d_s) = [d_0, ..., d_s]    n ∈ N_Arith    n.computedOffset = Δ
   296→  o_new = offset(d_s, D) + Δ
   297→  d_new = (freshDerivId(), d_s.did, Offset, o_new, d_s.targetRepD, pp)
   298→  ────────────────────────────────────────
   299→  chain(d_new) = [d_0, ..., d_s, d_new]
   300→```
   301→
   302→This rule records the offset computation as a new link in the derivation chain. The offset is accumulated: `d_new.offset = o_new = d_s.offset + Δ`. The RepD is preserved; a subsequent cast can change it. The IVE can verify the bounds invariant by checking `0 ≤ o_new ≤ r.size` where `r = resolveRegion(d_s, D, R)`.
   303→
   304→**Rule CHAIN-CAST** — Each cast creates a new branch:
   305→
   306→```
   307→  chain(d_s) = [d_0, ..., d_s]    n ∈ N_Cast    n.targetRepD = ρ_t
   308→  o = offset(d_s, D)    o mod ρ_t.alignment = 0
   309→  d_new = (freshDerivId(), d_s.did, Cast, o, ρ_t, pp)
   310→  ────────────────────────────────────────
   311→  chain(d_new) = [d_0, ..., d_s, d_new]
   312→```
   313→
   314→A cast creates a *branch* in the derivation chain: from the same source derivation `d_s`, multiple derivations with different RepDs may exist. For example, the same memory may be cast to both `uint32[]` and `float32[]` at the same offset — each cast creates a separate derivation chain branch. The alignment check is a side condition: if the target RepD requires 8-byte alignment and the offset is 4, the cast is invalid and the derivation is flagged.
   315→
   316→**Rule CHAIN-FIELD** — Field access creates a specialized offset derivation:
   317→
   318→```
   319→  chain(d_s) = [d_0, ..., d_s]    n accesses field f of d_s
   320→  ρ_s = d_s.targetRepD    ρ_s.fields[f] = (f_offset, f_size, f_repd)
   321→  d_new = (freshDerivId(), d_s.did, FieldAccess, offset(d_s, D) + f_offset, f_repd, pp)
   322→  ────────────────────────────────────────
   323→  chain(d_new) = [d_0, ..., d_s, d_new]
   324→```
   325→
   326→Field access is a special case of offset derivation where the offset and target RepD are determined by the field descriptor within the source RepD. This provides stronger verification: the IVE can check not only bounds and alignment but also that the field offset is valid within the source RepD's layout.
   327→
   328→### 2.3 Properties of Derivation Chains
   329→
   330→**Theorem (Origin Invariant).** For every derivation `d ∈ D`, `resolveRegion(d, D, R)` returns a region `r ∈ R` such that `r.owner` is the `N_Alloc` node that created the region.
   331→
   332→*Proof sketch.* By induction on the length of the derivation chain. Base case: `d_root.source = r.rid` and `r.owner = N_Alloc` by construction (Rule CHAIN-ROOT). Inductive step: if `d_i.source = d_{i-1}.did`, then `resolveRegion(d_i, D, R) = resolveRegion(d_{i-1}, D, R) = r` by the induction hypothesis. ∎
   333→
   334→**Theorem (Chain Finiteness).** For any derivation `d`, `|chain(d)|` is finite and bounded by the number of SCG nodes on the path from the allocation to the use.
   335→
   336→*Proof sketch.* Each derivation in the chain corresponds to a distinct SCG node (by the `programPoint` field). The SCG is a finite DAG, so the number of nodes on any path is finite. ∎
   337→
   338→**Theorem (Derivation Uniqueness).** If `chain(d_1) = chain(d_2)`, then `offset(d_1, D) = offset(d_2, D)` and `resolveRegion(d_1, D, R) = resolveRegion(d_2, D, R)`.
   339→
   340→*Proof.* Directly from the definition of `chain` and `offset`. Equal chains imply equal sequences of offsets and casts, producing equal cumulative offsets and the same root region. ∎

   344→## 3. Path-Sensitive MSG
   345→
   346→Different execution paths through the SCG may produce different memory states. A single monolithic MSG cannot capture this path-dependence without losing precision. We therefore define the MSG as a set of path-indexed memory states.
   347→
   348→### 3.1 Formal Definition
   349→
   350→A **Path-Sensitive MSG** is a finite set of pairs:
   351→
   352→```
   353→PS-MSG = { (π₁, M₁), (π₂, M₂), ..., (πₙ, Mₙ) }
   354→```
   355→
   356→where:
   357→- **Path** `πᵢ ∈ BranchDecision*` is a sequence of branch decisions. Each `BranchDecision` is a pair `(nodeId, outcome)` where `outcome ∈ {⊤, ⊥}`.
   358→- **MemoryState** `Mᵢ = (Rᵢ, Dᵢ, Aᵢ, φᵢ)` is a tuple of regions, derivations, accesses, and verification statuses, as defined in §0.2.
   359→
   360→We define the following operations on paths:
   361→
   362→```
   363→π₁ ≼ π₂  ≡  π₁ is a prefix of π₂       (path π₁ is an ancestor of π₂)
   364→π₁ ⊥ π₂   ≡  ¬(π₁ ≼ π₂) ∧ ¬(π₂ ≼ π₁)  (paths diverge at some branch)
   365→```
   366→
   367→### 3.2 Path-Sensitive Construction Rules
   368→
   369→**Rule PS-BRANCH:**
   370→
   371→```
   372→  (π, M) ∈ PS-MSG    n ∈ N_Control    n.kind = Branch
   373→  ────────────────────────────────────────────────────────────────────────
   374→  PS-MSG' = (PS-MSG \ {(π, M)}) ∪ { (π·(n.id, ⊤), M), (π·(n.id, ⊥), M) }
   375→```
   376→
   377→At a branch point, the current path `π` is split into two paths, each extended with the branch decision. The memory state `M` is duplicated for both paths — subsequent nodes will modify each copy independently.
   378→
   379→**Rule PS-MERGE:**
   380→
   381→```
   382→  (π₁, M₁) ∈ PS-MSG    (π₂, M₂) ∈ PS-MSG
   383→  π₁ ⊥ π₂    π₁ and π₂ converge at join node n_j
   384→  M₁ = (R₁, D₁, A₁, φ₁)    M₂ = (R₂, D₂, A₂, φ₂)
   385→  ────────────────────────────────────────────────────────────────────────
   386→  PS-MSG' = (PS-MSG \ {(π₁, M₁), (π₂, M₂)}) ∪ { (π₁⊙π₂, M_join) }
   387→
   388→  where M_join = (R_join, D₁ ∪ D₂, A₁ ∪ A₂, φ_join)
   389→        R_join = R₁ ∩_live R₂    (see below)
   390→        φ_join(a) = φ₁(a) ⊓ φ₂(a)  (glb in the lattice ✓ > ? > ✗)
   391→        π₁⊙π₂ = the merge of paths π₁ and π₂ at n_j
   392→```
   393→
   394→The intersection `R₁ ∩_live R₂` is defined as:
   395→
   396→```
   397→R₁ ∩_live R₂ = { r | r.rid ∈ (liveRegionIds(R₁) ∩ liveRegionIds(R₂)) }
   398→               ∪ { r_f | r_f.rid ∈ (freedOnOnePath(R₁, R₂)) }
   399→
   400→where:
   401→  liveRegionIds(R) = { r.rid | r ∈ R ∧ r.status = Allocated }
   402→  freedOnOnePath(R₁, R₂) = { r.rid | (r₁ ∈ R₁ ∧ r₁.status = Freed) ⊻ (r₂ ∈ R₂ ∧ r₂.status = Freed) }
   403→```
   404→
   405→For regions that are freed on one path but live on another, the merged region retains both possibilities: the merged region's status becomes `PathDependent(freedOn={π₁}, liveOn={π₂})`. This status is resolved at each access: if the access is on a path where the region is freed, it is flagged as a use-after-free; if on a path where the region is live, it proceeds normally.
   406→
   407→### 3.3 Access Verification Under Path Sensitivity
   408→
   409→When verifying an access `a` under a path-sensitive MSG, the IVE considers only those memory states whose paths are compatible with the access's path:
   410→
   411→```
   412→verify_path_sensitive(a, PS-MSG) = v
   413→
   414→  where S = { M | (π, M) ∈ PS-MSG ∧ π ≼ a.path }
   415→        V = { verify(a, M) | M ∈ S }
   416→        v = ⨓ V    (greatest lower bound of all verification results)
   417→```
   418→
   419→If the access is proven safe on *all* compatible paths, `v = ✓`. If it is proven unsafe on *any* compatible path, `v = ✗`. If it is unverified on some paths and safe on others, `v = ?`. This conservative merging ensures soundness: a path-sensitive MSG never proves an access safe that would be unsafe on some feasible execution path.
   420→
   421→### 3.4 Path Pruning
   422→
   423→To prevent path explosion (exponential growth in the number of path-state pairs), the PS-MSG employs **path pruning**:
   424→
   425→**Definition (Path Equivalence).** Two paths `π₁, π₂` are equivalent with respect to memory state if they produce identical memory states:
   426→
   427→```
   428→π₁ ≡_M π₂  ≡  M(π₁) = M(π₂)
   429→```
   430→
   431→where `M(π)` denotes the memory state associated with path `π`.
   432→
   433→**Rule PS-PRUNE:**
   434→
   435→```
   436→  (π₁, M₁) ∈ PS-MSG    (π₂, M₂) ∈ PS-MSG    M₁ = M₂
   437→  ────────────────────────────────────────────────────────────────────────
   438→  PS-MSG' = (PS-MSG \ {(π₁, M₁), (π₂, M₂)}) ∪ { (π₁ ∪ π₂, M₁) }
   439→```
   440→
   441→When two paths produce identical memory states, they are merged into a single path (represented as a set of branch decisions that lead to the same state). This is the key mechanism for preventing path explosion: after a join point, if the branch condition does not affect memory state (e.g., a branch that prints a debug message but does not modify memory), the two paths collapse into one.
   442→
   443→### 3.5 Widening for Loop Handling
   444→
   445→For loops, the PS-MSG applies a **widening operator** to ensure convergence:
   446→
   447→```
   448→  M_n = M_{n-1} ▽ (M_n^raw - M_{n-1})
   449→```
   450→
   451→where `▽` is the widening operator that abstracts the differences between the memory state at loop iteration `n-1` and the raw state at iteration `n`. The widening operator:
   452→- Replaces concrete offset ranges with intervals: if offset is 0, 4, 8 after three iterations, widen to [0, ∞).
   453→- Replaces finite region sets with region classes (see Section 4).
   454→- Replaces concrete derivation chains with pattern summaries.
   455→
   456→The widened state over-approximates all possible memory states reachable through the loop, ensuring that the PS-MSG converges in finite time while maintaining soundness.

   460→## 4. MSG Abstraction for Scalability
   461→
   462→The concrete MSG, as defined in Sections 1–3, can grow prohibitively large for real-world programs. This section defines an **abstract MSG** that groups similar regions and summarizes derivation chains, trading precision for scalability while preserving soundness.
   463→
   464→### 4.1 Abstract Domains
   465→
   466→**Abstract Region (Region Class).** A region class `rc` groups regions with similar properties:
   467→
   468→```
   469→rc = (rcid : RegionClassId,
   470→      allocSite : 𝒫(NodeId),        — set of allocation nodes
   471→      sizeRange : (Size, Size),       — (minSize, maxSize) interval
   472→      status : {Allocated, Freed, Mixed},
   473→      repdClass : RepDClass,          — abstract RepD
   474→      capdClass : CapDClass,          — abstract CapD
   475→      lifetimeRange : (PP, PP))       — (earliestBirth, latestDeath)
   476→```
   477→
   478→A region class `rc` **represents** a concrete region `r` (written `r ⊑ rc`) if:
   479→
   480→```
   481→r.owner ∈ rc.allocSite
   482→∧ r.size ∈ [rc.sizeRange.min, rc.sizeRange.max]
   483→∧ r.repd ⊑ rc.repdClass
   484→∧ r.capd ⊆ rc.capdClass
   485→∧ r.birth ≥ rc.lifetimeRange.min
   486→∧ (rc.status = Mixed ∨ r.status = rc.status)
   487→```
   488→
   489→**Abstract Derivation.** An abstract derivation summarizes a family of concrete derivation chains by their pattern:
   490→
   491→```
   492→d̂ = (did : DerivId,
   493→      sourceClass : RegionClassId,
   494→      pattern : DerivPattern,
   495→      offsetRange : (Offset, Offset),
   496→      repdClass : RepDClass)
   497→
   498→DerivPattern ::= Root
   499→               | OffsetPat(DerivPattern, OffsetRange)
   500→               | CastPat(DerivPattern, RepDClass)
   501→               | AssignPat(DerivPattern)
   502→               | FieldPat(DerivPattern, FieldName)
   503→               | BoundaryPat
   504→```
   505→
   506→An abstract derivation `d̂` **represents** a concrete derivation `d` (written `d ⊑ d̂`) if:
   507→
   508→```
   509→resolveRegion(d, D, R) ⊑ rc   (where rc = region class with rcid = d̂.sourceClass)
   510→∧ chain(d) matches d̂.pattern
   511→∧ offset(d, D) ∈ [d̂.offsetRange.min, d̂.offsetRange.max]
   512→∧ d.targetRepD ⊑ d̂.repdClass
   513→```
   514→
   515→### 4.2 Abstract MSG Definition
   516→
   517→An **Abstract MSG** is a tuple:
   518→
   519→```
   520→A-MSG = (RC, D̂, Â, φ̂)
   521→```
   522→
   523→where:
   524→- `RC` is a set of region classes
   525→- `D̂` is a set of abstract derivations
   526→- `Â` is a set of abstract accesses (grouped by region class and derivation class)
   527→- `φ̂ : Â → {✓, ?, ✗}` is the abstract verification function
   528→
   529→### 4.3 Abstraction Function
   530→
   531→The **abstraction function** `α : Concrete MSG → Abstract MSG` maps a concrete MSG to its abstract representation:
   532→
   533→```
   534→α(MSG) = A-MSG
   535→
   536→where:
   537→  RC = { rc | ∃r ∈ R : r ⊑ rc }           — group regions into classes
   538→  D̂ = { d̂ | ∃d ∈ D : d ⊑ d̂ }             — group derivations by pattern
   539→  Â = { â | ∃a ∈ A : a ⊑ â }               — group accesses by class
   540→  φ̂(â) = ⨓ { φ(a) | a ∈ A ∧ a ⊑ â }       — merge verification results
   541→```
   542→
   543→### 4.4 Concretization Function
   544→
   545→The **concretization function** `γ : Abstract MSG → 𝒫(Concrete MSG)` maps an abstract MSG to the set of all concrete MSGs it represents:
   546→
   547→```
   548→γ(A-MSG) = { MSG | α(MSG) ⊑ A-MSG }
   549→```
   550→
   551→where `⊑` on abstract MSGs is the pointwise ordering on region classes, derivations, and verification results.
   552→
   553→### 4.5 Galois Connection
   554→
   555→The abstraction and concretization functions form a **Galois connection**:
   556→
   557→```
   558→𝒫(Concrete MSG)  ⇄  Abstract MSG
   559→       γ                  α
   560→
   561→satisfying:  α(MSG) ⊑ A-MSG  ⇔  MSG ∈ γ(A-MSG)
   562→```
   563→
   564→This ensures that the abstraction is sound: any property proved about the abstract MSG holds for all concrete MSGs it represents.
   565→
   566→### 4.6 Soundness Theorem
   567→
   568→**Theorem (Abstract Soundness).** If the abstract MSG proves an invariant (i.e., `φ̂(â) = ✓` for some abstract access `â`), then the invariant also holds in every concrete MSG represented by the abstract MSG.
   569→
   570→*Formally:*
   571→
   572→```
   573→∀A-MSG, â ∈ Â : φ̂(â) = ✓ ⇒ ∀MSG ∈ γ(A-MSG), ∀a ∈ A : a ⊑ â ⇒ φ(a) = ✓
   574→```
   575→
   576→*Proof sketch.* Assume `φ̂(â) = ✓`. By definition of `φ̂`, `⨓ { φ(a) | a ∈ A ∧ a ⊑ â } = ✓`. Since `✓` is the greatest element of the verification lattice, every `φ(a)` in the meet must be `✓`. Therefore, for every concrete access `a ⊑ â`, `φ(a) = ✓`. ∎
   577→
   578→### 4.7 Precision vs. Performance Tradeoff
   579→
   580→The abstraction introduces imprecision in two ways:
   581→
   582→1. **Region class merging**: Two regions with different sizes or RepDs may be grouped into the same class, causing the abstract verification to be less precise than the concrete verification. Example: regions of size 64 and 128 merged into a class with `sizeRange = (64, 128)` — an offset of 100 is valid for the larger region but invalid for the smaller one; the abstract verification must conservatively report `?`.
   583→
   584→2. **Derivation pattern summarization**: Long derivation chains are summarized by patterns, losing information about intermediate steps. Example: a chain of 10 offset operations summarized as `OffsetPat(Root, [0, 1000))` — the abstract MSG knows the final offset is in [0, 1000) but not the exact value, potentially missing bounds violations at specific offsets.
   585→
   586→The **precision knob** is the granularity of region classes and derivation patterns. Finer granularity (more classes, more specific patterns) yields more precise verification but higher computational cost. Coarser granularity yields faster analysis but more `?` results. The IVE can dynamically adjust the precision: start with coarse abstractions, and refine only for accesses that yield `?` — a form of **counterexample-guided abstraction refinement (CEGAR)** adapted for memory verification.
   587→
   588→### 4.8 Widening and Narrowing
   589→
   590→To ensure termination of the abstract construction, we define widening and narrowing operators on the abstract domains:
   591→
   592→**Widening on region classes:**
   593→
   594→```
   595→rc₁ ▽ rc₂ = rc_merged
   596→
   597→where rc_merged.allocSite = rc₁.allocSite ∪ rc₂.allocSite
   598→      rc_merged.sizeRange = (min(rc₁.sizeRange.min, rc₂.sizeRange.min),
   599→                             max(rc₁.sizeRange.max, rc₂.sizeRange.max))
   600→      rc_merged.status = if rc₁.status = rc₂.status then rc₁.status else Mixed
   601→```
   602→
   603→**Narrowing on region classes:**
   604→
   605→```
   606→rc₁ △ rc₂ = rc_narrowed
   607→
   608→where rc_narrowed.sizeRange = intersect(rc₁.sizeRange, rc₂.sizeRange)
   609→      rc_narrowed.status = rc₂.status  (use the more precise status)
   610→```
   611→
   612→Widening is applied during fixpoint iteration to ensure convergence; narrowing is applied afterward to recover some of the precision lost by widening. This is the standard approach from abstract interpretation, adapted to the MSG domain.

   616→## 5. Incremental MSG Update
   617→
   618→When a node is added to the SCG (e.g., during interactive editing), reconstructing the entire MSG from scratch is wasteful. This section defines an **incremental update** mechanism that computes only the delta — the parts of the MSG that change.
   619→
   620→### 5.1 Delta-MSG Definition
   621→
   622→A **Delta-MSG** captures the changes to an MSG resulting from the addition of a single SCG node `n`:
   623→
   624→```
   625→Δ-MSG(n) = (ΔR, ΔD, ΔA, Δφ)
   626→```
   627→
   628→where:
   629→- `ΔR = (R⁺, R⁻, R~)` — regions added (`R⁺`), removed (`R⁻`), and modified (`R~`)
   630→- `ΔD = (D⁺, D⁻, D~)` — derivations added, removed, and modified
   631→- `ΔA = (A⁺, A⁻, A~)` — accesses added, removed, and modified
   632→- `Δφ` — changes to the verification function
   633→
   634→The **incremental update** applies the delta to the existing MSG:
   635→
   636→```
   637→MSG ⊕ Δ-MSG = MSG'
   638→
   639→where R' = (R \ R⁻) ∪ R⁺ ∪ R~
   640→      D' = (D \ D⁻) ∪ D⁺ ∪ D~
   641→      A' = (A \ A⁻) ∪ A⁺ ∪ A~
   642→      φ' = φ ∪ Δφ    (Δφ overrides existing entries)
   643→```
   644→
   645→### 5.2 Delta Computation Rules
   646→
   647→Each SCG node type produces a specific delta:
   648→
   649→**Delta-ALLOC:**
   650→
   651→```
   652→  n ∈ N_Alloc added to SCG
   653→  ────────────────────────────────────────
   654→  Δ-MSG(n) = ({r}, ∅, ∅, ∅)
   655→
   656→  where r = new region per Rule ALLOC
   657→```
   658→
   659→**Delta-DEALLOC:**
   660→
   661→```
   662→  n ∈ N_Dealloc added to SCG
   663→  ────────────────────────────────────────
   664→  Δ-MSG(n) = (∅, ∅, ∅, ∅, Δφ)
   665→
   666→  where Δφ = { (a.aid, ✗) | a ∈ A ∧ a.region = r.rid ∧ a.programPoint > n.programPoint }
   667→        r = target region of n
   668→```
   669→
   670→**Delta-ACCESS:**
   671→
   672→```
   673→  n ∈ N_Access added to SCG
   674→  ────────────────────────────────────────
   675→  Δ-MSG(n) = (∅, ∅, {a}, {(a.aid, v)})
   676→
   677→  where a = new access per Rule ACCESS
   678→        v = verify(a, R, D, A)
   679→```
   680→
   681→**Delta-CAST:**
   682→
   683→```
   684→  n ∈ N_Cast added to SCG
   685→  ────────────────────────────────────────
   686→  Δ-MSG(n) = (∅, {d_new}, ∅, ∅)
   687→
   688→  where d_new = new derivation per Rule CAST
   689→```
   690→
   691→**Delta-ARITH:**
   692→
   693→```
   694→  n ∈ N_Arith added to SCG
   695→  ────────────────────────────────────────
   696→  Δ-MSG(n) = (∅, {d_new}, ∅, ∅)
   697→
   698→  where d_new = new derivation per Rule ARITH
   699→```
   700→
   701→**Delta-CALL:**
   702→
   703→```
   704→  n ∈ N_Call added to SCG
   705→  ────────────────────────────────────────────────────────────────────────
   706→  Δ-MSG(n) = (R_f', D_f', A_f', φ_f')    (if inlining)
   707→
   708→  Δ-MSG(n) = (∅, {d_boundary}, {boundary_accesses}, Δφ_boundary)    (if boundary)
   709→```
   710→
   711→**Delta-CONTROL:**
   712→
   713→```
   714→  n ∈ N_Control (Branch) added to SCG
   715→  ────────────────────────────────────────
   716→  Δ-MSG(n) = path-split of existing MSG per Rule PS-BRANCH
   717→
   718→  n ∈ N_Control (Join) added to SCG
   719→  ────────────────────────────────────────
   720→  Δ-MSG(n) = path-merge of incoming MSGs per Rule PS-MERGE
   721→```
   722→
   723→### 5.3 Propagation Rules for Delta Updates
   724→
   725→When a delta modifies a region, derivation, or access that is referenced by other parts of the MSG, the change must be **propagated**. Propagation follows the dependency graph of the MSG.
   726→
   727→**Definition (Dependency Graph).** The MSG dependency graph `G_dep = (V, E)` has:
   728→- Vertices `V = R ∪ D ∪ A` (all regions, derivations, and accesses)
   729→- Edges `E = { (d, r) | d ∈ D, r = resolveRegion(d) } ∪ { (a, d) | a ∈ A, d = a.deriv } ∪ { (d, d') | d ∈ D, d' ∈ D, d.source = d'.did }`
   730→
   731→**Rule PROPAGATE-REGION:**
   732→
   733→```
   734→  r ∈ R~    (region r is modified)
   735→  D_r = { d ∈ D | resolveRegion(d) = r }    (all derivations into r)
   736→  A_r = { a ∈ A | a.region = r.rid }        (all accesses into r)
   737→  ────────────────────────────────────────────────────────────────────────
   738→  ∀d ∈ D_r : re-verify bounds(d, r)
   739→  ∀a ∈ A_r : re-verify φ(a)
   740→  Δφ_propagated = { (a.aid, verify(a, R, D, A)) | a ∈ A_r }
   741→```
   742→
   743→If a region's size, status, or RepD changes, all derivations that target the region must be re-verified for bounds, and all accesses must be re-verified for liveness, exclusivity, and interpretation.
   744→
   745→**Rule PROPAGATE-DERIVATION:**
   746→
   747→```
   748→  d ∈ D~    (derivation d is modified)
   749→  D_downstream = { d' ∈ D | d' is transitively derived from d }
   750→  A_downstream = { a ∈ A | a.deriv ∈ D_downstream ∪ {d} }
   751→  ────────────────────────────────────────────────────────────────────────
   752→  ∀d' ∈ D_downstream : recompute offset(d')
   753→  ∀a ∈ A_downstream : re-verify φ(a)
   754→  Δφ_propagated = { (a.aid, verify(a, R, D, A)) | a ∈ A_downstream }
   755→```
   756→
   757→If a derivation's offset or RepD changes, all downstream derivations must have their offsets recomputed, and all downstream accesses must be re-verified. The propagation is transitive: changing `d_0` affects `d_1` (which derives from `d_0`), which affects `d_2` (which derives from `d_1`), and so on.
   758→
   759→**Rule PROPAGATE-ACCESS:**
   760→
   761→```
   762→  a ∈ A~    (access a is modified)
   763→  ────────────────────────────────────────
   764→  Δφ_propagated = { (a.aid, verify(a, R, D, A)) }
   765→```
   766→
   767→An access modification only requires re-verification of that access — it does not affect other parts of the MSG (accesses are leaves in the dependency graph).
   768→
   769→### 5.4 Propagation Termination
   770→
   771→**Theorem (Delta Update Finiteness).** For any single SCG node addition, the delta update and its propagation produce a finite delta, bounded by O(|D_downstream| + |A_affected|) where `D_downstream` is the set of derivations transitively dependent on the modified derivation, and `A_affected` is the set of accesses into modified regions or through modified derivations.
   772→
   773→*Proof sketch.* The dependency graph is a DAG (derivations form chains, accesses are leaves). A modification to node `v` propagates to all nodes reachable from `v` in the dependency graph. Since the graph is finite (bounded by |R| + |D| + |A|), the propagation visits a finite number of nodes. Moreover, the propagation is monotone: each node is visited at most once (re-verification of a node does not modify its dependencies, only its verification status). Therefore, the total propagation work is O(|reachable from v|) which is bounded by |D_downstream| + |A_affected|. ∎
   774→
   775→**Theorem (Delta Update Boundedness).** The size of the delta `|Δ-MSG(n)|` is bounded by a constant for each node type (except `N_Call` with inlining, where it is bounded by the callee's MSG size).
   776→
   777→*Proof.* Directly from the delta computation rules: `N_Alloc` adds one region (constant), `N_Dealloc` modifies one region and re-verifies accesses (bounded by existing MSG structure), `N_Access` adds one access (constant), `N_Cast` and `N_Arith` each add one derivation (constant), `N_Control` splits or merges paths (bounded by number of active paths). The only non-constant case is `N_Call` with inlining, where the delta size equals the callee's MSG size — but this is bounded by the callee's own structure, which is fixed at call time. ∎
   778→
   779→### 5.5 Incremental Path-Sensitive Update
   780→
   781→For the path-sensitive MSG, incremental updates must account for the path structure:
   782→
   783→```
   784→Δ-PS-MSG(n) = { (π, Δ-MSG_π(n)) | π ∈ activePaths(n) }
   785→```
   786→
   787→where `activePaths(n)` is the set of paths that reach node `n` in the SCG. The delta for each path is computed independently, then merged at join points. This ensures that path-sensitive information is maintained incrementally without recomputing the entire PS-MSG.
   788→
   789→**Rule INCREMENTAL-JOIN:**
   790→
   791→```
   792→  Δ-PS-MSG at join node n_j
   793→  {(π₁, Δ₁), (π₂, Δ₂)} computed for incoming paths
   794→  ────────────────────────────────────────────────────────────────────────
   795→  Δ_join = mergeDeltas(Δ₁, Δ₂)
   796→
   797→  where mergeDeltas adds conflicting region statuses to PathDependent
   798→        and merges verification results with glb
   799→```
   800→
   801→### 5.6 Incremental Abstraction Refinement
   802→
   803→When an incremental update produces a `?` verification result, the IVE can **refine the abstraction** locally:
   804→
   805→```
   806→  Δ-MSG(n) produces φ(a) = ?
   807→  ────────────────────────────────────────
   808→  Refine: split the region class containing a.region
   809→          into sub-classes that distinguish the relevant property
   810→          re-verify a with the refined abstraction
   811→```
   812→
   813→This is the incremental version of CEGAR: instead of refining the entire abstract MSG, only the region classes and derivation patterns relevant to the unverified access are refined. The refinement is local and bounded by the size of the affected region class.

   838→## Appendix B: Invariant Summary
   839→
   840→| Invariant | Formal Statement | MSG Mechanism |
   841→|-----------|-----------------|---------------|
   842→| Liveness | `∀a ∈ A : a.programPoint ∈ [r.birth, r.death)` where `r = resolveRegion(a.deriv)` | DEALLOC updates `r.death`; ACCESS checks liveness |
   843→| Exclusivity | `∀a₁, a₂ ∈ A : a₁.mode = Write ∧ a₂.mode ∈ {Read, Write} ∧ sameRegion(a₁, a₂) ∧ concurrent(a₁, a₂) ⇒ a₁ = a₂` | ACCESS checks for conflicting concurrent accesses |
   844→| Interpretation | `∀a ∈ A : bytesAt(a.region, a.offset, a.size) ⊢ a.repd` | CAST checks alignment; ACCESS checks RepD compatibility |
   845→| Origin | `∀d ∈ D : ∃r ∈ R : r = resolveRegion(d)` | CHAIN-ROOT creates root derivation from region |
   846→| Cleanup | `∀r ∈ R : r.status = Freed ∨ r.death = ⊥ ∧ r is marked intentionally leaked` | DEALLOC transitions status; final scan checks for leaks |
