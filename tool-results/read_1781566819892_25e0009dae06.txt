     1→# Multi-Architecture ISA Research for VUMA Codegen
     2→
     3→**VUMA Project — Specification Document**
     4→**Purpose: Research all target ISAs for the multi-architecture compiler backend**
     5→**Date: March 2026**
     6→
     7→---
     8→
     9→## Overview
    10→
    11→This document provides a comprehensive reference for each instruction set architecture that VUMA should target as a multi-architecture compiler. For each ISA, we detail register files, calling conventions, instruction encoding, ELF types, base addresses, key instructions for basic codegen, relevance domains, implementation complexity, and toolchain/ecosystem readiness.
    12→
    13→VUMA's existing codegen (`src/codegen/`) currently targets only **AArch64** (ARM64/ARMv8-A) for the AArch64. The information below is intended to guide the design of a retargetable codegen backend that shares the IR layer (`ir.rs`) and register allocation framework (`regalloc.rs`) while adding per-ISA instruction selection, encoding, and emission modules.

    17→## 1. x86_64 (AMD64 / Intel 64)
    18→
    19→### 1.1 Register File
    20→
    21→| Category | Registers | Count | Notes |
    22→|----------|-----------|-------|-------|
    23→| GP regs | RAX, RCX, RDX, RBX, RSP, RBP, RSI, RDI, R8–R15 | 16 | RSP = stack pointer, RBP = frame pointer |
    24→| SIMD/FP (SSE) | XMM0–XMM15 | 16 (32 with AVX-512) | 128-bit each; YMM (256-bit) and ZMM (512-bit) with AVX/AVX-512 |
    25→| x87 FP | ST(0)–ST(7) | 8 | Legacy; avoid in new code |
    26→| Segment | CS, DS, ES, FS, GS, SS | 6 | FS/GS used for thread-local storage |
    27→| Control/Status | RFLAGS, RIP | 2 | RIP is instruction pointer; RFLAGS has condition codes |
    28→| System | CR0–CR4, DR0–DR7, MSRs | Many | Kernel-only |
    29→
    30→### 1.2 Calling Convention — System V AMD64 ABI (Linux/macOS)
    31→
    32→| Register(s) | Role |
    33→|-------------|------|
    34→| RDI, RSI, RDX, RCX, R8, R9 | Integer/pointer arguments 1–6 |
    35→| XMM0–XMM7 | Floating-point arguments 1–8 |
    36→| RAX | Return value (integer/pointer); also AL = number of vector args for variadics |
    37→| XMM0 | Return value (floating-point) |
    38→| RAX, RDX | 128-bit return (RDX = high half) |
    39→| RBX, RBP, R12–R15 | Callee-saved |
    40→| R10, R11 | Caller-saved (used by syscall too) |
    41→| RSP | 16-byte aligned at call sites (mandatory) |
    42→
    43→**Note:** Windows uses a different convention (Microsoft x64): RCX, RDX, R8, R9 for first 4 args; shadow store required.
    44→
    45→### 1.3 Instruction Encoding
    46→
    47→- **Variable-length:** 1–15 bytes per instruction
    48→- **Endianness:** Little-endian
    49→- **Prefix-based encoding:** REX/VEX/EVEX prefixes extend opcode space
    50→- **Complex encoding:** ModR/M + SIB bytes for addressing modes; immediates vary in size
    51→- **No fixed alignment:** Instructions can start at any byte boundary
    52→
    53→### 1.4 ELF & Binary
    54→
    55→| Property | Value |
    56→|----------|-------|
    57→| ELF machine type | `EM_X86_64` = 62 (0x3E) |
    58→| ELF class | `ELFCLASS64` |
    59→| ELF data | `ELFDATA2LSB` (little-endian) |
    60→| Typical base address (Linux) | `0x400000` (legacy), `0x555555554000` (PIE) |
    61→| Typical page size | 4 KB |
    62→
    63→### 1.5 Key Instructions for Basic Codegen
    64→
    65→| Category | Instructions |
    66→|----------|-------------|
    67→| Arithmetic | `ADD`, `SUB`, `IMUL`, `IDIV`, `INC`, `DEC`, `NEG` |
    68→| Logic | `AND`, `OR`, `XOR`, `NOT`, `SHL`, `SHR`, `SAR` |
    69→| Memory | `MOV`, `LEA`, `PUSH`, `POP`, `MOVSX`, `MOVZX` |
    70→| Control flow | `JMP`, `JE/JNE`, `JL/JG`, `CALL`, `RET`, `CMP`, `TEST` |
    71→| Conditional | `SETcc` (set byte on condition), `CMOVcc` (conditional move) |
    72→| Immediate loads | `MOV reg, imm32/64`; for 64-bit: `MOV reg, imm32` + sign-extend, or `MOVABS reg, imm64` |
    73→
    74→### 1.6 Relevance
    75→
    76→| Domain | Relevant? |
    77→|--------|-----------|
    78→| Embedded | No (too power-hungry) |
    79→| Mobile | No |
    80→| Desktop | **Yes** — dominant |
    81→| Server | **Yes** — dominant |
    82→| Browser | No (not a Wasm target) |
    83→
    84→### 1.7 Complexity: **8/10**
    85→
    86→Variable-length encoding is the primary difficulty. The ModR/M + SIB + displacement + immediate encoding scheme has hundreds of valid combinations. REX prefix handling, the 1–15 byte variable length, and the lack of a simple "template" for instruction encoding make binary emission significantly harder than fixed-length RISC ISAs. However, the massive x86_64 ecosystem (assemblers, disassemblers, emulators) provides excellent reference material.
    87→
    88→### 1.8 Toolchain & Cross-Compilation
    89→
    90→| Property | Status |
    91→|----------|--------|
    92→| Rust target triple | `x86_64-unknown-linux-gnu` (tier 1) |
    93→| Cross-compile from x86_64 Linux | N/A (native) |
    94→| QEMU emulator | `qemu-system-x86_64` (user-mode: `qemu-x86_64`) |
    95→| GNU cross-toolchain | Native; `x86_64-linux-gnu-*` available |
    96→| LLVM support | Excellent (primary target) |

   100→## 2. AArch64 (ARM64 / ARMv8-A)
   101→
   102→### 2.1 Register File
   103→
   104→| Category | Registers | Count | Notes |
   105→|----------|-----------|-------|-------|
   106→| GP regs | X0–X30 | 31 | 64-bit; W0–W30 are lower 32-bit views |
   107→| Special | SP (X31), XZR (X31) | 2 | SP in load/store context; XZR in arithmetic |
   108→| SIMD/FP | V0–V31 | 32 | 128-bit; D0–D31 (64-bit), S0–S31 (32-bit) views |
   109→| System | NZCV, FPCR, FPSR, ELR_EL1, SPSR_EL1 | Many | Condition flags, FP control, exception regs |
   110→| SVE (optional) | Z0–Z31 | 32 | Scalable Vector Extension, 128–2048 bits |
   111→| SVE predicates | P0–P15 | 16 | Predicate registers for SVE |
   112→
   113→### 2.2 Calling Convention — AAPCS64
   114→
   115→| Register(s) | Role |
   116→|-------------|------|
   117→| X0–X7 | Integer/pointer arguments 1–8 and return values |
   118→| X8 | Indirect result location register (large struct returns) |
   119→| X9–X15 | Caller-saved temporaries |
   120→| X16–X17 | Intra-procedure-call scratch (IP0/IP1, PLT veneers) |
   121→| X18 | Platform register (shadow stack on Android) |
   122→| X19–X28 | Callee-saved |
   123→| X29 | Frame pointer (FP) |
   124→| X30 | Link register (LR) |
   125→| V0–V7 | FP/SIMD arguments 1–8 and return values |
   126→| V8–V15 | Callee-saved (lower 64 bits only: D8–D15) |
   127→| V16–V31 | Caller-saved temporaries |
   128→
   129→**VUMA note:** This calling convention is already fully implemented in `src/codegen/src/ir.rs` (`compute_calling_conv`) and `src/codegen/src/arm64.rs`.
   130→
   131→### 2.3 Instruction Encoding
   132→
   133→- **Fixed-length:** 32 bits (4 bytes) for all instructions
   134→- **Endianness:** Little-endian (mandatory on Linux; big-endian possible but unused)
   135→- **Alignment:** Instructions must be 4-byte aligned
   136→- **Clean encoding:** Top-level classification by bits [28:25]; systematic bit-field layout
   137→
   138→### 2.4 ELF & Binary
   139→
   140→| Property | Value |
   141→|----------|-------|
   142→| ELF machine type | `EM_AARCH64` = 183 (0xB7) |
   143→| ELF class | `ELFCLASS64` |
   144→| ELF data | `ELFDATA2LSB` |
   145→| Typical base address (Linux) | `0x400000` |
   146→| Typical base address (bare-metal AArch64) | `0x80000` |
   147→| Typical page size | 4 KB (16 KB on Apple M-series) |
   148→
   149→**VUMA note:** Already implemented in `src/codegen/src/emit.rs` with `EM_AARCH64` constant and both Linux and bare-metal base addresses.
   150→
   151→### 2.5 Key Instructions for Basic Codegen
   152→
   153→| Category | Instructions |
   154→|----------|-------------|
   155→| Arithmetic | `ADD`, `SUB`, `MUL`, `SDIV`, `UDIV`, `MADD`, `MSUB` |
   156→| Logic | `AND`, `ORR`, `EOR`, `LSL`, `LSR`, `ASR`, `ROR` |
   157→| Memory | `LDR`, `STR`, `LDP`, `STP`, `LDRB/H/SW`, `STRB/H` |
   158→| Control flow | `B`, `BL`, `BR`, `BLR`, `RET`, `B.cond`, `CBZ`, `CBNZ`, `TBZ`, `TBNZ` |
   159→| Conditional | `CSEL`, `CSET`, `CSINC` |
   160→| Move | `MOV` (alias for ORR Xd, XZR, Xn), `MOVZ`, `MOVK` |
   161→| Extend | `SXTW`, `SXTB`, `SXTH`, `UXTB`, `UXTH` (aliases for SBFM/UBFM) |
   162→| Atomics | `LDXR`, `STXR`, `LDAXR`, `STLXR`, `CAS`, `LDAR`, `STLR` |
   163→| Barriers | `DMB`, `DSB`, `ISB` |
   164→
   165→**VUMA note:** All of the above are already implemented in `src/codegen/src/arm64.rs` with full binary encoding.
   166→
   167→### 2.6 Relevance
   168→
   169→| Domain | Relevant? |
   170→|--------|-----------|
   171→| Embedded | **Yes** — Cortex-A series |
   172→| Mobile | **Yes** — dominant |
   173→| Desktop | **Yes** — Apple M-series, Snapdragon X |
   174→| Server | **Yes** — AWS Graviton, Ampere Altra |
   175→| Browser | No (not a Wasm target) |
   176→
   177→### 2.7 Complexity: **4/10**
   178→
   179→Fixed-length 32-bit encoding with systematic bit-field layout makes binary emission straightforward. The instruction set is orthogonal and well-documented. The main complexity is the large number of instruction variants (shifted register, extended register, immediate forms), but each has a clean encoding template. This is VUMA's existing target and serves as the reference implementation for retargetable codegen.

   187→| QEMU emulator | `qemu-system-aarch64` (system), `qemu-aarch64` (user-mode) |
   188→| GNU cross-toolchain | `aarch64-linux-gnu-*` widely available |
   189→| LLVM support | Excellent |

   193→## 3. RISC-V 64 (RV64GC)
   194→
   195→### 3.1 Register File
   196→
   197→| Category | Registers | Count | Notes |
   198→|----------|-----------|-------|-------|
   199→| GP regs (x0–x31) | x0 (zero), x1 (ra), x2 (sp), x3 (gp), x4 (tp), x5–x7 (t0–t2), x8–x9 (s0–s1), x10–x17 (a0–a7), x18–x27 (s2–s11), x28–x31 (t3–t6) | 32 | x0 is hardwired zero |
   200→| FP regs (f0–f31) | f0–f7 (ft0–ft7), f8–f9 (fs0–fs1), f10–f17 (fa0–fa7), f18–f27 (fs2–fs11), f28–f31 (ft8–ft11) | 32 | 64-bit (D extension); F extension gives 32-bit single |
   201→| CSR | Various | Hundreds | mstatus, sstatus, mepc, sepc, satp, etc. |
   202→| Vector (V extension) | v0–v31 | 32 | Variable-length (128–4096 bits per reg, configurable) |
   203→
   204→### 3.2 Calling Convention — RV64G ABI (LP64D)
   205→
   206→| Register(s) | Role |
   207→|-------------|------|
   208→| a0–a7 (x10–x17) | Integer/pointer arguments 1–8 and return values (a0–a1) |
   209→| fa0–fa7 (f10–f17) | FP arguments 1–8 and return values (fa0–fa1) |
   210→| ra (x1) | Return address (link register) |
   211→| sp (x2) | Stack pointer |
   212→| gp (x3) | Global pointer (Global addresses) |
   213→| tp (x4) | Thread pointer (TLS) |
   214→| s0–s1 (x8–x9), s2–s11 (x18–x27) | Callee-saved (12 GP regs) |
   215→| fs0–fs1 (f8–f9), fs2–fs11 (f18–f27) | Callee-saved FP (12 FP regs) |
   216→| t0–t6 (x5–x7, x28–x31) | Caller-saved temporaries |
   217→| ft0–ft11 (f0–f7, f28–f31) | Caller-saved FP temporaries |
   218→
   219→**Stack alignment:** 16-byte aligned at call sites.
   220→
   221→### 3.3 Instruction Encoding
   222→
   223→- **Variable-length:** Base ISA is 32-bit fixed; RVC (compressed) extension adds 16-bit encodings
   224→- **Endianness:** Little-endian (dominant); big-endian also specified
   225→- **Alignment:** 32-bit instructions are 4-byte aligned; 16-bit (RVC) are 2-byte aligned
   226→- **Base ISA encoding:** 4 major formats: R-type, I-type, S-type, U-type; plus B-type (branch), J-type (jump)
   227→- **RVC encoding:** 8 major compressed formats; ~40 common instructions have 16-bit forms
   228→
   229→### 3.4 ELF & Binary
   230→
   231→| Property | Value |
   232→|----------|-------|
   233→| ELF machine type | `EM_RISCV` = 243 (0xF3) |
   234→| ELF class | `ELFCLASS64` |
   235→| ELF data | `ELFDATA2LSB` (LE) or `ELFDATA2MSB` (BE) |
   236→| ELF flags | Specify ISA extensions: `EF_RISCV_RVC`, `EF_RISCV_FLOAT_ABI_SOFT/SINGLE/DOUBLE/QUAD` |
   237→| Typical base address (Linux) | `0x10000` (default), `0x400000` (some distros) |
   238→| Typical page size | 4 KB |
   239→
   240→### 3.5 Key Instructions for Basic Codegen
   241→
   242→| Category | Instructions |
   243→|----------|-------------|
   244→| Arithmetic | `ADD`, `SUB`, `ADDI`, `MUL`, `DIV`, `DIVU`, `REM`, `REMU` (M extension) |
   245→| Logic | `AND`, `OR`, `XOR`, `ANDI`, `ORI`, `XORI` |
   246→| Shift | `SLL`, `SRL`, `SRA`, `SLLI`, `SRLI`, `SRAI` |
   247→| Memory | `LD`, `SD`, `LW`, `SW`, `LH`, `SH`, `LB`, `SB`, `LWU`, `LHU` |
   248→| Address | `LUI`, `AUIPC` (upper immediate for PC-relative addressing) |
   249→| Control flow | `JAL`, `JALR`, `BEQ`, `BNE`, `BLT`, `BGE`, `BLTU`, `BGEU` |
   250→| Compare | `SLT`, `SLTU`, `SLTI`, `SLTIU` (set on less-than) |
   251→| FP (D ext) | `FADD.D`, `FSUB.D`, `FMUL.D`, `FDIV.D`, `FCVT.D.S`, `FCVT.S.D`, `FMV.X.D`, `FMV.D.X` |
   252→| Extend | `SLLI` for zero-extend; `SRAI` for sign-extend |
   253→| Atomics (A ext) | `LR.D`, `SC.D`, `AMOSWAP.D`, `AMOADD.D`, etc. |
   254→| Fence | `FENCE`, `FENCE.I` |
   255→
   256→### 3.6 Relevance
   257→
   258→| Domain | Relevant? |
   259→|--------|-----------|
   260→| Embedded | **Yes** — rapidly growing (SiFive, StarFive) |
   261→| Mobile | Emerging (not yet mainstream) |
   262→| Desktop | Emerging (SiFive HiFive, RISC-V laptops) |
   263→| Server | Emerging (datacenter SoCs in development) |
   264→| Browser | No |
   265→
   266→### 3.7 Complexity: **3/10**
   267→
   268→RISC-V is arguably the easiest 64-bit ISA to implement a codegen for. The base ISA (RV64I) has only ~50 instructions with 4 clean encoding formats. No complex addressing modes — all memory ops are register + 12-bit signed immediate. The variable-length RVC extension adds moderate complexity but is optional. The main gotcha is that some operations require multi-instruction sequences (e.g., 32-bit immediate loads use LUI+ADDI; function calls may need AUIPC+JALR for range).

   276→| QEMU emulator | `qemu-system-riscv64` (system), `qemu-riscv64` (user-mode) |
   277→| GNU cross-toolchain | `riscv64-linux-gnu-*` available |
   278→| LLVM support | Good (RISC-V is a first-class LLVM target since LLVM 9) |

   282→## 4. WebAssembly (Wasm32 + Wasm64)
   283→
   284→### 4.1 Register File
   285→
   286→WebAssembly is a **stack machine**, not a register machine. There are no named registers.
   287→
   288→| Category | Description |
   289→|----------|-------------|
   290→| Value stack | Operand stack; implicitly popped/pushed by instructions |
   291→| Local variables | Per-function locals (typed); accessed by index; effectively infinite |
   292→| Global variables | Per-module globals (mutable or immutable); accessed by index |
   293→| Table | Indirect function references (for dynamic dispatch) |
   294→| Memory | Linear memory (byte-addressable array buffer); 1+ pages (64 KB each) |
   295→| PC | Program counter; implicit |
   296→
   297→**Wasm64 vs Wasm32:** Wasm64 uses 64-bit addresses/indices (pointers are i64) while Wasm32 uses 32-bit (pointers are i32). Everything else is identical.
   298→
   299→### 4.2 Calling Convention
   300→
   301→WebAssembly has no ABI in the traditional sense — the stack machine model abstracts it away:
   302→
   303→| Property | Description |
   304→|----------|-------------|
   305→| Arguments | Pushed on the value stack before `call`; callee receives them as locals |
   306→| Return values | Pushed on the value stack before `end`/`return`; callee's results |
   307→| Callee-saved | N/A — locals are per-function; there are no shared registers |
   308→| Stack | The WebAssembly value stack is separate from the "shadow stack" in linear memory |
   309→| Multi-value | Wasm MVP supports 1 return value; Multi-value extension allows multiple |
   310→
   311→**Interface to host:** When Wasm calls imported functions, the Wasm-to-native ABI is defined by the embedder (wasmi, Wasmtime, etc.). The standard Wasm C ABI (wasm32-wasi) maps to the stack-based calling convention with shadow stack.
   312→
   313→### 4.3 Instruction Encoding
   314→
   315→- **Variable-length:** 1+ bytes per instruction; LEB128-encoded immediates
   316→- **Endianness:** Little-endian for all multi-byte values
   317→- **Binary format:** Structured as a module with sections (Type, Import, Function, Table, Memory, Global, Export, Start, Element, Code, Data)
   318→- **Code section:** Sequence of functions; each function body is a byte vector
   319→- **Alignment:** No alignment requirement for instructions (byte-addressable)
   320→
   321→### 4.4 ELF & Binary
   322→
   323→| Property | Value |
   324→|----------|-------|
   325→| Format | **Not ELF** — `.wasm` binary format (magic: `\0asm`, version 1) |
   326→| Object format | `.wasm` files; or `*.o` with Wasm object format for linking |
   327→| No ELF machine type | N/A |
   328→| Base address | N/A — linear memory starts at address 0 by default |
   329→| Page size | 64 KB (minimum memory unit) |
   330→
   331→### 4.5 Key Instructions for Basic Codegen
   332→
   333→| Category | Instructions |
   334→|----------|-------------|
   335→| Arithmetic (i32) | `i32.add`, `i32.sub`, `i32.mul`, `i32.div_s`, `i32.div_u`, `i32.rem_s`, `i32.rem_u` |
   336→| Arithmetic (i64) | `i64.add`, `i64.sub`, `i64.mul`, `i64.div_s`, `i64.div_u`, `i64.rem_s`, `i64.rem_u` |
   337→| Logic | `i32.and`, `i32.or`, `i32.xor`, `i32.shl`, `i32.shr_u`, `i32.shr_s`, `i32.rotl`, `i32.rotr` |
   338→| FP | `f32.add`, `f32.sub`, `f32.mul`, `f32.div`, `f64.add`, etc. |
   339→| Memory | `i32.load`, `i32.store`, `i64.load`, `i64.store`, `i32.load8_u`, `i32.store8`, etc. |
   340→| Constants | `i32.const`, `i64.const`, `f32.const`, `f64.const` (LEB128-encoded) |
   341→| Control flow | `block`, `loop`, `if`/`else`/`end`, `br`, `br_if`, `br_table`, `return` |
   342→| Call | `call`, `call_indirect` |
   343→| Local | `local.get`, `local.set`, `local.tee` |
   344→| Global | `global.get`, `global.set` |
   345→| Compare | `i32.eq`, `i32.ne`, `i32.lt_s`, `i32.lt_u`, `i32.gt_s`, `i32.gt_u`, `i32.le_s`, `i32.ge_s`, etc. |
   346→| Convert | `i32.wrap_i64`, `i64.extend_i32_s`, `i64.extend_i32_u`, `i32.trunc_f64_s`, etc. |
   347→| Memory size/grow | `memory.size`, `memory.grow` |
   348→
   349→### 4.6 Relevance
   350→
   351→| Domain | Relevant? |
   352→|--------|-----------|
   353→| Embedded | Emerging (WASM micro runtimes) |
   354→| Mobile | Emerging |
   355→| Desktop | Emerging (WASM outside browser) |
   356→| Server | Emerging (WASM edge computing, Fermyon, Fastly) |
   357→| Browser | **Yes** — dominant (the only ISA that runs natively in browsers) |
   358→
   359→### 4.7 Complexity: **2/10**
   360→
   361→WebAssembly is by far the easiest target to implement. No register allocation needed (stack machine). No binary encoding complexity (LEB128 + simple opcodes). No calling convention to manage. The main work is: (1) emitting structured control flow (blocks/loops with branch labels), (2) managing the value stack depth, and (3) generating the module structure with sections. A basic Wasm32 codegen can be implemented in ~1000 lines of Rust.
   362→
   363→### 4.8 Toolchain & Cross-Compilation
   364→
   365→| Property | Status |
   366→|----------|--------|
   367→| Rust target triple | `wasm32-unknown-unknown` (tier 2), `wasm32-wasip1` (tier 2), `wasm32-wasip2` (tier 3) |
   368→| Cross-compile from x86_64 Linux | **Yes** — `rustup target add wasm32-unknown-unknown` |
   369→| Emulator/runtime | Wasmtime, Wasmer, wasmi, Node.js, Chrome/Firefox, WasmEdge |
   370→| Toolchain | `wasm-ld` (LLD), `wasm-opt`, `wasm2wat`, `wat2wasm` |
   371→| LLVM support | Excellent (primary Wasm backend) |

   375→## 5. ARM32 (AArch32 / ARMv7-A)
   376→
   377→### 5.1 Register File
   378→
   379→| Category | Registers | Count | Notes |
   380→|----------|-----------|-------|-------|
   381→| GP regs | R0–R12, SP (R13), LR (R14), PC (R15) | 16 | 32-bit each |
   382→| SIMD/FP (VFPv3/NEON) | D0–D31 (or S0–S63) | 32×64-bit | D0–D15 = S0–S31; D16–D31 = extended (VFPv3-D32) |
   383→| NEON | Q0–Q15 | 16×128-bit | Q0 = D0+D1, etc. |
   384→| CPSR | Current Program Status Register | 1 | N, Z, C, V flags + mode bits |
   385→| Banked | R8_fiq–R14_fiq, R13_svc, R14_svc, etc. | Per-mode | FIQ, SVC, ABT, IRQ, UND modes |
   386→
   387→### 5.2 Calling Convention — AAPCS (ARM32)
   388→
   389→| Register(s) | Role |

   534→| Server | No (historically SGI) |

   537→### 6.7 Complexity: **5/10**
   538→
   539→MIPS64 has a clean 32-bit fixed-length encoding and orthogonal instruction set. The main complications are: (1) **branch delay slots** — every branch/jump instruction is followed by an instruction that always executes, requiring the codegen to fill (or NOP) the slot, (2) multiply/divide results go to HI/LO registers rather than a destination register, requiring MFLO/MFHI to retrieve results, (3) no direct condition codes — comparisons write to a register (SLT/SLTU), (4) address construction requires LUI + DADDIU sequences for large constants.

   726→### 8.7 Complexity: **7/10**
   727→
   728→PowerPC64 has several features that complicate codegen: (1) **Condition Register (CR)** — 8 independent 4-bit CR fields; comparisons target a specific CR field; branches test CR fields; CR logical operations are available; (2) **TOC (Table of Contents)** — function prologues must set up R2 (TOC pointer) using a 2-instruction sequence from R12; (3) **rlwinm family** — bit-field operations use a complex rotate-left-then-mask encoding with mask start/end bits; (4) **Count Register (CTR)** — used for computed calls and decrement-and-branch loops; (5) **Function descriptors** (ELFv1 only); (6) **Little-endian quirks** — some bit-numbering conventions differ from other LE ISAs. The ELFv2 ABI is significantly cleaner than ELFv1.

   769→## Implementation Priority Recommendation
   770→
   771→Based on the combination of relevance, complexity, and ecosystem readiness, the recommended implementation order for VUMA's multi-arch codegen is:
   772→
   773→### Tier 1 — Immediate (leverage existing AArch64 codegen patterns)
   774→1. **AArch64** — Already implemented ✅
   775→2. **RISC-V64** — Cleanest RISC ISA; fixed 32-bit encoding; 3/10 complexity; growing ecosystem
   776→3. **WebAssembly (Wasm32)** — Simplest target; stack machine; 2/10 complexity; browser reach
   777→
   778→### Tier 2 — Near-term (medium effort, high value)
   779→4. **LoongArch64** — Very similar to RISC-V; 3/10 complexity; China market
   780→5. **x86_64** — Highest effort but largest installed base; 8/10 complexity; essential for server/desktop
   781→
   782→### Tier 3 — Medium-term (established ISAs, moderate complexity)
   783→6. **ARM32** — Legacy but still important for embedded; 6/10 complexity
   784→7. **MIPS64** — Branch delay slots add complexity; 5/10 complexity; embedded routers
   785→
   786→### Tier 4 — Later (specialized, high complexity)
   787→8. **PowerPC64** — Smallest market; highest complexity among RISC ISAs (7/10); CR manipulation; TOC overhead

   791→## Retargetable Codegen Architecture Recommendations
   792→
   793→Based on this research, VUMA's codegen should be refactored to support multiple backends with shared infrastructure:
   794→
   795→```
   796→IR (target-independent)
   797→    │
   798→    ├── Instruction Selection (per-ISA)
   799→    │     ├── arm64/isel.rs      (existing)
   800→    │     ├── riscv64/isel.rs    (new)
   801→    │     ├── wasm32/isel.rs     (new)
   802→    │     ├── x86_64/isel.rs     (new)
   803→    │     └── ...
   804→    │
   805→    ├── Register Allocation (shared framework)
   806→    │     ├── RegClass (ISA-specific pools)
   807→    │     ├── LinearScanAllocator (parameterized by ISA)
   808→    │     └── Spill/Reload (ISA-specific instructions)
   809→    │
   810→    ├── Binary Encoding (per-ISA)
   811→    │     ├── arm64/encode.rs    (existing)
   812→    │     ├── riscv64/encode.rs  (new)
   813→    │     ├── wasm32/encode.rs   (new)
   814→    │     ├── x86_64/encode.rs   (new)
   815→    │     └── ...
   816→    │
   817→    └── Emission (per-ISA ELF/binary format)
   818→          ├── arm64/emit.rs      (existing)
   819→          ├── riscv64/emit.rs    (new)
   820→          ├── wasm32/emit.rs     (new — .wasm format, not ELF)
   821→          ├── x86_64/emit.rs     (new)
   822→          └── ...
   823→```
   824→
   825→Key shared abstractions:
   826→- **`TargetDef` trait**: defines register counts, argument conventions, ELF machine type, base address, encoding width
   827→- **`InstructionSelector` trait**: maps IR nodes to target instructions
   828→- **`RegisterAllocator` trait**: parameterized by `TargetDef` for register pool sizes
   829→- **`Encoder` trait**: converts target instructions to bytes
   830→- **`Emitter` trait**: produces final binary output (ELF, .wasm, raw)
   831→
