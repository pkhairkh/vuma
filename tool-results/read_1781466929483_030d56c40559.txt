   176‚Üí
   177‚Üí1. **Register Definitions**

   182‚Üí

   187‚Üí   - `emit_rexw_reg_reg()` ‚Äî Common pattern for 64-bit reg-reg ALU ops
   188‚Üí   - `emit_rexw_opext_reg()` ‚Äî Common pattern for opcode-extension + reg ops
   189‚Üí
   190‚Üí3. **Instruction Encoding** (all producing exact bytes a real x86_64 CPU executes)

   216‚Üí   - `trampoline()` ‚Äî mov rax, imm64; jmp rax (12 bytes)

   315‚Üí### Summary

   317‚ÜíCreated `/home/z/my-project/vuma/src/codegen/src/control_flow.rs` ‚Äî a 2,616-line module that handles complex control flow lowering for ARM64 codegen. The module translates high-level control flow patterns into IR-level representations that the emitter can process.
   318‚Üí
   319‚Üí### Components Implemented
   320‚Üí
   321‚Üí1. **SwitchLowerer** (~300 lines)
   322‚Üí   - `choose_strategy()` ‚Äî selects between JumpTable, BinarySearch, and IfElseChain based on target count and density
   323‚Üí   - `lower_switch()` ‚Äî dispatches to the chosen strategy
   324‚Üí   - `lower_jump_table()` ‚Äî computes adjusted index, bounds check, sequential equality comparisons simulating table lookup
   325‚Üí   - `lower_binary_search()` ‚Äî recursive O(log n) partitioning with median comparisons
   326‚Üí   - `lower_if_else_chain()` ‚Äî linear equality comparison chain for few targets
   327‚Üí   - `is_dense_range()` ‚Äî checks if range/count ratio is below density threshold
   328‚Üí
   329‚Üí2. **ExceptionLowerer** (~250 lines)
   330‚Üí   - `LandingPad`, `ExceptionAction`, `ExceptionTableEntry` types
   331‚Üí   - `lower_invoke()` ‚Äî produces call block + landing pad block from an Invoke terminator
   332‚Üí   - `generate_exception_table()` ‚Äî walks function blocks to produce .gcc_except_table entries
   333‚Üí   - `collect_landing_pads()` ‚Äî scans for all Invoke terminators and builds LandingPad list
   334‚Üí
   335‚Üí3. **TailCallLowerer** (~200 lines)
   336‚Üí   - `is_tail_call_eligible()` ‚Äî checks return-value match, no allocas, register arg count, no invokes
   337‚Üí   - `lower_tail_call()` ‚Äî generates argument shuffle instructions with conflict detection
   338‚Üí   - `make_tail_call_terminator()` ‚Äî convenience constructor for TailCall terminator
   339‚Üí

   721‚Üí
   722‚Üí### Key Changes in mips64.rs
   723‚Üí
   724‚Üí1. **`lower_ir_instr` now accepts `frame_size: usize`** ‚Äî used by the Ret
   725‚Üí   handler to generate a proper function epilogue that restores $ra from the
   726‚Üí   stack and deallocates the frame.
   727‚Üí
   728‚Üí2. **`Ret` handler now emits full epilogue** ‚Äî Previously, Ret only emitted
   729‚Üí   `jr $ra; nop`, which is incorrect when the prologue saves $ra and adjusts
   730‚Üí   $sp. Now it emits:
   731‚Üí   - `ld $ra, frame_size-8($sp)` ‚Äî restore saved return address
   732‚Üí   - `daddiu $sp, $sp, frame_size` ‚Äî deallocate stack frame
   733‚Üí   - `jr $ra` ‚Äî return
   734‚Üí   - `nop` ‚Äî delay slot
   735‚Üí
   736‚Üí3. **`Free` handler now emits `break 0xFF`** instead of NOP ‚Äî Since Free is
   737‚Üí   heap deallocation that should be lowered to a runtime call, emitting a
   738‚Üí   `break` instruction with code 0xFF traps if accidentally executed, which is
   739‚Üí   safer than silently falling through.
   740‚Üí
   741‚Üí4. **`Phi` handler separated from `Free`** ‚Äî Phi nodes get their own match arm
   742‚Üí   with a comment explaining they should be eliminated by SSA deconstruction
   743‚Üí   before instruction selection.
   744‚Üí
   745‚Üí5. **`allocate_registers` passes `frame_size`** to `lower_ir_instr` so the
   746‚Üí   Ret handler can emit the correct epilogue.

   791‚Üí

   797‚ÜíAdded a cargo-fuzz style fuzzing harness for the VUMA parser, plus fixed an
   798‚Üíinfinite loop bug discovered during testing. The fuzzer generates semi-structured
   799‚ÜíVUMA-like source strings using the `arbitrary` crate approach and feeds them
   800‚Üíto `Parser::new(input).parse_program()`, verifying the parser never panics.
   801‚Üí32 edge-case tests were added covering deep nesting, Unicode, long identifiers,
   802‚Üíoperator soup, empty programs, keyword edge cases, and more. The fuzzer ran for
   803‚Üí1500 iterations with zero panics.
   804‚Üí
   805‚Üí### Bug Fixed
   806‚Üí
   807‚Üí**Infinite loop in `recover_to_item_boundary`**: When the parser encountered a
   808‚Üí`#` token not followed by `[` (e.g., `###$$$@@@`), `recover_to_item_boundary`
   809‚Üíwould break immediately because `TokenKind::Hash` is in `ITEM_STARTERS`,
   810‚Üícausing `parse_program`'s loop to retry the same token indefinitely. Fixed by
   811‚Üíchecking that `#` is followed by `[` before treating it as an item starter in
   812‚Üíthe recovery path.

   822‚Üí
   823‚Üí### Fuzzing Harness Design
   824‚Üí
   825‚ÜíThe fuzzer (`parse_program`) uses a `FuzzInput` struct that reads from a byte
   826‚Üíbuffer to make generation decisions. It generates VUMA-like source strings with:
   827‚Üí
   828‚Üí- **Item generators**: region, fn, struct, enum, let, assign, import, export,
   829‚Üí  const, free, expression statements, raw keyword injection
   830‚Üí- **Statement generators**: if/while/for/match/return/sync/bd-directive/expr
   831‚Üí- **Expression generators**: binary/unary ops, calls, grouping, allocate,
   832‚Üí  field access, casts, sizeof/alignof, derive, indexing
   833‚Üí- **Type generators**: named, pointer, array, generic, function types
   834‚Üí- **Atom generators**: integers (decimal/hex/binary/octal), identifiers,
   835‚Üí  booleans, null, string literals
   836‚Üí
   837‚ÜíEach fuzz iteration:
   838‚Üí1. Generates 16‚Äì512 random bytes
   839‚Üí2. Converts bytes into a VUMA-like source string via `FuzzInput`
   840‚Üí3. Feeds it to `Parser::new(input).parse_program()`
   841‚Üí4. Uses `panic::catch_unwind` to verify no panics
   842‚Üí
