     2в†’     2в†’

     4в†’     4в†’
     5в†’     5в†’This document is the definitive reference for the VUMA programming language, a memory-oriented, verification-first systems language targeting 8 backend architectures (AArch64, x86_64, RISC-V 64, ARM32, MIPS64, PPC64, LoongArch64, Wasm32). VUMA eliminates the need for `unsafe` blocks by making the Invariant Verification Engine (IVE) an integral part of compilation. Every pointer dereference, allocation, and free is automatically checked against five core invariants: **Liveness**, **Exclusivity**, **Interpretation**, **Origin**, and **Cleanup** (collectively "LIVE").

    26в†’    26в†’## 1. Lexical Structure
    27в†’    27в†’
    28в†’    28в†’The VUMA lexer transforms raw source text into a flat stream of tokens, each annotated with byte-offset spans and line/column positions for error reporting. The lexer supports full error recovery: it never stops on the first error, producing `Error` tokens and collecting diagnostic information that can be retrieved later. Whitespace and non-doc comments are silently skipped; doc comments (`///` and `//!`) are emitted as tokens for downstream tooling.
    29в†’    29в†’
    30в†’    30в†’### Keywords
    31в†’    31в†’
    32в†’    32в†’VUMA reserves the following keywords. They cannot be used as identifiers unless escaped. The keyword set is organized into six categories reflecting the language's design priorities:
    33в†’    33в†’
    34в†’    34в†’**Core language keywords:** `fn`, `let`, `if`, `else`, `while`, `for`, `return`, `as`, `match`, `struct`, `enum`, `loop`
    35в†’    35в†’
    36в†’    36в†’**Memory primitive keywords:** `ptr`, `region`, `alloc`, `allocate`, `free`, `derive`, `cast`, `read`, `write`
    37в†’    37в†’
    38в†’    38в†’**Concurrency and synchronization keywords:** `sync`, `async`, `await`, `spawn`, `lock`, `unlock`, `channel`, `send`, `recv`
    39в†’    39в†’
    40в†’    40в†’**Safety keywords:** `unsafe`, `safe`
    41в†’    41в†’
    42в†’    42в†’**Behavioral domain directive keywords:** `bd`, `repd`, `capd`, `reld`
    43в†’    43в†’
    44в†’    44в†’**Module system keywords:** `import`, `export`, `mod`, `use`, `self`, `super`
    45в†’    45в†’
    46в†’    46в†’**Literal keywords:** `true`, `false`
    47в†’    47в†’
    48в†’    48в†’**Type operator keywords:** `sizeof`, `alignof`

   112в†’   112в†’## 2. Types and BD Annotations
   113в†’   113в†’
   114в†’   114в†’VUMA's type system is built around the concept of **Behavioral Descriptors (BDs)**, which provide a complete specification of a value along three orthogonal axes. Every value in VUMA carries a BD that the Invariant Verification Engine (IVE) uses to prove memory safety without requiring an `unsafe` escape hatch. The type system covers primitive types, pointer types, region-annotated types, struct types, generic type applications, function types, and explicit BD annotations.
   115в†’   115в†’
   116в†’   116в†’### The BD Triple: RepD Г— CapD Г— RelD
   117в†’   117в†’
   118в†’   118в†’A Behavioral Descriptor is the composition of three sub-descriptors:
   119в†’   119в†’
   120в†’   120в†’```
   121в†’   121в†’BD = RepD Г— CapD Г— RelD
   122в†’   122в†’```
   123в†’   123в†’
   124в†’   124в†’Two BDs are **compatible** when all three layers are pairwise compatible. One BD **refines** another when every layer is at least as specific:
   125в†’   125в†’
   126в†’   126в†’```
   127в†’   127в†’bd1 вЉ‘ bd2  вџє  bd1.repd.subsumes(bd2.repd)
   128в†’   128в†’               в€§ bd1.capd.is_subset(bd2.capd)
   129в†’   129в†’               в€§ bd1.reld.refines(bd2.reld)
   130в†’   130в†’```
   131в†’   131в†’
   132в†’   132в†’### RepD вЂ” Representation Descriptor
   133в†’   133в†’
   134в†’   134в†’RepD describes the memory shape, size, and alignment of a value. It forms a subsumption lattice where more-specific RepDs subsume less-specific ones. The `Byte` variant is the universal bottom element that is compatible with everything.
   135в†’   135в†’
   136в†’   136в†’```vuma
   137в†’   137в†’// RepD variants (from the bd crate):
   138в†’   138в†’//   Byte(size, align)   вЂ” raw byte sequence, compatible with anything
   139в†’   139в†’//   Struct { fields }   вЂ” named field layout
   140в†’   140в†’//   Array { elem, len } вЂ” homogeneous fixed-size array
   141в†’   141в†’//   Enum { variants }   вЂ” tagged union
   142в†’   142в†’//   Ptr(pointee)        вЂ” pointer representation
   143в†’   143в†’//   Union { variants }  вЂ” untagged union
   144в†’   144в†’//   Func { params, ret }вЂ” function pointer representation
   145в†’   145в†’
   146в†’   146в†’// In source code, RepD is inferred from type annotations:
   147в†’   147в†’let x: i32 = 42;           // RepD: Byte(4, 4)
   148в†’   148в†’let p: Address = allocate(8); // RepD: Ptr(Byte(8, 8))
   149в†’   149в†’```
   150в†’   150в†’
   151в†’   151в†’### CapD вЂ” Capability Descriptor
   152в†’   152в†’
   153в†’   153в†’CapD specifies what operations are permitted on a value. It forms a lattice where the **meet** (intersection of capabilities, union of conditions) and **join** (union of capabilities, intersection of conditions) operations are well-defined. Capabilities can be conditioned on holding a specific lock, enabling fine-grained concurrency reasoning.
   154в†’   154в†’
   155в†’   155в†’```vuma
   156в†’   156в†’// CapD capabilities (from the bd crate):
   157в†’   157в†’//   Read     вЂ” the value may be read
   158в†’   158в†’//   Write    вЂ” the value may be written
   159в†’   159в†’//   Execute  вЂ” the value may be executed as code
   160в†’   160в†’//   Derive   вЂ” new pointers may be derived from this value
   161в†’   161в†’
   162в†’   162в†’// CapD can be conditioned on lock acquisition:
   163в†’   163в†’//   CapD { read: true, write: true, write_requires_lock: 42 }
   164в†’   164в†’//   This means write is only permitted when lock #42 is held.
   165в†’   165в†’
   166в†’   166в†’// CapD lattice operations:
   167в†’   167в†’//   meet(a, b) = intersection of caps, union of conditions (more restrictive)
   168в†’   168в†’//   join(a, b) = union of caps, intersection of conditions (less restrictive)
   169в†’   169в†’
   170в†’   170в†’// Explicit BD annotation syntax:
   171в†’   171в†’#bd(ReadOnly)    // CapD with only Read capability
   172в†’   172в†’#bd(ReadWrite)   // CapD with Read and Write capabilities
   173в†’   173в†’#bd(LockedWrite) // CapD where Write requires a specific lock
   174в†’   174в†’```
   175в†’   175в†’
   176в†’   176в†’### RelD вЂ” Relational Descriptor
   177в†’   177в†’
   178в†’   178в†’RelD captures temporal, dependency, and security relationships that a value participates in. It ensures that constraints such as lifetime ordering, information flow policies, and dependency tracking are preserved across operations. RelD composition is the union of relations, and consistency checking ensures that contradictory constraints (e.g., `Outlives` + `Succeeds`) are rejected.
   179в†’   179в†’
   180в†’   180в†’```vuma
   181в†’   181в†’// RelD relation types (from the bd crate):
   182в†’   182в†’//   Liveness      вЂ” value is guaranteed to be live
   183в†’   183в†’//   Outlives(a,b) вЂ” region a outlives region b
   184в†’   184в†’//   DependsOn(a,b)вЂ” computation depends on value b
   185в†’   185в†’//   Succeeds(a,b) вЂ” event a happens after event b
   186в†’   186в†’//   FlowPolicy(p) вЂ” information-flow policy label p
   187в†’   187в†’
   188в†’   188в†’// RelD is usually inferred, but can be annotated:
   189в†’   189в†’// #bd(Exclusive) carries an implicit RelD with Liveness
   190в†’   190в†’```
   191в†’   191в†’
   192в†’   192в†’### Type Syntax

   242в†’   242в†’## 3. Memory Model
   243в†’   243в†’
   244в†’   244в†’VUMA's memory model is built around four first-class operations: **allocate**, **write**, **read**, and **free**. Unlike languages where these are library calls, VUMA makes them primitive statements that the Invariant Verification Engine (IVE) automatically checks. Every pointer operation is verified against the five LIVE invariants, and there is no `unsafe` keyword to bypass verification. This section describes regions, allocations, derivations, and accesses вЂ” the core concepts that define how VUMA programs interact with memory.
   245в†’   245в†’
   246в†’   246в†’### Regions
   247в†’   247в†’
   248в†’   248в†’A **region** is a contiguous block of memory created by `allocate(size)` or `map_device(base, size)`. Regions are identified by their `Address` value, an opaque 64-bit handle. The IVE tracks the lifetime and bounds of every region, ensuring that all accesses fall within the region's allocated range.
   249в†’   249в†’
   250в†’   250в†’```vuma
   251в†’   251в†’// Allocate a region for one 64-bit integer
   252в†’   252в†’region = allocate(8);
   253в†’   253в†’
   254в†’   254в†’// Allocate a region large enough for a struct
   255в†’   255в†’node = allocate(24);  // sizeof(NodeHeader) = 24
   256в†’   256в†’
   257в†’   257в†’// Map a hardware device region (never freed, volatile semantics)
   258в†’   258в†’gpio = map_device(0x7e200000, 4096);
   259в†’   259в†’```
   260в†’   260в†’
   261в†’   261в†’Regions can be named using the `region` keyword for clarity:
   262в†’   262в†’
   263в†’   263в†’```vuma
   264в†’   264в†’region pool = allocate(4096);  // named region declaration
   265в†’   265в†’```
   266в†’   266в†’
   267в†’   267в†’### Allocations
   268в†’   268в†’
   269в†’   269в†’The `allocate(size)` statement reserves `size` bytes of memory and returns an `Address`. The IVE verifies at compile time that:
   270в†’   270в†’
   271в†’   271в†’1. **Liveness:** the returned address is live immediately after allocation
   272в†’   272в†’2. **Interpretation:** the allocation size is a positive, known value
   273в†’   273в†’3. **Cleanup:** every allocation is eventually freed (or mapped as a device that never needs freeing)
   274в†’   274в†’
   275в†’   275в†’```vuma
   276в†’   276в†’// Simple allocation
   277в†’   277в†’p = allocate(8);        // 8 bytes for a u64
   278в†’   278в†’*p = 42;                // write to the region
   279в†’   279в†’value: u64 = *p;        // read from the region
   280в†’   280в†’free(p);                // release the region
   281в†’   281в†’
   282в†’   282в†’// Arena-style allocation
   283в†’   283в†’arena = allocate(4096); // one large block
   284в†’   284в†’a = arena + 0;          // offset 0 within arena
   285в†’   285в†’b = arena + 64;         // offset 64 within arena
   286в†’   286в†’// IVE knows a and b are within the arena's 4096-byte range
   287в†’   287в†’free(arena);            // frees the entire arena; a and b are now dead
   288в†’   288в†’```
   289в†’   289в†’
   290в†’   290в†’### Derivations
   291в†’   291в†’
   292в†’   292в†’A **derivation** is the process of creating a new pointer from an existing one. VUMA tracks the full derivation chain so the IVE knows which original allocation a derived pointer comes from. This is critical for proving that offsets stay within bounds and that freeing a base allocation invalidates all derived pointers.
   293в†’   293в†’
   294в†’   294в†’```vuma
   295в†’   295в†’// Pointer arithmetic derives a new address
   296в†’   296в†’base = allocate(1024);
   297в†’   297в†’offset_ptr = base + 128;     // derived: base + 128
   298в†’   298в†’// IVE knows offset_ptr is within base's 1024-byte allocation
   299в†’   299в†’
   300в†’   300в†’// The derive expression is explicit:
   301в†’   301в†’derived = derive(ptr, region);  // formal derivation with region annotation
   302в†’   302в†’
   303в†’   303в†’// Arena allocation uses derivation chains:
   304в†’   304в†’current = (*arena).base + (*arena).offset;  // derived from arena.base
   305в†’   305в†’aligned = current.align_to(16);             // further derivation with alignment
   306в†’   306в†’// IVE tracks: arena.base в†’ current в†’ aligned
   307в†’   307в†’```
   308в†’   308в†’
   309в†’   309в†’When the base allocation is freed, the IVE marks **all** derived pointers as invalid. This prevents the most common bug in arena-based code: using a pointer after its arena has been destroyed.
   310в†’   310в†’
   311в†’   311в†’```vuma
   312в†’   312в†’arena = arena_create(4096);
   313в†’   313в†’a = arena_alloc(arena, 64, 8);
   314в†’   314в†’b = arena_alloc(arena, 128, 16);
   315в†’   315в†’arena_destroy(arena);
   316в†’   316в†’// IVE: a and b are now dead pointers
   317в†’   317в†’// x = *a;  // ERROR: use of dead pointer `a`
   318в†’   318в†’```
   319в†’   319в†’
   320в†’   320в†’### Accesses
   321в†’   321в†’
   322в†’   322в†’A **memory access** is a read or write through a dereferenced pointer. Every access is checked by the IVE against the five invariants. The IVE tracks byte-level granularity, so it can prove that two accesses to different fields of the same struct do not conflict вЂ” this is how VUMA verifies doubly-linked list operations that Rust's borrow checker rejects.
   323в†’   323в†’
   324в†’   324в†’```vuma
   325в†’   325в†’// Write access вЂ” stores a value into a region
   326в†’   326в†’*region = 42;
   327в†’   327в†’(*node).next = sentinel;
   328в†’   328в†’(*last).prev = node;
   329в†’   329в†’
   330в†’   330в†’// Read access вЂ” loads a value from a region
   331в†’   331в†’value: i32 = *region;
   332в†’   332в†’next_node = (*current).next;
   333в†’   333в†’
   334в†’   334в†’// The IVE verifies for each access:
   335в†’   335в†’//   Liveness:      the target address is live
   336в†’   336в†’//   Exclusivity:   no concurrent conflicting access (write-write or write-read)
   337в†’   337в†’//   Interpretation: the BD at the read matches the BD at the write
   338в†’   338в†’//   Origin:        the data came from a valid write
   339в†’   339в†’//   Cleanup:       the region will eventually be freed
   340в†’   340в†’```

   344в†’   344в†’## 4. Pointer Operations
   345в†’   345в†’
   346в†’   346в†’VUMA provides a small set of first-class pointer operations that cover all the functionality traditionally achieved through `unsafe` pointer arithmetic in Rust or C. These operations are not library calls; they are primitive language constructs that the IVE checks automatically. The key operations are **derive** (create a derived pointer), **offset** (pointer arithmetic), and **cast** (type reinterpretation with proof obligations).
   347в†’   347в†’
   348в†’   348в†’### Dereference (`*`)
   349в†’   349в†’
   350в†’   350в†’The dereference operator `*` reads or writes through a pointer. When used on the left side of an assignment, it performs a write; when used in an expression, it performs a read. The IVE verifies every dereference against all five invariants.
   351в†’   351в†’
   352в†’   352в†’```vuma
   353в†’   353в†’region = allocate(8);
   354в†’   354в†’
   355в†’   355в†’// Write via dereference
   356в†’   356в†’*region = 42;
   357в†’   357в†’
   358в†’   358в†’// Read via dereference
   359в†’   359в†’value: i32 = *region;
   360в†’   360в†’
   361в†’   361в†’// Dereference with field access
   362в†’   362в†’node = allocate(24);
   363в†’   363в†’*node = NodeHeader { prev: 0, next: 0, data: 0 };
   364в†’   364в†’(*node).prev = node;   // write to .prev field
   365в†’   365в†’(*node).next = node;   // write to .next field
   366в†’   366в†’prev_val = (*node).prev; // read from .prev field
   367в†’   367в†’```
   368в†’   368в†’
   369в†’   369в†’### Address-Of (`@`)
   370в†’   370в†’
   371в†’   371в†’The `@` operator takes the address of a value, producing a pointer. This is the inverse of dereference:
   372в†’   372в†’
   373в†’   373в†’```vuma
   374в†’   374в†’let x: i32 = 42;
   375в†’   375в†’ptr = @x;       // ptr has type *i32
   376в†’   376в†’*ptr = 100;     // modifies x through the pointer
   377в†’   377в†’```
   378в†’   378в†’
   379в†’   379в†’### Offset (Pointer Arithmetic)
   380в†’   380в†’
   381в†’   381в†’Pointer arithmetic is expressed using the `+` operator between an `Address` and an integer offset, or through the `Offset` expression. The IVE tracks the derivation chain and verifies that the resulting address remains within the bounds of the original allocation.
   382в†’   382в†’
   383в†’   383в†’```vuma
   384в†’   384в†’base = allocate(1024);
   385в†’   385в†’
   386в†’   386в†’// Offset by bytes
   387в†’   387в†’slot_0 = base + 0;     // first 8 bytes
   388в†’   388в†’slot_1 = base + 8;     // next 8 bytes
   389в†’   389в†’slot_2 = base + 16;    // and so on
   390в†’   390в†’
   391в†’   391в†’// IVE verifies: each slot is within base's 1024-byte allocation
   392в†’   392в†’*slot_0 = 10;
   393в†’   393в†’*slot_1 = 20;
   394в†’   394в†’*slot_2 = 30;
   395в†’   395в†’
   396в†’   396в†’// Hardware register access via offset
   397в†’   397в†’const GPIO_BASE: Address = 0x7e200000;
   398в†’   398в†’gpio = map_device(GPIO_BASE, 4096);
   399в†’   399в†’fsel = gpio + 0x00;    // GPFSEL0 offset
   400в†’   400в†’set = gpio + 0x1c;     // GPSET0 offset
   401в†’   401в†’clr = gpio + 0x28;     // GPCLR0 offset
   402в†’   402в†’```
   403в†’   403в†’
   404в†’   404в†’### Derive
   405в†’   405в†’
   406в†’   406в†’The `derive(ptr, region)` expression explicitly creates a derived pointer within a specific region. This is the formal mechanism for sub-allocation, where a pointer into a larger region is created and the IVE must track its provenance. Derive is implicit in pointer arithmetic but can be made explicit for annotation purposes.
   407в†’   407в†’
   408в†’   408в†’```vuma
   409в†’   409в†’// Explicit derivation
   410в†’   410в†’inner = derive(ptr, arena_region);
   411в†’   411в†’
   412в†’   412в†’// Implicit derivation via pointer arithmetic (equivalent)
   413в†’   413в†’inner = arena_base + offset;
   414в†’   414в†’
   415в†’   415в†’// The derivation chain is tracked:
   416в†’   416в†’//   arena.base в†’ current в†’ aligned в†’ ...
   417в†’   417в†’// When arena is freed, all derived pointers are invalidated.
   418в†’   418в†’```
   419в†’   419в†’
   420в†’   420в†’### Cast
   421в†’   421в†’
   422в†’   422в†’The `cast` operation (expressed with the `as` keyword or the `cast` statement) reinterprets a value as a different type. This is the only way to perform type punning in VUMA, and it carries proof obligations: the IVE checks that the RepD of the target type is compatible with the RepD of the source type. If the cast strengthens capabilities (CapD), a safety proof must be provided.
   423в†’   423в†’
   424в†’   424в†’```vuma
   425в†’   425в†’// Safe cast: narrowing capability (always allowed)
   426в†’   426в†’let raw: u64 = ptr as u64;  // pointer to integer (weakening)
   427в†’   427в†’
   428в†’   428в†’// Cast with proof obligation: widening capability
   429в†’   429в†’let ptr: *i32 = raw as *i32;  // integer to pointer (strengthening)
   430в†’   430в†’// IVE requires proof that `raw` originated from a valid pointer
   431в†’   431в†’
   432в†’   432в†’// Explicit cast statement
   433в†’   433в†’cast expr as Type;
   434в†’   434в†’
   435в†’   435в†’// sizeof and alignof for safe casting
   436в†’   436в†’let size = sizeof(NodeHeader);   // 24
   437в†’   437в†’let align = alignof(NodeHeader); // 8
   438в†’   438в†’```

   441в†’   441в†’
   442в†’   442в†’## 5. Control Flow
   443в†’   443в†’
   444в†’   444в†’VUMA provides the standard set of imperative control flow constructs, each with IVE integration to ensure that memory safety invariants are preserved across all execution paths. The IVE analyzes all branches of conditionals, all loop iterations, and all match arms to verify that every possible execution path satisfies the five LIVE invariants.
   445в†’   445в†’
   446в†’   446в†’### If / Else
   447в†’   447в†’
   448в†’   448в†’The `if` statement conditionally executes a block based on a boolean expression. An optional `else` block provides an alternative path. The IVE tracks the state of memory on both branches and merges them at the join point, ensuring that invariants hold regardless of which branch was taken.
   449в†’   449в†’
   450в†’   450в†’```vuma
   451в†’   451в†’if next_head % (*q).capacity == tail % (*q).capacity {
   452в†’   452в†’    return false;  // Queue is full
   453в†’   453в†’} else {
   454в†’   454в†’    *slot = value;
   455в†’   455в†’    (*q).head.store(next_head, Release);
   456в†’   456в†’    return true;
   457в†’   457в†’}
   458в†’   458в†’
   459в†’   459в†’// Nested conditionals are supported:
   460в†’   460в†’if x > 0 {
   461в†’   461в†’    if x > 100 {
   462в†’   462в†’        *ptr = 2;
   463в†’   463в†’    } else {
   464в†’   464в†’        *ptr = 1;
   465в†’   465в†’    }
   466в†’   466в†’} else {
   467в†’   467в†’    *ptr = 0;
   468в†’   468в†’}
   469в†’   469в†’```
   470в†’   470в†’
   471в†’   471в†’### While
   472в†’   472в†’
   473в†’   473в†’The `while` loop repeatedly executes a block as long as a condition is true. The IVE verifies that loop bodies preserve invariants on every iteration and that the loop terminates (when termination analysis is possible). For linked data structures, the IVE can prove that the loop will terminate because the list structure is acyclic aside from the sentinel.
   474в†’   474в†’
   475в†’   475в†’```vuma
   476в†’   476в†’// Traverse a linked list
   477в†’   477в†’current = (*list).next;
   478в†’   478в†’while current != list {
   479в†’   479в†’    print((*current).data);
   480в†’   480в†’    current = (*current).next;
   481в†’   481в†’}
   482в†’   482в†’
   483в†’   483в†’// Free all nodes in a list
   484в†’   484в†’current = (*list).next;
   485в†’   485в†’while current != list {
   486в†’   486в†’    next = (*current).next;
   487в†’   487в†’    free(current);
   488в†’   488в†’    current = next;
   489в†’   489в†’}
   490в†’   490в†’free(list);
   491в†’   491в†’```
   492в†’   492в†’
   493в†’   493в†’### For
   494в†’   494в†’
   495в†’   495в†’The `for` loop iterates over a range or iterable expression. The iterator variable is bound in the loop body's scope:
   496в†’   496в†’
   497в†’   497в†’```vuma
   498в†’   498в†’// Iterate over a range
   499в†’   499в†’for i in 0..10 {
   500в†’   500в†’    slot = base + i * 8;
   501в†’   501в†’    *slot = i;
   502в†’   502в†’}
   503в†’   503в†’
   504в†’   504в†’// Iterate over elements
   505в†’   505в†’for element in buffer {
   506в†’   506в†’    process(element);
   507в†’   507в†’}
   508в†’   508в†’```
   509в†’   509в†’
   510в†’   510в†’### Loop
   511в†’   511в†’
   512в†’   512в†’The infinite `loop` construct runs a block indefinitely. It is typically used in embedded systems for main event loops. The IVE accepts infinite loops in contexts where they are expected (e.g., bare-metal `main` functions) but flags them if they could leak resources.
   513в†’   513в†’
   514в†’   514в†’```vuma
   515в†’   515в†’// Bare-metal blink loop
   516в†’   516в†’loop {
   517в†’   517в†’    gpio_set(led_pin);
   518в†’   518в†’    delay_ms(500);
   519в†’   519в†’    gpio_clear(led_pin);
   520в†’   520в†’    delay_ms(500);
   521в†’   521в†’}
   522в†’   522в†’```
   523в†’   523в†’
   524в†’   524в†’### Match
   525в†’   525в†’
   526в†’   526в†’The `match` statement performs pattern matching on a value. Patterns can be literals, identifiers, wildcards, or struct-like destructuring. Each arm maps a pattern to an expression. The IVE ensures that all match arms preserve invariants and that the match is exhaustive when required.
   527в†’   527в†’
   528в†’   528в†’```vuma
   529в†’   529в†’match result {
   530в†’   530в†’    Some(value) => process(value),
   531в†’   531в†’    None => handle_empty(),
   532в†’   532в†’}
   533в†’   533в†’
   534в†’   534в†’match opcode {
   535в†’   535в†’    0x01 => execute_load(),
   536в†’   536в†’    0x02 => execute_store(),
   537в†’   537в†’    _ => handle_unknown(),  // wildcard catches everything else
   538в†’   538в†’}
   539в†’   539в†’
   540в†’   540в†’match node {
   541в†’   541в†’    Node { data, next } => traverse(next),
   542в†’   542в†’}
   543в†’   543в†’```
   544в†’   544в†’
   545в†’   545в†’### Return
   546в†’   546в†’
   547в†’   547в†’The `return` statement exits the current function, optionally with a value. The IVE verifies that returning does not leak any resources (Cleanup invariant) and that the returned value satisfies the function's declared return type.
   548в†’   548в†’
   549в†’   549в†’```vuma
   550в†’   550в†’fn add(a: i32, b: i32) -> i32 {
   551в†’   551в†’    return a + b;
   552в†’   552в†’}
   553в†’   553в†’
   554в†’   554в†’fn maybe_push(q: Address, value: u64) -> bool {
   555в†’   555в†’    if is_full(q) {
   556в†’   556в†’        return false;
   557в†’   557в†’    }
   558в†’   558в†’    push(q, value);
   559в†’   559в†’    return true;
   560в†’   560в†’}
   561в†’   561в†’```

   564в†’   564в†’
   565в†’   565в†’## 6. Functions
   566в†’   566в†’
   567в†’   567в†’Functions are the primary unit of code organization in VUMA. A function definition specifies a name, a list of typed parameters, an optional return type, and a body block. Functions are first-class items that appear at the top level of a program or within module declarations. The IVE analyzes each function independently and in the context of its callers to verify memory safety across function boundaries.
   568в†’   568в†’
   569в†’   569в†’### Function Definition
   570в†’   570в†’
   571в†’   571в†’The `fn` keyword introduces a function definition. Parameters may carry optional type annotations, and the return type is specified after `->`. If no return type is specified, the function returns `void`.
   572в†’   572в†’
   573в†’   573в†’```vuma
   574в†’   574в†’fn main() -> i32 {
   575в†’   575в†’    region = allocate(8);
   576в†’   576в†’    *region = 42;
   577в†’   577в†’    value: i32 = *region;
   578в†’   578в†’    free(region);
   579в†’   579в†’    return value;
   580в†’   580в†’}
   581в†’   581в†’
   582в†’   582в†’fn push_back(list: Address, value: u64) {
   583в†’   583в†’    sentinel = list;
   584в†’   584в†’    last = (*sentinel).prev;
   585в†’   585в†’    node = allocate(24);
   586в†’   586в†’    *node = NodeHeader { prev: last, next: sentinel, data: value };
   587в†’   587в†’    (*last).next = node;
   588в†’   588в†’    (*sentinel).prev = node;
   589в†’   589в†’}
   590в†’   590в†’
   591в†’   591в†’fn arena_alloc(arena: Address, alloc_size: u64, align: u64) -> Address {
   592в†’   592в†’    current = (*arena).base + (*arena).offset;
   593в†’   593в†’    aligned = current.align_to(align);
   594в†’   594в†’    new_offset = (aligned - (*arena).base) + alloc_size;
   595в†’   595в†’    (*arena).offset = new_offset;
   596в†’   596в†’    return aligned;
   597в†’   597в†’}
   598в†’   598в†’```
   599в†’   599в†’
   600в†’   600в†’### Function Calling
   601в†’   601в†’
   602в†’   602в†’Functions are called by name with parenthesized arguments. The IVE verifies that the arguments are compatible with the parameter types and that any `Address` arguments point to live regions. Function calls can appear as expressions or statements.
   603в†’   603в†’
   604в†’   604в†’```vuma
   605в†’   605в†’// Calling a function as a statement
   606в†’   606в†’push_back(list, 10);
   607в†’   607в†’free(region);
   608в†’   608в†’
   609в†’   609в†’// Calling a function as an expression
   610в†’   610в†’value: u64 = queue_pop(q);
   611в†’   611в†’size = sizeof(NodeHeader);
   612в†’   612в†’
   613в†’   613в†’// Method-style calls via namespace access
   614в†’   614в†’head = (*q).head.load(Acquire);
   615в†’   615в†’AtomicU64::new(0);
   616в†’   616в†’```
   617в†’   617в†’
   618в†’   618в†’### Async / Spawn
   619в†’   619в†’
   620в†’   620в†’VUMA supports lightweight concurrency through `async` blocks and `spawn` expressions. An `async` block creates a deferred computation that can be executed concurrently. A `spawn` expression launches an async block on a new task. The IVE extends its five invariants to concurrent code, verifying that no data races occur across tasks and that shared resources are properly synchronized.
   621в†’   621в†’
   622в†’   622в†’```vuma
   623в†’   623в†’// Async block вЂ” creates a deferred computation
   624в†’   624в†’task = async {
   625в†’   625в†’    let result = compute_value();
   626в†’   626в†’    return result;
   627в†’   627в†’};
   628в†’   628в†’
   629в†’   629в†’// Spawn вЂ” launches an async block as a concurrent task
   630в†’   630в†’handle = spawn async {
   631в†’   631в†’    process_data(buffer);
   632в†’   632в†’};
   633в†’   633в†’
   634в†’   634в†’// Await вЂ” waits for a spawned task to complete
   635в†’   635в†’result = await handle;
   636в†’   636в†’
   637в†’   637в†’// Spawning multiple concurrent tasks
   638в†’   638в†’h1 = spawn process_queue(q1);
   639в†’   639в†’h2 = spawn process_queue(q2);
   640в†’   640в†’await h1;
   641в†’   641в†’await h2;
   642в†’   642в†’```
   643в†’   643в†’
   644в†’   644в†’The IVE verifies async/spawn code by:
   645в†’   645в†’- Checking that no two concurrent tasks write to overlapping memory without synchronization
   646в†’   646в†’- Ensuring that shared reads are safe (concurrent reads never conflict)
   647в†’   647в†’- Tracking the derivation chain across task boundaries
   648в†’   648в†’- Verifying that all resources are freed even in the presence of concurrent execution
   649в†’   649в†’
   650в†’   650в†’---
   651в†’   651в†’
   652в†’   652в†’## 7. Concurrency
   653в†’   653в†’
   654в†’   654в†’VUMA provides language-level concurrency primitives that the IVE can reason about formally. Rather than relying on external libraries, VUMA bakes synchronization constructs into the language so that the IVE can prove the absence of data races, deadlocks, and use-after-free bugs in concurrent code. The key constructs are `sync` blocks, channels, locks, and atomic operations.
   655в†’   655в†’
   656в†’   656в†’### Sync Blocks
   657в†’   657в†’
   658в†’   658в†’The `sync` block establishes a happens-before ordering between accesses inside the block and accesses outside it. This is the fundamental mechanism for telling the IVE that certain operations are sequentially ordered. The IVE uses sync edges to build the ordered relation (transitive closure of happens-before), and skips conflict checks for ordered accesses.
   659в†’   659в†’
   660в†’   660в†’```vuma
   661в†’   661в†’// Sync block ensures sequential ordering
   662в†’   662в†’sync {
   663в†’   663в†’    *shared_data = 42;
   664в†’   664в†’    result = *shared_data;
   665в†’   665в†’}
   666в†’   666в†’// IVE: the write happens before the read, no data race
   667в†’   667в†’
   668в†’   668в†’// Without sync, concurrent accesses might conflict:
   669в†’   669в†’// Thread 1: *shared_data = 42;
   670в†’   670в†’// Thread 2: result = *shared_data;
   671в†’   671в†’// IVE would flag this as a write-read conflict without a sync edge
   672в†’   672в†’```
   673в†’   673в†’
   674в†’   674в†’### Channels
   675в†’   675в†’
   676в†’   676в†’Channels provide message-passing concurrency. The `channel` keyword creates a typed communication channel, `send` places a value into the channel, and `recv` extracts a value. Channels enforce ownership transfer: sending a value through a channel moves ownership to the receiver, preventing the sender from accessing it afterward.
   677в†’   677в†’
   678в†’   678в†’```vuma
   679в†’   679в†’// Create a channel
   680в†’   680в†’ch = channel<u64>();
   681в†’   681в†’
   682в†’   682в†’// Send a value (ownership transfers)
   683в†’   683в†’send(ch, computed_value);
   684в†’   684в†’
   685в†’   685в†’// Receive a value
   686в†’   686в†’value = recv(ch);
   687в†’   687в†’
   688в†’   688в†’// The IVE verifies:
   689в†’   689в†’// - After send(), the sender no longer has access to the value
   690в†’   690в†’// - recv() only succeeds after a matching send()
   691в†’   691в†’// - No two receivers can get the same value (Exclusivity)
   692в†’   692в†’```
   693в†’   693в†’
   694в†’   694в†’### Locks
   695в†’   695в†’
   696в†’   696в†’VUMA's `lock` and `unlock` statements provide mutual exclusion. The IVE integrates locks into the CapD lattice: when a CapD specifies `write_requires_lock: id`, the IVE knows that the write is safe as long as lock `id` is held. This enables the IVE to classify conflicting accesses as "probably safe" when they are both protected by the same mutex.
   697в†’   697в†’
   698в†’   698в†’```vuma
   699в†’   699в†’// Lock-based synchronization
   700в†’   700в†’lock(mutex_id);
   701в†’   701в†’// IVE: mutex_id is now held, write-locked CapDs are active
   702в†’   702в†’(*shared).counter = (*shared).counter + 1;
   703в†’   703в†’unlock(mutex_id);
   704в†’   704в†’// IVE: mutex_id is no longer held
   705в†’   705в†’
   706в†’   706в†’// The IVE tracks which locks are held and resolves CapD conditions:
   707в†’   707в†’// CapD { can_write: true, write_requires_lock: Some(42) }
   708в†’   708в†’// is_write_active(held_locks) = true when {42} вЉ† held_locks
   709в†’   709в†’```
   710в†’   710в†’
   711в†’   711в†’### Atomics
   712в†’   712в†’
   713в†’   713в†’VUMA provides atomic types such as `AtomicU64` with explicit memory ordering annotations. Atomic operations establish synchronization edges that the IVE uses to prove the absence of data races in lock-free code. The supported orderings are `Acquire`, `Release`, `AcqRel`, and `SeqCst`.
   714в†’   714в†’
   715в†’   715в†’```vuma
   716в†’   716в†’struct Queue<T> {
   717в†’   717в†’    buffer: Address,
   718в†’   718в†’    capacity: u64,
   719в†’   719в†’    head: AtomicU64,    // producer write index
   720в†’   720в†’    tail: AtomicU64,    // consumer read index
   721в†’   721в†’}
   722в†’   722в†’
   723в†’   723в†’fn queue_push(q: Address, value: u64) -> bool {
   724в†’   724в†’    head = (*q).head.load(Acquire);    // acquire: see all prior writes
   725в†’   725в†’    tail = (*q).tail.load(Acquire);
   726в†’   726в†’
   727в†’   727в†’    if next_head % (*q).capacity == tail % (*q).capacity {
   728в†’   728в†’        return false;
   729в†’   729в†’    }
   730в†’   730в†’
   731в†’   731в†’    slot = (*q).buffer + (head % (*q).capacity) * 8;
   732в†’   732в†’    *slot = value;
   733в†’   733в†’
   734в†’   734в†’    (*q).head.store(next_head, Release);  // release: make value visible
   735в†’   735в†’    return true;
   736в†’   736в†’}
   737в†’   737в†’
   738в†’   738в†’fn queue_pop(q: Address) -> Option<u64> {
   739в†’   739в†’    tail = (*q).tail.load(Acquire);
   740в†’   740в†’    head = (*q).head.load(Acquire);
   741в†’   741в†’
   742в†’   742в†’    if tail % (*q).capacity == head % (*q).capacity {
   743в†’   743в†’        return None;
   744в†’   744в†’    }
   745в†’   745в†’
   746в†’   746в†’    slot = (*q).buffer + (tail % (*q).capacity) * 8;
   747в†’   747в†’    value = *slot;
   748в†’   748в†’
   749в†’   749в†’    (*q).tail.store(tail + 1, Release);
   750в†’   750в†’    return Some(value);
   751в†’   751в†’}
   752в†’   752в†’```
   753в†’   753в†’
   754в†’   754в†’The IVE verifies lock-free code by:
   755в†’   755в†’- Checking that producer and consumer access non-overlapping slots (no data race)
   756в†’   756в†’- Verifying that Release ordering ensures the value write is visible before the index update
   757в†’   757в†’- Proving that Acquire ordering ensures the consumer sees the value before reading the slot
   758в†’   758в†’- Confirming that values read originated from valid writes (Origin invariant)
   759в†’   759в†’
   760в†’   760в†’---
   761в†’   761в†’
   762в†’   762в†’## 8. Memory Safety
   763в†’   763в†’
   764в†’   764в†’Memory safety in VUMA is not optional вЂ” it is guaranteed by the Invariant Verification Engine (IVE), which checks every program against five core invariants. There is no `unsafe` keyword to bypass verification; instead, VUMA provides verification annotations and proof hints that help the IVE discharge proof obligations that would otherwise be too complex to verify automatically. This section describes the five invariants, the verification pipeline, and the annotation mechanisms available to programmers.
   765в†’   765в†’
   766в†’   766в†’### The Five LIVE Invariants
   767в†’   767в†’
   768в†’   768в†’The IVE checks the following invariants for every pointer operation in a VUMA program:
   769в†’   769в†’
   770в†’   770в†’| Invariant | Meaning | Violation Example |
   771в†’   771в†’|---|---|---|
   772в†’   772в†’| **Liveness** | Every requested resource will eventually be provided | Use after free, null dereference |
   773в†’   773в†’| **Exclusivity** | At most one owner for exclusive resources | Data race (two concurrent writes) |
   774в†’   774в†’| **Interpretation** | Every read interprets data under the correct BD | Type confusion, reading a pointer as an integer |
   775в†’   775в†’| **Origin** | Every piece of data has a well-defined provenance | Reading uninitialized memory |
   776в†’   776в†’| **Cleanup** | Every acquired resource is eventually released | Memory leak, leaked lock |
   777в†’   777в†’
   778в†’   778в†’### Verification Pipeline
   779в†’   779в†’
   780в†’   780в†’The IVE operates through a multi-stage pipeline:
   781в†’   781в†’
   782в†’   782в†’1. **Parsing:** Source code is parsed into an AST with full span information
   783в†’   783в†’2. **SCG Construction:** The AST is lowered into a Semantic Compute Graph (SCG) вЂ” a DAG where nodes represent operations and edges represent data flow, control flow, and security boundaries
   784в†’   784в†’3. **BD Inference:** The InferenceEngine walks the SCG and propagates BDs from leaf nodes, resolving composition at each node:
   785в†’   785в†’   - Sequential composition: forward-propagate the output BD
   786в†’   786в†’   - Parallel composition: intersect or unify BDs
   787в†’   787в†’   - Conditional composition: take the union of branch BDs
   788в†’   788в†’4. **Invariant Verification:** The VerificationEngine checks each of the five invariants against the annotated SCG
   789в†’   789в†’5. **Debt Tracking:** Verification obligations that cannot be discharged immediately are recorded as debt items with priority levels
   790в†’   790в†’
   791в†’   791в†’```vuma
   792в†’   792в†’// After compilation, the IVE produces a verification summary:
   793в†’   793в†’// IVE verification: Liveness вњ“, Exclusivity вњ“, Interpretation вњ“, Origin вњ“, Cleanup вњ“
   794в†’   794в†’```
   795в†’   795в†’
   796в†’   796в†’### Verification Annotations
   797в†’   797в†’
   798в†’   798в†’VUMA provides several annotation mechanisms to help the IVE:
   799в†’   799в†’
   800в†’   800в†’**BD Annotations** (`#bd(Name)`): Explicitly specify a behavioral descriptor for a value or type. This overrides inferred BDs when the programmer has domain-specific knowledge:
   801в†’   801в†’
   802в†’   802в†’```vuma
   803в†’   803в†’// Mark a buffer as read-only
   804в†’   804в†’buffer: #bd(ReadOnly) Address = allocate(1024);
   805в†’   805в†’
   806в†’   806в†’// Mark a function as safe (manually verified)
   807в†’   807в†’#bd(Safe) fn trusted_operation() -> i32 { ... }
   808в†’   808в†’```
   809в†’   809в†’
   810в†’   810в†’**Proof Hints**: Provide the IVE with additional information about why an operation is safe. Proof hints can be explicit casts, runtime checks, or formal proof steps:
   811в†’   811в†’
   812в†’   812в†’```vuma
   813в†’   813в†’// Explicit cast provides proof of safe reinterpretation
   814в†’   814в†’let raw: u64 = ptr as u64;  // weakening: always safe
   815в†’   815в†’
   816в†’   816в†’// Runtime check provides proof of safe strengthening
   817в†’   817в†’if tag == TYPE_PTR {
   818в†’   818в†’    let recovered: *i32 = raw as *i32;  // safe because tag was checked
   819в†’   819в†’}
   820в†’   820в†’
   821в†’   821в†’// Formal proof (advanced)
   822в†’   822в†’// SafetyProof::FormalProof { steps: ["tag == TYPE_PTR implies raw is a valid pointer"] }
   823в†’   823в†’```
   824в†’   824в†’
   825в†’   825в†’**Safe/Unsafe Blocks**: While VUMA has no `unsafe` escape hatch, the `safe` keyword can be used to assert that a block has been manually verified, creating a proof obligation that must be discharged:
   826в†’   826в†’
   827в†’   827в†’```vuma
   828в†’   828в†’safe {
   829в†’   829в†’    // The programmer asserts this code is safe
   830в†’   830в†’    // IVE creates a proof obligation rather than checking automatically
   831в†’   831в†’    *raw_ptr = value;
   832в†’   832в†’}
   833в†’   833в†’```
   834в†’   834в†’
   835в†’   835в†’### Verification Results
   836в†’   836в†’
   837в†’   837в†’The IVE produces one of three verdicts for each invariant:
   838в†’   838в†’
   839в†’   839в†’- **Proven:** The invariant holds on all execution paths (formal proof or exhaustive analysis)
   840в†’   840в†’- **ProbablySafe:** The invariant holds under stated assumptions (e.g., lock protection, runtime checks)
   841в†’   841в†’- **Violated:** A counterexample was found demonstrating the invariant can be broken
   842в†’   842в†’
   843в†’   843в†’```vuma
   844в†’   844в†’// Example verification output:
   845в†’   845в†’// Liveness:     Proven (exhaustive analysis)
   846в†’   846в†’// Exclusivity:  ProbablySafe (2 conflicts protected by mutex locks)
   847в†’   847в†’// Interpretation: Proven (formal proof: 6 steps)
   848в†’   848в†’// Origin:       Proven (all reads trace to valid writes)
   849в†’   849в†’// Cleanup:      Proven (all allocations freed on all paths)
   850в†’   850в†’```
   851в†’   851в†’
   852в†’   852в†’---

   854в†’   854в†’## 9. Standard Library Overview
   855в†’   855в†’
   856в†’   856в†’VUMA's standard library provides essential data structures, memory management, I/O, formatting, mathematics, cryptography, string operations, concurrency, and filesystem primitives вЂ” all BD-annotated and verified by the IVE.    857в†’   857в†’
   858в†’   858в†’### Memory Management
   859в†’   859в†’
   860в†’   860в†’- **`allocate(size) -> Address`**: Reserve `size` bytes of memory. IVE verifies liveness and cleanup.

  1035в†’  1035в†’The `extern "C" { ... }` block declares external functions that are resolved at link time:
  1036в†’  1036в†’

  1076в†’  1076в†’

  1078в†’  1078в†’
  1079в†’  1079в†’### Device Memory Mapping
  1080в†’  1080в†’
  1081в†’  1081в†’The `map_device(base, size)` intrinsic maps a physical address range into the program's virtual address space. On AArch64, this is used to access hardware peripherals such as GPIO, UART, and DMA controllers. The IVE treats mapped device regions specially:
  1082в†’  1082в†’
  1083в†’  1083в†’- They never need to be freed (hardware is always present)
  1084в†’  1084в†’- They have volatile semantics (reads and writes have side effects)
  1085в†’  1085в†’- They have a known size for bounds checking
  1086в†’  1086в†’
