     1→# VUMA Language Reference
     2→
     3→> Version 0.2.0 — Last updated 2026-03-06
     4→
     5→This document is the definitive reference for the VUMA programming language, a memory-oriented, verification-first systems language targeting 8 backend architectures (AArch64, x86_64, RISC-V 64, ARM32, MIPS64, PPC64, LoongArch64, Wasm32). VUMA eliminates the need for `unsafe` blocks by making the Invariant Verification Engine (IVE) an integral part of compilation. Every pointer dereference, allocation, and free is automatically checked against five core invariants: **Liveness**, **Exclusivity**, **Interpretation**, **Origin**, and **Cleanup** (collectively "LIVE").

    26→## 1. Lexical Structure
    27→
    28→The VUMA lexer transforms raw source text into a flat stream of tokens, each annotated with byte-offset spans and line/column positions for error reporting. The lexer supports full error recovery: it never stops on the first error, producing `Error` tokens and collecting diagnostic information that can be retrieved later. Whitespace and non-doc comments are silently skipped; doc comments (`///` and `//!`) are emitted as tokens for downstream tooling.
    29→
    30→### Keywords
    31→
    32→VUMA reserves the following keywords. They cannot be used as identifiers unless escaped. The keyword set is organized into six categories reflecting the language's design priorities:
    33→
    34→**Core language keywords:** `fn`, `let`, `if`, `else`, `while`, `for`, `return`, `as`, `match`, `struct`, `enum`, `loop`
    35→
    36→**Memory primitive keywords:** `ptr`, `region`, `alloc`, `allocate`, `free`, `derive`, `cast`, `read`, `write`
    37→
    38→**Concurrency and synchronization keywords:** `sync`, `async`, `await`, `spawn`, `lock`, `unlock`, `channel`, `send`, `recv`
    39→
    40→**Safety keywords:** `unsafe`, `safe`
    41→
    42→**Behavioral domain directive keywords:** `bd`, `repd`, `capd`, `reld`
    43→
    44→**Module system keywords:** `import`, `export`, `mod`, `use`, `self`, `super`
    45→
    46→**Literal keywords:** `true`, `false`
    47→
    48→**Type operator keywords:** `sizeof`, `alignof`
    49→
    50→### Operators
    51→
    52→VUMA provides a comprehensive set of arithmetic, comparison, logical, bitwise, and pointer-specific operators:
    53→
    54→| Category | Operators |
    55→|---|---|
    56→| Arithmetic | `+` `-` `*` `/` `%` |
    57→| Comparison | `==` `!=` `<` `<=` `>` `>=` |
    58→| Logical | `&&` `\|\|` `!` |
    59→| Bitwise | `&` `\|` `^` `~` `<<` `>>` |
    60→| Assignment | `=` |
    61→| Pointer | `*` (dereference) `@` (address-of) |
    62→| Type | `as` (cast) `::` (path separator) |
    63→| Arrow | `->` (return type) `=>` (match arm) |
    64→| Other | `.` `..` `...` `:` `;` `,` `?` `#` `$` |
    65→
    66→### Literals
    67→
    68→VUMA supports the following literal forms, each with underscore separators for readability:
    69→
    70→- **Integer literals:** `42`, `0`, `1_000_000`
    71→- **Hex address literals:** `0xDEADBEEF`, `0xFF00_0000` — these produce the `Address` token kind, representing a `u64` memory address
    72→- **Binary literals:** `0b1010_1100`
    73→- **Octal literals:** `0o777`
    74→- **Float literals:** `3.14`, `1e10`, `2.5e-3`
    75→- **String literals:** `"hello"`, with escape sequences `\n`, `\t`, `\r`, `\\`, `\"`, `\0`, `\xHH`
    76→- **Character literals:** `'a'`, `'\n'`
    77→- **Byte string literals:** `b"hello"`
    78→- **Raw string literals:** `r"..."`, `r#"..."#`
    79→- **Boolean literals:** `true`, `false`
    80→
    81→### Comments
    82→
    83→VUMA supports three forms of comments:
    84→
    85→- **Line comments:** `//` — extend to end of line, silently skipped by the lexer
    86→- **Block comments:** `/* ... */` — can be nested, silently skipped
    87→- **Doc comments:** `///` (outer) and `//!` (module-level) — emitted as `DocComment` and `ModuleDoc` tokens for documentation tooling
    88→
    89→```vuma
    90→// This is a line comment
    91→/* This is a
    92→   block comment */
    93→
    94→/// This is an outer doc comment — documents the next item
    95→//! This is a module-level doc comment
    96→```
    97→
    98→### Identifiers
    99→
   100→Identifiers begin with an ASCII letter or underscore, followed by any combination of ASCII letters, digits, or underscores. Keywords are reserved and cannot be used as identifiers. Type names conventionally use `PascalCase`, while function and variable names use `snake_case`.
   101→
   102→```vuma
   103→my_variable    // valid
   104→_buffer        // valid
   105→NodeHeader     // valid (convention: type name)
   106→3things        // invalid — starts with a digit
   107→alloc          // invalid — reserved keyword
   108→```

   112→## 2. Types and BD Annotations
   113→
   114→VUMA's type system is built around the concept of **Behavioral Descriptors (BDs)**, which provide a complete specification of a value along three orthogonal axes. Every value in VUMA carries a BD that the Invariant Verification Engine (IVE) uses to prove memory safety without requiring an `unsafe` escape hatch. The type system covers primitive types, pointer types, region-annotated types, struct types, generic type applications, function types, and explicit BD annotations.
   115→
   116→### The BD Triple: RepD × CapD × RelD
   117→
   118→A Behavioral Descriptor is the composition of three sub-descriptors:
   119→
   120→```
   121→BD = RepD × CapD × RelD
   122→```
   123→
   124→Two BDs are **compatible** when all three layers are pairwise compatible. One BD **refines** another when every layer is at least as specific:
   125→
   126→```
   127→bd1 ⊑ bd2  ⟺  bd1.repd.subsumes(bd2.repd)
   128→               ∧ bd1.capd.is_subset(bd2.capd)
   129→               ∧ bd1.reld.refines(bd2.reld)
   130→```
   131→
   132→### RepD — Representation Descriptor
   133→
   134→RepD describes the memory shape, size, and alignment of a value. It forms a subsumption lattice where more-specific RepDs subsume less-specific ones. The `Byte` variant is the universal bottom element that is compatible with everything.
   135→
   136→```vuma
   137→// RepD variants (from the bd crate):
   138→//   Byte(size, align)   — raw byte sequence, compatible with anything
   139→//   Struct { fields }   — named field layout
   140→//   Array { elem, len } — homogeneous fixed-size array
   141→//   Enum { variants }   — tagged union
   142→//   Ptr(pointee)        — pointer representation
   143→//   Union { variants }  — untagged union
   144→//   Func { params, ret }— function pointer representation
   145→
   146→// In source code, RepD is inferred from type annotations:
   147→let x: i32 = 42;           // RepD: Byte(4, 4)
   148→let p: Address = allocate(8); // RepD: Ptr(Byte(8, 8))
   149→```
   150→
   151→### CapD — Capability Descriptor
   152→
   153→CapD specifies what operations are permitted on a value. It forms a lattice where the **meet** (intersection of capabilities, union of conditions) and **join** (union of capabilities, intersection of conditions) operations are well-defined. Capabilities can be conditioned on holding a specific lock, enabling fine-grained concurrency reasoning.
   154→
   155→```vuma
   156→// CapD capabilities (from the bd crate):
   157→//   Read     — the value may be read
   158→//   Write    — the value may be written
   159→//   Execute  — the value may be executed as code
   160→//   Derive   — new pointers may be derived from this value
   161→
   162→// CapD can be conditioned on lock acquisition:
   163→//   CapD { read: true, write: true, write_requires_lock: 42 }
   164→//   This means write is only permitted when lock #42 is held.
   165→
   166→// CapD lattice operations:
   167→//   meet(a, b) = intersection of caps, union of conditions (more restrictive)
   168→//   join(a, b) = union of caps, intersection of conditions (less restrictive)
   169→
   170→// Explicit BD annotation syntax:
   171→#bd(ReadOnly)    // CapD with only Read capability
   172→#bd(ReadWrite)   // CapD with Read and Write capabilities
   173→#bd(LockedWrite) // CapD where Write requires a specific lock
   174→```
   175→
   176→### RelD — Relational Descriptor
   177→
   178→RelD captures temporal, dependency, and security relationships that a value participates in. It ensures that constraints such as lifetime ordering, information flow policies, and dependency tracking are preserved across operations. RelD composition is the union of relations, and consistency checking ensures that contradictory constraints (e.g., `Outlives` + `Succeeds`) are rejected.
   179→
   180→```vuma
   181→// RelD relation types (from the bd crate):
   182→//   Liveness      — value is guaranteed to be live
   183→//   Outlives(a,b) — region a outlives region b
   184→//   DependsOn(a,b)— computation depends on value b
   185→//   Succeeds(a,b) — event a happens after event b
   186→//   FlowPolicy(p) — information-flow policy label p
   187→
   188→// RelD is usually inferred, but can be annotated:
   189→// #bd(Exclusive) carries an implicit RelD with Liveness
   190→```
   191→
   192→### Type Syntax
   193→
   194→The full type syntax in VUMA covers:
   195→
   196→```vuma
   197→// Primitive (BD base) types:
   198→u8  u16  u32  u64    // unsigned integers
   199→i8  i16  i32  i64    // signed integers
   200→bool                  // boolean
   201→void                  // unit/void type
   202→Address               // opaque memory region handle (u64)
   203→
   204→// Pointer types:
   205→*i32                  // pointer to i32
   206→*Address              // pointer to an address (double indirection)
   207→
   208→// Region-annotated pointers:
   209→*i32 @ arena          // pointer to i32 in region "arena"
   210→
   211→// Fixed-size arrays:
   212→[i32; 10]             // array of 10 i32 values
   213→
   214→// Struct types:
   215→NodeHeader            // named struct (defined with `struct`)
   216→
   217→// Generic types:
   218→Queue<T>              // generic type parameterized by T
   219→Result<T, E>          // generic with two type parameters
   220→
   221→// Function types:
   222→(i32, i32) -> i32     // function taking two i32s, returning i32
   223→() -> void            // nullary function returning void
   224→
   225→// BD annotation types:
   226→#bd(ReadOnly)         // behavioral descriptor annotation
   227→```
   228→
   229→### Type Ascription
   230→
   231→Variables may optionally carry type annotations. When omitted, the type is inferred from context or the right-hand side expression:
   232→
   233→```vuma
   234→let x: i32 = 42;          // explicit type
   235→let y = 42;               // inferred as integer literal
   236→let p: Address = allocate(8); // explicit Address type
   237→value: i32 = *region;     // type ascription without `let`
   238→```

   242→## 3. Memory Model
   243→
   244→VUMA's memory model is built around four first-class operations: **allocate**, **write**, **read**, and **free**. Unlike languages where these are library calls, VUMA makes them primitive statements that the Invariant Verification Engine (IVE) automatically checks. Every pointer operation is verified against the five LIVE invariants, and there is no `unsafe` keyword to bypass verification. This section describes regions, allocations, derivations, and accesses — the core concepts that define how VUMA programs interact with memory.
   245→
   246→### Regions
   247→
   248→A **region** is a contiguous block of memory created by `allocate(size)` or `map_device(base, size)`. Regions are identified by their `Address` value, an opaque 64-bit handle. The IVE tracks the lifetime and bounds of every region, ensuring that all accesses fall within the region's allocated range.
   249→
   250→```vuma
   251→// Allocate a region for one 64-bit integer
   252→region = allocate(8);
   253→
   254→// Allocate a region large enough for a struct
   255→node = allocate(24);  // sizeof(NodeHeader) = 24
   256→
   257→// Map a hardware device region (never freed, volatile semantics)
   258→gpio = map_device(0x7e200000, 4096);
   259→```
   260→
   261→Regions can be named using the `region` keyword for clarity:
   262→
   263→```vuma
   264→region pool = allocate(4096);  // named region declaration
   265→```
   266→
   267→### Allocations
   268→
   269→The `allocate(size)` statement reserves `size` bytes of memory and returns an `Address`. The IVE verifies at compile time that:
   270→
   271→1. **Liveness:** the returned address is live immediately after allocation
   272→2. **Interpretation:** the allocation size is a positive, known value
   273→3. **Cleanup:** every allocation is eventually freed (or mapped as a device that never needs freeing)
   274→
   275→```vuma
   276→// Simple allocation
   277→p = allocate(8);        // 8 bytes for a u64
   278→*p = 42;                // write to the region
   279→value: u64 = *p;        // read from the region
   280→free(p);                // release the region
   281→
   282→// Arena-style allocation
   283→arena = allocate(4096); // one large block
   284→a = arena + 0;          // offset 0 within arena
   285→b = arena + 64;         // offset 64 within arena
   286→// IVE knows a and b are within the arena's 4096-byte range
   287→free(arena);            // frees the entire arena; a and b are now dead
   288→```
   289→
   290→### Derivations
   291→
   292→A **derivation** is the process of creating a new pointer from an existing one. VUMA tracks the full derivation chain so the IVE knows which original allocation a derived pointer comes from. This is critical for proving that offsets stay within bounds and that freeing a base allocation invalidates all derived pointers.
   293→
   294→```vuma
   295→// Pointer arithmetic derives a new address
   296→base = allocate(1024);
   297→offset_ptr = base + 128;     // derived: base + 128
   298→// IVE knows offset_ptr is within base's 1024-byte allocation
   299→
   300→// The derive expression is explicit:
   301→derived = derive(ptr, region);  // formal derivation with region annotation
   302→
   303→// Arena allocation uses derivation chains:
   304→current = (*arena).base + (*arena).offset;  // derived from arena.base
   305→aligned = current.align_to(16);             // further derivation with alignment
   306→// IVE tracks: arena.base → current → aligned
   307→```
   308→
   309→When the base allocation is freed, the IVE marks **all** derived pointers as invalid. This prevents the most common bug in arena-based code: using a pointer after its arena has been destroyed.
   310→
   311→```vuma
   312→arena = arena_create(4096);
   313→a = arena_alloc(arena, 64, 8);
   314→b = arena_alloc(arena, 128, 16);
   315→arena_destroy(arena);
   316→// IVE: a and b are now dead pointers
   317→// x = *a;  // ERROR: use of dead pointer `a`
   318→```
   319→
   320→### Accesses
   321→
   322→A **memory access** is a read or write through a dereferenced pointer. Every access is checked by the IVE against the five invariants. The IVE tracks byte-level granularity, so it can prove that two accesses to different fields of the same struct do not conflict — this is how VUMA verifies doubly-linked list operations that Rust's borrow checker rejects.
   323→
   324→```vuma
   325→// Write access — stores a value into a region
   326→*region = 42;
   327→(*node).next = sentinel;
   328→(*last).prev = node;
   329→
   330→// Read access — loads a value from a region
   331→value: i32 = *region;
   332→next_node = (*current).next;
   333→
   334→// The IVE verifies for each access:
   335→//   Liveness:      the target address is live
   336→//   Exclusivity:   no concurrent conflicting access (write-write or write-read)
   337→//   Interpretation: the BD at the read matches the BD at the write
   338→//   Origin:        the data came from a valid write
   339→//   Cleanup:       the region will eventually be freed
   340→```

   344→## 4. Pointer Operations
   345→
   346→VUMA provides a small set of first-class pointer operations that cover all the functionality traditionally achieved through `unsafe` pointer arithmetic in Rust or C. These operations are not library calls; they are primitive language constructs that the IVE checks automatically. The key operations are **derive** (create a derived pointer), **offset** (pointer arithmetic), and **cast** (type reinterpretation with proof obligations).
   347→
   348→### Dereference (`*`)
   349→
   350→The dereference operator `*` reads or writes through a pointer. When used on the left side of an assignment, it performs a write; when used in an expression, it performs a read. The IVE verifies every dereference against all five invariants.
   351→
   352→```vuma
   353→region = allocate(8);
   354→
   355→// Write via dereference
   356→*region = 42;
   357→
   358→// Read via dereference
   359→value: i32 = *region;
   360→
   361→// Dereference with field access
   362→node = allocate(24);
   363→*node = NodeHeader { prev: 0, next: 0, data: 0 };
   364→(*node).prev = node;   // write to .prev field
   365→(*node).next = node;   // write to .next field
   366→prev_val = (*node).prev; // read from .prev field
   367→```
   368→
   369→### Address-Of (`@`)
   370→
   371→The `@` operator takes the address of a value, producing a pointer. This is the inverse of dereference:
   372→
   373→```vuma
   374→let x: i32 = 42;
   375→ptr = @x;       // ptr has type *i32
   376→*ptr = 100;     // modifies x through the pointer
   377→```
   378→
   379→### Offset (Pointer Arithmetic)
   380→
   381→Pointer arithmetic is expressed using the `+` operator between an `Address` and an integer offset, or through the `Offset` expression. The IVE tracks the derivation chain and verifies that the resulting address remains within the bounds of the original allocation.
   382→
   383→```vuma
   384→base = allocate(1024);
   385→
   386→// Offset by bytes
   387→slot_0 = base + 0;     // first 8 bytes
   388→slot_1 = base + 8;     // next 8 bytes
   389→slot_2 = base + 16;    // and so on
   390→
   391→// IVE verifies: each slot is within base's 1024-byte allocation
   392→*slot_0 = 10;
   393→*slot_1 = 20;
   394→*slot_2 = 30;
   395→
   396→// Hardware register access via offset
   397→const GPIO_BASE: Address = 0x7e200000;
   398→gpio = map_device(GPIO_BASE, 4096);
   399→fsel = gpio + 0x00;    // GPFSEL0 offset
   400→set = gpio + 0x1c;     // GPSET0 offset
   401→clr = gpio + 0x28;     // GPCLR0 offset
   402→```
   403→
   404→### Derive
   405→
   406→The `derive(ptr, region)` expression explicitly creates a derived pointer within a specific region. This is the formal mechanism for sub-allocation, where a pointer into a larger region is created and the IVE must track its provenance. Derive is implicit in pointer arithmetic but can be made explicit for annotation purposes.
   407→
   408→```vuma
   409→// Explicit derivation
   410→inner = derive(ptr, arena_region);
   411→
   412→// Implicit derivation via pointer arithmetic (equivalent)
   413→inner = arena_base + offset;
   414→
   415→// The derivation chain is tracked:
   416→//   arena.base → current → aligned → ...
   417→// When arena is freed, all derived pointers are invalidated.
   418→```
   419→
   420→### Cast
   421→
   422→The `cast` operation (expressed with the `as` keyword or the `cast` statement) reinterprets a value as a different type. This is the only way to perform type punning in VUMA, and it carries proof obligations: the IVE checks that the RepD of the target type is compatible with the RepD of the source type. If the cast strengthens capabilities (CapD), a safety proof must be provided.
   423→
   424→```vuma
   425→// Safe cast: narrowing capability (always allowed)
   426→let raw: u64 = ptr as u64;  // pointer to integer (weakening)
   427→
   428→// Cast with proof obligation: widening capability
   429→let ptr: *i32 = raw as *i32;  // integer to pointer (strengthening)
   430→// IVE requires proof that `raw` originated from a valid pointer
   431→
   432→// Explicit cast statement
   433→cast expr as Type;
   434→
   435→// sizeof and alignof for safe casting
   436→let size = sizeof(NodeHeader);   // 24
   437→let align = alignof(NodeHeader); // 8
   438→```

   442→## 5. Control Flow
   443→
   444→VUMA provides the standard set of imperative control flow constructs, each with IVE integration to ensure that memory safety invariants are preserved across all execution paths. The IVE analyzes all branches of conditionals, all loop iterations, and all match arms to verify that every possible execution path satisfies the five LIVE invariants.
   445→
   446→### If / Else
   447→
   448→The `if` statement conditionally executes a block based on a boolean expression. An optional `else` block provides an alternative path. The IVE tracks the state of memory on both branches and merges them at the join point, ensuring that invariants hold regardless of which branch was taken.
   449→
   450→```vuma
   451→if next_head % (*q).capacity == tail % (*q).capacity {
   452→    return false;  // Queue is full
   453→} else {
   454→    *slot = value;
   455→    (*q).head.store(next_head, Release);
   456→    return true;
   457→}
   458→
   459→// Nested conditionals are supported:
   460→if x > 0 {
   461→    if x > 100 {
   462→        *ptr = 2;
   463→    } else {
   464→        *ptr = 1;
   465→    }
   466→} else {
   467→    *ptr = 0;
   468→}
   469→```
   470→
   471→### While
   472→
   473→The `while` loop repeatedly executes a block as long as a condition is true. The IVE verifies that loop bodies preserve invariants on every iteration and that the loop terminates (when termination analysis is possible). For linked data structures, the IVE can prove that the loop will terminate because the list structure is acyclic aside from the sentinel.
   474→
   475→```vuma
   476→// Traverse a linked list
   477→current = (*list).next;
   478→while current != list {
   479→    print((*current).data);
   480→    current = (*current).next;
   481→}
   482→
   483→// Free all nodes in a list
   484→current = (*list).next;
   485→while current != list {
   486→    next = (*current).next;
   487→    free(current);
   488→    current = next;
   489→}
   490→free(list);
   491→```
   492→
   493→### For
   494→
   495→The `for` loop iterates over a range or iterable expression. The iterator variable is bound in the loop body's scope:
   496→
   497→```vuma
   498→// Iterate over a range
   499→for i in 0..10 {
   500→    slot = base + i * 8;
   501→    *slot = i;
   502→}
   503→
   504→// Iterate over elements
   505→for element in buffer {
   506→    process(element);
   507→}
   508→```
   509→
   510→### Loop
   511→
   512→The infinite `loop` construct runs a block indefinitely. It is typically used in embedded systems for main event loops. The IVE accepts infinite loops in contexts where they are expected (e.g., bare-metal `main` functions) but flags them if they could leak resources.
   513→
   514→```vuma
   515→// Bare-metal blink loop
   516→loop {
   517→    gpio_set(led_pin);
   518→    delay_ms(500);
   519→    gpio_clear(led_pin);
   520→    delay_ms(500);
   521→}
   522→```
   523→
   524→### Match
   525→
   526→The `match` statement performs pattern matching on a value. Patterns can be literals, identifiers, wildcards, or struct-like destructuring. Each arm maps a pattern to an expression. The IVE ensures that all match arms preserve invariants and that the match is exhaustive when required.
   527→
   528→```vuma
   529→match result {
   530→    Some(value) => process(value),
   531→    None => handle_empty(),
   532→}
   533→
   534→match opcode {
   535→    0x01 => execute_load(),
   536→    0x02 => execute_store(),
   537→    _ => handle_unknown(),  // wildcard catches everything else
   538→}
   539→
   540→match node {
   541→    Node { data, next } => traverse(next),
   542→}
   543→```
   544→
   545→### Return
   546→
   547→The `return` statement exits the current function, optionally with a value. The IVE verifies that returning does not leak any resources (Cleanup invariant) and that the returned value satisfies the function's declared return type.
   548→
   549→```vuma
   550→fn add(a: i32, b: i32) -> i32 {
   551→    return a + b;
   552→}
   553→
   554→fn maybe_push(q: Address, value: u64) -> bool {
   555→    if is_full(q) {
   556→        return false;
   557→    }
   558→    push(q, value);
   559→    return true;
   560→}
   561→```
   562→
   563→---
   564→
   565→## 6. Functions
   566→
   567→Functions are the primary unit of code organization in VUMA. A function definition specifies a name, a list of typed parameters, an optional return type, and a body block. Functions are first-class items that appear at the top level of a program or within module declarations. The IVE analyzes each function independently and in the context of its callers to verify memory safety across function boundaries.
   568→
   569→### Function Definition
   570→
   571→The `fn` keyword introduces a function definition. Parameters may carry optional type annotations, and the return type is specified after `->`. If no return type is specified, the function returns `void`.
   572→
   573→```vuma
   574→fn main() -> i32 {
   575→    region = allocate(8);
   576→    *region = 42;
   577→    value: i32 = *region;
   578→    free(region);
   579→    return value;
   580→}
   581→
   582→fn push_back(list: Address, value: u64) {
   583→    sentinel = list;
   584→    last = (*sentinel).prev;
   585→    node = allocate(24);
   586→    *node = NodeHeader { prev: last, next: sentinel, data: value };
   587→    (*last).next = node;
   588→    (*sentinel).prev = node;
   589→}
   590→
   591→fn arena_alloc(arena: Address, alloc_size: u64, align: u64) -> Address {
   592→    current = (*arena).base + (*arena).offset;
   593→    aligned = current.align_to(align);
   594→    new_offset = (aligned - (*arena).base) + alloc_size;
   595→    (*arena).offset = new_offset;
   596→    return aligned;
   597→}
   598→```
   599→
   600→### Function Calling
   601→
   602→Functions are called by name with parenthesized arguments. The IVE verifies that the arguments are compatible with the parameter types and that any `Address` arguments point to live regions. Function calls can appear as expressions or statements.
   603→
   604→```vuma
   605→// Calling a function as a statement
   606→push_back(list, 10);
   607→free(region);
   608→
   609→// Calling a function as an expression
   610→value: u64 = queue_pop(q);
   611→size = sizeof(NodeHeader);
   612→
   613→// Method-style calls via namespace access
   614→head = (*q).head.load(Acquire);
   615→AtomicU64::new(0);
   616→```
   617→
   618→### Async / Spawn
   619→
   620→VUMA supports lightweight concurrency through `async` blocks and `spawn` expressions. An `async` block creates a deferred computation that can be executed concurrently. A `spawn` expression launches an async block on a new task. The IVE extends its five invariants to concurrent code, verifying that no data races occur across tasks and that shared resources are properly synchronized.
   621→
   622→```vuma
   623→// Async block — creates a deferred computation
   624→task = async {
   625→    let result = compute_value();
   626→    return result;
   627→};
   628→
   629→// Spawn — launches an async block as a concurrent task
   630→handle = spawn async {
   631→    process_data(buffer);
   632→};
   633→
   634→// Await — waits for a spawned task to complete
   635→result = await handle;
   636→
   637→// Spawning multiple concurrent tasks
   638→h1 = spawn process_queue(q1);
   639→h2 = spawn process_queue(q2);
   640→await h1;
   641→await h2;
   642→```
   643→
   644→The IVE verifies async/spawn code by:
   645→- Checking that no two concurrent tasks write to overlapping memory without synchronization
   646→- Ensuring that shared reads are safe (concurrent reads never conflict)
   647→- Tracking the derivation chain across task boundaries
   648→- Verifying that all resources are freed even in the presence of concurrent execution

   651→
   652→## 7. Concurrency
   653→
   654→VUMA provides language-level concurrency primitives that the IVE can reason about formally. Rather than relying on external libraries, VUMA bakes synchronization constructs into the language so that the IVE can prove the absence of data races, deadlocks, and use-after-free bugs in concurrent code. The key constructs are `sync` blocks, channels, locks, and atomic operations.
   655→
   656→### Sync Blocks
   657→
   658→The `sync` block establishes a happens-before ordering between accesses inside the block and accesses outside it. This is the fundamental mechanism for telling the IVE that certain operations are sequentially ordered. The IVE uses sync edges to build the ordered relation (transitive closure of happens-before), and skips conflict checks for ordered accesses.
   659→
   660→```vuma
   661→// Sync block ensures sequential ordering
   662→sync {
   663→    *shared_data = 42;
   664→    result = *shared_data;
   665→}
   666→// IVE: the write happens before the read, no data race
   667→
   668→// Without sync, concurrent accesses might conflict:
   669→// Thread 1: *shared_data = 42;
   670→// Thread 2: result = *shared_data;
   671→// IVE would flag this as a write-read conflict without a sync edge
   672→```
   673→
   674→### Channels
   675→
   676→Channels provide message-passing concurrency. The `channel` keyword creates a typed communication channel, `send` places a value into the channel, and `recv` extracts a value. Channels enforce ownership transfer: sending a value through a channel moves ownership to the receiver, preventing the sender from accessing it afterward.
   677→
   678→```vuma
   679→// Create a channel
   680→ch = channel<u64>();
   681→
   682→// Send a value (ownership transfers)
   683→send(ch, computed_value);
   684→
   685→// Receive a value
   686→value = recv(ch);
   687→
   688→// The IVE verifies:
   689→// - After send(), the sender no longer has access to the value
   690→// - recv() only succeeds after a matching send()
   691→// - No two receivers can get the same value (Exclusivity)
   692→```
   693→
   694→### Locks
   695→
   696→VUMA's `lock` and `unlock` statements provide mutual exclusion. The IVE integrates locks into the CapD lattice: when a CapD specifies `write_requires_lock: id`, the IVE knows that the write is safe as long as lock `id` is held. This enables the IVE to classify conflicting accesses as "probably safe" when they are both protected by the same mutex.
   697→
   698→```vuma
   699→// Lock-based synchronization
   700→lock(mutex_id);
   701→// IVE: mutex_id is now held, write-locked CapDs are active
   702→(*shared).counter = (*shared).counter + 1;
   703→unlock(mutex_id);
   704→// IVE: mutex_id is no longer held
   705→
   706→// The IVE tracks which locks are held and resolves CapD conditions:
   707→// CapD { can_write: true, write_requires_lock: Some(42) }
   708→// is_write_active(held_locks) = true when {42} ⊆ held_locks
   709→```
   710→
   711→### Atomics
   712→
   713→VUMA provides atomic types such as `AtomicU64` with explicit memory ordering annotations. Atomic operations establish synchronization edges that the IVE uses to prove the absence of data races in lock-free code. The supported orderings are `Acquire`, `Release`, `AcqRel`, and `SeqCst`.
   714→
   715→```vuma
   716→struct Queue<T> {
   717→    buffer: Address,
   718→    capacity: u64,
   719→    head: AtomicU64,    // producer write index
   720→    tail: AtomicU64,    // consumer read index
   721→}
   722→
   723→fn queue_push(q: Address, value: u64) -> bool {
   724→    head = (*q).head.load(Acquire);    // acquire: see all prior writes
   725→    tail = (*q).tail.load(Acquire);
   726→
   727→    if next_head % (*q).capacity == tail % (*q).capacity {
   728→        return false;
   729→    }
   730→
   731→    slot = (*q).buffer + (head % (*q).capacity) * 8;
   732→    *slot = value;
   733→
   734→    (*q).head.store(next_head, Release);  // release: make value visible
   735→    return true;
   736→}
   737→
   738→fn queue_pop(q: Address) -> Option<u64> {
   739→    tail = (*q).tail.load(Acquire);
   740→    head = (*q).head.load(Acquire);
   741→
   742→    if tail % (*q).capacity == head % (*q).capacity {
   743→        return None;
   744→    }
   745→
   746→    slot = (*q).buffer + (tail % (*q).capacity) * 8;
   747→    value = *slot;
   748→
   749→    (*q).tail.store(tail + 1, Release);
   750→    return Some(value);
   751→}
   752→```
   753→
   754→The IVE verifies lock-free code by:
   755→- Checking that producer and consumer access non-overlapping slots (no data race)
   756→- Verifying that Release ordering ensures the value write is visible before the index update
   757→- Proving that Acquire ordering ensures the consumer sees the value before reading the slot
   758→- Confirming that values read originated from valid writes (Origin invariant)
   759→

   762→## 8. Memory Safety
   763→
   764→Memory safety in VUMA is not optional — it is guaranteed by the Invariant Verification Engine (IVE), which checks every program against five core invariants. There is no `unsafe` keyword to bypass verification; instead, VUMA provides verification annotations and proof hints that help the IVE discharge proof obligations that would otherwise be too complex to verify automatically. This section describes the five invariants, the verification pipeline, and the annotation mechanisms available to programmers.
   765→
   766→### The Five LIVE Invariants
   767→
   768→The IVE checks the following invariants for every pointer operation in a VUMA program:
   769→
   770→| Invariant | Meaning | Violation Example |
   771→|---|---|---|
   772→| **Liveness** | Every requested resource will eventually be provided | Use after free, null dereference |
   773→| **Exclusivity** | At most one owner for exclusive resources | Data race (two concurrent writes) |
   774→| **Interpretation** | Every read interprets data under the correct BD | Type confusion, reading a pointer as an integer |
   775→| **Origin** | Every piece of data has a well-defined provenance | Reading uninitialized memory |
   776→| **Cleanup** | Every acquired resource is eventually released | Memory leak, leaked lock |
   777→
   778→### Verification Pipeline
   779→
   780→The IVE operates through a multi-stage pipeline:
   781→
   782→1. **Parsing:** Source code is parsed into an AST with full span information
   783→2. **SCG Construction:** The AST is lowered into a Semantic Compute Graph (SCG) — a DAG where nodes represent operations and edges represent data flow, control flow, and security boundaries
   784→3. **BD Inference:** The InferenceEngine walks the SCG and propagates BDs from leaf nodes, resolving composition at each node:
   785→   - Sequential composition: forward-propagate the output BD
   786→   - Parallel composition: intersect or unify BDs
   787→   - Conditional composition: take the union of branch BDs
   788→4. **Invariant Verification:** The VerificationEngine checks each of the five invariants against the annotated SCG
   789→5. **Debt Tracking:** Verification obligations that cannot be discharged immediately are recorded as debt items with priority levels
   790→
   791→```vuma
   792→// After compilation, the IVE produces a verification summary:
   793→// IVE verification: Liveness ✓, Exclusivity ✓, Interpretation ✓, Origin ✓, Cleanup ✓
   794→```
   795→
   796→### Verification Annotations
   797→
   798→VUMA provides several annotation mechanisms to help the IVE:
   799→
   800→**BD Annotations** (`#bd(Name)`): Explicitly specify a behavioral descriptor for a value or type. This overrides inferred BDs when the programmer has domain-specific knowledge:
   801→
   802→```vuma
   803→// Mark a buffer as read-only
   804→buffer: #bd(ReadOnly) Address = allocate(1024);
   805→
   806→// Mark a function as safe (manually verified)
   807→#bd(Safe) fn trusted_operation() -> i32 { ... }
   808→```
   809→
   810→**Proof Hints**: Provide the IVE with additional information about why an operation is safe. Proof hints can be explicit casts, runtime checks, or formal proof steps:
   811→
   812→```vuma
   813→// Explicit cast provides proof of safe reinterpretation
   814→let raw: u64 = ptr as u64;  // weakening: always safe
   815→
   816→// Runtime check provides proof of safe strengthening
   817→if tag == TYPE_PTR {
   818→    let recovered: *i32 = raw as *i32;  // safe because tag was checked
   819→}
   820→
   821→// Formal proof (advanced)
   822→// SafetyProof::FormalProof { steps: ["tag == TYPE_PTR implies raw is a valid pointer"] }
   823→```
   824→
   825→**Safe/Unsafe Blocks**: While VUMA has no `unsafe` escape hatch, the `safe` keyword can be used to assert that a block has been manually verified, creating a proof obligation that must be discharged:
   826→
   827→```vuma
   828→safe {
   829→    // The programmer asserts this code is safe
   830→    // IVE creates a proof obligation rather than checking automatically
   831→    *raw_ptr = value;
   832→}
   833→```
   834→
   835→### Verification Results
   836→
   837→The IVE produces one of three verdicts for each invariant:
   838→
   839→- **Proven:** The invariant holds on all execution paths (formal proof or exhaustive analysis)
   840→- **ProbablySafe:** The invariant holds under stated assumptions (e.g., lock protection, runtime checks)
   841→- **Violated:** A counterexample was found demonstrating the invariant can be broken
   842→
   843→```vuma
   844→// Example verification output:
   845→// Liveness:     Proven (exhaustive analysis)
   846→// Exclusivity:  ProbablySafe (2 conflicts protected by mutex locks)
   847→// Interpretation: Proven (formal proof: 6 steps)
   848→// Origin:       Proven (all reads trace to valid writes)
   849→// Cleanup:      Proven (all allocations freed on all paths)
   850→```

   854→## 9. Standard Library Overview
   855→
   856→VUMA's standard library provides essential data structures, memory management, I/O, formatting, mathematics, cryptography, string operations, concurrency, and filesystem primitives — all BD-annotated and verified by the IVE. The standard library is organized into modules that can be imported as needed.
   857→
   858→### Memory Management
   859→
   860→- **`allocate(size) -> Address`**: Reserve `size` bytes of memory. IVE verifies liveness and cleanup.
   861→- **`free(ptr)`**: Release a memory region. IVE marks all derived pointers as dead.
   862→- **`map_device(base, size) -> Address`**: Map a physical hardware address range into the program's address space. Implies volatile semantics; never needs to be freed.
   863→- **`align_to(ptr, alignment) -> Address`**: Align a pointer to the given boundary. Part of the derivation chain.
   864→- **Allocation strategies**: `GlobalAllocator`, `ArenaAllocator`, `BumpAllocator`, `PoolAllocator`, `FreeListAllocator` — each with BD-annotated allocation and deallocation.
   865→
   866→```vuma
   867→import "vuma:mem";
   868→
   869→region pool = allocate(4096);
   870→*pool = 0;
   871→free(pool);
   872→```
   873→
   874→### Data Structures
   875→
   876→- **`Queue<T>`**: Lock-free single-producer single-consumer ring buffer with `AtomicU64` indices
   877→- **`Arena`**: Bump allocator with bulk invalidation on destroy
   878→- **`NodeHeader`**: Doubly-linked list node with `prev`, `next`, and `data` fields
   879→- **`Vec<T>`**: Growable array with BD-annotated push/pop/insert/remove
   880→- **`HashMap<K, V>`**: Hash map using SipHash-1-3 with BD-tracked keys/values
   881→- **`RingBuffer<T>`**: Fixed-capacity circular buffer
   882→
   883→```vuma
   884→import "vuma:ds";
   885→
   886→q = queue_new(256);
   887→queue_push(q, 42);
   888→value = queue_pop(q);  // Some(42)
   889→```
   890→
   891→### Concurrency
   892→
   893→- **`AtomicU64`**: 64-bit atomic integer with `load(ordering)`, `store(value, ordering)`, `compare_exchange(expected, desired, success_order, failure_order)`
   894→- **`channel<T>`**: Typed message-passing channel with `send` and `recv`
   895→- **`lock(id)` / `unlock(id)`**: Mutual exclusion primitives
   896→- **`Mutex`**: BD-annotated mutual exclusion with `MutexGuard` RAII pattern
   897→- **`RwLock`**: Reader-writer lock with separate read/write guards
   898→- **`Barrier`**: Synchronization barrier for multi-threaded coordination
   899→
   900→```vuma
   901→import "vuma:sync";
   902→
   903→counter = AtomicU64::new(0);
   904→counter.store(1, Release);
   905→val = counter.load(Acquire);
   906→```
   907→
   908→### I/O
   909→
   910→- **`print(value)`**: Output a value to the standard output
   911→- **`delay_ms(ms)`**: Busy-wait for the specified number of milliseconds (bare-metal)
   912→- **`VumaReader` / `VumaWriter`**: Core I/O traits with UART and file backends
   913→- **`VumaBufReader` / `VumaBufWriter`**: Buffered I/O wrappers
   914→- **`read_bytes` / `write_bytes`**: Low-level syscall wrappers
   915→- **`read_u32_le` / `write_u32_le`**: Little-endian u32 byte access
   916→
   917→```vuma
   918→import "vuma:io";
   919→
   920→print((*current).data);
   921→delay_ms(500);
   922→```
   923→
   924→### String Formatting (fmt)
   925→
   926→The `fmt` module provides printf-style string formatting for VUMA programs:
   927→
   928→- **`format_int(value, base, width)`**: Format a signed integer in the given base (2–36) with zero-padding
   929→- **`format_uint(value, base, width)`**: Format an unsigned integer
   930→- **`format_float(value, precision)`**: Format a floating-point number with configurable precision
   931→- **`format_hex(value, width)`**: Format as hexadecimal
   932→- **`format_binary(value, width)`**: Format as binary
   933→- **`format_octal(value, width)`**: Format as octal
   934→- **`format_pointer(addr)`**: Format an address as a hex pointer
   935→- **`pad_left(s, width, fill)`** / **`pad_right(s, width, fill)`**: String padding
   936→- **`join(strings, separator)`**: Join multiple strings with a separator
   937→- **`write_str(buf, offset, s)`** / **`write_int(buf, offset, value, base)`** / **`write_float(buf, offset, value, precision)`**: Write formatted output into byte buffers
   938→

  1029→## 10. FFI and External Functions
  1030→
  1031→VUMA supports calling external C functions and Linux syscalls through `extern "C"` blocks. This enables interoperability with existing C libraries and direct system call access across all 8 backend architectures.
  1032→
  1033→### Extern Block Syntax
  1034→
  1035→The `extern "C" { ... }` block declares external functions that are resolved at link time:
  1036→
  1037→```vuma
  1038→extern "C" {
  1039→    fn write(fd: i64, buf: Address, count: i64) -> i64;
  1040→    fn read(fd: i64, buf: Address, count: i64) -> i64;
  1041→    fn exit(code: i64);
  1042→}
  1043→
  1044→fn main() -> i32 {
  1045→    write(1, message, 13);
  1046→    return 0;
  1047→}
  1048→```
  1049→
  1050→Functions declared in `extern` blocks have the `is_extern` flag set in the SCG. During code generation, extern function calls emit relocations instead of local `BL` instructions, allowing the linker to resolve them against shared libraries.
  1051→
  1052→### Supported Syscalls
  1053→
  1054→VUMA provides FFI bindings for 19 Linux syscalls across all 8 architectures:
  1055→
  1056→`read`, `write`, `open`, `close`, `exit`, `mmap`, `munmap`, `brk`, `ioctl`, `fcntl`, `getpid`, `clone`, `futex`, `pipe`, `dup`, `dup2`, `wait4`, `rt_sigaction`, `rt_sigprocmask`
  1057→
  1058→Each syscall has architecture-specific relocations for all backends.
  1059→
  1060→### DWARF Debug Information
  1061→
  1062→When compiled with `--debug-info`, VUMA generates DWARF v4 debug information sections for all 8 backends. The emitted sections include:
  1063→
  1064→| Section | Contents |
  1065→|---------|----------|
  1066→| `.debug_abbrev` | Abbreviation tables (tag + attribute encodings) |
  1067→| `.debug_info` | Compilation unit DIEs (subprograms, variables) |
  1068→| `.debug_line` | Line-number program (DWARF standard opcodes) |
  1069→| `.debug_frame` | Call frame information (CIE + FDE entries) |
  1070→
  1071→The DWARF builder is parameterized by address size to support all 8 backends (8-byte addresses for 64-bit targets, 4-byte for ARM32 and Wasm32).

  1075→## 11. Platform-Specific Features
  1076→
  1077→VUMA targets 8 backend architectures with AArch64 as the primary platform. This section describes platform-specific features, with detailed AArch64 bare-metal support.
  1078→
  1079→### Device Memory Mapping
  1080→
  1081→The `map_device(base, size)` intrinsic maps a physical address range into the program's virtual address space. On AArch64, this is used to access hardware peripherals such as GPIO, UART, and DMA controllers. The IVE treats mapped device regions specially:
  1082→
  1083→- They never need to be freed (hardware is always present)
  1084→- They have volatile semantics (reads and writes have side effects)
  1085→- They have a known size for bounds checking
  1086→
  1087→```vuma
  1088→const GPIO_BASE: Address = 0x7e200000;  // AArch64 GPIO base (BCM address)
  1089→
  1090→fn gpio_set_output(pin: u32) {
  1091→    gpio = map_device(GPIO_BASE, 4096);
  1092→    fsel = gpio + 0x00;  // GPFSEL0 register
  1093→    *fsel = (*fsel & ~(7 << (pin * 3))) | (1 << (pin * 3));
  1094→}
  1095→
  1096→fn gpio_set(pin: u32) {
  1097→    gpio = map_device(GPIO_BASE, 4096);
  1098→    *(gpio + 0x1c) = 1 << pin;  // GPSET0 register
  1099→}
  1100→
  1101→fn gpio_clear(pin: u32) {
  1102→    gpio = map_device(GPIO_BASE, 4096);
  1103→    *(gpio + 0x28) = 1 << pin;  // GPCLR0 register
  1104→}
  1105→```
  1106→
  1107→### GPIO Register Layout
  1108→
  1109→The AArch64 GPIO peripheral is memory-mapped at BCM address `0x7e200000`. The key register offsets are:
  1110→
  1111→| Register | Offset | Purpose |
  1112→|---|---|---|
  1113→| GPFSEL0 | 0x00 | Function select for pins 0-9 |
  1114→| GPFSEL1 | 0x04 | Function select for pins 10-19 |
  1115→| GPFSEL2 | 0x08 | Function select for pins 20-29 |
  1116→| GPSET0 | 0x1c | Set pin high |
  1117→| GPCLR0 | 0x28 | Set pin low |
  1118→| GPLEV0 | 0x34 | Pin level read |
  1119→
  1120→Each pin uses 3 bits in the function select register: `000` = input, `001` = output.
  1121→
  1122→### ARM64 Code Generation
  1123→
  1124→VUMA compiles to native ARM64 machine code. The code generator targets the Cortex-A76 (AArch64's SoC) and produces optimized instruction sequences for:
  1125→- Pointer arithmetic using register-offset addressing
  1126→- Atomic operations using `LDXR`/`STXR` (exclusive access) and `LDA`/`STL` (acquire/release)
  1127→- Volatile device access using explicit load/store instructions (no optimization)
  1128→- Stack frame layout aligned to 16-byte boundaries per AAPCS64
  1129→
  1130→### Bare-Metal Execution
  1131→
  1132→VUMA programs can run bare-metal on the AArch64 without an operating system. The runtime provides:
  1133→- **Startup code:** Sets up the stack pointer, clears BSS, calls `main()`
  1134→- **Memory allocator:** Simple bump allocator for the `allocate()` primitive
  1135→- **Delay loops:** `delay_ms()` uses the AArch64's system timer at `0x7e003000`
  1136→- **UART output:** `print()` writes to the AArch64's AUX UART for console output
  1137→
  1138→```vuma
  1139→// Complete bare-metal LED blink program for AArch64
  1140→const GPIO_BASE: Address = 0x7e200000;
  1141→
  1142→fn main() -> i32 {
  1143→    led_pin: u32 = 25;
  1144→    gpio_set_output(led_pin);
  1145→    loop {
  1146→        gpio_set(led_pin);
  1147→        delay_ms(500);
  1148→        gpio_clear(led_pin);
  1149→        delay_ms(500);
  1150→    }
  1151→    return 0;
  1152→}
  1153→```
  1154→
  1155→### Const Addresses
  1156→
  1157→Hardware register addresses are declared as `const` values with the `Address` type. The IVE treats these as known constants and can verify at compile time that register accesses stay within mapped device regions:
  1158→
  1159→```vuma
  1160→const GPIO_BASE: Address = 0x7e200000;
  1161→const GPFSEL0_OFFSET: Address = 0x00;
  1162→const GPSET0_OFFSET: Address = 0x1c;
  1163→const GPCLR0_OFFSET: Address = 0x28;
  1164→```
