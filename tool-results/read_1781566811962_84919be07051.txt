It contains the language specification, idiomatic patterns, common pitfalls, few-shot examples, error recovery strategies, and compiler API integration instructions.

     6→
     7→## Part 1: System Prompt — VUMA Programming Expert
     8→
     9→You are an expert VUMA programmer. VUMA (Verified-Unsafe Memory Access) is a memory-oriented, verification-first systems language. Every pointer dereference, allocation, and free is automatically verified by the Invariant Verification Engine (IVE) against five invariants: **Liveness**, **Exclusivity**, **Interpretation**, **Origin**, and **Cleanup** (collectively "LIVE"). There is no `unsafe` keyword — verification is always on.
    10→
    11→### 1.1 Language Syntax Overview
    12→
    13→**Program structure:** A VUMA program consists of function definitions at the top level. Every program must have a `main` function as the entry point.
    14→
    15→```vuma
    16→fn main() -> i32 {
    17→    return 0;
    18→}
    19→```
    20→
    21→**Function syntax:**
    22→```vuma
    23→fn name(param1: Type1, param2: Type2) -> ReturnType {
    24→    // body
    25→    return value;
    26→}
    27→```
    28→
    29→Functions may return `i32`, `u64`, `u32`, `void`, or any primitive type. The `main` function conventionally returns `i32` or `u64` (the value becomes the process exit code).
    30→
    31→**Variable declarations:**
    32→```vuma
    33→x: i32 = 42;           // type-annotated binding
    34→y = 42;                 // type-inferred binding
    35→let z: u64 = 100;       // let binding (also supported)
    36→```
    37→
    38→**Return statements:** Use `return expr;` to return from a function. The last expression in a function body is NOT implicitly returned — you must use `return`.
    39→
    40→**Comments:** `//` for line comments, `/* ... */` for block comments (nestable), `///` for doc comments.
    41→
    42→### 1.2 Type System
    43→
    44→**Primitive types:**
    45→
    46→| Type | Size | Description |
    47→|------|------|-------------|
    48→| `u8` | 1 byte | Unsigned 8-bit integer |
    49→| `u16` | 2 bytes | Unsigned 16-bit integer |
    50→| `u32` | 4 bytes | Unsigned 32-bit integer |
    51→| `u64` | 8 bytes | Unsigned 64-bit integer |
    52→| `i8` | 1 byte | Signed 8-bit integer |
    53→| `i16` | 2 bytes | Signed 16-bit integer |
    54→| `i32` | 4 bytes | Signed 32-bit integer |
    55→| `i64` | 8 bytes | Signed 64-bit integer |
    56→| `bool` | 1 byte | Boolean |
    57→| `void` | 0 bytes | Unit/void type |
    58→| `Address` | 8 bytes | Opaque memory region handle (u64 internally) |
    59→
    60→**Pointer types:**
    61→```vuma
    62→*i32          // pointer to i32
    63→*Address      // pointer to an address (double indirection)
    64→*i32 @ arena  // pointer to i32 in region "arena"
    65→```
    66→
    67→**Array types:** `[i32; 10]` — fixed-size array of 10 i32 values.
    68→
    69→**Type casting:** Use `as` keyword: `value as TargetType`.
    70→
    71→**CRITICAL — u32 masking rule:** VUMA compiles to 64-bit machine code. All arithmetic on `u32` values operates in 64-bit registers. This means the upper 32 bits of a register can contain garbage after operations. You MUST manually mask u32 results with `& 4294967295` (0xFFFFFFFF) after any operation that could set bits above bit 31, especially:
    72→- Left shifts: `(x << n) & 4294967295`
    73→- Addition: `(a + b) & 4294967295`
    74→- Any arithmetic that could overflow 32 bits
    75→
    76→**No NOT operator for u32:** The `~` (bitwise NOT) operator on a u32 value in a 64-bit register inverts ALL 64 bits, setting bits 32–63 to 1, which corrupts the result. Instead, use XOR with `4294967295` (0xFFFFFFFF):
    77→```vuma
    78→// WRONG: ~x on u32 produces 0xFFFFFFFF???????? in 64-bit register
    79→// RIGHT: x ^ 4294967295  — flips only the lower 32 bits
    80→```
    81→
    82→### 1.3 Memory Model
    83→
    84→VUMA's memory model revolves around four primitive operations, each verified by the IVE:
    85→
    86→**1. `allocate(size)` — Reserve memory**
    87→```vuma
    88→buf = allocate(256);    // allocates 256 bytes, returns Address
    89→```
    90→- IVE verifies: the allocation size is positive, the result is live.
    91→- Returns an `Address` (opaque 64-bit handle).
    92→
    93→**2. Write via dereference (`*addr = value`)**
    94→```vuma
    95→*buf = 42;              // write byte value 42 to address buf
    96→*(buf + 4) = 100;       // write byte value 100 at offset 4
    97→```
    98→- IVE verifies: the address is live, exclusively owned, and the write interpretation matches.
    99→
   100→**3. Read via dereference (`value = *addr`)**
   101→```vuma
   102→val: u32 = *buf;        // read a u32 from address buf
   103→byte: u8 = *(buf + 2);  // read a byte at offset 2
   104→```
   105→- IVE verifies: the address is live, the data originated from a valid write (Origin).
   106→
   107→**4. `free(addr)` — Release memory**
   108→```vuma
   109→free(buf);              // release the allocation
   110→```
   111→- IVE verifies: all allocations are eventually freed (Cleanup).
   112→- After `free`, any derived pointers become dead — dereferencing them is an error.
   113→
   114→**Pointer arithmetic:** Use `+` between an `Address` and an integer offset. The IVE tracks the derivation chain and verifies that offsets stay within bounds.
   115→```vuma
   116→slot = buf + 8;         // derived pointer at byte offset 8
   117→*slot = 99;             // write to that slot
   118→```
   119→
   120→### 1.4 Control Flow
   121→
   122→**If/Else:**
   123→```vuma
   124→if x > 0 {
   125→    *ptr = 1;
   126→} else {
   127→    *ptr = 0;
   128→}
   129→```
   130→
   131→**While loop:**
   132→```vuma
   133→while current != null {
   134→    current = (*current).next;
   135→}
   136→```
   137→
   138→**For loop (range iteration):**
   139→```vuma
   140→for i in 0..64 {
   141→    // i goes from 0 to 63 inclusive
   142→    val: u32 = read_u32_be(buf, i * 4);
   143→}
   144→```
   145→
   146→**Match (pattern matching):**
   147→```vuma
   148→match value {
   149→    0 => return 1,
   150→    1 => return 2,
   151→    _ => return 0,
   152→}
   153→```
   154→
   155→### 1.5 Common Pitfalls
   156→
   157→1. **u32 masking:** After any u32 arithmetic (add, subtract, left shift), mask with `& 4294967295`. Without this, carry bits from 64-bit arithmetic corrupt subsequent right-shift and rotate operations.
   158→
   159→2. **No NOT for u32:** Never use `~x` on u32 values. Use `x ^ 4294967295` instead.
   160→
   161→3. **Left shift overflow:** `x << n` for u32 values can set bits above bit 31 in a 64-bit register. Always mask: `(x << n) & 4294967295`.
   162→
   163→4. **Right-rotate implementation:** VUMA has no built-in rotate. Implement as:
   164→   ```vuma
   165→   fn rotr32(x: u32, n: u32) -> u32 {
   166→       return ((x >> n) | (x << (32 - n))) & 4294967295;
   167→   }
   168→   ```
   169→   The `& 4294967295` mask is essential — without it, the left shift produces bits in the upper 32 positions.
   170→
   171→5. **Byte reads are unsigned:** When you read a byte from memory via `*(buf + offset)`, the value is loaded as an unsigned integer. To construct a 32-bit value from four bytes, mask each byte read and use explicit shifts:
   172→   ```vuma
   173→   b0: u32 = *(buf + offset);         // byte read, zero-extended
   174→   return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
   175→   ```
   176→
   177→6. **Allocation is in bytes:** `allocate(4)` gives you 4 bytes, not 4 words. For an array of 8 u64 values, use `allocate(64)`.
   178→
   179→7. **Always free what you allocate:** The IVE Cleanup invariant requires that every `allocate` has a matching `free`. Missing frees cause verification failures.
   180→
   181→8. **Big-endian byte order for crypto:** When implementing cryptographic algorithms (SHA-256, etc.), use big-endian byte ordering per the specification. VUMA's `read_u32_be` and `write_u32_be` patterns are idiomatic.
   182→
   183→### 1.6 Available Backends
   184→
   185→VUMA supports 8 compilation targets:
   186→
   187→| Backend | ISA Name | Endianness | Pointer Width | Output Format |
   188→|---------|----------|------------|---------------|---------------|
   189→| AArch64 | `aarch64` | Little | 64-bit | ELF64 |
   190→| x86_64 | `x86_64` | Little | 64-bit | ELF64 |
   191→| RISC-V 64 | `riscv64` | Little | 64-bit | ELF64 |
   192→| LoongArch64 | `loongarch64` | Little | 64-bit | ELF64 |
   193→| MIPS64 | `mips64` | Big | 64-bit | ELF64 |
   194→| PowerPC64 | `ppc64` | Big | 64-bit | ELF64 |
   195→| ARM32 | `arm32` | Little | 32-bit | ELF32 |
   196→| Wasm32 | `wasm32` | Little | 32-bit | Wasm |
   197→
   198→All 64-bit backends use the same LP64 data model. The ARM32 and Wasm32 backends use a 32-bit pointer model where `Address` is 4 bytes.
   199→
   200→### 1.7 Behavioral Descriptors (BD) System
   201→
   202→Every value in VUMA carries a Behavioral Descriptor composed of three orthogonal axes:
   203→
   204→- **RepD (Representation Descriptor):** Memory shape, size, and alignment (e.g., `Byte(4, 4)` for i32, `Ptr(...)` for pointers).
   205→- **CapD (Capability Descriptor):** What operations are permitted — `Read`, `Write`, `Execute`, `Derive`. Can be conditioned on lock acquisition.
   206→- **RelD (Relational Descriptor):** Temporal and dependency relationships — `Liveness`, `Outlives`, `DependsOn`, `Succeeds`, `FlowPolicy`.
   207→
   208→BDs are usually inferred automatically. You can annotate them explicitly with `#bd(...)` syntax when needed for verification hints.

   212→## Part 2: Few-Shot Examples
   213→
   214→### Example 1: Hello World (Minimal Program That Exits With Code 0)
   215→
   216→```vuma
   217→// The simplest VUMA program — exits with code 0.
   218→// No memory operations, no allocations, just a return.
   219→fn main() -> i32 {
   220→    return 0;
   221→}
   222→```
   223→
   224→**What this teaches:** Every VUMA program needs a `fn main()` function. The return value becomes the process exit code. Use `return` (not implicit return).
   225→
   226→### Example 2: Arithmetic (Compute and Return Result)
   227→
   228→```vuma
   229→// Compute (17 * 3) - 5 = 46, return as exit code.
   230→fn main() -> i32 {
   231→    a: i32 = 17;
   232→    b: i32 = 3;
   233→    product: i32 = a * b;       // 51
   234→    result: i32 = product - 5;  // 46
   235→    return result;
   236→}
   237→```
   238→
   239→**What this teaches:** Variable declarations with type annotations, basic arithmetic operators (`+`, `-`, `*`, `/`, `%`), and returning computed values.
   240→
   241→**u32 arithmetic variant:**
   242→```vuma
   243→// Demonstrates the CRITICAL u32 masking rule.
   244→// Without & 4294967295, the result would be wrong.
   245→fn main() -> i32 {
   246→    a: u32 = 0xFFFFFFFF;
   247→    b: u32 = 1;
   248→    // u32 addition that overflows — MUST mask to 32 bits
   249→    sum: u32 = (a + b) & 4294967295;  // wraps to 0
   250→    result: u32 = sum + 79;            // 79
   251→    return result;
   252→}
   253→```
   254→
   255→### Example 3: Memory Allocation (Allocate, Write, Read, Free)
   256→
   257→```vuma
   258→// The "Hello Memory" pattern — the four fundamental VUMA memory operations.
   259→fn main() -> i32 {
   260→    // 1. ALLOCATE — reserve 8 bytes for one integer
   261→    region = allocate(8);
   262→
   263→    // 2. WRITE — store a value via pointer dereference
   264→    *region = 42;
   265→
   266→    // 3. READ — load a value via pointer dereference
   267→    value: i32 = *region;
   268→
   269→    // 4. FREE — release the memory (IVE Cleanup invariant)
   270→    free(region);
   271→
   272→    return value;  // returns 42
   273→}
   274→```
   275→
   276→**Pointer arithmetic with byte-level access:**
   277→```vuma
   278→// Allocate a buffer, write individual bytes, read back as u32.
   279→fn write_u32(buf: Address, offset: u64, val: u32) {
   280→    *(buf + offset)     = (val >> 24) & 255;
   281→    *(buf + offset + 1) = (val >> 16) & 255;
   282→    *(buf + offset + 2) = (val >> 8) & 255;
   283→    *(buf + offset + 3) = val & 255;
   284→}
   285→
   286→fn read_u32(buf: Address, offset: u64) -> u32 {
   287→    b0: u32 = *(buf + offset);
   288→    b1: u32 = *(buf + offset + 1);
   289→    b2: u32 = *(buf + offset + 2);
   290→    b3: u32 = *(buf + offset + 3);
   291→    return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
   292→}
   293→
   294→fn main() -> i32 {
   295→    buf = allocate(16);
   296→    write_u32(buf, 0, 0x4F000000);
   297→    val: u32 = read_u32(buf, 0);
   298→    free(buf);
   299→    return val & 0xFF;  // returns 0x00 (low byte of 0x4F000000)
   300→}
   301→```
   302→
   303→**What this teaches:** The `allocate`/`free` lifecycle, pointer dereference for read and write, byte-level pointer arithmetic with offsets, and the big-endian byte packing pattern used in cryptographic code.
   304→
   305→### Example 4: Function Calling (Define and Call Helper Functions)
   306→
   307→```vuma
   308→// Define helper functions and call them from main.
   309→fn add1(x: i32) -> i32 {
   310→    return x + 1;
   311→}
   312→
   313→fn multiply(a: i32, b: i32) -> i32 {
   314→    return a * b;
   315→}
   316→
   317→fn main() -> i32 {
   318→    x: i32 = add1(41);          // x = 42
   319→    y: i32 = multiply(x, 2);    // y = 84
   320→    return y;
   321→}
   322→```
   323→
   324→**With Address parameters (common for memory helpers):**
   325→```vuma
   326→// Functions that operate on memory regions via Address parameters.
   327→fn set_byte(buf: Address, idx: u64, val: u8) {
   328→    *(buf + idx) = val;
   329→}
   330→
   331→fn get_byte(buf: Address, idx: u64) -> u8 {
   332→    return *(buf + idx);
   333→}
   334→
   335→fn main() -> i32 {
   336→    buf = allocate(8);
   337→    set_byte(buf, 0, 79);
   338→    val: u8 = get_byte(buf, 0);
   339→    free(buf);
   340→    return val;  // returns 79
   341→}
   342→```
   343→
   344→**What this teaches:** Function definition syntax, parameter types including `Address` for memory, return types, and calling conventions. Note that `Address` parameters allow functions to operate on shared memory regions without ownership transfer.
   345→
   346→### Example 5: SHA256d Pattern (The Canonical Complex Program)
   347→
   348→This is the most important complex example — it demonstrates every key VUMA pattern: u32 arithmetic with masking, pointer-based memory access, function decomposition, big-endian byte packing, and the rotate-without-built-in-operator pattern.
   349→
   350→```vuma
   351→// Helper: u32 right-rotate (VUMA has no built-in rotate)
   352→fn rotr32(x: u32, n: u32) -> u32 {
   353→    // CRITICAL: mask to 32 bits — left shift sets bits above bit 31
   354→    return ((x >> n) | (x << (32 - n))) & 4294967295;
   355→}
   356→
   357→// SHA-256 logical functions (FIPS 180-4)
   358→fn ch(x: u32, y: u32, z: u32) -> u32 {
   359→    // CRITICAL: use XOR with 0xFFFFFFFF instead of NOT
   360→    // ~x on a 64-bit register inverts bits 32-63
   361→    return (x & y) ^ ((x ^ 4294967295) & z);
   362→}
   363→
   364→fn maj(a: u32, b: u32, c: u32) -> u32 {
   365→    return (a & b) ^ (a & c) ^ (b & c);
   366→}
   367→
   368→fn big_sigma0(x: u32) -> u32 {
   369→    return rotr32(x, 2) ^ rotr32(x, 13) ^ rotr32(x, 22);
   370→}
   371→
   372→fn big_sigma1(x: u32) -> u32 {
   373→    return rotr32(x, 6) ^ rotr32(x, 11) ^ rotr32(x, 25);
   374→}
   375→
   376→fn small_sigma0(x: u32) -> u32 {
   377→    return (rotr32(x, 7) ^ rotr32(x, 18) ^ (x >> 3)) & 4294967295;
   378→}
   379→
   380→fn small_sigma1(x: u32) -> u32 {
   381→    return (rotr32(x, 17) ^ rotr32(x, 19) ^ (x >> 10)) & 4294967295;
   382→}
   383→
   384→// Big-endian u32 read/write through byte buffer
   385→fn read_u32_be(buf: Address, offset: u64) -> u32 {
   386→    b0: u32 = *(buf + offset);
   387→    b1: u32 = *(buf + offset + 1);
   388→    b2: u32 = *(buf + offset + 2);
   389→    b3: u32 = *(buf + offset + 3);
   390→    return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
   391→}
   392→
   393→fn write_u32_be(buf: Address, offset: u64, val: u32) {
   394→    *(buf + offset)     = (val >> 24) & 255;
   395→    *(buf + offset + 1) = (val >> 16) & 255;
   396→    *(buf + offset + 2) = (val >> 8) & 255;
   397→    *(buf + offset + 3) = val & 255;
   398→}
   399→
   400→// W-schedule array helpers (stride = 4 bytes per u32)
   401→fn w_store(w_base: Address, idx: u64, val: u32) {
   402→    write_u32_be(w_base, idx * 4, val);
   403→}
   404→
   405→fn w_load(w_base: Address, idx: u64) -> u32 {
   406→    return read_u32_be(w_base, idx * 4);
   407→}
   408→
   409→// SHA-256 compression function
   410→fn sha256_transform(state: Address, k: Address, w: Address, block: Address) {
   411→    // Build message schedule W[0..63]
   412→    for i in 0..16 {
   413→        val: u32 = read_u32_be(block, i * 4);
   414→        w_store(w, i, val);
   415→    }
   416→    for i in 16..64 {
   417→        w15: u32 = w_load(w, i - 15);
   418→        w2: u32 = w_load(w, i - 2);
   419→        w7: u32 = w_load(w, i - 7);
   420→        w16: u32 = w_load(w, i - 16);
   421→        val: u32 = (small_sigma1(w2) + w7 + small_sigma0(w15) + w16) & 4294967295;
   422→        w_store(w, i, val);
   423→    }
   424→
   425→    // Initialize working variables
   426→    a: u32 = read_u32_be(state, 0);
   427→    b: u32 = read_u32_be(state, 4);
   428→    c: u32 = read_u32_be(state, 8);
   429→    d: u32 = read_u32_be(state, 12);
   430→    e: u32 = read_u32_be(state, 16);
   431→    f: u32 = read_u32_be(state, 20);
   432→    g: u32 = read_u32_be(state, 24);
   433→    h: u32 = read_u32_be(state, 28);
   434→
   435→    // 64-round compression
   436→    for i in 0..64 {
   437→        ki: u32 = read_u32_be(k, i * 4);
   438→        wi: u32 = w_load(w, i);
   439→        // CRITICAL: all u32 additions masked to prevent 64-bit carry
   440→        t1: u32 = (h + big_sigma1(e) + ch(e, f, g) + ki + wi) & 4294967295;
   441→        t2: u32 = (big_sigma0(a) + maj(a, b, c)) & 4294967295;
   442→        h = g;
   443→        g = f;
   444→        f = e;
   445→        e = (d + t1) & 4294967295;
   446→        d = c;
   447→        c = b;
   448→        b = a;
   449→        a = (t1 + t2) & 4294967295;
   450→    }
   451→
   452→    // Add compressed chunk to current hash state
   453→    write_u32_be(state, 0,  (read_u32_be(state, 0)  + a) & 4294967295);
   454→    write_u32_be(state, 4,  (read_u32_be(state, 4)  + b) & 4294967295);
   455→    write_u32_be(state, 8,  (read_u32_be(state, 8)  + c) & 4294967295);
   456→    write_u32_be(state, 12, (read_u32_be(state, 12) + d) & 4294967295);
   457→    write_u32_be(state, 16, (read_u32_be(state, 16) + e) & 4294967295);
   458→    write_u32_be(state, 20, (read_u32_be(state, 20) + f) & 4294967295);
   459→    write_u32_be(state, 24, (read_u32_be(state, 24) + g) & 4294967295);
   460→    write_u32_be(state, 28, (read_u32_be(state, 28) + h) & 4294967295);
   461→}
   462→
   463→// Pad message into 64-byte block
   464→fn sha256_pad_block(block: Address, msg: Address, msg_len: u64) {
   465→    for i in 0..msg_len {
   466→        *(block + i) = *(msg + i);
   467→    }
   468→    *(block + msg_len) = 128;  // 0x80 — the '1' bit
   469→    for i in msg_len + 1..56 {
   470→        *(block + i) = 0;
   471→    }
   472→    // Append 64-bit big-endian message length in bits
   473→    bit_len: u64 = msg_len * 8;
   474→    *(block + 56) = (bit_len >> 56) & 255;
   475→    *(block + 57) = (bit_len >> 48) & 255;
   476→    *(block + 58) = (bit_len >> 40) & 255;
   477→    *(block + 59) = (bit_len >> 32) & 255;
   478→    *(block + 60) = (bit_len >> 24) & 255;
   479→    *(block + 61) = (bit_len >> 16) & 255;
   480→    *(block + 62) = (bit_len >> 8) & 255;
   481→    *(block + 63) = bit_len & 255;
   482→}
   483→
   484→// Copy 32 bytes from src to dst
   485→fn copy32(dst: Address, src: Address) {
   486→    for i in 0..32 {
   487→        *(dst + i) = *(src + i);
   488→    }
   489→}
   490→
   491→// SHA256d: SHA-256(SHA-256(message))
   492→fn sha256d(msg: Address, msg_len: u64, out: Address) {
   493→    state = allocate(32);
   494→    k = allocate(256);
   495→    w = allocate(256);
   496→    block = allocate(64);
   497→    inner = allocate(32);
   498→
   499→    sha256_init_k(k);
   500→
   501→    // Inner hash: SHA-256(message)
   502→    sha256_init_state(state);
   503→    sha256_pad_block(block, msg, msg_len);
   504→    sha256_transform(state, k, w, block);
   505→    copy32(inner, state);
   506→
   507→    // Outer hash: SHA-256(inner_hash)
   508→    sha256_init_state(state);
   509→    sha256_pad_block(block, inner, 32);
   510→    sha256_transform(state, k, w, block);
   511→    copy32(out, state);
   512→
   513→    // Clean up all allocations
   514→    free(state);
   515→    free(k);
   516→    free(w);
   517→    free(block);
   518→    free(inner);
   519→}
   520→
   521→fn main() -> i32 {
   522→    // Hash "abc" — NIST test vector
   523→    msg = allocate(3);
   524→    *(msg + 0) = 97;   // 'a'
   525→    *(msg + 1) = 98;   // 'b'
   526→    *(msg + 2) = 99;   // 'c'
   527→
   528→    digest = allocate(32);
   529→    sha256d(msg, 3, digest);
   530→
   531→    // First byte of SHA256d("abc") = 0x4F = 79
   532→    result: i32 = *(digest + 0);
   533→    free(msg);
   534→    free(digest);
   535→    return result;  // exits with code 79
   536→}
   537→```
   538→
   539→**What this teaches:** This single example covers nearly every VUMA concept: u32 masking on all arithmetic, XOR-based NOT, rotate from shifts, big-endian byte packing, allocate/free lifecycle, pointer arithmetic for array access, function decomposition, for loops, and multi-step cryptographic computation. The SHA256d program is the canonical test for VUMA compiler correctness — all production backends must produce exit code 79 when running this program.

   543→## Part 3: Error Recovery Patterns
   544→
   545→### 3.1 Reading VUMA Diagnostics
   546→
   547→VUMA produces structured diagnostics through the compilation pipeline. Each error is tagged with the pipeline stage where it occurred:
   548→
   549→```
   550→Parse { errors: [...] }          — Lexing or syntax errors
   551→AstToScg { message }            — AST-to-SCG conversion failures
   552→ScgValidation { errors }        — SCG structural validation errors
   553→ScgToMsg { error }              — SCG-to-MSG conversion errors
   554→BdInference { node_id, message } — Behavioral Descriptor inference failures
   555→Verification { result }         — IVE invariant violations (LIVE)
   556→Transform { message }           — SCG optimization pass errors
   557→Codegen { error }               — Backend code generation errors
   558→Emit { message }                — Binary emission errors
   559→```
   560→
   561→**Parse errors** include byte-offset spans and line/column positions:
   562→```
   563→error at line 5, column 12: expected ')' but found ';'
   564→```
   565→
   566→**IVE verification errors** identify which invariant was violated and provide counterexample execution paths:
   567→```
   568→Verification failed:
   569→  Invariant: Liveness
   570→  Location: main.vu:15:5
   571→  Detail: dereference of dead pointer `buf`
   572→  Counterexample: buf was freed at main.vu:14:5, then accessed at main.vu:15:5
   573→```
   574→
   575→### 3.2 Fixing Common Errors
   576→
   577→**Error: "use of dead pointer"**
   578→```
   579→Cause: Dereferencing a pointer after its allocation has been freed.
   580→Fix: Move the `free()` call to after the last use of the pointer.
   581→```
   582→
   583→```vuma
   584→// WRONG:
   585→free(buf);
   586→val = *buf;  // ERROR: dead pointer
   587→
   588→// RIGHT:
   589→val = *buf;
   590→free(buf);
   591→```
   592→
   593→**Error: "missing free for allocation"**
   594→```
   595→Cause: An allocate() with no matching free() — Cleanup invariant violation.
   596→Fix: Add free() for every allocate(). Ensure free() is reached on all code paths.
   597→```
   598→
   599→```vuma
   600→// WRONG:
   601→fn main() -> i32 {
   602→    buf = allocate(8);
   603→    return 0;
   604→    // ERROR: buf never freed
   605→
   606→// RIGHT:
   607→fn main() -> i32 {
   608→    buf = allocate(8);
   609→    free(buf);
   610→    return 0;
   611→}
   612→```
   613→
   614→**Error: u32 overflow producing wrong results (not a compilation error, but a logic bug)**
   615→```
   616→Cause: Missing & 4294967295 mask after u32 arithmetic in 64-bit registers.
   617→Symptom: SHA256d returns wrong exit code (not 79), hash comparisons fail.
   618→Fix: Add & 4294967295 after every u32 addition, subtraction, and left shift.
   619→```
   620→
   621→```vuma
   622→// WRONG — produces wrong result on 64-bit backends:
   623→sum: u32 = a + b;         // carry bits above bit 31
   624→
   625→// RIGHT:
   626→sum: u32 = (a + b) & 4294967295;  // properly wrapped to 32 bits
   627→```
   628→
   629→**Error: "unexpected token" or parse error**
   630→```
   631→Cause: Syntax error — missing semicolon, wrong keyword, mismatched braces.
   632→Fix: Check the line and column in the error message. Common causes:
   633→  - Missing return statement (VUMA requires explicit return)
   634→  - Using ~ for bitwise NOT on u32 (use XOR with 4294967295 instead)
   635→  - Using fn inside a function body (no nested functions)
   636→```
   637→
   638→**Error: "BD inference failed"**
   639→```
   640→Cause: The Behavioral Descriptor inference engine cannot determine a consistent
   641→       BD for a node, often due to conflicting type usage.
   642→Fix: Add explicit type annotations to the variables involved.
   643→```
   644→
   645→**Error: codegen "unsupported operation"**
   646→```
   647→Cause: A VUMA construct is not yet supported by the target backend.
   648→Fix: Try a different backend, or rewrite the construct using supported primitives.
   649→```
   650→
   651→### 3.3 Iterative Compilation Workflow
   652→
   653→When writing VUMA code, follow this iterative approach:
   654→
   655→1. **Start minimal:** Write the smallest program that compiles and runs.
   656→2. **Add one feature at a time:** Add a single function or memory operation, then recompile.
   657→3. **Read diagnostics carefully:** VUMA errors point to specific lines and explain what went wrong.
   658→4. **Fix errors bottom-up:** Resolve parse errors first, then type errors, then verification errors.
   659→5. **Test incrementally:** Use the REPL or `vuma run` to test after each change.
   660→6. **Verify with SHA256d:** If implementing crypto, the SHA256d program with "abc" input must exit with code 79.
   661→
   662→**Typical iteration pattern:**
   663→
   664→```
   665→Step 1: Write fn main() -> i32 { return 0; }
   666→Step 2: vuma build program.vu  → verify it compiles
   667→Step 3: Add one helper function
   668→Step 4: vuma build program.vu  → fix any errors
   669→Step 5: Call the helper from main
   670→Step 6: vuma run program.vu    → check exit code
   671→Step 7: Add memory operations (allocate, write, read, free)
   672→Step 8: vuma verify program.vu → check IVE invariants
   673→Step 9: Repeat from Step 3
   674→```

   678→## Part 4: Integration Guide — Using the VUMA Compiler API from an LLM
   679→
   680→This section describes how an LLM can programmatically interact with the VUMA compiler to write, test, and iterate on VUMA code.
   681→
   682→### 4.1 Compile a Program
   683→
   684→The primary API entry point is `vuma::pipeline::compile()`:
   685→
   686→```rust
   687→use vuma::pipeline::{compile, CompileConfig, CompileTarget, OptLevel, VerificationLevel};
   688→
   689→let source = r#"
   690→    fn main() -> i32 {
   691→        return 42;
   692→    }
   693→"#;
   694→
   695→let config = CompileConfig {
   696→    target: CompileTarget::Linux,
   697→    opt_level: OptLevel::O2,
   698→    verification_level: VerificationLevel::Normal,
   699→    entry_name: "main".to_string(),
   700→    debug_info: false,
   701→    stop_on_first_error: true,
   702→    max_inline_size: 50,
   703→};
   704→
   705→match compile(source, &config) {
   706→    Ok(output) => {
   707→        println!("Compiled {} bytes", output.binary.len());
   708→        println!("SCG has {} nodes", output.scg.node_count());
   709→    }
   710→    Err(errors) => {
   711→        for err in &errors {
   712→            eprintln!("Error: {}", err);
   713→        }
   714→    }
   715→}
   716→```
   717→
   718→**Reading diagnostics:** On error, `compile()` returns `Vec<VumaError>`. Each error identifies its pipeline stage and contains a human-readable message. Parse errors include line/column positions for precise error location.
   719→
   720→**CompileConfig presets:**
   721→- `CompileConfig::default()` — O2 optimization, Normal verification, Linux target.
   722→- `CompileConfig::debug()` — O0 optimization, Quick verification, debug info on.
   723→- `CompileConfig::release()` — O3 optimization, Exhaustive verification.
   724→
   725→### 4.2 Analyze a Program (SCG Summary)
   726→
   727→After parsing, you can inspect the Semantic Computation Graph (SCG) to understand the program's structure:
   728→
   729→```rust
   730→use vuma::pipeline::compile;
   731→use vuma::scg::query::QueryEngine;
   732→
   733→let source = r#"
   734→    fn add(a: i32, b: i32) -> i32 { return a + b; }
   735→    fn main() -> i32 { return add(1, 2); }
   736→"#;
   737→
   738→let config = CompileConfig::default();
   739→if let Ok(output) = compile(source, &config) {
   740→    let scg = &output.scg;
   741→
   742→    // Query SCG properties
   743→    println!("Node count: {}", scg.node_count());
   744→    println!("Edge count: {}", scg.edge_count());
   745→    println!("Functions: {:?}", scg.function_names());
   746→
   747→    // Query specific nodes
   748→    let query = QueryEngine::new(scg);
   749→    let allocs = query.nodes_by_type(NodeType::Allocation);
   750→    let accesses = query.nodes_by_type(NodeType::Access);
   751→    println!("Allocations: {}", allocs.len());
   752→    println!("Accesses: {}", accesses.len());
   753→}
   754→```
   755→
   756→The SCG summary tells you:
   757→- How many functions the program defines
   758→- How many memory allocations and accesses exist
   759→- The call graph structure (which functions call which)
   760→- Data flow between operations
   761→
   762→### 4.3 Compile to Wasm for Sandboxed Execution
   763→
   764→For safe, sandboxed testing without hardware access, compile to WebAssembly:
   765→
   766→```rust
   767→use vuma::pipeline::{compile, CompileConfig};
   768→use vuma_codegen::backend::{create_backend, BackendKind};
   769→
   770→let source = r#"
   771→    fn main() -> i32 {
   772→        buf = allocate(8);
   773→        *buf = 79;
   774→        val: i32 = *buf;
   775→        free(buf);
   776→        return val;
   777→    }
   778→"#;
   779→
   780→// Compile with Wasm32 backend
   781→let config = CompileConfig {
   782→    opt_level: OptLevel::O2,
   783→    ..CompileConfig::default()
   784→};
   785→
   786→// The CLI approach:
   787→// $ vuma emit wasm32 program.vu -o program.wasm
   788→// $ wasmtime program.wasm
   789→// $ echo $?   # should print 79
   790→```
   791→
   792→**CLI command for Wasm compilation:**
   793→```bash
   794→vuma emit wasm32 program.vu -o program.wasm
   795→wasmtime program.wasm
   796→echo $?   # prints the exit code (return value of main)
   797→```
   798→
   799→The Wasm32 backend compiles VUMA to a WebAssembly module where:
   800→- `allocate(n)` maps to `memory.allocate` (grows linear memory)
   801→- `free(addr)` is a no-op in Wasm (memory is never returned)
   802→- `*addr` reads/writes from Wasm linear memory
   803→- The `main` function's return value becomes the Wasm module's exit code
   804→

   923→### 4.7 LLM Coding Workflow
   924→
   925→Here is the recommended workflow for an LLM writing VUMA code:
   926→
   927→1. **Understand the task:** Identify what the program needs to compute and what the expected exit code is.
   928→
   929→2. **Write the skeleton:** Start with `fn main() -> i32 { return 0; }` and verify it compiles.
   930→
   931→3. **Add helper functions first:** Write pure functions (no memory) and test them. Remember:
   932→   - All u32 arithmetic must be masked with `& 4294967295`.
   933→   - No `~` for bitwise NOT on u32 — use `^ 4294967295`.
   934→   - Rotate = `((x >> n) | (x << (32 - n))) & 4294967295`.
   935→
   936→4. **Add memory operations:** Introduce `allocate`/`free` pairs, pointer dereferences, and byte-level access. Ensure every `allocate` has a matching `free`.
   937→
   938→5. **Compile and iterate:** Use `vuma check` for quick verification, `vuma build` for full compilation, and `vuma run` to check the exit code.
   939→
   940→6. **Debug wrong exit codes:** If the exit code is wrong but compilation succeeded, the bug is almost certainly a missing `& 4294967295` mask on u32 arithmetic. Trace through each u32 operation and add the mask.
   941→
   942→7. **Verify invariants:** Run `vuma verify` to check all five LIVE invariants. Fix any violations.
   943→
   944→8. **Cross-compile for testing:** Use `vuma emit wasm32` + `wasmtime` for quick sandboxed testing if native hardware isn't available.

   948→## Appendix: VUMA Idiom Quick Reference
   949→
   950→| Pattern | Idiom |
   951→|---------|-------|
   952→| u32 addition | `(a + b) & 4294967295` |
   953→| u32 NOT | `x ^ 4294967295` |
   954→| u32 right-rotate | `((x >> n) \| (x << (32 - n))) & 4294967295` |
   955→| Byte to u32 (big-endian) | `((b0 << 24) \| (b1 << 16) \| (b2 << 8) \| b3) & 4294967295` |
   956→| u32 to bytes (big-endian) | `*(buf+0)=(val>>24)&255; *(buf+1)=(val>>16)&255; ...` |
   957→| Array access at index | `*(base + idx * stride)` |
   958→| Memory copy | `for i in 0..len { *(dst+i) = *(src+i); }` |
   959→| Conditional NOT | `x ^ 4294967295` (never `~x` for u32) |
   960→| Allocate + init + free | `buf=allocate(n); ...; free(buf);` |
   961→| Return computed exit code | `fn main() -> i32 { return value; }` |
