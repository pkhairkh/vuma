     1‚Üí# VUMA Project Glossary
     2‚Üí
     3‚ÜíThis document defines every term used across the VUMA project. Each entry
     4‚Üíincludes the term, its pronunciation when non-obvious, a detailed definition
     5‚Üíof at least 50 words, and cross-references to related terms. This glossary is
     6‚Üíthe single source of truth for terminology; all other documents should use
     7‚Üíterms as defined here.
     8‚Üí
     9‚Üí---
    10‚Üí
    11‚Üí## Project Core Terms
    12‚Üí
    13‚Üí### SCG
    14‚Üí
    15‚Üí**Pronunciation:** /…õs-siÀê-d íiÀê/ (spell out: S-C-G)
    16‚Üí
    17‚Üí**Semantic Computation Graph** ‚Äî The primary representation of a program in the
    18‚ÜíVUMA framework. The SCG is a directed, acyclic graph where nodes represent
    19‚Üícomputational operations (function application, type construction, effect
    20‚Üíexecution, resource allocation, memory access, deallocation, control flow, type
    21‚Üícasts, and phantom markers for analysis), edges represent data flow, control
    22‚Üíflow, derivation, and annotation relationships between operations, annotations
    23‚Üícarry type information, constraints, invariants, and metadata attached to nodes
    24‚Üíand edges, and regions delineate scopes, phases, security boundaries, and
    25‚Üídeployment targets within the graph. The SCG is not derived from source code ‚Äî
    26‚Üíit *is* the program. Any textual or visual representation is a projection of the
    27‚ÜíSCG. Two programs with the same semantics have the same SCG, regardless of how
    28‚Üíthey were constructed (unique canonical form). The SCG is compositional
    29‚Üí(subgraphs combine through formally defined composition operators),
    30‚Üítransformable (semantics-preserving graph transformations are provably correct
    31‚Üíby construction), and queryable (the IVE can ask arbitrary questions about
    32‚Üíprogram properties through the SCG query engine). The SCG is implemented in the
    33‚Üí`vuma-scg` crate and defines node types (`NodeType` enum with variants
    34‚ÜíComputation, Allocation, Deallocation, Access, Cast, Effect, Control, Phantom),
    35‚Üíedge types (`EdgeKind` with DataFlow, ControlFlow, Derivation, Annotation),
    36‚Üíand region types (`SCGRegion` with `DeploymentTarget`).
    37‚Üí
    38‚Üí**See also:** IVE, Projection, BD, COR, MSG
    39‚Üí
    40‚Üí---
    41‚Üí
    42‚Üí### IVE
    43‚Üí
    44‚Üí**Pronunciation:** /a…™v/ (rhymes with "hive")
    45‚Üí
    46‚Üí**Inference and Verification Engine** ‚Äî The unified reasoning system that
    47‚Üíreplaces the traditional compiler's type checker, borrow checker, and static
    48‚Üíanalyzer. The IVE operates on the SCG and performs four core functions: (1) type
    49‚Üíinference, deriving all types from SCG structure without requiring human
    50‚Üíannotation by propagating Behavioral Descriptors through the graph; (2)
    51‚Üíconstraint inference, deriving temporal constraints, resource flow constraints,
    52‚Üísecurity boundaries, and complexity bounds from program structure; (3)
    53‚Üíverification, constructing formal proofs or counterexamples for properties that
    54‚Üícannot be inferred, checking the five VUMA global invariants (liveness,
    55‚Üíexclusivity, interpretation, origin, cleanup) against the Memory State Graph;
    56‚Üíand (4) gradual verification, maintaining a "verification debt" of properties
    57‚Üíbelieved true but not yet proven, and continuously working to reduce this debt.
    58‚ÜíThe IVE is the reasoning heart of the VUMA framework ‚Äî it is what makes
    59‚Üíverified-unsafe memory access possible, because it proves safety through global
    60‚Üíreasoning rather than local restriction. The IVE is implemented in the
    61‚Üí`vuma-ive` crate with per-invariant modules (`liveness.rs`, `exclusivity.rs`,
    62‚Üí`interpretation.rs`, `origin.rs`, `cleanup.rs`), an inference engine
    63‚Üí(`inference.rs`), a constraint system (`constraint.rs`), debt tracking
    64‚Üí(`debt.rs`), and an invariant aggregator that runs all passes and produces
    65‚Üíunified results (`invariant_aggregator.rs`).
    66‚Üí
    67‚Üí**See also:** SCG, VUMA, MSG, Verification Debt, Verification Confidence
    68‚Üí

    70‚Üí
    71‚Üí### COR
    72‚Üí
    73‚Üí**Pronunciation:** /k…îÀêr/ (rhymes with "more")
    74‚Üí
    75‚Üí**Continuous Optimization Runtime** ‚Äî The runtime system that replaces the
    76‚Üítraditional compile-link-run pipeline. The COR maintains the SCG in an
    77‚Üíalways-compiled state: edits trigger incremental recompilation of affected
    78‚Üísubgraphs, eliminating the concept of "build time." The COR performs
    79‚Üíprofile-guided optimization by collecting runtime profile data (including Pi 5
    80‚Üíhardware PMU counters for cycle count, instruction count, cache misses, and
    81‚Üíbranch misses) and feeding it back to the IVE for optimization decisions. It
    82‚Üísupports speculative optimization (pre-optimizing likely execution paths with
    83‚Üítransparent fallback via a snapshot-and-rollback mechanism) and adaptive
    84‚Üídeployment (moving computation between nodes in a distributed system based on
    85‚Üílatency, cost, and availability). The COR includes a deployment manager with
    86‚Üíhot-swap support (6-phase state machine: Idle, PreparingShadow,
    87‚ÜíAwaitingSafePoint, Swapping, Completed, Failed), delta deployment (block-level
    88‚Üíbinary diffing for minimal transfer), and version logging with rollback
    89‚Üícapability. The COR treats the SCG's region annotations as guidance for
    90‚Üídeployment topology decisions and is implemented in the `vuma-cor` crate.
    91‚Üí
    92‚Üí**See also:** SCG, IVE, Projection
    93‚Üí

    95‚Üí
    96‚Üí### BD
    97‚Üí
    98‚Üí**Pronunciation:** /biÀê-diÀê/ (spell out: B-D)
    99‚Üí
   100‚Üí**Behavioral Descriptor** ‚Äî A triple `(RepD, CapD, RelD)` that describes data
   101‚Üíby what it *does* rather than what it *is called*. The BD replaces traditional
   102‚Üínominal types in the VUMA framework. Where a traditional type bundles memory
   103‚Üílayout, valid operations, and relationships into a single nominal category, the
   104‚ÜíBD decomposes these into three orthogonal dimensions that can each vary
   105‚Üíindependently. The IVE infers and verifies all three components of a BD from
   106‚Üíprogram structure, function signatures, and execution context. The programmer
   107‚Üícan add explicit descriptors as optional refinements, but they are never
   108‚Üímandatory annotations. Two values with the same BD are interchangeable
   109‚Üíregardless of their nominal "type name," and two values with different BDs are
   110‚Üídistinct even if they share a name. The BD is implemented as a struct in the
   111‚Üí`vuma-bd` crate (`descriptor::BD`) with fields for each component, plus methods
   112‚Üífor compatibility checking (`compatible()`), refinement ordering (`refines()`),
   113‚Üíand composition. The crate also provides a context solver for BD inference and
   114‚Üía unification engine for BD compatibility.
   115‚Üí
   116‚Üí**See also:** RepD, CapD, RelD, IVE
   117‚Üí
   118‚Üí---
   119‚Üí
   120‚Üí### RepD
   121‚Üí
   122‚Üí**Pronunciation:** /r…õp-diÀê/ (rhymes with "step-dee")
   123‚Üí
   124‚Üí**Representation Descriptor** ‚Äî The component of a Behavioral Descriptor that
   125‚Üíspecifies the physical layout of data in memory: size, alignment, field offsets,
   126‚Üíand bit-level structure. A RepD is not a type; it is a *memory map*. Multiple
   127‚ÜíRepDs can describe the same memory at different granularities simultaneously ‚Äî
   128‚Üía 128-byte region can be described as `bytes[128]`, as `float32[32]`, or as
   129‚Üí`struct { header: uint32; payload: bytes[124] }`. The RepD does not choose
   130‚Üíamong these; it enumerates all valid interpretations. This multi-view capability
   131‚Üíis what enables zero-copy interoperation and eliminates type-conversion bugs,
   132‚Üíbecause there is no conversion ‚Äî only a shift in perspective verified by the
   133‚ÜíIVE. The RepD is implemented as an enum in `vuma-bd::repd::RepD` with variants
   134‚Üífor byte-level, pointer, struct, array, and primitive representations, along
   135‚Üíwith compatibility and refinement queries in `repd_compat.rs`. In the VUMA
   136‚Üístandard library, RepD is used to annotate the memory layout of primitive types
   137‚Üílike `Ptr<T>`, `RegionPtr<T>`, `Slice<T>`, `VumaResult<T, E>`, `VumaOption<T>`,
   138‚Üíand `Range`.
   139‚Üí
   140‚Üí**See also:** BD, CapD, RelD, VUMA, Interpretation
   141‚Üí
   142‚Üí---
   143‚Üí
   144‚Üí### CapD
   145‚Üí
   146‚Üí**Pronunciation:** /k√¶p-diÀê/ (rhymes with "map-dee")
   147‚Üí
   148‚Üí**Capability Descriptor** ‚Äî The component of a Behavioral Descriptor that
   149‚Üíspecifies what operations are valid on data in a given context. A CapD enumerates
   150‚Üípermissions: read, write, iterate, send over network, persist to disk, execute
   151‚Üías code, derive pointer from, compare for equality, hash, serialize. A CapD is
   152‚Üía set of *permissions*, not a type class. The same data can have different
   153‚Üícapabilities in different contexts ‚Äî a buffer is readable and writable during
   154‚Üíprocessing but only readable during transmission. CapD captures this
   155‚Üícontext-dependence natively, enabling polymorphism through capability matching:
   156‚Üíany value with the required capabilities satisfies the constraint, regardless
   157‚Üíof its nominal type. The CapD forms a lattice (implemented in
   158‚Üí`vuma-bd::capd_lattice`) where the join is union and the meet is intersection,
   159‚Üíwith a sub-capability ordering. The CapD is implemented as a struct in
   160‚Üí`vuma-bd::capd::CapD` with a bitflag-based `Capability` enum and methods for
   161‚Üístrengthening, weakening, union, intersection, and sub-capability checking.
   162‚Üí
   163‚Üí**See also:** BD, RepD, RelD, Capability Calculi
   164‚Üí
   165‚Üí---
   166‚Üí
   167‚Üí### RelD
   168‚Üí
   169‚Üí**Pronunciation:** /r…õl-diÀê/ (rhymes with "tell-dee")
   170‚Üí
   171‚Üí**Relational Descriptor** ‚Äî The component of a Behavioral Descriptor that
   172‚Üíspecifies relationships between data values: temporal co-occurrence, structural
   173‚Üícontainment, dependency ordering, semantic equivalence, and security-level flow.
   174‚ÜíA RelD captures the web of relationships that nominal types express through
   175‚Üíinheritance hierarchies and trait implementations, but with far greater
   176‚Üíexpressiveness. For example, a RelD can express: "this value is semantically
   177‚Üíequivalent to that value but represented differently" (a database row and a
   178‚Üíprotobuf message describing the same entity), or "this value must not outlive
   179‚Üíthat value" (a slice and its backing buffer), or "this value's security level
   180‚Üíis derived from the maximum security level of its sources" (a computed result
   181‚Üícombining public and secret inputs). The RelD is implemented as a struct in
   182‚Üí`vuma-bd::reld::RelD` with a `Relation` enum for relationship kinds
   183‚Üí(Containment, Liveness, Aliasing, DataFlow, RegionBound, Ownership) and methods
   184‚Üífor composition, refinement, and intersection, with a separate refinement
   185‚Üíengine in `reld_refine.rs`.
   186‚Üí
   187‚Üí**See also:** BD, RepD, CapD
   188‚Üí
   189‚Üí---
   190‚Üí
   191‚Üí### VUMA
   192‚Üí
   193‚Üí**Pronunciation:** /vuÀêm…ëÀê/ (rhymes with "llama" with a v)
   194‚Üí
   195‚Üí**Verified-Unsafe Memory Access** ‚Äî The memory model at the core of this
   196‚Üíproject, which permits unrestricted raw memory access (pointers, addresses,
   197‚Üímanual allocation, arbitrary casts) and establishes safety through global
   198‚Üíverification rather than local restriction. In the VUMA model, all data access
   199‚Üíis pointer-based. There are no "safe references" and "unsafe pointers" ‚Äî there
   200‚Üíare only addresses, and the IVE verifies that every access through every address
   201‚Üíis valid at the point of access. The memory model has three primitives: Address
   202‚Üí(a numeric value identifying a location, implemented as `vuma_core::address::Address`),
   203‚ÜíRegion (a contiguous range of addresses with allocation status, ownership
   204‚Üícontext, and access history, implemented as `vuma_core::region::Region`), and
   205‚ÜíAccess (a read or write operation targeting an address, verified by the IVE
   206‚Üíbefore execution, implemented as `vuma_core::access::Access`). VUMA inverts the
   207‚Üísafety model: instead of "restrict by default, permit with explicit unsafe
   208‚Üíblocks," it proposes "permit by default, verify by global reasoning, flag only
   209‚Üíwhat cannot be proven safe." The VUMA core is implemented in the `vuma` crate
   210‚Üíwith modules for regions, addresses, accesses, derivations, synchronization,
   211‚Üíthe Memory State Graph, and SCG-to-MSG conversion.
   212‚Üí
   213‚Üí**See also:** IVE, MSG, Liveness, Exclusivity, Interpretation, Origin, Cleanup
   214‚Üí
   215‚Üí---
   216‚Üí
   217‚Üí### MSG
   218‚Üí
   219‚Üí**Pronunciation:** /…õm-…õs-d íiÀê/ (spell out: M-S-G)
   220‚Üí
   221‚Üí**Memory State Graph** ‚Äî The IVE's formal model of the program's entire memory
   222‚Üíbehavior. The MSG captures every allocation point and the region it creates,
   223‚Üíevery pointer derivation and the path from allocation to dereference, every
   224‚Üídeallocation point and the region it destroys, every concurrent access and the
   225‚Üísynchronization that orders it, and every cast or reinterpretation and the
   226‚Üírepresentation descriptors involved. The IVE proves global memory invariants
   227‚Üí(liveness, exclusivity, interpretation, origin, cleanup) against the MSG. The
   228‚ÜíMSG is the data structure that makes VUMA possible ‚Äî it is the comprehensive
   229‚Üímodel against which all memory safety proofs are constructed. The MSG is
   230‚Üíimplemented as `vuma_core::msg::MSG` with an incremental update mechanism
   231‚Üí(`vuma_core::msg_incremental`) that supports delta computation and application
   232‚Üífor efficient incremental verification when the SCG changes. The MSG is
   233‚Üíconstructed from the SCG via `vuma_core::scg_to_msg::scg_to_msg()`, which
   234‚Üíperforms a topological walk of SCG nodes and creates the corresponding regions,
   235‚Üíderivations, accesses, and synchronization edges.
   236‚Üí
   237‚Üí**See also:** VUMA, IVE, Liveness, Exclusivity, Interpretation, Origin, Cleanup
   238‚Üí
   239‚Üí---
   240‚Üí
   241‚Üí### Projection
   242‚Üí
   243‚Üí**Pronunciation:** /pr…ôÀàd í…õk É…ôn/
   244‚Üí
   245‚ÜíA view of the SCG rendered for human consumption. Projections come in four
   246‚Üíforms: (1) Textual projections ‚Äî traditional code-like views customized to the
   247‚Üíviewer's role (systems programmer, domain expert, security auditor); (2) Visual
   248‚Üíprojections ‚Äî dataflow diagrams, call graphs, state machines, memory layout
   249‚Üíviews that are live and interactive; (3) Conversational projections ‚Äî natural
   250‚Üílanguage dialogue where the human describes intent and the system translates it
   251‚Üíinto SCG modifications; and (4) Diff projections ‚Äî change descriptions in human
   252‚Üíterms ("the authentication flow now requires 2FA for admin accounts") rather
   253‚Üíthan line-level diffs. Projections are bidirectional: changes to the projection
   254‚Üíare propagated back to the SCG and validated by the IVE before being applied.
   255‚ÜíThe projection system is implemented in the `vuma-projection` crate with
   256‚Üímodules for textual, visual, conversational, and diff projections, plus a
   257‚Üíbidirectional synchronization engine.
   258‚Üí
   259‚Üí**See also:** SCG, IVE
   260‚Üí
   261‚Üí---
   262‚Üí
   263‚Üí### Outcome Space
   264‚Üí
   265‚Üí**Pronunciation:** /Ààa ätk åm spe…™s/
   266‚Üí
   267‚ÜíThe complete set of possible outcomes of a computation, including all failure
   268‚Üímodes. In the VUMA framework, the type of a function includes its complete
   269‚Üíoutcome space ‚Äî not just `Result<T, E>` with a generic error type, but a
   270‚Üístructured enumeration of every possible outcome with its conditions and
   271‚Üíhandlers. The IVE verifies that every outcome is handled, either explicitly by
   272‚Üíthe programmer or implicitly by a verified safe default. As the program executes
   273‚Üíand invariants are established, the outcome space shrinks ‚Äî for example, after
   274‚Üíauthentication succeeds, the "unauthorized" outcome is removed from the
   275‚Üípossibility space for subsequent operations. The VUMA standard library provides
   276‚Üí`VumaResult<T, E>` and `VumaOption<T>` types with embedded Behavioral
   277‚ÜíDescriptors that track the outcome space, enabling the IVE to reason about
   278‚Üíwhich outcomes are possible at each program point and verify that all are
   279‚Üíhandled appropriately.
   280‚Üí
   281‚Üí**See also:** IVE, BD, CapD
   282‚Üí
   283‚Üí---
   284‚Üí
   285‚Üí### Verification Debt
   286‚Üí
   287‚Üí**Pronunciation:** /Àåv…õr…™f…™Ààke…™ É…ôn d…õt/
   288‚Üí
   289‚ÜíProperties believed true but not yet formally proven by the IVE. The IVE
   290‚Üímaintains verification debt as a prioritized work list: it continuously works
   291‚Üíto reduce debt, prioritizing properties that affect correctness and security.
   292‚ÜíThe concept mirrors technical debt in software engineering, but applies to
   293‚Üíformal guarantees rather than code quality. Verification debt is acceptable in
   294‚Üípractice because not all properties need to be proven at all times ‚Äî but the
   295‚Üísystem tracks and communicates the current debt level so that stakeholders can
   296‚Üímake informed decisions about risk. In the codebase, verification debt is
   297‚Üítracked by the `vuma_ive::debt::VerificationDebt` struct, which maintains a
   298‚Üílist of `DebtItem` entries with priorities, statuses, and descriptions. Code
   299‚Üíthat has not yet been verified by the IVE is marked with `// IVE-TODO`
   300‚Üíannotations, which are the code-level representation of verification debt.
   301‚Üí
   302‚Üí**See also:** IVE, Verification Confidence, IVE-TODO
   303‚Üí
   304‚Üí---
   305‚Üí
   306‚Üí### Verification Confidence
   307‚Üí
   308‚Üí**Pronunciation:** /Àåv…õr…™f…™Ààke…™ É…ôn Ààk…ínf…™d…ôns/
   309‚Üí
   310‚ÜíA tiered assessment of proof strength assigned to each verified property by the
   311‚ÜíIVE: (1) **Proven safe** ‚Äî the IVE has constructed a formal proof, represented
   312‚Üíby `VerificationStatus::ProvenSafe` with the proof steps and evidence; (2)
   313‚Üí**Probably safe given stated assumptions** ‚Äî the IVE has proven safety
   314‚Üíconditional on assumptions that it cannot verify but are documented and tracked,
   315‚Üírepresented by `VerificationStatus::ProbablySafe` with the list of assumptions;
   316‚Üí(3) **Unverified** ‚Äî the IVE has not yet been able to establish safety,
   317‚Üírepresented by `VerificationStatus::Unverified` with the reason why verification
   318‚Üícould not be completed. Deployment policies can require minimum confidence
   319‚Üílevels for different environments, allowing the system to gracefully degrade
   320‚Üíwhen full verification is not achievable. The verification confidence levels are
   321‚Üídefined in `vuma_ive::result::VerificationStatus` and are used throughout the
   322‚Üíverification pipeline to communicate proof strength to the programmer and the
   323‚Üíruntime.
   324‚Üí
   325‚Üí**See also:** IVE, Verification Debt, VUMA
   326‚Üí

   329‚Üí## Verification Invariant Terms
   330‚Üí
   331‚Üí### Liveness
   332‚Üí
   333‚Üí**Pronunciation:** /Ààla…™vn…ôs/
   334‚Üí
   335‚ÜíThe VUMA invariant requiring that every memory access targets a region that is
   336‚Üíallocated at that program point. The IVE proves liveness by tracking allocation
   337‚Üíand deallocation events through the MSG and verifying that no execution path
   338‚Üíleads to a dereference of a freed or unallocated region. If the IVE cannot
   339‚Üíprove liveness for a specific access, it flags that access and provides the
   340‚Üíexecution path that leads to the potential use-after-free. Liveness is the first
   341‚Üíof the five VUMA global invariants and corresponds to the elimination of
   342‚Üíuse-after-free bugs. The liveness verification pass is implemented in
   343‚Üí`vuma_ive::liveness::LivenessVerifier`, which performs a forward dataflow
   344‚Üíanalysis over the MSG tracking resource events (allocate, access, free) and
   345‚Üíchecking that every access is preceded by an allocation and not followed by a
   346‚Üídeallocation at the same program point. The pass produces a
   347‚Üí`LivenessVerificationResult` with either proof obligations satisfied or
   348‚Üíspecific `LivenessViolation` entries containing the offending address, program
   349‚Üípoint, and execution path.
   350‚Üí
   351‚Üí**See also:** VUMA, MSG, Exclusivity, Origin, Cleanup
   352‚Üí
   353‚Üí---
   354‚Üí
   355‚Üí### Exclusivity
   356‚Üí
   357‚Üí**Pronunciation:** /Àå…õkskluÀêÀàs…™v…™ti/
   358‚Üí
   359‚ÜíThe VUMA invariant requiring that every write access does not overlap with a
   360‚Üísimultaneous read or write access through a different address. The IVE proves
   361‚Üíexclusivity by analyzing concurrent access patterns through the MSG, verifying
   362‚Üíthat either accesses are properly ordered by synchronization (happens-before
   363‚Üíedges, atomic operations, mutex acquisitions) or that they target non-overlapping
   364‚Üíregions. If the IVE cannot prove exclusivity, it flags the potential data race
   365‚Üíand provides the concurrent execution paths. Exclusivity is the second VUMA
   366‚Üíinvariant and corresponds to the elimination of data races. The exclusivity
   367‚Üíverification pass is implemented in `vuma_ive::exclusivity::ExclusivityVerifier`,
   368‚Üíwhich constructs an interference graph of concurrent accesses and checks for
   369‚Üíconflicts. The pass considers synchronization edges (represented as
   370‚Üí`vuma_core::sync::SyncEdge` with orderings like HappensBefore, AcquireRelease,
   371‚ÜíSequentiallyConsistent) that establish ordering between accesses. ARM64 barrier
   372‚Üíinstructions (DMB, DSB, ISB) and atomic instructions (LDXR/STXR, CAS) are
   373‚Üímodeled as synchronization edges by the IVE.
   374‚Üí
   375‚Üí**See also:** VUMA, MSG, Liveness, Interpretation, DMB, DSB, LDXR, STXR
   376‚Üí
   377‚Üí---
   378‚Üí
   379‚Üí### Interpretation
   380‚Üí
   381‚Üí**Pronunciation:** /…™nÀåt…úÀêpr…™Ààte…™ É…ôn/
   382‚Üí
   383‚ÜíThe VUMA invariant requiring that every memory access interprets the target
   384‚Üíbytes according to a valid representation descriptor (RepD). If the IVE cannot
   385‚Üíprove the interpretation is valid (for example, reading uninitialized memory as
   386‚Üía pointer, or interpreting a floating-point bit pattern as an integer when the
   387‚Üísubsequent operations require an integer), it flags the access. Interpretation
   388‚Üíis the third VUMA invariant and ensures that the bytes at any accessed address
   389‚Üíare meaningful under the operation being performed, eliminating type confusion
   390‚Üíand uninitialized-memory bugs. The interpretation verification pass is
   391‚Üíimplemented in `vuma_ive::interpretation::InterpretationVerifier`, which traces
   392‚Üíwrite-read pairs across the MSG and verifies that the RepD at the write point
   393‚Üíis compatible with the RepD expected at the read point. The pass uses the
   394‚Üí`repd_compat` module from `vuma-bd` to check RepD compatibility, and produces
   395‚Üí`InterpretationViolation` entries with the write-read pair and the incompatible
   396‚ÜíRepDs when violations are found.
   397‚Üí
   398‚Üí**See also:** VUMA, MSG, RepD, Liveness
   399‚Üí
   400‚Üí---
   401‚Üí
   402‚Üí### Origin
   403‚Üí
   404‚Üí**Pronunciation:** /Àà…ír…™d í…™n/
   405‚Üí
   406‚ÜíThe VUMA invariant requiring that every address can be traced back to a valid
   407‚Üíallocation point through a well-formed derivation chain. If an address is
   408‚Üícomputed through arithmetic that the IVE cannot trace to an allocation (for
   409‚Üíexample, a hardcoded constant like `0xDEADBEEF`, or a value read from an
   410‚Üíuntrusted source without validation), the IVE flags the computation. Origin is
   411‚Üíthe fourth VUMA invariant and prevents the creation of "phantom pointers" that
   412‚Üípoint to memory the program never allocated, which could corrupt arbitrary
   413‚Üíprocess memory. The origin verification pass uses taint analysis: each source
   414‚Üíof pointer values is assigned a trust level, and the IVE verifies that every
   415‚Üípointer used in an access operation has a trust level that meets the required
   416‚Üíthreshold. The origin verification is implemented in
   417‚Üí`vuma_ive::origin` and supported by the proof infrastructure in
   418‚Üí`vuma_proof::origin_proofs`, which provides `OriginProof`, `DerivationChainProof`,
   419‚Üítaint labels, and proof tactics for establishing origin.
   420‚Üí
   421‚Üí**See also:** VUMA, MSG, Liveness, Cleanup, Derivation Chain
   422‚Üí
   423‚Üí---
   424‚Üí
   425‚Üí### Cleanup
   426‚Üí
   427‚Üí**Pronunciation:** /ÀàkliÀên åp/
   428‚Üí
   429‚ÜíThe VUMA invariant requiring that every allocated region is eventually freed,
   430‚Üíor explicitly marked as intentionally leaked (for example, a long-lived arena
   431‚Üíor a global static). If the IVE detects a potential leak, it flags the
   432‚Üíallocation. Cleanup is the fifth VUMA invariant and corresponds to the
   433‚Üíelimination of memory leaks. The "explicitly marked as intentionally leaked"
   434‚Üíexception accommodates patterns where memory is intentionally never freed
   435‚Üí(program-lifetime arenas, static globals), distinguishing them from accidental
   436‚Üíleaks. The cleanup verification pass is implemented in
   437‚Üí`vuma_ive::cleanup::CleanupVerifier`, which constructs a cleanup graph tracking
   438‚Üíallocation-deallocation pairs and verifies that every allocation has a matching
   439‚Üídeallocation on all execution paths. The pass produces `CleanupViolation`
   440‚Üíentries with the specific `ViolationKind` (Leaked, PartialLeak, DoubleFree) and
   441‚Üíthe offending allocation's program point. The proof infrastructure in
   442‚Üí`vuma_proof::cleanup_proofs` provides formal proof objects for cleanup
   443‚Üíverification results.
   444‚Üí
   445‚Üí**See also:** VUMA, MSG, Liveness, Origin
   446‚Üí
   447‚Üí---
   448‚Üí
   449‚Üí## ARM64 / AArch64 Terms
   450‚Üí
   451‚Üí### AAPCS64
   452‚Üí
   453‚Üí**Pronunciation:** /√¶pks-s…™kstiÀê-f…îÀêr/ (spell out acronym)
   454‚Üí
   455‚Üí**ARM Architecture Procedure Call Standard for 64-bit** ‚Äî The ABI (Application
   456‚ÜíBinary Interface) that governs how functions are called on ARM64 processors.
   457‚ÜíAAPCS64 defines which registers are used for arguments (x0‚Äìx7 for integer/pointer
   458‚Üíarguments, v0‚Äìv7 for floating-point/SIMD arguments), which registers are
   459‚Üícaller-saved (x0‚Äìx18, excluding SP and XZR) vs. callee-saved (x19‚Äìx28), stack
   460‚Üíframe layout with 16-byte stack alignment, and return value conventions (x0 for
   461‚Üíscalar returns, x8 for indirect result locations). VUMA codegen must comply with
   462‚ÜíAAPCS64 to interoperate with C libraries and the operating system on Pi 5. The
   463‚Üíregister classification is implemented in `vuma_codegen::arm64::Register` with
   464‚Üímethods `is_callee_saved()`, `is_caller_saved()`, `arg_register(index)`, and
   465‚Üí`arg_index()`. The codegen's register allocator (`vuma_codegen::regalloc`) must
   466‚Üírespect AAPCS64 calling conventions when allocating registers for function calls.
   467‚Üí
   468‚Üí**See also:** Cortex-A76, BCM2712, Register
   469‚Üí
   470‚Üí---
   471‚Üí
   472‚Üí### DMB
   473‚Üí
   474‚Üí**Pronunciation:** /diÀê-…õm-biÀê/ (spell out)
   475‚Üí
   476‚Üí**Data Memory Barrier** ‚Äî An ARM64 instruction that ensures that all explicit
   477‚Üímemory accesses before the DMB complete before any explicit memory accesses
   478‚Üíafter the DMB are observed. DMB is used to enforce ordering between memory
   479‚Üíoperations in multi-processor or DMA scenarios. DMB accepts a barrier option
   480‚Üíthat specifies the shareability domain: `SY` (full system), `ISH` (inner
   481‚Üíshareable), `OSH` (outer shareable), plus load/store variants (`ISHLD`,
   482‚Üí`ISHST`, `OSHLD`, `OSHST`, `LD`, `ST`). In VUMA, the IVE must understand DMB
   483‚Üísemantics to correctly verify the exclusivity invariant across concurrent
   484‚Üíaccess patterns. DMB does not cause a pipeline flush; it only orders memory
   485‚Üíoperations as observed by other observers. The DMB instruction is represented
   486‚Üías `Instruction::DMB { option: BarrierOption }` in the codegen, with encoding
   487‚Üíimplemented in the `Instruction::encode()` method.
   488‚Üí
   489‚Üí**See also:** DSB, ISB, Exclusivity, BarrierOption
   490‚Üí
   491‚Üí---
   492‚Üí
   493‚Üí### DSB
   494‚Üí
   495‚Üí**Pronunciation:** /diÀê-…õs-biÀê/ (spell out)
   496‚Üí
   497‚Üí**Data Synchronization Barrier** ‚Äî An ARM64 instruction stronger than DMB: it
   498‚Üíensures that all explicit memory accesses before the DSB complete before any
   499‚Üíinstruction after the DSB is executed. DSB is required when the CPU must wait
   500‚Üífor memory operations to be truly complete (for example, after writing to a
   501‚Üídevice register before reading a status register). In VUMA, DSB is used in
   502‚Üídevice-driver code for Pi 5 peripherals and in memory-mapped I/O scenarios where
   503‚Üíthe IVE must model device-visible memory ordering. The DSB instruction is
   504‚Üírepresented as `Instruction::DSB { option: BarrierOption }` in the codegen and
   505‚Üíaccepts the same barrier options as DMB (`SY`, `ISH`, `OSH`, and their load/store
   506‚Üívariants). The DSB is modeled as a full memory fence in the MSG's synchronization
   507‚Üímodel, creating happens-before edges between all preceding and subsequent memory
   508‚Üíoperations.
   509‚Üí
   510‚Üí**See also:** DMB, ISB, BCM2712
   511‚Üí
   512‚Üí---
   513‚Üí
   514‚Üí### ISB
   515‚Üí
   516‚Üí**Pronunciation:** /a…™-…õs-biÀê/ (spell out)
   517‚Üí
   518‚Üí**Instruction Synchronization Barrier** ‚Äî An ARM64 instruction that flushes the
   519‚Üíprocessor pipeline so that all instructions after the ISB are fetched from the
   520‚Üíinstruction cache or memory after the ISB completes. ISB is required after
   521‚Üíwriting to system registers (such as the SCTLR, TLB entries, or branch predictor
   522‚Üísettings) to ensure the change takes effect before subsequent instructions
   523‚Üíexecute. In VUMA, ISB is relevant for context-switching code and low-level
   524‚Üísystem programming on Pi 5, particularly in the exception vector table handlers
   525‚Üídefined in `src/pi5/src/boot.rs`. The ISB instruction is represented as
   526‚Üí`Instruction::ISB` (no operands) in the codegen. The IVE models ISB as a
   527‚Üífull execution barrier that also implies DSB semantics, ensuring all prior
   528‚Üímemory and system register writes are visible before subsequent instructions
   529‚Üíexecute.
   530‚Üí
   531‚Üí**See also:** DMB, DSB
   532‚Üí
   533‚Üí---
   534‚Üí
   535‚Üí### LDXR
   536‚Üí
   537‚Üí**Pronunciation:** /…õl-diÀê-…õks-…ëÀêr/ (spell out)
   538‚Üí
   539‚Üí**Load Exclusive Register** ‚Äî An ARM64 instruction that loads a value from
   540‚Üímemory and tags the memory address for exclusive access monitoring. LDXR is one
   541‚Üíhalf of the ARM64 exclusive access pair (with STXR) that implements lock-free
   542‚Üíatomic operations. The hardware tracks whether the tagged address has been
   543‚Üíwritten to by another agent between the LDXR and the subsequent STXR. In VUMA,
   544‚ÜíLDXR/STXR pairs are the codegen target for atomic compare-and-swap operations
   545‚Üí(the fallback path when ARMv8.1-LSE CAS is not available), and the IVE must
   546‚Üíunderstand their semantics to verify the exclusivity invariant for lock-free
   547‚Üídata structures. The LDXR instruction is represented as `Instruction::LDXR {
   548‚Üírt, rn }` in the codegen, encoding as a 32-bit word with the base register (Rn)
   549‚Üíand destination register (Rt). The exclusive monitor is modeled in the MSG as
   550‚Üía special synchronization edge that provides conditional ordering.
   551‚Üí
   552‚Üí**See also:** STXR, CAS, Exclusivity
   553‚Üí
   554‚Üí---
   555‚Üí
   556‚Üí### STXR
   557‚Üí
   558‚Üí**Pronunciation:** /…õs-tiÀê-…õks-…ëÀêr/ (spell out)
   559‚Üí
   560‚Üí**Store Exclusive Register** ‚Äî An ARM64 instruction that attempts to store a
   561‚Üívalue to memory, succeeding only if the target address has not been written to
   562‚Üíby another agent since the corresponding LDXR. STXR returns a status value in
   563‚Üía register: 0 indicates success, 1 indicates failure (the exclusive monitor was
   564‚Üílost). In VUMA, STXR is the codegen target for the store side of atomic
   565‚Üícompare-and-swap. Failed STXR operations typically cause a retry loop, and the
   566‚ÜíIVE must verify that such loops eventually succeed (a liveness property) and
   567‚Üíthat the retry logic correctly handles the failure case. The STXR instruction
   568‚Üíis represented as `Instruction::STXR { rs, rt, rn }` in the codegen, with the
   569‚Üístatus register (Rs), value register (Rt), and base address register (Rn).
   570‚Üí
   571‚Üí**See also:** LDXR, CAS, Liveness, Exclusivity
   572‚Üí
   573‚Üí---
   574‚Üí
   575‚Üí### CAS
   576‚Üí
   577‚Üí**Pronunciation:** /siÀê-e…™-…õs/ (spell out)
   578‚Üí
   579‚Üí**Compare and Swap** ‚Äî An ARM64 atomic instruction (ARMv8.1-LSE) that
   580‚Üíatomically compares the value at a memory address with an expected value and,
   581‚Üíif they match, replaces it with a new value. CAS is the single-instruction
   582‚Üíequivalent of the LDXR/STXR loop pair, providing better performance on
   583‚ÜíARMv8.1-capable processors like the Cortex-A76 in the Pi 5. CAS variants
   584‚Üíinclude acquire (`CASA`), release (`CASL`), and acquire-release (`CASAL`)
   585‚Üíordering semantics. In VUMA, the codegen prefers CAS over LDXR/STXR when the
   586‚Üítarget supports ARMv8.1-LSE, falling back to LDXR/STXR otherwise. The CAS
   587‚Üíinstruction is represented as `Instruction::CAS { rs, rt, rn }` in the codegen.
   588‚ÜíThe IVE models CAS as an atomic operation that provides both exclusivity and
   589‚Üíhappens-before ordering, making it the strongest synchronization primitive
   590‚Üíavailable in the ARM64 instruction set.
   591‚Üí
   592‚Üí**See also:** LDXR, STXR, Exclusivity
   593‚Üí
   594‚Üí---
   595‚Üí
   596‚Üí### Cortex-A76
   597‚Üí
   598‚Üí**Pronunciation:** /Ààk…îÀêt…õks e…™ s…õv…ônti-s…™ks/
   599‚Üí
   600‚ÜíThe ARM-compatible 64-bit CPU core used in the Raspberry Pi 5's BCM2712 SoC.
   601‚ÜíThe Cortex-A76 is an out-of-order, superscalar processor implementing the
   602‚ÜíARMv8.2-A architecture. It features a 4-wide decode pipeline, branch prediction,
   603‚Üíand a deep reorder buffer that allows significant instruction-level parallelism.
   604‚ÜíFor VUMA, the Cortex-A76's memory model (weakly ordered with multi-copy atomicity)
   605‚Üíis the target for the IVE's exclusivity verification, and its instruction set
   606‚Üí(including ARMv8.1-LSE atomic instructions like CAS, LDADD, and STLXR) is the
   607‚Üítarget for the ARM64 codegen backend. The Cortex-A76 supports the AAPCS64
   608‚Üícalling convention and has 31 general-purpose registers (X0‚ÄìX30), the stack
   609‚Üípointer (SP), and the zero register (XZR).
   610‚Üí
   611‚Üí**See also:** BCM2712, AAPCS64
   612‚Üí

   627‚Üícache hierarchy, and peripheral address map. The bare-metal boot code for the

   630‚Üítransferring control to the user's `main()` function. The linker script
   631‚Üí`src/pi5/link.ld` defines the memory layout with RAM at `0x80000` and an MMIO
   632‚Üíwindow for peripheral access.

   638‚Üí### GPIO
   639‚Üí

   641‚Üí
   642‚Üí**General-Purpose Input/Output** ‚Äî The configurable digital pins on the

   644‚Üítasks. The Pi 5 exposes a 40-pin GPIO header, with pins supporting digital I/O,

   650‚ÜíGPIO driver is implemented in `src/pi5/src/gpio.rs` with functions for pin
   651‚Üímode selection, digital read/write, and pull-up/pull-down configuration. All
   652‚ÜíGPIO register accesses are annotated with `// VUMA-VERIFIED` or `// IVE-TODO`.

   678‚Üí
   679‚Üí## Type Theory Terms
   680‚Üí
   681‚Üí### Nominal Types
   682‚Üí
   683‚Üí**Pronunciation:** /Ààn…ím…™n…ôl ta…™ps/
   684‚Üí
   685‚ÜíTypes defined by their *name* and explicit declaration. Two nominal types are
   686‚Üíconsidered distinct even if they have identical structure ‚Äî for example,
   687‚Üí`struct UserId(u64)` and `struct OrderId(u64)` are different types in Rust
   688‚Üídespite having the same memory layout, because they have different names.
   689‚ÜíNominal typing is the dominant paradigm in mainstream languages (C, Java, Rust,
   690‚ÜíSwift) because it aligns with how humans categorize the world. In VUMA, nominal
   691‚Üítypes are superseded by Behavioral Descriptors, which define data by what it
   692‚Üí*does* (representation, capabilities, relationships) rather than by what it is
   693‚Üí*called*. This means that two values with different names but the same BD are
   694‚Üíinterchangeable, and two values with the same name but different BDs are
   695‚Üídistinct ‚Äî a more flexible and precise system than nominal typing alone.
   696‚Üí
   697‚Üí**See also:** Structural Types, Behavioral Types, BD
   698‚Üí
   699‚Üí---
   700‚Üí
   701‚Üí### Structural Types
   702‚Üí
   703‚Üí**Pronunciation:** /Ààstr åkt É…ôr…ôl ta…™ps/
   704‚Üí
   705‚ÜíTypes defined by their *structure* rather than their name. Two structural types
   706‚Üíare considered equivalent if they have the same shape, regardless of what they
   707‚Üíare called. TypeScript interfaces, Go structural typing for interfaces, and ML
   708‚Üírecord types are examples. Structural typing is more flexible than nominal
   709‚Üítyping for interoperation (any value with the right shape satisfies the
   710‚Üíinterface), but it can accidentally equate types that happen to share structure
   711‚Üíbut have different semantic intent. VUMA's Behavioral Descriptors generalize
   712‚Üístructural types: instead of comparing shape, they compare capabilities and
   713‚Üírelationships, capturing the useful properties of structural typing while
   714‚Üíavoiding accidental equivalence through the RelD component that captures
   715‚Üísemantic relationships.
   716‚Üí
   717‚Üí**See also:** Nominal Types, Behavioral Types, CapD, RelD
   718‚Üí
   719‚Üí---
   720‚Üí
   721‚Üí### Behavioral Types
   722‚Üí
   723‚Üí**Pronunciation:** /b…™Ààhe…™vj…ôr…ôl ta…™ps/
   724‚Üí
   725‚ÜíA type formalism that describes the *behavior* of a component ‚Äî the sequence
   726‚Üíand conditions of interactions it may engage in ‚Äî rather than just its data
   727‚Üílayout. Behavioral types originate in concurrency theory (session types,
   728‚Üíbehavioral contracts) and specify communication protocols, ordering constraints,
   729‚Üíand resource usage patterns. VUMA's Behavioral Descriptors are a form of
   730‚Üíbehavioral type that extends the concept from communication protocols to all
   731‚Üíproperties of data: what operations are valid (CapD), what memory layout is used
   732‚Üí(RepD), and what relationships exist with other data (RelD). This makes VUMA's
   733‚Üítype system a full behavioral type system, not just a data classification system,
   734‚Üíenabling the IVE to reason about behavioral properties like protocol compliance
   735‚Üíand resource usage patterns in addition to memory safety.
   736‚Üí
   737‚Üí**See also:** Nominal Types, Structural Types, BD, CapD
   738‚Üí
   739‚Üí---
   740‚Üí
   741‚Üí### Capability Calculi
   742‚Üí
   743‚Üí**Pronunciation:** /Àåke…™p…ôÀàb…™l…™ti Ààk√¶lkj äla…™/
   744‚Üí
   745‚ÜíA family of formal calculi that model capabilities as first-class, unforgeable
   746‚Üítokens that grant specific access rights. Capability calculi (such as the
   747‚Üíobject-capability model, capability-based security, and the calculus of
   748‚Üícapabilities) provide formal foundations for reasoning about which components
   749‚Üícan access which resources. VUMA's CapD system draws on capability calculi
   750‚Üítheory: a CapD is essentially a capability set, and the IVE verifies that no
   751‚Üíoperation exceeds the capabilities granted to its inputs. The key difference
   752‚Üífrom traditional capability systems is that VUMA capabilities are *inferred* by
   753‚Üíthe IVE rather than explicitly managed by the programmer, reducing the
   754‚Üícognitive burden while maintaining the formal guarantees. The CapD lattice
   755‚Üístructure (join = union, meet = intersection, sub-capability ordering) is
   756‚Üíimplemented in `vuma-bd::capd_lattice` and provides the algebraic foundation
   757‚Üífor capability reasoning.
   758‚Üí
   759‚Üí**See also:** CapD, BD, IVE
   760‚Üí

   763‚Üí## Additional Project Terms
   764‚Üí
   765‚Üí### Derivation Chain
   766‚Üí
   767‚Üí**Pronunciation:** /d…õr…™Ààve…™ É…ôn t Ée…™n/
   768‚Üí
   769‚ÜíThe sequence of operations by which a pointer is computed from an original
   770‚Üíallocation. For example, if `base = allocate(bytes[1024])`, `offset = base +
   771‚Üí64`, and `field_ptr = offset as *Header`, then the derivation chain for
   772‚Üí`field_ptr` is `[base, offset, field_ptr]`. The IVE tracks derivation chains
   773‚Üíthrough the MSG to verify the origin invariant (every address traces back to an
   774‚Üíallocation) and to verify that derived pointers remain within the bounds of
   775‚Üítheir originating region. Derivation chains are modeled in the MSG as
   776‚Üí`vuma_core::derivation::Derivation` entries with kinds (Direct, Offset, Cast,
   777‚ÜíIndex) and provenance ranges (base address + size from the original allocation).
   778‚ÜíThe SCG query engine (`vuma_scg::query::find_derivation_chains`) can compute
   779‚Üíderivation chains from any node to its ancestors, and the proof infrastructure
   780‚Üí(`vuma_proof::origin_proofs::DerivationChainProof`) provides formal proof
   781‚Üíobjects for derivation chain well-formedness.
   782‚Üí
   783‚Üí**See also:** MSG, Origin, VUMA, Derivation
   784‚Üí
   785‚Üí---
   786‚Üí
   787‚Üí### Region (SCG)
   788‚Üí
   789‚Üí**Pronunciation:** /ÀàriÀêd í…ôn/
   790‚Üí
   791‚ÜíA subgraph of the SCG that delineates a scope, phase, security boundary, or
   792‚Üídeployment target. SCG regions are the unit of composition, optimization, and
   793‚Üídeployment. The COR uses region annotations to guide adaptive deployment
   794‚Üídecisions (moving computation between nodes based on latency and cost
   795‚Üíconstraints). In the VUMA memory model, a Region is also a contiguous range of
   796‚Üíaddresses with associated metadata (allocation status, ownership context, access
   797‚Üíhistory). The dual usage is intentional: SCG regions correspond to ownership
   798‚Üícontexts, and memory regions are the physical manifestation of those contexts.
   799‚ÜíSCG regions are implemented as `vuma_scg::region::SCGRegion` with a
   800‚Üí`DeploymentTarget` enum (Heap, Stack, Global, Mmap, Gpu, Device) and a set of
   801‚Üínode IDs. Memory regions are implemented as `vuma_core::region::Region` with
   802‚Üí`RegionId`, base address, size, `RegionStatus` (Allocated, Freed, Stack, Device,
   803‚ÜíStatic), and program points for allocation and deallocation.
   804‚Üí
   805‚Üí**See also:** SCG, COR, MSG
   806‚Üí
   807‚Üí---
   808‚Üí
   809‚Üí### VUMA-VERIFIED
   810‚Üí
   811‚Üí**Pronunciation:** /ÀàvuÀêm…ëÀê Ààv…õr…™fa…™d/
   812‚Üí
   813‚ÜíA code annotation comment (`// VUMA-VERIFIED`) used in the VUMA Rust codebase
   814‚Üíto mark methods and functions whose safety has been formally verified by the
   815‚ÜíIVE. This comment replaces the `unsafe` keyword in semantic intent: where Rust
   816‚Üíwould require an `unsafe` block to acknowledge that the programmer takes
   817‚Üíresponsibility for safety, `// VUMA-VERIFIED` indicates that the IVE has taken
   818‚Üíthat responsibility and proven safety. No VUMA stdlib code should contain bare
   819‚Üí`unsafe` blocks ‚Äî all raw memory access should be either `// VUMA-VERIFIED` or
   820‚Üí`// IVE-TODO`. The annotation must include a brief description of what was
   821‚Üíverified: `// VUMA-VERIFIED: IVE proves this dereference targets a live,
   822‚Üíaligned region`. The CI pipeline scans for `unsafe` blocks without the required
   823‚Üíannotations and rejects any PR that introduces an un-annotated `unsafe` block.
   824‚Üí
   825‚Üí**See also:** IVE-TODO, VUMA, Verification Confidence
   826‚Üí
   827‚Üí---
   828‚Üí
   829‚Üí### IVE-TODO
   830‚Üí
   831‚Üí**Pronunciation:** /a…™v tuÀêÀàduÀê/
   832‚Üí
   833‚ÜíA code annotation comment (`// IVE-TODO`) used in the VUMA Rust codebase to
   834‚Üímark methods and functions where IVE verification has not yet been implemented.
   835‚ÜíThis is the equivalent of Rust's `unsafe` block in terms of acknowledging that
   836‚Üísafety has not been formally proven, but it carries a different connotation: it
   837‚Üíis a *temporary* state, not an accepted risk. The project tracks IVE-TODO items
   838‚Üías verification debt that must be resolved before stable releases. The annotation
   839‚Üímust include a brief description of what needs to be verified:
   840‚Üí`// IVE-TODO: verify origin and liveness for this pointer arithmetic`. The
   841‚ÜíCI pipeline scans for bare `unsafe` blocks and rejects any that lack either
   842‚Üí`// VUMA-VERIFIED` or `// IVE-TODO` annotations. The verification debt tracker
   843‚Üí(`vuma_ive::debt::VerificationDebt`) maintains a list of all IVE-TODO items
   844‚Üíwith their priorities and statuses, enabling the project to track progress
   845‚Üítoward full verification coverage.
   846‚Üí
   847‚Üí**See also:** VUMA-VERIFIED, Verification Debt, IVE
   848‚Üí
   849‚Üí---
   850‚Üí
   851‚Üí### Invariant Aggregator
   852‚Üí
   853‚Üí**Pronunciation:** /…™nÀàv…õ…ôri…ônt Àà√¶…°r…™…°e…™t…ôr/
   854‚Üí
   855‚ÜíThe IVE component that runs all verification passes and produces a unified
   856‚Üíverification summary. The aggregator executes each invariant check (liveness,
   857‚Üíexclusivity, interpretation, origin, cleanup) against the MSG and combines the
   858‚Üíresults into an `AggregatedResult` with an `OverallVerdict` (AllPassed,
   859‚ÜíPartialPass, Failed), per-invariant deltas (`InvariantDelta` showing what
   860‚Üíchanged since the last verification run), and a `DiagnosticsReport` with
   861‚Üíactionable information. The aggregator is implemented in
   862‚Üí`vuma_ive::invariant_aggregator::InvariantAggregator` and is the primary entry
   863‚Üípoint for running full verification. It also supports incremental verification
   864‚Üíby comparing the current verification state against a previous snapshot and
   865‚Üíonly re-running the invariant checks that are affected by the changes.
   866‚Üí
   867‚Üí**See also:** IVE, MSG, Liveness, Exclusivity, Interpretation, Origin, Cleanup
   868‚Üí
   869‚Üí---
   870‚Üí
   871‚Üí### Proof Object
   872‚Üí
   873‚Üí**Pronunciation:** /pruÀêf Àà…íbd í…õkt/
   874‚Üí
   875‚ÜíA structured representation of a formal proof constructed by the IVE during
   876‚Üíverification. A proof object contains a goal (the property being proven), a
   877‚Üísequence of proof steps (each applying an inference rule to derive a new fact
   878‚Üífrom previous facts), and a conclusion (the final derived fact that establishes
   879‚Üíthe goal). Proof objects are checked by the `vuma_proof::checker::ProofChecker`,
   880‚Üíwhich verifies that each step follows from previous steps using the stated
   881‚Üíinference rules and detects circular reasoning. When a proof cannot be
   882‚Üíconstructed, the IVE generates a `CounterExample` from `vuma_proof::counterexample`
   883‚Üíthat describes a minimal execution trace demonstrating the violation, helping
   884‚Üíthe programmer understand and fix the issue.
   885‚Üí
   886‚Üí**See also:** IVE, Verification Confidence, CounterExample
   887‚Üí
