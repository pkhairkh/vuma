  1330‚Üí## Problem 1: sha256d UnknownVariable "v_296"
  1331‚Üí
  1332‚ÜíRoot cause: Wave 1-b correctly made `resolve_expr` return `UnknownVariable`
  1333‚Üíinstead of silently substituting 0. This exposed two real issues in how the
  1334‚ÜíSCG‚ÜíIR builder populates its `names` map:
  1335‚Üí
  1336‚Üí  (a) **Use-before-def ordering.** The SCG‚Üícodegen bridge
  1337‚Üí      (`pipeline::bridge_scg_to_codegen`) walks the control-flow graph and
  1338‚Üí      emits a flat statement list per function. DataFlow-only nodes (those
  1339‚Üí      with no incoming ControlFlow edge) are appended *after* the main body
  1340‚Üí      via the "remaining nodes" cleanup, so a statement that references
  1341‚Üí      `v_<id>` via DataFlow can appear in the flat list *before* the
  1342‚Üí      statement that defines `v_<id>`. `lower_statements` iterated in raw
  1343‚Üí      order, so the use hit `resolve_expr` before the def was registered.
  1344‚Üí
  1345‚Üí  (b) **Cross-function DataFlow references.** The bridge's "remaining nodes"
  1346‚Üí      cleanup creates a separate synthetic `main` function for unconsumed
  1347‚Üí      nodes. If a node N is DataFlow-only and referenced by a statement in
  1348‚Üí      function F, node N's def lands in the synthetic `main` while F's body
  1349‚Üí      references `v_N` ‚Äî a genuinely undefined reference *within F* that no
  1350‚Üí      amount of reordering inside F can fix (the def isn't in F at all).
  1351‚Üí
  1352‚Üí### Fix (scg_to_ir.rs)
  1353‚Üí
  1354‚Üí  1. **`topological_sort_statements`** ‚Äî extended the dependency-edge
  1355‚Üí     computation to handle use-before-def. Previously it only searched for
  1356‚Üí     the *last definition before j*; now, if none is found, it also searches
  1357‚Üí     *forward* for the first definition after j and records a dependency
  1358‚Üí     edge, so Kahn's algorithm lowers the def before the use. Existing
  1359‚Üí     in-order behavior (and the 3 existing topo-sort unit tests) is
  1360‚Üí     unchanged.
  1361‚Üí
  1362‚Üí  2. **`lower_statements`** ‚Äî now lowers statements in topological order
  1363‚Üí     (`Self::topological_sort_statements`) instead of raw list order. This
  1364‚Üí     reorders use-before-def instances so defs are registered before uses,
  1365‚Üí     eliminating the spurious `UnknownVariable` for case (a).
  1366‚Üí
  1367‚Üí  3. **Pre-pass in `lower_function`** ‚Äî before lowering the body, collect
  1368‚Üí     all variable names *used* but not *defined* anywhere in the body
  1369‚Üí     (recursively, via new helper `collect_defs_uses` which reuses
  1370‚Üí     `stmt_def_use`). For each such name that is a **synthetic**
  1371‚Üí     bridge-generated `v_<node_id>` (checked by new helper
  1372‚Üí     `is_synthetic_scg_var`), allocate an uninitialized virtual register
  1373‚Üí     and insert it into `names`. This handles case (b) ‚Äî cross-function
  1374‚Üí     DataFlow refs ‚Äî by giving `resolve_expr` a real (undefined) vreg
  1375‚Üí     instead of erroring. **Crucially**, only synthetic `v_\d+` names are

  1833‚Üí## Problem 1: AArch64 allocate path used generic "arm64_N" opcodes + empty
  1834‚Üíreads/writes (2-3 FP tests fail).
  1835‚Üí
  1836‚Üí### Root cause
  1837‚Üí`AArch64Backend::allocate_registers` in `backend.rs` (lines ~1851-1860
  1838‚Üíbefore this change) iterated the encoded code bytes and built one
  1839‚Üí`AllocatedInstruction` per 4-byte word with:
  1840‚Üí  - `opcode: format!("arm64_{}", i)` ‚Äî a generic placeholder
  1841‚Üí  - `reads: vec![]` / `writes: vec![]` ‚Äî always empty
  1842‚Üí
  1843‚ÜíThis is unlike mips64 (task 2-c) which decodes each chunk via
  1844‚Üí`Instruction::decode` and uses `inst.mnemonic()` as the opcode. As a result:
  1845‚Üí  - `test_fp_conversion_not_noop_all_backends` ‚Äî scans opcodes for
  1846‚Üí    "cvt"/"fcvt"/"scvtf"/"fcvtzs"/etc. substrings. With "arm64_N" opcodes,
  1847‚Üí    none matched. The test fails for the AArch64 arm.
  1848‚Üí  - `test_all_backends_float_to_int_not_just_move` ‚Äî requires
  1849‚Üí    `has_gpr && has_simd_fp` (both a GPR and a SimdFp register in
  1850‚Üí    `reads`/`writes` across the function). With empty reads/writes, both
  1851‚Üí    are false. Fails for AArch64.
  1852‚Üí  - `test_all_backends_fp_conversion_emit_real_instructions` ‚Äî was passing
  1853‚Üí    via the disasm fallback (task 3-a added the FP decode arms to arm64.rs),
  1854‚Üí    but the opcode path didn't match.
  1855‚Üí
  1856‚Üí### Fix
  1857‚Üí1. Added a new helper `arm64_instruction_regs(inst: &crate::arm64::Instruction)
  1858‚Üí   -> (Vec<PhysicalReg>, Vec<PhysicalReg>)` in `backend.rs` (before the
  1859‚Üí   AArch64 Backend impl block). It pattern-matches on the decoded
  1860‚Üí   `Instruction` and returns `(reads, writes)`:
  1861‚Üí   - For FP conversions (SCVTF/UCVTF/FCVTZS/FCVTZU/FCVT) and FP‚ÜîGPR moves
  1862‚Üí     (FMOV_DX/FMOV_XD), it classifies the FP side as `RegClass::SimdFp`
  1863‚Üí     and the GPR side as `RegClass::Gpr`, matching AAPCS64 (integer side
  1864‚Üí     = X0..X30, FP side = V0..V31).
  1865‚Üí     * SCVTF/UCVTF: reads=[Gpr(Rn)], writes=[SimdFp(Rd)]
  1866‚Üí     * FCVTZS/FCVTZU: reads=[SimdFp(Rn)], writes=[Gpr(Rd)]
  1867‚Üí     * FCVT: reads=[SimdFp(Rn)], writes=[SimdFp(Rd)]
  1868‚Üí     * FMOV_DX: reads=[Gpr(Rn)], writes=[SimdFp(Vd)]
  1869‚Üí     * FMOV_XD: reads=[SimdFp(Vn)], writes=[Gpr(Rd)]
  1870‚Üí   - For SIMD integer ops (CNT/ADDV/UMOV): both sides SimdFp (or Gpr for
  1871‚Üí     UMOV's destination).
  1872‚Üí   - For ordinary GPR instructions (ADD/SUB/MUL/LSL/LSR/ASR/AND/ORR/EOR/
  1873‚Üí     RORV/EXTR/LDR/STR/LDP/STP/LDXR/STXR/CAS/LDAR/STLR/BR/BLR/RET/CBZ/
  1874‚Üí     CBNZ/TBZ/TBNZ/CMP/CMN/TST/CSEL/CSET/MSUB/UBFM/SBFM/SXTW/CLZ/RBIT/
  1875‚Üí     MOV/MOVZ/MOVK): reads/writes are populated with `RegClass::Gpr`
  1876‚Üí     entries from the instruction's register operands.
  1877‚Üí   - For everything else (B, BL, BCond, DMB, DSB, ISB, SVC, NOP, RET
  1878‚Üí     without explicit Rn): empty reads/writes (matches previous
  1879‚Üí     behaviour ‚Äî these have no operands the tests care about).
  1880‚Üí
  1881‚Üí2. Replaced the body of the `code.iter().enumerate().map(...)` closure in
  1882‚Üí   `allocate_registers` so that for each 4-byte word it:
  1883‚Üí   a) calls `crate::arm64::Instruction::decode(word)` (the same decoder
  1884‚Üí      used by `disassemble`),
  1885‚Üí   b) on `Some(inst)`: uses `format!("{}", inst)` (the Display impl) as
  1886‚Üí      the opcode and `arm64_instruction_regs(&inst)` for reads/writes,
  1887‚Üí   c) on `None`: falls back to `format!("arm64_{}", i)` with empty
  1888‚Üí      reads/writes (defensive ‚Äî should not happen for any instruction
  1889‚Üí      emitted by the codegen).
  1890‚Üí
  1891‚Üí### Why `format!("{}", inst)` rather than a `mnemonic()` method
  1892‚ÜíThe arm64 `Instruction` enum (in arm64.rs) has no `mnemonic()` method (unlike

  1894‚Üílike `"scvtf d0, x0"`, `"fcvtzs x0, d0"`, `"extr x0, x1, x2, #5"`, etc.
  1895‚ÜíThe FP-conversion tests use `opcode.to_lowercase().contains("cvt")` /

  1900‚Üí## Problem 2: AArch64 ELF doesn't emit external symbols (2 tests fail).
  1901‚Üí
  1902‚Üí### Root cause
  1903‚ÜíThe AArch64 `encode_program` builds its ELF via `build_aarch64_elf_2seg`,
  1904‚Üía custom ELF builder that bypasses the shared `emit.rs::emit_elf` (which
  1905‚Üítask 3-c already fixed to emit SHN_UNDEF entries for externals in both
  1906‚ÜíET_REL and ET_EXEC). `build_aarch64_elf_2seg` only wrote the ELF header +
  1907‚Üí2 LOAD PHDRs + .text + (BSS) .data ‚Äî no section headers, no .symtab, no
  1908‚Üí.strtab. As a result:
  1909‚Üí  - `test_unresolved_reloc_not_offset_zero` ‚Äî calls an external function
  1910‚Üí    "external_callee" and asserts `find_bytes_in_elf(&binary,
  1911‚Üí    b"external_callee")` is true. The string never appeared in the ELF.
  1912‚Üí  - `prop_ffi_extern_symbols_are_undef` ‚Äî declares `extern "C" { fn NAME(...)
  1913‚Üí    }`, calls NAME, and asserts `find_undef_symbols(&binary).contains(&NAME)`.
  1914‚Üí    `find_undef_symbols` parses the ELF's SHT_SYMTAB section; with no
  1915‚Üí    section headers, it returned an empty list.
  1916‚Üí
  1917‚Üí### Fix
  1918‚Üí1. Added a new helper `append_aarch64_elf_sections(elf: &mut Vec<u8>,
  1919‚Üí   text_offset: u64, text_size: u64, extern_symbols: &[String])` in
  1920‚Üí   `backend.rs`. It appends (after the existing LOAD-segment file
  1921‚Üí   content):
  1922‚Üí   - `.shstrtab` content: `"\0.text\0.symtab\0.strtab\0.shstrtab\0"`
  1923‚Üí   - `.strtab` content: `"\0" + name1 + "\0" + name2 + "\0" + ...`
  1924‚Üí   - `.symtab` content: 1 NULL entry (24 zero bytes) + 1 entry per
  1925‚Üí     external symbol, each with `st_info = (STB_GLOBAL<<4)|STT_FUNC`,
  1926‚Üí     `st_shndx = SHN_UNDEF (0)`, `st_value = 0`, `st_size = 0`, and
  1927‚Üí     `st_name` = the offset of the name in `.strtab`.
  1928‚Üí   - Section header table (5 Elf64_Shdr entries, 64 bytes each):
  1929‚Üí     * Index 0: SHT_NULL (all zero)
  1930‚Üí     * Index 1: `.text` (SHT_PROGBITS, SHF_ALLOC|SHF_EXECINSTR=0x6,
  1931‚Üí       sh_addr = 0x400000 + text_offset, sh_offset = text_offset,
  1932‚Üí       sh_size = text_size, sh_addralign = 16)
  1933‚Üí     * Index 2: `.symtab` (SHT_SYMTAB, sh_link = 3 [.strtab index],
  1934‚Üí       sh_info = 1 [one local ‚Äî the NULL entry ‚Äî so the first global
  1935‚Üí       is at index 1; standard ELF convention], sh_addralign = 8,
  1936‚Üí       sh_entsize = 24)
  1937‚Üí     * Index 3: `.strtab` (SHT_STRTAB, sh_addralign = 1)
  1938‚Üí     * Index 4: `.shstrtab` (SHT_STRTAB, sh_addralign = 1)
  1939‚Üí   - Patches the ELF header in place: `e_shoff` (offset 40) =
  1940‚Üí     shdr_off, `e_shnum` (offset 60) = 5, `e_shstrndx` (offset 62) = 4.
  1941‚Üí   The section data is appended AFTER the existing LOAD-segment file
  1942‚Üí   content; it is NOT covered by any LOAD segment (section metadata is
  1943‚Üí   only used by linkers/tools, never loaded into memory).
  1944‚Üí
  1945‚Üí2. Changed `build_aarch64_elf_2seg`'s signature to take a third parameter
  1946‚Üí   `extern_symbols: &[String]`. When non-empty, it calls
  1947‚Üí   `append_aarch64_elf_sections` before returning.
  1948‚Üí
  1949‚Üí3. Modified `AArch64Backend::encode_program` to collect unresolved
  1950‚Üí   external symbols while walking the relocations (in the existing
  1951‚Üí   `else` branch where `func_offsets.get(&reloc.symbol)` returns None ‚Äî
  1952‚Üí   i.e. the BL target is not a defined function). Dedupes via a
  1953‚Üí   `contains` check. Passes the collected `external_symbols` slice to
  1954‚Üí   `build_aarch64_elf_2seg`.
  1955‚Üí
  1956‚Üí### Why this also fixes the cross_backend ELF section validation test

  1966‚Üí### Why `push_shdr` is a nested fn (not a closure)
