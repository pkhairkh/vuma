    24Ôćĺ## 1. Benchmark Categories
    25Ôćĺ
    26ÔćĺThe VUMA benchmark suite is organized into five distinct categories, each designed to isolate and measure specific aspects of language runtime performance, memory behavior, and the unique cost of IVE (Interactive Verification Engine) proof construction. These categories span the full spectrum from single-instruction-level overhead to whole-application throughput, ensuring that no performance characteristic escapes quantitative scrutiny. The categorization also reflects the layered architecture of VUMA itself: micro-benchmarks probe the codegen quality of the VUMA-to-LLVM pipeline, data structure benchmarks exercise the ownership and borrowing verifier, concurrency benchmarks stress the IVE's concurrent-memory-model proofs, real-world benchmarks validate end-to-end developer experience, and verification time benchmarks quantify the "price of safety" that IVE imposes at compile time.
    27Ôćĺ
    28ÔćĺEach category is designed to be independently runnable and reproducible. A benchmark harness script (`vuma-bench`) will orchestrate execution across all four language configurations (VUMA verified, C with `-O2`, Rust safe, Rust unsafe), collect raw cycle counts, compute statistical aggregates, and emit both machine-readable JSON and human-readable Markdown tables. The harness will also enforce environmental controls: CPU frequency scaling disabled, isolated CPU cores via `taskset`, transparent hugepages configured consistently, and no other user-space workloads running during measurement windows.
    29Ôćĺ
    30ÔćĺThe rationale for five categories rather than the traditional two (micro vs macro) is that VUMA introduces a novel axis -- verification time -- that no existing benchmark suite accounts for. Traditional benchmarks assume compilation cost is a one-time developer inconvenience; VUMA's IVE makes verification a measurable, repeatable cost that directly affects iteration speed and CI pipeline duration. By elevating verification time to a first-class benchmark category, we ensure that IVE performance regressions are caught as aggressively as runtime regressions. Additionally, separating concurrency from data structures acknowledges that AArch64's four Cortex-A76 cores introduce NUMA-like caching effects that would confound results if mixed with single-threaded collection benchmarks.

    34Ôćĺ## 2. Micro-Benchmarks
    35Ôćĺ
    36ÔćĺMicro-benchmarks measure the raw cost of individual operations, stripped of algorithmic complexity. Each benchmark performs a single, well-defined operation millions of times so that per-operation cost can be extracted with sub-nanosecond precision using the ARM64 PMU cycle counter. The goal is to isolate codegen differences between VUMA, C (compiled with `gcc -O2`), Rust safe (compiled with `cargo build --release`), and Rust unsafe (same but with `unsafe` blocks bypassing bounds checks). All micro-benchmarks operate on a pre-allocated 8 MiB buffer (2x L2 cache size on Cortex-A76) to ensure predictable cache behavior unless the benchmark explicitly tests cache-miss scenarios.
    37Ôćĺ
    38Ôćĺ### 2.1 alloc_free
    39Ôćĺ
    40Ôćĺ**Purpose:** Measure the full round-trip cost of heap allocation and deallocation. This benchmark reveals allocator overhead, which is a dominant cost in real-world programs that churn short-lived objects.
    41Ôćĺ
    42Ôćĺ**Parameters:**
    43Ôćĺ- Allocation sizes: 16 B, 64 B, 256 B, 1 KiB, 4 KiB, 16 KiB
    44Ôćĺ- Iterations: 1,000,000 per size
    45Ôćĺ- VUMA: uses `vuma::alloc::Box::new()` which IVE verifies for size correctness
    46Ôćĺ- C: uses `malloc()` / `free()`
    47Ôćĺ- Rust safe: uses `Box::new()` / drop
    48Ôćĺ- Rust unsafe: uses `alloc()` / `dealloc()` from `std::alloc`
    49Ôćĺ
    50Ôćĺ**Measurement:** Total cycles for all iterations, divided by iteration count. Report per-size. Expect VUMA to match C within 2% since VUMA's verification is compile-time only and the generated `malloc`/`free` calls should be identical.
    51Ôćĺ
    52Ôćĺ### 2.2 sequential_read
    53Ôćĺ
    54Ôćĺ**Purpose:** Measure sequential memory read throughput, which is dominated by cache line fill bandwidth and hardware prefetcher effectiveness. This benchmark validates that VUMA's generated load instructions are scheduled as efficiently as C's.
    55Ôćĺ
    56Ôćĺ**Parameters:**
    57Ôćĺ- Buffer size: 8 MiB (2,097,152 x 4-byte integers)
    58Ôćĺ- Access pattern: stride-1 sequential read
    59Ôćĺ- Iterations: 100 full passes over buffer
    60Ôćĺ- Prefetch: none (test hardware prefetcher only)
    61Ôćĺ- Data type: `i32`
    62Ôćĺ
    63Ôćĺ**Measurement:** Cycles per element read. Expect all four configurations within 1% of each other, since this is purely a memory subsystem benchmark and codegen differences should vanish for trivial load-add loops.
    64Ôćĺ
    65Ôćĺ### 2.3 sequential_write
    66Ôćĺ
    67Ôćĺ**Purpose:** Measure sequential write throughput, which exercises store buffers and write-combining. VUMA's store instructions should be identical to C after IVE verification completes at compile time.
    68Ôćĺ
    69Ôćĺ**Parameters:**
    70Ôćĺ- Buffer size: 8 MiB
    71Ôćĺ- Access pattern: stride-1 sequential write
    72Ôćĺ- Iterations: 100 full passes
    73Ôćĺ- Data type: `i32`
    74Ôćĺ
    75Ôćĺ**Measurement:** Cycles per element written. Expect near-identical results across all four configurations.
    76Ôćĺ
    77Ôćĺ### 2.4 random_read
    78Ôćĺ
    79Ôćĺ**Purpose:** Measure the cost of random memory accesses, which stress the TLB and L1/L2 cache miss pathways. This benchmark is critical for VUMA because it reveals whether IVE's bounds verification adds any indirect cost (e.g., by preventing aliasing optimizations that C enjoys).
    80Ôćĺ
    81Ôćĺ**Parameters:**
    82Ôćĺ- Buffer size: 64 MiB (16,777,216 x 4-byte integers), far larger than L2 cache
    83Ôćĺ- Access pattern: pre-generated random permutation of indices, fixed seed for reproducibility
    84Ôćĺ- Iterations: 1,000,000 random reads per measurement
    85Ôćĺ- Index generation: Fisher-Yates shuffle, seed 42
    86Ôćĺ
    87Ôćĺ**Measurement:** Cycles per random read. Expect VUMA and C within 3%, Rust safe slightly slower due to bounds checks on index operations, Rust unsafe matching C.
    88Ôćĺ
    89Ôćĺ### 2.5 random_write
    90Ôćĺ
    91Ôćĺ**Purpose:** Measure random write cost, which is higher than random read due to cache line allocation on write-miss. Same parameters as random_read but with store operations.
    92Ôćĺ
    93Ôćĺ**Parameters:**
    94Ôćĺ- Buffer size: 64 MiB
    95Ôćĺ- Access pattern: same random permutation as random_read
    96Ôćĺ- Iterations: 1,000,000 random writes per measurement
    97Ôćĺ- Write value: index XOR'd with constant
    98Ôćĺ
    99Ôćĺ**Measurement:** Cycles per random write. Same expectations as random_read.
   100Ôćĺ
   101Ôćĺ### 2.6 pointer_chase
   102Ôćĺ
   103Ôćĺ**Purpose:** Measure the cost of following a chain of pointers, which is the worst case for branch prediction and cache locality. This is the canonical "pointer dereference cost" benchmark and is highly relevant to linked data structures.
   104Ôćĺ
   105Ôćĺ**Parameters:**
   106Ôćĺ- Chain length: 1,048,576 nodes (1M)
   107Ôćĺ- Node size: 64 bytes (one cache line on Cortex-A76)
   108Ôćĺ- Layout: pre-shuffled to randomize cache behavior
   109Ôćĺ- Each node contains a single `next` pointer plus padding
   110Ôćĺ- Iterations: 10 full chases
   111Ôćĺ
   112Ôćĺ**Measurement:** Cycles per node traversal. Expect VUMA and C within 2%. Rust safe and unsafe both within 2% of C, since pointer chasing involves no bounds checks.
   113Ôćĺ
   114Ôćĺ### 2.7 arithmetic
   115Ôćĺ
   116Ôćĺ**Purpose:** Measure raw compute throughput for integer and floating-point operations, ensuring VUMA's arithmetic codegen matches C and Rust. Tests both register-pressure-light and register-pressure-heavy scenarios.
   117Ôćĺ
   118Ôćĺ**Parameters:**
   119Ôćĺ- Integer operations: add, sub, mul, div (i64), bitwise AND/OR/XOR/shift
   120Ôćĺ- Float operations: add, sub, mul, div (f64), sqrt, sin (libm)
   121Ôćĺ- Loop count: 100,000,000 iterations per operation
   122Ôćĺ- Accumulator pattern: prevent dead-code elimination
   123Ôćĺ
   124Ôćĺ**Measurement:** Cycles per operation. Expect near-identical results for integer ops; float ops may differ slightly due to libm implementation choices.
   125Ôćĺ
   126Ôćĺ### 2.8 cast_operations
   127Ôćĺ
   128Ôćĺ**Purpose:** Measure the cost of type reinterpretation casts, which is the critical differentiator for VUMA. In C, casts are free (zero runtime cost) but unsafe. In Rust, safe casts (`as`, `try_into`) may involve runtime checks. In VUMA, IVE verifies cast correctness at compile time, potentially adding verification cost but zero runtime cost.
   129Ôćĺ
   130Ôćĺ**Parameters:**
   131Ôćĺ- Cast types: i32-to-f32 bit reinterpret, i64-to-i32 truncation, f64-to-i64 conversion, pointer-to-integer, struct-to-bytes
   132Ôćĺ- Iterations: 10,000,000 per cast type
   133Ôćĺ- VUMA: uses `vuma::cast::verified_transmute()` which IVE proves safe
   134Ôćĺ- C: uses `*(target_type*)&value` (undefined behavior, but common)
   135Ôćĺ- Rust safe: uses `try_from()` / `try_into()` with error handling
   136Ôćĺ- Rust unsafe: uses `std::mem::transmute()`
   137Ôćĺ
   138Ôćĺ**Measurement:** Cycles per cast. Expect VUMA and C to match (both zero runtime overhead), Rust safe to be 2-5x slower on checked casts, Rust unsafe to match C.
   139Ôćĺ

   142Ôćĺ## 3. Data Structure Benchmarks
   143Ôćĺ
   144ÔćĺData structure benchmarks exercise VUMA's core value proposition: safe, verified data structures with zero runtime overhead compared to C. Each benchmark measures the three fundamental operations (insert, traverse, remove) on a collection, and additionally reports peak memory usage. The doubly-linked list benchmark is the flagship -- it is the canonical example where Rust requires `unsafe` for interior mutability, C is trivially unsafe, and VUMA provides verified safety with C-level performance.
   145Ôćĺ
   146Ôćĺ### 3.1 doubly_linked_list
   147Ôćĺ
   148Ôćĺ**Purpose:** The headline benchmark. A doubly-linked list is trivial in C (but unsafe), painful in safe Rust (requires `Rc<RefCell<Node>>` with reference counting overhead), ergonomic in unsafe Rust (raw pointers), and the showcase data structure for VUMA's ownership model. VUMA's IVE verifies that all pointer manipulations preserve list invariants (no dangling pointers, no cycles, proper sentinel handling) at compile time.
   149Ôćĺ
   150Ôćĺ**Parameters:**
   151Ôćĺ- List size: 10,000 nodes
   152Ôćĺ- Node payload: 64 bytes (cache-line sized)
   153Ôćĺ- Operations:
   154Ôćĺ  - `insert_head`: 10,000 insertions at head
   155Ôćĺ  - `insert_tail`: 10,000 insertions at tail
   156Ôćĺ  - `insert_sorted`: 10,000 insertions in sorted order
   157Ôćĺ  - `traverse_forward`: full forward traversal, sum payload
   158Ôćĺ  - `traverse_backward`: full backward traversal, sum payload
   159Ôćĺ  - `remove_head`: 10,000 removals from head
   160Ôćĺ  - `remove_tail`: 10,000 removals from tail
   161Ôćĺ  - `remove_middle`: 10,000 removals of random middle nodes
   162Ôćĺ- VUMA: `vuma::collections::DList<T>` with IVE-verified pointers
   163Ôćĺ- C: hand-written `struct Node { prev; next; payload; }` with `malloc`/`free`
   164Ôćĺ- Rust safe: `std::collections::LinkedList<T>` (which internally uses unsafe)
   165Ôćĺ- Rust unsafe: hand-rolled raw-pointer implementation
   166Ôćĺ
   167Ôćĺ**Measurement:** Cycles per operation. Memory: total bytes allocated including overhead. Expect VUMA to match C and unsafe Rust; safe Rust's `LinkedList` should also match since the standard library uses unsafe internally, but a naive safe Rust implementation with `Rc<RefCell<>>` would be 3-5x slower and use 2-3x more memory.
   168Ôćĺ
   169Ôćĺ### 3.2 hash_map
   170Ôćĺ
   171Ôćĺ**Purpose:** Measure hash table performance, which is dominated by hash computation quality and memory access patterns. VUMA's verified hash map should perform identically to C's because IVE only verifies memory safety, not algorithmic choices.
   172Ôćĺ
   173Ôćĺ**Parameters:**
   174Ôćĺ- Table size: 100,000 entries
   175Ôćĺ- Key type: `i64` (hash via SipHash-1-3, same as Rust's default)
   176Ôćĺ- Value type: `i64`
   177Ôćĺ- Load factors tested: 0.5, 0.75, 0.9
   178Ôćĺ- Operations:
   179Ôćĺ  - `insert`: 100,000 unique insertions
   180Ôćĺ  - `lookup_hit`: 100,000 lookups for existing keys
   181Ôćĺ  - `lookup_miss`: 100,000 lookups for absent keys
   182Ôćĺ  - `remove`: 50,000 removals
   183Ôćĺ- VUMA: `vuma::collections::HashMap<K,V>` with IVE-verified bucket access
   184Ôćĺ- C: hand-written open-addressing hash map with linear probing
   185Ôćĺ- Rust safe: `std::collections::HashMap<K,V>`
   186Ôćĺ- Rust unsafe: custom hash map with raw pointer bucket access
   187Ôćĺ
   188Ôćĺ**Measurement:** Cycles per operation at each load factor. Expect VUMA and C within 3%, Rust safe may be slightly slower due to SipHash overhead vs C's simpler hash function.
   189Ôćĺ
   190Ôćĺ### 3.3 vec
   191Ôćĺ
   192Ôćĺ**Purpose:** Measure dynamic array performance, focusing on the cost of bounds checking. This benchmark directly tests VUMA's claim that verified access eliminates bounds checks at runtime while maintaining safety. Rust's safe `vec[i]` performs a bounds check on every index operation; VUMA's IVE proves that indices are in-bounds and elides the check.
   193Ôćĺ
   194Ôćĺ**Parameters:**
   195Ôćĺ- Initial capacity: 0 (test growth strategy)
   196Ôćĺ- Maximum size: 1,000,000 elements
   197Ôćĺ- Element type: `i64`
   198Ôćĺ- Operations:
   199Ôćĺ  - `push`: 1,000,000 push operations (measures amortized growth cost)
   200Ôćĺ  - `pop`: 1,000,000 pop operations
   201Ôćĺ  - `index_sequential`: sequential access of all 1,000,000 elements
   202Ôćĺ  - `index_random`: random access of 1,000,000 elements (same permutation as micro-benchmark)
   203Ôćĺ  - `iterate`: sum all elements via iterator
   204Ôćĺ  - `resize`: 100 resize operations from 0 to 1,000,000 and back
   205Ôćĺ- VUMA: `vuma::collections::Vec<T>` with IVE-verified index elision
   206Ôćĺ- C: hand-written `realloc`-based dynamic array with no bounds checks
   207Ôćĺ- Rust safe: `std::vec::Vec<T>` with bounds checks on `v[i]`
   208Ôćĺ- Rust unsafe: `std::vec::Vec<T>` with `v.get_unchecked(i)`
   209Ôćĺ
   210Ôćĺ**Measurement:** Cycles per operation. The critical comparison is `index_sequential` and `index_random`: VUMA should match C and unsafe Rust (no bounds checks), while safe Rust should show 10-30% overhead on random index access due to the branch-prediction cost of bounds checks.
   211Ôćĺ
   212Ôćĺ### 3.4 arena_allocator
   213Ôćĺ
   214Ôćĺ**Purpose:** Measure arena (region-based) allocation performance, which is important for systems programming patterns where many objects share a lifetime. VUMA's IVE can verify that no arena-allocated reference escapes the arena's lifetime scope.
   215Ôćĺ
   216Ôćĺ**Parameters:**
   217Ôćĺ- Arena size: 64 MiB
   218Ôćĺ- Object sizes: 32 B, 64 B, 128 B, 256 B
   219Ôćĺ- Objects allocated: 100,000 per size
   220Ôćĺ- Operations:
   221Ôćĺ  - `alloc_many`: allocate 100,000 objects of each size
   222Ôćĺ  - `access_all`: touch every allocated object (read + write)
   223Ôćĺ  - `reset`: deallocate all at once (arena reset)
   224Ôćĺ- VUMA: `vuma::alloc::Arena` with IVE-verified lifetime scoping
   225Ôćĺ- C: hand-written bump allocator
   226Ôćĺ- Rust safe: `typed_arena` crate
   227Ôćĺ- Rust unsafe: hand-written bump allocator with raw pointers
   228Ôćĺ
   229Ôćĺ**Measurement:** Cycles per allocation, cycles per access, cycles per reset. Expect all configurations within 5% since arena allocation is inherently simple (bump a pointer).
   230Ôćĺ
   231Ôćĺ### 3.5 ring_buffer
   232Ôćĺ
   233Ôćĺ**Purpose:** Measure single-producer single-consumer (SPSC) ring buffer performance, which is the foundational primitive for concurrent communication. VUMA's IVE verifies that the producer and consumer never access the same slot simultaneously.
   234Ôćĺ
   235Ôćĺ**Parameters:**
   236Ôćĺ- Buffer capacity: 1024 elements (power of 2 for modulo optimization)
   237Ôćĺ- Element type: `i64`
   238Ôćĺ- Items produced/consumed: 10,000,000
   239Ôćĺ- Operations:
   240Ôćĺ  - `produce`: enqueue items from producer thread
   241Ôćĺ  - `consume`: dequeue items from consumer thread
   242Ôćĺ  - `round_trip`: time for 10M items from producer to consumer
   243Ôćĺ- VUMA: `vuma::sync::RingBuffer<T>` with IVE-verified SPSC invariant
   244Ôćĺ- C: hand-written lock-free SPSC ring with atomic indices
   245Ôćĺ- Rust safe: `crossbeam::channel::bounded(1024)`
   246Ôćĺ- Rust unsafe: hand-written lock-free SPSC ring with atomic raw pointers
   247Ôćĺ
   248Ôćĺ**Measurement:** Nanoseconds per item end-to-end. Expect VUMA and C within 5% (both use identical atomic-based ring logic); crossbeam may be slightly slower due to internal bookkeeping.

   252Ôćĺ## 4. Concurrency Benchmarks
   253Ôćĺ
   254ÔćĺConcurrency benchmarks exercise VUMA's IVE concurrent memory model verification on the AArch64's four Cortex-A76 cores. These benchmarks are designed to stress both the runtime performance of synchronization primitives and the IVE's ability to prove data-race freedom at compile time. Each benchmark pins threads to specific cores using `pthread_setaffinity_np` (or the VUMA equivalent) to eliminate scheduler noise. All benchmarks are run with the AArch64's CPU governor set to `performance` mode to eliminate frequency scaling variance.
   255Ôćĺ
   256Ôćĺ### 4.1 parallel_sum
   257Ôćĺ
   258Ôćĺ**Purpose:** Measure the simplest form of parallelism: partition an array into N chunks, sum each chunk on a separate core, then combine results. This benchmark tests thread creation overhead, cache coherency cost for read-only shared data, and the cost of the final reduction.
   259Ôćĺ
   260Ôćĺ**Parameters:**
   261Ôćĺ- Array size: 100,000,000 `i64` elements (800 MiB)
   262Ôćĺ- Thread counts: 1, 2, 4
   263Ôćĺ- Partition strategy: equal-sized contiguous chunks
   264Ôćĺ- Reduction: sequential sum of partial results
   265Ôćĺ- Thread pinning: core 0, core 1, core 2, core 3
   266Ôćĺ- VUMA: `vuma::thread::spawn` with IVE-verified thread safety
   267Ôćĺ- C: `pthread_create` / `pthread_join`
   268Ôćĺ- Rust safe: `std::thread::spawn` with `Arc<[i64]>`
   269Ôćĺ- Rust unsafe: `std::thread::spawn` with raw shared pointers
   270Ôćĺ
   271Ôćĺ**Measurement:** Total wall-clock time and speedup ratio vs single-threaded. Expect near-linear scaling up to 4 cores for all configurations. VUMA should match C within 3%.
   272Ôćĺ
   273Ôćĺ### 4.2 producer_consumer
   274Ôćĺ
   275Ôćĺ**Purpose:** Measure throughput of a single-producer single-consumer pipeline using a lock-free queue. This benchmark isolates the cost of inter-thread communication, which is dominated by cache-line bouncing between cores.
   276Ôćĺ
   277Ôćĺ**Parameters:**
   278Ôćĺ- Queue: lock-free MPMC queue (used in SPSC mode)
   279Ôćĺ- Message size: 8 bytes (`i64`)
   280Ôćĺ- Message count: 10,000,000
   281Ôćĺ- Producer pinned to core 0, consumer pinned to core 1
   282Ôćĺ- Backpressure: unbounded (producer never blocks)
   283Ôćĺ- VUMA: `vuma::sync::MpmcQueue<T>` with IVE-verified linearizability
   284Ôćĺ- C: hand-written lock-free queue using C11 atomics
   285Ôćĺ- Rust safe: `crossbeam::channel::unbounded()`
   286Ôćĺ- Rust unsafe: hand-written lock-free queue with raw atomics
   287Ôćĺ
   288Ôćĺ**Measurement:** Messages per second and median latency per message. Expect VUMA and C within 5%; crossbeam within 10% due to internal Select machinery.
   289Ôćĺ
   290Ôćĺ### 4.3 mutex_contention
   291Ôćĺ
   292Ôćĺ**Purpose:** Measure the cost of mutex contention when multiple threads repeatedly acquire and release the same lock. This is the worst-case scenario for lock-based concurrency and reveals the quality of each language's mutex implementation and the overhead of any language-specific bookkeeping.
   293Ôćĺ
   294Ôćĺ**Parameters:**
   295Ôćĺ- Thread counts: 1, 2, 4
   296Ôćĺ- Iterations per thread: 1,000,000
   297Ôćĺ- Critical section: increment a shared counter
   298Ôćĺ- Mutex type: normal (non-recursive, non-adaptive)
   299Ôćĺ- Threads pinned to separate cores
   300Ôćĺ- VUMA: `vuma::sync::Mutex<T>` with IVE-verified lock discipline
   301Ôćĺ- C: `pthread_mutex_t`
   302Ôćĺ- Rust safe: `std::sync::Mutex<T>`
   303Ôćĺ- Rust unsafe: `pthread_mutex_t` via FFI
   304Ôćĺ
   305Ôćĺ**Measurement:** Cycles per lock-unlock pair. Throughput (operations/second) per thread count. Expect VUMA and C within 3%; Rust's `std::sync::Mutex` may show 5-15% overhead due to poison checking.
   306Ôćĺ
   307Ôćĺ### 4.4 rw_lock_read_heavy
   308Ôćĺ
   309Ôćĺ**Purpose:** Measure read-write lock performance under a read-heavy workload (90% reads, 10% writes). This benchmark is representative of configuration data or cached metadata access patterns in embedded systems.
   310Ôćĺ
   311Ôćĺ**Parameters:**
   312Ôćĺ- Thread counts: 1, 2, 4
   313Ôćĺ- Read/write ratio: 90/10
   314Ôćĺ- Total operations per thread: 1,000,000
   315Ôćĺ- Shared data: array of 1000 `i64` values
   316Ôćĺ- Read operation: sum 10 random elements
   317Ôćĺ- Write operation: update 1 random element
   318Ôćĺ- VUMA: `vuma::sync::RwLock<T>` with IVE-verified reader/writer discipline
   319Ôćĺ- C: `pthread_rwlock_t`
   320Ôćĺ- Rust safe: `std::sync::RwLock<T>`
   321Ôćĺ- Rust unsafe: `pthread_rwlock_t` via FFI
   322Ôćĺ
   323Ôćĺ**Measurement:** Operations per second per thread count. Expect VUMA and C within 5%. Rust's `std::sync::RwLock` on Linux uses `pthread_rwlock_t` internally, so results should be similar.
   324Ôćĺ
   325Ôćĺ### 4.5 map_reduce
   326Ôćĺ
   327Ôćĺ**Purpose:** Measure a complete parallel map-reduce pipeline, which combines computation parallelism (map phase) with synchronization (reduce phase). This benchmark exercises thread pool management, work stealing, and barrier synchronization.
   328Ôćĺ
   329Ôćĺ**Parameters:**
   330Ôćĺ- Input size: 10,000,000 elements
   331Ôćĺ- Map function: `x -> x * x + x` (compute-bound)
   332Ôćĺ- Reduce function: sum
   333Ôćĺ- Thread counts: 1, 2, 4
   334Ôćĺ- Work partitioning: dynamic (work-stealing) and static (chunked)
   335Ôćĺ- VUMA: `vuma::parallel::map_reduce` with IVE-verified data partitioning
   336Ôćĺ- C: hand-written thread pool with work queue
   337Ôćĺ- Rust safe: `rayon::par_iter().map().reduce()`
   338Ôćĺ- Rust unsafe: custom thread pool with raw pointers
   339Ôćĺ
   340Ôćĺ**Measurement:** Wall-clock time and speedup ratio. Rayon's work-stealing scheduler may outperform static partitioning on heterogeneous workloads, but for uniform compute, all should scale similarly.
   341Ôćĺ

   344Ôćĺ## 5. Real-World Benchmarks
   345Ôćĺ
   346ÔćĺReal-world benchmarks measure end-to-end application performance, including I/O, parsing, system calls, and complex memory access patterns. These benchmarks are the most meaningful to developers evaluating whether VUMA is practical for production use. Each benchmark implements a realistic, self-contained application in all four language configurations, using idiomatic patterns for each language (no "writing C in Rust" or "writing Rust in VUMA").
   347Ôćĺ
   348Ôćĺ### 5.1 json_parser
   349Ôćĺ
   350Ôćĺ**Purpose:** Parse a JSON document into an in-memory tree structure, then serialize it back to a string. This benchmark exercises memory allocation (many small nodes), string processing, and recursive data structures -- all areas where VUMA's verified memory safety should shine.
   351Ôćĺ
   352Ôćĺ**Parameters:**
   353Ôćĺ- Input: synthetic JSON document, 10 MiB, with nested objects (depth 10), arrays (length 100), strings (average 32 bytes), numbers, and booleans
   354Ôćĺ- Parse: full DOM-style parse into tree of `JsonValue` nodes
   355Ôćĺ- Serialize: walk tree and emit JSON string
   356Ôćĺ- Validate: round-trip check (re-parse serialized output and compare)
   357Ôćĺ- Iterations: 100
   358Ôćĺ- VUMA: `vuma::json::parse()` with IVE-verified string handling (no buffer overread)
   359Ôćĺ- C: `cJSON` library (popular, lightweight, uses `malloc`/`free`)
   360Ôćĺ- Rust safe: `serde_json::from_str()` + `serde_json::to_string()`
   361Ôćĺ- Rust unsafe: custom parser with `unsafe` string slicing
   362Ôćĺ
   363Ôćĺ**Measurement:** Parse time (cycles), serialize time (cycles), peak memory (bytes). Expect VUMA within 5% of cJSON; serde_json may be faster due to zero-copy deserialization optimizations.
   364Ôćĺ
   365Ôćĺ### 5.2 http_server
   366Ôćĺ
   367Ôćĺ**Purpose:** Implement a minimal HTTP/1.1 server that handles GET requests for static content. This benchmark measures network I/O performance, request parsing, and response generation under load. It is the most complex benchmark in the suite and exercises nearly every subsystem.
   368Ôćĺ
   369Ôćĺ**Parameters:**
   370Ôćĺ- Server: listen on port 8080, serve files from memory (no disk I/O)
   371Ôćĺ- File set: 100 files, sizes from 100 B to 100 KiB, pre-loaded into memory
   372Ôćĺ- Client: `wrk` HTTP benchmarking tool, 2 threads, 100 connections, 30-second duration
   373Ôćĺ- Metrics: requests/second, latency P50/P95/P99, error rate
   374Ôćĺ- VUMA: `vuma::net::TcpListener` with IVE-verified buffer handling
   375Ôćĺ- C: `libevent`-based event loop
   376Ôćĺ- Rust safe: `tokio` async runtime with `hyper`
   377Ôćĺ- Rust unsafe: `mio`-based manual event loop with raw buffer handling
   378Ôćĺ
   379Ôćĺ**Measurement:** Requests per second, latency distribution. Expect all configurations within 10% since this is I/O-bound; differences will appear in CPU utilization during peak load.
   380Ôćĺ
   381Ôćĺ### 5.3 gpio_control
   382Ôćĺ
   383Ôćĺ**Purpose:** Toggle AArch64 GPIO pins at maximum frequency using memory-mapped I/O. This benchmark is AArch64-specific and measures the raw latency of MMIO operations, which is critical for real-time embedded applications. VUMA's IVE can verify that GPIO register accesses follow the correct sequence (set function select before writing data).
   384Ôćĺ
   385Ôćĺ**Parameters:**
   386Ôćĺ- GPIO: GPIO pin 18 (GPIO_GEN1, physical pin 12)
   387Ôćĺ- Access method: `/dev/gpiomem` memory-mapped register access
   388Ôćĺ- Toggle count: 10,000,000 on/off cycles
   389Ôćĺ- Measurement: toggle frequency (Hz) and jitter (standard deviation of half-period)
   390Ôćĺ- VUMA: `vuma::hal::gpio::Pin` with IVE-verified register access protocol
   391Ôćĺ- C: direct `volatile` pointer dereference on `/dev/gpiomem` mapping
   392Ôćĺ- Rust safe: `rppal` crate (uses underlying MMIO)
   393Ôćĺ- Rust unsafe: raw `volatile` pointer dereference via `write_volatile`
   394Ôćĺ
   395Ôćĺ**Measurement:** Toggle frequency (kHz) and cycle-to-cycle jitter (ns). Expect VUMA and C within 2% (identical MMIO instructions); rppal within 5%.
   396Ôćĺ
   397Ôćĺ### 5.4 image_transform
   398Ôćĺ
   399Ôćĺ**Purpose:** Apply a 3x3 convolution filter (Gaussian blur) to an image buffer. This benchmark measures compute throughput on regular data with SIMD potential. The ARM NEON SIMD unit on Cortex-A76 should be exercisable from all four language configurations.
   400Ôćĺ
   401Ôćĺ**Parameters:**
   402Ôćĺ- Image: 4096 x 4096 pixels, RGBA (16 MiB per channel, 64 MiB total)
   403Ôćĺ- Filter: 3x3 Gaussian kernel `[1,2,1; 2,4,2; 1,2,1] / 16`
   404Ôćĺ- Iterations: 10 full-image transformations
   405Ôćĺ- SIMD: test both scalar and NEON-vectorized paths
   406Ôćĺ- VUMA: `vuma::simd::neon` intrinsics with IVE-verified lane access
   407Ôćĺ- C: hand-written NEON intrinsics via `<arm_neon.h>`
   408Ôćĺ- Rust safe: `std::simd` or packed SIMD operations
   409Ôćĺ- Rust unsafe: raw NEON intrinsics via `core::arch::aarch64`
   410Ôćĺ
   411Ôćĺ**Measurement:** Megapixels per second. Expect VUMA and C within 3% for both scalar and NEON paths; Rust may show minor differences depending on autovectorization quality.
   412Ôćĺ
   413Ôćĺ### 5.5 sort_benchmark
   414Ôćĺ
   415Ôćĺ**Purpose:** Sort 1,000,000 integers using multiple sorting algorithms. This benchmark exercises comparison-based branching, memory access patterns (sequential for merge sort, random for quicksort), and in-place vs out-of-place memory behavior.
   416Ôćĺ
   417Ôćĺ**Parameters:**
   418Ôćĺ- Data: 1,000,000 `i64` values, pseudo-random (seed 42)
   419Ôćĺ- Algorithms: quicksort (in-place), merge sort (out-of-place), radix sort (non-comparison)
   420Ôćĺ- Pre-sorted input: random, nearly-sorted (10% out of order), reverse-sorted, all-equal
   421Ôćĺ- Iterations: 50 per algorithm per input pattern
   422Ôćĺ- VUMA: `vuma::sort::quicksort` / `merge_sort` / `radix_sort` with IVE-verified partition bounds
   423Ôćĺ- C: `qsort()` from libc + hand-written merge/radix sort
   424Ôćĺ- Rust safe: `vec.sort_unstable()` (pdqsort) + hand-written merge/radix
   425Ôćĺ- Rust unsafe: same algorithms with `get_unchecked` for partition swaps
   426Ôćĺ
   427Ôćĺ**Measurement:** Cycles per element sorted. Expect VUMA and C within 5% for all algorithms; Rust's `sort_unstable` (pdqsort) may outperform naive quicksort in C.

   431Ôćĺ## 6. Verification Time Benchmarks
   432Ôćĺ
   433ÔćĺVerification time benchmarks measure how long IVE takes to prove program correctness at compile time. This is the novel cost axis that VUMA introduces: unlike C and Rust, where compilation is a fixed cost regardless of program correctness, VUMA's IVE must construct a formal proof that the program satisfies its safety invariants. These benchmarks quantify the "price of verification" and help the VUMA team identify scalability bottlenecks in IVE's proof engine.
   434Ôćĺ
   435ÔćĺAll verification times are measured on the AArch64 itself (not cross-compiled), using `time vuma build --release` with IVE enabled. The measurement includes parsing, type checking, IVE proof construction, and LLVM codegen. For comparison, we also measure `gcc -O2` and `cargo build --release` compilation times for equivalent C and Rust programs.
   436Ôćĺ
   437Ôćĺ### 6.1 trivial_program
   438Ôćĺ
   439Ôćĺ**Purpose:** Measure IVE overhead on the simplest possible program: allocate an integer, read it, free it. This establishes the baseline IVE startup cost.
   440Ôćĺ
   441Ôćĺ**Parameters:**
   442Ôćĺ- Program: `fn main() { let x = Box::new(42i64); let y = *x; assert_eq!(y, 42); }`
   443Ôćĺ- Lines of code: ~5
   444Ôćĺ- IVE proof obligations: 3 (allocation valid, dereference valid, free valid)
   445Ôćĺ- Compilation iterations: 20 (to warm filesystem caches)
   446Ôćĺ- VUMA: `vuma build --release`
   447Ôćĺ- C: `gcc -O2 trivial.c`
   448Ôćĺ- Rust: `cargo build --release`
   449Ôćĺ
   450Ôćĺ**Measurement:** Wall-clock compilation time in milliseconds. Expect VUMA to be 2-5x slower than C and Rust due to IVE proof construction overhead.
   451Ôćĺ
   452Ôćĺ### 6.2 dlist_10_nodes
   453Ôćĺ
   454Ôćĺ**Purpose:** Measure IVE verification time for a 10-node doubly-linked list. This exercises IVE's pointer-aliasing analysis at a small scale.
   455Ôćĺ
   456Ôćĺ**Parameters:**
   457Ôćĺ- Program: create a `DList<i64>`, insert 10 nodes, traverse, remove all
   458Ôćĺ- Lines of code: ~50
   459Ôćĺ- IVE proof obligations: ~80 (10 inserts x 4 pointer updates + 10 removes + 1 traversal)
   460Ôćĺ- Compilation iterations: 20
   461Ôćĺ
   462Ôćĺ**Measurement:** Compilation time. Expect VUMA compilation to take 0.5-2 seconds; C and Rust should compile in <100ms.
   463Ôćĺ
   464Ôćĺ### 6.3 dlist_100_nodes
   465Ôćĺ
   466Ôćĺ**Purpose:** Scale the doubly-linked list to 100 nodes to measure IVE's scaling behavior. This is critical: if IVE scales quadratically with node count, large programs become impractical.
   467Ôćĺ
   468Ôćĺ**Parameters:**
   469Ôćĺ- Program: create a `DList<i64>`, insert 100 nodes, traverse, remove all
   470Ôćĺ- Lines of code: ~50 (same structure, different loop count)
   471Ôćĺ- IVE proof obligations: ~800
   472Ôćĺ- Compilation iterations: 20
   473Ôćĺ
   474Ôćĺ**Measurement:** Compilation time. Compare with dlist_10_nodes to determine scaling factor. Ideal: linear scaling (10x time for 10x nodes). Acceptable: O(n log n). Problematic: O(n^2) or worse.
   475Ôćĺ
   476Ôćĺ### 6.4 arena_1000_allocs
   477Ôćĺ
   478Ôćĺ**Purpose:** Measure IVE verification time for an arena allocator with 1000 allocations. This tests IVE's ability to handle bulk allocation patterns without per-allocation proof overhead.
   479Ôćĺ
   480Ôćĺ**Parameters:**
   481Ôćĺ- Program: create an `Arena`, allocate 1000 objects of varying sizes, access all, reset arena
   482Ôćĺ- Lines of code: ~30
   483Ôćĺ- IVE proof obligations: ~3000 (1000 allocs + 1000 accesses + 1 reset + lifetime checks)
   484Ôćĺ- Compilation iterations: 20
   485Ôćĺ
   486Ôćĺ**Measurement:** Compilation time. Arena patterns should amortize verification cost due to uniform lifetime scoping.
   487Ôćĺ
   488Ôćĺ### 6.5 concurrent_4_threads
   489Ôćĺ
   490Ôćĺ**Purpose:** Measure IVE verification time for a 4-thread concurrent program. This exercises IVE's concurrent memory model proofs, which are the most computationally expensive verification step.
   491Ôćĺ
   492Ôćĺ**Parameters:**
   493Ôćĺ- Program: 4 threads sharing a mutex-protected counter, each incrementing 1000 times
   494Ôćĺ- Lines of code: ~40
   495Ôćĺ- IVE proof obligations: ~4000 (thread safety + lock discipline + data-race freedom for each shared access)
   496Ôćĺ- Compilation iterations: 20
   497Ôćĺ
   498Ôćĺ**Measurement:** Compilation time. Expect this to be the most expensive verification benchmark; may take 5-30 seconds depending on IVE's concurrent proof engine efficiency.
   499Ôćĺ
   500Ôćĺ### 6.6 graph_100_nodes
   501Ôćĺ
   502Ôćĺ**Purpose:** Measure IVE verification time for a 100-node graph structure. Graphs are the hardest data structure for ownership-based verification because nodes have arbitrary connectivity.
   503Ôćĺ
   504Ôćĺ**Parameters:**
   505Ôćĺ- Program: create a directed graph with 100 nodes and ~200 edges, perform BFS and DFS, deallocate
   506Ôćĺ- Lines of code: ~80
   507Ôćĺ- IVE proof obligations: ~5000 (node lifetime + edge validity + traversal bounds + deallocation completeness)
   508Ôćĺ- Compilation iterations: 20
   509Ôćĺ
   510Ôćĺ**Measurement:** Compilation time. This benchmark may require IVE to use arena-based lifetime inference for graph nodes. Expect 10-60 seconds depending on IVE's graph analysis capabilities.

   514Ôćĺ## 7. Measurement Methodology
   515Ôćĺ
   516ÔćĺRigorous measurement methodology is essential for producing benchmark results that are reproducible, defensible, and free of confounding factors. The VUMA benchmark suite adopts the following methodology, which is designed for the AArch64's specific hardware characteristics (Cortex-A76, 4 cores, 8 GB LPDDR4X-4267 RAM, 512 KB L2 cache per core, shared 2 MB L3 cache).
   517Ôćĺ
   518Ôćĺ### 7.1 Hardware Configuration
   519Ôćĺ
   520Ôćĺ- **Platform:** AArch64 (BCM2712, Quad-core ARM Cortex-A76 @ 2.4 GHz)
   521Ôćĺ- **RAM:** 8 GB LPDDR4X-4267
   522Ôćĺ- **OS:** AArch64 Linux (64-bit, Linux kernel 6.6+)
   523Ôćĺ- **CPU governor:** Set to `performance` mode (`cpufreq-set -g performance`) to eliminate dynamic frequency scaling
   524Ôćĺ- **Isolation:** Disable WiFi, Bluetooth, and HDMI output to minimize interrupt noise
   525Ôćĺ- **CPU affinity:** Each benchmark thread pinned to a dedicated core via `taskset` or `pthread_setaffinity_np`
   526Ôćĺ- **Huge pages:** Transparent hugepages disabled (`echo never > /sys/kernel/mm/transparent_hugepage/enabled`) for consistent page fault behavior
   527Ôćĺ
   528Ôćĺ### 7.2 Timing Mechanism
   529Ôćĺ
   530Ôćĺ- **Primary timer:** ARM64 PMU cycle counter (`cntvct_el0`), accessed via `__builtin_arm_rsr64("cntvct_el0")` in C/VUMA and `std::arch::aarch64::_rdcntvct_el0()` in Rust
   531Ôćĺ- **Resolution:** 1 cycle at 2.4 GHz = ~0.42 ns per tick
   532Ôćĺ- **Overhead:** Timer read overhead measured and subtracted (typically 15-20 cycles on Cortex-A76)
   533Ôćĺ- **Secondary timer:** `clock_gettime(CLOCK_MONOTONIC_RAW)` for wall-clock measurements (verification time, I/O-bound benchmarks)
   534Ôćĺ- **Conversion:** Cycles to nanoseconds via `cycles / 2.4` (fixed frequency in performance mode)
   535Ôćĺ
   536Ôćĺ### 7.3 Iteration Protocol
   537Ôćĺ
   538Ôćĺ- **Warmup:** 10 iterations (discard results) to warm instruction cache, branch predictor, and TLB
   539Ôćĺ- **Measurement:** 100 iterations (record all results)
   540Ôćĺ- **Reporting:**
   541Ôćĺ  - Mean: arithmetic mean of 100 iterations
   542Ôćĺ  - Stddev: standard deviation of 100 iterations
   543Ôćĺ  - Median: 50th percentile
   544Ôćĺ  - P95: 95th percentile (captures tail latency)
   545Ôćĺ  - Min: minimum iteration (best achievable, useful for comparing codegen quality)
   546Ôćĺ  - Coefficient of variation (CV): stddev/mean; any benchmark with CV > 5% is flagged as unreliable
   547Ôćĺ
   548Ôćĺ### 7.4 Comparison Matrix
   549Ôćĺ
   550Ôćĺ| Configuration | Compiler | Flags | Notes |
   551Ôćĺ|---|---|---|---|
   552Ôćĺ| VUMA verified | `vuma build` | `--release` | IVE enabled, all safety proofs constructed |
   553Ôćĺ| C | `gcc` | `-O2 -march=armv8.2-a` | Standard optimization, no UB sanitizer |
   554Ôćĺ| Rust safe | `cargo` | `--release` | All safe Rust idioms, bounds checks enabled |
   555Ôćĺ| Rust unsafe | `cargo` | `--release` | `unsafe` blocks for pointer ops, `get_unchecked` for indexing |
   556Ôćĺ
   557Ôćĺ### 7.5 Metrics Collected
   558Ôćĺ
   559Ôćĺ- **Execution time:** Mean cycles per operation (micro), mean wall-clock time (real-world)
   560Ôćĺ- **Memory usage:** Peak RSS via `/proc/self/status` VmHWM field, measured per benchmark
   561Ôćĺ- **Verification time:** Wall-clock compilation time (`time vuma build --release`), including IVE
   562Ôćĺ- **Binary size:** Stripped ELF size via `size` command and `ls -l` on the final binary
   563Ôćĺ- **Code size:** `.text` section size, `.rodata` section size (for embedded relevance)
   564Ôćĺ
   565Ôćĺ### 7.6 Statistical Significance
   566Ôćĺ
   567Ôćĺ- All results reported with 95% confidence intervals
   568Ôćĺ- Outliers (>3 sigma from mean) are recorded but excluded from primary statistics
   569Ôćĺ- Benchmarks are run three times on different days to detect environmental drift
   570Ôćĺ- Any benchmark where day-to-day variance exceeds 5% is re-investigated for hidden confounds
   571Ôćĺ
   572Ôćĺ### 7.7 Reproducibility
   573Ôćĺ
   574Ôćĺ- All benchmark source code, build scripts, and raw results are stored in `vuma/benchmarks/`
   575Ôćĺ- A `Dockerfile` (ARM64) is provided to recreate the exact benchmark environment
   576Ôćĺ- Random seeds are fixed for all randomized benchmarks
   577Ôćĺ- GCC version: 13.2+, Rust version: 1.75+, VUMA version: git commit hash recorded

   581Ôćĺ## 8. Expected Results
   582Ôćĺ
   583ÔćĺThis section documents our hypotheses about benchmark outcomes. These are informed predictions based on VUMA's design principles (compile-time verification, zero-cost abstractions, C-equivalent codegen) and known performance characteristics of C and Rust on ARM64. Actual results may differ, and deviations from these expectations will guide VUMA's optimization roadmap.
   584Ôćĺ
   585Ôćĺ### 8.1 Execution Time
   586Ôćĺ
   587Ôćĺ**Hypothesis: VUMA execution time within 5% of C across all benchmarks, and equal to or better than Rust safe.**
   588Ôćĺ
   589ÔćĺRationale: VUMA compiles through LLVM (same backend as Clang/Rust) after IVE verification completes. Since IVE operates entirely at compile time, the generated machine code should be identical to what C would produce for equivalent logic. The only potential overhead sources are: (a) VUMA-specific calling conventions for verified functions, which we expect LLVM to optimize away; (b) extra move instructions for ownership tracking, which register allocation should eliminate; (c) any IVE-inserted runtime checks that couldn't be statically proven, which we expect to be zero for well-typed programs.
   590Ôćĺ
   591ÔćĺAgainst Rust safe, VUMA should win on bounds-checked operations (vec index, array access) by 10-30%, since IVE elides bounds checks that Rust's safe API must retain. Against Rust unsafe, VUMA should be within 1-2%, since both produce equivalent unchecked machine code.
   592Ôćĺ
   593ÔćĺThe 5% margin accounts for: (a) minor codegen differences due to VUMA's type layout rules, (b) linker differences (VUMA links against a minimal runtime vs glibc for C), and (c) instruction scheduling differences that may favor one compiler over another on specific Cortex-A76 pipeline configurations.
   594Ôćĺ
   595Ôćĺ### 8.2 Verification Time
   596Ôćĺ
   597Ôćĺ**Hypothesis: Sub-second for micro-benchmarks, 1-10 seconds for data structures, 10-60 seconds for concurrent/graph programs.**
   598Ôćĺ
   599ÔćĺRationale: IVE's proof engine uses an SMT-based approach (Z3 or custom solver) with incremental verification. Micro-benchmarks have trivial proof obligations (3-10 per benchmark) that should resolve in milliseconds. Data structure benchmarks have 100-1000 proof obligations with moderate complexity (pointer aliasing, lifetime scoping) that should resolve in 1-10 seconds. Concurrent benchmarks have 1000-5000 proof obligations with high complexity (happens-before relations, lock disciplines) that may require 10-60 seconds.
   600Ôćĺ
   601ÔćĺThe critical threshold is the developer iteration loop: if IVE adds more than 5 seconds to compilation, it begins to impact productivity. For programs up to ~10,000 lines, we expect IVE to stay under this threshold. For larger programs, incremental verification (re-proving only changed functions) should maintain acceptable iteration times.
   602Ôćĺ
   603ÔćĺComparison: C compilation is essentially free (gcc -O2 on a 100-line program: ~50ms). Rust compilation is slow but acceptable (cargo build --release on a 100-line program: ~2-5 seconds, mostly due to LLVM optimization and monomorphization). VUMA verification should fall between Rust and an order-of-magnitude slower than Rust for complex concurrent programs.
   604Ôćĺ
   605Ôćĺ### 8.3 Memory Usage
   606Ôćĺ
   607Ôćĺ**Hypothesis: VUMA memory usage identical to C, 10-30% less than Rust safe.**
   608Ôćĺ
   609ÔćĺRationale: VUMA has no garbage collector, no reference counting, no hidden vtables, and no panic unwinding tables (VUMA uses abort-on-error semantics for verified programs). Rust safe programs pay for: (a) `Rc<T>` / `Arc<T>` reference counting overhead (2 `usize` per reference: strong count + weak count), (b) `RefCell<T>` dynamic borrowing tracking (1 `usize` for borrow state), (c) panic unwinding tables (~5-10% of binary size, though not runtime memory), and (d) `enum` discriminant overhead for `Result<T, E>` error propagation.
   610Ôćĺ
   611ÔćĺVUMA avoids all of these because: (a) IVE proves ownership statically, eliminating reference counting, (b) IVE proves borrowing statically, eliminating `RefCell`, (c) verified programs cannot panic (IVE proves all error paths handled), and (d) VUMA uses explicit error returns without `Result` discriminant overhead when IVE can prove the error path is unreachable.
   612Ôćĺ
   613ÔćĺC achieves the same memory efficiency through the programmer's discipline; VUMA achieves it through IVE's mechanical verification. The difference is that VUMA's efficiency is guaranteed, while C's is aspirational.
   614Ôćĺ
   615Ôćĺ### 8.4 Binary Size
   616Ôćĺ
   617Ôćĺ**Hypothesis: VUMA binary size similar to C, 20-40% smaller than Rust.**
   618Ôćĺ
   619ÔćĺRationale: Rust binaries include: (a) panic unwinding code and `.eh_frame` sections (~50-100 KiB even for simple programs), (b) vtables for `dyn Trait` objects, (c) `std` library bloat (formatter, allocator, threading infrastructure), and (d) duplicate generic monomorphizations. VUMA eliminates: (a) panic unwinding (verified programs abort on impossible errors), (b) vtables (VUMA uses compile-time dispatch for verified trait objects), (c) minimal runtime (no `std`, only `vuma::rt` with ~5 KiB of startup code), and (d) IVE-aware monomorphization that eliminates dead code more aggressively than Rust's generic instantiation.
   620Ôćĺ
   621ÔćĺExpected stripped binary sizes for a "hello world" program:
   622Ôćĺ- C: ~15 KiB (dynamically linked to glibc)
   623Ôćĺ- VUMA: ~20 KiB (statically linked minimal runtime)
   624Ôćĺ- Rust: ~300 KiB (dynamically linked to libc, but includes unwinding)
   625Ôćĺ- Rust (no_std): ~5 KiB (bare metal, but impractical for most applications)
   626Ôćĺ
   627ÔćĺFor the JSON parser benchmark:
   628Ôćĺ- C (cJSON): ~50 KiB
   629Ôćĺ- VUMA: ~60 KiB
   630Ôćĺ- Rust (serde_json): ~500 KiB

   634Ôćĺ## Appendix A: Benchmark Execution Order
   635Ôćĺ
   636ÔćĺBenchmarks should be executed in the following order to minimize thermal throttling and cache interference:
   637Ôćĺ
   638Ôćĺ1. Micro-benchmarks (smallest memory footprint, shortest duration)
   639Ôćĺ2. Data structure benchmarks (moderate memory, moderate duration)
   640Ôćĺ3. Verification time benchmarks (CPU-intensive compilation, no runtime measurement)
   641Ôćĺ4. Concurrency benchmarks (highest thermal load, run last)
   642Ôćĺ5. Real-world benchmarks (mixed characteristics, run individually)
   643Ôćĺ
   644ÔćĺBetween each benchmark category, a 30-second cool-down period is enforced to allow the AArch64's thermal management to return to baseline.
