    11→## 1. Introduction
    12→
    13→This document provides a rigorous, operation-by-operation proof that a circular doubly-linked list with sentinel satisfies all five VUMA memory invariants under the Verified-Unsafe Memory Access model. The doubly-linked list is the canonical data structure that Rust's borrow checker cannot verify without `unsafe` — making it the ideal showcase for VUMA's global verification approach.
    14→
    15→The VUMA model defines five global invariants that the Inference and Verification Engine (IVE) must prove for every program:
    16→
    17→1. **Liveness** — Every pointer dereference targets allocated memory.
    18→2. **Exclusivity** — No two simultaneous mutable accesses target the same memory.
    19→3. **Interpretation** — Every memory access respects a valid representation descriptor.
    20→4. **Origin** — Every address traces back to a valid allocation.
    21→5. **Cleanup** — Every allocation is eventually freed.
    22→
    23→We prove each invariant for each operation of the doubly-linked list.
    24→
    25→---
    26→
    27→## 2. Data Structure Definition
    28→
    29→### 2.1 Node Layout
    30→
    31→```
    32→Node = { prev: Address, next: Address, data: u64 }
    33→```
    34→
    35→**Representation Descriptor (RepD):**
    36→- Size: 24 bytes (3 × 8-byte fields)
    37→- Alignment: 8 bytes
    38→- Field offsets: `prev` at +0, `next` at +8, `data` at +16
    39→
    40→### 2.2 List Structure
    41→
    42→```
    43→List = { sentinel: Address }
    44→```
    45→
    46→The list uses a **circular sentinel** design:
    47→- The sentinel node is a `Node` whose `data` field is unused.
    48→- `sentinel.next` points to the first real node (or back to sentinel if empty).
    49→- `sentinel.prev` points to the last real node (or back to sentinel if empty).
    50→- An empty list has `sentinel.prev = sentinel` and `sentinel.next = sentinel`.
    51→
    52→**Invariant (structural):** For every node `n` in the list, `n.next.prev = n` and `n.prev.next = n`. This holds for the sentinel as well.
    53→
    54→### 2.3 Memory State Graph (MSG) Notation
    55→
    56→We write `alloc(a, 24, 8)` to denote a 24-byte, 8-byte-aligned allocation at address `a`. We write `free(a)` to denote deallocation. We write `live(a)` to mean address `a` is within an allocated region at the current program point. We write `*(a + offset)` for a memory access at address `a` with the given byte offset.

    60→## 3. Operation Proofs
    61→
    62→### 3.1 Operation: `new_list()`
    63→
    64→**Specification:** Allocate a sentinel node. Set `sentinel.prev = sentinel` and `sentinel.next = sentinel`. Return the list handle.
    65→
    66→**Implementation:**
    67→```
    68→new_list() → List:
    69→    s = allocate(24, 8)        // allocate Node-sized region
    70→    s.prev = s                 // circular: points to self
    71→    s.next = s                 // circular: points to self
    72→    return List { sentinel: s }
    73→```
    74→
    75→#### 3.1.1 Liveness Proof
    76→
    77→**Claim:** After `new_list()`, the sentinel is allocated and all pointer dereferences within the operation target live memory.
    78→
    79→**Proof:**
    80→
    81→- `s = allocate(24, 8)`: `s` is freshly allocated. By the semantics of `allocate`, `live(s)` holds after this line. The region `[s, s+24)` is allocated.
    82→- `s.prev = s`: This is a write to `s + 0`. Since `live(s)` holds and `0 < 24`, this write targets live memory. ✓
    83→- `s.next = s`: This is a write to `s + 8`. Since `live(s)` holds and `8 < 24`, this write targets live memory. ✓
    84→- The return value contains `s`, which is live. ✓
    85→
    86→**Postcondition:** `live(s)` holds, `s.prev = s`, `s.next = s`. The sentinel points to itself, forming a valid empty circular list.
    87→
    88→**Conclusion:** Liveness holds for `new_list()`. ∎
    89→
    90→#### 3.1.2 Exclusivity Proof
    91→
    92→**Claim:** No two simultaneous mutable accesses target the same memory during `new_list()`.
    93→
    94→**Proof:**
    95→
    96→All operations are sequential and single-threaded. The writes occur in order:
    97→1. Write to `s + 0` (`s.prev`)
    98→2. Write to `s + 8` (`s.next`)
    99→
   100→These are non-overlapping regions within the same allocation (offset 0 and offset 8, each 8 bytes wide). They are also sequential, not concurrent. In single-threaded execution, exclusivity is trivially satisfied because only one access occurs at any point in time.
   101→
   102→**Conclusion:** Exclusivity holds for `new_list()`. ∎
   103→
   104→#### 3.1.3 Interpretation Proof
   105→
   106→**Claim:** Every memory access in `new_list()` interprets the target bytes according to a valid representation descriptor.
   107→
   108→**Proof:**
   109→
   110→- `allocate(24, 8)`: Allocates a region with the Node RepD (24 bytes, 8-byte alignment). The allocation itself creates no interpretation conflict.
   111→- `s.prev = s`: Writes an `Address` (8-byte pointer) to offset 0. The Node RepD specifies `prev: Address @ 0`. Interpretation matches. ✓
   112→- `s.next = s`: Writes an `Address` (8-byte pointer) to offset 8. The Node RepD specifies `next: Address @ 8`. Interpretation matches. ✓
   113→
   114→**Conclusion:** Interpretation holds for `new_list()`. ∎
   115→
   116→#### 3.1.4 Origin Proof
   117→
   118→**Claim:** Every address used in `new_list()` traces back to a valid allocation.
   119→
   120→**Proof:**
   121→
   122→- `s` is the return value of `allocate(24, 8)`. This is a root allocation — the origin of `s` is the allocation site itself.
   123→- `s.prev` and `s.next` are set to `s`, which has a valid origin (the allocation). So all stored pointers trace to a valid allocation.
   124→- No pointer arithmetic or address computation is performed beyond the initial allocation.
   125→
   126→**Conclusion:** Origin holds for `new_list()`. ∎
   127→
   128→#### 3.1.5 Cleanup Proof
   129→
   130→**Claim:** The allocation in `new_list()` will eventually be freed.
   131→
   132→**Proof:**
   133→
   134→This is an allocation-creating operation. Cleanup cannot be proven for `new_list()` in isolation — it depends on the caller eventually invoking `free_list()`. However, the IVE tracks this allocation as having no matching `free()` yet and records it as a **pending deallocation obligation**. When `free_list()` is later proven (§3.5), the IVE closes the obligation. The full proof of cleanup is established at the program level, not the operation level.
   135→
   136→**Partial conclusion:** Cleanup is conditionally satisfied, pending proof of `free_list()`. ∎

   140→### 3.2 Operation: `push_back(list, value)`
   141→
   142→**Specification:** Insert a new node containing `value` between the last node and the sentinel.
   143→
   144→**Implementation:**
   145→```
   146→push_back(list, value):
   147→    last = list.sentinel.prev        // current last node
   148→    n = allocate(24, 8)              // allocate new node
   149→    n.data = value                   // set data
   150→    n.prev = last                    // link backward to last
   151→    n.next = list.sentinel           // link forward to sentinel
   152→    last.next = n                    // link last forward to new node
   153→    list.sentinel.prev = n           // link sentinel backward to new node
   154→```
   155→
   156→#### 3.2.1 Liveness Proof
   157→
   158→**Claim:** After `push_back()`, all existing nodes remain allocated and the new node is allocated. All pointer dereferences target live memory.
   159→
   160→**Proof:**
   161→
   162→- `last = list.sentinel.prev`: Reads `list.sentinel + 0`. By the structural invariant, `list.sentinel` is live (allocated by `new_list()`). Reading from a live region is valid. `last` is then either the sentinel itself (empty list) or a previously pushed node (non-empty). By induction, all previously pushed nodes are still live. ✓
   163→- `n = allocate(24, 8)`: `n` is freshly allocated. `live(n)` holds. ✓
   164→- `n.data = value`: Write to `n + 16`. `live(n)` holds, `16 < 24`. ✓
   165→- `n.prev = last`: Write to `n + 0`. `live(n)` holds. `last` is live (proven above). ✓
   166→- `n.next = list.sentinel`: Write to `n + 8`. `live(n)` holds. `list.sentinel` is live. ✓
   167→- `last.next = n`: Write to `last + 8`. `live(last)` holds (by induction from structural invariant). `8 < 24`. ✓
   168→- `list.sentinel.prev = n`: Write to `list.sentinel + 0`. `live(list.sentinel)` holds. `0 < 24`. ✓
   169→
   170→**Postcondition:** The new node `n` is linked between `last` and `sentinel`. All existing nodes are untouched (only `last.next` is modified, which does not affect the liveness of other nodes). The structural invariant is maintained: `n.prev.next = n` (since `last.next = n`), `n.next.prev = n` (since `sentinel.prev = n`), and the old `last.next.prev` link (which was `last`) remains `last` because `n.prev = last`.
   171→
   172→**Conclusion:** Liveness holds for `push_back()`. ∎
   173→
   174→#### 3.2.2 Exclusivity Proof
   175→
   176→**Claim:** No two simultaneous mutable accesses target the same memory during `push_back()`.
   177→
   178→**Proof:**
   179→
   180→The writes in `push_back()` target four distinct nodes:
   181→1. `n.data` — write to node `n` at offset 16
   182→2. `n.prev` — write to node `n` at offset 0
   183→3. `n.next` — write to node `n` at offset 8
   184→4. `last.next` — write to node `last` at offset 8
   185→5. `list.sentinel.prev` — write to node `list.sentinel` at offset 0
   186→
   187→**Case 1: List has at least one element (last ≠ sentinel).**
   188→
   189→Nodes `n`, `last`, and `sentinel` are all distinct. Writes 1–3 target `n`, write 4 targets `last`, write 5 targets `sentinel`. No two writes target the same node. Since all operations are sequential, no conflicts arise. ✓
   190→
   191→**Case 2: List is empty (last = sentinel).**
   192→
   193→Writes 4 and 5 both target the sentinel node. Write 4 sets `sentinel.next` (offset 8), write 5 sets `sentinel.prev` (offset 0). These are **non-overlapping regions** within the same node (offset 0, size 8 vs. offset 8, size 8). Furthermore, they are **sequential**, not concurrent. Even within a single node, sequential writes to non-overlapping fields do not violate exclusivity.
   194→
   195→**Conclusion:** Exclusivity holds for `push_back()` in all cases. ∎
   196→
   197→#### 3.2.3 Interpretation Proof
   198→
   199→**Claim:** Every memory access in `push_back()` respects the Node RepD.
   200→
   201→**Proof:**
   202→
   203→All reads and writes access fields at their declared offsets:
   204→- `list.sentinel.prev`: read `Address @ +0` from a Node. ✓
   205→- `n.data`: write `u64 @ +16` to a Node. ✓
   206→- `n.prev`: write `Address @ +0` to a Node. ✓
   207→- `n.next`: write `Address @ +8` to a Node. ✓
   208→- `last.next`: write `Address @ +8` to a Node. ✓
   209→- `list.sentinel.prev`: write `Address @ +0` to a Node. ✓
   210→
   211→All interpretations match the Node RepD (size 24, align 8, fields at offsets 0, 8, 16).
   212→
   213→**Conclusion:** Interpretation holds for `push_back()`. ∎
   214→
   215→#### 3.2.4 Origin Proof
   216→
   217→**Claim:** Every address used in `push_back()` traces to a valid allocation.
   218→
   219→**Proof:**
   220→
   221→- `list.sentinel`: Origin is the `allocate` in `new_list()`. ✓
   222→- `last = list.sentinel.prev`: `last` is read from a pointer stored in the sentinel. By the structural invariant and induction, every `.prev`/`.next` pointer in the list points to a node that was allocated by a prior `push_back()` or `push_front()` call. Each such allocation is a valid origin. ✓
   223→- `n = allocate(24, 8)`: Root allocation. ✓
   224→- All stored pointers (`n.prev`, `n.next`) are set to `last` or `list.sentinel`, both of which have valid origins. ✓
   225→
   226→Pointer derivation chain: `list.sentinel` → (read `.prev`) → `last`. This is a one-hop derivation: read an `Address` field from a live node. Since the field stores a valid address (by the structural invariant), the derived pointer has a valid origin.
   227→
   228→**Conclusion:** Origin holds for `push_back()`. ∎
   229→
   230→#### 3.2.5 Cleanup Proof
   231→
   232→**Claim:** The allocation in `push_back()` will eventually be freed.
   233→
   234→**Proof:**
   235→
   236→Same argument as §3.1.5: the allocation of `n` creates a pending deallocation obligation. This obligation is discharged by `free_list()` (§3.5) or by `remove(n)` (§3.4). The IVE tracks this obligation globally.
   237→
   238→**Partial conclusion:** Cleanup is conditionally satisfied. ∎

   242→### 3.3 Operation: `push_front(list, value)`
   243→
   244→**Specification:** Insert a new node containing `value` between the sentinel and the first node.
   245→
   246→**Implementation:**
   247→```
   248→push_front(list, value):
   249→    first = list.sentinel.next       // current first node
   250→    n = allocate(24, 8)              // allocate new node
   251→    n.data = value                   // set data
   252→    n.prev = list.sentinel           // link backward to sentinel
   253→    n.next = first                   // link forward to first
   254→    first.prev = n                   // link first backward to new node
   255→    list.sentinel.next = n           // link sentinel forward to new node
   256→```
   257→
   258→#### 3.3.1 Liveness Proof
   259→
   260→**Claim:** After `push_front()`, all existing nodes remain allocated and the new node is allocated.
   261→
   262→**Proof:** Symmetric to §3.2.1. The key steps:
   263→- `first = list.sentinel.next`: `list.sentinel` is live; by the structural invariant, `first` is live. ✓
   264→- `n = allocate(24, 8)`: Fresh allocation, `live(n)` holds. ✓
   265→- All writes (`n.data`, `n.prev`, `n.next`, `first.prev`, `list.sentinel.next`) target live nodes at valid offsets. ✓
   266→
   267→**Conclusion:** Liveness holds for `push_front()`. ∎
   268→
   269→#### 3.3.2 Exclusivity Proof
   270→
   271→**Claim:** No two simultaneous mutable accesses target the same memory during `push_front()`.
   272→
   273→**Proof:** Symmetric to §3.2.2. The writes target `n` (three fields), `first` (one field), and `sentinel` (one field). In the empty-list case (`first = sentinel`), writes to `first.prev` and `list.sentinel.next` target the same node at offsets 0 and 8 — non-overlapping and sequential. In the non-empty case, all three nodes are distinct.
   274→
   275→**Conclusion:** Exclusivity holds for `push_front()`. ∎
   276→
   277→#### 3.3.3 Interpretation Proof
   278→
   279→**Claim:** Every access respects the Node RepD.
   280→
   281→**Proof:** Identical structure to §3.2.3. All field accesses match declared offsets and types. ✓
   282→
   283→**Conclusion:** Interpretation holds for `push_front()`. ∎
   284→
   285→#### 3.3.4 Origin Proof
   286→
   287→**Claim:** Every address traces to a valid allocation.
   288→
   289→**Proof:** Identical structure to §3.2.4. All pointers derive from `allocate()` or from `.prev`/`.next` reads of live nodes. ✓
   290→
   291→**Conclusion:** Origin holds for `push_front()`. ∎
   292→
   293→#### 3.3.5 Cleanup Proof
   294→
   295→**Claim:** The allocation will eventually be freed.
   296→
   297→**Proof:** Same as §3.2.5. Pending obligation discharged by `free_list()` or `remove()`. ✓

   301→### 3.4 Operation: `remove(node)`
   302→
   303→**Specification:** Unlink `node` from its list and free it. The node must be a real node (not the sentinel).
   304→
   305→**Implementation:**
   306→```
   307→remove(node):
   308→    prev = node.prev                 // node's predecessor
   309→    next = node.next                 // node's successor
   310→    prev.next = next                 // unlink: predecessor skips over node
   311→    next.prev = prev                 // unlink: successor skips over node
   312→    free(node)                       // deallocate the removed node
   313→```
   314→
   315→**This is the critical operation.** It is the one Rust's borrow checker cannot verify, because it involves simultaneous mutable access to adjacent nodes.
   316→
   317→#### 3.4.1 Liveness Proof
   318→
   319→**Claim:** After `remove(node)`, the removed node is freed, and all pointers to it were updated before the free. No use-after-free occurs.
   320→
   321→**Proof:**
   322→
   323→- `prev = node.prev`: Read from `node + 0`. `live(node)` holds (node has not been freed yet). By the structural invariant, `prev` is a live node. ✓
   324→- `next = node.next`: Read from `node + 8`. `live(node)` holds. By the structural invariant, `next` is a live node. ✓
   325→- `prev.next = next`: Write to `prev + 8`. `live(prev)` holds. This overwrites the pointer that previously pointed to `node`, replacing it with `next`. After this line, `prev` no longer points to `node`. ✓
   326→- `next.prev = prev`: Write to `next + 0`. `live(next)` holds. This overwrites the pointer that previously pointed to `node`, replacing it with `prev`. After this line, `next` no longer points to `node`. ✓
   327→- `free(node)`: Deallocates `node`. After this line, `live(node)` is **false**.
   328→
   329→**Critical analysis: Are there any dangling pointers after `free(node)`?**
   330→
   331→Before `free(node)`, there were exactly two pointers to `node`: `prev.next` and `next.prev`. Both were overwritten in lines 3 and 4 — **before** the `free()` call. Therefore, at the point of `free(node)`, no live pointer in the list targets `node`. The only remaining reference to `node` is the local variable `node` itself, which goes out of scope after `free()`. The `node` variable is never dereferenced after `free()`.
   332→
   333→**This is the key insight that Rust's borrow checker cannot prove.** Rust sees `(*node.prev).next = node.next` and `(*node.next).prev = node.prev` as two mutable borrows that might alias (when `node.prev == node.next`, i.e., the single-element case). But VUMA's global analysis sees that:
   334→1. Both borrows are to **adjacent** nodes, not the same node (in the general case).
   335→2. Even when they happen to be the same node (sentinel in the single-element case), the writes target **different fields** at different offsets, and are **sequential**.
   336→3. The borrows are completed (the writes finish) before `free()` is called, so no use-after-free is possible.
   337→
   338→**Conclusion:** Liveness holds for `remove()`. ∎
   339→
   340→#### 3.4.2 Exclusivity Proof
   341→
   342→**Claim:** No two simultaneous mutable accesses target the same memory during `remove()`.
   343→
   344→**Proof:**
   345→
   346→The writes in `remove()` are:
   347→1. `prev.next = next` — write to `prev + 8`
   348→2. `next.prev = prev` — write to `next + 0`
   349→
   350→**Case 1: `prev ≠ next` (list has ≥ 2 elements).**
   351→
   352→The two writes target different nodes at different addresses. No aliasing. Exclusivity holds trivially. ✓
   353→
   354→**Case 2: `prev = next` (list has exactly 1 element, so both point to sentinel).**
   355→
   356→Both writes target the same node (the sentinel). However:
   357→- Write 1 targets offset 8 (`next` field).
   358→- Write 2 targets offset 0 (`prev` field).
   359→- These are non-overlapping 8-byte regions within the same 24-byte node.
   360→- The writes are **sequential** — write 1 completes before write 2 begins.
   361→
   362→In VUMA, exclusivity is defined over **simultaneous** accesses. Sequential accesses to non-overlapping regions of the same allocation do not violate exclusivity, even if they occur within the same function invocation. This is because the memory model tracks accesses in program order, and two non-overlapping writes to the same allocation cannot interfere.
   363→
   364→**Contrast with Rust's borrow checker:** Rust would reject this code because `(*node.prev).next = node.next` creates a mutable borrow of `*node.prev`, and `(*node.next).prev = node.prev` creates a mutable borrow of `*node.next`. When `node.prev == node.next` (both are the sentinel), these are two simultaneous mutable borrows of the same data, which Rust forbids — even though the borrows target different, non-overlapping fields and are used sequentially. Rust's local, field-insensitive analysis cannot distinguish "two mutable borrows of different fields of the same struct" from "two mutable borrows of the same data."
   365→
   366→**Conclusion:** Exclusivity holds for `remove()` in all cases. ∎
   367→
   368→#### 3.4.3 Interpretation Proof
   369→
   370→**Claim:** Every memory access in `remove()` respects the Node RepD.
   371→
   372→**Proof:**
   373→
   374→- `node.prev`: read `Address @ +0` from a Node. ✓
   375→- `node.next`: read `Address @ +8` from a Node. ✓
   376→- `prev.next = next`: write `Address @ +8` to a Node. ✓
   377→- `next.prev = prev`: write `Address @ +0` to a Node. ✓
   378→- `free(node)`: deallocates a 24-byte, 8-byte-aligned Node region. ✓
   379→
   380→All accesses match the Node RepD.
   381→
   382→**Conclusion:** Interpretation holds for `remove()`. ∎
   383→
   384→#### 3.4.4 Origin Proof
   385→
   386→**Claim:** Every address used in `remove()` traces to a valid allocation.
   387→
   388→**Proof:**
   389→
   390→- `node`: The caller provides this address. By the precondition of `remove()`, `node` is a real node in the list, allocated by a prior `push_back()` or `push_front()`. Origin: the `allocate()` call in that push operation. ✓
   391→- `prev = node.prev`: Derived by reading a pointer from a live node. By the structural invariant, this pointer points to a valid allocated node. Origin: the `allocate()` call that created that node. ✓
   392→- `next = node.next`: Same argument as `prev`. ✓
   393→- All stored values (`prev.next`, `next.prev`) are addresses (`next`, `prev`) with valid origins. ✓
   394→
   395→Pointer derivation chains:
   396→- `node.prev` → one-hop read from `node` → `prev` (origin: push allocation)
   397→- `node.next` → one-hop read from `node` → `next` (origin: push allocation)
   398→
   399→All derivations stay within allocated regions (they read an 8-byte pointer field from a 24-byte node, which is within bounds).
   400→
   401→**Conclusion:** Origin holds for `remove()`. ∎
   402→
   403→#### 3.4.5 Cleanup Proof
   404→
   405→**Claim:** `free(node)` discharges the allocation obligation for `node`.
   406→
   407→**Proof:**
   408→
   409→The `free(node)` call exactly matches the `allocate(24, 8)` that created `node` in a prior push operation. The IVE verifies:
   410→1. `node` was allocated exactly once (by a push operation).
   411→2. `node` is freed exactly once (by this `remove()` or by `free_list()`).
   412→3. No `free()` occurs twice for the same address.
   413→4. After `free(node)`, no pointer to `node` is dereferenced (proven in §3.4.1).
   414→
   415→This discharges the pending deallocation obligation for `node`.
   416→
   417→**Conclusion:** Cleanup holds for `remove()`. ∎

   420→
   421→### 3.5 Operation: `free_list(list)`
   422→
   423→**Specification:** Iterate through the list, freeing every node, and free the sentinel last.
   424→
   425→**Implementation:**
   426→```
   427→free_list(list):
   428→    current = list.sentinel.next          // start at first node
   429→    while current != list.sentinel:
   430→        next = current.next               // save next pointer
   431→        free(current)                     // free current node
   432→        current = next                    // advance
   433→    free(list.sentinel)                   // free sentinel last
   434→```
   435→
   436→#### 3.5.1 Liveness Proof
   437→
   438→**Claim:** Every `free()` call targets a live node, and no freed node is accessed afterward.
   439→
   440→**Proof:**
   441→
   442→- `current = list.sentinel.next`: `list.sentinel` is live. Read from offset 8. By structural invariant, `current` is either `sentinel` (empty list) or a live node. ✓
   443→- **Loop invariant:** At the start of each iteration, `current` is a live, allocated node that is not the sentinel.
   444→- `next = current.next`: Read from `current + 8`. `current` is live (loop invariant). By structural invariant, `current.next` is a live node. ✓
   445→- `free(current)`: Deallocates `current`. After this line, `live(current)` is false. But `current` is never dereferenced again — the next line sets `current = next`, which is a different, live node. ✓
   446→- `current = next`: `next` was read from `current.next` before `current` was freed, so `next` is valid and live. ✓
   447→- When the loop exits, `current = list.sentinel`, which is still live (it was never freed inside the loop). ✓
   448→- `free(list.sentinel)`: Deallocates the sentinel. After this, the list is fully freed. ✓
   449→
   450→**Critical point:** The loop saves `next = current.next` **before** freeing `current`. This ensures that the traversal pointer is not invalidated by the free. This is a classic pattern that VUMA's IVE can verify but that simpler local analyses sometimes miss.
   451→
   452→**Conclusion:** Liveness holds for `free_list()`. ∎
   453→
   454→#### 3.5.2 Exclusivity Proof
   455→
   456→**Claim:** No two simultaneous mutable accesses occur.
   457→
   458→**Proof:**
   459→
   460→`free_list()` is single-threaded and sequential. Each `free()` call targets a different node (the loop advances before the next free). The `free()` calls are not simultaneous. No aliasing conflicts arise.
   461→
   462→**Conclusion:** Exclusivity holds for `free_list()`. ∎
   463→
   464→#### 3.5.3 Interpretation Proof
   465→
   466→**Claim:** Every access respects the Node RepD.
   467→
   468→**Proof:**
   469→
   470→- `list.sentinel.next`: read `Address @ +8`. ✓
   471→- `current.next`: read `Address @ +8`. ✓
   472→- `free(current)`: deallocates a 24-byte, 8-byte-aligned region (matching the Node allocation). ✓
   473→- `free(list.sentinel)`: same. ✓
   474→
   475→All accesses match the Node RepD.
   476→
   477→**Conclusion:** Interpretation holds for `free_list()`. ∎
   478→
   479→#### 3.5.4 Origin Proof
   480→
   481→**Claim:** Every address traces to a valid allocation.
   482→
   483→**Proof:**
   484→
   485→- `list.sentinel`: Origin is `allocate()` in `new_list()`. ✓
   486→- `current`: Initially `list.sentinel.next` (one-hop derivation from sentinel). In subsequent iterations, `current = next = current_old.next` (one-hop derivation from the previously visited node). By induction, every value of `current` traces back through a chain of `.next` reads to the sentinel, which has a valid origin. ✓
   487→- `next = current.next`: One-hop derivation from `current`. ✓
   488→
   489→All derivation chains terminate at valid `allocate()` sites.
   490→
   491→**Conclusion:** Origin holds for `free_list()`. ∎
   492→
   493→#### 3.5.5 Cleanup Proof
   494→
   495→**Claim:** Every node in the list is freed exactly once, and the sentinel is freed exactly once.
   496→
   497→**Proof:**
   498→
   499→The loop iterates through every real node between `sentinel.next` and `sentinel` (exclusive), freeing each one. By the structural invariant, these are exactly the nodes that were allocated by `push_back()` or `push_front()`. After the loop, `free(list.sentinel)` frees the sentinel allocated by `new_list()`.
   500→
   501→**Total allocations:**
   502→- 1 from `new_list()` (sentinel)
   503→- N from `push_back()`/`push_front()` (N real nodes)
   504→
   505→**Total deallocations:**
   506→- N from the loop body (real nodes)
   507→- 1 from `free(list.sentinel)` (sentinel)
   508→
   509→Total allocations = total deallocations = N + 1. ✓
   510→
   511→**No double-free:** Each node is visited exactly once by the loop (the list is acyclic in terms of real nodes — only the sentinel creates the cycle). The sentinel is freed after the loop exits. No node is freed twice.
   512→
   513→**Combined with §3.4.5:** If some nodes were removed by `remove()` before `free_list()` is called, those nodes have already been freed. The loop will skip over them (they are no longer in the list). The allocation count still balances: each node is freed either by `remove()` or by `free_list()`, never both.
   514→
   515→**Conclusion:** Cleanup holds for `free_list()`. ∎

   519→## 4. Comparison with Rust's Borrow Checker
   520→
   521→### 4.1 Why Rust Requires `unsafe`
   522→
   523→Rust's borrow checker enforces memory safety through **local, syntactic rules**:
   524→
   525→1. **Alias rule:** At any point, you may have either exactly one mutable reference (`&mut T`) or any number of immutable references (`&T`), but not both.
   526→2. **Lifetime rule:** A reference must not outlive the data it borrows.
   527→3. **Scope rule:** A borrow's lifetime is determined by its syntactic scope — the region of code where the reference is used.
   528→
   529→These rules are **sound but incomplete**: they reject some safe programs. The doubly-linked list is the canonical example.
   530→
   531→### 4.2 The Specific Conflict
   532→
   533→Consider the `remove()` operation in Rust-like pseudocode:
   534→
   535→```rust
   536→fn remove(node: &mut Node) {
   537→    let prev = node.prev;            // *mut Node
   538→    let next = node.next;            // *mut Node
   539→    unsafe {
   540→        (*prev).next = next;         // mutable borrow of *prev
   541→        (*next).prev = prev;         // mutable borrow of *next
   542→    }
   543→    free(node);
   544→}
   545→```
   546→
   547→Rust requires `unsafe` here because `prev` and `next` are raw pointers (`*mut Node`), not references (`&mut Node`). The borrow checker **refuses to verify** raw pointer dereferences because it cannot apply its local rules to them.
   548→
   549→But why can't we use references? Consider:
   550→
   551→```rust
   552→fn remove(node: &mut Node) {
   553→    let prev = &mut *node.prev;      // ERROR: cannot borrow *node.prev as mutable
   554→    let next = &mut *node.next;      // ERROR: cannot borrow *node.next as mutable
   555→    prev.next = next;
   556→    next.prev = prev;
   557→}
   558→```
   559→
   560→This fails for two reasons:
   561→
   562→1. **Self-referential borrows:** `node.prev` is an address stored in `node`. To create a reference to `*node.prev`, we need to read `node.prev` (which borrows `node`) while simultaneously creating a mutable reference to `*node.prev` (which borrows the previous node). The borrow checker sees two mutable borrows derived from overlapping data.
   563→
   564→2. **Aliasing in the single-element case:** When the list has one element, `node.prev == node.next == sentinel`. Then `prev` and `next` would be two mutable references to the same data (the sentinel). Rust's alias rule forbids this categorically — it has no mechanism to determine that the two mutable references target **different fields** of the same struct and are used **sequentially**.
   565→
   566→### 4.3 What Rust Cannot See
   567→
   568→Rust's analysis is **field-insensitive** at the borrow level. When it sees `&mut *sentinel`, it considers the borrow to cover the entire `sentinel` struct, not just the specific field being accessed. This is a deliberate design choice: field-sensitive borrow checking would dramatically increase the complexity of the analysis and is not included in Rust's borrow checker.
   569→
   570→More fundamentally, Rust's analysis is **local**: it examines each function in isolation (plus any lifetime annotations). It does not perform whole-program analysis to determine that:
   571→- The sentinel is the only node where `prev` and `next` could point to the same node.
   572→- Even in that case, the writes target different fields.
   573→- The writes are sequential and do not conflict.
   574→
   575→These facts require **global reasoning** — reasoning about the entire data structure and its invariants — which is beyond the scope of Rust's borrow checker by design.
   576→
   577→### 4.4 What VUMA Can See
   578→
   579→VUMA's IVE performs **global, field-sensitive, value-aware** analysis:
   580→
   581→1. **Field-sensitive:** The IVE tracks which field of a struct is being accessed. It knows that `(*prev).next` writes to offset 8 and `(*next).prev` writes to offset 0. Even when `prev == next` (sentinel case), these are non-overlapping accesses.
   582→
   583→2. **Value-aware:** The IVE tracks the actual values of pointers. It can determine that `prev == next` only in the single-element case, and that in all other cases the pointers are distinct. This enables it to prove exclusivity in both scenarios.
   584→
   585→3. **Global:** The IVE reasons about the entire program's memory state. It can prove that the structural invariant (`n.next.prev = n` and `n.prev.next = n`) is maintained across all operations, which allows it to verify liveness, exclusivity, and cleanup across function boundaries.
   586→
   587→4. **Sequentiality-aware:** The IVE understands that two mutable accesses to the same allocation are safe if they are sequential and non-overlapping. Rust's borrow checker conflates "multiple mutable borrows exist in the same scope" with "multiple mutable borrows are used simultaneously," rejecting code where the borrows are used sequentially.
   588→
   589→### 4.5 The Fundamental Trade-off
   590→
   591→Rust's design prioritizes **predictability and compilation speed**. The borrow checker's local rules are simple enough to explain to humans and fast enough to run on every compilation. The cost is that some safe programs (like doubly-linked lists) require `unsafe`.
   592→
   593→VUMA's design prioritizes **expressiveness and verification completeness**. The IVE's global analysis can verify a strictly larger class of safe programs, including all programs that Rust can verify plus programs that Rust rejects (like doubly-linked lists). The cost is that verification is more computationally expensive and less predictable for humans — but since the primary consumer is an AI agent, not a human, this cost is acceptable.
   594→
   595→The doubly-linked list proof above demonstrates this trade-off concretely: every operation satisfies all five VUMA invariants using raw, unrestricted pointer access, with no `unsafe` blocks, no borrow checker, and no runtime overhead. The safety is established entirely by the IVE's global reasoning.

   615→## 6. Appendix: Formal Definitions of VUMA Invariants
   616→
   617→For reference, the five VUMA invariants as defined in *Beyond Human Syntax* §3.6.2:
   618→
   619→1. **Liveness Invariant:** ∀ access `*(a)` at program point `p`, `live(a)` holds at `p`. That is, the address `a` falls within a region that is allocated (not freed) at the point of access.
   620→
   621→2. **Exclusivity Invariant:** ∀ pair of concurrent writes `*(a₁)` and `*(a₂)` at program point `p`, the regions `[a₁, a₁ + s₁)` and `[a₂, a₂ + s₂)` do not overlap. That is, no two simultaneous mutable accesses target overlapping memory.
   622→
   623→3. **Interpretation Invariant:** ∀ access `*(a)` at program point `p`, the RepD of the operation matches the RepD of the allocated region at `a`. That is, the bytes are interpreted according to a valid representation descriptor.
   624→
   625→4. **Origin Invariant:** ∀ address `a` used at program point `p`, there exists a derivation chain `a₀ → a₁ → ... → aₙ = a` where `a₀` is the return value of an `allocate()` call and each derivation step is either a field read (offset within an allocated region) or a pointer arithmetic operation that stays within bounds.
   626→
   627→5. **Cleanup Invariant:** ∀ `allocate(r)` at program point `p`, there exists a `free(r)` at some program point `q` reachable from `p`, and no access to `r` occurs after `q`. That is, every allocation is eventually freed, and no freed region is accessed.
