     3‚Üí## Task 3-d: Fix FP conversion casts in the ARM32 backend
     4‚Üí
     5‚Üí**Date**: 2026-03-05
     6‚Üí**File**: `src/codegen/src/arm32/mod.rs`
     7‚Üí
     8‚Üí### Bug Description
     9‚Üí
    10‚ÜíAll FP conversion casts in the ARM32 backend were broken. The two existing VCVT encoding functions (`encode_vcvt_f32_s32` and `encode_vcvt_s32_f32`) had **incorrect bit patterns**, causing them to emit garbage instructions instead of valid VCVT operations. Additionally:
    11‚Üí
    12‚Üí1. `UIntToFloat` incorrectly used `VCVT.F32.S32` (signed) instead of `VCVT.F32.U32` (unsigned).
    13‚Üí2. `FloatToUInt` incorrectly used `VCVT.S32.F32` (signed) instead of `VCVT.U32.F32` (unsigned).
    14‚Üí3. `FloatToFloat` was a complete no-op, not emitting any conversion instruction.
    15‚Üí
    16‚Üí### Root Cause
    17‚Üí
    18‚Üí#### Wrong VCVT Encoding Bit Patterns
    19‚Üí
    20‚ÜíThe original `encode_vcvt_f32_s32` function had the following encoding errors:
    21‚Üí
    22‚Üí| Bit Position | Old (wrong)   | Correct       | Field                |
    23‚Üí|--------------|---------------|---------------|----------------------|
    24‚Üí| [23]         | 0             | 1             | Always 1 for VCVT   |
    25‚Üí| [19:16]      | 0110 (0b011<<17) | 1000       | Opcode: int‚Üífloat   |
    26‚Üí| [8]          | Set via 1010  | 0             | sz = 0 (f32)        |
    27‚Üí| [7]          | Set via 01    | 0             | Signed               |
    28‚Üí
    29‚ÜíThe old encoding produced `0xEE360A40` for VCVT.F32.S32 S0,S0, but the correct encoding is `0xEEB80A40`. Similarly, `encode_vcvt_s32_f32` produced `0xEE360BC0` instead of the correct `0xEEBD0A40`.
    30‚Üí
    31‚ÜíThe root cause was using `0b011 << 17` for bits [19:17] (which gave 0110 at [19:16]) instead of the correct `0b1000 << 16` for bits [19:16], and missing the `1 << 23` bit.
    32‚Üí
    33‚Üí#### Signed/Unsigned Not Distinguished
    34‚Üí
    35‚ÜíThe old code grouped `IntToFloat | UIntToFloat` and `FloatToInt | FloatToUInt` into single match arms, always using the signed VCVT variant regardless of the actual cast kind. The ARM VFP encoding distinguishes signed vs unsigned via bit [7]: 0 = signed, 1 = unsigned.
    36‚Üí
    37‚Üí#### FloatToFloat Not Implemented
    38‚Üí
    39‚ÜíThe `FloatToFloat` case was a no-op with a comment saying "would need double-precision VCVT instructions". ARM32 VFP provides `VCVT.F64.F32` (f32‚Üíf64 promote) and `VCVT.F32.F64` (f64‚Üíf32 demote) with a different opcode at [19:16] = 0110 and the `sz` bit [8] indicating the destination precision.
    40‚Üí
    41‚Üí### ARM32 VFP VCVT Encoding Reference
    42‚Üí
    43‚ÜíAll VCVT variants share a common encoding base:
    44‚Üí
    45‚Üí```
    46‚Üícond 1110 1D11 xxxx Vd 101 x xx 1 M 0 Vm
    47‚Üí```
    48‚Üí
    49‚Üí| Variant         | [19:16] | [8] sz | [7] unsigned | Description              |
    50‚Üí|-----------------|---------|--------|--------------|--------------------------|
    51‚Üí| VCVT.F32.S32    | 1000    | 0      | 0            | signed i32 ‚Üí f32        |
    52‚Üí| VCVT.F32.U32    | 1000    | 0      | 1            | unsigned i32 ‚Üí f32      |
    53‚Üí| VCVT.S32.F32    | 1101    | 0      | 0            | f32 ‚Üí signed i32        |
    54‚Üí| VCVT.U32.F32    | 1101    | 0      | 1            | f32 ‚Üí unsigned i32      |
    55‚Üí| VCVT.F64.F32    | 0110    | 1      | 0            | f32 ‚Üí f64 (promote)     |
    56‚Üí| VCVT.F32.F64    | 0110    | 0      | 0            | f64 ‚Üí f32 (demote)      |
    57‚Üí
    58‚ÜíVerified reference encodings (S0,S0 or D0,S0 or S0,D0):
    59‚Üí- VCVT.F32.S32 S0, S0 = `0xEEB80A40`
    60‚Üí- VCVT.S32.F32 S0, S0 = `0xEEBD0A40`
    61‚Üí
    62‚Üí### Fixes Applied
    63‚Üí
    64‚Üí#### Fix 1: Corrected `encode_vcvt_f32_s32` encoding
    65‚Üí
    66‚ÜíChanged the bit pattern to use:
    67‚Üí- `(1 << 23)` ‚Äî required for VCVT encoding
    68‚Üí- `(0b1000 << 16)` ‚Äî correct opcode for int‚Üífloat
    69‚Üí- `(0b101 << 9)` + `(0 << 8)` ‚Äî sz=0 for f32, explicit
    70‚Üí- `(0 << 7)` ‚Äî signed
    71‚Üí- `(1 << 6)` ‚Äî required constant bit
    72‚Üí- Added `sm` parameter (was hardcoded to sd)
    73‚Üí
    74‚Üí#### Fix 2: Corrected `encode_vcvt_s32_f32` encoding
    75‚Üí
    76‚ÜíSame structural fix with `(0b1101 << 16)` for float‚Üíint opcode.
    77‚Üí
    78‚Üí#### Fix 3: Added `encode_vcvt_f32_u32` function
    79‚Üí
    80‚ÜíUnsigned variant of int‚Üífloat with bit [7] = 1.
    81‚Üí
    82‚Üí#### Fix 4: Added `encode_vcvt_u32_f32` function
    83‚Üí
    84‚ÜíUnsigned variant of float‚Üíint with bit [7] = 1.
    85‚Üí
    86‚Üí#### Fix 5: Added `encode_vcvt_f64_f32` function
    87‚Üí
    88‚Üíf32‚Üíf64 promotion with opcode [19:16] = 0110, sz [8] = 1.
    89‚Üí
    90‚Üí#### Fix 6: Added `encode_vcvt_f32_f64` function
    91‚Üí
    92‚Üíf64‚Üíf32 demotion with opcode [19:16] = 0110, sz [8] = 0.
    93‚Üí
    94‚Üí#### Fix 7: Added `encode_vldr_d` and `encode_vstr_d` functions
    95‚Üí
    96‚ÜíDouble-precision VLDR/VSTR using CP11 (bits [11:8] = 1011) instead of CP10 (1010) used by single-precision variants. Needed for f64 stack load/store in FloatToFloat handling.
    97‚Üí
    98‚Üí#### Fix 8: Separated CastKind match arms
    99‚Üí
   100‚ÜíSplit the `IntToFloat | UIntToFloat` arm into separate `IntToFloat` and `UIntToFloat` arms, using `encode_vcvt_f32_s32` and `encode_vcvt_f32_u32` respectively. Similarly split `FloatToInt | FloatToUInt`.
   101‚Üí
   102‚Üí#### Fix 9: Implemented `FloatToFloat` handling
   103‚Üí
   104‚ÜíUses `from_ty` and `to_ty` (previously ignored with `..`) to determine direction:
   105‚Üí- f32‚Üíf64: emit VCVT.F64.F32 with VLDR S0 ‚Üí VCVT ‚Üí VSTR D0
   106‚Üí- f64‚Üíf32: emit VCVT.F32.F64 with VLDR D0 ‚Üí VCVT ‚Üí VSTR S0
   107‚Üí- f32‚Üíf32 or unknown: no-op
   108‚Üí
   109‚Üí#### Fix 10: Added VCVT instruction variants to `Instruction` enum
   110‚Üí
   111‚ÜíAdded six new variants with `encode()`, `mnemonic()`, and `Display` implementations:
   112‚Üí- `VcvtF32S32 { sd, sm, cond }`
   113‚Üí- `VcvtF32U32 { sd, sm, cond }`
   114‚Üí- `VcvtS32F32 { sd, sm, cond }`
   115‚Üí- `VcvtU32F32 { sd, sm, cond }`
   116‚Üí- `VcvtF64F32 { dd, sm, cond }`

   191‚Üí

   198‚Üí## Task 2-f: Fix ARM32 calling convention for functions with >4 arguments
   199‚Üí
   200‚Üí**Date**: 2026-03-05
   201‚Üí**File**: `src/codegen/src/arm32/mod.rs`
   202‚Üí
   203‚Üí### Bug Description
   204‚Üí
   205‚ÜíThe ARM32 backend silently dropped function arguments beyond the first 4. Under AAPCS, the first 4 arguments are passed in registers R0‚ÄìR3, while arguments 5+ must be passed on the stack. The code at line 3594 had a `TODO: handle > 4 args via stack` comment, and arguments at index ‚â•4 were simply ignored in both the caller (call emission) and callee (prologue/argument setup).
   206‚Üí
   207‚Üí### Root Cause
   208‚Üí
   209‚ÜíThree related gaps in the ARM32 codegen:
   210‚Üí
   211‚Üí1. **Call emission** (`IRInstr::Call`): Only loaded args 0‚Äì3 into R0‚ÄìR3; args ‚â•4 were skipped entirely.
   212‚Üí2. **Function prologue**: Only stored R0‚ÄìR3 params to their vreg stack slots; stack-passed params (index ‚â•4) were never loaded from the stack into their vreg slots.
   213‚Üí3. **Missing helper**: No `emit_add_sp` counterpart to `emit_sub_sp`, needed for caller-side stack cleanup after calls with stack-passed args.
   214‚Üí
   215‚Üí### AAPCS Stack Layout
   216‚Üí
   217‚ÜíAfter the prologue (SUB SP, ADD R11=SP+frame_size, save LR/R11):
   218‚Üí```
   219‚ÜíHigher addresses
   220‚Üí  ‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê
   221‚Üí  ‚îÇ arg N-1      ‚îÇ  ‚Üê [R11 + 8 + (N-5)*4]
   222‚Üí  ‚îÇ ...          ‚îÇ
   223‚Üí  ‚îÇ arg 5        ‚îÇ  ‚Üê [R11 + 12]
   224‚Üí  ‚îÇ arg 4        ‚îÇ  ‚Üê [R11 + 8]
   225‚Üí  ‚îÇ saved LR     ‚îÇ  ‚Üê [R11 + 4]
   226‚Üí  ‚îÇ saved R11    ‚îÇ  ‚Üê [R11 + 0]
   227‚Üí  ‚îú‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚î§  ‚Üê R11 (frame pointer)
   228‚Üí  ‚îÇ vreg slots   ‚îÇ  ‚Üê [R11 - offset]
   229‚Üí  ‚îÇ ...          ‚îÇ
   230‚Üí  ‚îú‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚î§  ‚Üê SP
   231‚ÜíLower addresses
   232‚Üí```
   233‚Üí
   234‚ÜíStack-passed arg at 0-based index `i` (i ‚â• 4) is at `[R11 + 8 + (i - 4) * 4]`.
   235‚Üí
   236‚Üí### Fixes Applied
   237‚Üí
   238‚Üí#### Fix 1: Added `ss_load_from_r11_plus` helper (after line 3300)
   239‚Üí
   240‚ÜíNew closure that loads a 32-bit word from `[R11 + positive_offset]` into a destination register. Handles both the common case (offset ‚â§ 4095, single LDR instruction) and the rare large-offset case (compute address into R12 via ADD, then LDR indirect). This is the positive-offset counterpart to `ss_load_from_slot` which uses negative offsets.
   241‚Üí
   242‚Üí#### Fix 2: Added `emit_add_sp` helper (after line 3165)
   243‚Üí
   244‚ÜíNew closure that emits `ADD SP, SP, #imm` with the same large-value handling as the existing `emit_sub_sp` ‚Äî tries ARM rotated-immediate encoding first, falls back to loading the value into R12 and using `ADD SP, SP, R12`.
   245‚Üí
   246‚Üí#### Fix 3: Updated function prologue ‚Äî stack-passed args (lines 3444‚Äì3481)
   247‚Üí
   248‚ÜíExtended the parameter-storing loop to handle index ‚â•4. For each stack-passed param:
   249‚Üí1. Load the value from `[R11 + 8 + (i-4)*4]` into R0 using `ss_load_from_r11_plus`
   250‚Üí2. Store R0 to the param's vreg stack slot using `ss_store_to_slot`
   251‚Üí
   252‚ÜíUses R0 (not R12) as the scratch register because `ss_store_to_slot` documents that `src_reg` must NOT be R12 for large offsets (it uses R12 internally). R0 is safe because it was already saved for param 0 earlier in the loop.
   253‚Üí
   254‚Üí#### Fix 4: Updated call emission ‚Äî stack-passed args (lines 3862‚Äì3941)
   255‚Üí
   256‚ÜíCompletely rewrote the `IRInstr::Call` handler to support >4 arguments:
   257‚Üí
   258‚Üí1. **Allocate stack space**: `SUB SP, SP, #(num_stack_args * 4)` before setting up any args
   259‚Üí2. **Store args 5+ to stack**: For each arg at index ‚â•4:
   260‚Üí   - Normal case (offset ‚â§ 4095): Load arg into R12, `STR R12, [SP, #offset]`
   261‚Üí   - Large-offset case: Compute `SP + offset` into R12, load arg into R0, `STR R0, [R12, #0]`
   262‚Üí3. **Load args 0‚Äì3 into R0‚ÄìR3** (existing behavior, unchanged)
   263‚Üí4. **BL** (existing, unchanged)
   264‚Üí5. **Caller cleanup**: `ADD SP, SP, #(num_stack_args * 4)` after the call returns
   265‚Üí6. **Store return value** (existing, unchanged)
   266‚Üí
   267‚ÜíThe order (stack args first, then register args) ensures R12 is free for scratch use when storing stack args, and R0‚ÄìR3 are loaded last just before the BL.
   268‚Üí
   269‚Üí### Argument Access in Function Body
   270‚Üí
   271‚ÜíNo changes needed. The function body accesses all parameters (including stack-passed ones) through `vreg_stack_slots` using `ss_load_value`. The prologue is the bridge that copies stack-passed args from their AAPCS locations `[R11 + 8 + ...]` into the same vreg stack slot mechanism used by all other variables. After the prologue runs, the function body treats all params identically regardless of how they were passed.
   272‚Üí
   273‚Üí### Backward Compatibility
   274‚Üí
   275‚ÜíFunctions with ‚â§4 arguments are completely unaffected ‚Äî `num_stack_args` evaluates to 0, so no SUB SP, stack stores, or ADD SP instructions are emitted. The prologue loop only enters the `else` branch for params at index ‚â•4.

   277‚Üí## Task 2-c: Fix RISC-V 64 atomic CAS loop ‚Äî branch labels never inserted into label_offsets
   278‚Üí
   279‚Üí**Date**: 2026-03-05
   280‚Üí**File**: `src/codegen/src/riscv64.rs`
   281‚Üí
   282‚Üí### Bug Description
   283‚Üí
   284‚ÜíThe RISC-V 64 backend's atomic operations (AtomicStore and AtomicCas) had broken branch label handling. The code generated labels like `__atomic_store_retry_{offset}` and branch fixups referencing those labels, but never inserted the labels into the `label_offsets` HashMap. When Phase 4 (branch fixup) ran, it looked up each branch's `target_label` in `label_offsets`, got `None`, and silently skipped the fixup ‚Äî leaving all branch offsets at 0. This meant:
   285‚Üí
   286‚Üí- **AtomicStore**: SC.D failure was never retried; BNE with offset 0 branched to the next instruction (a no-op fall-through).
   287‚Üí- **AtomicCas**: No branch fixups were created at all (only a `TODO` comment). Both BNE instructions (not-equal‚Üídone and SC-failed‚Üíretry) kept offset 0, so the CAS loop never looped back on SC.D failure and never skipped to done on mismatch.
   288‚Üí
   289‚Üí### Additional Bug in AtomicStore
   290‚Üí
   291‚ÜíThe original AtomicStore code emitted `SC.D` without a preceding `LR.D`. Per the RISC-V A extension, SC.D requires a valid reservation established by a prior LR.D; without it, SC.D always reports failure (rd=1), making the retry loop infinite even if branching worked.
   292‚Üí
   293‚Üí### Root Cause Analysis
   294‚Üí
   295‚ÜíThe `label_offsets` HashMap is populated in two places:
   296‚Üí1. **Block labels** (line 4164): `label_offsets.insert(block.label.clone(), current_byte_offset)` ‚Äî works correctly for IR block labels.
   297‚Üí2. **Atomic labels**: Never inserted ‚Äî the code created label name strings and `BranchFixup` entries referencing them, but never called `label_offsets.insert(...)`.
   298‚Üí
   299‚Üí### Fixes Applied
   300‚Üí
   301‚Üí#### Fix 1: AtomicStore (lines 4659‚Äì4691)
   302‚Üí
   303‚Üí- **Added LR.D before SC.D** in the retry loop to properly establish a memory reservation. The loop is now: `LR.D T2, (T0)` ‚Üí `SC.D T2, T1, (T0)` ‚Üí `BNE T2, zero, retry`.
   304‚Üí- **Inserted the retry label into `label_offsets`** at the absolute byte offset of the LR.D instruction: `label_offsets.insert(retry_label.clone(), retry_abs_offset)`.
   305‚Üí- **Removed stale TODO comments** about "proper label support for atomic loops" since this IS the proper fix.
   306‚Üí- Cleaned up variable naming: `retry_offset` ‚Üí `retry_abs_offset` and `retry_label` for clarity.
   307‚Üí
   308‚Üí#### Fix 2: AtomicCas (lines 4693‚Äì4752)
   309‚Üí
   310‚Üí- **Inserted the retry label** (`__atomic_cas_retry_{offset}`) into `label_offsets` at the absolute byte offset of the LR.D instruction.
   311‚Üí- **Inserted the done label** (`__atomic_cas_done_{offset}`) into `label_offsets` at the absolute byte offset after the second BNE (where the dst store begins).
   312‚Üí- **Added branch fixup for BNE1** (current != expected ‚Üí done): Creates a `BranchFixup` targeting the `done_label` with `bne_rs1=T2, bne_rs2=T1`.
   313‚Üí- **Added branch fixup for BNE2** (SC.D failed ‚Üí retry): Creates a `BranchFixup` targeting the `retry_label` with `bne_rs1=T4, bne_rs2=Zero`.
   314‚Üí- **Removed stale TODO comment** about "proper label fixups for the retry/done branches".
   315‚Üí
   316‚Üí### How the Fix Works
   317‚Üí
   318‚ÜíBoth fixes follow the same pattern used by IR-level branches (e.g., `CondBranch`):
   319‚Üí
   320‚Üí1. Compute the absolute byte offset of the label position using `current_byte_offset + code.len() as u64`.
   321‚Üí2. Create a unique label name (e.g., `__atomic_cas_retry_128`) incorporating the absolute offset for uniqueness.
   322‚Üí3. Insert the label into `label_offsets` immediately ‚Äî no need to wait, since `label_offsets` is just a HashMap consulted later in Phase 4.
   323‚Üí4. Create `BranchFixup` entries referencing the label name.
   324‚Üí5. Phase 4's existing fixup loop resolves all labels correctly, computing `rel_offset = target_offset - branch_abs_offset` and patching the encoded BNE/JAL instructions.
   325‚Üí
   326‚Üí### Instruction Sequences After Fix
   327‚Üí
   328‚Üí**AtomicStore**:
   329‚Üí```
   330‚Üíretry:  LR.D   T2, (T0)       ; establish reservation
   331‚Üí        SC.D   T2, T1, (T0)   ; attempt store; T2=0 success, T2=1 failure
   332‚Üí        BNE    T2, zero, retry ; if failed, branch back to LR.D
   333‚Üí```
   334‚Üí
   335‚Üí**AtomicCas**:
   336‚Üí```
   337‚Üíretry:  LR.D   T2, (T0)       ; load current value & establish reservation
   338‚Üí        BNE    T2, T1, done   ; if current != expected, skip to done
   339‚Üí        SC.D   T4, T3, (T0)   ; attempt to store desired; T4=0 success, T4=1 failure
   340‚Üí        BNE    T4, zero, retry; if SC failed, branch back to LR.D
   341‚Üídone:   <store T2 to dst>     ; store the old (current) value to destination
   342‚Üí```

   350‚Üí

   354‚Üí

   431‚Üí### Key Design Decisions
   432‚Üí
   433‚Üí1. **No LL/SC loop needed**: Unlike AArch64 and RISC-V which require explicit load-linked/store-conditional loops, Wasm's `cmpxchg` instruction handles the retry loop internally. This is a major simplification.
   434‚Üí
   435‚Üí2. **Natural alignment enforced**: All atomic instructions use naturally aligned addresses (align = log2(access_width)). Misaligned atomic accesses trap in Wasm.
   436‚Üí
   437‚Üí3. **Sub-word support**: 8-bit and 16-bit atomic CAS is supported via the `rmw8.cmpxchg_u` and `rmw16.cmpxchg_u` instructions, which zero-extend the result to the Wasm stack type (i32 or i64).
   438‚Üí
   439‚Üí4. **Shared memory required**: The Wasm spec mandates that atomic instructions can only operate on shared memory. The production module now emits `shared: true` in its memory definition.

   442‚Üí
   443‚Üí**Date**: 2026-03-05
   444‚Üí**Files**: `src/codegen/src/mips64/mod.rs`, `src/codegen/src/loongarch64/mod.rs`, `src/codegen/src/loongarch64/stack_slot_isel.rs`, `src/codegen/src/loongarch64/reg_alloc_isel.rs`

   446‚Üí### Bug 1: MIPS64 Ror/Rol Incomplete Implementation
   447‚Üí
   448‚Üí#### Bug Description
   449‚Üí
   450‚ÜíThe MIPS64 backend had two separate ROR/ROL bugs:
   451‚Üí
   452‚Üí1. **`lower_binop` function (register-allocated path, dead code)**: ROR only emitted `DSRLV` and ROL only emitted `DSLLV` ‚Äî each was missing the complementary shift + OR to complete the rotation. For example, ROR should produce `(n >> r) | (n << (64-r))` but only produced `(n >> r)`.
   453‚Üí
   454‚Üí2. **Stack-slot ISel path (active code generation)**: ROR and ROL were hardcoded to 32-bit rotation (using `imm: 32` in ADDIU), ignoring the `is_32bit` flag. 64-bit types (I64/U64) would produce incorrect results because they need a 64-bit rotation width, and they don't need the zero-extend/mask-to-32-bits steps.
   455‚Üí
   456‚Üí#### Root Cause
   457‚Üí
   458‚ÜíThe `lower_binop` function had comments acknowledging the issue: "Simplified: just emit dsrlv as placeholder (lower_ir_instr is dead code)". The stack-slot ISel path was implemented only for the I32/U32 case and never updated for 64-bit types.
   459‚Üí
   460‚Üí#### Fixes Applied
   461‚Üí
   462‚Üí##### Fix 1: `lower_binop` ‚Äî Complete ROR/ROL sequences (lines ~1831‚Äì1922)
   463‚Üí
   464‚ÜíReplaced the single-instruction placeholder with proper multi-instruction sequences, following the same pattern as `SLe` (push to `result`, return early):
   465‚Üí
   466‚Üí**ROR (64-bit)**: `(n >> r) | (n << (64-r))`
   467‚Üí```
   468‚ÜíDSRLV T2, lhs, rhs       // T2 = lhs >> rhs
   469‚ÜíDADDIU T3, $zero, 64     // T3 = 64
   470‚ÜíDSUBU  T3, T3, rhs       // T3 = 64 - rhs
   471‚ÜíDSLLV  T3, lhs, T3       // T3 = lhs << (64-rhs)
   472‚ÜíOR     dst, T2, T3       // dst = (lhs >> rhs) | (lhs << (64-rhs))
   473‚Üí```
   474‚Üí
   475‚Üí**ROL (64-bit)**: `(n << r) | (n >> (64-r))`
   476‚Üí```
   477‚ÜíDSLLV  T2, lhs, rhs      // T2 = lhs << rhs
   478‚ÜíDADDIU T3, $zero, 64     // T3 = 64
   479‚ÜíDSUBU  T3, T3, rhs       // T3 = 64 - rhs
   480‚ÜíDSRLV  T3, lhs, T3       // T3 = lhs >> (64-rhs)
   481‚ÜíOR     dst, T2, T3       // dst = (lhs << rhs) | (lhs >> (64-rhs))
   482‚Üí```
   483‚Üí
   484‚ÜíUses T2 and T3 as scratch registers (caller-saved temporaries, safe to clobber).
   485‚Üí
   486‚Üí##### Fix 2: Stack-slot ISel ‚Äî 32-bit/64-bit dispatch (lines ~2536‚Äì2589)
   487‚Üí
   488‚ÜíAdded `is_32bit` check to ROR/ROL:
   489‚Üí
   490‚Üí- **32-bit path** (existing): Zero-extend value, rotate with width 32, mask result to 32 bits
   491‚Üí- **64-bit path** (new): Direct DSRLV/DSLLV + DADDIU(64) + DSUBU + DSLLV/DSRLV + OR sequence, no zero-extend or mask needed

   495‚Üí### Bug 2: LoongArch64 Unimplemented Terminators
   496‚Üí
   497‚Üí#### Bug Description
   498‚Üí
   499‚ÜíThe LoongArch64 backend's Switch, Invoke, TailCall, and Resume terminators all emitted a `BREAK` instruction (intentional crash), making any IR using these terminators immediately trap at runtime. Both `stack_slot_isel.rs` and `reg_alloc_isel.rs` had this issue.
   500‚Üí
   501‚ÜíAdditionally, the vreg collection phase in both files only tracked vregs used by `Branch` and `Return` terminators, missing vregs referenced by Switch (discr), Invoke (args), TailCall (args), and Resume (value). This could cause those vregs to not get stack slots, leading to incorrect code.
   502‚Üí
   503‚Üí#### Root Cause
   504‚Üí
   505‚ÜíThese terminators were never implemented ‚Äî the code had placeholder `BREAK` emissions from initial development.
   506‚Üí
   507‚Üí#### Fixes Applied
   508‚Üí
   509‚Üí##### Fix 0: Vreg collection (both stack_slot_isel.rs and reg_alloc_isel.rs)
   510‚Üí
   511‚ÜíExtended the terminator vreg tracking `match` to include:
   512‚Üí- `Switch { discr, .. }` ‚Üí track `discr` register
   513‚Üí- `Invoke { args, .. }` ‚Üí track all `args` registers
   514‚Üí- `TailCall { args, .. }` ‚Üí track all `args` registers
   515‚Üí- `Resume { value }` ‚Üí track `value` register
   516‚Üí
   517‚Üí##### Fix 1: Switch ‚Äî Cascade of BEQ comparisons (both isel files)
   518‚Üí
   519‚Üí```loongarch
   520‚Üí# Load discriminator into S0/T0
   521‚Üíld.d   S0, fp, discr_offset      # (or read_vreg for reg_alloc_isel)
   522‚Üí# For each (value, target):
   523‚Üílu12i.w S1, value_hi              # load comparison value
   524‚Üíori    S1, S1, value_lo
   525‚Üíbeq    S0, S1, target_label       # branch if match (patched later)
   526‚Üí# Default:
   527‚Üíb      default_label               # unconditional branch to default
   528‚Üí```
   529‚Üí
   530‚ÜíNo jump table is used (PCADDU12I + LD.W + JR would require emitting data in the code section, which the framework doesn't support). The cascade-of-comparisons approach is correct and handles all switch sizes, though it's O(n) in the number of targets.
   531‚Üí
   532‚Üí##### Fix 2: Invoke ‚Äî Call + branch to normal (both isel files)
   533‚Üí
   534‚ÜíInvoke is implemented identically to Call for the invocation part, with an additional unconditional branch to the `normal` continuation block after the call returns:
   535‚Üí
   536‚Üí1. Flush caller-saved registers (reg_alloc_isel) / set up args from stack slots (stack_slot_isel)
   537‚Üí2. Load arguments into $a0‚Äì$a7 (+ stack-passed args if >8)
   538‚Üí3. BL to function (with R_LARCH_B26 relocation)
   539‚Üí4. Deallocate stack arg space (if any)
   540‚Üí5. Store return value to dst's stack slot / assign to $a0 in reg cache
   541‚Üí6. B normal (branch to normal continuation, patched later)
   542‚Üí
   543‚ÜíThe `unwind` block is tracked for control flow correctness (it's a successor of the Invoke block) but doesn't get explicit landing-pad setup ‚Äî that requires DWARF exception info which the framework doesn't emit. The unwind handler would be reached by the unwinder at runtime.
   544‚Üí
   545‚Üí##### Fix 3: TailCall ‚Äî Epilogue + B to target (both isel files)
   546‚Üí
   547‚ÜíTail call reuses the caller's stack frame. The implementation:
   548‚Üí
   549‚Üí1. Load register args into $a0‚Äì$a7 (while frame is still intact)
   550‚Üí2. For stack-passed args (>8): place them at old_sp offsets (above current frame, in the caller's arg area which is no longer needed)
   551‚Üí3. Epilogue: restore $ra, $fp, deallocate frame (same as Return, minus JIRL)
   552‚Üí4. B to target (with R_LARCH_B26 relocation, no return address save)
   553‚Üí
   554‚ÜíSince `B` doesn't modify `$ra`, the callee returns directly to the tail-caller's caller ‚Äî the key semantic of a tail call.
   555‚Üí
   556‚ÜíIn `stack_slot_isel.rs`, the stack args are placed by computing `old_sp = $sp + frame_size` and storing at `old_sp + (i-8)*8`. In `reg_alloc_isel.rs`, we flush all registers and load args from the reg cache before the epilogue.
   557‚Üí
   558‚Üí##### Fix 4: Resume ‚Äî Call __Unwind_Resume (both isel files)
   559‚Üí
   560‚ÜíResume transfers control to the unwinder, which never returns:
   561‚Üí
   562‚Üí1. Load exception value into $a0
   563‚Üí2. BL __Unwind_Resume (with R_LARCH_B26 relocation)
   564‚Üí3. BREAK (safety trap ‚Äî __Unwind_Resume should never return)
   565‚Üí
   566‚ÜíThis is a minimal but correct implementation. In a full EH system, the personality function and unwinder would use DWARF info to find the correct landing pad.
   567‚Üí
   568‚Üí### Bonus Fix: LoongArch64 mod.rs ROL
   569‚Üí
   570‚ÜíWhile investigating, I also noticed that the `lower_binop_la64` function in `loongarch64/mod.rs` handled both Ror and Rol identically (both using `RotrD` directly), which is incorrect for Rol since `ROL(x, n) = ROTR(x, 64-n)`. Fixed by splitting the cases:

   584‚Üí## Task 2-e: Fix AtomicCas on ARM32 and MIPS64 backends
   585‚Üí
   586‚Üí**Date**: 2026-03-05
   587‚Üí**Files**: `src/codegen/src/arm32/mod.rs`, `src/codegen/src/mips64/mod.rs`
   588‚Üí
   589‚Üí### Bug Description
   590‚Üí
   591‚ÜíBoth the ARM32 and MIPS64 backends lowered `AtomicCas` to a simple load instruction, completely ignoring the `expected` and `desired` operands. This meant that compare-and-swap operations always returned the current value at the address without ever attempting the conditional store, making atomic CAS semantically equivalent to an atomic load ‚Äî useless for any concurrent algorithm.
   592‚Üí
   593‚Üí### Root Cause
   594‚Üí
   595‚ÜíThe ARM32 handler (line 3915) had a comment `// Lower as a simple load (placeholder)` and constructed a `Load` IR instruction that it never actually used, then loaded the address and performed a plain `LDR`. The `expected` and `desired` values were explicitly suppressed with `let _ = (load_ir, expected, desired);`.
   596‚Üí
   597‚ÜíThe MIPS64 handler (line 2867) had a similar `// Placeholder: simple load` comment, using a plain `LD` instruction with `expected: _` and `desired: _` patterns to suppress unused-variable warnings.
   598‚Üí
   599‚Üí### Fixes Applied
   600‚Üí
   601‚Üí#### ARM32: LDREX/STREX-based CAS loop
   602‚Üí
   603‚ÜíAdded 7 new instruction variants to the `Instruction` enum and corresponding encoding functions:
   604‚Üí
   605‚Üí| Instruction | Encoding | Description |
   606‚Üí|-------------|----------|-------------|
   607‚Üí| `Ldrex` | `cond 00011011 Rn Rd 1111 1001 1111` | Load Register Exclusive (32-bit) |
   608‚Üí| `Ldrexb` | `cond 00011101 Rn Rd 1111 1001 1111` | Load Register Exclusive Byte |
   609‚Üí| `Ldrexh` | `cond 00011111 Rn Rd 1111 1001 1111` | Load Register Exclusive Halfword |
   610‚Üí| `Strex` | `cond 00011000 Rn Rd 1111 1001 Rt` | Store Register Exclusive (32-bit) |
   611‚Üí| `Strexb` | `cond 00011100 Rn Rd 1111 1001 Rt` | Store Register Exclusive Byte |
   612‚Üí| `Strexh` | `cond 00011110 Rn Rd 1111 1001 Rt` | Store Register Exclusive Halfword |
   613‚Üí| `Dmb` | `cond 01010111 1111 1111 1111 0101 option` | Data Memory Barrier |
   614‚Üí
   615‚ÜíAlso added `encode()`, `mnemonic()`, `Display`, and disassembler support for all new instructions.
   616‚Üí
   617‚ÜíThe CAS loop uses a fixed-size instruction sequence with precomputed branch offsets:
   618‚Üí
   619‚Üí```
   620‚Üí; Register allocation: R3=addr, R1=expected, R2=desired, R0=old/result, R12=status
   621‚Üí  DMB SY                    ; +0  ‚Äî acquire barrier
   622‚Üíretry:
   623‚Üí  LDREX{,B,H} R0, [R3]     ; +4  ‚Äî load exclusive
   624‚Üí  CMP R0, R1                ; +8  ‚Äî compare old with expected
   625‚Üí  BNE done                  ; +12 ‚Äî offset_words = +2
   626‚Üí  STREX{,B,H} R12, R2, [R3]; +16 ‚Äî try store; R12 = status
   627‚Üí  CMP R12, #0               ; +20 ‚Äî check status
   628‚Üí  BNE retry                 ; +24 ‚Äî offset_words = -7
   629‚Üídone:
   630‚Üí  DMB SY                    ; +28 ‚Äî release barrier
   631‚Üí```
   632‚Üí
   633‚ÜíBranch offset derivation (ARM formula: `offset_words = (target - (branch + 8)) / 4`):
   634‚Üí- BNE done: `(28 - (12 + 8)) / 4 = +2`
   635‚Üí- BNE retry: `(4 - (24 + 8)) / 4 = -7`
   636‚Üí
   637‚ÜíType-specific variants:
   638‚Üí- I8/U8 ‚Üí LDREXB + STREXB
   639‚Üí- I16/U16 ‚Üí LDREXH + STREXH
   640‚Üí- I32/U32/I64/U64 ‚Üí LDREX + STREX
   641‚Üí
   642‚ÜíR0 holds the old value throughout the loop because STREX writes its status to R12 (not R0), so R0 is preserved for the result store.
   643‚Üí
   644‚Üí#### MIPS64: LLD/SCD-based CAS loop
   645‚Üí
   646‚ÜíAdded 5 new instruction variants to the `Instruction` enum, 4 new opcode constants, and 1 new function code:
   647‚Üí
   648‚Üí| Constant | Value | Description |
   649‚Üí|----------|-------|-------------|
   650‚Üí| `OPC_LL` | 0x30 | Load Linked opcode |
   651‚Üí| `OPC_LLD` | 0x34 | Load Linked Doubleword opcode |
   652‚Üí| `OPC_SC` | 0x38 | Store Conditional opcode |
   653‚Üí| `OPC_SCD` | 0x3C | Store Conditional Doubleword opcode |
   654‚Üí| `FN_SYNC` | 0x0F | SYNC function code |
   655‚Üí
   656‚ÜíNew instruction variants:
   657‚Üí- `Ll { rt, base, offset }` ‚Äî Load Linked (32-bit)
   658‚Üí- `Lld { rt, base, offset }` ‚Äî Load Linked Doubleword (64-bit)
   659‚Üí- `Sc { rt, base, offset }` ‚Äî Store Conditional (32-bit)
   660‚Üí- `Scd { rt, base, offset }` ‚Äî Store Conditional Doubleword (64-bit)
   661‚Üí- `Sync { stype }` ‚Äî Memory synchronization barrier
   662‚Üí
   663‚ÜíAlso added `encode()`, `mnemonic()`, and `Display` support for all new instructions.
   664‚Üí
   665‚ÜíThe CAS loop follows the MIPS64 LL/SC pattern with delay slot NOPs:
   666‚Üí
   667‚Üí```
   668‚Üí; Register allocation: T0=addr, T1=expected, T2=desired, V0=old/result, T3=temp/status
   669‚Üí  SYNC                      ; +0  ‚Äî acquire barrier
   670‚Üíretry:
   671‚Üí  LLD/LL V0, 0(T0)         ; +4  ‚Äî load linked
   672‚Üí  NOP                       ; +8  ‚Äî load-use hazard
   673‚Üí  BNE V0, T1, done          ; +12 ‚Äî if old != expected, skip
   674‚Üí  NOP                       ; +16 ‚Äî delay slot
   675‚Üí  DADDU T3, $zero, T2       ; +20 ‚Äî copy desired to temp (SCD overwrites rt)
   676‚Üí  SCD/SC T3, 0(T0)          ; +24 ‚Äî store conditional; T3 = 1 success, 0 failure
   677‚Üí  NOP                       ; +28 ‚Äî hazard
   678‚Üí  BEQ T3, $zero, retry      ; +32 ‚Äî if failed, retry
   679‚Üí  NOP                       ; +36 ‚Äî delay slot
   680‚Üídone:
   681‚Üí  SYNC                      ; +40 ‚Äî release barrier
   682‚Üí```
   683‚Üí
   684‚ÜíBranch offset derivation (MIPS formula: `offset_bytes = target - (branch + 4)`):
   685‚Üí- BNE done: `40 - (12 + 4) = +24` bytes ‚Üí 6 words
   686‚Üí- BEQ retry: `4 - (32 + 4) = -32` bytes ‚Üí -8 words
   687‚Üí
   688‚ÜíType-specific variants:
   689‚Üí- I32/U32 ‚Üí LL + SC
   690‚Üí- I64/U64/Ptr ‚Üí LLD + SCD
   691‚Üí
   692‚ÜíThe `DADDU T3, $zero, T2` instruction copies the desired value to T3 before SCD because SCD overwrites its `rt` register with the status flag (1=success, 0=failure). Without this copy, the desired value would be lost on retry. V0 holds the old value from LLD and is never overwritten by SCD (which writes to T3), so V0 is available for the result store after the loop.
   693‚Üí
   694‚Üí### Key Design Decisions
   695‚Üí
   696‚Üí1. **Precomputed branch offsets**: Both backends use fixed-size CAS loop sequences, allowing branch offsets to be computed at compile time rather than requiring label tracking and fixup passes. This simplifies the implementation and avoids interaction with the existing branch fixup mechanisms.
   697‚Üí
   698‚Üí2. **Memory barriers**: Both implementations use full memory barriers (DMB SY on ARM32, SYNC on MIPS64) before and after the CAS loop, providing sequential consistency semantics. The acquire barrier before the loop prevents reordering of the load with prior memory operations, and the release barrier after prevents reordering of the store with subsequent operations.
   699‚Üí
   700‚Üí3. **Sub-word support (ARM32 only)**: ARM32 has dedicated LDREXB/STREXB and LDREXH/STREXH instructions for byte and halfword CAS. MIPS64 does not have sub-word LL/SC variants; the implementation uses LL/SC for 32-bit types and LLD/SCD for 64-bit types only.
   701‚Üí
   702‚Üí4. **Conservative NOPs (MIPS64)**: The MIPS64 CAS loop includes NOPs after LLD/LL (load-use hazard) and SCD/SC (conservative hazard avoidance), matching the existing code generation pattern in the backend.

   837‚Üí
   838‚Üí### LoongArch64 Atomic Instructions
   839‚Üí
   840‚ÜíReferencing the LoongArch ISA manual:
   841‚Üí
   842‚Üí- **LL.D Rd, Rj, Si14** ‚Äî Load-Linked Doubleword (64-bit, address in Rj + Si14)
   843‚Üí- **SC.D Rd, Rj, Si14** ‚Äî Store-Conditional Doubleword (stores Rd to [Rj+Si14]; Rd ‚Üê 0 on success, 1 on failure)
   844‚Üí- **LL.W / SC.W** ‚Äî 32-bit variants
   845‚Üí- **AMSWAP.D Rd, Rj, Rk** ‚Äî Atomic Memory Swap Doubleword (Rd = old [Rj], [Rj] = Rk)
   846‚Üí- **AMSWAP.W** ‚Äî 32-bit variant
   847‚Üí- **DBAR hint** ‚Äî Data Barrier (hint=0 is full barrier)
   848‚Üí
   849‚Üí### New Instructions Added
   850‚Üí
   851‚ÜíAdded to `Instruction` enum in `mod.rs`:
   852‚Üí
   853‚Üí| Instruction | Format | Opcode | Encoding |
   854‚Üí|------------|--------|--------|----------|
   855‚Üí| `AmswapW { rd, rj, rk }` | 3R | 0x00C0 (bits[31:15]) | `encode_3r(0x00C0, rk, rj, rd)` |
   856‚Üí| `AmswapD { rd, rj, rk }` | 3R | 0x00C2 (bits[31:15]) | `encode_3r(0x00C2, rk, rj, rd)` |

   858‚Üí
   859‚Üí### Atomic Lowering Implementation
   860‚Üí
   861‚Üí#### AtomicLoad (dst, addr, ty)
   862‚Üí
   863‚Üí```
   864‚Üídbar 0                    ; acquire fence
   865‚Üíll.{d,w} S0, S0, 0       ; load-linked from [S0+0]
   866‚Üídbar 0                    ; fence after load
   867‚Üí; store S0 to dst vreg slot
   868‚Üí```
   869‚Üí
   870‚ÜíUses LL.D for 64-bit types (I64/U64/Ptr/Func) and LL.W for 32-bit types (I8/I16/I32/U8/U16/U32).
   871‚Üí
   872‚Üí#### AtomicStore (value, addr, ty)
   873‚Üí
   874‚Üí```
   875‚Üídbar 0                         ; release fence
   876‚Üíamswap.{d,w} $r0, S1, S0      ; atomically store S0 to [S1], discard old value in $r0
   877‚Üídbar 0                         ; fence after store
   878‚Üí```
   879‚Üí
   880‚ÜíUses AMSWAP.D for 64-bit and AMSWAP.W for 32-bit types. The `$r0` destination discards the old value.
   881‚Üí
   882‚Üí#### AtomicCas (dst, addr, expected, desired, ty)
   883‚Üí
   884‚Üí```
   885‚Üí; S0=addr, S1=expected, S2=desired
   886‚Üídbar 0                         ; offset 0:  full fence
   887‚Üíll.{d,w} S3, S0, 0             ; offset 4:  S3 = current value at [S0]
   888‚Üíbne S3, S1, +5                 ; offset 8:  if current != expected, skip to dbar
   889‚Üíor S3, S2, $r0                  ; offset 12: S3 = desired (reload before each SC)
   890‚Üísc.{d,w} S3, S0, 0             ; offset 16: store desired; S3 = 0 (ok) / 1 (fail)
   891‚Üíbnez S3, -4                    ; offset 20: retry if SC failed
   892‚Üíor S3, S1, $r0                  ; offset 24: SC succeeded: old value = expected = S1
   893‚Üídbar 0                         ; offset 28: fence after CAS
   894‚Üí; store S3 (old value) to dst vreg slot
   895‚Üí```
   896‚Üí
   897‚ÜíKey design decisions:
   898‚Üí- The `or S3, S2, $r0` before SC.D reloads the desired value (from S2, which is never overwritten) because SC.D overwrites its rd register with the success flag.
   899‚Üí- When BNE fires (current ‚âÝ expected), S3 still holds the old value from LL.D, and we jump directly to dbar at offset 28.
   900‚Üí- When SC succeeds, S3 = 0, and the old value = expected = S1. The `or S3, S1, $r0` at offset 24 moves the expected value into S3 for a unified store path.
   901‚Üí- Branch offsets are precomputed since the instruction sequence has fixed size (8 instructions = 32 bytes).
   902‚Üí
   903‚Üí### Disassembler Updates
   904‚Üí
   905‚ÜíEnhanced `decode_loongarch64_instruction()` in `mod.rs` with multi-level opcode decoding:
   906‚Üí1. **2RI12 format** (10-bit opcode at bits[31:22]): Added DBAR (0x0E7) decode
   907‚Üí2. **3R format** (17-bit opcode at bits[31:15]): Added AMSWAP.W (0x00C0) and AMSWAP.D (0x00C2) decode

   972‚Üí## Task 3-h: Fix FP conversion casts in the Wasm32 backend
   973‚Üí
   974‚Üí**Date**: 2026-03-05
   975‚Üí**Files**: `src/codegen/src/wasm32/mod.rs`, `src/codegen/src/wasm32/disasm.rs`
   976‚Üí
   977‚Üí### Bug Description
   978‚Üí
   979‚ÜíThe Wasm32 backend's FP conversion casts (IntToFloat, UIntToFloat, FloatToInt, FloatToUInt, FloatToFloat) were functionally broken in two ways:
   980‚Üí
   981‚Üí1. **`infer_wasm_type()` always returned `I32` for registers**, making it impossible to distinguish float source values from integer ones. Since the CastKind lowering only considered the source type, FloatToInt/FloatToUInt would always select `i32.trunc_f32_s`/`i32.trunc_f32_u` (which expects an F32 on the stack) even when the source was an F64 register ‚Äî producing invalid Wasm bytecode or silently wrong results.
   982‚Üí
   983‚Üí2. **Swapped signed/unsigned opcodes**: Eight conversion instructions had their signed/unsigned opcode pairs transposed in both the encoder and the decoder, violating the WebAssembly MVP spec:
   984‚Üí
   985‚Üí| Instruction | Wrong opcode | Correct opcode |
   986‚Üí|-------------|-------------|----------------|
   987‚Üí| `I32TruncF64S` | 0xA9 | 0xAA |
   988‚Üí| `I32TruncF32U` | 0xAA | 0xA9 |
   989‚Üí| `I64TruncF64S` | 0xAF | 0xB0 |
   990‚Üí| `I64TruncF32U` | 0xB0 | 0xAF |
   991‚Üí| `F32ConvertI64S` | 0xB3 | 0xB4 |
   992‚Üí| `F32ConvertI32U` | 0xB4 | 0xB3 |
   993‚Üí| `F64ConvertI64S` | 0xB8 | 0xB9 |
   994‚Üí| `F64ConvertI32U` | 0xB9 | 0xB8 |
   995‚Üí
   996‚Üí### Root Cause
   997‚Üí
   998‚Üí1. `infer_wasm_type()` ignored the `LoweringContext::vreg_types` map entirely, returning `WasmType::I32` for all register values regardless of their actual tracked type. This meant the CastKind match arms that branch on `src_ty` (e.g., `WasmType::F64 => ‚Ä¶`) could never fire for float source registers.
   999‚Üí
  1000‚Üí2. The opcode assignments were simply swapped ‚Äî likely a copy-paste error during initial implementation. The decoder (`disasm.rs`) had the same swaps and was also missing several opcodes (0xAA, 0xAB, 0xB0, 0xB1, 0xB4, 0xB5, 0xB8, 0xB9, 0xBA).

  1075‚Üí## Task 3-e: Fix FP conversion casts in the MIPS64 backend
  1076‚Üí
  1077‚Üí**Date**: 2026-03-05
  1078‚Üí**File**: `src/codegen/src/mips64/mod.rs`
  1079‚Üí
  1080‚Üí### Bug Description
  1081‚Üí
  1082‚ÜíThe MIPS64 backend treated all FP conversion casts (IntToFloat, UIntToFloat, FloatToInt, FloatToUInt, FloatToFloat) as no-ops in the register-allocated code path.   1083‚Üí
  1084‚ÜíAdditionally, the stack-slot code path had bugs:
  1085‚Üí1. **IntToFloat** used `LDC1` (64-bit load) instead of `MTC1` (32-bit GPR‚ÜíFPR move), causing incorrect data placement on big-endian MIPS since `CVT.D.W` reads the low 32 bits of the FPR.

  1087‚Üí3. Missing instruction variants for i64‚ÜîFP conversions (CVT.S.L, CVT.D.L, CVT.L.S, CVT.L.D).
  1088‚Üí4. Missing MTC1/MFC1/DMTC1/DMFC1 instructions needed for efficient GPR‚ÜîFPR transfers.
  1089‚Üí
  1090‚Üí### Root Cause
  1091‚Üí
  1092‚Üí1. **Register-allocated path** (`lower_ir_instr`): The `CastKind` discriminant was discarded with `..`, and all casts were treated as integer moves.
  1093‚Üí2. **Stack-slot path**: Used a roundabout stack-based transfer (SD‚ÜíLDC1‚ÜíCVT‚ÜíSDC1‚ÜíLD) with endianness bugs, instead of the direct MTC1/DMTC1 approach.

  1171‚Üí## Task 3-c: Fix FP conversion casts in the RISC-V 64 backend
  1172‚Üí
  1173‚Üí**Date**: 2026-03-05
  1174‚Üí**Files**: `src/codegen/src/ir.rs`, `src/codegen/src/riscv64.rs`, `src/codegen/src/scg_to_ir.rs`, `src/pipeline.rs`, `src/codegen/src/opt.rs`, `src/codegen/src/regalloc.rs`, `src/codegen/src/arm64.rs`, `src/codegen/src/emit.rs`, `src/codegen/src/arm32/mod.rs`, `src/codegen/src/ppc64/mod.rs`, `src/codegen/src/x86_64/mod.rs`, `src/codegen/src/x86_64/stack_slot_isel.rs`, `src/codegen/src/wasm32/mod.rs`, `src/codegen/src/mips64/mod.rs`
  1175‚Üí
  1176‚Üí### Bug Description
  1177‚Üí
  1178‚ÜíThe RISC-V 64 backend's `CastKind` handling for floating-point conversions only handled one type combination per cast kind (always the 64-bit variant), making conversions like `i32‚Üíf32`, `f32‚Üíi32`, `f32‚Üíf64`, etc. incorrect or impossible. Specifically:
  1179‚Üí
  1180‚Üí- `IntToFloat` always emitted `FCVT.D.L` (i64‚Üíf64), ignoring `FCVT.S.W` (i32‚Üíf32), `FCVT.S.L` (i64‚Üíf32), `FCVT.D.W` (i32‚Üíf64)
  1181‚Üí- `UIntToFloat` always emitted `FCVT.D.LU` (u64‚Üíf64), ignoring unsigned 32-bit and f32 variants
  1182‚Üí- `FloatToInt` always emitted `FCVT.L.D` (f64‚Üíi64), ignoring f32 source and i32 destination variants
  1183‚Üí- `FloatToUInt` always emitted `FCVT.LU.D` (f64‚Üíu64), ignoring f32 source and u32 destination variants
  1184‚Üí- `FloatToFloat` only handled `f64‚Üíf32` (narrowing), missing `f32‚Üíf64` (widening)
  1185‚Üí
  1186‚Üí### Root Cause
  1187‚Üí
  1188‚ÜíThe `IRInstr::Cast` instruction variant did not carry source/destination type information, so the RISC-V backend had no way to determine which specific FCVT instruction to emit. The ARM64 backend doesn't have this problem because ARM64's `SCVTF`/`UCVTF`/`FCVTZS`/`FCVTZU` handle different sizes via register width.
  1189‚Üí

  1264‚Üí## Task 3-a: Fix FP conversion casts in the x86_64 backend
  1265‚Üí
  1266‚Üí**Date**: 2026-03-05
  1267‚Üí**Files**: `src/codegen/src/x86_64/stack_slot_isel.rs`, `src/codegen/src/x86_64/mod.rs`, `src/codegen/src/arm32/mod.rs`
  1268‚Üí
  1269‚Üí### Bug Description
  1270‚Üí
  1271‚ÜíThe x86_64 backend's `CastKind` handling in `stack_slot_isel.rs` ignored the `from_ty` and `to_ty` fields on `IRInstr::Cast` and hardcoded specific type combinations for FP conversion casts. This meant:
  1272‚Üí
  1273‚Üí- **IntToFloat** always emitted `CVTSI2SD xmm, r32` (i32‚Üíf64), ignoring i32‚Üíf32, i64‚Üíf32, i64‚Üíf64 cases
  1274‚Üí- **UIntToFloat** always emitted `CVTSI2SD xmm, r64` (u32‚Üíf64 via zero-extend), ignoring u32‚Üíf32 and u64‚Üíf64/32 cases
  1275‚Üí- **FloatToInt** always emitted `CVTSD2SI r32, xmm` (f64‚Üíi32), ignoring f32‚Üíi32, f32‚Üíi64, f64‚Üíi64 cases
  1276‚Üí- **FloatToUInt** was identical to FloatToInt (no unsigned handling), ignoring f32‚Üíu32, f32‚Üíu64, f64‚Üíu32, f64‚Üíu64
  1277‚Üí- **FloatToFloat** always assumed f64‚Üíf32 (narrow), missing f32‚Üíf64 (widen)
  1278‚Üí
  1279‚Üí### Root Cause
  1280‚Üí
  1281‚ÜíThe `IRInstr::Cast` variant already had `from_ty: Option<IRType>` and `to_ty: Option<IRType>` fields (added previously for the RISC-V and LoongArch64 backends), but the x86_64 stack-slot ISel used `{ kind, dst, src, .. }` to destructure, discarding the type information.
