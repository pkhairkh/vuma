    82вҶ’    82вҶ’## Overview
    83вҶ’    83вҶ’
    84вҶ’    84вҶ’VUMA is a programming language framework built on a radical premise: **unsafe memory operations should not be forbidden вҖ” they should be made verifiable.** Instead of relying on a borrow checker to reject programs that cannot be statically proven safe, VUMA constructs a formal model of every memory operation and verifies global invariants against that model. Programs that pass verification run without runtime overhead; programs that fail receive precise counterexamples showing the exact execution path to the violation.
    85вҶ’    85вҶ’
    86вҶ’    86вҶ’### Why VUMA?
    87вҶ’    87вҶ’
    88вҶ’    88вҶ’Traditional languages face a fundamental trade-off: either restrict what programmers can express (Rust's borrow checker, Java's GC) or accept memory-unsafe programs (C, C++). VUMA breaks this trade-off through three innovations:
    89вҶ’    89вҶ’
    90вҶ’    90вҶ’1. **Semantic Computation Graphs (SCGs)** replace source code as the primary program representation. The SCG is a directed, acyclic, attributed multigraph where nodes are operations, edges are relationships, and regions delineate scopes and deployment targets.
    91вҶ’    91вҶ’
    92вҶ’    92вҶ’2. **Behavioral Descriptors (BDs)** replace nominal types. A BD is the triple (RepD, CapD, RelD) вҖ” representation, capabilities, and relationships вҖ” inferred from program structure rather than declared by the programmer.
    93вҶ’    93вҶ’
    94вҶ’    94вҶ’3. **Verification over restriction** вҖ” instead of rejecting programs that might be unsafe, VUMA verifies that they are safe and provides precise diagnostics when they are not.
    95вҶ’    95вҶ’
    96вҶ’    96вҶ’### The Five VUMA Invariants
    97вҶ’    97вҶ’
    98вҶ’    98вҶ’Every VUMA program is verified against five global memory-safety invariants:
    99вҶ’    99вҶ’
   100вҶ’   100вҶ’| Invariant       | Ensures                                            |
   101вҶ’   101вҶ’|-----------------|----------------------------------------------------|
   102вҶ’   102вҶ’| **Liveness**    | Every access targets allocated memory              |
   103вҶ’   103вҶ’| **Exclusivity** | No conflicting concurrent accesses                 |
   104вҶ’   104вҶ’| **Interpretation** | Every access uses a valid representation         |
   105вҶ’   105вҶ’| **Origin**      | Every address traces to a valid allocation         |
   106вҶ’   106вҶ’| **Cleanup**     | Every region is eventually freed or explicitly leaked |

   300вҶ’   300вҶ’## VUMA for LLMs
   301вҶ’   301вҶ’
   302вҶ’   302вҶ’VUMA is designed as an **AI-native** language framework. Every interface is designed for programmatic consumption by AI agents. This LLM-first design allows AI coding agents to use VUMA as a verified compilation sandbox: parse source, analyze structure, run verification, compile to any of 8 backends, and inspect results вҖ” all through clean API boundaries.
   303вҶ’   303вҶ’
   304вҶ’   304вҶ’### LLM-Facing Interfaces
   305вҶ’   305вҶ’
   306вҶ’   306вҶ’| Interface | Description | Use Case |
   307вҶ’   307вҶ’|-----------|-------------|----------|
   308вҶ’   308вҶ’| `VumaForLLM` API | `compile()`, `check()`, `analyze()`, `to_wasm()`, `explain_error()`, `suggest_fixes()` | LLM compiles and verifies code in a sandbox |
   309вҶ’   309вҶ’| `VumaCompiler` API | `compile()`, `parse()`, `analyze()`, `validate()`, `verify()` | Full pipeline with verification reports |
   310вҶ’   310вҶ’| LSP Server | Full LSP protocol: diagnostics, hover, go-to-definition, completion | IDE integration and LLM agent interaction |
   311вҶ’   311вҶ’| REPL `:wasm` | Compile to Wasm and show binary size | Quick Wasm compilation check |
   312вҶ’   312вҶ’| REPL `:backends` | List 8 backends with status | Discover compilation capabilities |
   313вҶ’   313вҶ’| REPL `:check` | Run IVE verification | Instant verification feedback |
   314вҶ’   314вҶ’| REPL `:diagnostics` | Show all diagnostics as JSON | Structured error analysis |
   315вҶ’   315вҶ’| REPL `:exports` | List all functions and signatures | Program structure inspection |
   316вҶ’   316вҶ’
   317вҶ’   317вҶ’### Wasm32 Sandbox for LLM Agents
   318вҶ’   318вҶ’
   319вҶ’   319вҶ’The Wasm32 backend enables LLM agents to compile VUMA programs into safe, sandboxed WebAssembly modules. This is the recommended execution path for LLM-generated code: the Wasm module runs in a sandboxed environment with no access to host memory or peripherals, ensuring that LLM-generated code cannot cause harm.
   320вҶ’   320вҶ’
   321вҶ’   321вҶ’```rust
   322вҶ’   322вҶ’// LLM agent compiles VUMA to Wasm
   323вҶ’   323вҶ’let result = VumaForLLM::compile(source);
   324вҶ’   324вҶ’if let Some(wasm) = &result.wasm_binary {
   325вҶ’   325вҶ’    println!("Wasm module: {} bytes", wasm.len());
   326вҶ’   326вҶ’}
   327вҶ’   327вҶ’```
   328вҶ’   328вҶ’
   329вҶ’   329вҶ’### Parser Hardening for LLM-Generated Code
   330вҶ’   330вҶ’
   331вҶ’   331вҶ’The VUMA parser is specifically hardened for code generated by LLMs:
   332вҶ’   332вҶ’
   333вҶ’   333вҶ’- **LLM type aliases**: `int` вҶ’ `i32`, `float` вҶ’ `f32`, `double` вҶ’ `f64`, `String` вҶ’ `string`
   334вҶ’   334вҶ’- **C/Rust construct detection**: `println!`, `vec!`, `format!`, `panic!` produce helpful errors
   335вҶ’   335вҶ’- **C-style for loop detection**: `for (i=0; i<n; i++)` вҶ’ specific error with VUMA suggestion
   336вҶ’   336вҶ’- **Reference type conversion**: `&T` / `&mut T` auto-convert to `*T` (pointer type)

   384вҶ’   384вҶ’
   385вҶ’   385вҶ’### Your First VUMA Program
   386вҶ’   386вҶ’
   387вҶ’   387вҶ’Create `hello.vuma`:
   388вҶ’   388вҶ’
   389вҶ’   389вҶ’```vuma
   390вҶ’   390вҶ’// hello.vuma вҖ” Basic memory allocation, write, read, and deallocation
   391вҶ’   391вҶ’
   392вҶ’   392вҶ’region main_region {
   393вҶ’   393вҶ’    let x: ptr<u8> = allocate(1) in main_region;
   394вҶ’   394вҶ’    write x, 42;
   395вҶ’   395вҶ’    let val: u8 = read x;
   396вҶ’   396вҶ’    deallocate x from main_region;
   397вҶ’   397вҶ’}
   398вҶ’   398вҶ’```
   399вҶ’   399вҶ’
   400вҶ’   400вҶ’Compile and verify:
   401вҶ’   401вҶ’
   402вҶ’   402вҶ’```bash
   403вҶ’   403вҶ’# Verify against all 5 VUMA invariants
   404вҶ’   404вҶ’vuma verify hello.vuma
   405вҶ’   405вҶ’
   406вҶ’   406вҶ’# Compile to a specific backend
   407вҶ’   407вҶ’vuma build --target x86_64 hello.vuma
   408вҶ’   408вҶ’vuma build --target aarch64 hello.vuma
   409вҶ’   409вҶ’vuma build --target wasm32 hello.vuma
   410вҶ’   410вҶ’
   411вҶ’   411вҶ’# Quick syntax and semantic check
   412вҶ’   412вҶ’vuma check hello.vuma
   413вҶ’   413вҶ’
   414вҶ’   414вҶ’# Run with memory safety analysis
   415вҶ’   415вҶ’vuma run --safe hello.vuma
   416вҶ’   416вҶ’
   417вҶ’   417вҶ’# Compile with DWARF debug info
   418вҶ’   418вҶ’vuma build --debug-info hello.vuma
   419вҶ’   419вҶ’```
   420вҶ’   420вҶ’
   421вҶ’   421вҶ’The verifier will check all five invariants against the SCG derived from your program. If verification passes, you get zero-overhead machine code for any of the 8 backends. If it fails, you get a precise counterexample showing the exact path to the violation.

   606вҶ’   606вҶ’## Key Concepts
   607вҶ’   607вҶ’
   608вҶ’   608вҶ’### SCG вҖ” Semantic Computation Graph
   609вҶ’   609вҶ’
   610вҶ’   610вҶ’The SCG is the single source of truth in VUMA. There is no "source code" that the compiler translates вҖ” the SCG *is* the program, and every other representation (textual, visual, conversational, machine code) is a **projection** of the SCG. Nodes represent computational operations (allocations, accesses, computations, casts, effects, control flow), edges represent relationships (data flow, control flow, derivation, annotation), and regions delineate scopes, phases, and deployment targets.
   611вҶ’   611вҶ’
   612вҶ’   612вҶ’### BD вҖ” Behavioral Descriptor
   613вҶ’   613вҶ’
   614вҶ’   614вҶ’A Behavioral Descriptor replaces traditional nominal types with the triple (RepD, CapD, RelD):
   615вҶ’   615вҶ’
   616вҶ’   616вҶ’- **RepD** (Representation Descriptor): memory layout вҖ” size, alignment, field offsets, multiple simultaneous interpretations
   617вҶ’   617вҶ’- **CapD** (Capability Descriptor): permitted operations вҖ” read, write, execute, serialize, send, persist, derive-pointer (context-dependent)
   618вҶ’   618вҶ’- **RelD** (Relational Descriptor): relationships вҖ” temporal co-occurrence, structural containment, dependency ordering, semantic equivalence, security-level flow
   619вҶ’   619вҶ’
   620вҶ’   620вҶ’BDs are **inferred**, not declared. The IVE derives them from SCG structure through iterative fixpoint computation.
   621вҶ’   621вҶ’
   622вҶ’   622вҶ’### MSG вҖ” Memory State Graph
   623вҶ’   623вҶ’
   624вҶ’   624вҶ’The Memory State Graph captures every allocation point, every pointer derivation, every deallocation point, every concurrent access, and every reinterpretation. It is constructed from the annotated SCG and serves as the formal model against which the five invariants are verified.
   625вҶ’   625вҶ’
   626вҶ’   626вҶ’### IVE вҖ” Inference & Verification Engine
   627вҶ’   627вҶ’
   628вҶ’   628вҶ’The IVE is the reasoning core. It reads the SCG, infers Behavioral Descriptors, constructs the MSG, and verifies the five global invariants. It operates through iterative fixpoint computation, resolving interdependencies between RepD, CapD, and RelD inference. It supports interprocedural analysis, escape analysis, verification caching, and incremental re-verification.
   629вҶ’   629вҶ’
   630вҶ’   630вҶ’### COR вҖ” Continuous Optimization Runtime
   631вҶ’   631вҶ’
   632вҶ’   632вҶ’The COR maintains an always-compiled invariant: every reachable SCG region is kept in compiled machine code at all times. It performs incremental compilation, profile-guided optimization (using PMU counters), speculative optimization with transparent deoptimization, and adaptive deployment across heterogeneous targets.

   737вҶ’   737вҶ’## Known Limitations
   738вҶ’   738вҶ’
   739вҶ’   739вҶ’This is an alpha release. We are transparent about what is not yet complete:
   740вҶ’   740вҶ’
   741вҶ’   741вҶ’| Area | Status | Details |
   742вҶ’   742вҶ’|------|--------|---------|
   743вҶ’   743вҶ’| **Self-hosting** | вқҢ Not started | VUMA cannot compile itself yet. The compiler is written in Rust. |
   744вҶ’   744вҶ’| **Structs / Enums** | вқҢ Not started | No user-defined struct or enum types. Programs operate on primitives, pointers, and regions. |
   745вҶ’   745вҶ’| **Stdlib is host-side only** | вҡ пёҸ Partial | Math, fmt, string, and crypto functions execute on the host side (Rust). They are not yet compiled to target machine code. |
   746вҶ’   746вҶ’| **BD inference completeness** | вҡ пёҸ Partial | Some complex BD inference scenarios (M2.3) are deferred to a future release. |
   747вҶ’   747вҶ’| **Doubly-linked list verification** | вҡ пёҸ Partial | Full verification of doubly-linked list patterns (M2.4) is not yet complete. |
   748вҶ’   748вҶ’| **Concurrent verification** | вҡ пёҸ Limited | Verification is limited to single-threaded programs. Full concurrent verification is planned. |
   749вҶ’   749вҶ’| **COR end-to-end** | вҡ пёҸ Partial | The Continuous Optimization Runtime is not yet integrated end-to-end. |
   750вҶ’   750вҶ’| **LoongArch64 performance** | вҡ пёҸ Slow | Full SHA256d on QEMU is too slow; should work on native hardware. |
   751вҶ’   751вҶ’| **Error recovery** | вҡ пёҸ Partial | Parser has known type mismatches in ASTвҶ’SCG lowering for some edge cases. |
   752вҶ’   752вҶ’
   753вҶ’   753вҶ’We believe in honest roadmapping. These limitations represent active development areas, not permanent constraints.
