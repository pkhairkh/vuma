    32вҶ’    32вҶ’## 1. Quick Start
    33вҶ’    33вҶ’
    34вҶ’    34вҶ’A VUMA program consists of one or more function definitions. The entry point is always `fn main()`. The compiler produces a native binary (or Wasm module) that can be executed directly or via QEMU for cross-architecture testing.
    35вҶ’    35вҶ’
    36вҶ’    36вҶ’### Minimal Program
    37вҶ’    37вҶ’
    38вҶ’    38вҶ’```vuma
    39вҶ’    39вҶ’fn main() -> i32 {
    40вҶ’    40вҶ’    return 0;
    41вҶ’    41вҶ’}
    42вҶ’    42вҶ’```
    43вҶ’    43вҶ’
    44вҶ’    44вҶ’This program exits with code 0.
    45вҶ’    45вҶ’
    46вҶ’    46вҶ’### Compile and Run
    47вҶ’    47вҶ’
    48вҶ’    48вҶ’VUMA compiles `.vuma` source files to native binaries. The build system uses Cargo under the hood. To build and test:
    49вҶ’    49вҶ’
    50вҶ’    50вҶ’```bash
    51вҶ’    51вҶ’# Build the VUMA compiler
    52вҶ’    52вҶ’cargo build --workspace
    53вҶ’    53вҶ’
    54вҶ’    54вҶ’# Run all tests
    55вҶ’    55вҶ’cargo test --workspace
    56вҶ’    56вҶ’
    57вҶ’    57вҶ’# Cross-architecture testing uses QEMU
    58вҶ’    58вҶ’# x86_64:
    59вҶ’    59вҶ’qemu-x86_64 ./output_binary
    60вҶ’    60вҶ’# AArch64:
    61вҶ’    61вҶ’qemu-aarch64 ./output_binary
    62вҶ’    62вҶ’# RISC-V 64:
    63вҶ’    63вҶ’qemu-riscv64-static ./output_binary
    64вҶ’    64вҶ’# ARM32:
    65вҶ’    65вҶ’qemu-arm ./output_binary
    66вҶ’    66вҶ’# MIPS64:
    67вҶ’    67вҶ’qemu-mips64-static ./output_binary
    68вҶ’    68вҶ’# PPC64:
    69вҶ’    69вҶ’qemu-ppc64 ./output_binary
    70вҶ’    70вҶ’# LoongArch64:
    71вҶ’    71вҶ’qemu-loongarch64 ./output_binary
    72вҶ’    72вҶ’```
    73вҶ’    73вҶ’
    74вҶ’    74вҶ’### Program with Memory
    75вҶ’    75вҶ’
    76вҶ’    76вҶ’```vuma
    77вҶ’    77вҶ’fn main() -> i32 {
    78вҶ’    78вҶ’    msg = allocate(3);
    79вҶ’    79вҶ’    *(msg + 0) = 97;   // 'a'
    80вҶ’    80вҶ’    *(msg + 1) = 98;   // 'b'
    81вҶ’    81вҶ’    *(msg + 2) = 99;   // 'c'
    82вҶ’    82вҶ’    free(msg);
    83вҶ’    83вҶ’    return 0;
    84вҶ’    84вҶ’}
    85вҶ’    85вҶ’```
    86вҶ’    86вҶ’
    87вҶ’    87вҶ’**Key facts**:
    88вҶ’    88вҶ’- `main()` returns `i32` (the process exit code) or `u64`.
    89вҶ’    89вҶ’- Every `allocate()` must be paired with a `free()`.
    90вҶ’    90вҶ’- Pointer dereference uses `*(ptr + offset)` syntax.
    91вҶ’    91вҶ’- Bytes are stored as integer values (e.g., `97` for ASCII `'a'`).

    95вҶ’    95вҶ’## 2. Types
    96вҶ’    96вҶ’
    97вҶ’    97вҶ’VUMA supports the following primitive types:
    98вҶ’    98вҶ’
    99вҶ’    99вҶ’| Type | Size (bytes) | Range | Signed |
   100вҶ’   100вҶ’|------|-------------|-------|--------|
   101вҶ’   101вҶ’| `i8`  | 1 | -128 to 127 | Yes |
   102вҶ’   102вҶ’| `i16` | 2 | -32768 to 32767 | Yes |
   103вҶ’   103вҶ’| `i32` | 4 | -2147483648 to 2147483647 | Yes |
   104вҶ’   104вҶ’| `i64` | 8 | -9223372036854775808 to 9223372036854775807 | Yes |
   105вҶ’   105вҶ’| `u8`  | 1 | 0 to 255 | No |
   106вҶ’   106вҶ’| `u16` | 2 | 0 to 65535 | No |
   107вҶ’   107вҶ’| `u32` | 4 | 0 to 4294967295 | No |
   108вҶ’   108вҶ’| `u64` | 8 | 0 to 18446744073709551615 | No |
   109вҶ’   109вҶ’| `Address` | 8 (4 on 32-bit targets) | N/A (opaque pointer) | No |
   110вҶ’   110вҶ’| `f32` | 4 | IEEE 754 single-precision | N/A |
   111вҶ’   111вҶ’| `f64` | 8 | IEEE 754 double-precision | N/A |
   112вҶ’   112вҶ’| `bool` | 1 | `true` or `false` | N/A |
   113вҶ’   113вҶ’| `void` | 0 | N/A | N/A |
   114вҶ’   114вҶ’
   115вҶ’   115вҶ’### Type Annotation Syntax
   116вҶ’   116вҶ’
   117вҶ’   117вҶ’```vuma
   118вҶ’   118вҶ’x: i32 = 42;
   119вҶ’   119вҶ’y: u32 = 0xFFFFFFFF;
   120вҶ’   120вҶ’ptr: Address = allocate(64);
   121вҶ’   121вҶ’big: u64 = 18446744073709551615;
   122вҶ’   122вҶ’flag: bool = true;
   123вҶ’   123вҶ’pi: f64 = 3.141592653589793;
   124вҶ’   124вҶ’angle: f32 = 1.5707964;
   125вҶ’   125вҶ’```
   126вҶ’   126вҶ’
   127вҶ’   127вҶ’### Critical Constraint: u32 on 64-bit Registers
   128вҶ’   128вҶ’
   129вҶ’   129вҶ’**This is the single most important thing to know about VUMA types.** All 64-bit target platforms (x86_64, AArch64, RISC-V 64, MIPS64, PPC64, LoongArch64) use 64-bit general-purpose registers. A `u32` value stored in a 64-bit register can have garbage bits in the upper 32 bits after arithmetic operations. You **must** manually mask u32 results with `& 4294967295` (which is `0xFFFFFFFF` in decimal) after any operation that could set bits above bit 31:
   130вҶ’   130вҶ’
   131вҶ’   131вҶ’```vuma
   132вҶ’   132вҶ’// WRONG: bits above bit 31 may be set
   133вҶ’   133вҶ’sum: u32 = a + b;
   134вҶ’   134вҶ’
   135вҶ’   135вҶ’// CORRECT: mask to 32 bits after addition
   136вҶ’   136вҶ’sum: u32 = (a + b) & 4294967295;
   137вҶ’   137вҶ’```
   138вҶ’   138вҶ’
   139вҶ’   139вҶ’See [Section 14: Pitfalls](#14-pitfalls) for the complete list of operations that require masking.
   140вҶ’   140вҶ’
   141вҶ’   141вҶ’### Address Type
   142вҶ’   142вҶ’
   143вҶ’   143вҶ’`Address` is the type for pointers and memory regions. It is 8 bytes on 64-bit targets and 4 bytes on 32-bit targets (ARM32, Wasm32). The `allocate()` built-in returns an `Address`.
   144вҶ’   144вҶ’
   145вҶ’   145вҶ’```vuma
   146вҶ’   146вҶ’buf: Address = allocate(256);
   147вҶ’   147вҶ’```

   150вҶ’   150вҶ’
   151вҶ’   151вҶ’## 3. Functions
   152вҶ’   152вҶ’
   153вҶ’   153вҶ’### Function Declaration
   154вҶ’   154вҶ’
   155вҶ’   155вҶ’```vuma
   156вҶ’   156вҶ’fn function_name(param1: type1, param2: type2) -> return_type {
   157вҶ’   157вҶ’    // body
   158вҶ’   158вҶ’    return value;
   159вҶ’   159вҶ’}
   160вҶ’   160вҶ’```
   161вҶ’   161вҶ’
   162вҶ’   162вҶ’- The `fn` keyword introduces a function.
   163вҶ’   163вҶ’- Parameters have mandatory type annotations when declared in the signature.
   164вҶ’   164вҶ’- The return type follows `->`. If omitted, the function returns `void`.
   165вҶ’   165вҶ’- The body is a block enclosed in `{ }`.
   166вҶ’   166вҶ’- Use `return expr;` to return a value. The last expression in a block without `return` is also the return value in some contexts, but `return` is preferred for clarity.
   167вҶ’   167вҶ’
   168вҶ’   168вҶ’### Examples
   169вҶ’   169вҶ’
   170вҶ’   170вҶ’```vuma
   171вҶ’   171вҶ’// Function returning a value
   172вҶ’   172вҶ’fn add(a: i32, b: i32) -> i32 {
   173вҶ’   173вҶ’    return a + b;
   174вҶ’   174вҶ’}
   175вҶ’   175вҶ’
   176вҶ’   176вҶ’// Function with no return value (void)
   177вҶ’   177вҶ’fn store_byte(buf: Address, offset: u64, val: u32) {
   178вҶ’   178вҶ’    *(buf + offset) = val & 255;
   179вҶ’   179вҶ’}
   180вҶ’   180вҶ’
   181вҶ’   181вҶ’// Function with Address parameter
   182вҶ’   182вҶ’fn read_u32_be(buf: Address, offset: u64) -> u32 {
   183вҶ’   183вҶ’    b0: u32 = *(buf + offset);
   184вҶ’   184вҶ’    b1: u32 = *(buf + offset + 1);
   185вҶ’   185вҶ’    b2: u32 = *(buf + offset + 2);
   186вҶ’   186вҶ’    b3: u32 = *(buf + offset + 3);
   187вҶ’   187вҶ’    return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
   188вҶ’   188вҶ’}
   189вҶ’   189вҶ’
   190вҶ’   190вҶ’// Main entry point вҖ” must return i32 or u64
   191вҶ’   191вҶ’fn main() -> i32 {
   192вҶ’   192вҶ’    return 0;
   193вҶ’   193вҶ’}
   194вҶ’   194вҶ’```
   195вҶ’   195вҶ’
   196вҶ’   196вҶ’### Function Call
   197вҶ’   197вҶ’
   198вҶ’   198вҶ’```vuma
   199вҶ’   199вҶ’result: i32 = add(10, 20);
   200вҶ’   200вҶ’store_byte(buffer, 0, 65);
   201вҶ’   201вҶ’```
   202вҶ’   202вҶ’
   203вҶ’   203вҶ’Functions are called by name with parenthesized arguments. Function calls can appear as expressions (when the function returns a value) or as statements (when the function returns void or the return value is discarded).

   207вҶ’   207вҶ’## 4. Variables
   208вҶ’   208вҶ’
   209вҶ’   209вҶ’### Declaration with Type Annotation
   210вҶ’   210вҶ’
   211вҶ’   211вҶ’```vuma
   212вҶ’   212вҶ’name: type = expression;
   213вҶ’   213вҶ’```
   214вҶ’   214вҶ’
   215вҶ’   215вҶ’Variables are declared with a name, a type annotation, and an initial value. The type annotation is required in most cases.
   216вҶ’   216вҶ’
   217вҶ’   217вҶ’```vuma
   218вҶ’   218вҶ’x: i32 = 42;
   219вҶ’   219вҶ’y: u32 = 0xFFFFFFFF;
   220вҶ’   220вҶ’ptr: Address = allocate(64);
   221вҶ’   221вҶ’sum: u32 = (a + b) & 4294967295;
   222вҶ’   222вҶ’```
   223вҶ’   223вҶ’
   224вҶ’   224вҶ’### Declaration Without Type Annotation (Inferred)
   225вҶ’   225вҶ’
   226вҶ’   226вҶ’VUMA can infer the type from the right-hand side in some cases, but explicit annotation is strongly recommended to avoid ambiguity:
   227вҶ’   227вҶ’
   228вҶ’   228вҶ’```vuma
   229вҶ’   229вҶ’result = add1(41);       // type inferred from function return type
   230вҶ’   230вҶ’msg = allocate(3);        // type inferred as Address
   231вҶ’   231вҶ’```
   232вҶ’   232вҶ’
   233вҶ’   233вҶ’**When writing VUMA code as an LLM, always use explicit type annotations** except for variables assigned from `allocate()` (which always returns `Address`).
   234вҶ’   234вҶ’
   235вҶ’   235вҶ’### Assignment
   236вҶ’   236вҶ’
   237вҶ’   237вҶ’Once declared, variables can be reassigned without repeating the type:
   238вҶ’   238вҶ’
   239вҶ’   239вҶ’```vuma
   240вҶ’   240вҶ’x: u32 = 10;
   241вҶ’   241вҶ’x = 20;              // reassignment
   242вҶ’   242вҶ’x = (x + 1) & 4294967295;  // reassignment with expression
   243вҶ’   243вҶ’```
   244вҶ’   244вҶ’
   245вҶ’   245вҶ’### Compound Assignment
   246вҶ’   246вҶ’
   247вҶ’   247вҶ’VUMA supports compound assignment operators:
   248вҶ’   248вҶ’
   249вҶ’   249вҶ’```vuma
   250вҶ’   250вҶ’x += 1;     // x = x + 1
   251вҶ’   251вҶ’x -= 1;     // x = x - 1
   252вҶ’   252вҶ’x *= 2;     // x = x * 2
   253вҶ’   253вҶ’x &= 255;   // x = x & 255
   254вҶ’   254вҶ’x |= mask;  // x = x | mask
   255вҶ’   255вҶ’x ^= key;   // x = x ^ key
   256вҶ’   256вҶ’x <<= 4;    // x = x << 4
   257вҶ’   257вҶ’x >>= 4;    // x = x >> 4
   258вҶ’   258вҶ’```
   259вҶ’   259вҶ’
   260вҶ’   260вҶ’**Note**: Compound assignments on `u32` values still need the `& 4294967295` mask. The compound form does **not** automatically mask. Prefer explicit assignment with masking:
   261вҶ’   261вҶ’
   262вҶ’   262вҶ’```vuma
   263вҶ’   263вҶ’// WRONG for u32 on 64-bit registers:
   264вҶ’   264вҶ’x += 1;
   265вҶ’   265вҶ’
   266вҶ’   266вҶ’// CORRECT:
   267вҶ’   267вҶ’x = (x + 1) & 4294967295;
   268вҶ’   268вҶ’```

   272вҶ’   272вҶ’## 5. Control Flow
   273вҶ’   273вҶ’
   274вҶ’   274вҶ’### For Loop (Range)
   275вҶ’   275вҶ’
   276вҶ’   276вҶ’The `for` loop iterates over an exclusive range `start..end`. The loop variable takes each integer value from `start` (inclusive) to `end` (exclusive).
   277вҶ’   277вҶ’
   278вҶ’   278вҶ’```vuma
   279вҶ’   279вҶ’for i in 0..10 {
   280вҶ’   280вҶ’    // i = 0, 1, 2, ..., 9
   281вҶ’   281вҶ’}
   282вҶ’   282вҶ’
   283вҶ’   283вҶ’for i in 0..64 {
   284вҶ’   284вҶ’    // i = 0, 1, 2, ..., 63
   285вҶ’   285вҶ’}
   286вҶ’   286вҶ’
   287вҶ’   287вҶ’for i in 16..64 {
   288вҶ’   288вҶ’    // i = 16, 17, ..., 63
   289вҶ’   289вҶ’}
   290вҶ’   290вҶ’```
   291вҶ’   291вҶ’
   292вҶ’   292вҶ’**The range is always exclusive on the upper bound**: `0..10` means `0 <= i < 10`.
   293вҶ’   293вҶ’
   294вҶ’   294вҶ’### For Loop with Memory Access
   295вҶ’   295вҶ’
   296вҶ’   296вҶ’```vuma
   297вҶ’   297вҶ’// Copy 32 bytes from src to dst
   298вҶ’   298вҶ’fn copy32(dst: Address, src: Address) {
   299вҶ’   299вҶ’    i: u64 = 0;
   300вҶ’   300вҶ’    for i in 0..32 {
   301вҶ’   301вҶ’        *(dst + i) = *(src + i);
   302вҶ’   302вҶ’    }
   303вҶ’   303вҶ’}
   304вҶ’   304вҶ’```
   305вҶ’   305вҶ’
   306вҶ’   306вҶ’### While Loop
   307вҶ’   307вҶ’
   308вҶ’   308вҶ’```vuma
   309вҶ’   309вҶ’current = (*list).next;
   310вҶ’   310вҶ’while current != list {
   311вҶ’   311вҶ’    process(current);
   312вҶ’   312вҶ’    current = (*current).next;
   313вҶ’   313вҶ’}
   314вҶ’   314вҶ’```
   315вҶ’   315вҶ’
   316вҶ’   316вҶ’### If / Else
   317вҶ’   317вҶ’
   318вҶ’   318вҶ’```vuma
   319вҶ’   319вҶ’if x > 100 {
   320вҶ’   320вҶ’    y = 2;
   321вҶ’   321вҶ’} else {
   322вҶ’   322вҶ’    y = 1;
   323вҶ’   323вҶ’}
   324вҶ’   324вҶ’```
   325вҶ’   325вҶ’
   326вҶ’   326вҶ’```vuma
   327вҶ’   327вҶ’if value & 1 == 1 {
   328вҶ’   328вҶ’    // odd
   329вҶ’   329вҶ’}
   330вҶ’   330вҶ’```
   331вҶ’   331вҶ’
   332вҶ’   332вҶ’Nested conditionals:
   333вҶ’   333вҶ’
   334вҶ’   334вҶ’```vuma
   335вҶ’   335вҶ’if x > 0 {
   336вҶ’   336вҶ’    if x > 100 {
   337вҶ’   337вҶ’        result = 2;
   338вҶ’   338вҶ’    } else {
   339вҶ’   339вҶ’        result = 1;
   340вҶ’   340вҶ’    }
   341вҶ’   341вҶ’} else {
   342вҶ’   342вҶ’    result = 0;
   343вҶ’   343вҶ’}
   344вҶ’   344вҶ’```
   345вҶ’   345вҶ’
   346вҶ’   346вҶ’### Infinite Loop
   347вҶ’   347вҶ’
   348вҶ’   348вҶ’```vuma
   349вҶ’   349вҶ’loop {

   370вҶ’   370вҶ’## 6. Memory
   371вҶ’   371вҶ’
   372вҶ’   372вҶ’VUMA provides two first-class memory operations: `allocate` and `free`. There is no garbage collector; memory management is manual.
   373вҶ’   373вҶ’
   374вҶ’   374вҶ’### allocate(n)
   375вҶ’   375вҶ’
   376вҶ’   376вҶ’Reserves `n` bytes of memory and returns an `Address` (pointer to the start of the block). The allocated memory is uninitialized.
   377вҶ’   377вҶ’
   378вҶ’   378вҶ’```vuma
   379вҶ’   379вҶ’buf = allocate(64);     // 64-byte buffer
   380вҶ’   380вҶ’state = allocate(32);   // 32-byte state buffer
   381вҶ’   381вҶ’k_table = allocate(256); // 256-byte constant table
   382вҶ’   382вҶ’```
   383вҶ’   383вҶ’
   384вҶ’   384вҶ’**The returned `Address` does not need a type annotation** вҖ” it is always inferred as `Address`:
   385вҶ’   385вҶ’
   386вҶ’   386вҶ’```vuma
   387вҶ’   387вҶ’state = allocate(32);   // state has type Address
   388вҶ’   388вҶ’```
   389вҶ’   389вҶ’
   390вҶ’   390вҶ’### free(ptr)
   391вҶ’   391вҶ’
   392вҶ’   392вҶ’Releases the memory block pointed to by `ptr`. After `free(ptr)`, the pointer is invalid and must not be used.
   393вҶ’   393вҶ’
   394вҶ’   394вҶ’```vuma
   395вҶ’   395вҶ’free(state);
   396вҶ’   396вҶ’free(k_table);
   397вҶ’   397вҶ’free(buf);
   398вҶ’   398вҶ’```
   399вҶ’   399вҶ’
   400вҶ’   400вҶ’**Every `allocate` must be matched with a `free`**. Failing to free memory is a resource leak. The IVE (Invariant Verification Engine) checks this at compile time.
   401вҶ’   401вҶ’
   402вҶ’   402вҶ’### Memory Layout Example
   403вҶ’   403вҶ’
   404вҶ’   404вҶ’```vuma
   405вҶ’   405вҶ’fn sha256d(msg: Address, msg_len: u64, out: Address) {
   406вҶ’   406вҶ’    // Allocate working memory
   407вҶ’   407вҶ’    state = allocate(32);
   408вҶ’   408вҶ’    k = allocate(256);
   409вҶ’   409вҶ’    w = allocate(256);
   410вҶ’   410вҶ’    block = allocate(64);
   411вҶ’   411вҶ’    inner = allocate(32);
   412вҶ’   412вҶ’
   413вҶ’   413вҶ’    // ... use the memory ...
   414вҶ’   414вҶ’
   415вҶ’   415вҶ’    // Clean up all allocations
   416вҶ’   416вҶ’    free(state);
   417вҶ’   417вҶ’    free(k);
   418вҶ’   418вҶ’    free(w);
   419вҶ’   419вҶ’    free(block);
   420вҶ’   420вҶ’    free(inner);
   421вҶ’   421вҶ’}
   422вҶ’   422вҶ’```
   423вҶ’   423вҶ’
   424вҶ’   424вҶ’### Region Declarations
   425вҶ’   425вҶ’
   426вҶ’   426вҶ’Named regions can be declared with the `region` keyword:
   427вҶ’   427вҶ’
   428вҶ’   428вҶ’```vuma
   429вҶ’   429вҶ’region pool = allocate(4096);
   430вҶ’   430вҶ’```

   433вҶ’   433вҶ’
   434вҶ’   434вҶ’## 7. Pointer Operations
   435вҶ’   435вҶ’
   436вҶ’   436вҶ’### Dereference: Load
   437вҶ’   437вҶ’
   438вҶ’   438вҶ’Read a byte from memory at `ptr + offset`:
   439вҶ’   439вҶ’
   440вҶ’   440вҶ’```vuma
   441вҶ’   441вҶ’b0: u32 = *(buf + offset);
   442вҶ’   442вҶ’```
   443вҶ’   443вҶ’
   444вҶ’   444вҶ’The result is a single byte zero-extended to the type of the variable (typically `u32`).
   445вҶ’   445вҶ’
   446вҶ’   446вҶ’### Dereference: Store
   447вҶ’   447вҶ’
   448вҶ’   448вҶ’Write a byte to memory at `ptr + offset`:
   449вҶ’   449вҶ’
   450вҶ’   450вҶ’```vuma
   451вҶ’   451вҶ’*(buf + offset) = value & 255;
   452вҶ’   452вҶ’```
   453вҶ’   453вҶ’
   454вҶ’   454вҶ’Only the low 8 bits of `value` are stored.
   455вҶ’   455вҶ’
   456вҶ’   456вҶ’### Pointer Arithmetic
   457вҶ’   457вҶ’
   458вҶ’   458вҶ’Pointer + integer produces a new `Address` offset by that many bytes:
   459вҶ’   459вҶ’
   460вҶ’   460вҶ’```vuma
   461вҶ’   461вҶ’ptr = base + 128;      // 128 bytes forward
   462вҶ’   462вҶ’byte3 = *(buf + 3);    // read byte at offset 3
   463вҶ’   463вҶ’*(buf + 4) = 99;       // write byte at offset 4
   464вҶ’   464вҶ’```
   465вҶ’   465вҶ’
   466вҶ’   466вҶ’The offset can be an expression:
   467вҶ’   467вҶ’
   468вҶ’   468вҶ’```vuma
   469вҶ’   469вҶ’*(buf + i * 4) = val;  // write at byte offset i*4
   470вҶ’   470вҶ’byte_val = *(buf + i); // read at byte offset i
   471вҶ’   471вҶ’```
   472вҶ’   472вҶ’
   473вҶ’   473вҶ’### Read/Write Multi-byte Values (u32)
   474вҶ’   474вҶ’
   475вҶ’   475вҶ’VUMA does not have native multi-byte load/store. You must compose byte-level accesses. For big-endian u32:
   476вҶ’   476вҶ’
   477вҶ’   477вҶ’```vuma
   478вҶ’   478вҶ’fn write_u32_be(buf: Address, offset: u64, val: u32) {
   479вҶ’   479вҶ’    *(buf + offset) = (val >> 24) & 255;
   480вҶ’   480вҶ’    *(buf + offset + 1) = (val >> 16) & 255;
   481вҶ’   481вҶ’    *(buf + offset + 2) = (val >> 8) & 255;
   482вҶ’   482вҶ’    *(buf + offset + 3) = val & 255;
   483вҶ’   483вҶ’}
   484вҶ’   484вҶ’
   485вҶ’   485вҶ’fn read_u32_be(buf: Address, offset: u64) -> u32 {
   486вҶ’   486вҶ’    b0: u32 = *(buf + offset);
   487вҶ’   487вҶ’    b1: u32 = *(buf + offset + 1);
   488вҶ’   488вҶ’    b2: u32 = *(buf + offset + 2);
   489вҶ’   489вҶ’    b3: u32 = *(buf + offset + 3);
   490вҶ’   490вҶ’    return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
   491вҶ’   491вҶ’}
   492вҶ’   492вҶ’```
   493вҶ’   493вҶ’
   494вҶ’   494вҶ’**Why the `& 4294967295` mask?** After left-shifting `b0` by 24 bits in a 64-bit register, bits above bit 31 may be set. The mask clears them. This is the most common source of bugs in VUMA code. See [Section 14: Pitfalls](#14-pitfalls).
   495вҶ’   495вҶ’
   496вҶ’   496вҶ’### Address-Of Operator
   497вҶ’   497вҶ’
   498вҶ’   498вҶ’The `@` operator takes the address of a variable:
   499вҶ’   499вҶ’
   500вҶ’   500вҶ’```vuma
   501вҶ’   501вҶ’let x: i32 = 42;
   502вҶ’   502вҶ’ptr = @x;       // ptr has type Address, points to x
   503вҶ’   503вҶ’*ptr = 100;     // modifies x through the pointer
   504вҶ’   504вҶ’```
   505вҶ’   505вҶ’
   506вҶ’   506вҶ’### Derive
   507вҶ’   507вҶ’
   508вҶ’   508вҶ’The `derive(ptr, region)` expression explicitly creates a derived pointer within a specific region. This is the formal mechanism for sub-allocation:
   509вҶ’   509вҶ’
   510вҶ’   510вҶ’```vuma
   511вҶ’   511вҶ’inner = derive(ptr, arena_region);
   512вҶ’   512вҶ’```
   513вҶ’   513вҶ’
   514вҶ’   514вҶ’In practice, pointer arithmetic (`base + offset`) implicitly derives a new pointer.
   515вҶ’   515вҶ’
   516вҶ’   516вҶ’---

   518вҶ’   518вҶ’## 8. Bitwise Operations
   519вҶ’   519вҶ’
   520вҶ’   520вҶ’VUMA supports the standard bitwise operations:
   521вҶ’   521вҶ’
   522вҶ’   522вҶ’| Operator | Name | Example |
   523вҶ’   523вҶ’|----------|------|---------|
   524вҶ’   524вҶ’| `&` | Bitwise AND | `a & 0xFF` |
   525вҶ’   525вҶ’| `\|` | Bitwise OR | `a \| b` |
   526вҶ’   526вҶ’| `^` | Bitwise XOR | `a ^ 0xFFFFFFFF` |
   527вҶ’   527вҶ’| `<<` | Left shift | `a << 8` |
   528вҶ’   528вҶ’| `>>` | Right shift | `a >> 4` |
   529вҶ’   529вҶ’
   530вҶ’   530вҶ’### No Built-in NOT Operator
   531вҶ’   531вҶ’
   532вҶ’   532вҶ’**VUMA does not have a reliable bitwise NOT (`~`) operator for u32 values.** The `~x` operator inverts all 64 bits of the register, which corrupts the upper 32 bits. Instead, use XOR with `0xFFFFFFFF`:
   533вҶ’   533вҶ’
   534вҶ’   534вҶ’```vuma
   535вҶ’   535вҶ’// WRONG: ~x inverts bits 32-63 which corrupts u32 values
   536вҶ’   536вҶ’not_x = ~x;
   537вҶ’   537вҶ’
   538вҶ’   538вҶ’// CORRECT: XOR with 0xFFFFFFFF inverts only the lower 32 bits
   539вҶ’   539вҶ’not_x: u32 = x ^ 4294967295;
   540вҶ’   540вҶ’```
   541вҶ’   541вҶ’
   542вҶ’   542вҶ’**Always use `x ^ 4294967295` instead of `~x` for u32 bitwise NOT.**
   543вҶ’   543вҶ’
   544вҶ’   544вҶ’### Bitwise AND for Masking
   545вҶ’   545вҶ’
   546вҶ’   546вҶ’```vuma
   547вҶ’   547вҶ’low_byte: u32 = value & 255;          // mask to 8 bits
   548вҶ’   548вҶ’low_16: u32 = value & 65535;          // mask to 16 bits
   549вҶ’   549вҶ’low_32: u32 = value & 4294967295;     // mask to 32 bits
   550вҶ’   550вҶ’```
   551вҶ’   551вҶ’
   552вҶ’   552вҶ’### Bitwise OR for Combining
   553вҶ’   553вҶ’
   554вҶ’   554вҶ’```vuma
   555вҶ’   555вҶ’combined: u32 = (high << 8) | low;
   556вҶ’   556вҶ’```
   557вҶ’   557вҶ’
   558вҶ’   558вҶ’### Left Shift
   559вҶ’   559вҶ’
   560вҶ’   560вҶ’```vuma
   561вҶ’   561вҶ’shifted: u32 = (value << 24) & 4294967295;  // MUST mask after left shift
   562вҶ’   562вҶ’```
   563вҶ’   563вҶ’
   564вҶ’   564вҶ’**Left shift on u32 values always requires masking** because the result can have bits set above bit 31.
   565вҶ’   565вҶ’
   566вҶ’   566вҶ’### Right Shift
   567вҶ’   567вҶ’
   568вҶ’   568вҶ’```vuma
   569вҶ’   569вҶ’upper: u32 = value >> 24;   // no mask needed for right shift
   570вҶ’   570вҶ’```
   571вҶ’   571вҶ’
   572вҶ’   572вҶ’Right shift does not require masking because it only clears bits (never sets new bits above the existing ones).
   573вҶ’   573вҶ’
   574вҶ’   574вҶ’### Rotate Right (Compose from Shifts)
   575вҶ’   575вҶ’
   576вҶ’   576вҶ’VUMA does not have a built-in rotate operator. Compose it from shifts:
   577вҶ’   577вҶ’
   578вҶ’   578вҶ’```vuma
   579вҶ’   579вҶ’fn rotr32(x: u32, n: u32) -> u32 {
   580вҶ’   580вҶ’    return ((x >> n) | (x << (32 - n))) & 4294967295;
   581вҶ’   581вҶ’}
   582вҶ’   582вҶ’```
   583вҶ’   583вҶ’
   584вҶ’   584вҶ’The mask is required because `x << (32 - n)` sets bits above bit 31.
   585вҶ’   585вҶ’
   586вҶ’   586вҶ’---

   588вҶ’   588вҶ’## 9. Arithmetic
   589вҶ’   589вҶ’
   590вҶ’   590вҶ’### Basic Operators
   591вҶ’   591вҶ’
   592вҶ’   592вҶ’| Operator | Name | Example |
   593вҶ’   593вҶ’|----------|------|---------|
   594вҶ’   594вҶ’| `+` | Addition | `a + b` |
   595вҶ’   595вҶ’| `-` | Subtraction | `a - b` |
   596вҶ’   596вҶ’| `*` | Multiplication | `a * b` |
   597вҶ’   597вҶ’| `/` | Division | `a / b` |
   598вҶ’   598вҶ’| `%` | Modulo | `a % b` |
   599вҶ’   599вҶ’
   600вҶ’   600вҶ’### u32 Masking Rule
   601вҶ’   601вҶ’
   602вҶ’   602вҶ’**All u32 arithmetic results must be masked with `& 4294967295`.** VUMA uses 64-bit registers on 64-bit targets. Arithmetic on u32 values can produce carries into the upper 32 bits:
   603вҶ’   603вҶ’
   604вҶ’   604вҶ’```vuma
   605вҶ’   605вҶ’// WRONG: overflow bits pollute upper 32 bits
   606вҶ’   606вҶ’sum: u32 = a + b;
   607вҶ’   607вҶ’
   608вҶ’   608вҶ’// CORRECT: mask to 32 bits
   609вҶ’   609вҶ’sum: u32 = (a + b) & 4294967295;
   610вҶ’   610вҶ’```
   611вҶ’   611вҶ’
   612вҶ’   612вҶ’This applies to **all** arithmetic operations on u32 values:
   613вҶ’   613вҶ’
   614вҶ’   614вҶ’```vuma
   615вҶ’   615вҶ’sum: u32 = (a + b) & 4294967295;
   616вҶ’   616вҶ’diff: u32 = (a - b) & 4294967295;
   617вҶ’   617вҶ’prod: u32 = (a * b) & 4294967295;
   618вҶ’   618вҶ’quot: u32 = (a / b) & 4294967295;
   619вҶ’   619вҶ’rem: u32 = (a % b) & 4294967295;
   620вҶ’   620вҶ’```
   621вҶ’   621вҶ’
   622вҶ’   622вҶ’Division and modulo technically cannot overflow for u32, but masking is still recommended for consistency and safety.
   623вҶ’   623вҶ’
   624вҶ’   624вҶ’### u64 Arithmetic
   625вҶ’   625вҶ’
   626вҶ’   626вҶ’u64 arithmetic does **not** require masking:
   627вҶ’   627вҶ’
   628вҶ’   628вҶ’```vuma
   629вҶ’   629вҶ’big_sum: u64 = a + b;    // OK, no mask needed
   630вҶ’   630вҶ’```
   631вҶ’   631вҶ’
   632вҶ’   632вҶ’### i32 Arithmetic
   633вҶ’   633вҶ’
   634вҶ’   634вҶ’Signed 32-bit arithmetic should also be masked for correctness:
   635вҶ’   635вҶ’
   636вҶ’   636вҶ’```vuma
   637вҶ’   637вҶ’signed_result: i32 = (a + b) & 4294967295;
   638вҶ’   638вҶ’```
   639вҶ’   639вҶ’
   640вҶ’   640вҶ’### Chained Arithmetic
   641вҶ’   641вҶ’
   642вҶ’   642вҶ’When multiple u32 operations are chained, mask the **final** result:
   643вҶ’   643вҶ’
   644вҶ’   644вҶ’```vuma
   645вҶ’   645вҶ’// SHA-256 T1 computation
   646вҶ’   646вҶ’t1: u32 = (h + big_sigma1(e) + ch(e, f, g) + ki + wi) & 4294967295;
   647вҶ’   647вҶ’```
   648вҶ’   648вҶ’
   649вҶ’   649вҶ’Intermediate results in sub-expressions (like `big_sigma1(e)`) should also be masked in their function definitions. Every function that returns u32 must mask its return value.
   650вҶ’   650вҶ’

   743вҶ’   743вҶ’**In existing VUMA code, the decimal form `4294967295` is conventional for u32 masking.** When generating VUMA code, prefer the decimal form for consistency with the existing codebase.

   756вҶ’   756вҶ’## 12. Calling Convention
   757вҶ’   757вҶ’
   758вҶ’   758вҶ’### Argument Passing
   759вҶ’   759вҶ’
   760вҶ’   760вҶ’On 64-bit targets, the first 8 integer/pointer arguments are passed in registers (X0-X7 on AArch64, RDI/RSI/RDX/RCX/R8/R9 on x86_64, a0-a7 on RISC-V). Additional arguments are passed on the stack.
   761вҶ’   761вҶ’
   762вҶ’   762вҶ’On 32-bit targets (ARM32, Wasm32), the first 4 arguments use registers (r0-r3 on ARM32, the value stack on Wasm32).
   763вҶ’   763вҶ’
   764вҶ’   764вҶ’### Return Values
   765вҶ’   765вҶ’
   766вҶ’   766вҶ’The return value is passed in the first return register (X0 on AArch64, RAX on x86_64, a0 on RISC-V). For void functions, no return register is used.
   767вҶ’   767вҶ’
   768вҶ’   768вҶ’### Address Parameters
   769вҶ’   769вҶ’
   770вҶ’   770вҶ’`Address` parameters are passed like integer values (pointer-sized). On 64-bit targets, they occupy 8 bytes; on 32-bit targets, 4 bytes.
   771вҶ’   771вҶ’
   772вҶ’   772вҶ’### Function Call Example
   773вҶ’   773вҶ’
   774вҶ’   774вҶ’```vuma
   775вҶ’   775вҶ’fn add1(x: i32) -> i32 {
   776вҶ’   776вҶ’    return x + 1;
   777вҶ’   777вҶ’}
   778вҶ’   778вҶ’
   779вҶ’   779вҶ’fn main() -> i32 {
   780вҶ’   780вҶ’    return add1(41);  // returns 42
   781вҶ’   781вҶ’}
   782вҶ’   782вҶ’```
   783вҶ’   783вҶ’
   784вҶ’   784вҶ’### Cross-Function u32 Values
   785вҶ’   785вҶ’
   786вҶ’   786вҶ’When a function returns `u32`, the caller receives the value in a 64-bit register. The upper 32 bits may contain garbage. The **returning function** must mask the value before returning, and the **calling function** should not assume the upper bits are clear unless the callee guarantees it.
   787вҶ’   787вҶ’
   788вҶ’   788вҶ’```vuma
   789вҶ’   789вҶ’fn safe_add(a: u32, b: u32) -> u32 {
   790вҶ’   790вҶ’    return (a + b) & 4294967295;  // callee masks the result
   791вҶ’   791вҶ’}
   792вҶ’   792вҶ’
   793вҶ’   793вҶ’fn main() -> i32 {
   794вҶ’   794вҶ’    result: u32 = safe_add(0xFFFFFFFF, 1);  // result = 0
   795вҶ’   795вҶ’    return 0;
   796вҶ’   796вҶ’}
   797вҶ’   797вҶ’```
   798вҶ’   798вҶ’
   799вҶ’   799вҶ’### Stack Layout (AArch64)
   800вҶ’   800вҶ’
   801вҶ’   801вҶ’```
   802вҶ’   802вҶ’Higher addresses
   803вҶ’   803вҶ’  +----------------------+
   804вҶ’   804вҶ’  | Incoming stack args   |  FP+16, FP+24, ...
   805вҶ’   805вҶ’  +----------------------+
   806вҶ’   806вҶ’  | Saved FP (X29)       |  FP+0
   807вҶ’   807вҶ’  | Saved LR (X30)       |  FP+8
   808вҶ’   808вҶ’  +----------------------+
   809вҶ’   809вҶ’  | Callee-saved regs    |  FP-8, FP-16, ...
   810вҶ’   810вҶ’  | (X19..X28)           |
   811вҶ’   811вҶ’  +----------------------+
   812вҶ’   812вҶ’  | Local variables      |
   813вҶ’   813вҶ’  | (aligned)            |
   814вҶ’   814вҶ’  +----------------------+
   815вҶ’   815вҶ’  | Outgoing stack args  |
   816вҶ’   816вҶ’  | (for nested calls)   |
   817вҶ’   817вҶ’  +----------------------+  SP
   818вҶ’   818вҶ’Lower addresses
   819вҶ’   819вҶ’```

   823вҶ’   823вҶ’## 13. Standard Library Modules
   824вҶ’   824вҶ’
   825вҶ’   825вҶ’VUMA provides multiple standard library modules with VUMA-verified, BD-annotated functions: `fmt` (formatting), `math` (mathematical functions), `crypto` (cryptographic primitives), `string` (string/memory operations), and more.
   826вҶ’   826вҶ’
   827вҶ’   827вҶ’### 13.1 The `fmt` Module (String Formatting)
   828вҶ’   828вҶ’
   829вҶ’   829вҶ’The `fmt` module provides VUMA-verified string formatting primitives вҖ” the equivalent of C's `printf`/`sprintf` family. These functions convert numeric values to text representations in various bases, format floating-point numbers, pad strings, and write formatted output into byte buffers.
   830вҶ’   830вҶ’
   831вҶ’   831вҶ’All `fmt` functions are pure (CapD: { Read, Compare }) except for buffer-writing functions which also declare { Write }.
   832вҶ’   832вҶ’
   833вҶ’   833вҶ’#### Integer Formatting
   834вҶ’   834вҶ’
   835вҶ’   835вҶ’```vuma
   836вҶ’   836вҶ’// Format a signed integer in the given base (2вҖ“36) with minimum width
   837вҶ’   837вҶ’s = fmt::format_int(value: i64, base: u32, width: u32);
   838вҶ’   838вҶ’
   839вҶ’   839вҶ’// Format an unsigned integer in the given base with minimum width
   840вҶ’   840вҶ’s = fmt::format_uint(value: u64, base: u32, width: u32);
   841вҶ’   841вҶ’
   842вҶ’   842вҶ’// Convenience wrappers for common bases:
   843вҶ’   843вҶ’s = fmt::format_hex(value: u64, width: u32);     // base 16, e.g. "0x000000ff"
   844вҶ’   844вҶ’s = fmt::format_binary(value: u64, width: u32);  // base 2, e.g. "11111111"
   845вҶ’   845вҶ’s = fmt::format_octal(value: u64, width: u32);   // base 8, e.g. "377"
   846вҶ’   846вҶ’
   847вҶ’   847вҶ’// Format a pointer address as "0x" + 16-char hex
   848вҶ’   848вҶ’s = fmt::format_pointer(addr: u64);
   849вҶ’   849вҶ’```
   850вҶ’   850вҶ’
   851вҶ’   851вҶ’**Key behaviors:**
   852вҶ’   852вҶ’- `base` must be 2вҖ“36; values outside that range are treated as base 10.
   853вҶ’   853вҶ’- `width` specifies the minimum number of digits; the output is left-padded with `'0'` to reach this width.
   854вҶ’   854вҶ’- For `format_int`, a negative value is prefixed with `'-'`; the sign occupies one character but is not counted as a digit for width purposes.
   855вҶ’   855вҶ’- `format_hex` uses lowercase hex digits (`a`вҖ“`f`).
   856вҶ’   856вҶ’
   857вҶ’   857вҶ’**Examples:**
   858вҶ’   858вҶ’
   859вҶ’   859вҶ’```vuma
   860вҶ’   860вҶ’// Decimal formatting
   861вҶ’   861вҶ’s = fmt::format_int(42, 10, 0);          // "42"
   862вҶ’   862вҶ’s = fmt::format_int(-7, 10, 4);          // "-0007"
   863вҶ’   863вҶ’s = fmt::format_uint(255, 10, 0);        // "255"
   864вҶ’   864вҶ’
   865вҶ’   865вҶ’// Hex formatting
   866вҶ’   866вҶ’s = fmt::format_hex(255, 4);             // "00ff"
   867вҶ’   867вҶ’s = fmt::format_hex(0xDEADBEEF, 8);      // "deadbeef"
   868вҶ’   868вҶ’
   869вҶ’   869вҶ’// Binary formatting
   870вҶ’   870вҶ’s = fmt::format_binary(5, 8);            // "00000101"
   871вҶ’   871вҶ’
   872вҶ’   872вҶ’// Pointer formatting
   873вҶ’   873вҶ’s = fmt::format_pointer(0x7FFE4321);     // "0x000000007ffe4321"
   874вҶ’   874вҶ’```
   875вҶ’   875вҶ’
   876вҶ’   876вҶ’#### Floating-Point Formatting
   877вҶ’   877вҶ’
   878вҶ’   878вҶ’```vuma
   879вҶ’   879вҶ’// Format a float with exactly `precision` digits after the decimal point
   880вҶ’   880вҶ’s = fmt::format_float(value: f64, precision: u32);
   881вҶ’   881вҶ’```
   882вҶ’   882вҶ’
   883вҶ’   883вҶ’**Special values:**
   884вҶ’   884вҶ’- NaN вҶ’ `"nan"`
   885вҶ’   885вҶ’- Positive infinity вҶ’ `"inf"`
   886вҶ’   886вҶ’- Negative infinity вҶ’ `"-inf"`
   887вҶ’   887вҶ’- Zero with precision 0 вҶ’ `"0"`, with precision > 0 вҶ’ `"0.00..."`
   888вҶ’   888вҶ’
   889вҶ’   889вҶ’**Examples:**
   890вҶ’   890вҶ’
   891вҶ’   891вҶ’```vuma
   892вҶ’   892вҶ’s = fmt::format_float(3.14159, 2);       // "3.14"
   893вҶ’   893вҶ’s = fmt::format_float(3.14159, 6);       // "3.141590"
   894вҶ’   894вҶ’s = fmt::format_float(0.0, 0);           // "0"
   895вҶ’   895вҶ’s = fmt::format_float(0.0/0.0, 2);       // "nan"
   896вҶ’   896вҶ’```
   897вҶ’   897вҶ’
   898вҶ’   898вҶ’#### String Padding and Joining
   899вҶ’   899вҶ’
   900вҶ’   900вҶ’```vuma
   901вҶ’   901вҶ’// Left-pad a string to `width` characters with `fill`
   902вҶ’   902вҶ’s = fmt::pad_left(s: &str, width: u32, fill: char);
   903вҶ’   903вҶ’
   904вҶ’   904вҶ’// Right-pad a string to `width` characters with `fill`
   905вҶ’   905вҶ’s = fmt::pad_right(s: &str, width: u32, fill: char);
   906вҶ’   906вҶ’
   907вҶ’   907вҶ’// Join string slices with a separator
   908вҶ’   908вҶ’s = fmt::join(parts: &[&str], separator: &str);
   909вҶ’   909вҶ’```
   910вҶ’   910вҶ’
   911вҶ’   911вҶ’**Examples:**
   912вҶ’   912вҶ’
   913вҶ’   913вҶ’```vuma
   914вҶ’   914вҶ’s = fmt::pad_left("42", 6, '0');         // "000042"
   915вҶ’   915вҶ’s = fmt::pad_left("hello", 3, ' ');      // "hello" (already wider)
   916вҶ’   916вҶ’s = fmt::pad_right("abc", 6, '-');        // "abc---"
   917вҶ’   917вҶ’s = fmt::join(["a", "b", "c"], ", ");     // "a, b, c"
   918вҶ’   918вҶ’```

  1036вҶ’  1036вҶ’
  1037вҶ’  1037вҶ’The `crypto` module provides SHA-256 constants, logical functions, byte access helpers, and constant-time operations:
  1038вҶ’  1038вҶ’
  1039вҶ’  1039вҶ’```vuma
  1040вҶ’  1040вҶ’// SHA-256 logical functions
  1041вҶ’  1041вҶ’result: u32 = crypto::sha256_ch(x: u32, y: u32, z: u32);
  1042вҶ’  1042вҶ’result: u32 = crypto::sha256_maj(a: u32, b: u32, c: u32);
  1043вҶ’  1043вҶ’result: u32 = crypto::sha256_big_sigma0(x: u32);
  1044вҶ’  1044вҶ’result: u32 = crypto::sha256_big_sigma1(x: u32);
  1045вҶ’  1045вҶ’result: u32 = crypto::sha256_small_sigma0(x: u32);
  1046вҶ’  1046вҶ’result: u32 = crypto::sha256_small_sigma1(x: u32);
  1047вҶ’  1047вҶ’
  1048вҶ’  1048вҶ’// Big-endian byte access
  1049вҶ’  1049вҶ’val: u32 = crypto::sha256_read_u32_be(buf: Address, offset: u64);
  1050вҶ’  1050вҶ’crypto::sha256_write_u32_be(buf: Address, offset: u64, val: u32);
  1051вҶ’  1051вҶ’
  1052вҶ’  1052вҶ’// Constant-time operations (no data-dependent branches вҖ” safe against timing attacks)
  1053вҶ’  1053вҶ’equal: u32 = crypto::ct_eq_u32(a: u32, b: u32);     // 1 if equal, 0 otherwise
  1054вҶ’  1054вҶ’neq: u32 = crypto::ct_ne_u32(a: u32, b: u32);        // 1 if not equal
  1055вҶ’  1055вҶ’selected: u32 = crypto::ct_select_u32(flag: u32, a: u32, b: u32);  // a if flag=1, b if flag=0
  1056вҶ’  1056вҶ’lt: u32 = crypto::ct_lt_u32(a: u32, b: u32);         // 1 if a < b
  1057вҶ’  1057вҶ’gte: u32 = crypto::ct_gte_u32(a: u32, b: u32);       // 1 if a >= b
  1058вҶ’  1058вҶ’
  1059вҶ’  1059вҶ’// SHA-256 constants
  1060вҶ’  1060вҶ’// crypto::SHA256_K вҖ” 64 u32 round constants
  1061вҶ’  1061вҶ’// crypto::SHA256_H вҖ” 8 u32 initial hash values
  1062вҶ’  1062вҶ’```
  1063вҶ’  1063вҶ’
  1064вҶ’  1064вҶ’**Critical:** Constant-time functions execute in the same number of cycles regardless of input values. Use them whenever comparing secret data to prevent timing side-channels. All 8 backends implement these with branchless codegen.
  1065вҶ’  1065вҶ’
  1066вҶ’  1066вҶ’### 13.4 The `string` Module (String and Memory Operations)

  1096вҶ’  1096вҶ’## 14. Foreign Function Interface (FFI)
  1097вҶ’  1097вҶ’
  1098вҶ’  1098вҶ’VUMA supports calling external C functions through the `extern "C"` block syntax. This allows VUMA programs to interface with the system's C library, Linux syscalls, and other native code.
  1099вҶ’  1099вҶ’
  1100вҶ’  1100вҶ’### 14.1 The `extern "C"` Block
  1101вҶ’  1101вҶ’
  1102вҶ’  1102вҶ’Use an `extern "C" { ... }` block at the top level to declare external functions:
  1103вҶ’  1103вҶ’
  1104вҶ’  1104вҶ’```vuma
  1105вҶ’  1105вҶ’extern "C" {
  1106вҶ’  1106вҶ’    fn write(fd: i64, buf: Address, count: i64) -> i64;
  1107вҶ’  1107вҶ’    fn read(fd: i64, buf: Address, count: i64) -> i64;
  1108вҶ’  1108вҶ’    fn exit(code: i64);
  1109вҶ’  1109вҶ’}
  1110вҶ’  1110вҶ’```
  1111вҶ’  1111вҶ’
  1112вҶ’  1112вҶ’**Syntax rules:**
  1113вҶ’  1113вҶ’- The block must appear at the top level (not inside a function).
  1114вҶ’  1114вҶ’- The convention string is `"C"` (the only currently supported convention; `"system"` is reserved).
  1115вҶ’  1115вҶ’- Each function signature ends with `;` (no body вҖ” the implementation is external).
  1116вҶ’  1116вҶ’- Parameters use the same type syntax as regular VUMA functions.
  1117вҶ’  1117вҶ’- Functions without a return type are `void` (like `exit` above).
  1118вҶ’  1118вҶ’
  1119вҶ’  1119вҶ’### 14.2 Calling External Functions
  1120вҶ’  1120вҶ’
  1121вҶ’  1121вҶ’Once declared, extern functions are called like any other VUMA function:
  1122вҶ’  1122вҶ’
  1123вҶ’  1123вҶ’```vuma
  1124вҶ’  1124вҶ’fn main() -> i64 {
  1125вҶ’  1125вҶ’    let msg_addr: Address = 0x400000;
  1126вҶ’  1126вҶ’    let msg_len: i64 = 21;
  1127вҶ’  1127вҶ’    let result: i64 = write(1, msg_addr, msg_len);
  1128вҶ’  1128вҶ’    exit(0);
  1129вҶ’  1129вҶ’    return result;   // unreachable вҖ” exit() does not return
  1130вҶ’  1130вҶ’}
  1131вҶ’  1131вҶ’```
  1132вҶ’  1132вҶ’
  1133вҶ’  1133вҶ’### 14.3 How It Works (Codegen Details)
  1134вҶ’  1134вҶ’
  1135вҶ’  1135вҶ’When the compiler encounters a call to an `extern` function:
  1136вҶ’  1136вҶ’
  1137вҶ’  1137вҶ’1. **No local `BL` instruction** вҖ” Instead of emitting a direct branch, the codegen produces a **relocation** entry in the `.rela.text` section of the ELF object file.
  1138вҶ’  1138вҶ’2. **Symbol table** вҖ” The extern function is recorded as an `SHN_UNDEF` symbol in the ELF symbol table, indicating it must be resolved by the linker.
  1139вҶ’  1139вҶ’3. **Linking** вҖ” The system linker (`ld`) resolves the relocations against the C library or other object files.
  1140вҶ’  1140вҶ’
  1141вҶ’  1141вҶ’**Compilation pipeline for FFI programs:**
  1142вҶ’  1142вҶ’
  1143вҶ’  1143вҶ’```bash
  1144вҶ’  1144вҶ’# Compile VUMA source to a relocatable object (.o)
  1145вҶ’  1145вҶ’vuma compile --format obj --target aarch64 ffi_demo.vuma -o ffi_demo.o
  1146вҶ’  1146вҶ’
  1147вҶ’  1147вҶ’# Link with the C library
  1148вҶ’  1148вҶ’ld -o ffi_demo ffi_demo.o -lc
  1149вҶ’  1149вҶ’```
  1150вҶ’  1150вҶ’
  1151вҶ’  1151вҶ’### 14.4 Built-in Syscall Bindings
  1152вҶ’  1152вҶ’
  1153вҶ’  1153вҶ’The VUMA compiler includes built-in bindings for common Linux syscalls and C library functions. These are available without explicit `extern` declarations when compiling for bare-metal:
  1154вҶ’  1154вҶ’
  1155вҶ’  1155вҶ’**Linux syscalls:**
  1156вҶ’  1156вҶ’- `write(fd, buf, count)` вҖ” write bytes to a file descriptor
  1157вҶ’  1157вҶ’- `read(fd, buf, count)` вҖ” read bytes from a file descriptor
  1158вҶ’  1158вҶ’- `exit(code)` вҖ” terminate the process
  1159вҶ’  1159вҶ’- `mmap(addr, len, prot, flags, fd, offset)` вҖ” map memory
  1160вҶ’  1160вҶ’- `munmap(addr, len)` вҖ” unmap memory
  1161вҶ’  1161вҶ’- `brk(addr)` вҖ” change data segment size
  1162вҶ’  1162вҶ’
  1163вҶ’  1163вҶ’**C library functions:**
  1164вҶ’  1164вҶ’- `memcpy(dst, src, n)` вҖ” copy memory
  1165вҶ’  1165вҶ’- `memset(dst, c, n)` вҖ” fill memory
  1166вҶ’  1166вҶ’- `malloc(size)` вҖ” allocate memory
  1167вҶ’  1167вҶ’- `free(ptr)` вҖ” free memory
  1168вҶ’  1168вҶ’
  1169вҶ’  1169вҶ’### 14.5 Pitfalls for FFI
  1170вҶ’  1170вҶ’
  1171вҶ’  1171вҶ’- **No type checking across the boundary**: The compiler cannot verify the types of external C functions. If you declare `fn write(fd: i32, ...)` but the C function expects `i64`, the behavior is undefined.
  1172вҶ’  1172вҶ’- **Calling convention must match**: The `"C"` convention uses the platform's standard C ABI (System V AMD64 on Linux/x86_64, AAPCS64 on AArch64, etc.).
  1173вҶ’  1173вҶ’- **Linking is required**: FFI programs cannot run as standalone ET_REL objects. They must be linked with `ld` or `gcc`.
  1174вҶ’  1174вҶ’- **E037 relocation errors**: If a relocation cannot be resolved (e.g., symbol not found, overflow), the linker emits error E037.
  1175вҶ’  1175вҶ’

  1178вҶ’  1178вҶ’## 15. Atomic Operations
  1179вҶ’  1179вҶ’
  1180вҶ’  1180вҶ’VUMA provides three atomic memory operations for concurrent programming. These compile to target-specific atomic instructions.
  1181вҶ’  1181вҶ’
  1182вҶ’  1182вҶ’### 15.1 AtomicLoad
  1183вҶ’  1183вҶ’
  1184вҶ’  1184вҶ’Atomically loads a value from memory with acquire semantics.
  1185вҶ’  1185вҶ’
  1186вҶ’  1186вҶ’```vuma
  1187вҶ’  1187вҶ’value = AtomicLoad(addr);
  1188вҶ’  1188вҶ’```
  1189вҶ’  1189вҶ’
  1190вҶ’  1190вҶ’**Target-specific lowering:**
  1191вҶ’  1191вҶ’- AArch64: `LDAXR`
  1192вҶ’  1192вҶ’- x86_64: `LOCK` prefix or plain `MOV` (x86 is already atomic for aligned loads)
  1193вҶ’  1193вҶ’- RISC-V: `LR.D`
  1194вҶ’  1194вҶ’
  1195вҶ’  1195вҶ’### 15.2 AtomicStore
  1196вҶ’  1196вҶ’
  1197вҶ’  1197вҶ’Atomically stores a value to memory with release semantics.
  1198вҶ’  1198вҶ’
  1199вҶ’  1199вҶ’```vuma
  1200вҶ’  1200вҶ’AtomicStore(value, addr);
  1201вҶ’  1201вҶ’```
  1202вҶ’  1202вҶ’
  1203вҶ’  1203вҶ’**Target-specific lowering:**
  1204вҶ’  1204вҶ’- AArch64: `STLXR`
  1205вҶ’  1205вҶ’- x86_64: `LOCK` prefix or plain `MOV`
  1206вҶ’  1206вҶ’- RISC-V: `SC.D`
  1207вҶ’  1207вҶ’
  1208вҶ’  1208вҶ’### 15.3 AtomicCas (Compare-and-Swap)
  1209вҶ’  1209вҶ’
  1210вҶ’  1210вҶ’Atomically compares the value at `addr` with `expected`. If equal, writes `desired` to `addr`. Returns the old value at `addr`.
  1211вҶ’  1211вҶ’
  1212вҶ’  1212вҶ’```vuma
  1213вҶ’  1213вҶ’old_value = AtomicCas(addr, expected, desired);
  1214вҶ’  1214вҶ’```
  1215вҶ’  1215вҶ’
  1216вҶ’  1216вҶ’**Target-specific lowering:**
  1217вҶ’  1217вҶ’- AArch64: `LDAXR` / `CMP` / `B.NE` / `STLXR` loop
  1218вҶ’  1218вҶ’- x86_64: `LOCK CMPXCHG`
  1219вҶ’  1219вҶ’- RISC-V: `LR.D` / `BNE` / `SC.D` loop
  1220вҶ’  1220вҶ’
  1221вҶ’  1221вҶ’### 15.4 Example: Atomic Counter
  1222вҶ’  1222вҶ’
  1223вҶ’  1223вҶ’```vuma
  1224вҶ’  1224вҶ’fn atomic_increment(counter: Address) -> u64 {
  1225вҶ’  1225вҶ’    old: u64 = AtomicLoad(counter);
  1226вҶ’  1226вҶ’    loop {
  1227вҶ’  1227вҶ’        new_val: u64 = old + 1;
  1228вҶ’  1228вҶ’        prev: u64 = AtomicCas(counter, old, new_val);
  1229вҶ’  1229вҶ’        if prev == old {
  1230вҶ’  1230вҶ’            return new_val;
  1231вҶ’  1231вҶ’        }
  1232вҶ’  1232вҶ’        old = prev;
  1233вҶ’  1233вҶ’    }
  1234вҶ’  1234вҶ’}
  1235вҶ’  1235вҶ’```
  1236вҶ’  1236вҶ’
  1237вҶ’  1237вҶ’**Note:** Atomic operations are only available on 64-bit backends (x86_64, AArch64, RISC-V 64). The ARM32 and Wasm32 backends have limited atomic support.
  1238вҶ’  1238вҶ’

  1273вҶ’  1273вҶ’| ARM32 | SP (13) | LR (14) | FP (11) | 2 | вҲ’4 |
  1274вҶ’  1274вҶ’| MIPS64 | SP (29) | RA (31) | вҖ” | 4 | вҲ’8 |
  1275вҶ’  1275вҶ’| PPC64 | R1 | LR (65) | вҖ” | 4 | вҲ’8 |
  1276вҶ’  1276вҶ’| LoongArch64 | r3 (SP) | r1 (RA) | r22 (FP) | 4 | вҲ’8 |
  1277вҶ’  1277вҶ’| Wasm32 | вҖ” | вҖ” | вҖ” | 1 | вҖ” |

  1280вҶ’  1280вҶ’

  1300вҶ’  1300вҶ’## 17. Common Patterns
  1301вҶ’  1301вҶ’
  1302вҶ’  1302вҶ’### Pattern 1: SHA256d (Double SHA-256)
  1303вҶ’  1303вҶ’
  1304вҶ’  1304вҶ’This is the canonical VUMA program. It exercises all major language features: functions, memory, pointer arithmetic, bitwise operations, u32 masking, and loops.
  1305вҶ’  1305вҶ’
  1306вҶ’  1306вҶ’```vuma
  1307вҶ’  1307вҶ’fn rotr32(x: u32, n: u32) -> u32 {
  1308вҶ’  1308вҶ’    return ((x >> n) | (x << (32 - n))) & 4294967295;
  1309вҶ’  1309вҶ’}
  1310вҶ’  1310вҶ’
  1311вҶ’  1311вҶ’fn ch(x: u32, y: u32, z: u32) -> u32 {
  1312вҶ’  1312вҶ’    return (x & y) ^ ((x ^ 4294967295) & z);
  1313вҶ’  1313вҶ’}
  1314вҶ’  1314вҶ’
  1315вҶ’  1315вҶ’fn maj(a: u32, b: u32, c: u32) -> u32 {
  1316вҶ’  1316вҶ’    return (a & b) ^ (a & c) ^ (b & c);
  1317вҶ’  1317вҶ’}
  1318вҶ’  1318вҶ’
  1319вҶ’  1319вҶ’fn big_sigma0(x: u32) -> u32 {
  1320вҶ’  1320вҶ’    return rotr32(x, 2) ^ rotr32(x, 13) ^ rotr32(x, 22);
  1321вҶ’  1321вҶ’}
  1322вҶ’  1322вҶ’
  1323вҶ’  1323вҶ’fn big_sigma1(x: u32) -> u32 {
  1324вҶ’  1324вҶ’    return rotr32(x, 6) ^ rotr32(x, 11) ^ rotr32(x, 25);
  1325вҶ’  1325вҶ’}
  1326вҶ’  1326вҶ’
  1327вҶ’  1327вҶ’fn small_sigma0(x: u32) -> u32 {
  1328вҶ’  1328вҶ’    return (rotr32(x, 7) ^ rotr32(x, 18) ^ (x >> 3)) & 4294967295;
  1329вҶ’  1329вҶ’}
  1330вҶ’  1330вҶ’
  1331вҶ’  1331вҶ’fn small_sigma1(x: u32) -> u32 {
  1332вҶ’  1332вҶ’    return (rotr32(x, 17) ^ rotr32(x, 19) ^ (x >> 10)) & 4294967295;
  1333вҶ’  1333вҶ’}
  1334вҶ’  1334вҶ’
  1335вҶ’  1335вҶ’fn write_u32_be(buf: Address, offset: u64, val: u32) {
  1336вҶ’  1336вҶ’    *(buf + offset) = (val >> 24) & 255;
  1337вҶ’  1337вҶ’    *(buf + offset + 1) = (val >> 16) & 255;
  1338вҶ’  1338вҶ’    *(buf + offset + 2) = (val >> 8) & 255;
  1339вҶ’  1339вҶ’    *(buf + offset + 3) = val & 255;
  1340вҶ’  1340вҶ’}
  1341вҶ’  1341вҶ’
  1342вҶ’  1342вҶ’fn read_u32_be(buf: Address, offset: u64) -> u32 {
  1343вҶ’  1343вҶ’    b0: u32 = *(buf + offset);
  1344вҶ’  1344вҶ’    b1: u32 = *(buf + offset + 1);
  1345вҶ’  1345вҶ’    b2: u32 = *(buf + offset + 2);
  1346вҶ’  1346вҶ’    b3: u32 = *(buf + offset + 3);
  1347вҶ’  1347вҶ’    return ((b0 << 24) | (b1 << 16) | (b2 << 8) | b3) & 4294967295;
  1348вҶ’  1348вҶ’}
  1349вҶ’  1349вҶ’
  1350вҶ’  1350вҶ’fn main() -> i32 {
  1351вҶ’  1351вҶ’    msg = allocate(3);
  1352вҶ’  1352вҶ’    *(msg + 0) = 97;
  1353вҶ’  1353вҶ’    *(msg + 1) = 98;
  1354вҶ’  1354вҶ’    *(msg + 2) = 99;
  1355вҶ’  1355вҶ’    // ... SHA-256 compression ...
  1356вҶ’  1356вҶ’    free(msg);
  1357вҶ’  1357вҶ’    return 0;
  1358вҶ’  1358вҶ’}
  1359вҶ’  1359вҶ’```
  1360вҶ’  1360вҶ’
  1361вҶ’  1361вҶ’### Pattern 2: Memory Buffer (Read/Write Bytes)
  1362вҶ’  1362вҶ’
  1363вҶ’  1363вҶ’```vuma
  1364вҶ’  1364вҶ’fn main() -> i32 {
  1365вҶ’  1365вҶ’    buf = allocate(16);
  1366вҶ’  1366вҶ’
  1367вҶ’  1367вҶ’    // Write bytes
  1368вҶ’  1368вҶ’    *(buf + 0) = 0x4f;
  1369вҶ’  1369вҶ’    *(buf + 1) = 0x8b;
  1370вҶ’  1370вҶ’
  1371вҶ’  1371вҶ’    // Read bytes back
  1372вҶ’  1372вҶ’    b0: u32 = *(buf + 0);
  1373вҶ’  1373вҶ’    b1: u32 = *(buf + 1);
  1374вҶ’  1374вҶ’
  1375вҶ’  1375вҶ’    free(buf);
  1376вҶ’  1376вҶ’    return 0;
  1377вҶ’  1377вҶ’}
  1378вҶ’  1378вҶ’```
  1379вҶ’  1379вҶ’
  1380вҶ’  1380вҶ’### Pattern 3: Byte Manipulation (Extract/Insert)
  1381вҶ’  1381вҶ’
  1382вҶ’  1382вҶ’```vuma
  1383вҶ’  1383вҶ’// Extract byte n from a u32 (big-endian)
  1384вҶ’  1384вҶ’fn get_byte(val: u32, n: u32) -> u32 {
  1385вҶ’  1385вҶ’    return (val >> (n * 8)) & 255;
  1386вҶ’  1386вҶ’}
  1387вҶ’  1387вҶ’
  1388вҶ’  1388вҶ’// Set byte n in a u32 (big-endian)
  1389вҶ’  1389вҶ’fn set_byte(val: u32, n: u32, byte: u32) -> u32 {
  1390вҶ’  1390вҶ’    shift: u32 = n * 8;
  1391вҶ’  1391вҶ’    mask: u32 = 255 << shift;
  1392вҶ’  1392вҶ’    return (val & (mask ^ 4294967295)) | ((byte & 255) << shift) & 4294967295;
  1393вҶ’  1393вҶ’}
  1394вҶ’  1394вҶ’```
  1395вҶ’  1395вҶ’
  1396вҶ’  1396вҶ’### Pattern 4: Memory Copy
  1397вҶ’  1397вҶ’
  1398вҶ’  1398вҶ’```vuma
  1399вҶ’  1399вҶ’fn copy_bytes(dst: Address, src: Address, len: u64) {
  1400вҶ’  1400вҶ’    i: u64 = 0;
  1401вҶ’  1401вҶ’    for i in 0..len {
  1402вҶ’  1402вҶ’        *(dst + i) = *(src + i);
  1403вҶ’  1403вҶ’    }
  1404вҶ’  1404вҶ’}
  1405вҶ’  1405вҶ’```
  1406вҶ’  1406вҶ’
  1407вҶ’  1407вҶ’### Pattern 5: Memory Set (Fill with Value)
  1408вҶ’  1408вҶ’
  1409вҶ’  1409вҶ’```vuma
  1410вҶ’  1410вҶ’fn mem_set(buf: Address, len: u64, val: u32) {
  1411вҶ’  1411вҶ’    i: u64 = 0;
  1412вҶ’  1412вҶ’    for i in 0..len {
  1413вҶ’  1413вҶ’        *(buf + i) = val & 255;
  1414вҶ’  1414вҶ’    }
  1415вҶ’  1415вҶ’}
  1416вҶ’  1416вҶ’```
  1417вҶ’  1417вҶ’
  1418вҶ’  1418вҶ’### Pattern 6: Array-style Access via Stride
  1419вҶ’  1419вҶ’
  1420вҶ’  1420вҶ’When storing an array of u32 values in a byte buffer, each element occupies 4 bytes (stride = 4):
  1421вҶ’  1421вҶ’
  1422вҶ’  1422вҶ’```vuma
  1423вҶ’  1423вҶ’fn w_store(w_base: Address, idx: u64, val: u32) {
  1424вҶ’  1424вҶ’    write_u32_be(w_base, idx * 4, val);
  1425вҶ’  1425вҶ’}
  1426вҶ’  1426вҶ’
  1427вҶ’  1427вҶ’fn w_load(w_base: Address, idx: u64) -> u32 {
  1428вҶ’  1428вҶ’    return read_u32_be(w_base, idx * 4);
  1429вҶ’  1429вҶ’}
  1430вҶ’  1430вҶ’```
  1431вҶ’  1431вҶ’
  1432вҶ’  1432вҶ’### Pattern 7: SHA-256 Compression Round
  1433вҶ’  1433вҶ’
  1434вҶ’  1434вҶ’```vuma
  1435вҶ’  1435вҶ’for i in 0..64 {
  1436вҶ’  1436вҶ’    ki: u32 = read_u32_be(k, i * 4);
  1437вҶ’  1437вҶ’    wi: u32 = w_load(w, i);
  1438вҶ’  1438вҶ’    t1: u32 = (h + big_sigma1(e) + ch(e, f, g) + ki + wi) & 4294967295;
  1439вҶ’  1439вҶ’    t2: u32 = (big_sigma0(a) + maj(a, b, c)) & 4294967295;
  1440вҶ’  1440вҶ’    h = g;
  1441вҶ’  1441вҶ’    g = f;
  1442вҶ’  1442вҶ’    f = e;
  1443вҶ’  1443вҶ’    e = (d + t1) & 4294967295;
  1444вҶ’  1444вҶ’    d = c;
  1445вҶ’  1445вҶ’    c = b;
  1446вҶ’  1446вҶ’    b = a;
  1447вҶ’  1447вҶ’    a = (t1 + t2) & 4294967295;
  1448вҶ’  1448вҶ’}
  1449вҶ’  1449вҶ’```

  1452вҶ’  1452вҶ’
  1453вҶ’  1453вҶ’## 18. Pitfalls
  1454вҶ’  1454вҶ’
  1455вҶ’  1455вҶ’This section documents the most common mistakes when writing VUMA code. **Read this section carefully.**
  1456вҶ’  1456вҶ’
  1457вҶ’  1457вҶ’### Pitfall 1: u32 Arithmetic Without Masking
  1458вҶ’  1458вҶ’
  1459вҶ’  1459вҶ’**Problem**: VUMA uses 64-bit registers. When two u32 values are added, the 64-bit result can have bits set above bit 31 (the carry). Subsequent operations on this value (especially right shifts) will produce wrong results.
  1460вҶ’  1460вҶ’
  1461вҶ’  1461вҶ’```vuma
  1462вҶ’  1462вҶ’// WRONG
  1463вҶ’  1463вҶ’sum: u32 = a + b;        // upper 32 bits may have carry
  1464вҶ’  1464вҶ’rotated: u32 = sum >> 4;  // wrong! shift includes garbage bits
  1465вҶ’  1465вҶ’
  1466вҶ’  1466вҶ’// CORRECT
  1467вҶ’  1467вҶ’sum: u32 = (a + b) & 4294967295;  // clear upper 32 bits
  1468вҶ’  1468вҶ’rotated: u32 = sum >> 4;          // now correct
  1469вҶ’  1469вҶ’```
  1470вҶ’  1470вҶ’
  1471вҶ’  1471вҶ’**Rule**: After every arithmetic operation (`+`, `-`, `*`) on u32 values, apply `& 4294967295`.
  1472вҶ’  1472вҶ’
  1473вҶ’  1473вҶ’### Pitfall 2: Using `~` for Bitwise NOT
  1474вҶ’  1474вҶ’
  1475вҶ’  1475вҶ’**Problem**: The `~` operator inverts all 64 bits. For a u32 value in a 64-bit register, this means bits 32-63 are also inverted, producing `0xFFFFFFFF????????` instead of `0x00000000????????`.
  1476вҶ’  1476вҶ’
  1477вҶ’  1477вҶ’```vuma
  1478вҶ’  1478вҶ’// WRONG
  1479вҶ’  1479вҶ’not_x = ~x;  // produces 0xFFFFFFFF???????? for u32 x
  1480вҶ’  1480вҶ’
  1481вҶ’  1481вҶ’// CORRECT
  1482вҶ’  1482вҶ’not_x: u32 = x ^ 4294967295;  // flips only lower 32 bits
  1483вҶ’  1483вҶ’```
  1484вҶ’  1484вҶ’
  1485вҶ’  1485вҶ’**Rule**: Never use `~` for u32 bitwise NOT. Always use `x ^ 4294967295`.
  1486вҶ’  1486вҶ’
  1487вҶ’  1487вҶ’### Pitfall 3: Left Shift Without Masking
  1488вҶ’  1488вҶ’
  1489вҶ’  1489вҶ’**Problem**: Left-shifting a u32 value by N bits in a 64-bit register moves bits into the upper 32 bits.
  1490вҶ’  1490вҶ’
  1491вҶ’  1491вҶ’```vuma
  1492вҶ’  1492вҶ’// WRONG
  1493вҶ’  1493вҶ’shifted: u32 = value << 24;  // bits above bit 31 are set
  1494вҶ’  1494вҶ’
  1495вҶ’  1495вҶ’// CORRECT
  1496вҶ’  1496вҶ’shifted: u32 = (value << 24) & 4294967295;  // clear upper 32 bits
  1497вҶ’  1497вҶ’```
  1498вҶ’  1498вҶ’
  1499вҶ’  1499вҶ’**Rule**: After any left shift of a u32 value, apply `& 4294967295`.
  1500вҶ’  1500вҶ’
  1501вҶ’  1501вҶ’### Pitfall 4: Rotate Without Masking
  1502вҶ’  1502вҶ’
  1503вҶ’  1503вҶ’**Problem**: A rotate-right composed from shifts produces bits above bit 31 from the left-shift component.
  1504вҶ’  1504вҶ’
  1505вҶ’  1505вҶ’```vuma
  1506вҶ’  1506вҶ’// WRONG
  1507вҶ’  1507вҶ’rotated: u32 = (x >> n) | (x << (32 - n));  // upper bits polluted
  1508вҶ’  1508вҶ’
  1509вҶ’  1509вҶ’// CORRECT
  1510вҶ’  1510вҶ’rotated: u32 = ((x >> n) | (x << (32 - n))) & 4294967295;
  1511вҶ’  1511вҶ’```
  1512вҶ’  1512вҶ’
  1513вҶ’  1513вҶ’**Rule**: Always mask the result of a rotate operation.
  1514вҶ’  1514вҶ’
  1515вҶ’  1515вҶ’### Pitfall 5: Forgetting to Free Memory
  1516вҶ’  1516вҶ’
  1517вҶ’  1517вҶ’**Problem**: Every `allocate()` must be matched with a `free()`. The IVE checks this, but it is easy to forget in complex control flow.
  1518вҶ’  1518вҶ’
  1519вҶ’  1519вҶ’```vuma
  1520вҶ’  1520вҶ’// WRONG вҖ” memory leak
  1521вҶ’  1521вҶ’fn bad() -> i32 {
  1522вҶ’  1522вҶ’    buf = allocate(64);
  1523вҶ’  1523вҶ’    return 0;  // buf is never freed!
  1524вҶ’  1524вҶ’}
  1525вҶ’  1525вҶ’
  1526вҶ’  1526вҶ’// CORRECT
  1527вҶ’  1527вҶ’fn good() -> i32 {
  1528вҶ’  1528вҶ’    buf = allocate(64);
  1529вҶ’  1529вҶ’    // ... use buf ...
  1530вҶ’  1530вҶ’    free(buf);
  1531вҶ’  1531вҶ’    return 0;
  1532вҶ’  1532вҶ’}
  1533вҶ’  1533вҶ’```
  1534вҶ’  1534вҶ’
  1535вҶ’  1535вҶ’**Rule**: Always pair `allocate()` with `free()`. Free in reverse order of allocation to match the typical stack discipline.
  1536вҶ’  1536вҶ’
  1537вҶ’  1537вҶ’### Pitfall 6: Using `!x` for Bitwise NOT
  1538вҶ’  1538вҶ’
  1539вҶ’  1539вҶ’**Problem**: The `!` operator is logical NOT (converts non-zero to 0 and 0 to 1), not bitwise NOT.
  1540вҶ’  1540вҶ’
  1541вҶ’  1541вҶ’```vuma
  1542вҶ’  1542вҶ’// WRONG вҖ” logical NOT, not bitwise NOT
  1543вҶ’  1543вҶ’not_x = !x;  // produces 0 or 1, not bitwise complement
  1544вҶ’  1544вҶ’
  1545вҶ’  1545вҶ’// CORRECT вҖ” bitwise NOT via XOR
  1546вҶ’  1546вҶ’not_x: u32 = x ^ 4294967295;
  1547вҶ’  1547вҶ’```
  1548вҶ’  1548вҶ’
  1549вҶ’  1549вҶ’**Rule**: Use `x ^ 4294967295` for bitwise NOT, never `!x` or `~x`.
  1550вҶ’  1550вҶ’
  1551вҶ’  1551вҶ’### Pitfall 7: Hex Literals as Byte Values in Stores
  1552вҶ’  1552вҶ’
  1553вҶ’  1553вҶ’**Problem**: When storing a value to memory via `*(ptr + offset) = value`, only the low 8 bits are stored. Make sure you are not accidentally storing a truncated value.
  1554вҶ’  1554вҶ’
  1555вҶ’  1555вҶ’```vuma
  1556вҶ’  1556вҶ’// This stores only the low 8 bits (0xC2)
  1557вҶ’  1557вҶ’*(buf + 0) = 0x4FC2;  // stores 0xC2, NOT 0x4FC2
  1558вҶ’  1558вҶ’
  1559вҶ’  1559вҶ’// To store a specific byte, mask explicitly:
  1560вҶ’  1560вҶ’*(buf + 0) = (0x4FC2 >> 8) & 255;  // stores 0x4F
  1561вҶ’  1561вҶ’*(buf + 1) = 0x4FC2 & 255;        // stores 0xC2
  1562вҶ’  1562вҶ’```
  1563вҶ’  1563вҶ’
  1564вҶ’  1564вҶ’### Pitfall 8: Exit Code Truncation
  1565вҶ’  1565вҶ’
  1566вҶ’  1566вҶ’**Problem**: The process exit code is typically 8 bits (0-255) on Linux. When `main()` returns a u32 or i32, only the low byte is used as the exit code.
  1567вҶ’  1567вҶ’
  1568вҶ’  1568вҶ’```vuma
  1569вҶ’  1569вҶ’fn main() -> i32 {
  1570вҶ’  1570вҶ’    // To exit with code 79 (0x4F):
  1571вҶ’  1571вҶ’    return 79;  // exit code = 79
  1572вҶ’  1572вҶ’}
  1573вҶ’  1573вҶ’```
  1574вҶ’  1574вҶ’
  1575вҶ’  1575вҶ’If you need to verify a specific byte from a u32 computation, mask it and return:
  1576вҶ’  1576вҶ’
  1577вҶ’  1577вҶ’```vuma
  1578вҶ’  1578вҶ’fn main() -> i32 {
  1579вҶ’  1579вҶ’    result: u32 = compute_something();
  1580вҶ’  1580вҶ’    return result & 255;  // return low byte as exit code
  1581вҶ’  1581вҶ’}
  1582вҶ’  1582вҶ’```
  1583вҶ’  1583вҶ’
  1584вҶ’  1584вҶ’### Pitfall 9: Reading Bytes as u32 Without Masking
  1585вҶ’  1585вҶ’
  1586вҶ’  1586вҶ’When you read a byte from memory into a `u32` variable, the value is zero-extended by the load. However, if you then shift and combine bytes, the result must be masked:
  1587вҶ’  1587вҶ’
  1588вҶ’  1588вҶ’```vuma
  1589вҶ’  1589вҶ’// Each byte read is zero-extended to u32, but after shifting
  1590вҶ’  1590вҶ’// and combining, upper bits may be set
  1591вҶ’  1591вҶ’b0: u32 = *(buf + 0);  // e.g., 0x61
  1592вҶ’  1592вҶ’combined: u32 = (b0 << 24);  // 0x61000000 in 64-bit register, but
  1593вҶ’  1593вҶ’                               // might have upper bits
  1594вҶ’  1594вҶ’combined: u32 = (b0 << 24) & 4294967295;  // CORRECT
  1595вҶ’  1595вҶ’```
  1596вҶ’  1596вҶ’
  1597вҶ’  1597вҶ’### Pitfall 10: Range Bounds in For Loops
  1598вҶ’  1598вҶ’
  1599вҶ’  1599вҶ’The range `a..b` is exclusive on the upper bound. `0..64` means 0 through 63, not 0 through 64:
  1600вҶ’  1600вҶ’
  1601вҶ’  1601вҶ’```vuma
  1602вҶ’  1602вҶ’for i in 0..64 {
  1603вҶ’  1603вҶ’    // i goes from 0 to 63 inclusive (64 iterations)
  1604вҶ’  1604вҶ’}
  1605вҶ’  1605вҶ’
  1606вҶ’  1606вҶ’for i in 16..64 {
  1607вҶ’  1607вҶ’    // i goes from 16 to 63 inclusive (48 iterations)
  1608вҶ’  1608вҶ’}
  1609вҶ’  1609вҶ’```
  1610вҶ’  1610вҶ’
  1611вҶ’  1611вҶ’### Pitfall 11: FFI Type Mismatches
  1612вҶ’  1612вҶ’
  1613вҶ’  1613вҶ’**Problem**: The compiler cannot verify that the parameter types declared in an `extern "C"` block match the actual C function signature. A mismatch causes undefined behavior at runtime.
  1614вҶ’  1614вҶ’
  1615вҶ’  1615вҶ’```vuma
  1616вҶ’  1616вҶ’// WRONG: C's write() expects size_t (u64) for count, not i32
  1617вҶ’  1617вҶ’extern "C" {
  1618вҶ’  1618вҶ’    fn write(fd: i64, buf: Address, count: i32) -> i64;  // BUG: count should be i64
  1619вҶ’  1619вҶ’}
  1620вҶ’  1620вҶ’
  1621вҶ’  1621вҶ’// CORRECT: match the C ABI exactly
  1622вҶ’  1622вҶ’extern "C" {
  1623вҶ’  1623вҶ’    fn write(fd: i64, buf: Address, count: i64) -> i64;
  1624вҶ’  1624вҶ’}
  1625вҶ’  1625вҶ’```
  1626вҶ’  1626вҶ’
  1627вҶ’  1627вҶ’**Rule**: Always consult the C header documentation when declaring `extern "C"` functions. Use `i64`/`u64` for pointer-sized C types (`size_t`, `ssize_t`, `intptr_t`).
  1628вҶ’  1628вҶ’
  1629вҶ’  1629вҶ’### Pitfall 12: AtomicCas Retry Loop
  1630вҶ’  1630вҶ’
  1631вҶ’  1631вҶ’**Problem**: `AtomicCas` can fail spuriously (the value at `addr` was changed by another thread between the load and the CAS). You must loop until the CAS succeeds:
  1632вҶ’  1632вҶ’
  1633вҶ’  1633вҶ’```vuma
  1634вҶ’  1634вҶ’// WRONG: single CAS attempt may fail
  1635вҶ’  1635вҶ’fn bad_increment(counter: Address) -> u64 {
  1636вҶ’  1636вҶ’    old: u64 = AtomicLoad(counter);
  1637вҶ’  1637вҶ’    new_val: u64 = old + 1;
  1638вҶ’  1638вҶ’    prev: u64 = AtomicCas(counter, old, new_val);
  1639вҶ’  1639вҶ’    return new_val;  // may return wrong value if CAS failed!
  1640вҶ’  1640вҶ’}
  1641вҶ’  1641вҶ’
  1642вҶ’  1642вҶ’// CORRECT: retry loop
  1643вҶ’  1643вҶ’fn atomic_increment(counter: Address) -> u64 {
  1644вҶ’  1644вҶ’    old: u64 = AtomicLoad(counter);
  1645вҶ’  1645вҶ’    loop {
  1646вҶ’  1646вҶ’        new_val: u64 = old + 1;
  1647вҶ’  1647вҶ’        prev: u64 = AtomicCas(counter, old, new_val);
  1648вҶ’  1648вҶ’        if prev == old {
  1649вҶ’  1649вҶ’            return new_val;
  1650вҶ’  1650вҶ’        }
  1651вҶ’  1651вҶ’        old = prev;
  1652вҶ’  1652вҶ’    }
  1653вҶ’  1653вҶ’}
  1654вҶ’  1654вҶ’```
  1655вҶ’  1655вҶ’
  1656вҶ’  1656вҶ’**Rule**: Always wrap `AtomicCas` in a retry loop. Check that the returned old value matches the expected value.

  1745вҶ’  1745вҶ’
  1746вҶ’  1746вҶ’VUMA compiles to native machine code for 8 architectures plus Wasm. Each target has its own backend that translates the IR to machine code.

  1790вҶ’  1790вҶ’### Target-Specific Considerations
  1791вҶ’  1791вҶ’
  1792вҶ’  1792вҶ’**64-bit targets (x86_64, AArch64, RISC-V 64, MIPS64, PPC64, LoongArch64)**:
  1793вҶ’  1793вҶ’- All use 64-bit general-purpose registers
  1794вҶ’  1794вҶ’- u32 values stored in 64-bit registers **must** be masked with `& 4294967295` after arithmetic and left shifts
  1795вҶ’  1795вҶ’- `Address` is 8 bytes
  1796вҶ’  1796вҶ’- Pointer arithmetic uses 64-bit addition
  1797вҶ’  1797вҶ’
  1798вҶ’  1798вҶ’**32-bit targets (ARM32, Wasm32)**:
  1799вҶ’  1799вҶ’- Use 32-bit registers
  1800вҶ’  1800вҶ’- u32 masking is technically unnecessary but recommended for portability
  1801вҶ’  1801вҶ’- `Address` is 4 bytes
  1802вҶ’  1802вҶ’- Pointer arithmetic uses 32-bit addition
  1803вҶ’  1803вҶ’
  1804вҶ’  1804вҶ’**Wasm32 specifics**:
  1805вҶ’  1805вҶ’- No native ROR/ROL instructions; rotates are synthesized from shift+or sequences
  1806вҶ’  1806вҶ’- Type inference can be tricky; explicit type annotations are important
  1807вҶ’  1807вҶ’- Stack machine architecture (no registers); the backend manages a virtual value stack
  1808вҶ’  1808вҶ’
  1809вҶ’  1809вҶ’### Portability Rules
  1810вҶ’  1810вҶ’
  1811вҶ’  1811вҶ’When writing VUMA code, follow these rules to ensure it works on all 8 targets:
  1812вҶ’  1812вҶ’
  1813вҶ’  1813вҶ’1. **Always mask u32 results** with `& 4294967295` after addition, subtraction, multiplication, and left shift.
  1814вҶ’  1814вҶ’2. **Use `x ^ 4294967295`** for bitwise NOT, never `~x`.
  1815вҶ’  1815вҶ’3. **Compose rotates from shifts** with a final mask: `((x >> n) | (x << (32 - n))) & 4294967295`.
  1816вҶ’  1816вҶ’4. **Use explicit type annotations** for all variable declarations.
  1817вҶ’  1817вҶ’5. **Use `u64` for offsets and lengths**, not `u32`, to avoid truncation on large buffers.
  1818вҶ’  1818вҶ’6. **Always pair `allocate()` with `free()`**.
