    11â†’## Overview
    12â†’
    13â†’This document specifies the verification algorithms used by the Inference and Verification Engine (IVE) to prove or refute the five global memory-safety invariants of the Verified-Unsafe Memory Access (VUMA) model. Each algorithm operates on the **Memory State Graph (MSG)** â€” a formal model of the program's entire memory behavior â€” and produces for each invariant either a proof that the invariant holds or a counterexample demonstrating a violation.
    14â†’
    15â†’The MSG captures every allocation point, every pointer derivation chain, every deallocation point, every concurrent access with its synchronization ordering, and every cast or reinterpretation with its representation descriptor. The algorithms below describe how the IVE systematically verifies each of the five invariants â€” Liveness, Exclusivity, Interpretation, Origin, and Cleanup â€” against this graph, along with the overall pipeline that orchestrates them and the incremental verification strategy that minimizes rework on program edits.
    16â†’
    17â†’---
    18â†’
    19â†’## 1. Liveness Verification Algorithm
    20â†’
    21â†’### 1.1 Purpose and Intuition
    22â†’
    23â†’The liveness invariant asserts that every memory access targets a region that is allocated at the program point where the access occurs. This is the VUMA analogue of the "use-after-free" check, but it operates on a far richer model than the borrow checker's local lifetime analysis. Where the borrow checker reasons about lexical scopes and ownership transfers within a single function, the liveness verifier reasons about the entire program's allocation lifecycle across all execution paths. A region is "live" from its allocation point up to (but not including) its deallocation point. An access that occurs after the deallocation point on any reachable execution path constitutes a liveness violation.
    24â†’
    25â†’### 1.2 Input
    26â†’
    27â†’- **MSG** containing:
    28â†’  - `regions`: set of all memory regions, each with `alloc_point`, `free_point` (optional), and `address_range`
    29â†’  - `accesses`: set of all memory accesses, each with `target` (address expression), `program_point`, `kind` (Read/Write/Execute)
    30â†’  - `derivations`: set of pointer derivations linking addresses to their source regions
    31â†’  - `cfg`: control flow graph of the program for path-sensitive reasoning
    32â†’
    33â†’### 1.3 Algorithm
    34â†’
    35â†’```
    36â†’function VERIFY_LIVENESS(msg: MSG) -> Map<Access, LivenessResult>:
    37â†’    results = {}
    38â†’    for each access a in msg.accesses:
    39â†’        // Step 1: Determine which region the access targets
    40â†’        target_region = RESOLVE_BASE_REGION(a.target, msg.derivations)
    41â†’        if target_region is None:
    42â†’            results[a] = LivenessResult(Violated,
    43â†’                reason="Access targets unresolvable address",
    44â†’                counterexample=a.target)
    45â†’            continue
    46â†’
    47â†’        // Step 2: Get allocation and deallocation points
    48â†’        alloc_pp = target_region.alloc_point
    49â†’        free_pp  = target_region.free_point   // None if never freed
    50â†’
    51â†’        // Step 3: Trivial case â€” region never freed
    52â†’        if free_pp is None:
    53â†’            results[a] = LivenessResult(Proven,
    54â†’                reason="Region never freed; liveness holds trivially")
    55â†’            continue
    56â†’
    57â†’        // Step 4: Path-sensitive liveness check
    58â†’        if IS_REACHABLE(cfg, alloc_pp, a.program_point) and
    59â†’           not IS_REACHABLE_ON_ANY_PATH_THROUGH(cfg, free_pp, a.program_point):
    60â†’            // The access is reachable from alloc but NOT reachable
    61â†’            // by any path that goes through free first
    62â†’            results[a] = LivenessResult(Proven,
    63â†’                reason="Access occurs on paths where region is still live")
    64â†’        else if IS_REACHABLE_ON_ANY_PATH_THROUGH(cfg, free_pp, a.program_point):
    65â†’            // There exists at least one path: alloc -> free -> access
    66â†’            violating_path = FIND_PATH(cfg, alloc_pp, free_pp, a.program_point)
    67â†’            results[a] = LivenessResult(Violated,
    68â†’                reason="Use-after-free: access occurs after free on reachable path",
    69â†’                counterexample=violating_path)
    70â†’        else:
    71â†’            // Access is unreachable from alloc â€” dead code
    72â†’            results[a] = LivenessResult(Proven,
    73â†’                reason="Access is unreachable from allocation; dead code")
    74â†’
    75â†’    return results
    76â†’
    77â†’
    78â†’function IS_REACHABLE_ON_ANY_PATH_THROUGH(cfg, waypoint, target):
    79â†’    // Returns true if there exists any path from entry to target
    80â†’    // that passes through waypoint
    81â†’    return IS_REACHABLE(cfg, waypoint, target)
    82â†’
    83â†’
    84â†’function RESOLVE_BASE_REGION(target, derivations):
    85â†’    // Walk derivation chain until we reach a region allocation
    86â†’    current = target
    87â†’    while current is not a Region:
    88â†’        derivation = FIND_DERIVATION(current, derivations)
    89â†’        if derivation is None:
    90â†’            return None   // Cannot trace back to any allocation
    91â†’        current = derivation.source
    92â†’    return current
    93â†’```
    94â†’
    95â†’### 1.4 Path-Sensitive Analysis
    96â†’
    97â†’The simple reachability check above may produce false positives in cases where the deallocation is conditional. For example, if `free(r)` only occurs on error-handling paths, and an access occurs on the happy path, the access is safe even though a path from `free` to the access technically exists through an infeasible combination of branches. The IVE addresses this with **path feasibility analysis**:
    98â†’
    99â†’```
   100â†’function PATH_SENSITIVE_LIVENESS(cfg, region, access):
   101â†’    // Enumerate paths from alloc_point to access.program_point
   102â†’    paths = ENUMERATE_FEASIBLE_PATHS(cfg, region.alloc_point, access.program_point)
   103â†’
   104â†’    for each path p in paths:
   105â†’        if region.free_point in p.nodes:
   106â†’            // This path goes through free before the access
   107â†’            // Check if free_point dominates the access on this path
   108â†’            free_idx = p.index_of(region.free_point)
   109â†’            access_idx = p.index_of(access.program_point)
   110â†’            if free_idx < access_idx:
   111â†’                return LivenessResult(Violated,
   112â†’                    counterexample=p)
   113â†’
   114â†’    return LivenessResult(Proven)
   115â†’```
   116â†’
   117â†’Path enumeration is bounded to prevent exponential blowup. The IVE uses a k-limiting strategy (default k=5) on loop unrolling and a symbolic execution engine with constraint solving to prune infeasible paths. For programs where full path sensitivity is infeasible, the IVE falls back to flow-sensitive analysis and reports a **confidence level** indicating that the result is sound but imprecise.
   118â†’
   119â†’### 1.5 Complexity
   120â†’
   121â†’- **Worst case:** O(|accesses| Ã— |regions|) for region resolution, multiplied by O(|paths|) for path-sensitive analysis.
   122â†’- **Practical:** Most accesses resolve to a single region in O(1) via the derivation chain index. Path-sensitive analysis is bounded by k-limiting, making the practical cost O(|accesses| Ã— k Ã— |region_lifecycle|).
   123â†’- **Optimization:** The IVE caches region liveness intervals and reuses them across accesses that target the same region, reducing the effective cost toward O(|accesses| + |regions| Ã— |paths|).
   124â†’
   125â†’### 1.6 Output
   126â†’
   127â†’For each access, the algorithm produces one of:
   128â†’- **Proven**: The access targets a live region on all feasible execution paths.
   129â†’- **Violated**: A specific execution path leads from `free` to the access, constituting a use-after-free. The counterexample path is provided.
   130â†’- **Conditional**: The access is safe under specific path conditions that the IVE can enumerate but not prove universally. A confidence score is attached.

   134â†’## 2. Exclusivity Verification Algorithm
   135â†’
   136â†’### 2.1 Purpose and Intuition
   137â†’
   138â†’The exclusivity invariant asserts that no two memory accesses that conflict â€” i.e., at least one is a write and their target address ranges overlap â€” can occur simultaneously without being ordered by a synchronization edge. This is the VUMA answer to data-race freedom. Unlike Rust's borrow checker, which prevents aliasing at the structural level (no two `&mut` references to the same data), the exclusivity verifier checks the actual dynamic access patterns and their synchronization ordering. Two writes to the same address are perfectly acceptable if they are ordered by a happens-before relation â€” the problem only arises when two conflicting accesses might occur concurrently with no ordering constraint.
   139â†’
   140â†’### 2.2 Input
   141â†’
   142â†’- **MSG** containing:
   143â†’  - `accesses`: set of all memory accesses with `target`, `size`, `kind` (Read/Write), and `program_point`
   144â†’  - `sync_edges`: set of synchronization edges (lock acquires/releases, barrier arrivals, channel sends/receives, fork/joins)
   145â†’  - `cfg`: control flow graph
   146â†’
   147â†’### 2.3 Algorithm
   148â†’
   149â†’```
   150â†’function VERIFY_EXCLUSIVITY(msg: MSG) -> Map<(Access, Access), ExclusivityResult>:
   151â†’    results = {}
   152â†’
   153â†’    // Step 1: Build conflict pairs
   154â†’    conflict_pairs = BUILD_CONFLICT_PAIRS(msg.accesses)
   155â†’
   156â†’    // Step 2: Compute happens-before relation
   157â†’    hb = COMPUTE_HAPPENS_BEFORE(msg.sync_edges, msg.cfg)
   158â†’
   159â†’    // Step 3: Check each conflict pair
   160â†’    for each (a1, a2) in conflict_pairs:
   161â†’        if hb.happens_before(a1, a2) or hb.happens_before(a2, a1):
   162â†’            results[(a1, a2)] = ExclusivityResult(Proven,
   163â†’                reason=format("a1 and a2 ordered by happens-before"))
   164â†’        else:
   165â†’            // Check if they can actually execute concurrently
   166â†’            if CAN_EXECUTE_CONCURRENTLY(a1, a2, msg.cfg):
   167â†’                results[(a1, a2)] = ExclusivityResult(Violated,
   168â†’                    reason="Data race: conflicting accesses with no synchronization",
   169â†’                    counterexample=(a1, a2))
   170â†’            else:
   171â†’                results[(a1, a2)] = ExclusivityResult(Proven,
   172â†’                    reason="Accesses are mutually exclusive on all paths")
   173â†’
   174â†’    return results
   175â†’
   176â†’
   177â†’function BUILD_CONFLICT_PAIRS(accesses):
   178â†’    pairs = []
   179â†’    // Optimization: group accesses by region class
   180â†’    region_classes = GROUP_BY_REGION_CLASS(accesses)
   181â†’
   182â†’    for each region_class rc in region_classes:
   183â†’        class_accesses = region_classes[rc]
   184â†’        for i in 0..len(class_accesses):
   185â†’            for j in (i+1)..len(class_accesses):
   186â†’                a1 = class_accesses[i]
   187â†’                a2 = class_accesses[j]
   188â†’                // At least one must be a write
   189â†’                if a1.kind == Read and a2.kind == Read:
   190â†’                    continue
   191â†’                // Address ranges must overlap
   192â†’                if RANGES_OVERLAP(a1.target, a1.size, a2.target, a2.size):
   193â†’                    pairs.append((a1, a2))
   194â†’    return pairs
   195â†’
   196â†’
   197â†’function COMPUTE_HAPPENS_BEFORE(sync_edges, cfg):
   198â†’    // Build transitive closure of synchronization ordering
   199â†’    hb = HappensBeforeGraph()
   200â†’
   201â†’    // Add intra-thread program order
   202â†’    for each thread t in cfg.threads:
   203â†’        for each consecutive pair (n1, n2) in t.execution_order:
   204â†’            hb.add_edge(n1, n2)
   205â†’
   206â†’    // Add inter-thread synchronization order
   207â†’    for each sync_edge (src, dst, kind) in sync_edges:
   208â†’        hb.add_edge(src, dst)
   209â†’        // Propagate: everything before src happens-before everything after dst
   210â†’        for each predecessor p of src:
   211â†’            for each successor s of dst:
   212â†’                hb.add_edge(p, s)
   213â†’
   214â†’    // Compute transitive closure
   215â†’    hb.compute_transitive_closure()
   216â†’    return hb
   217â†’
   218â†’
   219â†’function CAN_EXECUTE_CONCURRENTLY(a1, a2, cfg):
   220â†’    // Two accesses can execute concurrently if they are in different threads
   221â†’    // and neither happens-before the other
   222â†’    if a1.thread == a2.thread:
   223â†’        return false   // Same thread: ordered by program order
   224â†’    return not (hb.happens_before(a1, a2) or hb.happens_before(a2, a1))
   225â†’```
   226â†’
   227â†’### 2.4 Happens-Before Computation Details
   228â†’
   229â†’The happens-before relation is the heart of the exclusivity verification. It is computed as the transitive closure of two ordering relations:
   230â†’
   231â†’1. **Intra-thread program order**: Within a single thread of execution, all operations are totally ordered. This ordering is extracted directly from the CFG for each thread.
   232â†’
   233â†’2. **Inter-thread synchronization order**: Synchronization primitives establish ordering between threads. A `mutex_lock(acquire)` happens-after the corresponding `mutex_unlock(release)`. A `channel_send` happens-before the corresponding `channel_recv`. A `fork` happens-before the first operation of the child thread. A `join` happens-after the last operation of the joined thread. A `barrier` arrival happens-before all subsequent operations on any thread that passed the barrier.
   234â†’
   235â†’The transitive closure is computed using a modified Floyd-Warshall algorithm over the set of program points, yielding an O(|nodes|^3) cost. In practice, the IVE uses a sparse representation and incremental closure computation, reducing the cost to O(|sync_edges| Ã— |threads|).
   236â†’
   237â†’### 2.5 Region Class Optimization
   238â†’
   239â†’The naive algorithm considers all pairs of accesses, yielding O(|accesses|^2) conflict pair candidates. The IVE dramatically reduces this by **grouping accesses by region class** â€” two accesses can only conflict if they target overlapping address ranges, which requires that they target the same region (or aliasing regions). The IVE partitions accesses by their base region (computed during origin verification, Section 4), and only checks pairs within the same partition. This reduces the effective cost to O(Î£_rc |accesses_rc|^2), where the sum is over region classes. For well-structured programs where regions are small and numerous, this is close to O(|accesses|).
   240â†’
   241â†’### 2.6 Complexity
   242â†’
   243â†’- **Worst case:** O(|accesses|^2 Ã— |sync_edges|) â€” quadratic in accesses, linear in synchronization edges.
   244â†’- **With region class optimization:** O(Î£_rc |accesses_rc|^2 Ã— |sync_edges_rc|), typically near-linear in practice.
   245â†’- **Happens-before computation:** O(|program_points|^3) for full transitive closure, but sparse representations and incremental computation reduce this to O(|sync_edges| Ã— |threads|) in practice.
   246â†’
   247â†’### 2.7 Output
   248â†’
   249â†’For each conflicting access pair, the algorithm produces one of:
   250â†’- **Proven**: The accesses are ordered by happens-before, or they are mutually exclusive on all paths (same thread, different branches).
   251â†’- **Violated**: The accesses can execute concurrently with no ordering â€” a data race. Both access program points and the missing synchronization are reported.
   252â†’- **Conditional**: The accesses are ordered under some path conditions but not others. The IVE enumerates the conditions under which safety holds.

   256â†’## 3. Interpretation Verification Algorithm
   257â†’
   258â†’### 3.1 Purpose and Intuition
   259â†’
   260â†’The interpretation invariant asserts that every memory access interprets the target bytes according to a valid Representation Descriptor (RepD). This is the VUMA replacement for type safety. In a traditional language, the type system ensures that you never read an `int` from memory that currently holds a `float` â€” the type of the pointer constrains the interpretation. In VUMA, there are no types; there are only RepDs, and the IVE must verify that the RepD assumed by the access matches the RepD that the memory actually has at that program point.
   261â†’
   262â†’The key challenge is that a single region of memory may be interpreted through multiple RepDs at different times. A buffer allocated as `bytes[128]` may be written as `struct Header` at offset 0 and `float32[28]` at offset 16. The IVE must verify that each read uses a RepD that is compatible with the most recent write's RepD â€” or, if the memory is uninitialized, that the read's RepD is valid for uninitialized memory (typically, only `bytes[N]` or `u8[N]`).
   263â†’
   264â†’### 3.2 Input
   265â†’
   266â†’- **MSG** containing:
   267â†’  - `accesses`: set of all accesses with `target`, `size`, `kind`, `program_point`, and `expected_RepD`
   268â†’  - `derivations`: set of pointer derivations with associated RepD information
   269â†’  - `region_repd_history`: for each region, a chronological log of RepD-altering operations (writes, casts, initializations)
   270â†’
   271â†’### 3.3 Algorithm
   272â†’
   273â†’```
   274â†’function VERIFY_INTERPRETATION(msg: MSG) -> Map<Access, InterpretationResult>:
   275â†’    results = {}
   276â†’
   277â†’    for each access a in msg.accesses:
   278â†’        // Step 1: Determine the target region and offset
   279â†’        target_region = RESOLVE_BASE_REGION(a.target, msg.derivations)
   280â†’        offset = COMPUTE_OFFSET(a.target, msg.derivations)
   281â†’
   282â†’        // Step 2: Get the RepD at the target location at this program point
   283â†’        actual_repd = GET_REPD_AT_POINT(
   284â†’            target_region, offset, a.size, a.program_point, msg)
   285â†’
   286â†’        // Step 3: Get the expected RepD of the access operation
   287â†’        expected_repd = a.expected_RepD
   288â†’
   289â†’        // Step 4: Check compatibility
   290â†’        compat = CHECK_REPD_COMPATIBILITY(actual_repd, expected_repd)
   291â†’
   292â†’        switch compat:
   293â†’            case Compatible:
   294â†’                results[a] = InterpretationResult(Proven,
   295â†’                    reason="RepD compatible: actual matches expected")
   296â†’            case Subsuming:
   297â†’                // actual RepD is a superset of expected (e.g., reading i32 from bytes[4])
   298â†’                results[a] = InterpretationResult(Proven,
   299â†’                    reason="RepD subsuming: expected is a valid projection of actual")
   300â†’            case Reinterpretation:
   301â†’                // A cast is involved; check that it's a valid reinterpretation
   302â†’                if IS_VALID_REINTERPRETATION(actual_repd, expected_repd, offset):
   303â†’                    results[a] = InterpretationResult(Proven,
   304â†’                        reason="Valid RepD reinterpretation after cast")
   305â†’                else:
   306â†’                    results[a] = InterpretationResult(Violated,
   307â†’                        reason="Invalid cast: expected RepD not a valid reinterpretation",
   308â†’                        counterexample=(actual_repd, expected_repd))
   309â†’            case Incompatible:
   310â†’                results[a] = InterpretationResult(Violated,
   311â†’                    reason="RepD incompatible: cannot interpret actual as expected",
   312â†’                    counterexample=(actual_repd, expected_repd))
   313â†’
   314â†’        // Step 5: For reads, check initialization
   315â†’        if a.kind == Read and actual_repd.includes_uninitialized:
   316â†’            // Check that the specific bytes being read have been written
   317â†’            // at the expected RepD before this read
   318â†’            if not HAS_BEEN_WRITTEN_AS(target_region, offset, a.size,
   319â†’                                       expected_repd, a.program_point, msg):
   320â†’                results[a] = InterpretationResult(Violated,
   321â†’                    reason="Read of uninitialized memory as non-trivial RepD",
   322â†’                    counterexample=a)
   323â†’
   324â†’    return results
   325â†’
   326â†’
   327â†’function GET_REPD_AT_POINT(region, offset, size, point, msg):
   328â†’    // Walk the region's RepD history backward from `point`
   329â†’    // to find the most recent RepD-affecting operation
   330â†’    history = msg.region_repd_history[region.id]
   331â†’
   332â†’    // Filter to operations at or before `point` affecting [offset, offset+size)
   333â†’    relevant = [op for op in history
   334â†’                if op.point <= point and
   335â†’                   RANGES_OVERLAP(op.offset, op.size, offset, size)]
   336â†’
   337â†’    if relevant is empty:
   338â†’        // No prior operation: memory is uninitialized
   339â†’        return RepD.Uninitialized(size)
   340â†’
   341â†’    // Merge the RepDs of the most recent writes
   342â†’    // (multiple writes may cover different sub-ranges)
   343â†’    return MERGE_REPDS(relevant, offset, size)
   344â†’
   345â†’
   346â†’function CHECK_REPD_COMPATIBILITY(actual, expected):
   347â†’    if actual == expected:
   348â†’        return Compatible
   349â†’    if actual.is_bytes and expected.size == actual.size:
   350â†’        return Subsuming    // Reading typed data from raw bytes is OK
   351â†’    if actual.is_struct and expected.is_struct and
   352â†’       STRUCTURE_COMPATIBLE(actual, expected):
   353â†’        return Reinterpretation
   354â†’    return Incompatible
   355â†’
   356â†’
   357â†’function IS_VALID_REINTERPRETATION(actual, expected, offset):
   358â†’    // A reinterpretation is valid if:
   359â†’    // 1. The sizes match (no truncation or extension)
   360â†’    // 2. The alignment constraints of expected are satisfied at offset
   361â†’    // 3. The endianness interpretation is consistent
   362â†’    // 4. Any sub-field RepDs of expected are compatible with actual's sub-fields
   363â†’    if expected.size > actual.available_size_at(offset):
   364â†’        return false
   365â†’    if offset % expected.alignment != 0:
   366â†’        return false
   367â†’    if expected.has_pointer_fields and not actual.was_written_as_pointer:
   368â†’        return false    // Can't reinterpret integer as pointer without provenance
   369â†’    return true
   370â†’```
   371â†’
   372â†’### 3.4 Initialization Tracking
   373â†’
   374â†’A critical aspect of interpretation verification is tracking whether memory has been **initialized** at the expected RepD. Reading uninitialized bytes as raw `u8` is always safe (the bytes exist, they just have indeterminate values). But reading uninitialized bytes as a pointer, a floating-point value, or a structure with invariants is unsafe â€” it violates the interpretation invariant because the RepD assumes properties that the uninitialized memory does not have.
   375â†’
   376â†’The IVE maintains a **write log** for each region: the sequence of write operations and their associated RepDs. For each read, the IVE checks whether a prior write at a compatible RepD covers the bytes being read. If the bytes were written as `u8[4]` and are being read as `i32`, the read is valid (subsuming). If the bytes were never written and are being read as `*u8` (a pointer), the read is invalid â€” the pointer provenance cannot be established for uninitialized memory.
   377â†’
   378â†’```
   379â†’function HAS_BEEN_WRITTEN_AS(region, offset, size, expected_repd, point, msg):
   380â†’    writes = msg.region_repd_history[region.id]
   381â†’    // Find all writes before `point` covering [offset, offset+size)
   382â†’    covering_writes = [w for w in writes
   383â†’                       if w.point < point and
   384â†’                          w.kind == Write and
   385â†’                          RANGES_OVERLAP(w.offset, w.size, offset, size)]
   386â†’
   387â†’    // Check that the union of covering writes spans [offset, offset+size)
   388â†’    // and each write's RepD is compatible with expected_repd
   389â†’    covered = IntervalSet()
   390â†’    for w in covering_writes:
   391â†’        if CHECK_REPD_COMPATIBILITY(w.repd, expected_repd) != Incompatible:
   392â†’            covered.add(w.offset, w.size)
   393â†’
   394â†’    return covered.covers(offset, size)
   395â†’```
   396â†’
   397â†’### 3.5 Complexity
   398â†’
   399â†’- **Worst case:** O(|accesses| Ã— |RepD_checks|), where |RepD_checks| is the number of historical RepD entries that must be examined per access.
   400â†’- **With indexing:** The IVE maintains a spatial index (interval tree) over each region's RepD history, reducing lookup to O(log |history|) per access. Total cost: O(|accesses| Ã— log |max_history|).
   401â†’- **Initialization check:** O(|writes_in_region|) per access in the worst case, but the interval tree reduces this to O(log |writes| + |covering_writes|).
   402â†’
   403â†’### 3.6 Output
   404â†’
   405â†’For each access, the algorithm produces one of:
   406â†’- **Proven**: The access's expected RepD is compatible with the actual RepD at the target location, and the memory is properly initialized for the read.
   407â†’- **Violated**: The RepD is incompatible (e.g., reading a pointer from integer memory), or the memory is uninitialized at the expected RepD. The actual RepD and the expected RepD are reported as the counterexample.
   408â†’- **Conditional**: The RepD compatibility depends on path conditions (e.g., a cast occurs on only some paths). The IVE enumerates the conditions.
   409â†’
   410â†’---
   411â†’
   412â†’## 4. Origin Verification Algorithm
   413â†’
   414â†’### 4.1 Purpose and Intuition
   415â†’
   416â†’The origin invariant asserts that every address used in an access can be traced back through a chain of derivations to a valid region allocation. This is the VUMA answer to pointer provenance. In C, a pointer computed by casting an arbitrary integer (e.g., `(int*)0xDEADBEEF`) has no provenance â€” the compiler cannot verify that it points to valid memory. In Rust, such operations require `unsafe`. In VUMA, the IVE traces every address back to its origin and flags any address whose derivation chain cannot be connected to a valid allocation.
   417â†’
   418â†’The derivation chain is a central data structure in the MSG. Every address expression is represented as a node in the derivation graph, with edges connecting derived addresses to their source addresses. The edge labels specify the derivation kind: offset (`base + N`), element access (`base[i]`), field access (`base.field`), arithmetic (`base + stride * index`), or cast (`base as T`). The origin verifier walks this graph from each access's target address back to a root node, which must be a region allocation.
   419â†’
   420â†’### 4.2 Input
   421â†’
   422â†’- **MSG** containing:
   423â†’  - `derivations`: set of all pointer derivations, each with `source`, `operation`, `parameters`, `result`
   424â†’  - `regions`: set of all region allocations
   425â†’
   426â†’### 4.3 Algorithm
   427â†’
   428â†’```
   429â†’function VERIFY_ORIGIN(msg: MSG) -> Map<Access, OriginResult>:
   430â†’    results = {}
   431â†’
   432â†’    for each derivation d in msg.derivations:
   433â†’        // Step 1: Trace derivation chain to root
   434â†’        chain = TRACE_DERIVATION_CHAIN(d, msg.derivations)
   435â†’
   436â†’        // Step 2: Check chain terminates at a valid allocation
   437â†’        root = chain.last()
   438â†’        if root is not a RegionAllocation:
   439â†’            if root is a ConstantAddress:
   440â†’                results[d] = OriginResult(Violated,
   441â†’                    reason="Address derived from constant; no allocation provenance",
   442â†’                    confidence=Tier1_Unsafe,
   443â†’                    counterexample=chain)
   444â†’            else if root is an ExternalInput:
   445â†’                results[d] = OriginResult(Conditional,
   446â†’                    reason="Address from external input; safety depends on runtime value",
   447â†’                    confidence=Tier2_Conditional)
   448â†’            continue
   449â†’
   450â†’        // Step 3: For offset derivations, verify bounds
   451â†’        target_region = root.region
   452â†’        for each step in chain:
   453â†’            if step.operation == Offset:
   454â†’                if step.offset + step.size > target_region.size:
   455â†’                    results[d] = OriginResult(Violated,
   456â†’                        reason="Offset derivation exceeds source region bounds",
   457â†’                        counterexample=(step, target_region.size))
   458â†’                    break
   459â†’
   460â†’            // Step 4: For arithmetic derivations, classify and verify
   461â†’            else if step.operation == Arithmetic:
   462â†’                pattern = CLASSIFY_ARITHMETIC(step)
   463â†’
   464â†’                switch pattern:
   465â†’                    case Linear(base, constant, index):
   466â†’                        // base + constant * index: verify index bounds
   467â†’                        max_index = COMPUTE_MAX_INDEX(index, msg.cfg)
   468â†’                        if constant * max_index + base > target_region.size:
   469â†’                            results[d] = OriginResult(Violated,
   470â†’                                reason="Linear index exceeds region bounds",
   471â†’                                counterexample=(step, max_index))
   472â†’                        else:
   473â†’                            results[d] = OriginResult(Proven,
   474â†’                                reason="Linear derivation within bounds")
   475â†’
   476â†’                    case NonLinear:
   477â†’                        // Non-linear arithmetic: cannot prove bounds statically
   478â†’                        results[d] = OriginResult(Conditional,
   479â†’                            reason="Non-linear pointer arithmetic; cannot verify statically",
   480â†’                            confidence=Tier3_RequiresRuntimeCheck)
   481â†’
   482â†’                    case Symbolic:
   483â†’                        // Symbolic expression: attempt to solve constraints
   484â†’                        bounds = SOLVE_SYMBOLIC_BOUNDS(step, msg.constraints)
   485â†’                        if bounds.is_within(target_region):
   486â†’                            results[d] = OriginResult(Proven,
   487â†’                                reason="Symbolic bounds verified within region")
   488â†’                        else:
   489â†’                            results[d] = OriginResult(Conditional,
   490â†’                                reason="Symbolic bounds cannot be fully resolved",
   491â†’                                confidence=Tier2_Conditional)
   492â†’
   493â†’        if d not in results:
   494â†’            results[d] = OriginResult(Proven,
   495â†’                reason="All derivation steps within bounds")
   496â†’
   497â†’    return results
   498â†’
   499â†’
   500â†’function TRACE_DERIVATION_CHAIN(d, derivations):
   501â†’    // Walk the derivation graph backward from d to the root
   502â†’    chain = [d]
   503â†’    current = d.source
   504â†’
   505â†’    while current is not None and current not in RegionAllocations:
   506â†’        parent = FIND_DERIVATION_BY_RESULT(current, derivations)
   507â†’        if parent is None:
   508â†’            chain.append(current)   // Unresolved: external or constant
   509â†’            break
   510â†’        chain.append(parent)
   511â†’        current = parent.source
   512â†’
   513â†’    return chain
   514â†’
   515â†’
   516â†’function CLASSIFY_ARITHMETIC(step):
   517â†’    // Classify the arithmetic expression into categories:
   518â†’    // - Linear: base + constant * index  (verifiable)
   519â†’    // - NonLinear: any expression with non-linear terms  (flagged)
   520â†’    // - Symbolic: expression with symbolic variables  (constraint-solved)
   521â†’    expr = step.expression
   522â†’
   523â†’    if expr.is_linear():
   524â†’        return Linear(expr.base, expr.coefficient, expr.index)
   525â†’    elif expr.is_symbolic():
   526â†’        return Symbolic(expr)
   527â†’    else:
   528â†’        return NonLinear(expr)
   529â†’```
   530â†’
   531â†’### 4.4 Tiered Confidence System
   532â†’
   533â†’The origin verifier uses a **tiered confidence system** for derivations it cannot fully verify:
   534â†’
   535â†’| Tier | Label | Meaning | Action |
   536â†’|------|-------|---------|--------|
   537â†’| Tier 1 | **Unsafe** | Address has no allocation provenance (constant address, unresolvable derivation) | Flag as violation; access is potentially unsafe |
   538â†’| Tier 2 | **Conditional** | Address derivation is valid but depends on runtime values or external inputs | Flag for runtime check insertion; access is safe under stated conditions |
   539â†’| Tier 3 | **RequiresRuntimeCheck** | Non-linear arithmetic; bounds cannot be proven statically | Insert runtime bounds check; access is safe if check passes |
   540â†’
   541â†’This tiered approach ensures that the IVE never silently allows an unverified access. Every access is either proven safe, flagged as a violation, or guarded by an explicit runtime check that the IVE inserts. The programmer is informed of all Tier 2 and Tier 3 results and can choose to add stronger constraints to promote them to Tier Proven.
   542â†’
   543â†’### 4.5 Complexity
   544â†’
   545â†’- **Worst case:** O(|derivations| Ã— max_chain_length), where max_chain_length is the maximum derivation chain depth.
   546â†’- **Practical:** Most derivation chains are short (1â€“5 steps for typical pointer arithmetic). The IVE caches chain traces, so re-verification of unchanged derivations is O(1).
   547â†’- **Arithmetic classification:** Linear classification is O(1). Symbolic constraint solving is O(|constraints|^2) in the worst case (SMT solver complexity), but the IVE uses incremental solving and caches results.
   548â†’
   549â†’### 4.6 Output
   550â†’
   551â†’For each derivation, the algorithm produces one of:
   552â†’- **Proven**: The derivation chain traces to a valid allocation, and all steps are within bounds.
   553â†’- **Violated**: The chain terminates at a constant address (no provenance) or an intermediate step exceeds region bounds.
   554â†’- **Conditional (Tier 2)**: The chain is valid but depends on runtime conditions.
   555â†’- **RequiresRuntimeCheck (Tier 3)**: Non-linear arithmetic prevents static verification; a runtime bounds check is inserted.
   556â†’
   557â†’---
   558â†’
   559â†’## 5. Cleanup Verification Algorithm
   560â†’
   561â†’### 5.1 Purpose and Intuition
   562â†’
   563â†’The cleanup invariant asserts that every allocated region is eventually freed, unless it is explicitly marked as intentionally leaked. This is the VUMA answer to memory leak detection. Unlike garbage-collected languages, which handle cleanup automatically, and unlike Rust, which enforces cleanup through the ownership model, VUMA permits manual memory management and verifies that cleanup actually occurs. The IVE does not force the programmer to free every allocation â€” long-lived arenas, global caches, and intentionally persistent data structures are marked as `Leaked` by annotation or inference â€” but it does require that every unmarked allocation has a reachable deallocation point.
   564â†’
   565â†’The cleanup verifier also handles **double-free detection**: verifying that no region is freed more than once on any execution path. A double-free is a safety violation because the second free may corrupt the allocator's internal data structures, leading to undefined behavior.
   566â†’
   567â†’### 5.2 Input
   568â†’
   569â†’- **MSG** containing:
   570â†’  - `regions`: set of all regions with `alloc_point`, `free_point` (optional), `leak_status` (optional)
   571â†’  - `cfg`: control flow graph
   572â†’
   573â†’### 5.3 Algorithm
   574â†’
   575â†’```
   576â†’function VERIFY_CLEANUP(msg: MSG) -> Map<Region, CleanupResult>:
   577â†’    results = {}
   578â†’
   579â†’    for each region r in msg.regions:
   580â†’        // --- Leak detection ---
   581â†’
   582â†’        if r.free_point is not None:
   583â†’            // Region has an explicit deallocation point
   584â†’            // Verify the free_point is reachable from alloc_point
   585â†’            if IS_REACHABLE(msg.cfg, r.alloc_point, r.free_point):
   586â†’                results[r] = CleanupResult(Proven,
   587â†’                    reason="Region freed at reachable program point")
   588â†’            else:
   589â†’                results[r] = CleanupResult(Violated,
   590â†’                    reason="Free point is unreachable from allocation (dead free)",
   591â†’                    counterexample=FIND_UNREACHABLE_REASON(msg.cfg, r.alloc_point, r.free_point))
   592â†’
   593â†’        else if r.leak_status == ExplicitlyLeaked:
   594â†’            // Programmer or IVE has marked this region as intentionally leaked
   595â†’            results[r] = CleanupResult(Proven,
   596â†’                reason="Region explicitly marked as intentionally leaked")
   597â†’
   598â†’        else if r.leak_status == InferredLeaked:
   599â†’            // IVE has inferred that this region is a long-lived arena or global cache
   600â†’            // Verify that the inference is sound
   601â†’            if INFERENCE_IS_SOUND(r, msg):
   602â†’                results[r] = CleanupResult(Proven,
   603â†’                    reason="Region inferred as intentionally leaked; inference sound")
   604â†’            else:
   605â†’                results[r] = CleanupResult(Conditional,
   606â†’                    reason="Leak inference may be unsound; review recommended")
   607â†’
   608â†’        else:
   609â†’            // Region has no free point and no leak annotation: potential leak
   610â†’            results[r] = CleanupResult(Violated,
   611â†’                reason="Potential memory leak: no deallocation point and no leak annotation",
   612â†’                counterexample=r.alloc_point)
   613â†’
   614â†’    // --- Double-free detection ---
   615â†’
   616â†’    double_free_results = VERIFY_NO_DOUBLE_FREE(msg)
   617â†’    results.merge(double_free_results)
   618â†’
   619â†’    return results
   620â†’
   621â†’
   622â†’function VERIFY_NO_DOUBLE_FREE(msg: MSG) -> Map<Region, CleanupResult>:
   623â†’    results = {}
   624â†’
   625â†’    for each region r in msg.regions:
   626â†’        if r.free_point is None:
   627â†’            continue   // No free means no double-free
   628â†’
   629â†’        // Find all program points that free this region
   630â†’        free_points = FIND_ALL_FREE_POINTS(r, msg)
   631â†’
   632â†’        if len(free_points) > 1:
   633â†’            // Multiple free points: check if any two can execute on the same path
   634â†’            for each pair (fp1, fp2) in free_points:
   635â†’                if EXISTS_PATH_THROUGH_BOTH(msg.cfg, fp1, fp2):
   636â†’                    violating_path = FIND_PATH(msg.cfg, r.alloc_point, fp1, fp2)
   637â†’                    results[r] = CleanupResult(Violated,
   638â†’                        reason="Double-free: two free points reachable on same path",
   639â†’                        counterexample=violating_path)
   640â†’                    break
   641â†’            else:
   642â†’                // Multiple free points exist but are mutually exclusive
   643â†’                results[r] = CleanupResult(Proven,
   644â†’                    reason="Multiple free points are mutually exclusive (e.g., if-else branches)")
   645â†’
   646â†’        // Check for re-free after free on the same path
   647â†’        // This handles the case where a single free_point is executed,
   648â†’        // then execution loops back and reaches it again
   649â†’        if IS_IN_LOOP(msg.cfg, r.free_point):
   650â†’            // The free point is inside a loop: check if the allocation
   651â†’            // is also inside the same loop (each iteration allocates and frees)
   652â†’            if IS_IN_SAME_LOOP(msg.cfg, r.alloc_point, r.free_point):
   653â†’                results[r] = CleanupResult(Proven,
   654â†’                    reason="Alloc and free in same loop iteration: no double-free")
   655â†’            else:
   656â†’                results[r] = CleanupResult(Violated,
   657â†’                    reason="Free inside loop but alloc outside: potential double-free",
   658â†’                    counterexample=DESCRIBE_LOOP_STRUCTURE(msg.cfg, r.free_point))
   659â†’
   660â†’    return results
   661â†’
   662â†’
   663â†’function FIND_ALL_FREE_POINTS(region, msg):
   664â†’    // Collect all program points that deallocate this region
   665â†’    // This includes explicit free() calls and implicit deallocations
   666â†’    // (e.g., end of scope for stack regions, drop() calls)
   667â†’    free_points = []
   668â†’    for each access a in msg.accesses:
   669â†’        if a.kind == Free and a.target_region == region:
   670â†’            free_points.append(a.program_point)
   671â†’    return free_points
   672â†’```
   673â†’
   674â†’### 5.4 Leak Inference
   675â†’
   676â†’The IVE can automatically infer that a region should be marked as `Leaked` based on several heuristics:
   677â†’
   678â†’1. **Global scope**: If a region is allocated at program initialization and its address is stored in a global variable, the IVE infers it is a long-lived arena.
   679â†’2. **No free in any reachable code**: If the IVE can prove that no `free()` call targeting the region is reachable from any program point, it infers that the region is intentionally leaked.
   680â†’3. **Arena pattern**: If a region is used as an arena (many sub-allocations, bulk deallocation at shutdown), the IVE infers the arena is intentionally leaked until the bulk free.
   681â†’4. **Static lifetime**: If the region's address is captured by a closure or data structure whose lifetime is `'static` (in Rust terms) or equivalent, the IVE infers it is intentionally leaked.
   682â†’
   683â†’These inferences are always marked as `InferredLeaked` (not `ExplicitlyLeaked`) and are subject to the soundness check in the main algorithm. If the inference is later invalidated by a program change, the IVE re-verifies the cleanup invariant for the affected region.
   684â†’
   685â†’### 5.5 Complexity
   686â†’
   687â†’- **Leak detection:** O(|regions| Ã— |paths|), where |paths| is the cost of reachability analysis per region. With CFG caching, this is O(|regions| Ã— |cfg_nodes|) in the worst case.
   688â†’- **Double-free detection:** O(|regions| Ã— |free_points_per_region|^2 Ã— |paths|) â€” quadratic in the number of free points per region, multiplied by path analysis cost.
   689â†’- **Practical:** Most regions have exactly one free point, reducing double-free detection to O(|regions| Ã— |paths|). The total cost is dominated by path analysis, which is linear in CFG size per reachability query.
   690â†’
   691â†’### 5.6 Output
   692â†’
   693â†’For each region, the algorithm produces one of:
   694â†’- **Proven**: The region has a reachable deallocation point, or is explicitly/inferrentially leaked. No double-free exists.
   695â†’- **Violated**: The region has no deallocation point and no leak annotation (leak), or has multiple free points reachable on the same path (double-free). A counterexample path or allocation point is provided.
   696â†’- **Conditional**: The leak inference may be unsound, requiring programmer review.
   697â†’

   699â†’
   700â†’## 6. Overall Verification Pipeline
   701â†’
   702â†’### 6.1 Purpose and Intuition
   703â†’
   704â†’The five verification algorithms do not execute in isolation. They form a **pipeline** with dependencies: the output of one algorithm feeds into the input of another. The pipeline ordering is critical because later algorithms rely on results computed by earlier ones. For example, the liveness verifier needs to know which region an access targets, which is computed by the origin verifier. The exclusivity verifier needs to know which accesses are live, which is computed by the liveness verifier. The interpretation verifier needs the RepD information computed by BD inference, which runs before any VUMA verification.
   705â†’
   706â†’### 6.2 Pipeline Definition
   707â†’
   708â†’```
   709â†’function VERIFY_PROGRAM(scg: SCG) -> VerificationReport:
   710â†’
   711â†’    // Phase 0: Construct the Memory State Graph from the SCG
   712â†’    // This is linear in SCG size: traverse all nodes and edges,
   713â†’    // extract allocation/deallocation/derivation/access information
   714â†’    msg = BUILD_MSG(scg)
   715â†’    // Complexity: O(|scg_nodes| + |scg_edges|)
   716â†’
   717â†’    // Phase 1: Infer Behavioral Descriptors
   718â†’    // The BD inference engine computes RepD, CapD, and RelD for every
   719â†’    // value in the program. This is polynomial in SCG size.
   720â†’    bd_results = INFER_BEHAVIORAL_DESCRIPTORS(scg)
   721â†’    // Complexity: O(|scg_nodes|^2) in worst case for constraint propagation
   722â†’    // Annotate the MSG with inferred RepDs
   723â†’    msg.annotate_with_bd(bd_results)
   724â†’
   725â†’    // Phase 2: Verify Interpretation (uses BD results)
   726â†’    // Check that every access uses a valid RepD for its target memory
   727â†’    interpretation_results = VERIFY_INTERPRETATION(msg)
   728â†’    // Complexity: O(|accesses| * |RepD_checks|)
   729â†’
   730â†’    // Phase 3: Verify Origin (uses derivation chains)
   731â†’    // Trace every address back to a valid allocation
   732â†’    origin_results = VERIFY_ORIGIN(msg)
   733â†’    // Complexity: O(|derivations| * max_chain_length)
   734â†’
   735â†’    // Phase 4: Verify Liveness (uses origin results for region resolution)
   736â†’    // Check that every access targets a live region
   737â†’    // Origin results tell us which region each access targets
   738â†’    msg.annotate_with_origin_results(origin_results)
   739â†’    liveness_results = VERIFY_LIVENESS(msg)
   740â†’    // Complexity: O(|accesses| * |regions|) + path analysis
   741â†’
   742â†’    // Phase 5: Verify Exclusivity (uses liveness results)
   743â†’    // Check that no two concurrent conflicting accesses exist
   744â†’    // Only check accesses that are proven live (dead accesses are irrelevant)
   745â†’    msg.annotate_with_liveness_results(liveness_results)
   746â†’    exclusivity_results = VERIFY_EXCLUSIVITY(msg)
   747â†’    // Complexity: O(|live_accesses|^2 * |sync_edges|)
   748â†’
   749â†’    // Phase 6: Verify Cleanup (independent of other invariants)
   750â†’    // Check that every region is freed or marked leaked
   751â†’    cleanup_results = VERIFY_CLEANUP(msg)
   752â†’    // Complexity: O(|regions| * |paths|)
   753â†’
   754â†’    // Phase 7: Generate proofs for verified invariants
   755â†’    proofs = {}
   756â†’    for each (invariant, results) in [
   757â†’        ("interpretation", interpretation_results),
   758â†’        ("origin",         origin_results),
   759â†’        ("liveness",       liveness_results),
   760â†’        ("exclusivity",    exclusivity_results),
   761â†’        ("cleanup",        cleanup_results)]:
   762â†’        for each (entity, result) in results:
   763â†’            if result.status == Proven:
   764â†’                proofs[(invariant, entity)] = GENERATE_PROOF(invariant, entity, result)
   765â†’
   766â†’    // Phase 8: Generate counterexamples for violations
   767â†’    counterexamples = {}
   768â†’    for each (invariant, results) in [
   769â†’        ("interpretation", interpretation_results),
   770â†’        ("origin",         origin_results),
   771â†’        ("liveness",       liveness_results),
   772â†’        ("exclusivity",    exclusivity_results),
   773â†’        ("cleanup",        cleanup_results)]:
   774â†’        for each (entity, result) in results:
   775â†’            if result.status == Violated:
   776â†’                counterexamples[(invariant, entity)] = GENERATE_COUNTEREXAMPLE(
   777â†’                    invariant, entity, result)
   778â†’
   779â†’    // Phase 9: Compute confidence levels
   780â†’    confidence_map = COMPUTE_CONFIDENCE_LEVELS(
   781â†’        interpretation_results, origin_results,
   782â†’        liveness_results, exclusivity_results, cleanup_results)
   783â†’
   784â†’    return VerificationReport(
   785â†’        proofs=proofs,
   786â†’        counterexamples=counterexamples,
   787â†’        confidence=confidence_map,
   788â†’        summary=SUMMARIZE_RESULTS(all_results))
   789â†’```
   790â†’
   791â†’### 6.3 Pipeline Stage Dependencies
   792â†’
   793â†’```
   794â†’SCG â”€â”€â–º MSG Construction
   795â†’            â”‚
   796â†’            â–¼
   797â†’        BD Inference â”€â”€â–º Interpretation Verification
   798â†’            â”‚
   799â†’            â–¼
   800â†’        Origin Verification â”€â”€â–º Liveness Verification
   801â†’                                        â”‚
   802â†’                                        â–¼
   803â†’                              Exclusivity Verification
   804â†’                                        â”‚
   805â†’        Cleanup Verification â—„â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”˜
   806â†’            â”‚
   807â†’            â–¼
   808â†’        Proof / Counterexample Generation
   809â†’            â”‚
   810â†’            â–¼
   811â†’        Confidence Level Computation
   812â†’```
   813â†’
   814â†’The dependency structure shows that:
   815â†’- **Interpretation** depends only on BD inference (no other VUMA algorithm).
   816â†’- **Origin** depends only on the MSG (derivation chains are self-contained).
   817â†’- **Liveness** depends on origin results (needs to know which region each access targets).
   818â†’- **Exclusivity** depends on liveness results (only live accesses can participate in data races).
   819â†’- **Cleanup** is independent of all other invariants and can run in parallel.
   820â†’
   821â†’This means that interpretation, origin, and cleanup can all begin as soon as the MSG is constructed and BD inference completes. Liveness must wait for origin, and exclusivity must wait for liveness. The critical path is: MSG â†’ BD â†’ Origin â†’ Liveness â†’ Exclusivity.
   822â†’
   823â†’### 6.4 Proof Generation
   824â†’
   825â†’For each proven invariant, the IVE generates a formal proof witness that can be independently checked. The proof format depends on the invariant:
   826â†’
   827â†’| Invariant | Proof Format |
   828â†’|-----------|-------------|
   829â†’| Liveness | Path derivation showing access is between alloc and free on all feasible paths |
   830â†’| Exclusivity | Happens-before chain ordering the conflicting accesses |
   831â†’| Interpretation | RepD compatibility derivation and initialization log |
   832â†’| Origin | Complete derivation chain from access to allocation |
   833â†’| Cleanup | Reachability derivation from alloc to free, or leak annotation record |
   834â†’
   835â†’### 6.5 Counterexample Generation
   836â†’
   837â†’For each violated invariant, the IVE generates a concrete counterexample â€” a specific execution path or pair of paths that leads to the violation. Counterexamples are presented to the programmer through the projection system in human-readable form (not as raw graph paths). The counterexample includes:
   838â†’
   839â†’1. **The violating entity**: The specific access, derivation, or region that fails the invariant.
   840â†’2. **The violation path**: The execution path from program entry to the violation point.
   841â†’3. **The root cause**: The specific step in the path where the invariant breaks (e.g., the free that makes the access use-after-free, the missing lock that creates a data race).
   842â†’4. **Suggested fix**: A machine-generated suggestion for how to fix the violation (e.g., "add lock acquisition before this access" or "move free after last use").
   843â†’
   844â†’### 6.6 Confidence Level Computation
   845â†’
   846â†’Not all verification results are equally certain. The IVE assigns a confidence level to each result:
   847â†’
   848â†’| Level | Meaning | Condition |
   849â†’|-------|---------|-----------|
   850â†’| **High** | Proven by exhaustive analysis | All feasible paths checked, no approximations |
   851â†’| **Medium** | Proven with bounded approximation | k-limited path analysis, sound but incomplete |
   852â†’| **Low** | Provisionally proven | Relies on unsolved constraints or external assumptions |
   853â†’| **Unverified** | Cannot prove or disprove | IVE lacks information to determine safety |
   854â†’
   855â†’The overall program confidence is the minimum confidence across all invariants. The IVE continuously works to raise low-confidence results toward high confidence by performing deeper analysis when resources are available.

   859â†’## 7. Incremental Verification
   860â†’
   861â†’### 7.1 Purpose and Intuition
   862â†’
   863â†’Programs are not static â€” they evolve continuously as the programmer (or AI agent) edits the semantic model. Re-running the full verification pipeline on every edit would be prohibitively expensive. Instead, the IVE supports **incremental verification**: when a single node is added, removed, or modified in the SCG, the IVE rebuilds only the affected portion of the MSG and re-verifies only the invariants that depend on the changed subgraph.
   864â†’
   865â†’Incremental verification is essential for the "always-compiled" model described in the proposal. The programmer should never have to wait for a full re-verification after making a change. The IVE must determine the minimal set of invariants that need re-verification, perform only those checks, and cache the results of all unaffected checks.
   866â†’
   867â†’### 7.2 Change Impact Analysis
   868â†’
   869â†’The first step in incremental verification is determining the **impact scope** of a change â€” which parts of the MSG are affected and which invariants need re-verification.
   870â†’
   871â†’```
   872â†’function ANALYZE_CHANGE_IMPACT(delta: SCGDelta, msg: MSG) -> ImpactReport:
   873â†’    affected_regions = {}
   874â†’    affected_derivations = {}
   875â†’    affected_accesses = {}
   876â†’    invariant_impacts = set()
   877â†’
   878â†’    for each change in delta.changes:
   879â†’        switch change.kind:
   880â†’            case NodeAdded(node):
   881â†’                // A new SCG node may introduce new allocations, accesses, or derivations
   882â†’                if node.is_allocation:
   883â†’                    affected_regions.add(node.region)
   884â†’                    invariant_impacts.add(Cleanup)   // New region needs cleanup check
   885â†’                if node.is_access:
   886â†’                    affected_accesses.add(node.access)
   887â†’                    invariant_impacts.add(Liveness)
   888â†’                    invariant_impacts.add(Interpretation)
   889â†’                    invariant_impacts.add(Exclusivity)
   890â†’                if node.is_derivation:
   891â†’                    affected_derivations.add(node.derivation)
   892â†’                    invariant_impacts.add(Origin)
   893â†’
   894â†’            case NodeRemoved(node):
   895â†’                // Removal may invalidate previously proven invariants
   896â†’                // that depended on this node's region/derivation/access
   897â†’                dependents = FIND_DEPENDENTS(node, msg)
   898â†’                for each dep in dependents:
   899â†’                    if dep.is_region:
   900â†’                        affected_regions.add(dep)
   901â†’                    if dep.is_access:
   902â†’                        affected_accesses.add(dep)
   903â†’                    if dep.is_derivation:
   904â†’                        affected_derivations.add(dep)
   905â†’                invariant_impacts.add(ALL_INVARIANTS)  // Conservative: removal may affect anything
   906â†’
   907â†’            case NodeModified(node, old_props, new_props):
   908â†’                // Modification may change region bounds, access targets, derivation chains
   909â†’                if old_props.region_bounds != new_props.region_bounds:
   910â†’                    affected_regions.add(node.region)
   911â†’                    invariant_impacts.add(Origin)   // Bounds change affects origin
   912â†’                    invariant_impacts.add(Cleanup)  // Bounds change may affect cleanup
   913â†’                if old_props.access_target != new_props.access_target:
   914â†’                    affected_accesses.add(node.access)
   915â†’                    invariant_impacts.add(Liveness)
   916â†’                    invariant_impacts.add(Interpretation)
   917â†’                    invariant_impacts.add(Exclusivity)
   918â†’                if old_props.derivation != new_props.derivation:
   919â†’                    affected_derivations.add(node.derivation)
   920â†’                    invariant_impacts.add(Origin)
   921â†’                    invariant_impacts.add(Liveness)   // Origin change affects liveness
   922â†’
   923â†’            case EdgeAdded(edge) | EdgeRemoved(edge) | EdgeModified(edge):
   924â†’                // Edges represent control flow or synchronization
   925â†’                if edge.is_sync_edge:
   926â†’                    invariant_impacts.add(Exclusivity)   // Sync change affects exclusivity
   927â†’                if edge.is_control_flow:
   928â†’                    // Control flow changes affect all path-dependent invariants
   929â†’                    invariant_impacts.add(Liveness)
   930â†’                    invariant_impacts.add(Cleanup)
   931â†’
   932â†’    return ImpactReport(
   933â†’        affected_regions=affected_regions,
   934â†’        affected_derivations=affected_derivations,
   935â†’        affected_accesses=affected_accesses,
   936â†’        invariant_impacts=invariant_impacts)
   937â†’```
   938â†’
   939â†’### 7.3 Incremental MSG Rebuild
   940â†’
   941â†’Once the impact scope is determined, the IVE rebuilds only the affected portion of the MSG. This is done by:
   942â†’
   943â†’1. **Identifying the affected subgraph**: The MSG subgraph that includes all regions, derivations, and accesses in the impact report.
   944â†’2. **Rebuilding the subgraph**: Re-extracting allocation, derivation, and access information from the changed SCG nodes.
   945â†’3. **Splicing the subgraph**: Replacing the old subgraph in the MSG with the rebuilt version, while preserving all unaffected subgraphs and their cached verification results.
   946â†’
   947â†’```
   948â†’function INCREMENTAL_MSG_REBUILD(delta, msg, impact):
   949â†’    // Step 1: Remove affected subgraph
   950â†’    for each region r in impact.affected_regions:
   951â†’        msg.remove_region_subgraph(r)
   952â†’    for each derivation d in impact.affected_derivations:
   953â†’        msg.remove_derivation_subgraph(d)
   954â†’    for each access a in impact.affected_accesses:
   955â†’        msg.remove_access_subgraph(a)
   956â†’
   957â†’    // Step 2: Rebuild from changed SCG nodes
   958â†’    for each change in delta.changes:
   959â†’        new_subgraph = EXTRACT_MSG_SUBGRAPH(change.node, scg)
   960â†’        msg.splice_subgraph(new_subgraph)
   961â†’
   962â†’    // Step 3: Update cross-references
   963â†’    // Derivations that cross region boundaries may need updating
   964â†’    for each derivation d in msg.derivations:
   965â†’        if d.crosses_boundary(impact.affected_regions):
   966â†’            d.update_cross_references(msg)
   967â†’
   968â†’    return msg
   969â†’```
   970â†’
   971â†’### 7.4 Incremental Re-verification
   972â†’
   973â†’With the rebuilt MSG and the impact report, the IVE re-verifies only the affected invariants:
   974â†’
   975â†’```
   976â†’function INCREMENTAL_VERIFY(delta, msg, cached_results):
   977â†’    impact = ANALYZE_CHANGE_IMPACT(delta, msg)
   978â†’    msg = INCREMENTAL_MSG_REBUILD(delta, msg, impact)
   979â†’
   980â†’    new_results = copy(cached_results)
   981â†’
   982â†’    // Re-verify only impacted invariants, in dependency order
   983â†’    if Interpretation in impact.invariant_impacts:
   984â†’        new_results.interpretation = VERIFY_INTERPRETATION(msg)
   985â†’            .restricted_to(impact.affected_accesses)
   986â†’        // Merge: keep cached results for unaffected accesses
   987â†’        new_results.interpretation = MERGE_RESULTS(
   988â†’            cached_results.interpretation,
   989â†’            new_results.interpretation,
   990â†’            unaffected=cached_results.interpretation.keys() - impact.affected_accesses)
   991â†’
   992â†’    if Origin in impact.invariant_impacts:
   993â†’        new_results.origin = VERIFY_ORIGIN(msg)
   994â†’            .restricted_to(impact.affected_derivations)
   995â†’        new_results.origin = MERGE_RESULTS(
   996â†’            cached_results.origin,
   997â†’            new_results.origin,
   998â†’            unaffected=cached_results.origin.keys() - impact.affected_derivations)
   999â†’
  1000â†’    if Liveness in impact.invariant_impacts:
  1001â†’        // Liveness depends on origin; if origin changed, re-verify liveness
  1002â†’        // for accesses whose region resolution may have changed
  1003â†’        live_accesses = impact.affected_accesses
  1004â†’        if Origin in impact.invariant_impacts:
  1005â†’            live_accesses += FIND_ACCESSES_DEPENDENT_ON(impact.affected_derivations, msg)
  1006â†’        new_results.liveness = VERIFY_LIVENESS(msg)
  1007â†’            .restricted_to(live_accesses)
  1008â†’        new_results.liveness = MERGE_RESULTS(
  1009â†’            cached_results.liveness,
  1010â†’            new_results.liveness,
  1011â†’            unaffected=cached_results.liveness.keys() - live_accesses)
  1012â†’
  1013â†’    if Exclusivity in impact.invariant_impacts:
  1014â†’        // Exclusivity depends on liveness; if liveness changed, re-verify
  1015â†’        // exclusivity for conflict pairs involving changed accesses
  1016â†’        live_changed = impact.affected_accesses
  1017â†’        if Liveness in impact.invariant_impacts:
  1018â†’            live_changed += FIND_ACCESSES_WITH_CHANGED_LIVENESS(new_results, cached_results)
  1019â†’        conflict_pairs = FIND_CONFLICT_PAIRS_INVOLVING(live_changed, msg)
  1020â†’        new_results.exclusivity = VERIFY_EXCLUSIVITY(msg)
  1021â†’            .restricted_to(conflict_pairs)
  1022â†’        new_results.exclusivity = MERGE_RESULTS(
  1023â†’            cached_results.exclusivity,
  1024â†’            new_results.exclusivity,
  1025â†’            unaffected=cached_results.exclusivity.keys() - conflict_pairs)
  1026â†’
  1027â†’    if Cleanup in impact.invariant_impacts:
  1028â†’        new_results.cleanup = VERIFY_CLEANUP(msg)
  1029â†’            .restricted_to(impact.affected_regions)
  1030â†’        new_results.cleanup = MERGE_RESULTS(
  1031â†’            cached_results.cleanup,
  1032â†’            new_results.cleanup,
  1033â†’            unaffected=cached_results.cleanup.keys() - impact.affected_regions)
  1034â†’
  1035â†’    return new_results
  1036â†’```
  1037â†’
  1038â†’### 7.5 Caching Strategy
  1039â†’
  1040â†’The IVE maintains a **verification cache** that stores the results of all invariant checks keyed by the MSG subgraph they depend on. When a change occurs:
  1041â†’
  1042â†’1. **Cache invalidation**: Results that depend on the changed subgraph are invalidated. The IVE uses a fine-grained dependency map (each result records which MSG elements it depends on) to minimize invalidation.
  1043â†’2. **Cache reuse**: Results that do not depend on the changed subgraph are reused without re-computation. This is the primary source of incremental speedup.
  1044â†’3. **Cache compaction**: Over time, the cache may accumulate results for subgraphs that are no longer part of the MSG (e.g., after a deletion). The IVE periodically compacts the cache by removing entries whose key subgraphs are no longer present.
  1045â†’
  1046â†’The cache is implemented as a content-addressed store: the key is a hash of the MSG subgraph, and the value is the verification result. This allows the IVE to detect when a rebuilt subgraph is identical to a previously verified one (e.g., after an undo operation) and reuse the cached result directly.
  1047â†’
  1048â†’### 7.6 Complexity of Incremental Update
  1049â†’
  1050â†’- **Change impact analysis:** O(|delta| Ã— |dependency_map_lookup|) â€” linear in the size of the change, with constant-time dependency lookups.
  1051â†’- **MSG rebuild:** O(|affected_subgraph|) â€” proportional to the size of the changed subgraph, not the entire MSG.
  1052â†’- **Re-verification:** O(|affected_entities| Ã— verification_cost_per_entity) â€” only the affected accesses, derivations, and regions are re-verified.
  1053â†’- **Overall:** O(delta_size Ã— log(program_size)) â€” the log factor comes from dependency map lookups and cache indexing. This is exponentially faster than full re-verification, which is O(program_size).
  1054â†’
  1055â†’In practice, for a single-line edit that affects one region and a handful of accesses, the incremental verification completes in milliseconds, enabling the "always-verified" experience described in the proposal.
